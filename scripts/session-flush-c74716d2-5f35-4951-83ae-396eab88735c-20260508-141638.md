**CLAUDE.md is for project truth that applies to everyone.** Build commands (`npm run build`, `npm test`), quality gates that must pass before commit, coding style rules, architectural decisions, known gotchas, and documentation requirements. If a new developer joined your team tomorrow, they'd need this information. If you removed it, Claude would make incorrect assumptions about your project. CLAUDE.md content is objective and verifiable — you can point to the codebase and say "this rule exists because of that code."

**Claude Memory is for context that's personal, temporal, or soft.** Your role as a data scientist investigating logging. The fact that you prefer terse responses without trailing summaries. That the team has a merge freeze starting Thursday for a mobile release. That pipeline bugs are tracked in the "INGEST" Linear project. These facts are true and useful, but they don't belong in a file that every contributor reads, and they may become stale as circumstances change.

Claude Memory also has explicit exclusions — it will not save code patterns, architecture details, file paths, git history, or debugging solutions that can be derived from the current project state. This is deliberate: memory supplements the codebase, it doesn't duplicate it. If you ask Claude to "remember that we use the repository pattern," it should recognize that this is observable from the code and doesn't need a memory file.

The boundary gets tested most often with project decisions. "We chose Postgres over DynamoDB because of complex query requirements" — does this belong in CLAUDE.md or memory? The answer depends on whether the decision is still load-bearing. If it's an active architectural constraint that affects how Claude writes code, it belongs in CLAUDE.md. If it's historical context explaining a past choice, a project memory captures it without cluttering the shared instruction file.

As detailed in our [deep dive on the Claude Code memory system](/blog/claude-code-memory), the auto memory architecture uses a file-per-memory approach with frontmatter metadata (name, description, type) so that Claude can quickly assess relevance when loading context for a new conversation. This design means individual memories can be updated or deleted without affecting others — unlike a monolithic CLAUDE.md where removing one section risks breaking the flow of surrounding instructions.

## Team Collaboration and Scaling

For solo developers, the distinction between Claude Memory and CLAUDE.md is mostly organizational convenience. For teams, it becomes essential infrastructure.

**CLAUDE.md scales horizontally across people.** When your team's CLAUDE.md says "run `npm test` before every commit" and "never import Next.js modules in pipeline scripts," every developer's Claude Code session enforces those rules identically. New team members get the same guardrails from their first session. Code review becomes easier because Claude follows the same conventions regardless of who's prompting it. Changes to standards go through pull requests, creating an audit trail of why rules were added or modified.

**Claude Memory scales vertically within one person.** Over weeks of working in a codebase, Claude accumulates understanding of your specific workflow, expertise gaps, communication preferences, and current focus areas. A senior backend engineer and a junior frontend developer working in the same repo will have completely different memory profiles — and Claude will adapt its explanations, code suggestions, and autonomy level accordingly. This is the kind of personalization that would be inappropriate in a shared CLAUDE.md.

The interaction between these two systems creates a powerful dynamic. CLAUDE.md provides the "what" — what standards to follow, what commands to run, what patterns to use. Auto memory provides the "how" — how to communicate with this specific developer, how to prioritize based on their current project, how to calibrate technical depth to their expertise level.

Teams that try to use only CLAUDE.md often end up with bloated instruction files full of individual preferences that conflict with each other. Teams that rely only on auto memory find that each developer's Claude behaves differently on basic conventions, leading to inconsistent code quality. The [Claude Code extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) is designed for these layers to work together, not in isolation.

## Practical Setup: Getting Both Systems Right

Setting up CLAUDE.md and Claude Memory correctly from the start saves significant friction later. Here's how to approach each system in practice.

### Setting Up CLAUDE.md

Start minimal. A CLAUDE.md with three sections covers 80% of what you need:

```markdown
# Project Name

## Commands
npm run dev          # Local dev server
npm run build        # Production build
npm test             # Test suite
npm run lint         # Linting

## Rules
- Run build + tests before every commit
- Never skip failing tests

## Gotchas
- Don't import server-only modules in shared code
```

Add rules only when Claude gets something wrong that it shouldn't. If Claude uses the wrong test framework, add it. If Claude imports a banned dependency, add it. Let CLAUDE.md grow organically from real problems rather than trying to anticipate every scenario. The article [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) covers this iterative approach in detail.

For larger projects, consider subdirectory CLAUDE.md files for module-specific rules. A `packages/api/CLAUDE.md` can specify API-layer conventions without cluttering the root file with rules that don't apply to the frontend.

### Working With Claude Memory

Auto memory requires less setup because Claude manages it. But you can influence what gets remembered:

- **Explicitly ask Claude to remember** things that affect future sessions: "Remember that I'm a data scientist, not a frontend developer" or "Remember that our staging environment uses a different database"
- **Correct Claude and it will save feedback**: When you say "don't mock the database in tests," Claude stores this as a feedback memory with the reasoning, so it applies the rule in future conversations
- **Review your memories periodically**: Check `~/.claude/projects/*/memory/` to see what Claude has accumulated. Delete stale memories, update incorrect ones

The most valuable memories tend to be feedback corrections and reference pointers — the kind of information that prevents you from repeating yourself across sessions. User memories about your role and expertise also compound well, as they help Claude calibrate its responses over weeks and months of collaboration.

## When to Choose CLAUDE.md

Use CLAUDE.md when the information meets all three criteria:

1. **It applies to everyone** — any developer on the team needs this, not just you
2. **It's stable** — the rule won't change week to week (coding standards, build commands, architectural constraints)
3. **It's actionable** — Claude should follow this rule deterministically, not use it as a soft signal

Specific scenarios where CLAUDE.md is the right choice:

- Build, test, and lint commands
- Quality gates and pre-commit requirements
- Coding style rules (naming conventions, file organization, import patterns)
- Banned patterns or dependencies
- Known bugs and workarounds that affect AI-generated code
- Documentation requirements and frontmatter schemas
- Deployment procedures and environment-specific gotchas

If you've used [Claude Code Skills](/blog/5-claude-code-skills-i-use-every-single-day) (SKILL.md files), note that these complement CLAUDE.md by providing task-specific instructions — a newsletter skill, a code review skill, a test generation skill — while CLAUDE.md provides the project-wide baseline. The relationship between Skills and CLAUDE.md is similar to the relationship between function-level and module-level configuration.

## When to Choose Claude Memory

Use Claude Memory when the information meets any of these criteria:

1. **It's personal** — it's about you, not the project
2. **It's temporal** — it reflects current state that will change (deadlines, current sprint focus, ongoing incidents)
3. **It's observational** — Claude learned it from your behavior patterns, not from a written rule
4. **It points elsewhere** — it's a reference to where information lives (Linear, Slack, Grafana), not the information itself

Specific scenarios where Claude Memory is the right choice:

- Your role, expertise level, and communication preferences
- Current project focus ("I'm working on the auth migration this week")
- Corrections to Claude's approach that reflect your personal workflow
- Pointers to external systems (bug trackers, dashboards, documentation portals)
- Team dynamics and deadlines ("merge freeze starts Thursday")
- Historical decisions that explain why code looks the way it does

The key insight: if removing this information would make Claude worse for **you specifically** but wouldn't affect a teammate's experience, it belongs in memory. If removing it would make Claude worse for **everyone on the project**, it belongs in CLAUDE.md.

## Verdict

**Use both — they solve different problems.** CLAUDE.md is your project's instruction manual: deterministic, shared, version-controlled. Claude Memory is your personal context layer: adaptive, private, conversation-derived. Trying to use one without the other either limits personalization (CLAUDE.md only) or sacrifices team consistency (memory only).

**Start with CLAUDE.md** — even a 10-line file with build commands and top rules dramatically improves Claude Code's output. Let auto memory accumulate naturally as you work. Review both periodically: prune CLAUDE.md rules that no longer apply, delete stale memories about completed projects or past deadlines.

For teams, establish a convention: CLAUDE.md changes go through code review just like any other config file. Individual memories stay personal. This separation keeps the shared foundation clean while letting each developer build their own productive relationship with Claude Code. For a deeper look at how these systems fit into the broader architecture, see our coverage of [what makes Claude Code special](/blog/whats-so-special-about-the-claude-code) as a programmable AI platform.

## Frequently Asked Questions

### Can Claude Memory override rules in CLAUDE.md?

No. CLAUDE.md instructions take precedence as explicit project rules. Claude Memory provides soft context that informs Claude's judgment within the boundaries set by CLAUDE.md. If CLAUDE.md says "use Vitest," a memory saying you prefer Jest won't override that — Claude will follow the project rule and may note the conflict.

### Does Claude Memory sync across machines?

No. Claude Memory is stored locally in `~/.claude/projects/` and is not synced across devices or shared with teammates. If you work from multiple machines, each will develop its own memory profile. CLAUDE.md, being in the repo, syncs everywhere you pull the code.

### How do I migrate information from Claude Memory to CLAUDE.md?

Check your memory files at `~/.claude/projects/*/memory/` for patterns that apply to the whole team. If you see feedback memories like "don't mock the database" or "always run integration tests," those are candidates for CLAUDE.md rules. Copy the rule (not the personal framing) into CLAUDE.md, commit it, and optionally delete the memory file since CLAUDE.md will now handle enforcement.

### Should I commit CLAUDE.md to my repository?

Yes. The project-level CLAUDE.md at your repo root should be committed and code-reviewed like any configuration file. This ensures all team members and all Claude Code sessions follow the same rules. Your personal global CLAUDE.md at `~/.claude/CLAUDE.md` stays local — it contains preferences that apply across all your projects.

### How many memories can Claude store before performance degrades?

Claude loads the `MEMORY.md` index at the start of each conversation, with a practical limit of around 200 lines before truncation. Individual memory files are loaded based on relevance. Keep memories focused and prune outdated ones — a dozen well-maintained memories outperform a hundred stale ones.
