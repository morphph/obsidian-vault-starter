**CLAUDE.md ages like code.** It's updated deliberately, reviewed in PRs, and maintained alongside the codebase. When a build command changes, someone updates CLAUDE.md. When a convention is retired, someone removes the rule. The maintenance burden scales with project complexity but follows established code review practices. Teams that treat CLAUDE.md as a living document — updating it when conventions change, pruning outdated rules — get consistent AI behavior. Teams that write it once and forget it accumulate stale instructions that confuse more than they help.

**Claude Memory ages like notes.** Memories accumulate automatically, and while Claude Code is designed to update or remove stale memories, the process is imperfect. A memory noting "currently working on the auth refactor" might persist long after the refactor is complete. A feedback memory about a since-fixed bug might incorrectly prevent Claude from using a now-valid pattern.

The mitigation for memory staleness is built into the system: Claude Code is instructed to verify memory content against current state before acting on it. If a memory says "function X exists in file Y," Claude should check that it still does before recommending it. But this verification adds latency and isn't foolproof.

For CLAUDE.md, the mitigation is human discipline — the same discipline that keeps README files and documentation current. The advantage is that CLAUDE.md staleness is visible (anyone can read the file and spot outdated rules), while memory staleness is hidden (only the individual user encounters it).

## Context Window Economics

Both systems consume tokens from Claude's context window, but their consumption patterns differ significantly.

**CLAUDE.md has a fixed cost.** Every word in your CLAUDE.md files is loaded into every conversation turn. A 500-line CLAUDE.md file might consume 2,000-3,000 tokens per turn. For a complex project with detailed conventions, this is a worthwhile investment — the instructions prevent mistakes that would cost far more tokens to diagnose and fix. But overstuffed CLAUDE.md files waste context on rarely-relevant information, leaving less room for actual code and conversation.

Best practice: keep CLAUDE.md focused on **rules and commands**, not explanations. "Use `pnpm` not `npm`" is better than a three-paragraph explanation of why your monorepo uses pnpm. If the reasoning matters, link to an ADR or documentation.

**Claude Memory has a variable cost.** The `MEMORY.md` index is always loaded (typically 50-200 lines), but individual memory files are only loaded when relevant. This means 30 detailed memories about different project aspects might only cost 100 tokens on average per turn (the index), spiking to 500-800 tokens when a relevant memory is loaded. This is far more efficient than putting all that information in CLAUDE.md.

The practical implication: **move contextual information out of CLAUDE.md and into memory**. If your CLAUDE.md contains sections like "Current team priorities" or "Recent architecture decisions" that are only relevant sometimes, they're better stored as memory entries that load on demand.

## Integration with Claude Code's Extension Stack

Both CLAUDE.md and Claude Memory interact with Claude Code's broader [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp), but they play different roles.

**CLAUDE.md integrates tightly with skills and hooks.** When you define a skill in `skills/*/SKILL.md`, the skill instructions layer on top of CLAUDE.md — the project-level rules still apply while the skill-specific instructions add domain behavior. Hooks defined in `settings.json` can enforce CLAUDE.md rules deterministically (e.g., a pre-commit hook that runs the linter, matching the CLAUDE.md rule that requires linting). This creates a layered system where CLAUDE.md defines the rules, skills define specialized workflows, and hooks enforce critical constraints. For practical examples, see our [guide to skills that improve agent output](/blog/do-skills-actually-improve-your-agents-output).

**Claude Memory integrates with conversation flow.** Memories shape how Claude Code approaches tasks — a feedback memory saying "user prefers single bundled PRs over many small ones" influences how Claude Code proposes work organization. A project memory about an upcoming deadline influences urgency and scope decisions. These effects are subtle and contextual, influencing behavior rather than dictating it.

The relationship between the two: CLAUDE.md sets the **floor** (minimum standards, required behaviors), while memory raises the **ceiling** (personalized, context-aware assistance). A well-configured project has strict CLAUDE.md rules preventing mistakes and rich memory stores enabling efficiency.

## Migration Path: Moving Information Between Layers

As projects mature, information often needs to move between CLAUDE.md and memory.

**Memory → CLAUDE.md** (promotion): When a personal preference turns out to be a team standard, promote it. If your memory says "always run `validate-pipeline.ts` before committing pipeline changes" and the team agrees this should be universal, add it to CLAUDE.md and remove it from your personal memory. The signal that promotion is needed: you find yourself telling multiple team members the same thing.

**CLAUDE.md → Memory** (demotion): When a CLAUDE.md rule is too specific or temporary, demote it. A rule like "merge freeze until March 5 for mobile release" doesn't belong in the permanent project configuration. Store it as a project memory with an expiration date, or better yet, remove it entirely once the freeze lifts. The signal that demotion is needed: the CLAUDE.md file is growing past 100 lines with context that's only relevant this quarter.

**Neither → Documentation**: Some information doesn't belong in either system. Architectural explanations, onboarding guides, and API documentation belong in actual docs — not crammed into an AI instruction file. CLAUDE.md should reference docs ("see `docs/ARCHITECTURE.md` for the data flow"), not replace them.

## When to Choose Claude Memory

Choose Claude Memory as your primary persistence layer when:

- **You work solo** and don't need to share AI configuration with a team. Memory accumulates naturally and doesn't require deliberate maintenance.
- **Context is personal** — your role, your preferences, your feedback about how Claude Code should behave with you specifically.
- **Information is temporal** — current priorities, in-flight decisions, and project state that will change within weeks.
- **You want zero maintenance** — auto memory builds itself. You don't need to write or update anything; Claude Code learns from your conversations.
- **Context is reference-heavy** — pointers to external systems (dashboards, issue trackers, Slack channels) that are useful when relevant but shouldn't consume context tokens every turn.

The risk: memory staleness and the bus factor. Mitigate by periodically reviewing your memory files and promoting team-relevant information to CLAUDE.md or documentation.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence layer when:

- **You work on a team** and need consistent AI behavior across all developers. CLAUDE.md is the only mechanism that guarantees every session follows the same rules.
- **Rules are non-negotiable** — quality gates, coding standards, security constraints, and architectural decisions that must never be forgotten or misapplied.
- **Information is stable** — build commands, project structure, and conventions that change infrequently and only through deliberate decisions.
- **Reliability matters more than personalization** — CLAUDE.md content is always present; it can't be missed by a retrieval heuristic.
- **You need auditability** — CLAUDE.md changes are tracked in git history, reviewable in PRs, and attributable to specific authors. For details on crafting effective project instructions, see the [complete guide to Claude Code](/blog/claude-code-complete-guide).

The risk: bloat and staleness. Mitigate by treating CLAUDE.md like code — review it regularly, prune outdated rules, and keep it focused on actionable instructions.

## Verdict

**Use both — but be intentional about what goes where.** CLAUDE.md is your project's constitution: the shared, explicit, version-controlled rules that define how Claude Code must behave for everyone on the team. Claude Memory is your personal assistant's notebook: the accumulated context about you, your preferences, and the current state of your work that makes Claude Code increasingly effective over time.

The decision rule is simple: **if it's a rule, put it in CLAUDE.md. If it's context, let memory handle it.** If your CLAUDE.md is growing past 100 lines, you're probably stuffing context into the wrong layer. If your memory contains rules that the whole team should follow, promote them to CLAUDE.md. The [memory system architecture](/blog/claude-code-memory) supports both approaches — the key is matching the persistence mechanism to the information type.

Start with a focused CLAUDE.md covering build commands, quality gates, and critical conventions. Let memory accumulate naturally as you work. Review both quarterly to prune staleness and promote or demote information between layers.

## Frequently Asked Questions

### Can Claude Memory and CLAUDE.md conflict with each other?

Yes. If CLAUDE.md says "use npm" and a feedback memory says "user prefers pnpm," Claude Code will follow CLAUDE.md because explicit project instructions override personal memory. CLAUDE.md acts as the authoritative source; memory provides supplementary context that yields to explicit rules when they conflict.

### Does Claude Memory work across different machines?

No. Claude Memory is stored locally in `.claude/projects/` on your machine and is not synced across devices. If you work from multiple machines, your memory stores will diverge. CLAUDE.md, being checked into the repo, follows you anywhere you clone the project. Anthropic's [memory import features](/blog/anthropic-claude-memory-upgrades-importing) may address this gap over time.

### How much should I put in CLAUDE.md before it becomes counterproductive?

Keep project-level CLAUDE.md under 100 lines of actionable instructions. Beyond that, you're likely including contextual information that would be better served as memory entries or documentation. Every line in CLAUDE.md costs tokens on every conversation turn — make each line earn its place by being a rule that applies universally and frequently.

### Can I manually edit Claude Memory files?

Yes. Memory files are plain markdown stored in `.claude/projects/*/memory/`. You can edit, delete, or reorganize them at any time. This is useful for cleaning up stale memories, correcting inaccurate information, or bulk-importing context when starting a new project.

### Should skills (SKILL.md) go in CLAUDE.md or memory?

Neither — skills have their own dedicated system. **SKILL.md** files live in `skills/*/` directories and are loaded on demand when invoked. They sit between CLAUDE.md (always loaded) and memory (contextually loaded) in the persistence hierarchy. CLAUDE.md sets global rules, skills define specialized workflows, and memory provides personal context. See [5 Claude Code skills I use every day](/blog/5-claude-code-skills-i-use-every-single-day) for practical examples.
