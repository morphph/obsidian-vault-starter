**CLAUDE.md** is a markdown file checked into the root of your repository that gives Claude Code project-level instructions. Think of it as a README for your AI agent — it defines build commands, coding standards, architectural constraints, testing requirements, and anything else Claude needs to know to work effectively in your codebase. Every developer on the team gets the same instructions because the file travels with the repo.

CLAUDE.md follows a simple convention: place a `CLAUDE.md` file at your project root, and Claude Code reads it automatically at the start of every session. You can also place additional CLAUDE.md files in subdirectories for module-specific instructions. The file is plain markdown with no required schema — you write whatever instructions matter for your project. Common contents include build/test commands, style guidelines, forbidden patterns, and links to relevant documentation.

The key property of CLAUDE.md is that it is **deterministic and shared**. Every team member, every CI run, every Claude Code session gets the same instructions. For a comprehensive walkthrough of how CLAUDE.md fits into Claude Code's architecture, see our [complete guide to Claude Code](/blog/claude-code-complete-guide).

## How Context Flows: The Mental Model

Understanding when to use each system requires understanding how Claude Code processes context at the start of a session. The loading order matters because it determines what takes priority.

When you launch Claude Code in a project directory, it reads context in this order:

1. **Global CLAUDE.md** (`~/.claude/CLAUDE.md`) — your personal defaults for all projects
2. **Project CLAUDE.md** (`./CLAUDE.md`) — repo-level instructions
3. **Auto-memory** (`~/.claude/projects/<project>/memory/`) — learned context from prior sessions
4. **Subdirectory CLAUDE.md files** — loaded as you navigate into specific directories

This layered system means CLAUDE.md sets the rules, and Memory fills in the personal context. A CLAUDE.md file might say "use Vitest for testing," while Memory knows you prefer to see test output in verbose mode. They don't compete — they operate at different levels of specificity.

The practical implication: if you find yourself repeatedly telling Claude the same thing across sessions, ask whether that information belongs in CLAUDE.md (if it applies to anyone working on this project) or Memory (if it's specific to you). Getting this distinction right eliminates most repeated-instruction friction.

## Project-Level Instructions: Where CLAUDE.md Excels

CLAUDE.md is the right tool whenever the instruction applies to the project, not the person. This covers a wide range of engineering context that would otherwise live only in team members' heads.

**Build and test commands** are the most common CLAUDE.md content. Instead of Claude guessing whether your project uses `npm test`, `pnpm test`, or `make test`, CLAUDE.md states it explicitly. This eliminates a category of errors where Claude runs the wrong command and produces confusing failures.

**Coding standards and forbidden patterns** prevent Claude from introducing inconsistencies. If your team uses a specific error-handling pattern, a particular naming convention, or avoids certain libraries, CLAUDE.md enforces these across every Claude Code session. Without it, each developer's Claude might write code in slightly different styles.

**Architecture constraints** capture decisions that aren't obvious from the code alone. "Don't import Next.js modules in pipeline scripts" or "always use CJK word count for Chinese content" are the kinds of rules that developers learn through painful debugging. CLAUDE.md encodes them once. The [skills system](/blog/5-claude-code-skills-i-use-every-single-day) extends this further with task-specific instruction files.

**Quality gates** define what must pass before Claude commits code. A CLAUDE.md that says "run build, tests, and lint before any commit" turns Claude Code from a code generator into a standards-enforcing workflow. See our guide on [writing effective Claude Code skills](/blog/9-principles-writing-claude-code-skills) for techniques that apply equally to CLAUDE.md authoring.

The limitation of CLAUDE.md is maintenance. It's a static file that you must update manually when your project evolves. If you change your test framework and forget to update CLAUDE.md, Claude will keep running the old commands. Memory doesn't have this problem — it updates itself.

## Personal Context: Where Claude Memory Excels

Claude Memory handles the information that varies by person rather than by project. Two developers working in the same repo need different context from Claude — one is a senior backend engineer who wants terse explanations, the other is a frontend developer new to the codebase who needs more guidance.

**Role and expertise calibration** is Memory's highest-value function. When Claude knows you've been writing Go for ten years but are new to the project's React frontend, it frames explanations accordingly — using backend analogies to explain frontend concepts, skipping basics in your area of expertise. This calibration happens automatically as you interact, though you can accelerate it with explicit statements.

**Feedback corrections** accumulate over time into a behavioral profile. If you tell Claude "don't mock the database in integration tests," Memory records not just the rule but the reason — and applies it in future sessions without being reminded. This is the mechanism that makes Claude Code feel like it "gets" you over time. Our coverage of [how Claude Code's memory system works](/blog/claude-code-memory) breaks down the technical implementation.

**Project state and ongoing work** gets captured as project-type memories. Sprint goals, architectural decisions in progress, ongoing migrations — information that's relevant for weeks or months but doesn't belong in a permanent CLAUDE.md instruction. Memory's temporal nature makes it better suited for this kind of evolving context.

**External references** point Claude to information sources outside the codebase. "Pipeline bugs are tracked in Linear project INGEST" or "the oncall dashboard is at grafana.internal/d/api-latency" — details that help Claude give better suggestions without cluttering CLAUDE.md with non-instruction content.

The limitation of Memory is that it's personal and local. Your carefully accumulated context doesn't help a new team member. And because Claude writes memories automatically, they can occasionally capture outdated information that needs pruning.

## Team Dynamics: Shared vs Personal

The shared-vs-personal distinction is the most important factor when deciding where to put context.

**CLAUDE.md is team infrastructure.** It goes through code review. When someone adds a new quality gate or changes a build command, the whole team sees it in the PR diff. This auditability matters for engineering teams — you can trace when an instruction was added, who added it, and why. If a CLAUDE.md instruction causes problems, you revert it like any other code change.

**Memory is individual tooling.** It's the equivalent of your personal IDE configuration or shell aliases. It makes you more productive but doesn't constrain or guide anyone else. There's no review process, no approval — Claude just learns from your interactions and adjusts.

This creates a useful decision framework:

- "Everyone working on this project needs to know X" → CLAUDE.md
- "Claude should treat me differently because of X" → Memory
- "This decision was made last week and affects current work" → Memory (project type)
- "This pattern must never be used in this codebase" → CLAUDE.md

For teams adopting Claude Code, the recommendation is to start with CLAUDE.md. Get your build commands, coding standards, and architecture constraints documented. Memory will accumulate naturally as individual developers work with Claude. The [extension stack guide](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) covers how CLAUDE.md fits alongside skills, hooks, and MCP servers in the full Claude Code platform.

## Common Mistakes

**Putting personal preferences in CLAUDE.md.** If only you care about verbose test output or a specific commit message style, that belongs in your global `~/.claude/CLAUDE.md` or in Memory — not in the project's CLAUDE.md where it affects everyone.

**Relying on Memory for critical project rules.** If a rule must be followed by every developer and every CI run (like "never skip the build check"), it must be in CLAUDE.md. Memory is per-user and can be cleared — it's not a reliable enforcement mechanism for team-wide standards.

**Duplicating instructions across both systems.** If CLAUDE.md says "use pnpm" and you also tell Claude to remember "we use pnpm," you create potential for drift when one gets updated and the other doesn't. Let CLAUDE.md be authoritative for project rules.

**Overloading CLAUDE.md with reference information.** CLAUDE.md should contain instructions and rules, not encyclopedic documentation. Links to dashboards, wiki pages, or ticket trackers belong in Memory (reference type) unless the whole team needs them for Claude Code specifically.

## When to Choose Claude Memory

Use Memory when the context is:

- **Personal**: Your role, expertise, preferred explanation depth, or communication style
- **Temporal**: Ongoing work, sprint context, in-progress decisions that will resolve
- **Corrective**: Behavioral adjustments you've made to Claude's approach ("don't do X because Y happened")
- **Referential**: Pointers to external systems only you use — your monitoring dashboards, your ticket filters

Memory is the right default for anything you'd normally keep in your head during a working session. It bridges the gap between sessions so you don't restart from zero.

## When to Choose CLAUDE.md

Use CLAUDE.md when the context is:

- **Universal**: Applies to every developer and every Claude Code session in this project
- **Durable**: Won't change next week — build commands, coding standards, architecture boundaries
- **Enforceable**: Must be followed, not just preferred — quality gates, forbidden patterns, security constraints
- **Auditable**: Should go through code review and be traceable in git history

CLAUDE.md is the right default for anything that would cause bugs, inconsistencies, or standards violations if a developer's Claude didn't know it. For practical guidance on structuring these files, read [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code).

## Verdict

**Use both — they solve different problems.** CLAUDE.md is your project's AI instruction manual: deterministic, shared, version-controlled. Claude Memory is your personal AI context: automatic, adaptive, individual. Start with CLAUDE.md to establish project-level rules and commands. Let Memory accumulate naturally as you work. The combination means Claude Code understands both your project's constraints and your personal working style — which is how you get from "helpful AI tool" to "productive AI collaborator."

If you're forced to choose one, **start with CLAUDE.md**. Shared project context prevents more errors than personal preferences optimize. You can always add Memory later; you can't retroactively prevent the bugs caused by missing project instructions.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions take priority because they represent explicit, reviewed project rules. Memory supplements these instructions with personal context but cannot contradict them. If Memory contains a preference that conflicts with a CLAUDE.md rule, the CLAUDE.md rule wins.

### Does CLAUDE.md work without Claude Code?

CLAUDE.md is specific to Claude Code's terminal agent. It has no effect in the Claude web interface at claude.ai or in the Claude API directly. The file is read by Claude Code's harness layer before each session, making it a Claude Code-specific feature.

### How do I see what Claude has stored in Memory?

Memory files are stored as markdown in `~/.claude/projects/<project-hash>/memory/`. You can read them directly, and the `MEMORY.md` index file lists all stored memories with one-line summaries. You can also ask Claude "what do you remember about me?" during a session.

### Should I commit my global CLAUDE.md to a dotfiles repo?

Your global `~/.claude/CLAUDE.md` contains personal defaults for all projects — commit message preferences, response style, workflow habits. Committing it to a dotfiles repo is reasonable if you work across multiple machines, but it won't affect your team. The project-level CLAUDE.md in your repo root is what your team shares.

### How often should I update CLAUDE.md?

Update CLAUDE.md whenever project fundamentals change: new build system, different test framework, added quality gates, changed architecture constraints. Treat it like you'd treat updating a CI configuration — if the change affects how code gets built, tested, or reviewed, CLAUDE.md should reflect it.
