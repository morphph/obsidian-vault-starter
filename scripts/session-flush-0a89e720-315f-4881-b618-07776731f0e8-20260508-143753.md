**Claude Code's agent architecture** is hierarchical and synchronous. The main Claude Code session acts as an orchestrator. When it encounters a task that benefits from delegation — a broad codebase search, an independent implementation subtask, a design review — it spawns a subagent with a specific type and prompt. The subagent runs, completes its work, and returns a result to the parent. The parent then synthesizes the results and continues. This is not unlike a senior engineer delegating tasks to team members and integrating their work.

Each subagent type comes with a predefined tool set. The Explore agent has access to Glob, Grep, Read, and Bash (read-only) — it can search but not modify. The Plan agent can read and analyze but not edit files. General-purpose agents get the full tool set. This typed approach prevents a subagent from accidentally overstepping its role — an Explore agent physically cannot edit your files, eliminating an entire class of errors.

Custom agents in Claude Code are defined through [agent SDK](/glossary/agent-sdk) configurations or inline prompt specifications. You can create a "security-reviewer" agent that only has access to read tools and specific security-scanning commands, or a "test-writer" agent with file editing restricted to `test/` directories. These definitions live in your repository and travel with it, meaning any team member using Claude Code gets the same agent configurations. For more on how this extensibility works, see our [breakdown of Claude Code's extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp).

**Codex's agent architecture** is flat and asynchronous. Each custom agent is an independent unit — there is no built-in parent-child orchestration. You define an agent configuration (instructions, tools, environment setup), assign it a task, and it runs to completion in its own cloud sandbox. If you want multiple agents working on related tasks, you launch them separately and coordinate the results yourself, typically by reviewing the pull requests they produce.

This flat architecture has a significant advantage: simplicity. There is no orchestration logic to debug, no parent-child communication to manage, no shared state to reason about. Each agent is a self-contained unit of work. The disadvantage is that complex multi-step workflows — where the output of one agent informs the input of another — require manual coordination or external tooling. Codex does not natively support agent-to-agent communication within a single workflow.

The practical implication: Claude Code subagents are better for tightly coupled tasks (refactor a module, then update its tests, then fix the CI), while Codex custom agents are better for loosely coupled tasks (review this PR, write docs for that module, scan for security issues in this package — all independently).

## Workflow Integration: Detailed Analysis

How these agent systems fit into your daily development workflow matters as much as their technical architecture. A powerful agent system that disrupts your flow is worse than a simpler one that fits seamlessly.

**Claude Code subagents integrate into an interactive terminal session.** You are actively working — reading code, running tests, making decisions — and you delegate subtasks as they arise. The workflow feels like pair programming with a team: you stay in the driver's seat while agents handle parallel investigation or implementation work. A typical flow looks like:

1. You ask Claude Code to refactor a module
2. Claude Code spawns an Explore agent to find all callers of the module
3. Simultaneously, it spawns a Plan agent to design the refactoring approach
4. Both agents return results; Claude Code synthesizes them into a plan
5. You approve the plan; Claude Code (or a general-purpose subagent) executes it
6. Another subagent runs the test suite to verify nothing broke

This entire flow happens in a single terminal session. You can intervene at any step — redirect the plan, add constraints, or abort. The cost is attention: you need to be present and engaged. Claude Code subagents are not designed for "come back tomorrow and review the results." For teams thinking about how this orchestration model scales, our [analysis of agent harnesses](/blog/agent-harnesses-2026) covers the broader landscape.

**Codex custom agents integrate into an async task queue.** You define your agents once, then assign tasks as they come up — through the Codex UI, the [VS Code extension](/blog/codex-vscode), or the API. Each task runs in the background. You check results when you are ready, review the proposed changes, and merge or reject. The workflow feels like managing a distributed team: you assign work, context-switch to something else, and review deliverables later.

A typical Codex flow:

1. You create a custom agent called "test-writer" with instructions to follow your project's testing conventions
2. You assign it a task: "Write unit tests for the auth module"
3. Codex spins up a sandbox, clones your repo, and the agent works
4. 10-20 minutes later, you get a notification with a proposed PR
5. You review the diff, request changes or merge

This flow is inherently less interactive but more scalable. You can have dozens of Codex agents working on different tasks across multiple repositories simultaneously. The tradeoff is review overhead — every agent's output needs human review before merging, and without mid-task interaction, agents sometimes go down the wrong path and waste a full execution cycle.

## Defining Custom Agents: Configuration Compared

Both platforms let you create custom agent configurations, but the mechanism and expressiveness differ.

**In Claude Code**, custom agents are defined through agent definition files or inline specifications. An agent definition includes the agent type, available tools, system instructions, and optional model overrides. These definitions can be checked into your repository alongside your code, versioned and reviewed like any other configuration. The `subagent_type` parameter selects from built-in types or custom definitions, and the `prompt` parameter provides task-specific instructions each time the agent is spawned.

Claude Code's agent customization is code-level: you control exactly which tools the agent can access, what instructions it receives, and how its output is handled. This gives maximum flexibility but requires understanding the agent system's internals. The [skills and hooks system](/blog/claude-code-hooks-a-complete-guide-to-automating-your-ai-coding-workflow) adds another layer — you can trigger specific agent configurations automatically based on file patterns or events.

**In Codex**, custom agents are configured through the platform interface or API. You specify a name, description, instructions (as a system prompt), and environment setup (dependencies, build commands, environment variables). The configuration is stored at the organization level, meaning any team member can use the agent. Codex also supports "agent templates" — pre-built configurations for common tasks like code review, test generation, and documentation.

Codex's configuration model is higher-level and more accessible. You do not need to understand the underlying agent architecture to create a useful custom agent. The tradeoff is less granular control — you cannot restrict an agent's tool access to specific directories or commands the way you can in Claude Code. Codex agents in their sandbox get broad filesystem and shell access within the container.

| Configuration Aspect | Claude Code | Codex |
|----------------------|-------------|-------|
| Where defined | Repo files or inline | Platform UI/API |
| Versioned with code | Yes | No (org-level) |
| Tool-level restrictions | Yes (per agent type) | Limited (sandbox-level) |
| Model selection | Per-agent override | Platform default |
| Sharing mechanism | Git (checked-in files) | Org-level platform config |
| Template library | Built-in types + community skills | Platform templates |

## When to Choose Claude Code Subagents

Choose Claude Code's subagent system when your workflow demands **interactive, tightly-coordinated multi-agent execution**:

- **Complex refactoring** that requires exploration, planning, and implementation in sequence — the hierarchical orchestration handles dependency chains between agents automatically
- **Active development sessions** where you want to delegate subtasks without leaving your terminal or waiting for async results
- **Fine-grained agent control** — you need agents restricted to specific tools, directories, or operations (e.g., a read-only auditor agent that physically cannot modify files)
- **Local-first development** — you work offline, on private codebases, or in environments where cloud execution is restricted
- **Fast iteration** — subagents share the local filesystem, so there is no clone/build/setup overhead per task

Claude Code subagents are ideal for senior developers who think in terms of delegation patterns and want the AI to mirror how a well-run engineering team operates. The [agent teams feature](/blog/claude-code-agent-teams) extends this further with parallel execution across large codebases. If you already use Claude Code as your primary [coding agent](/blog/claude-code-complete-guide), subagents are a natural extension of your existing workflow.

## When to Choose Codex Custom Agents

Choose Codex custom agents when your workflow benefits from **async, isolated, team-shared agent execution**:

- **Team-wide agent definitions** — you want every engineer to use the same "code-reviewer" or "test-writer" agent without checking configuration files into every repo
- **Fire-and-forget tasks** — assign work and context-switch; review results when ready. No need to stay in a terminal session
- **Sandboxed safety** — experimental or untrusted tasks run in isolated containers with no access to your local machine or credentials
- **Multiple repositories** — Codex agents can be assigned tasks across different repos from a single interface. Good for platform teams managing many services
- **Non-developer stakeholders** — Codex's UI-first approach means product managers or QA engineers can assign tasks to custom agents without terminal access

Codex custom agents shine for teams that want to standardize AI-assisted workflows across an organization. The [student program](/blog/codex-for-students) and [open source initiative](/blog/codex-for-open-source) also make Codex accessible to individual developers who want cloud-based agent execution without infrastructure setup.

## Combining Both Systems

Claude Code and Codex are not mutually exclusive. A practical hybrid workflow:

1. **Use Claude Code subagents for active development** — exploration, planning, implementation, and testing during your coding session
2. **Use Codex custom agents for background tasks** — documentation generation, dependency updates, security scans, and other work that does not require your active attention
3. **Use Claude Code for local prototyping**, then hand off proven patterns to Codex custom agents for team-wide automation

The key decision point is synchronicity. If you need the agent's output before you can continue your current task, use a Claude Code subagent (it returns results in seconds to minutes within your session). If the task is independent and you will review results later, use a Codex custom agent (it runs in the background and produces a reviewable PR).

## Verdict

**For interactive, developer-driven multi-agent workflows, choose Claude Code subagents.** The typed agent system, hierarchical orchestration, and local execution give you unmatched control and speed during active coding sessions. You stay in the terminal, delegate intelligently, and integrate results in real time.

**For async, team-shared agent automation, choose Codex custom agents.** The cloud sandbox model, platform-level configuration, and fire-and-forget execution make it the better choice for organizations standardizing AI-assisted workflows across teams and repositories.

Most teams will benefit from using both. Claude Code subagents for the work you are doing right now; Codex custom agents for the work you want done while you focus on something else. The real question is not which system is better — it is which execution model fits each task. ## Frequently Asked Questions

### Can Claude Code subagents and Codex custom agents work together?

Yes. They operate independently — Claude Code runs locally in your terminal, Codex runs in the cloud. You can use Claude Code subagents for interactive tasks during development and assign background work to Codex custom agents simultaneously. There is no built-in integration between the two, but they can work on the same repository through standard git workflows.

### How many subagents can Claude Code run in parallel?

Claude Code can spawn multiple subagents concurrently within a single session. The practical limit depends on your API rate limits and local system resources rather than a hard platform cap. For most workflows, 2-4 parallel subagents (e.g., one Explore, one Plan, two general-purpose) is the effective sweet spot before context management becomes complex.

### Do Codex custom agents persist between sessions?

Yes. Custom agent configurations in Codex are stored at the organization level and persist indefinitely. Once you define a custom agent — its instructions, tools, and environment setup — any team member can invoke it for any task. The agent configurations are independent of individual sessions or repositories.

### Which is more cost-effective for small teams?

Claude Code subagents use per-token API billing with no platform fee, making them cost-effective for small teams with moderate usage. Codex custom agents require a platform subscription (Pro or Team tier) plus usage-based charges for cloud compute. For teams under five developers doing fewer than 20 agent tasks per week, Claude Code's pay-per-use model is typically cheaper. At higher volumes, Codex's bundled pricing may offer better value.

### Can I restrict what a custom agent can access in my codebase?

In Claude Code, yes — each subagent type has explicit tool restrictions, and custom agents can be configured with directory-level and command-level access controls. In Codex, agents run in isolated sandboxes with full access to the cloned repository but no access to your local machine or other repositories. Codex does not currently support per-directory restrictions within the sandbox.
