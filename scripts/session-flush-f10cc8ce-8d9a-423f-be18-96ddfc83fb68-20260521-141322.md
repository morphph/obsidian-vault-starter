**CLAUDE.md lives in your repository.** It's a regular file — you edit it in your IDE, review it in pull requests, and track its history with `git log`. When someone joins the team, they clone the repo and immediately get the same Claude Code behavior as everyone else. This is powerful for instructions that must be consistent: build commands, test requirements, forbidden patterns, deployment workflows. If two developers' Claude Code sessions disagree about whether to run tests before committing, you have a bug. CLAUDE.md prevents that class of bug entirely.

**Claude Memory lives in your local `.claude/` directory structure.** It's organized by project path, with individual markdown files for each memory entry. The system supports four typed categories: `user` memories (your role, expertise, preferences), `feedback` memories (corrections and confirmed approaches), `project` memories (decisions, deadlines, ongoing work), and `reference` memories (pointers to external systems like Linear boards or Grafana dashboards). Each memory file includes frontmatter with a name, description, and type, making them searchable and maintainable.

The practical consequence: CLAUDE.md is infrastructure, memory is personal context. You wouldn't put "I'm a backend engineer who prefers verbose error messages" in a shared CLAUDE.md — that's a personal preference that might conflict with a frontend colleague's needs. Similarly, you wouldn't rely on memory to enforce "all PRs must pass CI" — that's a team-wide rule that belongs in version control where it can be reviewed and enforced consistently.

One subtlety worth noting: Claude Memory can become stale. A project memory about a merge freeze that ended two weeks ago is noise, not signal. Claude Code is designed to treat memories as hints rather than absolute truth — it verifies against current state before acting on recalled information. CLAUDE.md entries, by contrast, are treated as directives. If CLAUDE.md says "run `npm test` before committing," Claude Code will do it every time, no verification needed. For more on how Claude Code's memory system works in practice, see our deep dive on the [Claude Code memory system](/blog/claude-code-memory).

## Use Cases and Workflow Integration: Detailed Analysis

The real question isn't "which is better" — it's "which information goes where." Here's a decision framework based on how teams actually use both systems.

**Put it in CLAUDE.md when:**

- It's a rule that applies to every developer on the team. Build commands, quality gates, lint configurations, and deployment workflows belong here. Your CLAUDE.md should be the single source of truth for "how Claude Code behaves in this project."
- It's a constraint that must never be violated. Forbidden patterns (like importing Next.js modules in pipeline scripts), required pre-commit checks, or security rules should be directives in CLAUDE.md, not suggestions in memory.
- It needs code review. If a rule change should go through a pull request — say, relaxing a test requirement or changing the commit message format — it belongs in a version-controlled file.
- It's architectural context that shapes every task. "This is a Next.js + SQLite stack," "Chinese content uses CJK word count," "the newsletter pipeline has seven stages" — these are foundational facts that every session needs.

**Put it in Memory when:**

- It's about you, not the project. Your role, your expertise level, your communication preferences, your workflow habits. These are personal context that helps Claude Code tailor its responses — explaining React concepts differently to a Go expert versus a frontend specialist.
- It's a correction or confirmed approach. When you tell Claude Code "don't mock the database in integration tests" or confirm that a single bundled PR was the right call, that's feedback memory. It prevents you from having to give the same guidance twice.
- It's ephemeral project state with an expiration date. A merge freeze, a sprint goal, an ongoing incident — these matter now but will be irrelevant in two weeks. Memory's automatic update mechanism handles this better than a CLAUDE.md entry you'll forget to remove.
- It's a pointer to an external system. "Pipeline bugs are tracked in Linear project INGEST" or "the oncall dashboard is at grafana.internal/d/api-latency" — these references help Claude Code find context outside the repo without cluttering the shared instruction file.

Teams building sophisticated Claude Code workflows often layer both systems with [skills files](/blog/5-claude-code-skills-i-use-every-single-day), which provide task-specific instructions that sit between CLAUDE.md's project-wide rules and memory's personal context. The [nine principles for writing effective skills](/blog/9-principles-writing-claude-code-skills-that-actually-work) cover how to structure this layered approach.

## Scope and Hierarchy: How They Interact

Claude Code doesn't choose between memory and CLAUDE.md — it reads both and merges them into a unified context. Understanding the priority order matters when they contain conflicting information.

**CLAUDE.md operates at three levels:**

1. **Global** (`~/.claude/CLAUDE.md`): Personal defaults that apply to all projects — your git workflow preferences, general coding style, preferred tools.
2. **Project** (`/repo/CLAUDE.md`): Team-shared instructions checked into the repository. This is the most important level for consistency.
3. **Directory** (`/repo/src/components/CLAUDE.md`): Subsystem-specific rules, like UI component conventions or API endpoint patterns.

Project-level CLAUDE.md overrides global, and directory-level overrides project for files within that directory. This hierarchy lets you set sensible defaults while allowing subsystem-specific exceptions.

**Memory operates at two levels:**

1. **Global** (`~/.claude/memory/`): Cross-project context about you as a developer.
2. **Project** (`~/.claude/projects/<project-path>/memory/`): Context specific to the current project.

When Claude Code starts a session, it loads the CLAUDE.md hierarchy first (these are directives), then loads relevant memories (these are context). If a memory contradicts a CLAUDE.md instruction, the CLAUDE.md instruction wins — it's deterministic and team-approved. Memory provides additional context within the boundaries CLAUDE.md sets.

This means you can think of CLAUDE.md as the constitution and memory as institutional knowledge. The constitution defines the rules; institutional knowledge helps you work effectively within them. A senior engineer who's been on the team for years has both: they know the rules (CLAUDE.md) and they know the unwritten context (memory) — why a particular architecture decision was made, which module is fragile, who to ask about the billing system.

## When to Choose Claude Memory

Choose Claude Memory as your primary context mechanism when you're working solo or when personal adaptation matters more than team consistency.

**Solo developers** benefit enormously from memory. Without a team to share CLAUDE.md with, the version-control advantage is less compelling. Memory's automatic persistence means you spend less time maintaining instruction files and more time coding. Claude Code learns your patterns — your preferred testing approach, your commit message style, your debugging methodology — and applies them without being told each session.

**Developers new to a codebase** should lean on memory during the ramp-up period. As you learn about the project — which modules are fragile, what the deployment process looks like, where the documentation lives — memory captures these discoveries automatically. Later, insights that are universally relevant can be promoted to CLAUDE.md via a pull request.

**Cross-project workers** who jump between multiple codebases benefit from memory's project-scoped organization. Your global memories travel with you, while project-specific context activates only when you're working in that repo. ## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary context mechanism when team consistency, reproducibility, and auditability are priorities.

**Teams of any size** need CLAUDE.md. The moment two developers use Claude Code on the same project, you need a shared source of truth for how the AI should behave. Without CLAUDE.md, each developer's Claude Code operates with different context, leading to inconsistent code style, missed quality gates, and conflicting approaches. CLAUDE.md eliminates this by design.

**Projects with strict quality requirements** — regulated industries, security-critical code, or high-stakes production systems — should encode their requirements in CLAUDE.md. Memory is helpful context; CLAUDE.md is enforceable policy. "All changes must pass security review" belongs in a file that's auditable, reviewable, and impossible to accidentally forget.

**Open-source projects** benefit from CLAUDE.md because external contributors get the same AI behavior as maintainers. A new contributor who clones the repo and runs Claude Code immediately follows the project's conventions. ## Verdict

**Use both.** CLAUDE.md and Claude Memory aren't competing systems — they're complementary layers in Claude Code's context architecture. **CLAUDE.md is your project's constitution**: shared, versioned, deterministic instructions that every team member's Claude Code follows identically. **Claude Memory is your personal institutional knowledge**: learned preferences, corrections, project state, and external references that make your individual sessions more effective.

The decision rule is simple: **if the information should be consistent across all developers, put it in CLAUDE.md. If it's personal, evolving, or too granular for a shared file, let memory handle it.** When in doubt, start with memory — if you find yourself telling Claude Code the same thing every session, promote it to CLAUDE.md. ## Frequently Asked Questions

### Can Claude Memory and CLAUDE.md conflict with each other?

Yes, but Claude Code handles conflicts with a clear priority: CLAUDE.md instructions always override memory. CLAUDE.md entries are treated as directives that must be followed, while memories are treated as context that informs behavior. If your memory says "use tabs for indentation" but CLAUDE.md says "use spaces," Claude Code will use spaces.

### Should I commit my Claude Memory files to the repository?

No. Memory files in `.claude/projects/` are personal and per-user — they contain your individual preferences, role information, and conversation-derived context. Committing them would impose one developer's personal context on the entire team. CLAUDE.md is the shared context mechanism; memory stays local.

### How do I migrate useful memories into CLAUDE.md?

When you notice a memory that applies to all team members — a recurring correction, a project convention, a build requirement — manually add it to CLAUDE.md and submit a pull request. Then delete or let the memory entry go stale naturally. Think of memory as a staging area for patterns that might deserve promotion to team-wide rules.

### Does Claude Memory work across different machines?

Memory files are stored locally in your `~/.claude/` directory. They don't sync automatically across machines. If you work from multiple devices, you'll build up separate memory contexts on each. CLAUDE.md, being in the repository, syncs everywhere you clone the repo — another reason to put critical project context in CLAUDE.md rather than relying solely on memory.

### What happens to memories when a project changes significantly?

Claude Code treats memories as hints, not facts. Before acting on a recalled memory, it verifies against the current codebase state. If a memory references a file that no longer exists or a function that was renamed, Claude Code discovers the discrepancy and updates or ignores the stale memory. This self-correcting behavior makes memory resilient to project evolution, though periodic manual cleanup is still worthwhile.
