# Codex Subagents vs Custom Agents：哪种多智能体方案更适合你？

**TL;DR：** 如果你需要快速将大任务拆解为并行子任务、且不想自己管理 Agent 生命周期，**Codex Subagents** 是开箱即用的选择。如果你的工作流需要定制工具集、专属 system prompt 或与外部系统深度集成，**Custom Agents** 提供完全的控制权。多数生产级项目最终会两者结合使用。

## 概述：Codex Subagents

Codex Subagents 是 OpenAI Codex 平台内置的子智能体机制，允许主 Agent 在执行复杂任务时自动派生出多个子任务执行者。每个 Subagent 运行在独立的沙箱环境中，拥有自己的文件系统快照和执行上下文，完成后将结果汇报给主 Agent。

这种模式的核心优势在于**零配置的并行能力**。你不需要定义 Agent 的结构或通信协议——Codex 会根据任务的自然结构自动拆分。例如，当你要求 Codex 重构一个模块并同时更新相关测试时，它可以派生两个 Subagent 分别处理，最后合并结果。这对处理大型代码仓库中的跨文件任务特别有效。

Subagents 的计费跟随 Codex 主任务，不需要单独管理 API 调用。每个 Subagent 继承主任务的权限边界和仓库上下文，降低了安全配置的复杂度。详细的平台能力可参考 [Codex 完全指南](/zh/blog/codex-complete-guide)。

## 概述：Custom Agents

Custom Agents 是指开发者在 Codex 生态中自行定义的智能体——通过 [Agent SDK](/zh/glossary/agent-sdk) 或 Codex API 构建，拥有自定义的 system prompt、工具集和执行逻辑。与平台内置的 Subagents 不同，Custom Agents 的行为完全由开发者控制。

Custom Agents 的价值在于**可编程的专业化**。你可以为代码审查、安全扫描、文档生成等不同任务分别创建专属 Agent，每个 Agent 携带领域特定的指令和工具。这类似于组建一支各有专长的工程团队，而不是依赖一个全能选手。

在 Codex 的 VS Code 扩展中，Custom Agents 可以作为工作流的一部分被触发。开发者可以通过配置文件定义 Agent 的触发条件和上下文范围，实现 IDE 内的[智能体编码](/zh/glossary/agentic-coding)工作流。更多 IDE 集成细节参见 [Codex VS Code 扩展指南](/zh/blog/codex-vscode)。

## 功能对比

| 维度 | Codex Subagents | Custom Agents | 推荐 |
|------|-----------------|---------------|------|
| **配置成本** | 零配置，平台自动管理 | 需定义 prompt、工具、触发逻辑 | Subagents |
| **并行执行** | 原生支持，自动拆分 | 需自行编排并发逻辑 | Subagents |
| **行为可控性** | 受限于平台默认行为 | 完全自定义 | Custom Agents |
| **工具集成** | 仅限 Codex 内置工具 | 支持自定义工具和 MCP | Custom Agents |
| **上下文隔离** | 每个子任务独立沙箱 | 开发者自行管理隔离边界 | Subagents |
| **复用性** | 一次性，随任务销毁 | 可跨项目、跨任务复用 | Custom Agents |
| **调试难度** | 平台日志，黑箱居多 | 完全可观测 | Custom Agents |
| **计费模式** | 包含在主任务中 | 独立 API 调用计费 | Subagents |

## 任务拆解与并行执行：深度对比

Codex Subagents 的最大卖点是**自动任务拆解**。当主 Agent 识别到一个任务可以被分解为独立的子问题时，它会自动创建 Subagents。每个 Subagent 获得仓库的只读快照，在隔离环境中执行，完成后提交差异（diff）给主 Agent 合并。整个过程对用户透明，你只需要提交一个高层任务描述。

这种方式的局限也很明显：**你无法干预拆分逻辑**。如果 Codex 对任务的理解与你的预期不同——比如它把一个应该串行执行的重构拆成了并行子任务，导致合并冲突——你只能重新描述任务，而不能直接修改拆分策略。

Custom Agents 的并行执行需要开发者自己编排。你需要定义哪些 Agent 可以并行、如何分配上下文、结果如何合并。这显然更费工夫，但也意味着你可以实现 Subagents 做不到的复杂工作流。例如，一个 Custom Agent 负责代码变更，另一个实时审查变更是否符合安全策略，第三个更新相关文档——这种有依赖关系的[多智能体工作流](/zh/blog/con-u-pour-des-workflows-multi-agents)在纯 Subagents 模式下很难实现。

对比 Claude Code 的 Agent Teams 机制（参见 [Claude Code Agent Teams 详解](/zh/blog/claude-code-agent-teams)），Codex Subagents 更偏向自动化，而 Claude Code 允许开发者在 CLAUDE.md 中预定义子 Agent 的角色和职责，在自动化与可控性之间取得了不同的平衡点。

## 定制化与工具集成：深度对比

Custom Agents 真正拉开差距的地方在于**工具生态的开放性**。Codex Subagents 只能使用平台内置的工具——文件读写、shell 命令、代码搜索——这对大多数编码任务足够了，但一旦你需要调用外部 API、查询数据库、或与内部系统交互，就会碰壁。

Custom Agents 可以挂载任意工具。通过 Agent SDK 定义工具接口，你的 Agent 可以调用 Jira API 自动创建 ticket、查询 Grafana 获取性能数据、或者调用内部的代码质量平台获取历史指标。这种灵活性对企业级工作流至关重要。

在 system prompt 层面，Custom Agents 允许你注入完整的领域知识。你可以为前端 Agent 配置组件库规范，为后端 Agent 配置 API 设计指南，为 DevOps Agent 配置部署策略。这种专业化是 Subagents 的通用 prompt 无法替代的。关于如何设计高质量的 Agent 指令，可以参考 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)——虽然是 Claude Code 生态的经验，但 system prompt 设计原则是通用的。

一个实际的权衡：Custom Agents 的灵活性伴随着**维护成本**。每个 Agent 的 prompt 需要随代码库演进而更新，工具接口需要与外部系统保持兼容，Agent 之间的交互逻辑需要测试。如果你的团队只有几个人，维护一套 Custom Agent 体系的成本可能超过收益。

## 什么时候选 Codex Subagents

Subagents 适合以下场景：

- **快速原型和探索阶段**：你还不确定最终的工作流长什么样，需要快速试验。Subagents 零配置的特性让你专注于任务本身而不是基础设施
- **标准编码任务**：重构、测试生成、bug 修复——这些任务的模式相对固定，Subagents 的自动拆解通常能给出合理的结果
- **小团队或个人项目**：没有足够的工程资源来维护 Custom Agent 体系时，Subagents 是性价比最高的选择
- **Codex 平台深度用户**：如果你的主要工作流已经在 Codex 生态内，Subagents 的无缝集成比引入外部 Agent 框架更自然

想了解 Codex 在开源项目中的应用模式，参见 [Codex for Open Source 分析](/zh/blog/codex-for-open-source)。

## 什么时候选 Custom Agents

Custom Agents 适合以下场景：

- **企业级多步骤工作流**：代码变更→安全审查→合规检查→文档更新→部署审批——这种有严格顺序和依赖的流程需要精确编排
- **跨系统集成需求**：Agent 需要与 CI/CD、项目管理、监控系统等外部服务交互
- **领域专业化要求高**：不同类型的任务需要截然不同的 prompt 和工具集，一个通用 Agent 无法胜任
- **可观测性和审计要求**：需要精确追踪每个 Agent 的决策路径和工具调用，满足合规审计需求
- **跨平台复用**：同一套 Agent 定义需要在 Codex、[Claude Code 子 Agent](/zh/blog/claude-code-subagents-examples) 等不同平台间迁移

## 最佳实践：混合使用

实际生产环境中，最有效的方案往往是**分层混合**：

1. **日常编码任务**交给 Subagents 处理——写代码、跑测试、修小 bug，不需要过度工程化
2. **关键路径工作流**用 Custom Agents 精确编排——PR 审查、安全扫描、发布流程
3. **逐步迁移**：先用 Subagents 跑通流程，识别出需要定制化的环节后，再逐个替换为 Custom Agents

这种渐进式的方法避免了过早投入 Custom Agent 开发的风险，同时保留了在关键环节实现精确控制的能力。

## 总结

**Codex Subagents 是默认起点，Custom Agents 是进阶选择。** 对于 80% 的编码任务，Subagents 的自动拆解和并行执行已经足够。当你发现自己反复调整任务描述试图"引导" Subagent 的行为时，就是切换到 Custom Agents 的信号。两者不是非此即彼的关系——成熟的工程团队会在同一个工作流中同时使用两种模式，让每种机制在最擅长的场景中发挥作用。

## 常见问题

### Codex Subagents 和 Custom Agents 可以在同一个任务中混用吗？

可以。主 Agent 可以在自动派生 Subagents 的同时调用预定义的 Custom Agents。典型模式是让 Subagents 处理代码变更，同时触发一个 Custom Agent 执行安全审查。两者共享同一个仓库上下文，但各自在隔离环境中执行。

### Custom Agents 需要多少工程投入才能上手？

最简单的 Custom Agent 只需要一个 system prompt 和一组工具定义，30 分钟即可搭建。但要做到生产级别——包括错误处理、重试逻辑、可观测性——通常需要 1-2 周的初始投入。建议从单一用途的 Agent 开始，验证价值后再扩展。

### Subagents 的并行执行会导致合并冲突吗？

会。当多个 Subagents 修改了同一文件的相邻区域时，主 Agent 需要解决冲突。Codex 内置了基本的冲突解决能力，但对于复杂冲突，可能需要人工介入。降低冲突概率的方法是在任务描述中明确模块边界。
