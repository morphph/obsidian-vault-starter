**Put it in CLAUDE.md if:**

- Every developer on the project needs to know it
- It's a rule, constraint, or standard (not a preference)
- It changes infrequently (quarterly or less)
- It's derivable from or related to the codebase itself
- Violating it would break the build, fail CI, or introduce bugs

Examples that belong in CLAUDE.md:
- Build and test commands (`npm run build`, `npm test`)
- Coding standards ("use snake_case for database columns")
- Architectural constraints ("never import Next.js modules in pipeline scripts")
- Known gotchas ("ZH content must use CJK word count, not English space tokenization")
- Required quality gates before committing
- Git workflow rules

**Put it in Memory if:**

- It's about you, not the project
- It's about how you want Claude to interact with you
- It's transient project state with a shelf life
- It's a pointer to an external resource only you use
- It emerged from conversation, not from deliberate documentation

Examples that belong in Memory:
- Your role and expertise level
- Communication preferences ("be terse," "explain like I'm new to this")
- Corrections to Claude's behavior ("don't mock the database in tests")
- Current project initiatives and deadlines
- Links to external tracking systems, dashboards, or docs

**The gray zone:**

Some information could reasonably go in either place. The tiebreaker is always: **would a new team member benefit from this?** If yes, CLAUDE.md. If it's only useful in the context of your personal workflow, Memory.

For example, "we're migrating from REST to GraphQL" is project context a new team member needs — put it in CLAUDE.md. "The migration is blocked on the auth team's review, expected to unblock by April 20" is transient state that only matters to people actively working on it — that's a Memory.

## Team Workflows: Scaling the Two Systems

For solo developers, the distinction between Memory and CLAUDE.md matters less — you're the only audience for both. But the separation pays dividends as teams grow.

**Solo developer (1 person):** Use CLAUDE.md for project setup and build commands. Let Memory handle everything else organically. You'll naturally accumulate feedback memories as you correct Claude's behavior, and project memories as you share context about ongoing work. Periodically review your memory files and promote anything that's become a permanent project rule into CLAUDE.md.

**Small team (2-5 people):** CLAUDE.md becomes critical. Invest time in documenting coding standards, architectural decisions, and workflow rules. Each team member's Memory will diverge — one developer might prefer detailed explanations while another wants terse responses — and that's the point. The shared CLAUDE.md ensures consistency in what Claude does; individual Memory customizes how it communicates.

**Larger team (5+ people):** Layer CLAUDE.md files by directory. A monorepo might have a root CLAUDE.md with global standards, a `frontend/CLAUDE.md` with React conventions, and a `backend/CLAUDE.md` with API design rules. Memory becomes essential for encoding role-specific context — the frontend lead's Memory captures different project priorities than the infrastructure engineer's.

At every team size, the principle holds: **CLAUDE.md is the team's shared brain; Memory is your personal brain.** Mixing them creates confusion — team instructions in Memory means inconsistent AI behavior across developers, and personal preferences in CLAUDE.md means noisy instruction files that no one maintains.

## Common Mistakes and How to Avoid Them

**Mistake 1: Putting everything in CLAUDE.md.**
Solo developers often dump their entire working context — role, preferences, project state, external links — into CLAUDE.md. This works initially but creates problems: the file grows unwieldy, personal preferences clutter team instructions, and transient information (deadlines, current bugs) becomes stale. Fix: Use CLAUDE.md for durable project rules only. Let Memory handle the rest.

**Mistake 2: Ignoring Memory entirely.**
Some developers treat CLAUDE.md as the only persistence mechanism and repeat their preferences every conversation. Claude's Memory system exists precisely to avoid this. If you find yourself re-explaining your role, your communication preferences, or your current project focus every session, you're doing Memory's job manually. Fix: Let Claude save memories naturally, or explicitly say "remember that I prefer X" when correcting behavior.

**Mistake 3: Duplicating information across both.**
If your CLAUDE.md says "run tests before committing" and you also have a Memory that says "always run tests before committing," you've created a maintenance burden with no benefit. The CLAUDE.md instruction loads deterministically every time — the Memory is redundant. Fix: Memory should contain information that CLAUDE.md doesn't, not echo it.

**Mistake 4: Using Memory for code patterns.**
Memory is for context that can't be derived from the codebase. Code conventions, file naming patterns, and architecture decisions are better placed in CLAUDE.md where they're version-controlled and team-visible. If Claude can learn it by reading the code, it doesn't need to be in Memory. Fix: Reserve Memory for information about you, your workflow, and external context.

**Mistake 5: Never reviewing or pruning Memory.**
Memory files accumulate over time. Project memories about deadlines pass, reference memories point to moved resources, and feedback memories become redundant as your workflow evolves. Stale memories can mislead Claude — acting on an outdated deadline or a deprecated dashboard link. Fix: Periodically scan your `memory/` directory and remove or update outdated entries.

## When to Choose Claude Memory

Choose Memory as your primary persistence investment when:

- **You're a solo developer** and the distinction between personal and team context doesn't matter yet — Memory's automatic accumulation requires zero setup effort
- **You're joining an existing project** that already has a mature CLAUDE.md — your Memory captures your personal onboarding context (what confused you, what you're responsible for, which external tools you use)
- **You work across many projects** — global Memory captures your universal preferences (communication style, role, expertise) so every project benefits without editing each CLAUDE.md
- **Project context is changing rapidly** — during an incident, a migration, or a crunch, Memory captures the fast-moving state that would be stale in CLAUDE.md by tomorrow

Memory's strength is zero-friction persistence. You don't plan it, you don't maintain it (much), and it adapts to your evolving workflow automatically. The tradeoff is that it's personal and non-deterministic — Claude decides what's relevant to load, and no one else on your team sees it.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence investment when:

- **You're setting up a team project** — CLAUDE.md is the single most impactful file you can create for consistent AI-assisted development across your team
- **You need deterministic behavior** — every conversation must know the build commands, the coding standards, and the architectural constraints, with no exceptions and no relevance filtering
- **You're encoding institutional knowledge** — gotchas, migration notes, and "why we do it this way" explanations that must survive team turnover
- **You want version-controlled AI configuration** — CLAUDE.md changes show up in PRs, get reviewed, and create a history of how your project's AI instructions evolved

CLAUDE.md's strength is reliability and shareability. It loads every time, for every developer, exactly as written. The tradeoff is maintenance overhead — you have to write it, update it, and keep it from becoming a dumping ground. ## Verdict

**Use both — they're complementary, not competing.** CLAUDE.md is your project's shared instruction manual: build commands, coding standards, architectural constraints, and workflow rules that every developer and every Claude session must follow. Claude Memory is your personal context layer: your role, your preferences, your corrections, and the transient project state that changes too fast for version-controlled documentation.

If you're forced to prioritize, **start with CLAUDE.md**. A well-written CLAUDE.md file has more impact on Claude Code's usefulness than any other single configuration. Memory accumulates naturally over time with no upfront effort. But CLAUDE.md requires deliberate investment — and that investment pays off immediately for every conversation and every team member.

The developers getting the most from Claude Code treat these as a stack: CLAUDE.md sets the project floor, Memory adds the personal ceiling, and [skills files](/blog/9-principles-writing-claude-code-skills) handle task-specific instructions. Together, they give Claude the full context it needs to operate as an effective [agentic coding](/glossary/agentic-coding) partner rather than a stateless autocomplete engine.

## Frequently Asked Questions

### Can Claude Memory and CLAUDE.md conflict with each other?

Yes, but Claude is designed to resolve conflicts in favor of CLAUDE.md. If your Memory says "prefer tabs" and the project CLAUDE.md says "use spaces," Claude follows CLAUDE.md. Memory provides context and preferences; CLAUDE.md provides rules. When they disagree, rules win — which is the correct behavior for team consistency.

### Does Claude Memory work across different projects?

Claude Memory is scoped per project by default, stored in project-specific directories under `~/.claude/projects/`. However, a global `~/.claude/CLAUDE.md` applies across all projects for user-wide preferences, and user-type memories (your role, communication style) carry across projects through the global memory scope. Project-specific memories stay isolated to that project.

### How often should I update CLAUDE.md?

Update CLAUDE.md whenever a project rule changes — a new build step, a deprecated pattern, a shifted architectural constraint. Treat it like documentation that's read by an AI on every interaction: stale instructions cause incorrect behavior. A quarterly review is a good minimum cadence; high-velocity projects may need monthly updates.

### Can I see what Claude has saved in Memory?

Yes. Memory files are plain markdown stored in `~/.claude/projects/<project-id>/memory/`. You can read, edit, or delete them directly. The `MEMORY.md` index file shows all saved memories at a glance. Periodically reviewing this directory helps you catch stale or incorrect memories before they mislead Claude.

### Should CLAUDE.md contain information about team members?

Generally no. Team member details — who's responsible for what, who's on vacation, who's leading a migration — are transient project state that belongs in Memory. CLAUDE.md should contain rules and instructions that apply regardless of who's working. The exception is role-based instructions like "frontend changes require review from the design team" — that's a process rule, not personal context.
