# Codex Subagents vs Custom Agents：哪种多智能体方案更适合你？

**TL;DR:** **Codex Subagents** 是平台原生的任务拆分机制，适合将大型编码任务自动分解为并行子任务，开箱即用、无需额外配置。**Custom Agents** 是用户自定义的智能体配置，通过指定系统提示词、工具集和行为约束来构建专用 Agent，灵活度更高但需要更多前期投入。**如果你需要快速分发标准编码任务，选 Subagents；如果你需要针对特定工作流深度定制，选 Custom Agents。**

## 概览：Codex Subagents

**Codex Subagents** 是 OpenAI Codex 平台内置的子智能体调度能力——当一个编码任务足够复杂时，系统会自动将其拆解为多个独立子任务，每个子任务在独立的沙箱环境中并行执行。这种机制的核心优势在于零配置：你只需提交一个任务描述，Codex 自行决定如何分解、分配和汇总。

Subagents 运行在 Codex 的云端容器中，每个子 Agent 拥有独立的文件系统快照和执行环境。任务完成后，结果会被自动合并回主任务。这对于大规模代码重构、批量文件修改、跨模块测试生成等场景尤其高效。关于 Codex 的完整架构设计，参见[《OpenAI Codex 完全指南》](/zh/blog/codex-complete-guide)。

Subagents 的典型使用门槛很低——你不需要理解 Agent 编排框架，也不需要编写额外的配置文件。但相应地，你对子任务的拆分逻辑、执行顺序和工具调用几乎没有控制权。

## 概览：Custom Agents

**Custom Agents** 指的是用户在编码 Agent 平台上自定义的智能体实例。通过编写系统提示词（System Prompt）、指定可用工具集、设定行为约束和输出格式，你可以构建出专门用于代码审查、文档生成、安全扫描或特定业务逻辑的 Agent。

Custom Agents 的核心价值在于可控性和可复用性。你可以精确定义 Agent 的行为边界——它能访问哪些文件、能执行哪些命令、遇到歧义时如何处理。这种模式在 Claude Code 生态中已经被广泛验证，例如通过 [SKILL.md 和 Agent Teams](/zh/blog/claude-code-agent-teams) 实现团队级的 Agent 协作。

构建 Custom Agents 需要更多前期投入：你需要设计提示词、测试边界情况、迭代配置。但一旦调试完成，这些 Agent 可以作为团队资产持续复用，且行为一致性远高于通用型子任务分发。

## 功能对比

| 维度 | Codex Subagents | Custom Agents | 优势方 |
|------|----------------|---------------|--------|
| **配置复杂度** | 零配置，平台自动拆分 | 需编写提示词和工具配置 | Subagents |
| **任务拆分控制** | 平台决定拆分逻辑 | 用户完全控制 | Custom Agents |
| **并行执行** | 原生支持 | 需自行编排 | Subagents |
| **工具调用范围** | 平台预设工具集 | 用户自定义工具集 | Custom Agents |
| **行为一致性** | 依赖模型判断 | 通过提示词强约束 | Custom Agents |
| **适用任务粒度** | 中大型、可拆分任务 | 任意粒度 | 平手 |
| **学习曲线** | 低 | 中高 | Subagents |
| **团队复用性** | 低（每次重新拆分） | 高（配置可版本化） | Custom Agents |

## 任务编排与执行模型：深度分析

Subagents 和 Custom Agents 在执行模型上的差异，决定了它们各自的天花板。

Codex Subagents 采用的是**隐式编排**——你提交一个高层任务，平台的调度层决定拆成几个子任务、每个子任务的 scope 是什么、执行顺序如何安排。这种模式在标准场景下效率极高：比如「给这个模块的所有公开函数补充单元测试」，Codex 会自动识别函数列表、为每个函数生成独立的测试子任务、并行执行后合并结果。整个过程无需人工干预。

但隐式编排的局限也很明显：当任务之间存在复杂依赖关系时（比如「先重构数据层，再更新所有调用方，最后修改集成测试」），自动拆分可能无法正确识别执行顺序。你可能会看到子任务 B 在子任务 A 完成前就开始执行，导致产出基于过时的代码状态。

Custom Agents 采用的是**显式编排**——你精确定义每个 Agent 的职责边界和交互协议。在 [Claude Code 的多 Agent 实践](/zh/blog/claude-code-subagents-examples)中，开发者通过 Agent Teams 机制实现了类似的显式编排：一个 Agent 负责代码生成，另一个负责测试验证，第三个负责文档更新，彼此通过明确的输入输出接口协作。这种模式的前期成本更高，但在复杂工作流中的可靠性远超隐式编排。

如果你的团队已经在使用 [Claude Code 的 Skills 体系](/zh/blog/9-principles-writing-claude-code-skills)，那么 Custom Agents 的配置模式会非常熟悉——本质上就是把 SKILL.md 的理念扩展到了 Agent 级别。

## 可控性与安全边界：深度分析

在生产环境中使用 AI 编码 Agent，可控性不是加分项，而是硬性要求。

Codex Subagents 运行在平台托管的沙箱中，安全边界由平台统一管理。每个子 Agent 只能访问你提交的代码仓库快照，无法访问外部网络或执行未授权的系统命令。这种「安全默认」的模型降低了风险，但也限制了灵活性——如果你需要子 Agent 调用内部 API、读取私有包注册表或执行自定义 lint 规则，平台预设的沙箱可能无法满足需求。

Custom Agents 的安全边界由用户定义。这意味着你可以精确控制每个 Agent 能做什么、不能做什么。在 [Claude Code Hooks 体系](/zh/blog/claude-code-hooks-mastery)中，开发者通过 Hook 机制实现了细粒度的行为拦截——比如禁止 Agent 修改特定目录、强制每次文件写入都通过 lint 检查、在执行危险命令前要求人工确认。这种控制粒度是平台托管方案目前无法提供的。

但自定义安全边界也是一把双刃剑：配置不当可能导致 Agent 获得过大的权限。团队需要建立 Agent 配置的审查流程，就像审查代码一样审查 Agent 的权限设定。

## 什么时候选 Codex Subagents

Subagents 是「快速出活」的最佳选择，特别适合以下场景：

- **批量标准任务**：为整个目录补充类型注解、生成 API 文档、批量修复 lint 告警——这类任务结构化程度高、子任务之间独立性强，Subagents 的自动拆分机制能显著提升效率。
- **原型阶段的快速迭代**：项目早期不需要精细的 Agent 配置，Subagents 的零配置特性让你把精力集中在业务逻辑上。了解更多 Codex 在团队中的使用方式，参见[《Codex VS Code 扩展》](/zh/blog/codex-vscode)。
- **个人开发者或小团队**：没有专门的 Agent 运维能力时，平台托管的 Subagents 是最低风险的选择。

简而言之：如果你的任务可以用一句话描述清楚、子任务之间没有复杂依赖，Subagents 几乎总是更高效的选择。

## 什么时候选 Custom Agents

Custom Agents 的投入回报比在以下场景中最高：

- **复杂工作流编排**：任务涉及多步骤依赖、条件分支或人工审批节点时，显式编排比隐式拆分更可靠。[多智能体工作流的设计](/zh/blog/con-u-pour-des-workflows-multi-agents)一文详细讨论了这类场景的架构选择。
- **团队级标准化**：当你需要确保所有开发者使用的 AI Agent 遵循相同的编码规范、安全策略和输出格式时，版本化的 Agent 配置是唯一可靠的方案。
- **垂直领域深度定制**：安全审计、合规检查、特定框架的代码生成——这些场景需要 Agent 具备领域知识和专用工具，通用 Subagents 无法胜任。
- **生产环境的可观测性要求**：Custom Agents 可以集成日志、指标和告警，Subagents 的执行过程对用户来说更像黑盒。

## 结论

**选择 Codex Subagents 还是 Custom Agents，本质上是在「开箱即用」和「深度可控」之间做取舍。** 对于 80% 的标准编码任务，Subagents 的自动拆分和并行执行已经足够好，且几乎不需要额外投入。但当你的工作流涉及复杂依赖、团队协作规范或生产级的可靠性要求时，Custom Agents 的前期投入会在后续迭代中持续产生回报。

务实的建议：**从 Subagents 开始，在遇到控制力不足时再迁移到 Custom Agents**。两者并非互斥——在同一个项目中，简单任务用 Subagents 分发，关键路径用 Custom Agents 把控，是目前最高效的混合策略。

## 常见问题

### Codex Subagents 需要额外付费吗？

Codex Subagents 的计费包含在 Codex 平台的整体使用费用中。每个子任务消耗独立的计算资源和 Token 额度，因此并行执行多个子任务会比单任务消耗更多配额，但不会产生额外的「子任务调度费」。具体费率以 OpenAI 官方定价页面为准。

### Custom Agents 的配置可以跨项目复用吗？

可以。Custom Agents 的配置本质上是提示词、工具定义和行为约束的集合，可以版本化存储在代码仓库中，像代码一样进行审查和复用。[Claude Code 的 Skills 体系](/zh/blog/do-skills-actually-improve-your-agents-output)已经验证了这种模式的有效性。

### 两者可以混合使用吗？

可以，且推荐混合使用。典型模式是：用 Custom Agents 处理需要精确控制的关键任务（代码审查、安全扫描），用 Subagents 处理标准化的批量任务（测试生成、格式修复）。两种模式互补而非互斥。
