# Claude Memory vs CLAUDE.md：两套记忆系统到底有什么区别？

**TL;DR：** **Claude Memory** 是 Anthropic 的跨对话持久记忆功能，自动记住你的偏好、背景和习惯，适用于所有 Claude 对话场景。**CLAUDE.md** 是 Claude Code 的项目级指令文件，随代码仓库版本管理，定义编码规范、架构约束和工作流规则。两者不是竞争关系——Memory 记住「你是谁」，CLAUDE.md 定义「项目怎么做」。需要个人化体验选 Memory，需要团队级工程规范选 CLAUDE.md，大多数开发者应该同时使用两者。

## 概述：Claude Memory

**Claude Memory** 是 Anthropic 在 Claude 对话产品中内置的持久化记忆功能，让 Claude 在多次对话之间记住用户的关键信息。它解决的核心问题是：每次开新对话都要重新介绍自己的角色、偏好和上下文。

Memory 的工作方式是自动从对话中提取值得记住的信息——你的职业背景、技术偏好、项目上下文、沟通风格——存储为结构化的记忆条目。下次对话时，Claude 会自动加载相关记忆来调整回应方式。2026 年初，Anthropic [大幅升级了 Memory 系统](/zh/blog/anthropic-claude-memory-upgrades-importing)，免费用户也可以使用，并且支持从 ChatGPT 导入历史数据。

Memory 的定位是**个人化**：它属于用户账户，跟随用户跨对话、跨项目、跨设备。

## 概述：CLAUDE.md

**CLAUDE.md** 是 [Claude Code](/glossary/claude-code) 的项目级配置文件，用 Markdown 格式定义项目的编码规范、架构决策、工作流约束和技术上下文。它不是 Claude 平台的通用功能，而是 Claude Code 终端工具的专属机制。

CLAUDE.md 的工作方式更接近传统的工程文档：它是一个签入代码仓库的文件，Claude Code 在每次会话开始时自动读取并遵循其中的指令。你可以在里面写构建命令、测试要求、代码风格约束、禁止的模式等。团队所有成员共享同一份 CLAUDE.md，确保 AI 行为的一致性。

关于 CLAUDE.md 的完整架构和层级设计，可以参考我们的[Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

CLAUDE.md 的定位是**项目标准化**：它属于代码仓库，跟随项目走，跟随团队走。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **归属** | 用户账户 | 代码仓库 | 取决于场景 |
| **作用域** | 跨所有对话和项目 | 单个项目 | Memory 更广 |

| **可审计性** | 用户可查看和删除 | 团队可 code review | CLAUDE.md |

## 核心差异：自动记忆 vs 显式指令

这是两者最根本的区别。Claude Memory 是一个**隐式系统**——它在对话过程中自动判断哪些信息值得记住，用户不需要主动管理。这很方便，但也意味着你对记忆内容的控制力较弱。你可能不确定 Claude 到底记住了什么，或者记住的信息是否准确。

CLAUDE.md 是一个**显式系统**——每一条规则都是你（或你的团队）明确写下来的，经过 code review，有版本历史。没有猜测，没有歧义。当 Claude Code 的行为不符合预期时，你可以直接检查 CLAUDE.md 找到原因。

在实际工程中，这个区别很重要。Memory 适合存储「我喜欢简洁的代码风格」这类个人偏好。但「本项目所有测试必须通过才能提交」「禁止在 pipeline 脚本中直接 import Next.js 模块」这类团队级约束，必须用 CLAUDE.md 来确保每个人（包括每个 AI 会话）都遵守同样的规则。

这也是为什么 Anthropic 在 Claude Code 中同时保留了两套系统——[Claude Code 的记忆系统](/zh/blog/claude-code-memory)实际上包含 CLAUDE.md（规则手册）和 Auto Memory（自动笔记）两个层次，分别处理不同类型的知识持久化需求。

## 层级与组合：实际使用中的协作关系

在 Claude Code 的实际使用中，Memory 和 CLAUDE.md 不是二选一的关系，而是分层协作。Claude Code 的上下文加载顺序是：

1. **全局 CLAUDE.md**（`~/.claude/CLAUDE.md`）：用户级别的通用偏好，如 Git 工作流习惯、提交消息风格
2. **项目 CLAUDE.md**（项目根目录）：项目级别的编码规范和约束
3. **Auto Memory**（`~/.claude/projects/*/memory/`）：自动记录的项目上下文和用户反馈

这个分层设计意味着：全局偏好用 CLAUDE.md 写一次，所有项目生效；项目规范用项目级 CLAUDE.md，团队共享；个人在项目中积累的经验和上下文，由 Auto Memory 自动处理。

关于如何有效利用这套分层系统，我们在[Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)中有更详细的讨论。Skills（SKILL.md 文件）是 CLAUDE.md 之上的另一层——它们定义特定任务的执行方式，进一步细化了指令系统的粒度。

## 什么时候选 Claude Memory

以下场景中，Claude Memory 是更好的选择：

- **跨项目的个人偏好**：你喜欢的沟通风格、技术栈偏好、角色背景。这些信息不属于任何单个项目，而是属于你个人。
- **非编码场景**：在 Claude.ai 上写邮件、做分析、头脑风暴。这些场景没有代码仓库，CLAUDE.md 无从谈起。
- **新项目冷启动**：当你开始一个新项目时，Memory 中已有的个人偏好可以立即生效，不需要从零配置。
- **渐进式学习**：你不想一开始就写一份完整的规则文档，而是希望 AI 在合作过程中逐渐了解你的习惯。

## 什么时候选 CLAUDE.md

以下场景中，CLAUDE.md 是更好的选择：

- **团队协作**：多人共同开发的项目，需要所有人（和所有 AI 会话）遵守相同的规范。Memory 是个人的，无法共享。
- **关键约束的可靠执行**：「构建必须通过才能提交」「测试覆盖率不能低于 80%」这类硬约束，不能依赖自动记忆的不确定性。
- **可审计和可追溯**：CLAUDE.md 在 Git 中有完整的修改历史，可以做 code review，可以回滚。Memory 没有这些保障。
- **CI/CD 集成**：CLAUDE.md 可以被自动化工具读取和验证，成为工程流水线的一部分。
- **复杂的项目特定规则**：比如数据库 schema 约定、API 设计规范、特定模块的架构决策。这些信息过于具体和技术化，不适合存在通用记忆里。

如果你正在考虑如何系统性地提升 Claude Code 的使用效率，[我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)这篇文章从实战角度覆盖了包括 CLAUDE.md 在内的多种优化策略。

## 结论

**Claude Memory 和 CLAUDE.md 解决的是不同层次的问题，最佳实践是同时使用两者。** Memory 处理个人化和跨项目的上下文延续，CLAUDE.md 处理项目级的工程规范和团队协作。把个人偏好写进 Memory（或全局 CLAUDE.md），把项目规则写进项目 CLAUDE.md，把任务级指令写进 SKILL.md——这就是 Claude Code 记忆系统的完整分层策略。

对于刚接触 Claude Code 的开发者，建议从项目 CLAUDE.md 开始：写下你的构建命令、测试要求和最重要的三条编码规范。这比任何其他配置都能更快地提升 AI 辅助编码的质量。Memory 会在你使用过程中自然积累，不需要刻意管理。

## 常见问题

### Claude Memory 的内容会影响 CLAUDE.md 的规则吗？

不会产生冲突。在 Claude Code 中，CLAUDE.md 的指令具有更高的优先级。如果 Memory 中记录了「用户喜欢宽松的代码风格」，但项目 CLAUDE.md 要求严格的 lint 规则，Claude Code 会遵循 CLAUDE.md 的约束。两者是分层关系，不是竞争关系。

### CLAUDE.md 可以替代 Claude Memory 吗？

部分场景可以。全局 CLAUDE.md（`~/.claude/CLAUDE.md`）可以存放跨项目的个人偏好，功能上与 Memory 有重叠。但 Memory 的优势在于自动化——你不需要手动维护，它会在对话中自动学习和更新。对于非 Claude Code 的场景（如 Claude.ai 网页端），CLAUDE.md 完全不适用。

### 团队项目中应该如何分配两者的职责？

推荐的分工是：项目级的强制规范（构建流程、测试要求、架构约束）写入 CLAUDE.md 并签入仓库；个人级的偏好（沟通风格、代码风格微调）交给 Memory 或全局 CLAUDE.md。原则是：需要团队一致性的用 CLAUDE.md，纯属个人习惯的用 Memory。
