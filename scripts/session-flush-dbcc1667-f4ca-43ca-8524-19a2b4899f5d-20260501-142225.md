# Claude Memory vs CLAUDE.md：哪个才是你需要的上下文管理方案？

**TL;DR：** **CLAUDE.md** 是项目级的规则手册——版本控制、团队共享、每次会话确定性加载，适合编码规范和架构约束。**Claude Memory** 是用户级的自动笔记——跨会话持久化、个人专属、按相关性召回，适合偏好记录和工作习惯。两者不是替代关系，而是互补层。大多数认真使用 Claude Code 的开发者应该同时用好这两个机制。

## 概述：Claude Memory

**Claude Memory** 是 Claude Code 的自动记忆系统，它在对话过程中学习并持久化用户偏好、项目上下文和工作反馈。记忆文件存储在 `~/.claude/projects/` 目录下，以 Markdown 文件形式组织，包含 frontmatter 元数据和正文内容。

Memory 的核心设计理念是「不需要用户主动维护」。当你纠正 Claude 的行为（"不要在测试里 mock 数据库"）、确认某个做法有效（"对，就用这种单 PR 的方式"）、或分享项目背景（"我们在做合规迁移"），Claude 会自动判断是否值得记住，并写入对应类型的记忆文件。

Memory 分为四种类型：用户记忆（角色和知识背景）、反馈记忆（工作方式偏好）、项目记忆（进行中的工作上下文）、引用记忆（外部资源指针）。每种类型有不同的保存触发条件和使用场景。关于记忆系统的最新升级——包括免费用户开放和 ChatGPT 数据导入——可以参考我们的[详细报道](/zh/blog/anthropic-claude-memory-upgrades-importing)。

## 概述：CLAUDE.md

**CLAUDE.md** 是一个放在项目根目录的 Markdown 文件，定义了 Claude Code 在该项目中的行为规则。它是确定性的——每次 Claude Code 启动时都会完整加载，不存在「是否被召回」的问题。

CLAUDE.md 的内容通常包括：构建命令（`npm run build`、`npm test`）、代码风格规范、架构约束（"不要在 pipeline 脚本里 import Next.js 模块"）、提交规范、以及项目特定的注意事项。因为它是一个普通的代码文件，所以天然支持 Git 版本控制、代码审查和团队共享。

CLAUDE.md 还支持层级结构：全局级（`~/.claude/CLAUDE.md`）覆盖所有项目，项目级（项目根目录）覆盖当前仓库，子目录级覆盖特定模块。这套层级设计在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中属于最基础也最关键的一层。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 胜出 |
|------|--------------|-----------|------|
| **作用域** | 用户级，跨项目 | 项目级，per-repo | 各有所长 |
| **加载方式** | 按相关性召回 | 每次会话完整加载 | CLAUDE.md |
| **维护成本** | 自动写入，低维护 | 手动编写和更新 | Memory |
| **版本控制** | 不在 Git 中 | 可 Git 跟踪 | CLAUDE.md |
| **团队共享** | 个人专属 | 团队共享 | CLAUDE.md |
| **内容类型** | 偏好、反馈、上下文 | 规则、命令、约束 | 各有所长 |
| **可靠性** | 可能遗漏或过时 | 确定性加载 | CLAUDE.md |
| **灵活性** | 对话中随时学习 | 需要手动编辑文件 | Memory |

## 核心差异：确定性 vs 自适应

这是两个系统之间最根本的区别。

CLAUDE.md 是**确定性**的。你写了 "commit 前必须通过 `npm test`"，Claude 每次都会看到这条规则——不存在遗忘的可能。这对于团队级的编码规范、质量门禁、架构红线来说至关重要。一条关键规则被遗忘一次，可能就是一个生产事故。

Memory 是**自适应**的。它通过对话学习，根据上下文决定哪些记忆相关并召回。这意味着 Memory 能捕捉到你不会主动写进规则文件的东西——比如你是数据科学家而不是前端工程师，比如你更喜欢简洁的回复风格，比如上周的合规审查还没完成。但自适应也意味着不确定性：一条记忆可能在关键时刻没被召回，或者已经过时但仍被引用。

在实际使用中，这个差异决定了内容分配策略：**必须遵守的规则放 CLAUDE.md，可以灵活参考的上下文放 Memory**。想深入了解两者如何在实际项目中协作，推荐阅读 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 团队协作 vs 个人效率

CLAUDE.md 的团队价值是不可替代的。当一个五人团队共享同一个 CLAUDE.md，所有人的 Claude Code 实例都遵循相同的编码规范、使用相同的构建命令、避开相同的已知坑点。新成员 clone 仓库后立即获得完整的 AI 行为上下文，不需要任何额外配置。

Memory 则解决了个人效率问题。资深后端工程师和刚接触 React 的新人，即使在同一个项目里工作，需要的 AI 交互方式也完全不同。Memory 让 Claude 记住「这个用户有十年 Go 经验但第一次碰前端」，从而自动调整解释的深度和角度。这种个性化不适合写进项目级的 CLAUDE.md。

一个实际的例子：团队的 CLAUDE.md 里写着 "使用 Vitest 进行单元测试"，这是所有人必须遵循的规则。而 Memory 里可能记着 "这个用户更喜欢先写测试再写实现"——这是个人工作风格，不需要强制到团队层面。

## 内容组织与生命周期

CLAUDE.md 的内容应该是**长期稳定**的。频繁变动的信息不适合放在这里——如果你的 CLAUDE.md 每天都需要更新，说明有些内容应该交给 Memory 或其他机制处理。好的 CLAUDE.md 像项目的宪法：精炼、权威、不轻易修改。

Memory 的内容是**动态演进**的。项目记忆会随着工作推进而更新（"合规迁移已完成"），反馈记忆会随着新偏好被发现而积累，用户记忆会随着角色变化而调整。Memory 系统有内置的索引文件（MEMORY.md）来管理所有记忆条目，但记忆本身会自然衰减——过时的记忆应该被更新或删除。

在 Claude Code 的[扩展栈体系](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中，CLAUDE.md 和 Memory 分别占据不同的层级。CLAUDE.md 是基础配置层，与 Skills 和 Hooks 并列；Memory 是学习层，随使用而增长。理解这个层级关系有助于决定每条信息应该放在哪里。

## 什么时候用 Claude Memory

选择 Memory 的典型场景：

- **个人工作偏好**：回复风格、commit 习惯、代码审查标准等个人化设置
- **跨项目上下文**：你的技术背景、角色、专业领域——这些在切换项目时依然有用
- **临时项目状态**：当前冲刺的目标、正在调查的 bug、待完成的迁移——这些有时效性，不值得写进 CLAUDE.md
- **外部资源指针**：Linear 看板地址、Grafana 仪表盘 URL、Slack 频道——这些容易变化，且通常是个人需要的
- **纠错反馈**：当你纠正 Claude 的行为后，Memory 自动记录，下次不再犯同样的错误

## 什么时候用 CLAUDE.md

选择 CLAUDE.md 的典型场景：

- **构建和测试命令**：`npm run build`、`npm test` 等必须每次执行的命令
- **代码规范和约束**："不要在 pipeline 脚本里直接 import Next.js 模块"、"ZH 内容必须用 CJK word count"
- **架构决策**：技术栈选择、文件组织约定、API 设计原则
- **质量门禁**：提交前必须通过的检查清单，不容许遗忘
- **团队共识**：所有团队成员的 Claude Code 都应该遵循的行为规则
- **已知坑点**：反复踩到的问题，需要确定性地提醒每一次会话

如果你还在犹豫要不要为项目建立 CLAUDE.md，[这篇文章](/zh/blog/how-to-effectively-prompt-a-claude-code)提供了从零开始的实用指南。

## 最佳实践：组合使用策略

最有效的做法是同时使用两个系统，但有明确的分工：

**CLAUDE.md 负责「法律」**——不可违反的规则、必须执行的流程、团队级的标准。写进 CLAUDE.md 的每一条都应该是你愿意在代码审查中强制执行的。

**Memory 负责「文化」**——工作方式、沟通偏好、项目背景、经验教训。Memory 中的内容影响 Claude 的判断和行为方式，但不构成硬性约束。

一个反面模式是把所有东西都塞进 CLAUDE.md。当文件超过几百行时，信息密度下降，Claude 的注意力被稀释。关于如何精心组织 Claude Code 的上下文层级，[七层架构解析](/zh/blog/claude-code-seven-programmable-layers)值得一读。

另一个反面模式是完全依赖 Memory 而不建 CLAUDE.md。Memory 的概率性召回意味着关键规则可能在最需要的时候缺席。对于团队项目来说，这几乎必然导致不一致。

## 判决

**两者必须同时使用，但职责不同。** 如果只能选一个，先建 **CLAUDE.md**——它提供的确定性加载和团队共享能力是 AI 辅助开发的基础设施。Memory 是锦上添花，它让 Claude 从「遵循规则的工具」变成「理解你的协作者」。但没有规则手册的自适应记忆，就像没有代码规范的团队——短期灵活，长期混乱。

对于独立开发者，CLAUDE.md 定义项目规则，Memory 积累个人经验。对于团队，CLAUDE.md 是共享的项目宪法，Memory 是每个人的私人教练。两层配合，才能让 Claude Code 既一致又个性化。

## 常见问题

### Claude Memory 会自动覆盖 CLAUDE.md 里的规则吗？

不会。CLAUDE.md 的优先级更高——它在每次会话开始时完整加载，作为 Claude 行为的硬性约束。Memory 提供的是补充上下文，当两者信息冲突时，CLAUDE.md 中的规则优先。

### Memory 中的内容会过时吗？怎么处理？

会。Memory 系统不会自动清理过期内容。Claude Code 在使用记忆前会尝试验证其有效性（比如检查文件是否还存在），但最佳实践是定期审查 MEMORY.md 索引文件，手动删除或更新不再准确的记忆条目。

### 团队成员之间可以共享 Memory 吗？

不能直接共享。Memory 文件存储在用户本地的 `~/.claude/` 目录下，绑定到个人账户。如果你发现某条记忆对整个团队都有价值，正确的做法是把它提炼成规则写进 CLAUDE.md，通过 Git 共享。
