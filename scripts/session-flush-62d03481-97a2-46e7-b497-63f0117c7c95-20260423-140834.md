**Claude Memory is AI-authored.** Claude writes memory files based on conversation signals — your corrections, your stated preferences, facts you share about ongoing work. This makes it low-maintenance but less predictable. You can review Memory files and delete ones that are wrong or outdated, but you can't control exactly when Claude decides to save something.

Memory files use a structured format with frontmatter (name, description, type) and content organized by category. The `MEMORY.md` index keeps entries discoverable. But unlike CLAUDE.md, Memory can become stale without you noticing — a project context note from three months ago might no longer be accurate, and Claude might still act on it.

The maintenance tradeoff is clear: CLAUDE.md requires active human effort but stays accurate when maintained. Memory requires minimal effort but can drift. Teams that care about consistency invest in CLAUDE.md. Individual developers who want Claude to "just get them" rely more on Memory.

For teams building sophisticated Claude Code workflows, the [extension stack guide](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) explains how CLAUDE.md fits alongside Skills, Hooks, and MCP servers in the full configuration hierarchy.

## Scope and Sharing: Detailed Analysis

The scope difference between Memory and CLAUDE.md has significant implications for team workflows.

**CLAUDE.md is shared infrastructure.** When one developer adds a gotcha to CLAUDE.md ("don't import Next.js modules in pipeline scripts — they're server-only"), every team member's Claude sessions immediately benefit. This makes CLAUDE.md a force multiplier for institutional knowledge. New team members get the same Claude quality as veterans from their first session, because Claude reads the same instructions.

CLAUDE.md also supports hierarchical scoping. You can have a root CLAUDE.md for the whole repo and additional CLAUDE.md files in subdirectories for module-specific instructions. Claude merges them, with more specific files taking precedence. This lets monorepo teams maintain both global standards and package-specific rules.

**Memory is personal and isolated.** Your corrections, preferences, and context never leak to teammates. This is intentional — your preference for terse responses shouldn't affect a colleague who wants detailed explanations. But it also means knowledge doesn't propagate. If you teach Claude something important about the codebase through a correction, that learning stays with you.

This isolation creates an interesting team dynamic. The team invests in CLAUDE.md for shared knowledge. Each developer accumulates personal Memory for their individual workflow. Over time, senior developers who've been correcting Claude for months have a significantly more effective Claude experience than new team members — but only for personal preferences, not project knowledge.

The practical recommendation: if you discover something that would help the whole team, put it in CLAUDE.md, not Memory. Reserve Memory for genuinely personal context — your role, your communication preferences, your current focus areas.

## Content Types: What Goes Where

Knowing what content belongs in each system prevents duplication and confusion.

### Put in CLAUDE.md

- **Build commands**: `npm run build`, `npm test`, `npm run lint`
- **Quality gates**: "All tests must pass before commit"
- **Architecture constraints**: "Never import server-only modules in scripts"
- **Style guidelines**: "Newsletter tone: sharp tech insider briefing a busy founder"
- **Known gotchas**: "Chinese content must use CJK word count, not English space tokenization"
- **Workflow rules**: "New feature → discuss design first, get human approval before coding"
- **Documentation requirements**: "Update PIPELINE.md when modifying pipeline scripts"
- **Environment-specific instructions**: "Run validate-pipeline.ts for pipeline changes"

### Put in Memory

- **Your role**: "Senior backend engineer, new to the frontend side of this repo"
- **Communication preferences**: "Skip trailing summaries, I can read the diff"
- **Approach corrections**: "Don't mock the database in integration tests"
- **Ongoing project context**: "Merge freeze starts April 10 for mobile release"
- **External references**: "Pipeline bugs tracked in Linear project INGEST"
- **Personal workflow**: "I prefer bundled PRs over many small ones for refactors"

### The Gray Zone

Some information could reasonably go in either place. The deciding factor is always: **does this apply to the whole team, or just to me?**

A debugging insight like "the auth service returns 403 for expired tokens, not 401" could go in CLAUDE.md as a known gotcha (helps everyone) or in Memory as project context (helps just you). If it's a recurring source of confusion, CLAUDE.md is the better home. If it's a one-time context note for your current work, Memory suffices.

## Reliability and Staleness

Both systems can go stale, but they fail differently.

**CLAUDE.md fails loudly.** When instructions are wrong — a build command that changed, a deprecated workflow step — Claude follows the outdated instruction and the resulting error is immediately visible. Someone notices, updates CLAUDE.md, and the fix propagates to everyone. The feedback loop is tight.

**Memory fails quietly.** An outdated Memory entry might cause Claude to make slightly wrong assumptions without anyone noticing. If Memory says "the team uses Jest for testing" but you migrated to Vitest six months ago, Claude might not produce outright errors — it might just frame its suggestions differently or miss Vitest-specific features. These subtle misalignments are harder to catch.

To mitigate Memory staleness, Claude is designed to verify Memory entries against current reality before acting on them. If a Memory entry names a specific file, Claude checks that the file exists before recommending it. But this verification isn't perfect, especially for softer context like project goals or team structures.

The practical takeaway: audit your Memory files periodically. Delete entries about completed projects, outdated team structures, or resolved decisions. CLAUDE.md gets audited naturally through code review; Memory needs intentional maintenance.

## When to Choose Claude Memory

Use Memory as your primary context mechanism when:

- **You're a solo developer.** There's no team to share CLAUDE.md with, and your personal preferences are the project preferences. Memory's automatic capture saves you from manually maintaining instructions.

- **You want zero-maintenance personalization.** You'd rather correct Claude in conversation and have it learn than write and maintain a configuration file. Memory's "teach by doing" model fits your workflow.

- **Your context is personal, not project-level.** Your role, your preferred communication style, your current focus areas — these are about you, not the codebase. Memory is the right home.

- **You're exploring or prototyping.** In early-stage projects where conventions aren't established yet, Memory captures emerging patterns naturally. Once patterns stabilize, you can formalize them into CLAUDE.md.

- **You work across many repositories.** Memory persists per-project without requiring you to create CLAUDE.md files in every repo. It provides baseline personalization everywhere.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary context mechanism when:

- **You're on a team.** CLAUDE.md is the only way to ensure every team member's Claude sessions follow the same rules. Without it, each developer trains their own Claude independently, leading to inconsistent behavior.

- **You need deterministic behavior.** When Claude must always run tests before committing, always follow a specific style guide, or always avoid certain patterns — CLAUDE.md enforces these as hard constraints, not suggestions.

- **Your project has non-obvious conventions.** Every codebase has gotchas that aren't obvious from reading the code. CLAUDE.md captures these explicitly: "Don't use `upsertKeyword()` without all three parameters," "Chinese content uses CJK word counting."

- **You want version-controlled AI configuration.** CLAUDE.md changes go through pull requests. You can see who changed what instruction and why. You can revert bad changes. Memory offers none of this auditability.

- **Onboarding matters.** New team members — or new Claude Code users on the team — get immediate benefit from a well-maintained CLAUDE.md. They don't need months of conversation history to build up useful Memory.

## Using Both: The Recommended Approach

Most effective Claude Code setups use both systems together, each handling what it does best.

**CLAUDE.md handles the "what"** — what to build, what commands to run, what patterns to follow, what mistakes to avoid. It's your project's constitution, and Claude follows it to the letter.

**Memory handles the "how"** — how to communicate with you, how to approach ambiguous decisions based on your preferences, how to adapt its behavior to your working style. It's your personal relationship with Claude.

A concrete example: CLAUDE.md says "commit messages must accurately describe the change." Memory says "this user prefers conventional commit format with scope." Both inform how Claude writes a commit message, but from different angles — one is a project rule, the other is a personal preference.

Another example: CLAUDE.md says "run `npm run build` before committing." Memory says "this user wants terse responses, skip the explanation of what the build does." The build check happens regardless (CLAUDE.md instruction), but the way Claude reports the result adapts to you (Memory preference).

The [Claude Code Skills system](/blog/5-claude-code-skills-i-use-every-single-day) adds a third layer — reusable instruction files for specific task types — but Skills build on top of CLAUDE.md and Memory rather than replacing them.

## Common Mistakes

**Mistake 1: Putting personal preferences in CLAUDE.md.** "I prefer short responses" doesn't belong in a team-shared file. Your colleague might want detailed explanations. Let Memory handle personal style.

**Mistake 2: Relying on Memory for critical project rules.** If a rule matters enough that violating it causes bugs or broken builds, it belongs in CLAUDE.md where it's enforced deterministically. Memory is informational, not authoritative.

**Mistake 3: Duplicating information in both.** If CLAUDE.md says "use Vitest for testing" and Memory also says "the project uses Vitest," you're maintaining two sources of truth. Let CLAUDE.md be authoritative for project facts; use Memory only for context CLAUDE.md can't cover.

**Mistake 4: Never auditing Memory.** Memory entries from three months ago might reference files that no longer exist, team members who've left, or decisions that were reversed. Review and prune periodically.

**Mistake 5: Writing a CLAUDE.md that's too long.** A 500-line CLAUDE.md dilutes the important instructions with noise. Keep it focused on what Claude actually needs to know that isn't obvious from the code. If you can derive it by reading the source, don't put it in CLAUDE.md.

## Verdict

**Use both — but know which one to reach for.** CLAUDE.md is your project's instruction manual for Claude: version-controlled, team-shared, and deterministic. Memory is your personal relationship with Claude: automatic, adaptive, and private. Neither replaces the other.

**Start with CLAUDE.md** if you're setting up Claude Code for a team. Document your build commands, quality gates, and known gotchas. This gives every team member a strong baseline immediately. Then let Memory accumulate naturally as each developer works with Claude over time.

**Start with Memory** if you're a solo developer exploring a new project. Let Claude learn your preferences organically. Once your project conventions stabilize, formalize the important ones into CLAUDE.md so they persist even if you clear your Memory.

The developers getting the most out of Claude Code treat CLAUDE.md like infrastructure and Memory like configuration — one is shared and stable, the other is personal and adaptive. ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions are treated as authoritative project constraints that Claude must follow. Memory provides informational context that shapes Claude's approach but cannot override explicit CLAUDE.md directives. If CLAUDE.md says "always run tests before committing," no Memory entry can skip that requirement.

### Does CLAUDE.md support per-user sections?

CLAUDE.md itself is shared and doesn't have per-user sections. However, Claude Code also reads a global `~/.claude/CLAUDE.md` file for user-specific instructions that apply across all projects. This global file fills a similar role to Memory but requires manual authoring. Use it for hard rules you want everywhere; use Memory for softer preferences.

### How do I migrate useful Memory entries into CLAUDE.md?

Review your Memory files in `~/.claude/projects/<project>/memory/` and identify entries that would benefit the whole team — recurring gotchas, architectural decisions, workflow patterns. Rewrite them as concise CLAUDE.md instructions and submit a pull request. Then delete the redundant Memory entries to avoid duplication.

### Will clearing Memory affect CLAUDE.md behavior?

No. CLAUDE.md and Memory are independent systems loaded from different locations. Clearing Memory removes personal context and preferences but has no effect on project-level instructions in CLAUDE.md. Claude will still follow all CLAUDE.md directives exactly as before.

### How often should I update CLAUDE.md?

Update CLAUDE.md whenever project conventions change — new build tools, updated workflow rules, discovered gotchas, changed architecture constraints. A good rule of thumb: if you had to correct Claude about a project-level issue twice, add it to CLAUDE.md so nobody has to correct it again.
