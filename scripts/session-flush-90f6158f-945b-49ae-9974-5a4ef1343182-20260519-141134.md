# Claude Memory vs CLAUDE.md: Which Persistence Layer Should You Use?

**TL;DR:** **CLAUDE.md** is your deterministic instruction layer — explicit rules, conventions, and constraints that every session follows without variation. **Claude Memory** is your adaptive context layer — observations, preferences, and learned facts that accumulate across conversations. Use CLAUDE.md for team-shared project rules; use Claude Memory for personal workflow context that evolves over time. Most effective setups use both together.

## Overview: Claude Memory

Claude Memory is [Claude Code's](/glossary/claude-code) automatic persistence system that stores learned context across conversations in `.claude/projects/*/memory/` directories. Each memory is a markdown file with typed frontmatter (user, feedback, project, or reference) that Claude reads at session start to recall prior context.

The system works like a colleague's notebook — Claude observes your preferences, corrections, and project context during conversations, then saves relevant details for future sessions. You can also explicitly ask Claude to remember something. Memories are personal to your machine and not shared with teammates or checked into version control.

Key characteristics: automatic accumulation, personal scope, adaptive over time, structured by type (user preferences, feedback corrections, project state, external references).

## Overview: CLAUDE.md

**CLAUDE.md** is a deterministic instruction file placed at your project root (or in `~/.claude/CLAUDE.md` for global rules). It's loaded into every Claude Code session as part of the system context — think of it as a constitution that Claude must follow. Unlike memory, CLAUDE.md is explicit: you write every line, it's version-controlled, and the entire team shares the same instructions.

CLAUDE.md typically contains build commands, quality gates, coding style rules, workflow constraints, and architectural decisions. As described in our [guide to Claude Code's seven programmable layers](/blog/claude-code-seven-programmable-layers), it sits at the project instruction layer — above the model but below user prompts in the priority hierarchy.

Key characteristics: manually authored, team-shared, version-controlled, deterministic, loaded every session without variation.

## Persistence Model: How Each System Stores and Retrieves Context

Claude Memory and CLAUDE.md solve different halves of the persistence problem. CLAUDE.md answers "what rules should Claude always follow?" while Memory answers "what has Claude learned about this project and this user?"

**CLAUDE.md** loads as part of the system prompt every session. There's no retrieval logic — the entire file is always present. This makes it completely reliable but size-constrained. A 2,000-line CLAUDE.md wastes context window on instructions that may not be relevant to the current task. Best practice is keeping CLAUDE.md focused on universal project rules (build commands, quality gates, style constraints) that apply to every session.

**Claude Memory** uses an index file (`MEMORY.md`) that's always loaded, with individual memory files read on demand when relevant. This two-tier architecture means memory scales better — you can accumulate hundreds of observations without bloating every session's context. The tradeoff is retrieval uncertainty: a memory might not surface when relevant, or might surface when stale.

The [Claude Code memory system](/blog/claude-code-memory) documentation details how these layers interact in practice — CLAUDE.md provides the floor (minimum context), while memory fills in adaptive context above it.

## Collaboration Model: Team vs Individual

The most consequential difference between these systems is their collaboration boundary.

**CLAUDE.md is team infrastructure.** When you add `npm run build` as a required pre-commit check in CLAUDE.md, every team member's Claude Code sessions enforce that gate. When you document that "ZH content must use CJK word count," the entire team benefits. This makes CLAUDE.md the right place for:

- Build and test commands
- Coding conventions and style rules
- Architecture decisions and constraints
- Workflow requirements (PR templates, commit message formats)
- Known gotchas that affect everyone

**Claude Memory is individual context.** Your memories about your role, your debugging preferences, your project responsibilities — these don't belong in a shared file. Memory handles:

- Your expertise level and communication preferences
- Feedback you've given about Claude's behavior
- Project status updates relevant to your work
- References to external systems you use

Teams building with Claude Code typically encode shared standards in CLAUDE.md and let individual memory handle personal workflow adaptation. The [extension stack guide](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) explains how these layers compose with skills, hooks, and MCP servers for a complete programmable surface.

## Reliability and Staleness: When Each System Fails

**CLAUDE.md fails when it's outdated.** If your build command changes from `npm run build` to `turbo build` and nobody updates CLAUDE.md, Claude will run the wrong command every session. But this failure mode is visible — it's in version control, it's reviewable, and it breaks loudly.

**Claude Memory fails when it's stale.** A memory might record "the auth module is in `src/auth/`" — but if that module was moved to `packages/auth/`, the memory silently misleads. Memory files include no automatic expiration. Claude is instructed to verify memories against current state before acting on them, but this verification isn't guaranteed.

The practical implication: put anything that Claude should follow *without question* in CLAUDE.md. Put context that Claude should *consider but verify* in memory. If a wrong instruction would break the build or violate team standards, it belongs in the deterministic layer.

## When to Choose Claude Memory

Use Claude Memory when the context is:

- **Personal** — your role, your preferences, your communication style
- **Evolving** — project status that changes weekly, team responsibilities that shift
- **Observational** — "the user prefers terse responses" or "deploys go through staging first"
- **Corrective** — feedback about Claude's behavior that should persist ("don't mock the database in integration tests")

Memory shines for reducing repetition across sessions. Instead of re-explaining your role, your testing philosophy, or your project's deployment quirks every conversation, Claude remembers and adapts. For teams using Claude Code extensively, the [skills and workflow patterns](/blog/5-claude-code-skills-i-use-every-single-day) guide shows how memory complements skill files for personalized automation.

## When to Choose CLAUDE.md

Use CLAUDE.md when the instruction is:

- **Universal** — applies to every team member, every session, without exception
- **Deterministic** — must be followed exactly, not interpreted or verified
- **Stable** — doesn't change week-to-week (build commands, style rules, architecture boundaries)
- **Auditable** — should be reviewable in PRs and traceable in git history

CLAUDE.md is particularly valuable for [agentic coding](/glossary/agentic-coding) workflows where Claude operates with significant autonomy. When Claude Code is running builds, committing code, and creating PRs, the constraints in CLAUDE.md act as guardrails that prevent drift. Our [guide to prompting Claude Code effectively](/blog/how-to-effectively-prompt-a-claude-code) covers how to write CLAUDE.md rules that are specific enough to be reliable.

## Using Both Together

The strongest Claude Code configurations layer both systems:

1. **CLAUDE.md** defines the project constitution — build commands, quality gates, forbidden patterns, documentation rules
2. **Claude Memory** accumulates operational context — who's working on what, what decisions were made, what feedback you've given

A practical example: CLAUDE.md says "run `npm test` before every commit." Memory knows "the user is working on the auth refactor this week and cares about test coverage in `packages/auth/`." Together, Claude both follows the universal rule and applies it with context-appropriate focus.

For teams scaling Claude Code across multiple developers, the [complete guide](/blog/claude-code-complete-guide) details how CLAUDE.md, memory, skills, and hooks compose into a reproducible development environment.

## Verdict

**Choose CLAUDE.md for rules, choose Memory for context.** If the information is a constraint that must be enforced identically across every session and every team member, write it in CLAUDE.md. If it's adaptive context that helps Claude work smarter over time — preferences, project state, learned corrections — let memory handle it. The two systems are complementary, not competing. Teams that use both effectively get deterministic guardrails (CLAUDE.md) with personalized intelligence (memory) — the combination that makes [Claude Code](/glossary/claude-code) most effective as a long-term engineering partner.

## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md loads as system-level instructions with higher priority than memory. If CLAUDE.md says "always use tabs" and a memory records "user prefers spaces," the CLAUDE.md rule wins. Memory provides context; CLAUDE.md provides constraints.

### Does Claude Memory sync across machines?

No. Memory files live in `.claude/projects/` on your local machine and are not version-controlled by default. If you switch machines, your memories don't follow. CLAUDE.md, being checked into git, syncs everywhere the repo exists.

### How large should CLAUDE.md be?

Keep it under 200-300 lines of essential rules. Every line consumes context window in every session. If you're adding context that only applies sometimes, consider whether it belongs in a skill file or memory instead.

### Can I convert memories into CLAUDE.md rules?

Yes — this is a natural workflow. When you notice Claude Memory has accumulated a correction that should apply team-wide (like "never import Next.js modules in pipeline scripts"), promote it to a CLAUDE.md rule and delete the memory. The memory served as a draft; CLAUDE.md makes it permanent and shared.
