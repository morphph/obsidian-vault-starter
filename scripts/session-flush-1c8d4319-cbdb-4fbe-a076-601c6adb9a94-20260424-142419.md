# Claude Memory vs CLAUDE.md：两套持久化机制该怎么选？

**TL;DR：** **Claude Memory** 是跨会话的用户级自动记忆——Claude 自己决定记什么、什么时候用。**CLAUDE.md** 是项目级的指令文件——你手动写规则，Claude 每次启动时读取并严格遵守。两者不是竞争关系，而是互补关系：Memory 记住"你是谁"，CLAUDE.md 规定"这个项目怎么做"。如果只能选一个先配，**CLAUDE.md 的 ROI 更高**，因为它直接影响每一次代码输出的质量。

## 概述：Claude Memory

Claude Memory 是 Anthropic 在 2026 年初大幅升级的跨会话持久化系统。它的核心设计理念是让 Claude 像一个长期合作的同事一样，记住你的偏好、角色、技术栈和工作习惯，而不需要你每次对话都重复交代背景。

Memory 的运作方式分为两层。第一层是 Claude.ai 平台级的记忆，所有 Claude 对话（包括网页端、桌面端、移动端）共享一套记忆库，Claude 会在对话中自动识别值得记住的信息并存储。第二层是 Claude Code 的 auto memory，它以文件形式保存在 `~/.claude/projects/` 目录下，按项目隔离，支持用户手动管理。

Memory 的关键特征是**自动化**——你不需要刻意维护它。Claude 会在你提到"我是数据工程师"、"我们团队用 Go"、"不要在测试里 mock 数据库"这类信息时，自主决定是否存储。[Anthropic 在 2026 年 3 月的记忆升级](/zh/blog/anthropic-claude-memory-upgrades-importing)中还加入了从 ChatGPT 导入记忆的功能，降低了迁移成本。

## 概述：CLAUDE.md

CLAUDE.md 是 Claude Code 的项目级指令文件，功能类似于一份"项目规则手册"。它是一个 Markdown 文件，放在项目根目录下，随代码一起提交到 Git 仓库，团队所有成员共享同一套规则。

CLAUDE.md 解决的问题很具体：确保 Claude Code 在每次会话中都遵守项目的编码规范、构建流程、架构约束和质量门禁。比如"提交前必须跑 `npm test`"、"不要在 pipeline 脚本里 import Next.js 模块"、"中文内容必须用 CJK 分词计数"这类规则，写一次就永久生效。

CLAUDE.md 的关键特征是**确定性**——它不是建议，是指令。Claude Code 在每次启动时都会读取并严格遵守，不存在"忘记"或"选择性执行"的问题。在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中，CLAUDE.md 位于最基础的项目上下文层，是整个扩展栈的根基。

## 持久化机制：核心差异深度解析

这两套系统的根本区别在于**谁控制内容**和**内容如何被使用**。

Claude Memory 是一个由 AI 管理的知识库。Claude 在对话过程中自动判断哪些信息值得记住，将其分类存储（用户信息、反馈、项目背景、外部引用），并在后续对话中自主决定何时调用。这意味着 Memory 的内容是渐进式积累的——你用得越多，Claude 对你的理解越深。但它也有一个固有局限：你无法保证某条记忆在下次对话中一定会被使用。Claude 的记忆调用是概率性的，取决于它对当前上下文的相关性判断。

CLAUDE.md 是一个由人类管理的指令文件。它的加载时机是确定的（每次 Claude Code 启动时），执行方式是强制的（不是"参考"，是"遵守"），内容是显式的（你写什么它就读什么）。这种确定性使得 CLAUDE.md 特别适合编码规范和质量门禁——你不会希望"提交前跑测试"这条规则被 Claude 自主判断为"这次不太相关"而跳过。

在[Claude Code 记忆系统的详细解析](/zh/blog/claude-code-memory)中，这两者被定义为互补关系：CLAUDE.md 是"规则手册"，Memory 是"自动笔记本"。规则手册规定做事的标准，笔记本记录做事的上下文。

## 团队协作：CLAUDE.md 的结构化优势

对于团队场景，CLAUDE.md 的优势是压倒性的。

首先是**一致性**。当 CLAUDE.md 写明"commit message 必须准确描述改了什么"、"新功能先讨论设计再编码"，团队中每个使用 Claude Code 的成员都会收到相同的约束。Memory 是私有的——你的 Claude 知道你喜欢简洁的回复，但你同事的 Claude 不知道。

其次是**可审计性**。CLAUDE.md 是代码仓库的一部分，每次修改都有 Git 记录。你可以 code review 规则变更，回滚不合理的约束，追踪谁在什么时候加了什么规则。Memory 虽然可以通过文件系统查看，但它分散在多个文件中，没有统一的变更追踪。

第三是**可组合性**。除了根目录的 CLAUDE.md，Claude Code 还支持 `skills/*/SKILL.md` 文件——这是一套可复用的指令模块。你可以为不同任务（写测试、做 code review、生成 SEO 内容）编写专门的 SKILL.md，按需激活。这在[Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)中有详细介绍。Memory 不支持这种模块化组织。

[多家企业的工程实践](/zh/blog/claude-code-enterprise-engineering-ramp-shopify-spotify)表明，CLAUDE.md 是大规模 Claude Code 部署的核心治理工具。Shopify、Spotify 等团队都将项目规范写入 CLAUDE.md，确保 AI 生成的代码符合团队标准。

## 什么时候该用 Claude Memory

选择 Memory 作为主要持久化手段的场景：

- **个人偏好和工作习惯**：你喜欢简洁回复、不要 emoji、代码注释用英文——这些跨项目通用的偏好，让 Memory 自动记住比每个项目的 CLAUDE.md 都写一遍更合理
- **角色和技术背景**：你是有十年 Go 经验但第一次写 React 的后端工程师——这条信息帮助 Claude 调整解释的深度和类比方式
- **外部资源指引**：bug 在 Linear 的 INGEST 项目里追踪、告警看 Grafana 的某个 dashboard——这类"去哪里找信息"的指引，Memory 比 CLAUDE.md 更适合
- **跨平台使用**：如果你同时使用 Claude.ai 网页端和 Claude Code 终端，平台级 Memory 是唯一的跨端持久化方案

## 什么时候该用 CLAUDE.md

选择 CLAUDE.md 作为主要持久化手段的场景：

- **构建和部署流程**：`npm run build`、`npm test`、`npm run lint`——这些命令写在 CLAUDE.md 里，Claude Code 每次都会遵守
- **质量门禁**：提交前必须通过的检查、不允许跳过的测试、不允许注释掉的 lint 规则——需要确定性执行，不能交给概率性调用
- **架构约束和已知陷阱**：不要在 pipeline 脚本里 import Next.js 模块、`upsertKeyword()` 必须传三个参数——这类项目特定的"坑"，写在 CLAUDE.md 里一劳永逸
- **团队规范统一**：所有使用 Claude Code 的团队成员需要遵守相同的编码标准和工作流程
- **编程 Agent 工作流**：当 Claude Code 以 [Agent 模式](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)运行子任务时，CLAUDE.md 确保每个子 Agent 都读取并遵守项目规则

## 结论

**CLAUDE.md 和 Claude Memory 不是二选一的关系，而是不同层次的持久化工具。** CLAUDE.md 负责项目级的规则和约束，以确定性方式加载，适合团队共享和质量治理。Memory 负责用户级的偏好和背景，以自动化方式积累，适合个性化体验和跨项目上下文。

实际操作建议：**先把 CLAUDE.md 写好**。一个结构清晰的 CLAUDE.md 能立刻提升 Claude Code 在你项目中的输出质量。Memory 则让它自然积累——用得越多，Claude 对你的理解越准确。如果你正在为团队部署 Claude Code，CLAUDE.md 是第一优先级；如果你是个人开发者想让 Claude 更懂你，Memory 会在几次对话后自动生效。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。两者作用于不同层次：CLAUDE.md 在项目层面提供确定性指令，Memory 在用户层面提供个性化上下文。Claude Code 启动时会同时加载两者，不存在冲突。如果两者内容矛盾，CLAUDE.md 的显式指令通常优先于 Memory 的隐式偏好。

### CLAUDE.md 的内容会影响 Claude.ai 网页端吗？

不会。CLAUDE.md 仅在 Claude Code 终端环境中生效。Claude.ai 网页端、桌面端和移动端使用平台级 Memory 系统，不读取本地文件系统中的 CLAUDE.md。如果你需要跨平台的持久化，只能依赖 Memory。

### 如何判断某条规则应该写在 CLAUDE.md 还是让 Memory 记住？

判断标准是**确定性需求**和**共享需求**。如果这条规则必须每次都执行（质量门禁、构建命令），写 CLAUDE.md。如果这条规则只是你的个人偏好（回复风格、解释深度），让 Memory 记住。如果这条规则需要团队共享，一定写 CLAUDE.md。
