**CLAUDE.md** is a single file (or hierarchy of files) living in your repository. It's read in its entirety at session start — no filtering, no relevance matching. This makes it deterministic and predictable: if a rule is in CLAUDE.md, Claude follows it. But it also means CLAUDE.md has a practical size limit. An excessively long CLAUDE.md wastes context window on every interaction, whether or not the instructions are relevant to the current task.

The storage difference has real consequences for teams. CLAUDE.md travels with `git clone` — a new team member gets full project instructions immediately. Memory starts empty and builds over sessions, which means a new developer's Claude Code won't know their preferences or prior context until several conversations have passed.

## Content and Purpose: What Belongs Where

The most common mistake teams make is putting the wrong type of information in the wrong system. Here's the decision framework.

**Put it in CLAUDE.md when:**

- It's a rule everyone on the team must follow (coding standards, required test commands, forbidden patterns)
- It's a fact about the project that doesn't change often (tech stack, architecture decisions, deployment targets)
- It needs to be enforced deterministically every session (quality gates, pre-commit checks, style requirements)
- A new team member needs to know it on day one (build commands, environment setup, key gotchas)

Examples: "Run `npm run build` before every commit." "Use Tailwind v4 utility classes, never inline styles." "The API uses snake_case, the frontend uses camelCase." "Never import Next.js server modules in pipeline scripts."

**Put it in Memory when:**

- It's specific to you, not the team (your role, your preferences, how you like explanations)
- It emerged from conversation, not from a deliberate decision (a debugging insight, a clarification about how something works)
- It changes frequently as the project evolves (current sprint goals, ongoing incidents, who's working on what)
- It's a correction to Claude's behavior that only you care about (response length preferences, verbosity level, explanation style)

Examples: "User is a data scientist, not a frontend developer — frame frontend explanations using backend analogues." "Don't summarize changes at the end of responses, user reads the diff directly." "Current priority is the auth migration, driven by legal compliance deadline of March 15."

The [Claude Code memory system documentation](/blog/claude-code-memory) covers both systems in detail, including how they interact during a session.

## Team Workflows: Shared vs Individual Context

For solo developers, the distinction between Memory and CLAUDE.md is less critical — you can put everything in CLAUDE.md and it works fine. The distinction becomes essential once a team is involved.

**CLAUDE.md as team alignment tool.** When three engineers work on the same codebase, CLAUDE.md ensures Claude Code behaves consistently for all of them. Without it, each developer's Claude might format code differently, skip tests, or make different architectural assumptions. CLAUDE.md eliminates this variance. It's particularly valuable for [agentic coding](/glossary/agentic-coding) workflows where Claude operates with significant autonomy — the rules in CLAUDE.md are the guardrails that keep autonomous actions predictable.

**Memory as individual adaptation.** Each developer's Memory captures their unique working relationship with Claude. A senior architect's Memory might note "user prefers high-level design discussions before implementation" while a junior developer's might note "user is learning TypeScript, explain type system concepts when they come up." These shouldn't be in CLAUDE.md because they're individual preferences, not team standards.

**The overlap zone.** Some information legitimately fits in either system. A decision made during a conversation ("we decided to use SQLite instead of Postgres for the pipeline database") could go in Memory as a project-type memory, or it could be added to CLAUDE.md as an architectural constraint. The rule of thumb: if the decision affects how Claude should behave for anyone on the team, put it in CLAUDE.md. If it's context that helps Claude understand why things are the way they are, Memory is fine.

Teams using Claude Code's [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) — skills, hooks, MCP servers, and agent teams — should be especially intentional about CLAUDE.md. These systems compose with CLAUDE.md instructions, so unclear or contradictory rules create unpredictable behavior across the stack.

## Maintenance and Lifecycle: Who Updates What

**CLAUDE.md requires active maintenance.** It's a file in your repo, so it only changes when someone edits it and commits the change. This is a strength (predictable, reviewable via PR) and a weakness (can become stale if the project evolves faster than the documentation). Teams should treat CLAUDE.md updates as part of their development workflow — when you change the build process, update CLAUDE.md in the same commit.

Best practices for CLAUDE.md maintenance:

- Keep it under 500 lines — longer files waste context on every session
- Review it quarterly for stale instructions
- Use directory-level CLAUDE.md files for module-specific rules instead of bloating the root file
- Write rules as imperatives ("Run X before Y") not descriptions ("The project uses X")

**Memory is self-maintaining but not infallible.** Claude automatically updates or removes memories that become outdated, and it checks memory accuracy before acting on recalled information. But the system isn't perfect — memories can become stale if the underlying project changes without Claude being involved (e.g., a colleague refactors a module while you're on vacation). When you return, Claude's memories about that module's structure may be wrong.

Memory maintenance tips:

- Periodically tell Claude to "review and clean up stale memories"
- If something feels wrong, ask Claude "what do you remember about X?" and correct inaccuracies
- When starting a new project phase, explicitly note it so Claude can update project-type memories

## How They Interact: The Loading Order

Understanding the loading order clarifies how conflicts are resolved:

1. **CLAUDE.md loads first** — the full file, unconditionally, at session start
2. **MEMORY.md index loads next** — the one-line-per-entry index file
3. **Individual memories load on demand** — when contextually relevant during the session

If CLAUDE.md says "use tabs for indentation" and a Memory record says "user prefers spaces," CLAUDE.md takes precedence because it's treated as authoritative project configuration. Memory provides context and preferences; CLAUDE.md provides rules and constraints.

This hierarchy is intentional. CLAUDE.md represents the team's agreed-upon standards. Memory represents one individual's accumulated context. When they conflict, the team standard wins. This prevents a situation where one developer's preference, captured in Memory, overrides a deliberate team decision.

The practical implication: never rely on Memory for critical project rules. If something must be followed every time, it belongs in CLAUDE.md. Memory is for the "nice to know" context that makes Claude more helpful, not the "must follow" rules that keep the project consistent.

## When to Choose Claude Memory

Choose Claude Memory as your primary persistence mechanism when:

- **You're a solo developer** and don't need team-shared rules. Memory captures everything organically without you having to write and maintain a separate file. Over a few sessions, Claude learns your preferences, your project's quirks, and your workflow patterns.

- **Context is evolving rapidly.** During an active sprint or incident, the relevant context changes daily. Memory captures these shifts naturally through conversation. Writing and rewriting CLAUDE.md for transient project state adds friction without value.

- **You want Claude to adapt to you.** Memory's feedback type is specifically designed for this — when you correct Claude's approach or confirm a non-obvious choice, it records the guidance for future sessions. This creates a progressively more personalized experience that CLAUDE.md can't replicate.

- **You need reference pointers.** Memory's reference type stores where to find information in external systems — which Linear project tracks bugs, which Grafana dashboard to check, which Slack channel has context. These are individual workflow pointers that don't belong in project configuration.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence mechanism when:

- **You're on a team** and need consistent Claude behavior across developers. CLAUDE.md is the only way to guarantee that Claude Code follows the same rules for everyone. Without it, each developer gets a slightly different AI assistant shaped by their individual conversation history.

- **Reliability is non-negotiable.** CLAUDE.md loads fully every session. There's no risk of relevant instructions being missed due to poor relevance matching. For critical rules — "never push to main," "always run the full test suite," "use the staging database for pipeline testing" — this determinism is essential.

- **You're onboarding new team members.** A developer cloning the repo for the first time gets full Claude Code instructions immediately via CLAUDE.md. Memory starts from zero and takes sessions to build useful context. For teams with frequent rotation or contractors, CLAUDE.md is the faster path to productive AI assistance.

- **You want reviewable, auditable configuration.** CLAUDE.md changes go through your normal PR review process. You can see exactly what changed, who changed it, and why. Memory changes happen silently during conversations and aren't tracked in version control.

- **You use [skills files](/blog/5-claude-code-skills-i-use-every-single-day)** for specialized workflows. Skills reference CLAUDE.md conventions and compose with project-level rules. A well-structured CLAUDE.md makes skills more effective by providing consistent baseline context.

## The Right Approach: Use Both

For most projects beyond a single-person prototype, the answer is both. Here's a concrete setup:

**CLAUDE.md handles:**
- Build and test commands
- Coding standards and forbidden patterns
- Architecture decisions and constraints
- Documentation rules
- Deployment workflow
- Quality gates (what must pass before commit)

**Memory handles:**
- Your role and expertise level
- Your communication preferences
- Ongoing project context (current sprint, active incidents)
- Feedback corrections (how you want Claude to behave differently)
- External system references (where to find things outside the repo)
- Decisions made in conversation that haven't been formalized into CLAUDE.md yet

**Migration path:** When a memory captures something that the whole team should follow, promote it to CLAUDE.md. When CLAUDE.md contains a personal preference that only you care about, move it to Memory. Over time, this natural migration keeps both systems focused on what they do best.

The [nine principles for writing effective skills](/blog/9-principles-writing-claude-code-skills) apply equally to writing effective CLAUDE.md files — clarity, specificity, and avoiding over-prescription lead to better AI behavior regardless of which persistence system you use.

## Verdict

**CLAUDE.md is the foundation; Memory is the personalization layer.** Start with CLAUDE.md — even a minimal one with build commands and key constraints makes Claude Code dramatically more effective. Add Memory naturally over time by working with Claude and letting it learn your context.

If you're forced to choose one: **pick CLAUDE.md for teams, Memory for solo work.** But you rarely need to choose. The two systems operate at different levels of the stack and complement each other well. CLAUDE.md tells Claude how your project works. Memory tells Claude how you work. Both make the AI more useful, in different ways that don't conflict.

For a complete walkthrough of setting up both systems, see the [Claude Code memory system guide](/blog/claude-code-memory).

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md rules?

No. CLAUDE.md loads first and is treated as authoritative project configuration. If a memory contradicts a CLAUDE.md instruction, the CLAUDE.md rule takes precedence. Memory provides supplementary context and preferences, not override-capable rules. This hierarchy is intentional — team-agreed standards in CLAUDE.md should never be overridden by one individual's accumulated conversation context.

### Does CLAUDE.md work without Claude Memory enabled?

Yes. CLAUDE.md is a standalone configuration file that works independently of the memory system. Claude Code reads it at session start regardless of whether auto-memory is active. Many teams use CLAUDE.md exclusively and never interact with the memory system. The reverse is also true — Memory works without a CLAUDE.md file, though Claude Code is less effective without explicit project instructions.

### How do I move a memory into CLAUDE.md?

If Claude has learned something through conversation that the whole team should follow, manually copy the relevant guidance from the memory file (found in `~/.claude/projects/<project>/memory/`) into your CLAUDE.md, then tell Claude to remove the now-redundant memory. Treat this like any other CLAUDE.md change — commit it, get it reviewed. The promoted rule now applies to all team members, not just you.

### Is there a size limit for CLAUDE.md?

There's no hard character limit, but practical performance degrades with length. CLAUDE.md loads entirely into every session's context window, so a 2,000-line file wastes context on instructions that aren't relevant to the current task. Keep the root CLAUDE.md under 500 lines. Use directory-level CLAUDE.md files for module-specific rules, and use skill files for task-specific workflows like testing or deployment.

### Can I share my Claude Memory with teammates?

Not directly — Memory files are stored in your local `~/.claude/` directory and aren't designed for sharing. If you discover valuable context that teammates should have, the right path is to promote it to CLAUDE.md (for rules) or to a shared documentation file (for reference context). Anthropic's [memory import feature](/blog/anthropic-claude-memory-upgrades-importing) supports importing from other AI tools, but not cross-user memory sharing within Claude Code.
