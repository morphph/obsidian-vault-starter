**Codex custom agents** are configured through the Codex UI or API. You specify a name, a system prompt (natural language instructions), an optional setup script (shell commands that run before the task begins), and environment variables. The setup script handles dependency installation, database seeding, or any environment preparation the agent needs. Once saved, any team member with access can invoke the agent by name.

The strength of this approach is simplicity. A team lead writes the agent definition once, and it becomes a shared tool. The limitation is rigidity — custom agents are static templates. They don't adapt based on what they discover during execution, and they can't spawn sub-tasks or coordinate with other agents mid-run. Each Codex task runs in isolation.

**Claude Code subagents** use a typed system. When spawning an agent, you specify a `subagent_type` — each type comes with predefined tool access and behavioral patterns:

- **Explore**: Read-only search agent for locating code across the codebase
- **Plan**: Software architect agent that designs implementation strategies
- **code-reviewer**: Reviews changes against project standards
- **general-purpose**: Full-capability agent for complex multi-step tasks
- **pipeline-reviewer**: Specialized for CI/CD pipeline script changes

Beyond built-in types, behavior is shaped by SKILL.md files — reusable instruction sets checked into your repository. A SKILL.md for "write integration tests" defines exactly how the agent should approach test generation, what patterns to follow, and what to avoid. Because these files live in version control, they evolve with your codebase and are consistent across all team members using Claude Code.

The tradeoff: Claude Code's system is more powerful but requires more upfront investment in writing good SKILL.md files. Codex custom agents are faster to set up but less flexible. For teams that have already invested in [writing effective skills](/blog/9-principles-writing-claude-code-skills), Claude Code's approach compounds in value over time.

## Execution Model: Detailed Analysis

The execution model — where and how agents run — has practical implications for speed, safety, and workflow integration.

**Codex runs agents in the cloud.** When you submit a task, Codex provisions a sandboxed container, clones your repository at the specified branch, runs the setup script, and executes the task. The agent has no internet access by default, which prevents supply-chain attacks and data exfiltration but also means it cannot fetch documentation, call external APIs, or install packages not already in the setup script. Tasks run asynchronously — you submit them and check back later for results delivered as pull requests or code diffs.

This model excels for batch workflows. You can queue ten tasks, close your laptop, and review ten PRs the next morning. It also provides strong isolation guarantees — a misbehaving agent cannot affect your local environment or other running tasks.

**Claude Code runs agents locally.** Subagents execute in your terminal session (or in git worktrees for isolation). They have the same access as your shell — file system, network, installed tools, running services. Results stream back to the parent session in real time, allowing you to review intermediate output, redirect the agent, or spawn follow-up agents based on what was discovered.

This model excels for interactive investigation. A common pattern: spawn an Explore agent to find relevant code, review its findings, then spawn a general-purpose agent to implement changes based on what was found. The parent session maintains context across the entire workflow. The tradeoff is that local execution means agents can affect your environment — though the worktree isolation option mitigates this for file-system changes.

For teams that need both models, it is worth noting that Claude Code can approximate Codex's async behavior by running agents in the background with `run_in_background: true`, though the session must remain active.

## Multi-Agent Coordination: Detailed Analysis

The ability to coordinate multiple agents — not just run them independently — separates simple task delegation from true [agentic coding](/glossary/agentic-coding) workflows.

**Codex custom agents operate independently.** Each task submitted to Codex runs in its own sandbox with no awareness of other running tasks. You cannot have one agent's output feed into another agent's input within the Codex system itself. Coordination happens externally — you review Agent A's PR, then submit a follow-up task to Agent B referencing those changes. This is by design: isolation is a safety feature, not a limitation Codex is trying to solve.

For workflows that genuinely require independent, parallelizable tasks — "write tests for these five modules" — this works well. Submit five tasks, get five PRs. But for workflows where agents need to collaborate — "analyze the codebase architecture, then plan a refactor, then implement it" — the manual handoff between steps creates friction.

**Claude Code subagents can coordinate within a session.** The parent session acts as an orchestrator. You can spawn multiple agents in parallel (by including multiple Agent calls in a single message), wait for all results, synthesize findings, and then spawn the next round of agents with refined instructions. Agents can also be addressed by name using `SendMessage` to continue a previously spawned agent's work with additional context.

A practical example from our [subagent examples guide](/blog/claude-code-subagents-examples): spawn three Explore agents in parallel to search for all usages of a deprecated API across frontend, backend, and test code. When all three return, the parent session compiles a migration plan and spawns implementation agents for each affected area — each with precise file paths and replacement patterns from the exploration phase.

This orchestration capability is Claude Code's strongest advantage for complex refactoring and investigation workflows. The [extension stack architecture](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) — skills, hooks, agents, and MCP servers — provides the coordination primitives that make this practical.

## When to Choose Codex Custom Agents

Codex custom agents are the right choice when your workflow matches this pattern: **define once, delegate many times, review results asynchronously.**

Specific scenarios where Codex excels:

- **Team standardization**: You want every developer using the same agent definitions for common tasks like test generation, migration, or code review, without requiring them to learn a new tool
- **Batch task processing**: You have multiple independent tasks (bug fixes, test coverage expansion, documentation updates) that can run in parallel without coordination
- **Security-sensitive environments**: The sandboxed, no-internet execution model provides strong guarantees against supply-chain attacks or accidental data exposure
- **Non-technical stakeholders**: Product managers or designers can submit tasks through the ChatGPT interface without needing terminal access
- **CI/CD integration**: Codex tasks can be triggered programmatically, fitting into existing automation pipelines

The [Codex VS Code extension](/blog/codex-vscode) also provides IDE integration for developers who prefer not to use the ChatGPT web interface, bridging the gap between cloud execution and local development experience.

## When to Choose Claude Code Subagents

Claude Code subagents are the right choice when your workflow requires **real-time orchestration, contextual awareness, and iterative refinement.**

Specific scenarios where Claude Code excels:

- **Complex investigation**: You need to explore a codebase, understand architecture, and then plan and execute changes — all in one session with maintained context
- **Multi-step refactoring**: The task requires coordination between agents — one analyzes, another plans, a third implements — with human review at each stage
- **Local environment dependence**: Your workflow requires running tests, building the project, or interacting with local services during agent execution
- **Skill-driven teams**: You have already invested in SKILL.md files and CLAUDE.md project context, and you want agents that automatically inherit these standards
- **Real-time oversight**: You need to monitor agent progress, redirect when something goes off track, and make decisions based on intermediate results

For teams already using Claude Code's [hooks system](/blog/claude-code-hooks-a-complete-guide-to-automating-your-ai-coding-workflow) and [memory system](/blog/claude-code-memory), subagents inherit all of these configurations automatically — no additional setup required.

## Verdict

**Choose Codex custom agents if you need cloud-based, fire-and-forget task delegation with strong isolation and team-wide standardization.** Codex's strength is making multi-agent workflows accessible — define an agent once, and anyone on your team can use it without terminal expertise.

**Choose Claude Code subagents if you need interactive, orchestrated multi-agent workflows with real-time coordination and deep project context.** Claude Code's typed agent system, combined with SKILL.md files and session-level orchestration, provides more powerful primitives for complex engineering tasks.

For many teams, the practical recommendation is to **use both**. Codex for batch delegation of well-defined tasks (test generation, migrations, documentation). Claude Code for interactive investigation, complex refactoring, and any workflow where agents need to coordinate. The two systems are complementary, not competing — they address different points on the autonomy-vs-control spectrum that every [agentic coding](/glossary/agentic-coding) workflow must navigate.

## Frequently Asked Questions

### Can Codex custom agents call other agents during execution?

No. Each Codex task runs in an isolated sandbox with no ability to spawn sub-tasks or communicate with other running agents. Coordination between tasks must happen externally — you review one agent's output and manually submit follow-up tasks. This isolation is a deliberate safety design choice.

### How many Claude Code subagents can run in parallel?

Claude Code can spawn multiple subagents simultaneously by including multiple Agent calls in a single message. There is no hard-coded limit on concurrent agents, but practical limits come from your machine's resources and the context window. For most workflows, two to five parallel agents is the productive sweet spot before coordination overhead outweighs parallelism gains.

### Do Codex custom agents work with private repositories?

Yes. Codex integrates with GitHub and can access private repositories that your organization has connected. The agent clones the repository into its sandboxed environment at the branch you specify. Access controls follow your existing GitHub permissions.

### Can Claude Code subagents access MCP servers?

Yes. Subagents inherit the MCP server configuration from the parent session. This means a subagent can query databases, call external APIs, or interact with any tool connected via the Model Context Protocol — giving it capabilities beyond the local file system and shell.

### Is there a cost difference between the two approaches?

Codex custom agents are included in ChatGPT Pro and Team subscription plans, making costs predictable regardless of usage volume. Claude Code subagents use Anthropic's usage-based API billing — you pay per token across all agents in a session. For light usage, Claude Code may cost less; for heavy batch processing, Codex's flat-rate plans can be more economical.
