**CLAUDE.md uses a flat-file, always-loaded model.** The entire contents of CLAUDE.md are injected into Claude Code's system prompt at the start of every session. There's no relevance filtering — every instruction is present every time. This makes it deterministic but creates a scaling constraint: a 2,000-line CLAUDE.md consumes context window space even when you're working on something that only needs 10 lines of those instructions.

Claude Code actually supports a hierarchy of CLAUDE.md files: a global `~/.claude/CLAUDE.md` for cross-project preferences, a project-level `CLAUDE.md` at the repo root, and directory-level CLAUDE.md files deeper in the tree. All applicable files are merged and loaded. This hierarchy gives you some organizational control, but everything still loads every time.

**The practical difference:** CLAUDE.md is like environment variables — always set, always available, zero retrieval latency. Memory is like a database — stores more, retrieves selectively, but can miss. For rules that must never be skipped (build gates, security constraints, style mandates), CLAUDE.md is strictly more reliable. For context that's situationally useful (your role, past decisions, external references), Memory is more efficient.

## Content Boundaries: What Belongs Where

**This is where most developers get confused.** Both systems persist information across sessions, so the question becomes: when you learn something that should stick around, where does it go?

The boundary is clearer than it looks. CLAUDE.md answers the question: *"How should Claude Code behave in this project?"* Memory answers the question: *"What does Claude Code need to know about me and the current situation?"*

### What belongs in CLAUDE.md

- **Build and validation commands**: `npm run build`, `npm test`, lint configurations
- **Quality gates**: "All tests must pass before commit," "run validate-pipeline.ts for pipeline changes"
- **Coding conventions**: naming patterns, file organization rules, import restrictions
- **Known gotchas**: "Don't import Next.js modules in pipeline scripts," "use CJK word count for Chinese content"
- **Workflow rules**: "New feature → discuss design first," "bug fix → systematic debug, not trial-and-error"
- **Documentation requirements**: which docs to update when specific files change

These are project-level truths. They apply to every developer, every session, every task. They don't change based on who's typing. As detailed in our [deep dive on Claude Code's programmable layers](/blog/claude-code-seven-programmable-layers), CLAUDE.md sits at the project configuration layer — it's infrastructure, not personalization.

### What belongs in Memory

- **Your role and expertise**: "I'm a data scientist investigating logging" or "deep Go expertise, new to React"
- **Behavioral feedback**: "Don't mock the database in tests — we got burned last quarter" or "stop summarizing at the end of responses"
- **Project context not in code**: "Merge freeze begins March 5 for mobile release," "auth rewrite is driven by legal compliance, not tech debt"
- **External references**: "Pipeline bugs tracked in Linear project INGEST," "oncall latency dashboard at grafana.internal/d/api-latency"

These are personal, situational, or ephemeral. They change based on who you are, what's happening right now, or where to find information outside the repo. They don't belong in version control because they'd either clutter the project config or expose individual preferences to the whole team.

### The gray zone

Some information legitimately could go in either place. A team convention like "we prefer bundled PRs over split ones for refactors" could be a CLAUDE.md rule or a feedback memory. The deciding factor: **does every developer on the team need to follow this?** If yes, CLAUDE.md. If it's one developer's preference that might not apply to others, Memory.

## Team Dynamics: Shared vs Personal Persistence

**CLAUDE.md creates alignment across a team.** When you add a rule to CLAUDE.md and push it to the repository, every developer's Claude Code sessions immediately follow that rule. This is powerful for engineering standards — you define your quality gates once, and the AI enforces them consistently whether a senior engineer or a new hire is driving.

This also means CLAUDE.md changes go through code review. A teammate can't silently change the AI's behavior for everyone without a PR. The instruction history lives in git log. You can bisect to find when a rule was added, who added it, and why. For organizations that care about AI governance — and in 2026, most do — this auditability matters.

**Memory creates personalization without interference.** A frontend specialist and a backend specialist working on the same repo should get different explanations, different levels of detail on different topics, and different workflow recommendations. Memory enables this without either developer's preferences polluting the other's experience.

The risk with Memory is drift. Over months of accumulated memories, a developer might have outdated project context ("merge freeze starts March 5" is stale by April) or contradictory feedback ("prefer small PRs" from January vs "bundle refactors into one PR" from March). Memory files require occasional pruning, but there's no built-in mechanism to prompt this. Claude Code does flag when a memory conflicts with current code state, but it can't catch all staleness.

**The practical recommendation:** Use CLAUDE.md as your source of truth for project behavior. Use Memory as a personal optimization layer on top. If a memory keeps influencing Claude Code's behavior in ways that should apply to the whole team, promote it to CLAUDE.md. If a CLAUDE.md rule is really just one person's preference, demote it to Memory.

## Reliability and Failure Modes

**CLAUDE.md fails predictably.** If an instruction in CLAUDE.md is wrong, it's wrong every time, for everyone. This makes bugs easy to identify and fix — you see consistent misbehavior, check CLAUDE.md, find the bad rule, fix it, push. The failure mode is *wrong instructions consistently applied*.

**Memory fails stochastically.** A memory might not be recalled when it should be (the index description didn't match the current context well enough), or it might be recalled when it shouldn't be (a stale project memory about a deadline that already passed). The failure mode is *right information inconsistently applied*. This is harder to debug because the same task might succeed one session and fail the next depending on whether the relevant memory was loaded.

Memory also has a freshness problem. Claude Code warns that "memory records can become stale over time" and instructs itself to "verify that the memory is still correct and up-to-date by reading the current state." But verification depends on Claude Code recognizing that a memory needs checking — which is itself a heuristic judgment.

For critical project rules — security constraints, build gates, deployment procedures — CLAUDE.md's determinism is non-negotiable. You don't want "run tests before commit" to be a memory that might not load on a busy Friday afternoon. That's a CLAUDE.md rule, full stop.

## Practical Setup: Configuring Both Systems

### Setting up CLAUDE.md

Create a `CLAUDE.md` file at your repository root. Start minimal and add rules as you discover what Claude Code needs to know:

```markdown
# CLAUDE.md

## Commands
npm run dev          # Local dev server
npm run build        # Production build
npm test             # Run tests
npm run lint         # Lint check

## Quality Gates
Before ANY commit: npm run build && npm test must pass.

## Conventions
- Use TypeScript strict mode
- Database columns use snake_case
- API routes follow REST naming
```

For a detailed breakdown of how CLAUDE.md fits into the broader extension stack, see our [guide to Claude Code's skills, hooks, agents, and MCP](/blog/claude-code-extension-stack-skills-hooks-agents-mcp).

### Working with Memory

Memory doesn't require setup — Claude Code creates the memory directory automatically. You interact with it conversationally:

- **Explicit save**: "Remember that I'm a frontend developer new to this backend codebase"
- **Explicit recall**: "What do you remember about our deploy process?"
- **Explicit forget**: "Forget what I said about preferring small PRs"
- **Automatic save**: Claude Code saves memories proactively when you share role information, correct its behavior, or mention project context

You can also inspect and edit memory files directly in `~/.claude/projects/<project>/memory/`. Each file is plain markdown with YAML frontmatter — no proprietary format, no database, no lock-in.

### The combined workflow

The most effective setup uses both systems in their respective strengths:

1. **CLAUDE.md** for everything the team needs to agree on — build commands, quality gates, coding standards, known gotchas
2. **Memory** for everything personal — your expertise level, your behavioral preferences, your knowledge of ongoing project context
3. **SKILL.md files** for reusable task instructions that go beyond what CLAUDE.md covers — editorial guidelines, review checklists, pipeline configurations. See our [guide to writing effective skills](/blog/9-principles-writing-claude-code-skills) for details.

## When to Choose Claude Memory

Choose Claude Memory as your primary persistence mechanism when:

- **You're a solo developer** who doesn't need team-shared instructions and wants the lowest-friction persistence. Memory's automatic saving means you rarely need to manually configure anything.
- **You work across many projects** and want your personal preferences (explanation style, commit message format, response length) to follow you. Global memory applies across all projects without duplicating CLAUDE.md files.
- **Your context is inherently personal** — your role, your expertise gaps, your feedback on Claude Code's behavior. These shouldn't be in a shared project file.
- **You need to track ephemeral project state** — ongoing incidents, upcoming deadlines, who's working on what. This information changes frequently and doesn't belong in version-controlled config.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence mechanism when:

- **You work on a team** and need consistent AI behavior across all developers. CLAUDE.md is the only mechanism that guarantees every Claude Code session follows the same rules.
- **Your rules are critical and must never be skipped** — security constraints, build validation, deployment procedures. CLAUDE.md's deterministic loading means these rules are always active.
- **You want auditability** — who added this rule, when, and why. CLAUDE.md changes go through git history and code review.
- **You're onboarding new team members** who need Claude Code to immediately understand the project's conventions without building up months of personal memory.

For most teams, the answer is both. CLAUDE.md is the foundation; Memory is the personalization layer. Read our [Claude Code memory system explainer](/blog/claude-code-memory) for implementation details on how the two systems interact at runtime.

## Verdict

**Use both, but know which one is load-bearing.** CLAUDE.md is your project's source of truth — deterministic, version-controlled, team-shared. Every project using Claude Code should have one, even if it starts as five lines of build commands. Memory is your personal optimization layer — it makes Claude Code better at working with *you specifically*, but it should never be the only place a critical project rule lives.

The most common mistake is putting team conventions in Memory instead of CLAUDE.md. If you've told Claude Code something three times across different sessions and it keeps being relevant, promote it to CLAUDE.md. The second most common mistake is cluttering CLAUDE.md with personal preferences that only apply to you — that's what Memory is for.

**Start with CLAUDE.md.** Add your build commands, quality gates, and top three coding conventions. Let Memory accumulate naturally as you work. Review your memories quarterly and promote anything that should be shared to CLAUDE.md. This gives you the reliability of explicit configuration with the convenience of automatic personalization. ## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md instructions are loaded into the system prompt and take priority over recalled memories. If CLAUDE.md says "always use TypeScript strict mode" and a memory says "the user prefers loose TypeScript," the CLAUDE.md rule wins. Memory provides additional context but cannot contradict explicit project configuration.

### Does CLAUDE.md consume context window space even when irrelevant?

Yes. The entire contents of all applicable CLAUDE.md files are loaded into every session regardless of the current task. A large CLAUDE.md (1,000+ lines) can consume meaningful context window space. Keep it concise — move detailed task-specific instructions into SKILL.md files that are loaded on demand instead.

### Can I share Claude Memory files across a team?

Technically yes — memory files are plain markdown and could be committed to a shared location. But this defeats the purpose. Memory is designed for personal context that varies between developers. If information should be shared, put it in CLAUDE.md. If you want shared task instructions beyond what CLAUDE.md covers, use [SKILL.md files](/blog/5-claude-code-skills-i-use-every-single-day) instead.

### How do I know if a memory is stale or outdated?

Claude Code attempts to verify memories against current code state before acting on them, but this check is heuristic. Review your memory directory (`~/.claude/projects/<project>/memory/`) periodically. Delete memories about completed deadlines, resolved incidents, or roles that have changed. The `MEMORY.md` index gives you a quick overview of all active memories.

### What happens to CLAUDE.md during Claude Code context compression?

CLAUDE.md content is part of the system prompt and is preserved during context compression. Even when earlier conversation messages are summarized to free space, your project instructions remain fully intact. This is another reliability advantage over Memory, where individual memory files loaded mid-conversation might be summarized along with other context.
