**Codex custom agents** are configured through the Codex dashboard or API. Each agent definition includes a name, a system prompt describing its role and constraints, an optional setup script that runs before each task (installing dependencies, configuring tools, seeding data), and permissions controlling which tools the agent can access. Once defined, a custom agent appears as a selectable option when you submit tasks. This model works well for teams standardizing workflows — your "security reviewer" agent always runs the same static analysis tools, your "docs writer" agent always has access to your style guide.

The limitation is rigidity. Modifying a Codex agent means updating its configuration in the dashboard, and those changes apply to all future tasks. There's no way to tweak an agent's behavior for a single task without creating a variant.

**Claude Code subagents** are defined through a combination of built-in types and project-level files. The built-in types cover common patterns: Explore for fast codebase search, Plan for architecture design, and general-purpose for flexible research and execution. Beyond these, you customize agent behavior through [CLAUDE.md files](/blog/claude-code-memory) and [skill files](/blog/5-claude-code-skills-i-use-every-single-day) that travel with your repository. A `CLAUDE.md` in your project root sets global context — coding standards, architecture constraints, forbidden patterns — that every subagent inherits automatically. Skill files in `skills/*/SKILL.md` define task-specific instructions: how to write tests, how to review PRs, how to generate documentation.

This file-based approach means agent customization is version-controlled, portable across team members, and adjustable per-session. You can override behavior by passing a detailed prompt when spawning the agent, without changing any persistent configuration. The [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) — skills, hooks, agents, and MCP — layers these customizations predictably.

**Bottom line:** Codex agents are better for standardized, repeatable workflows managed centrally. Claude Code subagents are better for adaptive, context-sensitive work where agent behavior needs to flex per task.

## Execution Model and Workflow Integration: Detailed Analysis

How agents execute tasks determines how they fit into your daily development workflow. This is where the Codex vs Claude Code divide is sharpest.

**Codex runs asynchronously in cloud sandboxes.** When you submit a task, Codex clones your repository into a fresh container, runs your agent's setup script, and begins working. You don't need to keep a terminal open or stay online. Codex works through the task — reading files, writing code, running tests — and produces a result (typically a PR diff or a set of changes) that you review later. This is the same model as CI/CD: fire and forget, review the output.

The async model unlocks powerful batch workflows. You can submit a task per module — "add error handling to the user service," "add error handling to the payment service," "add error handling to the notification service" — and Codex works all three simultaneously in separate sandboxes. For large codebases, this parallelism can compress hours of work into minutes of wall-clock time.

The cost is context staleness. Codex works on a snapshot of your repo at submission time. If you push new commits while Codex is working, its changes might conflict. There's no way to tell Codex "actually, I just changed the interface for that function" mid-task.

**Claude Code subagents run interactively in your local environment.** When you spawn a subagent, it starts working immediately with access to your current file system state — including uncommitted changes, running servers, and local databases. You can monitor progress, send follow-up messages to a running agent using `SendMessage`, and course-correct in real time. Background agents notify you when they complete, so you can continue your own work in the foreground while subagents handle parallel tasks.

The [worktree isolation feature](/blog/anatomy-of-git-worktree-add) bridges the gap between interactivity and safety. Spawning a subagent with `isolation: "worktree"` creates a temporary git worktree — the agent works on an isolated copy of your repo, preventing conflicts with your working tree, while still running locally with full environment access. If the agent makes no changes, the worktree is automatically cleaned up.

**Bottom line:** Codex excels at batch, offline task processing where you want to parallelize heavily across cloud resources. Claude Code excels at interactive sessions where real-time steering and live environment access matter. Teams doing large-scale migrations lean toward Codex; teams doing iterative feature development lean toward Claude Code.

## Built-in Specializations and Agent Types

One area where Claude Code has a clear structural advantage is built-in agent specializations. Claude Code ships with pre-configured agent types that encode best practices for common development tasks:

- **Explore agent**: Read-only search agent optimized for finding files, grepping for symbols, and answering "where is X defined" questions. It reads excerpts rather than whole files, making it fast for targeted lookups across large codebases.
- **Plan agent**: Software architect agent that designs implementation plans, identifies critical files, and considers architectural tradeoffs before you write code.
- **Pipeline-reviewer agent**: Reviews changes to pipeline scripts against a project's known-issues registry to prevent re-introducing past bugs — a specialized quality gate.
- **General-purpose agent**: Flexible agent for research, code search, and multi-step execution tasks.

These specializations mean you don't start from scratch when defining agent behavior. The Explore agent already knows to use glob patterns and grep efficiently; the Plan agent already structures output as actionable implementation steps. You extend and customize these defaults rather than building from zero.

Codex, by contrast, provides a single generic agent runtime. All specialization comes from your custom configuration — the system prompt, setup script, and tool permissions you define. This is more flexible in theory (you can build anything) but requires more upfront investment to match the out-of-the-box capabilities Claude Code provides.

For teams building specialized agents for unique workflows — compliance checking, domain-specific code generation, proprietary tool integration — Codex's blank-slate approach may be preferable. For teams that want productive multi-agent workflows immediately, Claude Code's built-in types provide a faster path.

## Pricing and Resource Considerations

Pricing models differ significantly and can be the deciding factor for many teams.

**Codex custom agents** are included in ChatGPT Pro ($200/month), Team ($30/user/month), and Enterprise plans. The compute cost of running sandboxed agents in the cloud is absorbed into the subscription. This bundled pricing is attractive for teams already on ChatGPT plans — adding Codex agents doesn't increase the bill. However, Pro users may face rate limits on concurrent tasks during peak periods, and the actual throughput depends on your plan tier.

**Claude Code subagents** use Anthropic's usage-based API billing. You pay per token — input and output — for every agent and subagent in your session. Spawning five parallel subagents means paying for five concurrent API streams. For light usage, this can be cheaper than a $200/month subscription. For heavy usage with many parallel agents on large codebases, costs can add up quickly. The advantage is granular control — you pay only for what you use, and there are no plan-tier restrictions on concurrency.

**Resource tradeoffs beyond price:**

- Codex uses cloud compute, so your local machine stays free. Useful when running agents on large repos that would strain a laptop.
- Claude Code uses your local CPU, memory, and disk. Worktree isolation creates copies of your repo, which means disk usage scales with the number of parallel agents.
- Codex latency includes container spin-up time (typically 30-90 seconds before work begins). Claude Code agents start immediately.

## When to Choose Codex Custom Agents

Codex custom agents fit best when you need scalable, repeatable, offline task processing:

- **Batch operations across many repos**: Submitting the same task (dependency updates, code style migration, security patches) to dozens of repositories simultaneously. Codex's cloud parallelism handles this without taxing your local machine.
- **Standardized team workflows**: When you want every engineer to use the same agent configuration for code review, test writing, or documentation — managed centrally through the Codex dashboard.
- **Low-interactivity tasks**: Work that doesn't require mid-task steering. "Add comprehensive tests for this module" is a good Codex task; "help me debug this flaky test by iterating on theories" is not.
- **Resource-constrained local environments**: If you're working on a laptop with limited compute, offloading heavy agent tasks to cloud sandboxes keeps your machine responsive.

Teams already invested in the OpenAI ecosystem (ChatGPT Team or Enterprise plans) get Codex agents at no additional cost, making adoption frictionless. The [Codex VS Code extension](/blog/codex-vscode) integrates task submission directly into the editor, reducing context switching.

## When to Choose Claude Code Subagents

Claude Code subagents are the better fit for interactive, context-sensitive development:

- **Iterative feature development**: When you're building something and need to research, prototype, test, and refine in a tight loop — spawning Explore agents to find relevant code, Plan agents to design the approach, and general-purpose agents to implement in parallel.
- **Live environment access**: When agents need to interact with running services, local databases, or environment-specific configuration that can't be replicated in a cloud sandbox.
- **Complex debugging sessions**: When you need to steer the investigation — "check the logs, now look at this function, now run this specific test with these flags." Claude Code's interactive model supports this; Codex's async model doesn't.
- **Custom agent behavior per task**: When you need different agent behavior for each task without creating persistent agent configurations. Claude Code's prompt-based customization lets you specify behavior at invocation time.
- **Projects using [Claude Code's extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp)**: If your repo already has CLAUDE.md files, skill definitions, hooks, and MCP servers configured, subagents inherit all of this context automatically — no duplicate setup needed.

The [agent teams pattern](/blog/claude-code-agent-teams) — spawning multiple specialized subagents that work in parallel and report back — is particularly effective for large refactoring tasks where you want human oversight at each decision point.

## Verdict

**If you need scalable batch processing with minimal supervision, choose Codex custom agents.** They excel at parallelizing standardized tasks across cloud infrastructure, and the bundled pricing on ChatGPT plans makes them cost-effective for teams already in the OpenAI ecosystem.

**If you need interactive, context-rich multi-agent sessions with real-time steering, choose Claude Code subagents.** The built-in specializations (Explore, Plan, pipeline-reviewer), file-based customization via skills and CLAUDE.md, and live local environment access create a more adaptive development experience.

Many teams will benefit from using both. Codex handles the batch work — dependency updates, boilerplate generation, cross-repo migrations — while Claude Code handles the nuanced work that requires human-in-the-loop guidance and deep project context. The choice isn't binary; it's about matching the right execution model to each type of task.

## Frequently Asked Questions

### Can Codex custom agents access my local development environment?

No. Codex agents run in isolated cloud sandboxes with a cloned copy of your repository. They cannot access local services, databases, or environment variables on your machine. If your task requires live environment interaction, Claude Code subagents — which run locally in your terminal — are the better fit.

### How many Claude Code subagents can I run in parallel?

There is no hard limit on concurrent subagents, but each one consumes local compute resources and API tokens. Practical parallelism depends on your machine's CPU and memory, and your Anthropic API rate limits. Using background agents and worktree isolation helps manage resource contention across multiple parallel tasks.

### Do I need to configure custom agents in Codex, or can I use it out of the box?

Codex works out of the box with a default agent configuration. Custom agents are optional — they let you specialize behavior with tailored system prompts, setup scripts, and tool permissions. For simple tasks like "write tests for this file," the default agent is sufficient. Custom agents become valuable when you need repeatable, domain-specific workflows.

### Can Claude Code subagents create pull requests automatically?

Yes. Claude Code subagents have full git access and can stage changes, create commits, push branches, and open pull requests using the GitHub CLI. When running with worktree isolation, the subagent works on a separate branch automatically, making PR creation a natural final step.

### Which option is more cost-effective for a small team?

For teams of 1-3 developers with moderate usage, Claude Code's pay-per-token model is typically cheaper than a ChatGPT Pro subscription. For teams of 5+ developers already on ChatGPT Team or Enterprise plans, Codex custom agents are effectively free to use since they're bundled with the subscription. Run the numbers based on your expected token volume and team size.
