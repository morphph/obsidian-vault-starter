# Claude Memory vs CLAUDE.md：两套记忆系统怎么选？

**TL;DR：** **Claude Memory**（Auto Memory）是 Claude Code 在对话中自动积累的个人笔记，记录你的偏好、反馈和项目背景；**CLAUDE.md** 是你手动编写、提交到代码仓库的指令文件，定义编码规范、构建命令和项目约束。两者不是竞争关系——**Memory 解决"记住我是谁"的问题，CLAUDE.md 解决"这个项目怎么做"的问题**。大多数团队应该同时使用两者。

## 概览：Claude Memory

**Claude Memory** 是 Claude Code 内置的自动持久化记忆系统，存储在 `~/.claude/projects/` 目录下。它在对话过程中自动观察和学习，将值得跨会话保留的信息写入独立的 Markdown 文件，并通过 `MEMORY.md` 索引管理。

Memory 记录四类信息：用户信息（角色、技能水平、工作偏好）、反馈记录（"不要在测试里 mock 数据库"这类纠正）、项目动态（正在进行的工作、截止日期、决策背景）和外部引用（Linear 项目地址、Grafana 看板链接）。这些信息自动积累，无需手动维护——你和 Claude Code 合作越多，它对你的理解越准确。

关于 Memory 系统的最新动态，包括免费用户开放和跨平台导入功能，可以参考我们的 [Claude 记忆系统升级详解](/zh/blog/anthropic-claude-memory-upgrades-importing)。

## 概览：CLAUDE.md

**CLAUDE.md** 是放在项目根目录（或子目录）的指令文件，开发者手动编写，随代码一起版本控制。它告诉 Claude Code 这个项目的规则：用什么构建命令、遵循什么编码风格、有哪些已知陷阱需要避开。

CLAUDE.md 的设计哲学是"规则手册"——它不关心你是谁，只关心这个仓库怎么工作。典型内容包括：构建和测试命令（`npm run build`、`npm test`）、代码风格要求（"commit message 用英文"、"不要在 pipeline 脚本里 import Next.js 模块"）、架构约束和已知 bug。整个团队共享同一份 CLAUDE.md，确保每个成员的 Claude Code 表现一致。

想深入了解 CLAUDE.md 在 Claude Code 整体架构中的位置，推荐阅读 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 差异关键 |
|------|--------------|-----------|---------|
| **创建方式** | 自动生成 | 手动编写 | Memory 零维护成本 |
| **存储位置** | `~/.claude/projects/` | 项目根目录 | Memory 不进仓库 |
| **版本控制** | 不受 Git 管理 | 随代码提交 | CLAUDE.md 可 Code Review |
| **作用范围** | 个人级（每个用户独立） | 团队级（全员共享） | 个人偏好 vs 团队规范 |
| **内容类型** | 偏好、反馈、上下文 | 规则、命令、约束 | 软记忆 vs 硬指令 |
| **生命周期** | 跨会话持续积累 | 随项目演进手动更新 | 自动 vs 显式 |
| **可审计性** | 仅本人可见 | 团队透明 | CLAUDE.md 胜出 |
| **灵活性** | 高（自然语言描述） | 中（结构化指令） | Memory 更适合模糊知识 |

## 核心差异详解：自动化 vs 显式控制

这是两套系统最根本的区别。

Claude Memory 是一个**观察-学习系统**。当你告诉 Claude Code "不要用 `any` 类型"，它自动将这条反馈记录为一条 feedback 类型的 memory，下次对话时自动加载。你不需要打开任何文件、不需要记住在哪里配置——它就是默默地学会了。Memory 还会记录你纠正过的做法和确认过的好做法，形成一个越来越精准的个人偏好画像。

CLAUDE.md 是一个**声明式规则文件**。它不会自动生成，不会自动更新。你写下 `npm run build — SSG build succeeds` 和 `Before ANY commit, ALL of these must pass`，这就是铁律。所有读到这份文件的 Claude Code 实例都会严格遵守，不存在"我记得上次你说过"这种模糊性。

实际影响：Memory 的优势在于**零配置的个性化**，CLAUDE.md 的优势在于**可审计的确定性**。当你的偏好出了错，Memory 可能在你不知情的情况下影响行为；而 CLAUDE.md 的每一条规则都能在 PR Review 中被审查。

## 协同使用：最佳实践

理解了两者定位后，正确的做法不是二选一，而是让它们各司其职。

**CLAUDE.md 负责"法律"——团队必须遵守的硬性规则：**
- 构建命令和质量门禁（build、test、lint 必须通过）
- 编码规范和风格要求（命名约定、注释规则、commit 格式）
- 项目架构约束和已知陷阱（"不要在 pipeline 脚本里 import Next.js 模块"）
- 文档更新要求（改了什么就更新对应的文档）

**Memory 负责"习惯"——个人工作方式的微调：**
- 你偏好的解释风格（"我是数据科学家，用数据管道的类比解释后端概念"）
- 你对特定做法的反馈（"上次把 PR 拆成多个的做法很好，继续保持"）
- 当前项目上下文（"正在做 auth 模块重构，法务要求的合规改造"）
- 外部系统的指引（"pipeline bug 在 Linear 的 INGEST 项目里追踪"）

这个分工模式在 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory) 中有更完整的讨论，包括 Memory 的四种类型和 CLAUDE.md 的层级加载机制。

## 什么时候该选 Claude Memory

以下场景中，Memory 比 CLAUDE.md 更合适：

- **个人偏好不适合强制给团队**：你喜欢函数式风格，但团队没有统一要求。写进 CLAUDE.md 会影响所有人，记在 Memory 里只影响你。
- **信息频繁变化**：当前 sprint 的目标、正在调查的 bug、临时的工作安排——这些写进版本控制的文件只会增加噪音。
- **跨项目通用的习惯**：你在所有项目中都希望 Claude Code 给出简洁回复、不加 emoji，这种偏好适合存在全局 Memory 中。
- **你不想花时间维护配置文件**：Memory 自动积累，不需要你刻意管理。

## 什么时候该选 CLAUDE.md

以下场景中，CLAUDE.md 是唯一正确的选择：

- **团队规范必须统一**：代码风格、测试要求、构建流程——这些不能因人而异。CLAUDE.md 提交到仓库，新成员加入时自动生效。
- **规则需要可审计**：安全约束、部署流程、数据处理规范——必须有明确的文件记录，可以在 Code Review 中检查。
- **项目有复杂的构建系统**：多步骤的构建命令、环境变量要求、特定的部署流程——这些必须精确记录，不能依赖"我记得上次怎么做的"。
- **跨仓库的组织规范**：如果你的公司有统一的 Claude Code 使用规范，放在 `~/.claude/CLAUDE.md` 中可以全局生效，且版本可控。

理解 CLAUDE.md 如何与 Skills、Hooks 等其他扩展机制协作，可以参考 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 结论

**CLAUDE.md 是团队的法规，Memory 是个人的笔记本。** 不要在 CLAUDE.md 里写个人偏好（"我喜欢用 Vim"），也不要把构建命令存在 Memory 里（"npm run build"应该是确定性的指令，不是模糊的记忆）。

对于刚开始使用 Claude Code 的开发者：先写好 CLAUDE.md，把项目的硬性规则固定下来；Memory 会在你工作过程中自然积累，不需要刻意配置。对于团队：确保 CLAUDE.md 经过 Code Review 并定期更新——它和代码一样重要。

## 常见问题

### Claude Memory 的内容会被发送给 Anthropic 吗？

Memory 文件存储在你本地的 `~/.claude/projects/` 目录下，作为对话上下文的一部分加载。它遵循 Claude Code 的标准数据处理策略，不会被单独上传或用于模型训练。

### CLAUDE.md 可以放在子目录里吗？

可以。Claude Code 支持层级加载：根目录的 CLAUDE.md 定义全局规则，子目录的 CLAUDE.md 可以添加或覆盖特定模块的规则。这对 monorepo 特别有用。

### Memory 记错了怎么办？

你可以直接告诉 Claude Code "忘掉关于 X 的记忆"，它会找到并删除对应的 memory 文件。也可以手动编辑 `~/.claude/projects/` 下的 Markdown 文件。Memory 系统在使用前会验证记忆是否仍然准确——如果代码现状和记忆冲突，以代码为准。

### 两者冲突时谁优先？

CLAUDE.md 的指令优先级高于 Memory。如果 CLAUDE.md 规定"所有函数必须写 JSDoc"，而 Memory 里记录了你说"不要写注释"，Claude Code 会遵守 CLAUDE.md 的规则。这是设计如此——团队规范不应被个人偏好覆盖。
