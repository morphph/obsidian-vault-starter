# Claude Memory vs CLAUDE.md：两套持久化机制，该怎么选？

**TL;DR：** **CLAUDE.md** 是你主动写给 Claude Code 的规则手册——项目约定、构建命令、代码风格，签入仓库，团队共享。**Claude Memory** 是 Claude Code 在对话中自动积累的笔记——你的角色、偏好、项目背景，存在本地，仅你可见。两者不是替代关系，而是互补：**CLAUDE.md 管「怎么做」，Memory 管「跟谁做、为什么做」**。

## 概述：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 [Claude Code](/glossary/claude-code) 内置的持久化记忆系统，存储在 `~/.claude/projects/<项目>/memory/` 目录下。它的核心设计目标是让 Claude Code 在跨会话场景中记住你——不需要你每次重新解释自己是谁、项目处于什么阶段、之前做过哪些决策。

Memory 按类型分为四类：**user**（用户画像）、**feedback**（行为反馈）、**project**（项目动态）、**reference**（外部资源指向）。Claude Code 会在对话中自动判断何时保存——比如你说「我是数据工程师，主要负责 pipeline」，它会存为 user 类型的记忆；你说「不要在这些测试里 mock 数据库」，它会存为 feedback 类型。

每条记忆是一个独立的 Markdown 文件，带有 frontmatter 元数据。`MEMORY.md` 是索引文件，每次对话开始时自动加载，Claude Code 根据相关性决定是否读取具体记忆文件。

关键特征：**自动写入、选择性读取、用户私有、本地存储**。更多细节参见我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件，放在仓库根目录，随代码一起签入 Git。它的角色类似于 `.editorconfig` 或 `CONTRIBUTING.md`——告诉 Claude Code 这个项目的规矩是什么。

典型的 CLAUDE.md 包含：构建和测试命令（`npm run build`、`npm test`）、代码风格要求（命名规范、注释策略）、架构约束（不要在 pipeline 脚本里 import Next.js 模块）、工作流规则（提交前必须通过所有质量门禁）。

CLAUDE.md 还支持层级结构：全局级 `~/.claude/CLAUDE.md` 适用于所有项目，项目级 `./CLAUDE.md` 适用于当前仓库，子目录级 `./src/CLAUDE.md` 适用于特定模块。Claude Code 在每次会话开始时自动加载所有层级的 CLAUDE.md，无条件执行。

关键特征：**手动编写、每次全量加载、团队共享、版本控制**。想深入理解 Claude Code 的多层架构，可以看 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 胜出 |
|------|--------------|-----------|------|
| **写入方式** | Claude Code 自动写入 | 开发者手动编写 | 各有优势 |
| **作用域** | 用户私有，本地存储 | 团队共享，签入 Git | 按场景 |
| **加载方式** | 索引常驻，内容按需读取 | 每次会话全量加载 | CLAUDE.md |
| **内容类型** | 用户画像、行为偏好、项目动态 | 规则、命令、约束、风格 | 按场景 |
| **可编辑性** | 可手动编辑，但通常由 AI 维护 | 完全由人类控制 | CLAUDE.md |
| **版本控制** | 不纳入 Git | 签入仓库，有变更历史 | CLAUDE.md |
| **跨项目通用** | 全局 Memory 跨项目生效 | 每个项目独立维护 | Memory |
| **团队协作** | 每人独立，不共享 | 全团队统一标准 | CLAUDE.md |

## 写入机制：自动积累 vs 显式声明

Memory 的核心优势在于**零维护成本**。你不需要专门花时间去写「我喜欢简洁的 commit message」——在三次对话中表现出这个偏好后，Claude Code 就会自动记住。对于那些难以提前预见的隐性知识，这种方式远比手动维护高效。

但自动写入也有代价：你无法精确控制 Claude Code 记住了什么。Memory 文件可能包含过时信息（三个月前的项目状态）、错误推断（把一次偶然选择当成固定偏好）、或冗余条目。Memory 系统虽然有更新和删除机制，但需要你主动审查。

CLAUDE.md 走的是相反路线：**每条规则都是你深思熟虑后写下的**。它不会自己膨胀，不会产生过时内容（除非你不维护），不会因为推断错误而做出意外行为。但代价是所有知识都需要人工编写和更新。

实践建议：把 CLAUDE.md 当作「宪法」——少而精的硬性规则。把 Memory 当作「工作笔记」——丰富但允许模糊的上下文信息。了解 Claude Code 扩展机制的完整图景，推荐阅读 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 作用域与协作：私有笔记 vs 团队契约

这是两者最根本的区别。

CLAUDE.md 签入 Git，意味着团队中每个人的 Claude Code 都遵循同一套规则。新成员 clone 仓库后，不需要任何额外配置，Claude Code 就知道用什么命令跑测试、提交前要过哪些门禁、代码风格有什么要求。这对团队一致性至关重要。

Memory 是纯私有的。你的 Claude Code 知道你是后端工程师、偏好 TypeScript、不喜欢冗长的注释——但你同事的 Claude Code 对此一无所知。这很合理，因为这些信息本来就因人而异。

一个常见陷阱：把本该写在 CLAUDE.md 里的团队规则存进了 Memory。比如「我们这个项目不用 mock 数据库做测试」——这是团队约定，应该写进 CLAUDE.md，而不是只存在某个人的 Memory 里。反过来，「我是这个项目的 tech lead，主要负责 API 设计」——这是个人上下文，应该由 Memory 管理。

如果你正在构建 Claude Code 的工作流，[Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)这篇文章值得一读——Skills 是 CLAUDE.md 的进阶版，解决的是更具体的任务场景。

## 什么时候用 Memory

选择 Memory 管理的信息通常具有这些特征：

- **因人而异**：你的技术背景、偏好的工作方式、关注的领域
- **渐进式积累**：不是一次性能写清楚的，需要在多次对话中逐步完善
- **可能过时**：项目动态、当前冲刺目标、截止日期等时效性信息
- **不适合签入代码仓库**：个人工作风格不应该成为团队规则

典型 Memory 内容：「用户是高级后端工程师，React 经验较少——前端解释需要更多上下文」、「当前在做 auth 模块重构，deadline 是 4 月 30 日」、「用户偏好一个大 PR 而不是拆成多个小 PR」。

## 什么时候用 CLAUDE.md

选择 CLAUDE.md 管理的信息通常具有这些特征：

- **团队通用**：所有人都应该遵守的规则和约定
- **稳定不变**：构建命令、质量门禁、代码风格——不会每周变一次
- **需要强制执行**：不是建议，是必须遵守的硬性要求
- **需要版本历史**：规则的变更应该有记录，可追溯

典型 CLAUDE.md 内容：「提交前必须通过 `npm run build` 和 `npm test`」、「中文内容必须用 CJK 字数统计」、「不允许跳过 pre-commit hook」。

## 最佳搭配策略

最有效的用法是把两者当作互补层：

1. **CLAUDE.md 定义规则边界**——什么必须做、什么绝对不能做。这些规则在每次会话中无条件加载，确保 Claude Code 不会越界。
2. **Memory 提供执行上下文**——在规则框架内，根据你的角色、偏好和当前状态来调整行为。
3. **Skills 处理具体任务**——CLAUDE.md 说「我们的代码要有测试」，[SKILL.md](/zh/blog/5-claude-code-skills-i-use-every-single-day) 定义「测试应该怎么写」。
4. **定期审查 Memory**——每月花 5 分钟看一眼 `~/.claude/projects/<项目>/memory/` 目录，删掉过时条目，修正错误推断。

不要过度依赖任何一方。一个只有 CLAUDE.md 没有 Memory 的 Claude Code，每次对话都像跟陌生人合作；一个只有 Memory 没有 CLAUDE.md 的 Claude Code，知道你是谁，但不知道项目的规矩。两者搭配才能让 Claude Code 既懂规则又懂人。

## 结论

**如果信息是团队应该共享的硬性规则，写进 CLAUDE.md。如果信息是个人特定的、会变化的上下文，交给 Memory。** 大多数团队的问题不是选错了工具，而是把所有东西都塞进一个地方。CLAUDE.md 和 Memory 的分工越清晰，Claude Code 的表现就越稳定。

## 常见问题

### Claude Memory 会自动同步到团队其他成员吗？

不会。Claude Memory 存储在本地 `~/.claude/projects/` 目录下，完全私有。团队成员之间不共享 Memory 内容。需要团队共享的规则应写入 CLAUDE.md 并签入 Git。

### CLAUDE.md 写太长会影响性能吗？

会。CLAUDE.md 在每次会话开始时全量加载到上下文窗口中，过长的文件会占用宝贵的 token 空间。建议控制在 200-500 行以内，把详细的任务指令拆分到 SKILL.md 文件中按需加载。

### 能不能让 Claude Code 把 Memory 里的内容自动整理到 CLAUDE.md？

不建议这样做。Memory 包含个人偏好和时效性信息，直接迁移到 CLAUDE.md 会污染团队共享的规则文件。正确做法是手动审查 Memory 中有团队价值的内容，精简后写入 CLAUDE.md。

### 删掉所有 Memory 文件会怎样？

Claude Code 会正常工作，但每次对话都从零开始——不记得你的角色、偏好或之前的项目背景。相当于跟一个完全陌生的高级工程师合作：能力没变，但默契消失了。

### CLAUDE.md 支持条件逻辑吗？

CLAUDE.md 是纯 Markdown 文本，没有条件分支语法。但你可以通过目录层级实现类似效果：根目录的 CLAUDE.md 放通用规则，`src/backend/CLAUDE.md` 放后端特定规则，Claude Code 会根据工作目录自动加载对应层级。
