**Claude Code custom agents** are defined as markdown files in `.claude/agents/`. Each file specifies the agent's name, description, system instructions, available tools, and optional model override. Because these files live in your repo, they're version-controlled, code-reviewed, and shared across the team. A typical setup might include:

- A `review-agent` that runs security and code-quality checks with read-only permissions
- An `explore-agent` optimized for codebase search with the `Explore` sub-agent type
- A `docs-agent` that generates documentation following your team's style guide
- A `test-agent` that writes and runs tests with full shell access

This is an extension of Claude Code's broader programmability philosophy — the same pattern used by [SKILL.md files](/blog/5-claude-code-skills-i-use-every-single-day) and [hooks](/blog/claude-code-hooks-a-complete-guide-to-automating-your-ai-coding-workflow). You encode team standards into declarative configuration files that the AI follows consistently.

**Codex custom agents** work through environment configuration rather than agent definition files. You customize a Codex task by specifying the sandbox setup (which packages to install, which files to include), the system prompt (what instructions the agent follows), and the model. The Codex dashboard lets you save task templates for repeated use. The [Codex for open source](/blog/codex-for-open-source) initiative shows this pattern at scale — maintainers configure Codex environments tuned to their project's build system and contribution guidelines.

The key difference: Claude Code's custom agents are composable building blocks within a larger orchestration. Codex's custom configurations are independent task templates. If you need Agent A to call Agent B, Claude Code supports that natively. In Codex, you'd dispatch two separate tasks and stitch the results together manually or via external scripting.

## Parallel Execution and Orchestration: Detailed Analysis

Both platforms support running multiple agents or tasks in parallel, but the semantics differ significantly.

**Claude Code parallel agents** are spawned within a single conversation turn. You can launch multiple `Agent` tool calls in one message, and they execute concurrently — each with its own context and toolset. When all agents complete, their results are available to the parent agent in the same conversation. This enables patterns like:

- Fan-out: spawn five agents to search different parts of the codebase, then synthesize their findings
- Speculative execution: run two alternative approaches in parallel, pick the better result
- Pipeline stages: Agent 1 researches, Agent 2 plans based on Agent 1's output, Agent 3 implements based on Agent 2's plan

The `run_in_background` parameter lets you fire-and-forget sub-agents while continuing other work — you're notified when they complete. Combined with git worktree isolation, each background agent can work on its own branch without file conflicts.

**Codex parallel tasks** are dispatched independently from the dashboard or API. Each task runs in its own sandbox, consuming cloud resources rather than your local machine. This makes Codex excellent for batch operations — "fix all 47 linting issues, one task per file" — where tasks are independent and results can be merged later. Codex surfaces results as pull requests, so the merge step happens through your normal code review process.

Where Codex struggles is conditional orchestration. If you need "research the API, then based on what you find, generate the client code, then based on the generated code, write tests," that's three sequential tasks with data dependencies. In Claude Code, a single parent agent handles this flow naturally, spawning sub-agents at each step and passing context forward. In Codex, you'd need external scripting or manual intervention between tasks.

For teams already using [effective harnesses for long-running agents](/blog/effective-harnesses-for-long-running-agents), Claude Code's sub-agent system integrates directly into those harness patterns. Codex requires a separate orchestration layer — though its API makes building one straightforward for teams with infrastructure engineering capacity.

## Developer Experience and Workflow Integration

The day-to-day experience of working with agents differs substantially between the two platforms.

**Claude Code** is terminal-native. You interact with agents through text commands, approve or deny tool calls, and see results streamed back in real time. The feedback loop is tight — you can redirect a sub-agent mid-task, ask it to explain its approach, or kill it and try a different strategy. Custom agents are just markdown files, so creating a new agent type takes seconds. The [Claude Code skills system](/blog/9-principles-writing-claude-code-skills) and [hooks](/blog/claude-code-hooks-mastery) mean you can build sophisticated automation that triggers custom agents based on specific conditions.

**Codex** is dashboard-and-API-native. You dispatch tasks from a web UI or programmatically, then check back for results. The asynchronous model suits workflows where you want to batch tasks overnight or during off-hours — fire 20 tasks before lunch, review the PRs after. The [Codex VS Code extension](/blog/codex-vscode) brings this into the IDE, but the interaction model is still dispatch-and-review rather than interactive orchestration.

For [students](/blog/codex-for-students) and developers new to agent-based workflows, Codex's simpler model — "describe a task, get a PR" — has a gentler learning curve. Claude Code's sub-agent system is more powerful but requires understanding agent types, prompt engineering for sub-agent delegation, and the nuances of parallel execution with shared state.

## Cost and Resource Considerations

Multi-agent workflows multiply resource consumption, so cost modeling matters.

**Claude Code** charges per API token across all agents. A parent agent spawning three sub-agents means four concurrent Claude model instances, each consuming input and output tokens. Extended thinking in sub-agents adds to the bill. However, you can use cheaper models (Haiku) for simple sub-agents like search or validation, reserving Opus for complex reasoning tasks. The model-per-agent flexibility is a significant cost optimization lever.

**Codex** includes a task allowance in ChatGPT Pro subscriptions, with API-based billing for higher volumes. Each sandbox has compute costs (container startup, execution time, package installation), but these are absorbed by OpenAI's infrastructure rather than your local machine. For teams dispatching dozens of tasks daily, the economics depend on subscription tier and task complexity.

Neither platform is categorically cheaper. Claude Code is more cost-efficient for interactive workflows where sub-agents share context (less redundant token processing). Codex is more cost-efficient for independent batch tasks where container isolation prevents cascading failures that waste tokens.

## When to Choose Claude Code Subagents

Claude Code's sub-agent system is the stronger choice when your workflow requires coordination between agents:

- **Complex refactoring**: You need one agent to analyze the dependency graph, another to plan the migration, and a third to execute changes — with each step informed by the previous one
- **Code review pipelines**: A security review agent, a performance review agent, and a style review agent all examine the same PR, and a parent agent synthesizes their findings into a unified report
- **Team-specific workflows**: Your organization has specific agent configurations (code style enforcers, documentation generators, test strategists) that should be version-controlled and shared across engineers via [custom agent definitions](/blog/claude-code-subagents-examples)
- **Interactive development**: You want to steer agents in real time — redirecting, asking follow-up questions, or composing results on the fly
- **Codebase-wide operations**: Tasks that require understanding relationships across files benefit from sub-agents that can share discoveries through the parent context

## When to Choose Codex

Codex's task-dispatch model is the stronger choice when isolation and simplicity matter more than orchestration:

- **Batch operations**: Fix 50 similar issues across a monorepo by dispatching one task per issue — each runs in isolation, results arrive as PRs
- **Untrusted or experimental changes**: Every task runs in a sandbox, so a bad task can't corrupt your working tree or leak environment variables
- **Team-wide task distribution**: Non-technical stakeholders can dispatch Codex tasks from the dashboard without terminal access or agent configuration knowledge
- **Resource-constrained local environments**: All computation happens in the cloud, so a laptop with limited RAM can orchestrate complex work
- **CI/CD integration**: Codex tasks can be triggered from pipelines, creating PRs that go through your normal review process — no interactive session required

## Verdict

**For multi-agent orchestration, Claude Code is the clear winner.** Its sub-agent system — with typed agents, custom definitions, parallel execution, context sharing, and model-per-agent flexibility — is purpose-built for workflows where agents need to coordinate. If your work involves complex refactoring, multi-perspective code review, or any task where one agent's output feeds another agent's input, Claude Code's architecture handles this natively.

**For independent, parallelizable task dispatch, Codex is simpler and more isolated.** Its cloud sandbox model eliminates local resource concerns and provides stronger isolation guarantees. If your workflow is "send 20 independent coding tasks, review the PRs tomorrow," Codex's model is a better fit.

Many teams will use both. Claude Code for interactive, orchestrated development sessions where sub-agents collaborate on complex problems. Codex for batch operations and fire-and-forget tasks where isolation matters more than coordination. The [Claude Code `codex-rescue` agent type](/blog/claude-code-subagents-examples) even bridges the two — spawning a Codex task from within a Claude Code session when you want cloud isolation for a specific sub-task.

The broader trend is clear: single-agent coding tools are giving way to [multi-agent workflows](/blog/con-u-pour-des-workflows-multi-agents) where specialized agents handle different aspects of development. Whether you start with Claude Code's orchestration or Codex's dispatch model, building fluency with multi-agent patterns is where AI-assisted development is heading.

## Frequently Asked Questions

### Can you use subagents and custom agents in Codex the same way as Claude Code?
Not directly. Codex uses a task-dispatch model where each coding task runs in an isolated cloud sandbox. You can run multiple tasks in parallel and customize each task's environment and instructions, but Codex does not support in-session sub-agent spawning or agent-to-agent communication the way Claude Code does. Orchestration between Codex tasks requires external scripting or manual coordination.

### How do you define a custom agent in Claude Code?
Custom agents are markdown files placed in the `.claude/agents/` directory of your repository. Each file specifies the agent's name, description, system instructions, tool permissions, and optional model override. Because they live in git, custom agents are version-controlled and shared across your team automatically.

### Is Codex or Claude Code better for large-scale batch refactoring?
It depends on task independence. If each refactoring change is independent — fix the same lint issue across 50 files — Codex's parallel sandbox model handles this well with strong isolation. If the refactoring involves coordinated changes where one file's modification affects another, Claude Code's sub-agents with shared context are more effective because they can pass information between steps.

### Can Claude Code sub-agents use different AI models?
Yes. Each sub-agent can specify a model override — Opus for complex reasoning, Sonnet for balanced performance, or Haiku for fast, simple tasks like codebase search. This lets you optimize cost and speed per agent role within a single workflow.

### Do Codex tasks share context with each other?
No. Each Codex task runs in its own isolated container with its own filesystem snapshot. Tasks cannot read each other's state or communicate during execution. Results are returned independently, typically as separate pull requests or code patches.
