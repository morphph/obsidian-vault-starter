**If removing it would break a teammate's workflow → CLAUDE.md.**
**If removing it would only affect your own experience → Memory.**

For example, "the auth middleware rewrite is driven by legal compliance, not tech debt" is project context that affects design decisions anyone might make. It could go in either place. But since it affects architectural choices across the team, CLAUDE.md is safer — it ensures everyone's Claude Code sessions respect the constraint.

Conversely, "user prefers one bundled PR over many small ones for refactors" is a validated workflow preference. Your teammate might prefer the opposite. That's Memory.

## Team Collaboration: Shared vs Personal Context

For solo developers, the distinction between Memory and CLAUDE.md matters less — you're the only consumer of both. For teams, the distinction is critical.

**CLAUDE.md creates team alignment.** When your team writes "commit message must accurately describe what changed" in CLAUDE.md, every developer's Claude Code session follows that rule. New team members get the same instructions on day one. The rules evolve through pull requests, with the same review process as any other code change. This is powerful for engineering standards, but it requires consensus — you can't experiment with personal workflow optimizations in a shared file.

**Memory enables personal optimization.** Each developer's Memory reflects their individual working style, role, and accumulated corrections. A frontend specialist's Memory might note "frame backend explanations in terms of React analogues." A backend engineer's Memory might note "skip UI testing steps, focus on API contracts." These personalizations make Claude Code more effective for each individual without requiring team-wide agreement.

The interaction between the two systems is additive. CLAUDE.md sets the floor — minimum standards everyone must follow. Memory raises the ceiling — personal context that makes interactions more efficient. A well-configured team has a tight, focused CLAUDE.md (under 200 lines) and lets each developer's Memory handle the rest.

## Update Lifecycle: Manual Curation vs Automatic Learning

**CLAUDE.md requires deliberate maintenance.** Someone writes it, someone reviews it, someone merges it. When the build system changes, someone must update the commands. When a new architectural constraint emerges, someone must add it. This manual process is both a strength (nothing changes without review) and a weakness (stale instructions persist until someone notices).

Well-maintained CLAUDE.md files include documentation update rules — "when modifying pipeline scripts, update `docs/specs/PIPELINE.md`" — to prevent drift. But enforcement depends on developer discipline and code review thoroughness.

**Claude Memory updates itself.** When you correct Claude Code's behavior, it saves a `feedback` memory without you explicitly asking. When it learns about your role or a project deadline, it saves the relevant type. You can also explicitly ask Claude Code to remember something, and it saves immediately.

This automatic learning means Memory improves over the first few weeks of use with minimal effort. But it also means Memory can accumulate stale entries — a project deadline that passed, a preference you've changed, a reference to a renamed file. Claude Code is designed to verify memories against current state before acting on them, but periodic manual review of `MEMORY.md` keeps the system clean.

The healthiest pattern: treat CLAUDE.md like infrastructure (review quarterly, update on major changes) and treat Memory like a living document (trust the automation, prune occasionally).

## Scope and Hierarchy: How Instructions Layer

Both systems support hierarchical scoping, but they implement it differently.

**CLAUDE.md hierarchy is directory-based.** A global `~/.claude/CLAUDE.md` applies to all projects. A project-root `CLAUDE.md` applies to everything in the repo. A `src/frontend/CLAUDE.md` applies only when working in that subdirectory. Instructions cascade — subdirectory files add to (and can override) parent instructions. This maps naturally to monorepo structures where different modules have different conventions.

**Memory hierarchy is type-based.** `user` memories apply globally (your role and preferences). `feedback` memories apply to behavioral patterns (how Claude Code should respond). `project` memories apply to the current project's state. `reference` memories point to external systems. The retrieval system uses descriptions to judge relevance, not directory paths.

In practice, this means CLAUDE.md handles **spatial** scoping ("these rules apply in this part of the codebase") while Memory handles **temporal** and **personal** scoping ("this fact is true right now" or "this preference is mine").

For teams building sophisticated [Claude Code skill systems](/blog/5-claude-code-skills-i-use-every-single-day), the hierarchy extends further — `SKILL.md` files provide task-specific instructions that layer on top of CLAUDE.md and Memory. The three systems together form a complete context stack: project rules (CLAUDE.md) + personal context (Memory) + task instructions (SKILL.md).

## Common Mistakes and How to Avoid Them

### Mistake 1: Using CLAUDE.md as a Knowledge Base

CLAUDE.md loads in full every session. Stuffing it with architectural documentation, API references, or historical decisions bloats token usage without improving instruction quality. Keep CLAUDE.md under 200 lines of actionable instructions. Move reference material to `docs/` files that Claude Code can read on demand, or save project context as Memory entries with descriptive tags for selective retrieval.

### Mistake 2: Duplicating CLAUDE.md Instructions in Memory

If CLAUDE.md says "run `npm test` before committing," saving a Memory that says the same thing creates confusion when one gets updated and the other doesn't. Memory should never duplicate what CLAUDE.md already states. The Memory system is explicitly designed to exclude "code patterns, conventions, architecture, file paths, or project structure" — anything derivable from the current project state.

### Mistake 3: Storing Ephemeral Task State in Memory

Memory persists across conversations, so saving "currently debugging the auth timeout issue" creates a stale entry that confuses future sessions. Use Claude Code's in-conversation task tracking for current-session work. Only save to Memory when the information will be useful in a future conversation — and convert relative dates to absolute dates ("Thursday" → "2026-03-05") so the entry remains interpretable later.

### Mistake 4: Never Reviewing Memory

Auto-saved memories can reference files that have been renamed, preferences you've changed, or project states that no longer apply. Claude Code verifies memories before acting on them, but stale entries still waste retrieval bandwidth. Review your `MEMORY.md` index every few weeks and delete entries that no longer apply.

### Mistake 5: Skipping CLAUDE.md on Solo Projects

Even if you're the only developer, CLAUDE.md provides value that Memory cannot: deterministic, always-loaded instructions. Your Memory might not surface a quality gate at the right moment, but CLAUDE.md always will. Solo projects benefit from a minimal CLAUDE.md (build commands, test commands, key gotchas) even when Memory handles everything else.

## When to Choose Claude Memory

Use Claude Memory when:

- **You want Claude Code to learn your working style** over multiple sessions without repeating instructions
- **The information is personal** — your role, experience level, communication preferences, or individual workflow habits
- **The context is temporal** — project deadlines, in-flight initiatives, or recent decisions that will become irrelevant in weeks
- **You need to store external references** — links to Linear projects, Slack channels, Grafana dashboards, or documentation portals
- **You've corrected Claude Code's behavior** and want the correction to persist — feedback memories are the most impactful type for improving session quality over time

Memory shines for developers who use Claude Code daily and want it to feel like a colleague who remembers your previous conversations, not a stranger you brief from scratch each time.

## When to Choose CLAUDE.md

Use CLAUDE.md when:

- **The instruction must apply to every team member** — coding standards, build procedures, quality gates, and architectural constraints
- **Reliability matters more than flexibility** — CLAUDE.md always loads, Memory might not surface at the right moment
- **The instruction should go through code review** — putting workflow rules in a reviewable file means the team agrees on them
- **You need directory-scoped rules** — different standards for `src/frontend/` vs `src/backend/` vs `scripts/`
- **The information is about the codebase, not a person** — function signatures, known gotchas, module boundaries, and dependency constraints

CLAUDE.md is essential for any project where more than one person (or more than one Claude Code session) touches the code. It's the source of truth for "how we work here."

## Verdict

**Use both.** CLAUDE.md and Claude Memory are not competing systems — they're complementary layers in Claude Code's [persistence architecture](/blog/claude-code-memory). CLAUDE.md provides the deterministic foundation: shared rules that every session enforces identically. Memory provides the adaptive layer: personal context that makes each developer's experience better over time.

**If you're forced to pick one:** start with CLAUDE.md. A project without CLAUDE.md means Claude Code has no persistent project context at all — every session starts cold. A project without Memory still works fine; you just repeat yourself more often. CLAUDE.md is the foundation; Memory is the optimization.

**The decision framework is simple:** if your teammate needs this information, put it in CLAUDE.md. If only you need it, let Memory handle it. If you're unsure, CLAUDE.md is the safer default — you can always move personal preferences to Memory later, but team-critical instructions buried in one person's Memory create a bus-factor risk.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions are treated as project-level rules that always apply. If a Memory entry conflicts with CLAUDE.md, Claude Code follows CLAUDE.md. Memory adds context and personalization on top of project rules — it doesn't override them. If you find a conflict, update or remove the stale Memory entry.

### Does Claude Memory sync across devices?

Claude Memory is stored locally in `~/.claude/projects/` on your machine. It does not sync across devices automatically. If you work from multiple machines, each will build its own Memory over time. CLAUDE.md, being a git-tracked file, syncs everywhere your repo does — another reason critical instructions belong there.

### How large should CLAUDE.md be?

Keep it under 200 lines of actionable instructions. Every line loads into every conversation, consuming tokens. Focus on build commands, quality gates, style rules, and known gotchas. Move lengthy documentation to `docs/` files and reference material to Memory. A concise CLAUDE.md that Claude Code actually follows is better than a comprehensive one it partially ignores due to context limits.

### Can I use CLAUDE.md without Claude Memory enabled?

Yes. CLAUDE.md works independently — it's just a file Claude Code reads at startup. Memory is a separate, optional system. Many teams start with CLAUDE.md only and adopt Memory as individual developers discover they're repeating the same corrections across sessions.

### How do I migrate content between Memory and CLAUDE.md?

If you've saved something in Memory that the whole team needs, copy it into CLAUDE.md and submit a PR. If CLAUDE.md contains personal preferences that only apply to you, move them to a Memory file and remove them from CLAUDE.md. There's no automated migration tool — the systems use different formats and scoping rules, so manual review during migration ensures content ends up in the right place.
