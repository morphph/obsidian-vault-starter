# Codex 子代理 vs Claude Code 子代理：多 Agent 编程工作流对比

**TL;DR：** 两个平台都支持多 Agent 并行工作，但路径截然不同。**Codex** 的子代理运行在云端沙箱里，适合异步、批量的代码任务分发；**Claude Code** 的子代理运行在本地终端，适合实时、交互式的多文件协作。如果你的团队需要把任务「丢出去跑」，选 Codex；如果你需要 Agent 在你眼皮底下协同操作同一个代码库，选 Claude Code。

## 概述：Codex 子代理与自定义 Agent

OpenAI Codex 是一个云端编程智能体平台，核心特点是每个任务在独立的沙箱环境中运行。当你在 Codex 中创建一个任务，它会启动一个隔离的云端容器，拉取你的代码仓库，然后让 AI 在里面自主工作——读代码、改代码、跑测试、生成 PR。

Codex 的「子代理」概念体现在任务级别的并行化：你可以同时发起多个独立任务，每个任务各自在自己的沙箱里运行，互不干扰。自定义 Agent 则允许你通过配置文件定义 Agent 的行为约束和工具权限，让不同类型的任务遵循不同的执行策略。

这种架构的优势在于隔离性和可扩展性——100 个任务可以同时跑，不会互相踩脚。更多关于 Codex 平台能力的分析，参见我们的 [Codex 完全指南](/zh/blog/codex-complete-guide)。

## 概述：Claude Code 子代理

Claude Code 是 Anthropic 推出的终端 AI 编程工具，它的子代理系统（Agent Teams）允许主 Agent 在执行过程中动态生成多个子代理，每个子代理负责特定的子任务。这些子代理可以并行运行在独立的 git worktree 中，共享同一个代码库但互不阻塞。

与 Codex 不同，Claude Code 的子代理运行在你的本地机器上，主 Agent 充当协调者——它决定何时分拆任务、如何分配给子代理、怎样合并结果。这种模式更接近一个技术主管带着几个工程师协同工作。

Claude Code 还支持通过 `subagent_type` 指定专门的子代理类型（如代码搜索、代码审查、计划制定等），每种类型有不同的工具权限和能力范围。详细的实战案例可以参考 [Claude Code 子代理使用示例](/zh/blog/claude-code-subagents-examples)。

## 功能对比

| 功能维度 | Codex 子代理 | Claude Code 子代理 | 优势方 |
|---------|-------------|-------------------|--------|
| **运行环境** | 云端沙箱容器 | 本地终端 / git worktree | 各有所长 |
| **并行能力** | 多任务同时运行，完全隔离 | 多子代理并行，共享代码库 | Codex |
| **上下文共享** | 任务间不共享上下文 | 主代理可向子代理传递完整上下文 | Claude Code |
| **执行模式** | 异步：提交任务后等结果 | 同步/异步混合：可实时观察 | Claude Code |
| **自定义程度** | 通过配置文件定义 Agent 行为 | 通过 SKILL.md + subagent_type 定义 | 平手 |
| **代码库访问** | 拉取仓库快照 | 直接操作本地文件系统 | Claude Code |
| **结果交付** | 自动生成 PR | 直接修改文件 / 提交 / 推送 | 各有所长 |
| **IDE 集成** | VS Code 扩展 + Web 界面 | 终端原生 + IDE 扩展 | 平手 |

## 架构模式：云端隔离 vs 本地协同

这是两个平台最根本的分歧。

Codex 选择了「云端隔离」路线：每个子任务在独立的容器中运行，拥有自己的文件系统副本。这意味着一个子代理改了文件 A，另一个子代理完全不受影响——它们甚至不知道彼此的存在。最终结果通过 PR 合并回主分支。这种模式天然适合那些可以清晰拆分、互不依赖的任务，比如「给这 10 个模块分别写单元测试」。

Claude Code 选择了「本地协同」路线：子代理通过 git worktree 在同一仓库的不同分支上工作，主代理负责协调。关键区别在于主代理保持全局视野——它知道每个子代理在做什么，可以根据一个子代理的结果调整另一个子代理的策略。这种模式更适合需要上下文关联的复杂任务，比如「重构认证模块并同步更新所有依赖它的测试和文档」。

对于想深入了解 Claude Code 多 Agent 协作机制的读者，推荐阅读 [Claude Code Agent Teams 详解](/zh/blog/claude-code-agent-teams)。

## 自定义 Agent 能力：配置 vs 编排

在 Codex 中，自定义 Agent 主要通过项目根目录的配置文件实现。你可以定义 Agent 的系统提示词、可用工具列表、执行超时时间等参数。Codex 还支持在 [VS Code 扩展](/zh/blog/codex-vscode)中使用这些自定义 Agent，让你在 IDE 内直接发起任务。

Claude Code 的自定义能力更偏向「编排」而非「配置」。通过 [Skills 系统](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)，你可以定义可复用的指令文件（SKILL.md），规定 Agent 在特定任务类型下的行为规范。子代理的类型系统（Explore、Plan、code-reviewer 等）则提供了开箱即用的专业化能力。主代理可以在运行时动态决定使用哪种子代理类型，甚至让子代理运行在隔离的 worktree 中以防止副作用。

两种方式各有取舍：Codex 的配置更声明式、更容易标准化；Claude Code 的编排更灵活、更适合应对复杂多变的工程场景。

## 适合选择 Codex 子代理的场景

如果你的工作模式是「批量分发、异步收割」，Codex 是更好的选择：

- **大规模批量任务**：需要同时处理几十个独立的代码变更，比如跨仓库的依赖升级、批量 bug 修复
- **CI/CD 集成**：希望 Agent 工作流融入现有的 GitHub PR 流程，每个任务自动生成可审查的 PR
- **团队协作**：多人团队需要一个共享的任务队列，任何人都可以提交任务、查看进度、审查结果
- **安全敏感场景**：代码不能在开发者本地机器上被 AI 随意修改，需要云端沙箱隔离

Codex 开放了对开源项目的支持（参见 [Codex for Open Source](/zh/blog/codex-for-open-source)），如果你维护开源项目并需要批量处理 issue，这是一个值得关注的方向。

## 适合选择 Claude Code 子代理的场景

如果你的工作模式是「实时协作、迭代推进」，Claude Code 更合适：

- **复杂重构**：任务之间有强依赖关系，需要主代理根据子代理的中间结果动态调整策略
- **探索性开发**：不确定最终方案，需要 Agent 边做边商量，你在旁边实时提供反馈
- **全栈操作**：需要 Agent 同时操作代码、运行测试、检查构建结果、修改配置，而不仅仅是改代码
- **本地工具链依赖**：项目依赖本地环境的特定工具、数据库或服务，无法在云端沙箱完整复现

Claude Code 的子代理可以使用 [MCP Server](/zh/glossary/agent-sdk) 连接外部工具和数据源，这意味着子代理的能力边界不限于代码编辑，还能扩展到数据库查询、API 调用等操作。

## 结论

**选 Codex 子代理**：你需要的是一个任务分发平台——提交清晰定义的任务，让 AI 在云端独立完成，通过 PR 交付结果。优势在于规模化、隔离性和团队协作。

**选 Claude Code 子代理**：你需要的是一个智能协作伙伴——在你的终端里实时工作，子代理之间上下文互通，主代理全局调度。优势在于灵活性、上下文关联和本地工具链集成。

实际上，两者并不互斥。越来越多的工程团队在日常开发中使用 Claude Code 的实时子代理做探索和原型，然后用 Codex 的异步子代理做批量的测试补全、文档生成和代码迁移。关键不在于选哪一个，而在于理解每种模式的边界——把正确的任务交给正确的工具。

## 常见问题

### Codex 子代理和 Claude Code 子代理可以一起用吗？

可以。两者面向不同的工作场景，没有冲突。你可以在 Claude Code 中完成交互式开发和重构，然后用 Codex 批量处理需要独立运行的后续任务（如补测试、升依赖）。团队中不同角色也可以根据偏好选择不同工具。

### Codex 的自定义 Agent 支持什么程度的定制？

Codex 允许通过项目配置文件定义 Agent 的系统提示词、工具权限、超时设置等参数。你可以为不同类型的任务创建不同的 Agent 配置。但目前 Codex 的自定义更偏向声明式配置，不像 Claude Code 那样支持运行时动态编排子代理。

### 哪个平台的子代理更适合处理大型代码库？

两者各有优势。Codex 的云端沙箱可以分配更多计算资源，适合需要大量并行处理的场景。Claude Code 的子代理直接操作本地文件系统，避免了代码同步延迟，且主代理的全局上下文理解在大型代码库的跨模块重构中更有优势。选择取决于你的具体需求——并行度优先选 Codex，上下文关联优先选 Claude Code。
