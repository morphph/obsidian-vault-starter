**Claude Memory is Claude-written, Claude-read.** During conversations, Claude detects information worth persisting — your expertise level, a correction you gave, a project deadline you mentioned — and writes it to disk. On the next conversation, Claude reads those memories back to inform its behavior. This is a feedback loop: Claude both produces and consumes the content. The benefit is zero-effort persistence. The tradeoff is less direct control — you're trusting Claude's judgment about what's worth remembering.

This distinction has practical consequences. CLAUDE.md is the right place for anything you'd want to review and approve before it influences Claude's behavior. Memory is the right place for context that would be tedious to maintain manually — like the fact that you're a senior backend engineer who prefers terse responses without trailing summaries.

The two systems also differ in **load behavior**. CLAUDE.md is always loaded in full at the start of every session. Claude Memory's index (`MEMORY.md`) is loaded, but individual memory files are only read when Claude determines they're relevant. This means CLAUDE.md has a guaranteed delivery model — every instruction is seen every time — while memory has a relevance-filtered delivery model. For critical rules (build commands, quality gates, forbidden patterns), CLAUDE.md's guaranteed delivery is what you want.

## Scope and Sharing: Detailed Analysis

The second critical difference is scope. CLAUDE.md is project-scoped and team-shared. Claude Memory is user-scoped and personal. This isn't a minor implementation detail — it determines the entire category of information each system should hold.

**Project-scoped, team-shared (CLAUDE.md):**
- Build and test commands that every developer needs
- Coding standards and lint rules
- Architectural decisions and constraints
- Known gotchas that apply regardless of who's coding
- Quality gates that must pass before any commit

These belong in CLAUDE.md because they're objective project facts. It doesn't matter whether you're a new intern or the tech lead — `npm test` must pass before committing, and the API routes live in `src/app/api/`. Every team member needs the same instructions.

**User-scoped, personal (Claude Memory):**
- Your role and expertise level (senior backend engineer vs. junior frontend developer)
- Communication preferences (terse vs. detailed, with or without summaries)
- Feedback corrections ("don't mock the database in integration tests — we got burned last quarter")
- Current project status ("merge freeze starts April 25 for mobile release")
- Pointers to external systems you use ("pipeline bugs are tracked in Linear project INGEST")

These belong in memory because they're subjective or transient. Your preference for terse responses shouldn't be imposed on your teammate who's new to the codebase and wants detailed explanations. A merge freeze that starts next week is useful context now but will be stale in a month — it shouldn't live in a version-controlled file that persists forever.

There's a subtle overlap case: **personal global CLAUDE.md** (`~/.claude/CLAUDE.md`). This file is personal (not committed to any repo) and applies across all projects. It occupies a middle ground — it's human-written like CLAUDE.md but personal like memory. Use it for cross-project personal rules that you want guaranteed delivery on: "always use conventional commits," "never push to main without asking," or workflow preferences that are important enough to be explicit instructions rather than learned context.

## Information Lifecycle: Detailed Analysis

Understanding how information flows through each system helps you decide where to put new context.

**CLAUDE.md lifecycle:**
1. A developer identifies a rule, command, or constraint
2. They write it into CLAUDE.md (or submit a PR if the team reviews changes)
3. It takes effect immediately for all subsequent Claude Code sessions
4. It persists indefinitely until someone explicitly edits or removes it
5. History is tracked in git — you can see when rules were added, changed, or removed

This lifecycle favors **stable, authoritative information**. Build commands don't change weekly. Coding standards evolve slowly. Architectural decisions are made once and hold for months or years. CLAUDE.md is the right home for information with a long shelf life.

**Claude Memory lifecycle:**
1. During a conversation, you say something that reveals useful context (or explicitly ask Claude to remember something)
2. Claude writes a memory file with structured frontmatter and content
3. Claude adds an index entry to MEMORY.md
4. In future conversations, Claude loads the memory index and reads relevant memory files
5. Memories can become stale — Claude is instructed to verify memories against current state before acting on them
6. You can ask Claude to forget something, and it will remove the memory file and index entry

This lifecycle favors **evolving, contextual information**. Your current project focus changes. Deadlines pass. Feedback accumulates. The memory system handles this naturally — Claude updates or removes memories as context changes, without requiring PRs or code review.

The key risk with memory is **staleness**. A memory that says "we're using React 17" is actively harmful once you've upgraded to React 19. Claude is designed to verify memories against current state, but this isn't foolproof. For critical facts that must be current, CLAUDE.md is safer because humans update it deliberately. For context that's useful even if slightly stale — like your role or communication preferences — memory is fine.

## Interaction Between the Two Systems

Claude Memory and CLAUDE.md aren't isolated — they interact at runtime in ways that matter for your workflow.

**CLAUDE.md takes precedence.** If a memory says "this user prefers tabs" but CLAUDE.md says "use 2-space indentation," CLAUDE.md wins. This is by design — project rules override personal preferences. The team's coding standard is more important than your individual habit.

**Memory fills gaps CLAUDE.md can't.** CLAUDE.md doesn't know who you are. It can't say "explain things at a senior level" because some team members are junior. It can't say "this user was burned by mocked tests last quarter" because that's personal history. Memory provides the per-user context layer that makes Claude's behavior feel personalized without imposing personal preferences on the team.

**The two systems cover different time horizons.** CLAUDE.md captures what's true about the project now and for the foreseeable future. Memory captures what's true about the current situation — ongoing work, recent decisions, temporary constraints. A merge freeze goes in memory. A permanent branch protection rule goes in CLAUDE.md.

## Common Mistakes

Developers frequently put information in the wrong system. Here are the patterns to avoid:

**Mistake 1: Putting personal preferences in CLAUDE.md.** "I prefer functional components over class components" is a personal preference. If it's a team standard, word it as a project rule: "Use functional components with hooks. No class components." If it's just your preference, let memory learn it from your corrections.

**Mistake 2: Putting build commands in memory.** "npm run build" belongs in CLAUDE.md where it's guaranteed to load every session. If Claude learns it through memory, there's a risk of it not loading in a future session, or loading a stale version after you've changed build tools.

**Mistake 3: Duplicating information across both.** If CLAUDE.md says "run npm test before committing" and you also save a memory saying "always run tests before commits," you've created a maintenance burden. When the testing command changes, you'd need to update both. Keep each fact in exactly one place.

**Mistake 4: Putting transient project status in CLAUDE.md.** "We're in a merge freeze until April 25" will be stale by April 26. If it's in CLAUDE.md, someone has to remember to remove it. Memory is better for time-bound context because Claude can recognize when it's no longer relevant.

**Mistake 5: Never using memory at all.** Some developers put everything in CLAUDE.md, including personal workflow notes, role descriptions, and temporary project context. This bloats the file with per-user information that shouldn't be shared. Trust memory for the personal layer.

## When to Choose Claude Memory

Claude Memory is the right choice when:

- **The information is about you, not the project.** Your role, your expertise, your communication preferences, your working hours — these are personal context that shouldn't be in a shared file.
- **The information changes frequently.** Current sprint goals, active incidents, temporary constraints — memory handles the update lifecycle naturally.
- **You want zero-effort persistence.** If you're tired of repeating "I'm a backend engineer, please don't explain what a REST API is," memory solves this automatically after one conversation.
- **The context is conversational.** "Don't mock the database — we got burned last quarter" is feedback that Claude should remember. It's not a project rule; it's a learned correction specific to your experience.
- **You need external system pointers.** "Bugs are tracked in Linear project INGEST" is a reference that helps Claude find information but doesn't belong in project instructions.

Memory works best as a **personalization layer** — it makes Claude behave as if it knows you, without requiring you to maintain that context manually.

## When to Choose CLAUDE.md

CLAUDE.md is the right choice when:

- **The instruction applies to everyone on the team.** Build commands, coding standards, quality gates, architectural constraints — anything that should be consistent regardless of who's working.
- **The information must load every session.** Critical rules that Claude must never forget — like "never skip failing tests" or "always run the validator before committing pipeline changes" — need CLAUDE.md's guaranteed delivery.
- **You want version-controlled history.** If you need to know when a rule was added, who added it, and why, CLAUDE.md's git history provides this. Memory has no audit trail.
- **The content is stable.** Information that won't change for months or years — language versions, framework choices, directory structure conventions — belongs in the stable, deliberately-maintained CLAUDE.md.
- **You need hierarchical scoping.** A root-level CLAUDE.md for the whole project plus subdirectory CLAUDE.md files for module-specific rules gives you fine-grained control that memory doesn't offer.

CLAUDE.md works best as a **project contract** — it defines how Claude should behave in this codebase, and every team member gets the same contract. ## Verdict

**Claude Memory and CLAUDE.md are not alternatives — they're layers.** Use CLAUDE.md for team-shared project rules that must load every session: build commands, coding standards, quality gates, and architectural constraints. Use Claude Memory for personal context that evolves across conversations: your role, your preferences, your feedback corrections, and current project status.

The decision rule is straightforward: **if a teammate needs the same instruction, put it in CLAUDE.md. If it's specific to you or temporary, let memory handle it.** If you're unsure, ask yourself whether the information should survive a `git clone` to a new machine. If yes, CLAUDE.md. If no, memory.

The developers getting the most out of Claude Code use both systems deliberately — a well-structured CLAUDE.md for the project foundation, and accumulated memories for the personal layer that makes every conversation feel like a continuation rather than a cold start. ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions take precedence over memory at runtime. If your CLAUDE.md specifies 2-space indentation but a memory records your preference for tabs, Claude follows the CLAUDE.md rule. This is by design — project standards override personal preferences to keep team consistency.

### Does Claude Memory sync across machines?

No. Claude Memory files are stored locally under `~/.claude/projects/` and do not sync to the repository or any cloud service. If you switch machines, your memories don't follow. CLAUDE.md, by contrast, travels with the repository because it's checked into git. For context that must persist across machines, put it in CLAUDE.md or your global `~/.claude/CLAUDE.md`.

### How do I see what Claude has remembered about me?

Check the `MEMORY.md` index file under `~/.claude/projects/` for your project. It lists all saved memories with one-line descriptions. Each memory is stored as a separate markdown file with frontmatter (name, description, type) and a content body. You can read, edit, or delete these files directly. You can also ask Claude to show or forget specific memories during a conversation.

### Should I commit my personal CLAUDE.md to the repo?

Your personal global CLAUDE.md at `~/.claude/CLAUDE.md` should not be committed — it's for cross-project personal instructions. The project-level CLAUDE.md in the repo root should absolutely be committed and shared. If you have personal project-specific rules, use the project-scoped memory system rather than creating a separate personal CLAUDE.md in the repo.

### What happens if a memory becomes outdated?

Claude is instructed to verify memories against current state before acting on them. If a memory says a function exists but Claude can't find it in the codebase, it should trust the current code over the stale memory and update or remove the memory. However, this verification isn't perfect — for critical, fast-changing facts, prefer CLAUDE.md where humans manage updates deliberately.
