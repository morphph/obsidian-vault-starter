# Codex Subagents 与 Claude Code Subagents：多智能体编码工作流怎么选

**TL;DR：** 如果你需要在云端沙箱中并行处理多个独立任务，**OpenAI Codex** 的多任务模型更直接——每个任务是一个隔离容器，天然并行。如果你需要在本地终端中让主 agent 动态拆分子任务、共享项目上下文，**Claude Code** 的 subagent 机制更灵活，支持自定义 agent 类型和 git worktree 隔离。两者不是同一层面的竞争，而是云端异步 vs 本地实时两种编程范式的分野。

## 概述：OpenAI Codex 的多智能体模式

**OpenAI Codex** 是 OpenAI 于 2025 年推出的云端编程智能体，每个任务运行在一个独立的沙箱容器中。Codex 的"多智能体"并不是传统意义上的 subagent 分派——它的核心思路是**多任务并行**：你同时提交多个任务，每个任务独立运行在自己的环境里，互不干扰。

这种模型适合批量处理：比如同时修复 5 个 bug、给 10 个模块写测试、或者并行审查多个 PR。每个任务有自己的文件系统快照，完成后生成 PR 或 diff 供你审查。Codex 目前集成在 ChatGPT 界面和 [VS Code 扩展](/zh/blog/codex-vscode)中，支持通过 `AGENTS.md` 文件配置全局指令。

关键限制：Codex 的任务之间**无法通信**。没有"主 agent 拆分子任务"的编排层——你就是那个编排者。想了解完整架构，参阅我们的 [Codex 完全指南](/zh/blog/codex-complete-guide)。

## 概述：Claude Code 的 Subagent 系统

**Claude Code** 是 Anthropic 的终端编程智能体，它的 subagent 机制是一套真正的**agent 编排系统**。主 agent 可以通过 `Agent()` 工具动态生成子 agent，每个子 agent 有独立的上下文窗口、指定的工具集，甚至可以运行在隔离的 git worktree 中。

Claude Code 的 subagent 支持多种内置类型：`Explore`（只读搜索）、`Plan`（架构设计）、`codex:codex-rescue`（调用 Codex 作为辅助）、以及默认的通用 agent。你还可以在项目的 `.claude/agents/` 目录下定义自定义 agent，指定系统提示、可用工具和模型。

这套系统的核心优势是**动态编排**：主 agent 根据任务复杂度自行决定是否拆分、如何拆分、分派给哪种 agent。子 agent 完成后，结果返回主 agent 做综合判断。详细案例可参考 [Claude Code Subagents 实战](/zh/blog/claude-code-subagents-examples)。

## 功能对比

| 维度 | OpenAI Codex | Claude Code | 优势方 |
|------|-------------|-------------|--------|
| **运行环境** | 云端沙箱容器 | 本地终端（可选远程） | 各有所长 |
| **并行方式** | 多任务并行提交 | 主 agent 动态分派 subagent | Claude Code |
| **任务间通信** | 不支持 | 子 agent 结果返回主 agent | Claude Code |
| **自定义 agent** | 通过 AGENTS.md 配置指令 | `.claude/agents/` 自定义类型 + 工具集 | Claude Code |
| **隔离机制** | 每任务独立容器 | git worktree 隔离 | Codex |
| **审查模式** | 生成 PR/diff，人工审查 | 实时交互，逐步确认 | 各有所长 |
| **上下文共享** | 各任务独立快照 | 子 agent 继承项目上下文 | Claude Code |
| **定价** | ChatGPT Pro/Team 订阅包含 | 按 token 用量计费 | Codex |

## 架构差异深度分析

两者在"多智能体"概念上的理解截然不同，这是选型时最关键的分水岭。

Codex 采用的是**任务队列模型**。你把多个独立任务推入队列，Codex 为每个任务创建一个完整的沙箱环境——包括代码快照、依赖安装、环境变量。每个任务从头到尾独立运行，没有中间状态可以干预（除了取消）。这种模型的优势是**天然隔离**：一个任务崩溃不影响其他任务，不存在子 agent 之间相互污染的风险。但代价是**无法编排**——如果任务 B 依赖任务 A 的结果，你必须等 A 完成后手动启动 B。

Claude Code 的 subagent 是一个**递归编排模型**。主 agent 在对话过程中分析任务，决定拆分策略，生成带有详细 prompt 的子 agent。子 agent 可以进一步生成自己的子 agent（虽然实际中很少超过两层）。每个子 agent 可以被指定不同的 agent 类型——比如用 `Explore` agent 做代码搜索、用 `Plan` agent 做方案设计、用通用 agent 做实际修改。这种灵活性在处理复杂的跨模块重构时非常有价值。

对于想深入理解 agent 编排架构的读者，我们的 [Agent Harness 深度分析](/zh/blog/agent-harnesses-2026)详细拆解了这两种范式的工程权衡。

## 自定义 Agent 能力对比

自定义能力是评估两个平台"多智能体"成熟度的关键指标。

**Codex 的自定义方式**相对简单：在项目根目录放一个 `AGENTS.md` 文件，写入全局指令——比如编码规范、测试要求、禁止使用的库。所有任务都会读取这个文件作为系统提示的一部分。这本质上是**指令注入**，不是真正的 agent 类型定义。你不能为不同类型的任务创建不同的 agent 配置。

**Claude Code 的自定义**更加结构化。你可以在 `.claude/agents/` 目录下创建 `.md` 文件，每个文件定义一种 agent 类型，包括：

- 系统提示（角色、行为约束）
- 可用工具列表（比如只给代码审查 agent 读权限）
- 模型选择（用 Haiku 做快速搜索、用 Opus 做复杂推理）

这意味着你可以为团队构建一套可复用的 agent 库：代码审查 agent、安全扫描 agent、文档生成 agent——每个都有精确的能力边界。结合 [Claude Code Skills 系统](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)，自定义 agent 可以调用项目级别的 skill 文件，形成完整的工作流。

## 实际工作流场景

### 场景一：批量修复同类 bug

**Codex 更优。** 你有 8 个类似的 bug（比如 API 返回格式不一致），每个涉及不同模块。把 8 个任务同时提交给 Codex，每个在独立沙箱运行，15 分钟后收到 8 个 PR。你批量审查、合并。Claude Code 也能做，但需要串行处理或手动管理多个 subagent 的协调。

### 场景二：跨模块架构重构

**Claude Code 更优。** 你需要把认证模块从 session-based 迁移到 JWT——涉及中间件、数据库 schema、API 路由、前端 token 管理。这需要一个主 agent 理解全局架构，拆分任务（schema 迁移、中间件重写、前端适配），确保子任务之间的接口一致。Codex 的独立任务模型无法处理这种有依赖关系的编排。

### 场景三：开源项目贡献

**Codex 更优，尤其是有免费额度的情况下。** OpenAI 为[开源项目提供免费 Codex 额度](/zh/blog/codex-for-open-source)，适合处理 issue 分拣、简单 PR 生成等重复性工作。如果你是学生，还有[专属的免费计划](/zh/blog/codex-for-students)。

## 什么时候选 Codex

选择 Codex 的多任务模式，当你的场景满足这些条件：

- **任务相互独立**：bug 修复、测试生成、代码格式化——不需要任务间通信
- **需要强隔离**：每个任务有自己的完整环境，不会相互影响
- **异步工作流**：提交任务后去做别的事，稍后回来审查结果
- **已有 ChatGPT 订阅**：Codex 包含在 Pro/Team 计划中，无需额外付费

Codex 的理想用户是需要把重复性编程工作外包给 AI 的团队——尤其是那些可以被标准化、模板化的任务。详细的使用方法见 [Codex 多 Agent 工作流指南](/zh/blog/con-u-pour-des-workflows-multi-agents)。

## 什么时候选 Claude Code

选择 Claude Code 的 subagent 系统，当你需要：

- **动态编排**：任务复杂度不确定，需要 agent 自行判断是否拆分
- **上下文传递**：子任务需要理解主任务的上下文和目标
- **自定义 agent 类型**：不同任务需要不同的工具集和行为约束
- **实时交互**：你想在 agent 工作过程中介入、调整方向

Claude Code 的 subagent 更适合**高复杂度、高上下文依赖**的工程任务。团队可以通过 [Agent Teams 功能](/zh/blog/claude-code-agent-teams)让多个 subagent 协作处理大型代码库的重构。

## 结论

**没有"更好"的多智能体方案，只有更匹配你工作流的选择。** Codex 的多任务模型是云端批处理的标杆——简单、隔离、异步。Claude Code 的 subagent 系统是本地智能编排的代表——灵活、可定制、有上下文感知。

如果你的工作以独立任务为主（修 bug、写测试、处理 issue），Codex 的并行模型更高效。如果你经常做跨模块重构、架构迁移这类需要全局理解的工程任务，Claude Code 的 subagent 编排是更成熟的选择。很多团队两者都用——Codex 处理批量任务，Claude Code 处理复杂编排。

## 常见问题

### Codex 支持 subagent 之间通信吗？

目前不支持。Codex 的每个任务运行在独立沙箱中，任务之间没有通信机制。如果任务有依赖关系，需要人工按顺序提交。这是架构设计的取舍——强隔离换来了简单性和可靠性。

### Claude Code 的自定义 agent 怎么创建？

在项目的 `.claude/agents/` 目录下创建 `.md` 文件，用 frontmatter 定义 agent 的名称、系统提示、可用工具和模型。主 agent 在对话中可以通过 `subagent_type` 参数指定使用哪种自定义 agent，实现不同任务类型的精确分工。

### 两个平台的 subagent 可以混用吗？

可以。Claude Code 内置了 `codex:codex-rescue` agent 类型，允许在 Claude Code 工作流中调用 Codex 作为辅助——比如让 Codex 在沙箱中跑一个耗时的测试套件，同时 Claude Code 继续做其他工作。这种混合模式正在被越来越多的团队采用。
