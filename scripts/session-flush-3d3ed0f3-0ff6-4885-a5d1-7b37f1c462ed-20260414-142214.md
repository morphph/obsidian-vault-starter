# Claude Memory vs CLAUDE.md：两套记忆系统该怎么选？

**TL;DR：** **CLAUDE.md** 是你手写的项目规则手册——确定性强、版本可控、团队共享。**Claude Memory（Auto Memory）** 是 Claude 自动积累的个人笔记——跨会话记住你的偏好和上下文。两者不是替代关系，而是互补：**CLAUDE.md 管规则，Memory 管偏好**。团队协作选 CLAUDE.md，个人效率选 Memory，生产项目两个都用。

## 概述：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Claude Code 的持久化记忆系统，它会在对话过程中自动学习并保存你的工作偏好、项目背景和反馈。这些记忆以 Markdown 文件的形式存储在 `~/.claude/projects/` 目录下，跨会话持续生效。

Memory 的核心价值在于「不用重复说」。你纠正过一次 Claude 的行为——比如「不要在测试里 mock 数据库」——它就会记住，下次自动遵守。Memory 分为四种类型：用户画像（user）、行为反馈（feedback）、项目上下文（project）、外部引用（reference）。每种类型有不同的触发条件和使用场景。

根据 [Claude 记忆系统的最新升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，Memory 现在支持从其他 AI 助手导入数据，免费用户也已开放使用。这意味着 Memory 正在成为 Claude 生态的基础设施层，而不仅仅是一个实验性功能。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件，放在仓库根目录，随代码一起版本控制。它告诉 Claude「在这个项目里应该怎么做」——构建命令、代码规范、架构约束、质量门禁，所有确定性的规则都写在这里。

CLAUDE.md 的设计哲学是「显式优于隐式」。团队里每个人启动 Claude Code 时都会加载同一份 CLAUDE.md，确保 AI 行为的一致性。你不需要依赖 Claude 是否「记得住」——规则就在那里，每次都会被读取。详细的分层设计可参考 [Claude Code 七层架构解析](/zh/blog/claude-code-seven-programmable-layers)。

CLAUDE.md 存在多个层级：全局级（`~/.claude/CLAUDE.md`）、项目级（仓库根目录）、目录级（子目录中的 CLAUDE.md）。层级越深，优先级越高，支持精细化的规则覆盖。

| **生效范围** | 跨项目可用 | 单项目/单目录 | Memory |

## 核心差异：确定性 vs 适应性

这是两套系统最根本的区别。CLAUDE.md 提供的是**确定性保证**——「每次 commit 前必须跑 `npm run build` 和 `npm test`」，这条规则写在 CLAUDE.md 里，100% 会被执行。你不需要担心 Claude 某天「忘了」这条规则。

Memory 提供的是**适应性智能**——Claude 记住你喜欢简洁的 commit message、你是后端工程师所以前端解释要用后端类比、你上周说过这个模块下周要重构所以现在别动它。这些偏好很难（也不应该）写成硬性规则，但它们对协作体验影响巨大。

用一个类比：CLAUDE.md 是公司的员工手册，Memory 是和你搭档了三个月的同事脑子里对你的了解。手册保证了底线一致性，同事的了解提升了协作效率。两者缺一不可。

关于 Claude Code 如何将这些层次整合到一起，[Claude Code 扩展栈解析](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 一文中有详细的架构拆解。

## 实际工作流：该往哪里写？

判断一条信息该放 CLAUDE.md 还是让 Memory 自动处理，用这个决策树：

1. **这条信息是不是团队所有人都需要遵守的？** → CLAUDE.md
2. **这条信息明天还成立吗？下个月呢？** → 如果长期稳定 → CLAUDE.md；如果会变 → Memory
3. **这条信息如果被忽略，会不会导致 bug 或事故？** → CLAUDE.md（确定性保障）
4. **这条信息是关于「我个人怎么工作」的？** → Memory

举几个例子：

- 「Python 项目用 ruff 格式化，不用 black」→ **CLAUDE.md**（团队规范，需确定性）
- 「我更喜欢函数式写法，少用 class」→ **Memory**（个人偏好，不需要强制）
- 「数据库迁移必须先写回滚脚本」→ **CLAUDE.md**（安全规则，不能遗忘）
- 「最近在做性能优化，关注 N+1 查询」→ **Memory**（临时上下文，会过期）

如果你对 Claude Code 的整体工作方式还不太熟悉，[Claude Code 不是一个编程工具](/zh/blog/claude-code-is-not-a-coding-tool) 这篇文章会帮你建立正确的心智模型。

## 什么时候选 Claude Memory

Memory 最适合**个人开发者**和**高频使用者**。如果你每天和 Claude Code 协作几个小时，Memory 的价值会快速累积——它记住你的技术栈偏好、代码风格、常用工具链，省去反复说明的摩擦。

具体场景：

- **独立开发者**管理多个项目，希望 Claude 在所有项目中都保持一致的行为
- **新加入团队的成员**，让 Claude 逐步学习你对这个代码库的理解
- **做探索性工作**，项目方向还不确定，规则还没定型，让 Memory 先积累再固化

Memory 的风险在于它的概率性：Claude 可能在关键时刻没有召回某条记忆。所以**安全关键的规则永远不要只靠 Memory**。

## 什么时候选 CLAUDE.md

CLAUDE.md 最适合**团队项目**和**需要一致性保障的场景**。当你有 3 个以上开发者同时用 Claude Code 工作时，CLAUDE.md 是唯一能保证 AI 行为一致的机制。

具体场景：

- **团队协作**项目，需要所有人的 Claude 遵守相同的构建流程和质量门禁
- **生产环境**代码，部署脚本、数据库操作等不能出错的规则
- **开源项目**，新贡献者 clone 仓库就能获得完整的 AI 协作上下文
- **CI/CD 集成**，CLAUDE.md 里的规则可以被自动化流程验证

想了解 CLAUDE.md 和 SKILL.md 如何配合使用，可以参考 [我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

## 最佳实践：两者配合使用

成熟的 Claude Code 工作流会同时利用两套系统。推荐的分工方式：

**CLAUDE.md 负责「法律」**：构建命令、测试要求、代码规范、安全约束、部署流程。这些是不可协商的硬性规则。

**Memory 负责「文化」**：沟通风格、技术偏好、项目背景、工作节奏。这些是提升体验的软性上下文。

一个常见的演进路径：先让 Memory 自然积累，当某条反馈被触发三次以上时，把它固化到 CLAUDE.md 里。这样 CLAUDE.md 不会膨胀成一份无人维护的巨型文档，Memory 也不会承担超出其能力的确定性保障。

## 结论

**CLAUDE.md 和 Claude Memory 解决的是不同层面的问题。** CLAUDE.md 是你的项目宪法——确定、共享、可审计。Memory 是你的个人助理——灵活、自适应、零维护。选择不是二选一：**用 CLAUDE.md 守住底线，用 Memory 提升体验**。如果只能选一个，团队项目先写 CLAUDE.md，个人项目先让 Memory 跑起来。

## 常见问题

### Claude Memory 会不会泄露个人信息给团队成员？

不会。Memory 存储在你本地的 `~/.claude/` 目录下，不进入 Git 仓库，不会被其他团队成员访问。它是完全个人化的存储，只有你自己的 Claude Code 实例会读取。

### CLAUDE.md 写太长会不会影响性能？

会。CLAUDE.md 每次会话都会全量加载到上下文窗口中，过长的文件会占用宝贵的 token 额度。建议保持在 200 行以内，用 SKILL.md 拆分特定任务的指令，避免把所有规则都塞进一个文件。

### 可以关闭 Auto Memory 吗？

可以。在 Claude Code 的设置中可以禁用 Auto Memory 功能。但建议先试用一段时间再决定——大多数用户在使用两周后会发现 Memory 显著减少了重复沟通的成本。

### CLAUDE.md 支持哪些层级？

三个层级：全局级（`~/.claude/CLAUDE.md`，所有项目生效）、项目级（仓库根目录，当前项目生效）、目录级（子目录中的 CLAUDE.md，该目录下生效）。层级越深优先级越高，子目录规则会覆盖项目级规则。
