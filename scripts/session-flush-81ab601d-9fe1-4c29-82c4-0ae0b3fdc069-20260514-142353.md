# Claude Memory vs CLAUDE.md：两套记忆系统，该用哪个？

**TL;DR：** **CLAUDE.md** 是你写给 Claude 的规则手册——确定性强、版本可控、团队共享。**Claude Memory**（Auto Memory）是 Claude 自己积累的笔记——跨会话持久化、个人专属、自动生成。两者不是竞争关系，而是互补：**CLAUDE.md 管"必须怎么做"，Memory 管"上次聊到哪"**。如果你只能选一个先搞好，先写 CLAUDE.md。

## 概览：Claude Memory

**Claude Memory** 是 Anthropic 在 Claude Code 中内置的持久化记忆系统，解决的核心问题是"每次新对话都要从零开始介绍自己和项目"。它通过 `~/.claude/projects/` 目录下的 markdown 文件，在多次会话之间保存用户偏好、项目背景和工作上下文。

Memory 分为四种类型：`user`（用户画像）、`feedback`（行为纠偏）、`project`（项目动态）、`reference`（外部资源指针）。Claude 会在对话中自动识别值得记住的信息并写入 Memory 文件，你也可以通过"记住这个"之类的指令手动触发保存。详细机制可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

Memory 的最大优势是**零配置**——你不需要提前写任何东西，Claude 会随着交互自动学习你的工作模式。但它也有明显局限：Memory 是个人的，不能在团队成员之间共享；内容由 AI 决定记什么、怎么记，你无法完全控制。

## 概览：CLAUDE.md

**CLAUDE.md** 是放在项目根目录的指令文件，本质上是一份你写给 Claude 的操作手册。它随代码仓库一起版本控制，团队每个人 clone 下来都能用。

CLAUDE.md 的内容通常包括：构建命令（`npm run build`、`npm test`）、代码规范（命名约定、提交信息格式）、架构约束（"不要在 pipeline 脚本里 import Next.js 模块"）、质量门禁（"提交前必须通过所有测试"）。它是确定性的——你写什么 Claude 就遵守什么，没有 AI 推断的模糊空间。

CLAUDE.md 还支持层级结构：项目根目录的 `CLAUDE.md` 定义全局规则，子目录可以放各自的 `CLAUDE.md` 覆盖或补充。配合 [Skills 文件](/zh/blog/do-skills-actually-improve-your-agents-output)（`skills/*/SKILL.md`），可以为不同任务类型定义专用指令集。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 胜出 |
|------|--------------|-----------|------|
| **持久性** | 跨会话自动保存 | 随代码仓库永久保存 | 平手 |
| **共享性** | 仅限个人 | 团队共享（Git） | CLAUDE.md |
| **可控性** | AI 决定记什么 | 你决定写什么 | CLAUDE.md |
| **上手成本** | 零配置 | 需要手动编写 | Memory |
| **版本控制** | 有文件但不在 Git 中 | 随代码一起版本控制 | CLAUDE.md |
| **适应性** | 自动学习偏好变化 | 需要手动更新 | Memory |
| **确定性** | AI 推断，可能不一致 | 明确规则，确定性执行 | CLAUDE.md |
| **个性化** | 记住你的角色和习惯 | 定义项目规则，不涉及个人 | Memory |

## 记忆机制：深入分析

Claude Memory 和 CLAUDE.md 的根本区别在于**谁在做决策**。

CLAUDE.md 是声明式的：你告诉 Claude "提交前必须跑 `npm test`"，Claude 就会在每次准备提交时执行测试。规则明确、可预测、可审计。如果某条规则不再适用，你删掉那一行，所有人立即生效。这种模式适合团队协作——新成员不需要"训练" Claude，clone 仓库就能获得完整的 AI 行为规范。

Claude Memory 是归纳式的：Claude 在对话中观察你的行为模式，提取它认为有价值的信息，写入持久化文件。比如你纠正过一次"不要在响应末尾总结"，Claude 会记住这个偏好并在未来的对话中自动遵守。这种机制的好处是省心，缺点是你无法完全预测 Claude 记住了什么、会怎么应用。

从 [Claude Code 的七层架构](/zh/blog/claude-code-seven-programmable-layers) 来看，CLAUDE.md 属于"Project Context"层，是最早被加载到上下文中的指令；Memory 属于"Persistent State"层，在项目上下文之后被注入。两者在 Claude 的注意力分配中具有不同的权重——通常 CLAUDE.md 中的显式规则优先级更高。

## 实际使用场景：CLAUDE.md 更合适

以下场景应该优先使用 CLAUDE.md：

**团队协作项目**。当三个工程师共用一个仓库时，你不可能指望每个人的 Memory 都一致。把"commit message 用英文""API 路由放在 `src/app/api/`""数据库迁移需要更新 SYSTEM-OVERVIEW.md"这些规则写进 CLAUDE.md，所有人立即对齐。

**质量门禁和流程控制**。"提交前必须通过 build、test、lint"这类硬性要求必须写在 CLAUDE.md 中。Memory 是建议性的，Claude 可能在某些上下文中忘记应用；CLAUDE.md 是指令性的，Claude 会严格遵守。

**项目架构约束**。"不要在 pipeline 脚本里 import Next.js 模块""upsertKeyword() 必须传三个参数"——这些容易犯错的技术陷阱，写入 CLAUDE.md 的 Known Gotchas 部分，可以避免 Claude 反复踩坑。

## 实际使用场景：Claude Memory 更合适

以下场景 Memory 的优势更明显：

**个人工作偏好**。你喜欢简洁回复还是详细解释？习惯先看 diff 还是先看总结？这些个人偏好不应该写进团队共享的 CLAUDE.md，而应该让 Memory 自然积累。当你说"别每次结尾都做总结，我会看 diff"，Claude 记住后在所有项目中都会遵守。

**跨会话上下文接续**。上一次对话中你正在调试某个数据库迁移问题，中途关掉了终端。下次打开时，Memory 可以帮 Claude 快速回忆"上次我们在处理什么"，省去重新解释的时间。

**用户角色理解**。你是资深后端工程师但刚接触前端？Memory 会记住这一点，在解释 React 概念时用后端类比，在讨论 Go 代码时直接用高级术语。这种个性化是 CLAUDE.md 无法提供的。

Anthropic 在最近的 [记忆系统升级](/zh/blog/anthropic-claude-memory-upgrades-importing) 中还增加了从 ChatGPT 导入记忆的功能，进一步降低了迁移成本。

## 最佳实践：两者配合使用

实际上，成熟的 Claude Code 工作流会同时使用两套系统，各司其职：

**CLAUDE.md 管规则**：构建命令、代码规范、质量门禁、架构约束、已知陷阱。这些是客观的、确定的、团队共享的。

**Memory 管偏好**：沟通风格、个人习惯、角色背景、项目动态。这些是主观的、演化的、个人专属的。

一个实用的检验标准：**如果某条信息在你离职后仍然有用，写进 CLAUDE.md；如果只对你个人有意义，交给 Memory。**

更高级的用法是将 CLAUDE.md 与 [Skills 系统和 Hooks](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 结合。CLAUDE.md 定义全局行为约束，SKILL.md 定义任务级专用指令，Hooks 在工具调用前后执行自动化检查。三层叠加，构成完整的 [agentic coding](/zh/glossary/agentic-coding) 控制体系。

## 结论

**如果你只做一件事，先写好项目的 CLAUDE.md**——它的投入产出比最高，10 分钟的配置能节省每次对话中反复解释的时间，而且团队每个人都受益。Memory 作为补充，让 Claude 在遵守团队规则的前提下，进一步适应你个人的工作方式。两者互补，不要只用其中一个。

对于正在建设 [AI 驱动的开发工作流](/zh/blog/claude-code-is-not-a-coding-tool) 的团队，建议的优先级是：先写好 CLAUDE.md → 再通过日常使用积累 Memory → 最后根据任务类型拆分 SKILL.md。

## 常见问题

### Claude Memory 的内容会被其他人看到吗？

不会。Claude Memory 存储在你本地的 `~/.claude/projects/` 目录下，仅限你个人的 Claude Code 会话读取。它不会上传到云端，也不会出现在 Git 仓库中。团队成员之间的 Memory 完全隔离。

### CLAUDE.md 写太多规则会不会影响 Claude 的性能？

会有影响。CLAUDE.md 的内容会被注入到每次对话的上下文中，过长的文件会占用有限的上下文窗口。建议控制在 200 行以内，把详细的任务指令拆分到 SKILL.md 文件中，按需加载而非全局加载。

### 两者的规则冲突时，Claude 听哪个？

CLAUDE.md 的优先级更高。作为显式的项目指令，CLAUDE.md 中的规则会覆盖 Memory 中可能存在的矛盾偏好。例如，如果 CLAUDE.md 要求"commit message 用英文"，即使你的 Memory 中记录了"用户偏好中文"，Claude 在提交时仍会使用英文。
