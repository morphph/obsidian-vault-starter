# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **CLAUDE.md** 是你手写的规则手册，签入仓库、团队共享、确定性执行——适合编码规范、架构约束、工作流指令。**Claude Memory（Auto Memory）** 是 Claude 自动记录的个人笔记本，存在本地、跨会话持久化——适合用户偏好、项目背景、反馈记忆。两者不是替代关系，而是互补：CLAUDE.md 管「团队该怎么做」，Memory 管「我是谁、我喜欢怎么做」。

## 概览：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Claude Code 的自动持久化记忆系统，存储在 `~/.claude/projects/` 目录下。它的核心机制是：Claude 在对话过程中自动识别值得记住的信息——你的角色、偏好、项目上下文、工作反馈——然后写入独立的 markdown 文件，并在 `MEMORY.md` 索引中维护指针。

Memory 的设计哲学是「对话即配置」。你不需要手动编辑配置文件，只需要在对话中自然表达：「我是数据科学家」「不要在测试里 mock 数据库」「这个项目的 deadline 是下周四」。Claude 会自动分类（user / feedback / project / reference）并持久化。下次打开新会话时，这些记忆会被加载到上下文中，让 Claude 从你上次离开的地方继续。

关于 Memory 系统的完整机制，可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。Anthropic 近期还对记忆系统做了重大升级，包括免费用户开放和 [ChatGPT 数据导入支持](/zh/blog/anthropic-claude-memory-upgrades-importing)。

## 概览：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件，本质上是一个纯文本的规则手册。它存在于项目根目录（或 `~/.claude/CLAUDE.md` 作为全局配置），会在每次会话启动时被自动加载到系统提示词中。

CLAUDE.md 的设计哲学是「显式优于隐式」。所有规则都是你手写的、可审查的、可版本控制的。团队成员 `git pull` 之后，每个人的 Claude Code 都会遵循同一套规范——编码风格、提交流程、质量门禁、架构约束。这一点对团队协作至关重要。

CLAUDE.md 配合 `skills/*/SKILL.md` 文件形成完整的指令体系。CLAUDE.md 定义全局规则，SKILL.md 定义特定任务的执行标准。想了解技巧文件的最佳实践，推荐阅读 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **存储位置** | `~/.claude/projects/` 本地目录 | 项目根目录（可签入 Git） | CLAUDE.md |
| **创建方式** | Claude 自动生成 | 开发者手动编写 | Memory |
| **团队共享** | 不共享（个人本地） | Git 版本控制，团队共享 | CLAUDE.md |
| **内容类型** | 用户偏好、反馈、项目背景 | 编码规范、架构约束、工作流 | 各有所长 |
| **可审查性** | 可读但松散组织 | 结构化、可 code review | CLAUDE.md |
| **维护成本** | 零——自动维护 | 需要手动更新 | Memory |
| **确定性** | 上下文相关，可能不加载 | 每次会话必定加载 | CLAUDE.md |
| **跨项目** | `~/.claude/CLAUDE.md` 全局 Memory | 每个项目独立 | 平手 |

## 核心差异：手动规则 vs 自动笔记

这是两套系统最本质的区别。CLAUDE.md 里的每一行都是你有意识写下的指令，具有命令性质——Claude **必须**遵循。Memory 里的内容是 Claude 从对话中提取的上下文，具有参考性质——Claude **应该**考虑，但不是硬性约束。

举个具体例子：

你在 CLAUDE.md 里写「所有 API 端点必须有 Zod schema 验证」，这是一条强制规则。Claude 在生成任何 API 代码时都会加上 Zod 验证，没有例外。

你在对话中说「我更喜欢用 early return 而不是嵌套 if」，Memory 会记住这个偏好。下次写代码时 Claude 会倾向于用 early return，但如果上下文更适合其他写法，它可能灵活处理。

这种区分很重要：**把偏好写进 CLAUDE.md 会让系统变得僵硬，把规则丢给 Memory 会让系统变得不可靠。** 选错存储位置是最常见的配置错误。

## 协作场景：团队 vs 个人

CLAUDE.md 是团队协作的基础设施。当你在 CLAUDE.md 中定义「commit message 必须以动词开头」「测试覆盖率不低于 80%」「PR 描述必须包含 Test Plan」，每个团队成员的 Claude Code 实例都会自动执行这些规范。不需要口头约定，不需要 code review 时反复纠正——规范直接内嵌在 AI 工作流中。

Memory 是个人效率的加速器。它记住你是前端还是后端、你习惯用 vim 还是 VS Code、你上周在做的功能分支叫什么名字。这些信息对你个人非常有用，但对团队其他成员毫无意义。

从 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 的角度看，CLAUDE.md 和 SKILL.md 构成了「指令层」，Memory 构成了「上下文层」，Hooks 构成了「自动化层」，MCP 构成了「工具层」。四层各司其职，不应混淆。

## 实战建议：什么该放哪里

**放 CLAUDE.md 的内容：**
- 编码规范和风格约束（缩进、命名、import 排序）
- 构建和质量门禁（`npm run build` 必须通过才能 commit）
- Git 工作流规则（commit message 格式、分支命名）
- 架构约束（禁止在 pipeline 脚本里 import Next.js 模块）
- 常见陷阱提醒（中文内容必须用 CJK word count）

**放 Memory 的内容：**
- 个人角色和背景（「我是数据科学家，主要关注 observability」）
- 交互偏好（「不要在回答末尾加总结」「我喜欢简洁的回复」）
- 项目背景知识（「auth 重写是因为合规要求，不是技术债」）
- 外部资源指向（「pipeline bug 在 Linear 的 INGEST 项目里」）

**不该放在任何一个里的内容：**
- 代码结构和文件路径——直接读代码更准确
- Git 历史和变更记录——`git log` 是权威来源
- 调试方案和修复步骤——fix 在代码里，context 在 commit message 里

如果你刚开始使用 Claude Code，建议先从 CLAUDE.md 入手。花 15 分钟写清楚项目的基本规范，比花一周「训练」Memory 的效果好得多。关于如何有效配置 Claude Code 的工作方式，可以参考 [如何有效提示 Claude Code](/zh/blog/how-to-effectively-prompt-a-claude-code)。

## 该选哪个

不存在「选哪个」的问题——**两者应该同时使用**。

CLAUDE.md 是你的团队宪法，Memory 是你的个人助理笔记。宪法确保一致性和可预测性，笔记提升个人效率和上下文连续性。

如果你是独立开发者，CLAUDE.md 的团队共享优势弱化了，但它的「确定性执行」依然不可替代。你不会希望关键的架构约束依赖 Memory 的模糊匹配，而不是 CLAUDE.md 的强制加载。

如果你在团队中工作，CLAUDE.md 是必须的基础设施，Memory 是锦上添花的个人优化。团队中每个人的 Memory 不同，但 CLAUDE.md 相同——这正是设计意图。想了解更多 Claude Code 的团队工程实践，可以看 [Claude Code 企业落地实录](/zh/blog/claude-code-enterprise-engineering-ramp-shopify-spotify)。

**决策规则很简单：如果这条信息换一个人也需要遵循，放 CLAUDE.md；如果只对你自己有用，让 Memory 自动处理。**

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。CLAUDE.md 在每次会话启动时强制加载，提供确定性的规则执行。Memory 在上下文相关时被召回，提供个性化的背景信息。两者互不冲突，分别解决不同层面的问题。

### CLAUDE.md 的内容会被 Memory 覆盖吗？

不会。CLAUDE.md 中的指令优先级高于 Memory。如果 CLAUDE.md 规定「使用 4 空格缩进」而 Memory 记录了你偏好 2 空格，Claude 会遵循 CLAUDE.md 的规则。CLAUDE.md 是强制指令，Memory 是参考建议。

### Memory 记错了怎么办？

Memory 文件存储在 `~/.claude/projects/` 目录下，都是纯文本 markdown 文件。你可以直接编辑或删除。也可以在对话中告诉 Claude「忘掉关于 X 的记忆」，它会自动清理对应的 Memory 条目。

### 新团队成员如何快速上手？

让新成员 clone 仓库后，CLAUDE.md 和 SKILL.md 会自动生效——这是 CLAUDE.md 最大的团队价值。Memory 需要通过实际使用逐步积累，无法批量导入团队经验。如果团队有重要的上下文知识（如「为什么选了这个架构」），应该写进 CLAUDE.md 而不是依赖个人 Memory。
