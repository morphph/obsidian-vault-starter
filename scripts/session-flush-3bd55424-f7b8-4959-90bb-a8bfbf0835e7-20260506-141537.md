**Put it in CLAUDE.md if:**
- It applies to every developer on the project
- It should survive when a new team member clones the repo
- It's a rule that must be enforced (not a preference)
- It relates to build commands, test requirements, or quality gates
- It documents architecture decisions or known gotchas
- It needs to work in CI/CD where no memory exists

**Put it in Memory if:**
- It's about you specifically — your role, expertise level, preferences
- It's feedback on Claude's behavior ("stop summarizing," "be more concise")
- It's temporary project context that changes frequently
- It references external systems or resources (Slack channels, dashboards, issue trackers)
- It's something you'd tell a new pair-programming partner but wouldn't write in a README

**Examples:**

| Context | Where | Why |
|---------|-------|-----|
| "Run `npm run build` before committing" | CLAUDE.md | Project rule, applies to everyone |
| "I'm a data scientist, not a frontend dev" | Memory | Personal context, shapes explanations |
| "Use Vitest, not Jest" | CLAUDE.md | Project standard, not personal preference |
| "Don't add emoji to commit messages" | Memory (if personal) or CLAUDE.md (if team rule) | Depends on scope |
| "Auth migration is due by March 15" | Memory | Temporary project context, will expire |
| "Never import Next.js modules in pipeline scripts" | CLAUDE.md | Architecture constraint, must be enforced |
| "Bugs are tracked in Linear project INGEST" | Memory | External reference, varies by team setup |

## Team Workflows: How They Complement Each Other

For teams, CLAUDE.md and Memory create a two-layer system that handles both shared standards and individual context. The team lead writes CLAUDE.md with project conventions. Each developer's Memory accumulates their personal working style.

A practical team workflow looks like this: the tech lead maintains a CLAUDE.md that covers build commands, testing requirements, code style, deployment procedures, and architecture constraints. Individual developers don't need to think about Memory — it builds itself as they work. Over a few sessions, each developer's Claude Code instance learns their communication preferences, areas of expertise, and current focus areas.

The separation prevents a common failure mode in AI-assisted development: context drift. Without CLAUDE.md, each developer's Claude Code instance could gradually develop different assumptions about project conventions. One developer's Claude might learn "we use single quotes" from their codebase interactions, while another's learns "we use double quotes" because they worked on a different module first. CLAUDE.md eliminates this by making conventions explicit and shared.

Memory handles the equally important but harder-to-standardize dimension: how each person works. The developer who wants Claude to explain every change gets explanations. The developer who just wants diffs gets diffs. Both follow the same commit message format because CLAUDE.md says so.

## Maintenance and Lifecycle

CLAUDE.md requires active maintenance. When you change your build system from webpack to Vite, you update CLAUDE.md. When you add a new testing requirement, you update CLAUDE.md. When you deprecate an API pattern, you update CLAUDE.md. This maintenance cost is real but manageable — treat CLAUDE.md updates as part of the same PR that changes the thing it documents.

Memory maintains itself, but that's both a strength and a weakness. Memories can become stale. If Claude saved "the user is working on the auth migration" three months ago and the migration is done, that memory is noise. Claude Code does prune memories when it detects contradictions, but it can't always tell when context has expired. Periodic manual review of your memory files helps — check `~/.claude/projects/` occasionally and delete entries that are no longer relevant.

The version control aspect of CLAUDE.md creates an underappreciated benefit: you can see how your project instructions evolved over time. `git blame` on CLAUDE.md shows you when each rule was added and (via the commit message) why. Memory has no equivalent history — once a memory is updated, the previous version is gone.

## Scaling: From Solo to Enterprise

For a **solo developer**, start with CLAUDE.md. Write your build commands, testing requirements, and the two or three project-specific gotchas that trip you up. Let Memory handle everything else organically. You'll have a working system in under an hour.

For a **small team (2-10 developers)**, CLAUDE.md becomes essential. Without it, each developer's Claude Code instance develops independently, and you lose consistency. Write a thorough CLAUDE.md during project setup, require CLAUDE.md updates in PR reviews, and let each developer's Memory handle their individual preferences.

For **larger teams and enterprise**, CLAUDE.md becomes part of your engineering standards infrastructure — alongside linting configs, CI pipelines, and style guides. Some organizations maintain a global CLAUDE.md template that all projects inherit from, plus project-specific overrides. Memory at this scale is purely individual and isn't something the organization manages.

The [Claude Code memory system](/blog/claude-code-memory) documentation covers the technical details of how both systems are loaded and prioritized during a session.

## When to Choose Claude Memory

Choose Memory as your primary context mechanism when:

- You're the sole developer and your preferences are the project's preferences
- You want zero-configuration context that improves over time
- Your workflow involves frequent context-switching between projects and you need Claude to remember where you left off
- You're exploring or prototyping and don't want the overhead of maintaining instruction files
- Your feedback to Claude is primarily about communication style and interaction preferences

Memory shines in long-term solo relationships with Claude. After a few weeks of regular use, Claude Code knows your stack preferences, your debugging approach, your tolerance for explanation, and your current project priorities — all without you writing a single instruction file.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary context mechanism when:

- Multiple developers work on the project and need consistent AI behavior
- You need Claude Code to work in CI/CD pipelines where no user memory exists
- Your project has strict coding standards, quality gates, or architecture constraints that must be enforced
- You want auditable, reviewable instructions — you need to know exactly what Claude is being told
- You're onboarding new team members and want them to get correct AI behavior immediately

CLAUDE.md is non-negotiable for any shared codebase. The question isn't whether to use it — it's how thorough to make it. Start with build commands and testing requirements, then add architecture constraints and known gotchas as they come up. ## Verdict

**Use both.** CLAUDE.md and Claude Memory aren't competing systems — they're complementary layers that solve different problems. **CLAUDE.md is your project's constitution**: the shared rules that ensure consistent, correct AI behavior across every developer and every CI run. **Claude Memory is your personal assistant's notebook**: the accumulated understanding of who you are and how you prefer to work.

If you're forced to pick one, pick CLAUDE.md. Deterministic, shared instructions are more valuable than adaptive personal context, especially on teams. But you shouldn't have to pick — Claude Code uses both simultaneously, and the best workflows leverage the strengths of each.

## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md instructions take precedence over memory for project-level rules. If CLAUDE.md says "run tests before every commit," memory cannot override that requirement. Memory influences tone, communication style, and personal preferences — not project constraints that apply to the whole team.

### Do I need to manually write Claude Memory entries?

No. Claude Code's auto-memory system creates and updates memory entries automatically based on your interactions. You can explicitly ask Claude to remember something, and you can edit or delete memory files in `~/.claude/projects/`, but the system works without any manual intervention. Most developers never touch their memory files directly.

### Does CLAUDE.md work in CI/CD pipelines?

Yes. Since CLAUDE.md is a file checked into your repository, it's available anywhere your code is — including CI/CD pipelines, cloud development environments, and fresh clones. This is one of its key advantages over Memory, which is tied to a specific user's local machine and isn't available in automated pipelines.

### How often should I update CLAUDE.md?

Update CLAUDE.md whenever you change something it documents — a new build command, a deprecated pattern, a changed testing requirement. The best practice is to include CLAUDE.md updates in the same PR that changes the underlying system. Stale CLAUDE.md instructions are worse than no instructions because they actively mislead Claude Code.

### Is Claude Memory shared across devices?

Claude Code's auto-memory is stored locally in `~/.claude/projects/` and does not sync across devices by default. If you work on multiple machines, each will develop its own memory independently. CLAUDE.md, being a repository file, is automatically available on every device where you clone the project.
