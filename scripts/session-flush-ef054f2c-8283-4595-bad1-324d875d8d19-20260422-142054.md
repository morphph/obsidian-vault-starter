# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **CLAUDE.md** 是你主动编写的规则手册，每次对话都会完整加载，适合团队共享的编码规范和项目约束。**Claude Memory** 是自动积累的上下文笔记，跨会话持久化但选择性召回，适合记录个人偏好和项目背景。两者不是替代关系——最佳实践是同时使用，让 CLAUDE.md 定义「必须做什么」，让 Memory 记住「上次聊了什么」。

## 概述：Claude Memory

**Claude Memory**（自动记忆）是 Claude Code 内置的文件级持久化记忆系统。它将对话中学到的信息——用户偏好、项目背景、工作反馈——自动写入 `.claude/projects/` 目录下的 Markdown 文件，并通过 `MEMORY.md` 索引管理。

这套系统的核心设计思路是「积累」：你不需要手动维护，Claude 会在对话中识别值得记住的信息并主动保存。记忆分为四种类型——user（用户画像）、feedback（行为纠正）、project（项目动态）、reference（外部资源指针）。每条记忆都有 frontmatter 元数据，方便后续检索和过期清理。

关于 Memory 系统的完整机制，可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的显式指令文件，相当于给 AI 写的「员工手册」。它可以存在于三个层级：全局（`~/.claude/CLAUDE.md`）、项目根目录（`./CLAUDE.md`）、以及项目级私有（`.claude/CLAUDE.md`）。每次 Claude Code 启动时，所有层级的 CLAUDE.md 会被合并加载到上下文中。

CLAUDE.md 的内容通常包括：构建命令、代码风格要求、Git 工作流规范、技术栈约束、已知问题清单等。它是确定性的——写了什么就加载什么，不存在「可能被遗忘」的情况。因为它是普通文件，可以提交到 Git 仓库，整个团队共享同一套规则。

Anthropic 在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers) 中将 CLAUDE.md 定位为最基础的配置层，是所有高级功能（Skills、Hooks、MCP）的地基。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md |
|------|--------------|-----------|
| **创建方式** | AI 自动写入或用户指令触发 | 人工手动编写 |
| **加载机制** | 选择性召回（按相关性） | 每次对话完整加载 |
| **存储位置** | `.claude/projects/*/memory/` | 项目根目录或 `~/.claude/` |
| **版本控制** | 通常不提交到 Git | 提交到 Git，团队共享 |
| **内容类型** | 用户偏好、反馈纠正、项目动态 | 编码规范、构建命令、约束规则 |
| **持久性** | 跨会话持久，可能过期 | 永久有效，除非手动修改 |
| **团队协作** | 个人私有 | 团队共享 |
| **确定性** | 低——可能不被召回 | 高——始终加载 |

## 加载机制：确定性 vs 启发式

这是两套系统最根本的区别。CLAUDE.md 采用确定性加载——文件存在就会被读取，内容完整注入上下文窗口，没有任何筛选或过滤。你写了「所有测试必须通过才能提交」，Claude 每次都会看到这条规则。

Claude Memory 则是启发式召回。`MEMORY.md` 索引会被加载，但具体的记忆文件只在系统判断相关时才读取。这意味着一条三周前保存的 feedback 记忆，在当前对话中可能不会被主动想起——除非你明确提到相关话题，或者 Claude 在 `MEMORY.md` 的索引描述中匹配到了关联关键词。

实际影响：**关键规则必须放在 CLAUDE.md 里**。如果一条规范被遗忘的后果是代码质量下降或构建失败，它不能只存在于 Memory 中。Memory 适合「锦上添花」的上下文——知道了更好，不知道也不会出错。

## 内容边界：该写什么，不该写什么

CLAUDE.md 的内容应该是**不随对话变化的稳定规则**：构建命令（`npm run build`）、代码风格（Tailwind v4 + TypeScript）、Git 工作流（commit 前必须跑测试）、已知陷阱（「ZH 内容必须用 CJK 字数统计」）。这些信息对每次对话都适用，写在 CLAUDE.md 里最高效。

Memory 的内容应该是**从对话中自然涌现的上下文**：用户角色（「数据科学家，关注可观测性」）、行为偏好（「不要在回复末尾加总结」）、项目动态（「周四之后冻结非关键合并」）、外部资源（「pipeline bug 在 Linear 的 INGEST 项目里追踪」）。这些信息具有个人性、时效性，不适合硬编码在版本控制文件中。

有一类信息两者都不该存：代码结构、文件路径、函数签名——这些直接读代码就能获取，记忆只会增加过期风险。

## 团队场景：共享规则 vs 个人上下文

在团队协作中，CLAUDE.md 是**唯一可靠的共享规则通道**。当你把编码规范写进 `CLAUDE.md` 并提交到仓库，团队中每个使用 Claude Code 的成员都会自动遵循同一套规则——不需要每个人重复同样的提示词。

结合 [Skills 系统](/zh/blog/9-principles-writing-claude-code-skills)，CLAUDE.md 构成了团队 AI 协作的基础设施：CLAUDE.md 定义全局约束，`skills/*/SKILL.md` 定义具体任务的执行标准。两者都通过 Git 版本控制，可审查、可回溯。

Memory 则是完全个人化的。它不会被提交到仓库，不会被其他团队成员看到。这恰恰是它的优势——你可以让 Claude 记住「我是前端新手，解释 React 概念时用后端类比」，而不必把个人学习背景暴露在团队共享文件中。

## 什么时候用 Claude Memory

选择 Memory 的典型场景：

- **跨会话连续性**：你在多天内开发同一个功能，需要 Claude 记住之前的讨论结论和决策背景
- **行为纠正**：你发现 Claude 总是做某个你不喜欢的事情（比如过度注释），告诉它一次之后希望它永远记住
- **个人工作习惯**：你偏好的代码风格、沟通方式、输出格式等主观偏好
- **项目实时状态**：「正在做 auth 模块重构，周五前完成」——有时效性、不适合写进 CLAUDE.md 的临时上下文

Anthropic 最近还在 [Memory 系统升级](/zh/blog/anthropic-claude-memory-upgrades-importing) 中增加了从 ChatGPT 导入记忆的功能，降低了迁移成本。

## 什么时候用 CLAUDE.md

选择 CLAUDE.md 的典型场景：

- **构建和验证命令**：`npm run build`、`npm test`——每次都要用，不能靠记忆
- **代码规范**：命名约定、文件结构、禁止的模式——违反的后果是代码质量下降
- **Git 工作流**：commit message 格式、分支策略、CI/CD 要求
- **已知陷阱和技术债**：「不要在 pipeline 脚本里直接 import Next.js 模块」——被遗忘的后果是构建失败
- **团队统一规则**：所有成员都必须遵循的约束，不能依赖每个人自己的 Memory

一条好的判断标准：**如果这条规则被遗忘会导致错误或返工，它属于 CLAUDE.md；如果被遗忘只是体验略差，它可以放在 Memory 里。**

## 最佳实践：两者搭配使用

成熟的 Claude Code 工作流同时使用两套系统，各司其职：

**CLAUDE.md 作为「宪法」**——不可违反的规则。项目级 CLAUDE.md 定义技术栈、构建流程、质量门禁。全局 CLAUDE.md 定义个人的通用工作习惯。这些内容变更频率低，每次修改都应该像代码变更一样经过审查。

**Memory 作为「笔记本」**——随时积累的上下文。Claude 在对话中自动记录你的偏好、纠正、项目动态。你也可以主动说「记住这个」来触发保存。定期检查 `MEMORY.md` 索引，清理过期条目。

关于 Claude Code 完整的可编程层级——从 CLAUDE.md 到 Skills、Hooks、MCP——参考我们的 [扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 结论

**CLAUDE.md 和 Claude Memory 解决的是不同的问题**。CLAUDE.md 是规则引擎——确定性加载、团队共享、版本控制，适合一切「必须遵守」的规范。Memory 是上下文引擎——自动积累、个人私有、启发式召回，适合一切「知道了更好」的背景信息。把关键规则写在 CLAUDE.md 里确保不被遗忘，把个人偏好和项目动态交给 Memory 自动管理——这才是两套系统的正确打开方式。

## 常见问题

### Claude Memory 会自动过期吗？

Memory 文件本身不会自动删除，但内容可能变得过时。Claude 在使用记忆前会尝试验证其准确性——如果记忆中提到的文件或函数已不存在，它会优先信任当前代码状态。建议定期检查 `MEMORY.md` 索引，手动清理不再相关的条目。

### CLAUDE.md 的内容会占用上下文窗口吗？

会。CLAUDE.md 的全部内容在每次对话开始时注入上下文窗口，这意味着过长的 CLAUDE.md 会减少可用于实际任务的上下文空间。建议保持简洁——聚焦关键规则和约束，避免写入可以通过读代码获取的信息（如完整文件结构、函数列表）。

### 团队中每个人的 Memory 会冲突吗？

不会。Memory 是完全个人化的，存储在各自的 `.claude/projects/` 目录下，不会提交到 Git。团队成员之间的 Memory 互不可见、互不影响。共享规则通过 CLAUDE.md 和 Skills 文件在仓库层面同步。
