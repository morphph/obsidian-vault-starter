# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR:** **Claude Memory** 是 Anthropic 的跨会话自动记忆系统，Claude 会自行决定记住什么；**CLAUDE.md** 是存放在代码仓库里的项目级指令文件，由开发者手动编写和维护。前者解决「每次开新对话都要重新解释背景」的问题，后者解决「AI 不了解项目规范和架构约束」的问题。**需要个人偏好跨会话延续，用 Memory；需要团队共享的项目规则，用 CLAUDE.md；大多数严肃的工程场景，两者同时使用。**

## 概述：Claude Memory

**Claude Memory** 是 Anthropic 在 Claude 产品中内置的持久化记忆功能，让 Claude 能够在不同对话之间记住用户的偏好、背景和工作习惯。用户无需每次都重复说明自己的角色、技术栈或沟通偏好——Claude 会自动提取关键信息并存储。

Memory 的核心定位是**个人化**。它记录的是「你是谁」和「你喜欢怎样协作」，而不是某个项目的具体技术细节。例如，Claude 可能记住你是一位偏好简洁回答的高级工程师、你的主力语言是 TypeScript、你不喜欢代码里加注释。这些记忆会自动应用到你未来的所有对话中。

Anthropic 在 2026 年初对 Memory 做了重大升级，[免费用户也可以使用记忆功能，并支持从 ChatGPT 导入数据](/zh/blog/anthropic-claude-memory-upgrades-importing)。用户可以在设置中查看、编辑和删除已保存的记忆条目。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级配置文件，以 Markdown 格式存放在代码仓库的根目录。它的作用相当于一份「项目规则手册」，告诉 Claude Code 这个项目的构建命令、代码风格、架构约束和工作流程规范。

CLAUDE.md 的核心定位是**项目化和团队化**。它跟随代码仓库走，任何打开这个仓库的人（无论是你还是你的同事）都会自动获得相同的 AI 行为。关于 CLAUDE.md 在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中的定位，它是最基础的上下文层——项目级指令优先于个人记忆。

CLAUDE.md 由开发者手动编写和维护，内容通常包括：构建和测试命令、代码风格约定、提交前检查清单、项目特有的注意事项（gotchas）。它被 check in 到 Git 仓库，版本可追溯。

## 记忆机制：运作方式对比

Claude Memory 和 CLAUDE.md 的底层机制完全不同，理解这个差异是正确使用它们的关键。

**Claude Memory 是隐式的。** 在对话过程中，Claude 会自主判断哪些信息值得长期记住，然后静默保存。你说「我是数据科学家，主要用 Python」，Claude 可能会自动把这条记为记忆。下次你开新对话时，Claude 已经知道你的背景。这种机制的优势是零摩擦，但缺点也很明显：你不完全控制 Claude 记住了什么、遗忘了什么。

**CLAUDE.md 是显式的。** 每一行规则都是开发者有意写下的。它不会自动生成（虽然你可以让 Claude 帮你起草），也不会自动更新。这意味着更高的维护成本，但也意味着更高的确定性——你精确知道 AI 读到了什么指令。

在 Claude Code 中，还有一个 **Auto Memory** 机制——它会将对话中学到的信息自动保存到 `~/.claude/` 目录下的文件中。这实际上是 Memory 和 CLAUDE.md 之间的中间地带：自动写入，但存储在本地文件系统。详细机制可参考 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 适用场景：什么时候用 Memory

Claude Memory 的价值在需要**跨项目、跨会话的个人一致性**时最为明显。

**个人工作偏好。** 你喜欢简洁回答还是详细解释？代码里要不要加注释？commit message 用英文还是中文？这些偏好不属于任何一个项目，而是属于「你」。Memory 是存放这类信息的正确位置。

**背景知识传递。** 如果你是一位前端工程师正在学习 Rust，你不需要在每次对话开始时都解释自己的技术背景。Memory 会记住你的知识结构，让 Claude 自动调整解释的深度和角度。

**跨产品连续性。** Memory 在 Claude.ai 网页端、Claude Desktop 和 Claude Code 之间共享。如果你在网页端告诉 Claude 你的偏好，切换到 Claude Code 时这些偏好依然生效。

**不适合用 Memory 的场景：** 项目特定的构建命令、团队代码规范、架构决策——这些应该进 CLAUDE.md，因为它们是项目属性，不是个人属性。

## 适用场景：什么时候用 CLAUDE.md

CLAUDE.md 的价值在需要**项目级的确定性和团队一致性**时最为明显。

**团队协作。** 当五个工程师同时使用 Claude Code 开发同一个项目时，你需要所有人的 AI 行为一致。CLAUDE.md 随仓库走，每个人 clone 下来就自动获得相同的规则。没有人需要手动配置自己的 Memory。

**质量守卫。** CLAUDE.md 可以定义提交前必须通过的检查清单——先跑 `npm test`、再跑 `npm run build`、不允许跳过失败的测试。这种硬性约束不适合放在 Memory 里，因为 Memory 的记忆可能被遗忘或覆盖。

**项目特有知识。** 每个项目都有自己的坑：这个 API 必须传三个参数、那个模块不能直接 import Next.js 服务端代码、中文内容必须用 CJK 分词计数。这些 gotchas 写在 CLAUDE.md 里，能防止 AI（和新成员）反复踩坑。

**可审计性。** CLAUDE.md 的每次修改都有 Git commit 记录。如果 AI 行为突然变了，你可以 `git blame` 找到是谁改了规则。Memory 没有这种追溯能力。

结合 [Claude Code 的 Skills 系统](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)，CLAUDE.md 可以进一步分层——高层级规则放在 CLAUDE.md，具体任务的详细指令放在 `skills/*/SKILL.md` 文件里。

## 协同使用：推荐的工程实践

在实际工程中，Memory 和 CLAUDE.md 不是二选一的关系，而是两个不同层次的配置。

**推荐分层策略：**

1. **CLAUDE.md（项目层）**：构建命令、代码风格、提交规范、架构约束、已知 gotchas
2. **SKILL.md（任务层）**：特定任务的详细指令，如写测试、做 code review、生成内容
3. **Auto Memory（个人项目层）**：你在这个项目中的工作习惯和发现，自动积累
4. **Claude Memory（全局个人层）**：你的角色、偏好、沟通风格

**优先级从高到低：** 当 Memory 和 CLAUDE.md 冲突时，CLAUDE.md 的指令优先。例如，你的 Memory 可能记住了「用户喜欢简洁的 commit message」，但 CLAUDE.md 里写着「commit message 必须包含 JIRA ticket 编号」——后者会覆盖前者。

**实际操作建议：**

- 新建项目时，先花 10 分钟写好 CLAUDE.md，把构建命令和核心约束写清楚
- 不要试图在 CLAUDE.md 里写个人偏好——那些自然会进入 Memory
- 定期检查 Memory 列表，删除过时的条目
- CLAUDE.md 随项目演进持续更新，每次改架构或加约束时同步更新

更多关于如何编写高效 Claude Code 指令的实践，可以参考 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)。

## 结论

**如果你只用 Claude.ai 聊天，Memory 已经足够**——它自动记住你的偏好，零配置成本。**如果你用 Claude Code 做工程开发，CLAUDE.md 是必需品**——没有它，AI 不了解你的项目规范，输出质量不可控。**对于认真使用 Claude Code 的开发者，两者同时使用是最佳实践**：CLAUDE.md 管项目规则，Memory 管个人偏好，Auto Memory 自动积累项目经验。

关键决策点：信息属于「项目」还是「个人」？属于项目的进 CLAUDE.md，属于个人的让 Memory 处理。想深入了解 Claude Code 的完整记忆体系，推荐阅读 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 常见问题

### Claude Memory 和 CLAUDE.md 会冲突吗？

当两者包含矛盾指令时，CLAUDE.md 的项目级规则优先于 Memory 中的个人偏好。例如，Memory 记住你偏好 JavaScript，但 CLAUDE.md 指定项目使用 TypeScript，Claude Code 会遵循 CLAUDE.md。这种优先级设计是有意为之——项目规范应当覆盖个人习惯。

### 不用 Claude Code，只用 Claude.ai，需要关心 CLAUDE.md 吗？

不需要。CLAUDE.md 是 Claude Code 特有的机制，只在终端环境中生效。如果你只使用 Claude.ai 网页端或 Claude Desktop 进行对话，Memory 是你唯一需要关注的持久化机制。但如果你未来计划使用 Claude Code，提前了解 CLAUDE.md 的设计理念会有帮助。

### Auto Memory 算 Memory 还是算 CLAUDE.md？

Auto Memory 是两者的中间地带。它像 Memory 一样自动生成，但像 CLAUDE.md 一样以本地文件形式存储在 `~/.claude/` 目录下。它的作用范围是项目级的（绑定到特定工作目录），优先级介于全局 Memory 和 CLAUDE.md 之间。可以把它理解为「Claude Code 自动写的项目笔记」。

### 团队里每个人的 Memory 不同，会不会导致 AI 行为不一致？

会，但这正是 CLAUDE.md 存在的原因。团队共享的规范（代码风格、提交流程、架构约束）应当写在 CLAUDE.md 里，确保每个人的 Claude Code 行为一致。个人 Memory 里的差异（沟通偏好、解释详细程度）不影响代码产出质量，反而让每个人获得定制化的交互体验。
