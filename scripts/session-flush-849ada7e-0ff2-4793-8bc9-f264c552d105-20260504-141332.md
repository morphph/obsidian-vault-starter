**CLAUDE.md** is loaded deterministically. At the start of every session, Claude Code reads the global `~/.claude/CLAUDE.md`, then the project-level `CLAUDE.md`, then any subdirectory-scoped `CLAUDE.md` files relevant to the current working path. This content becomes part of Claude's system instructions — it's always active, always applied. There's no relevance filtering. If you write "never use `var` in JavaScript" in your project CLAUDE.md, that rule applies to every session, every contributor, every task.

**Claude Memory** is loaded contextually. The `MEMORY.md` index is always read, but individual memory files are accessed based on relevance to the current task. Each memory file includes a `description` field in its frontmatter specifically to help Claude decide whether to load it. A memory about your team's deploy freeze won't be surfaced during a routine refactoring session unless it's relevant.

This creates a critical practical distinction: CLAUDE.md is for **rules you never want skipped**, while memory is for **context that's sometimes useful**. Putting a critical build command in memory risks it being ignored when Claude judges it irrelevant. Putting your personal formatting preferences in CLAUDE.md forces them on every team member.

The storage formats also differ meaningfully. CLAUDE.md is freeform markdown — write whatever structure makes sense for your project. Memory files use typed frontmatter (`user`, `feedback`, `project`, `reference`) that helps Claude categorize and retrieve information. The `feedback` type, for instance, records corrections you've made to Claude's behavior, complete with reasoning. This structured approach means Claude can learn from past mistakes systematically rather than relying on a single flat config file.

As described in the [Claude Code complete guide](/blog/claude-code-complete-guide), these two systems form complementary layers: CLAUDE.md sets the baseline rules, and memory personalizes the experience on top.

## Team Collaboration and Sharing: Detailed Analysis

The collaboration story is where CLAUDE.md and Claude Memory diverge most sharply, and where choosing the wrong system causes the most friction.

**CLAUDE.md lives in git.** When you commit a CLAUDE.md file, every team member who clones the repo gets the same AI behavior. This is essential for engineering standards: test requirements, build commands, naming conventions, forbidden patterns. A new team member's first Claude Code session behaves identically to a veteran's — at least for project-level rules. Changes to CLAUDE.md go through code review, which means the team can discuss and approve modifications to AI behavior just like they'd review any other config change.

**Claude Memory is inherently personal.** Your memory files live in your local `.claude/` directory (within a project-specific subdirectory) and are not shared. Your preference for verbose explanations doesn't affect your colleague who wants terse output. Your memory that "the payments module is fragile and needs extra test coverage" reflects your experience, not a team-wide rule.

This separation is a feature, not a limitation. Consider what happens when you collapse them:

- If you put personal preferences in CLAUDE.md, you force your style on the team. "Always explain changes in detail" is your preference — your senior colleague who's been on the project for three years doesn't need explanations.
- If you put project rules in memory, new team members don't get them. "Run `npm test` before every commit" in your memory means the new hire's Claude Code session happily commits untested code.

The decision rule is straightforward: **if a new team member needs this information on day one, it belongs in CLAUDE.md.** If it's specific to how you personally work with Claude, it belongs in memory.

For teams adopting Claude Code at scale, this distinction becomes even more important. Our coverage of [Claude Code in enterprise engineering at Ramp, Shopify, and Spotify](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify) shows how larger organizations standardize CLAUDE.md files across repositories while letting individual engineers accumulate personal memory organically.

## Content That Belongs in Each System: Detailed Analysis

The most practical question isn't "what can each system do?" but "where does this specific piece of information belong?" Here's a concrete classification framework.

### What belongs in CLAUDE.md

- **Build and test commands**: `npm run build`, `npm test`, `cargo clippy` — anything Claude needs to validate changes
- **Quality gates**: "All tests must pass before commit," "Run linting before push"
- **Coding standards**: Naming conventions, import ordering, forbidden patterns
- **Architecture constraints**: "Never import Next.js modules in pipeline scripts," "Use snake_case for DB columns"
- **Workflow rules**: "Discuss design before implementing new features," "Create a new branch for each feature"
- **Project-specific gotchas**: Known footguns that affect anyone working on the codebase
- **Documentation requirements**: What to update when certain files change

### What belongs in Claude Memory

- **Your role and expertise**: "Senior backend engineer, new to the React frontend" — so Claude calibrates explanations
- **Behavioral corrections**: "Don't summarize changes at the end of responses" — your personal style preference
- **Project context not in code**: "The auth rewrite is driven by legal compliance, not tech debt" — motivation that explains priorities
- **External resource locations**: "Pipeline bugs tracked in Linear project INGEST" — pointers Claude can't derive from code
- **Temporal project state**: "Merge freeze starts May 10 for mobile release" — time-bound context

### What belongs in neither

- **Code patterns and architecture**: Claude can read the code directly
- **Git history**: `git log` and `git blame` are authoritative
- **Fix recipes**: The fix is in the committed code; the commit message has the context

One way to stress-test your placement: imagine deleting all memory files. Would the project break? If yes, that information belongs in CLAUDE.md. Now imagine a new team member starting tomorrow. Would they be confused without this context? If it's personal to you, it's memory. If it's universal, it's CLAUDE.md.

## Maintenance and Lifecycle

**CLAUDE.md requires deliberate maintenance.** When your build system changes, someone needs to update the CLAUDE.md. When you adopt a new testing framework, the test commands need updating. This is the same maintenance burden as any documentation — and carries the same risk of drift. Stale CLAUDE.md instructions are worse than no instructions, because Claude follows them confidently. A CLAUDE.md that says "run `yarn test`" when the project migrated to `pnpm test` will cause silent failures.

**Claude Memory is self-maintaining — mostly.** Claude updates and creates memory files during normal conversations. When you correct Claude's behavior, it saves the correction as a `feedback` memory. When you mention a deadline, it saves a `project` memory. But self-maintenance has limits: Claude won't remove a memory about a deadline that passed, or update a memory about a team member who left. Memory files include a staleness warning in the system — Claude is instructed to verify memories against current state before acting on them — but this verification isn't guaranteed.

The practical advice: **review CLAUDE.md quarterly**, just like you'd review any project documentation. For memory, periodically scan `MEMORY.md` and remove entries that are clearly outdated. Both systems degrade without maintenance, but they degrade differently: stale CLAUDE.md actively misleads, while stale memory passively clutters.

## When to Choose Claude Memory

Choose Claude Memory as your primary context mechanism when:

- **You're a solo developer** who wants Claude to learn your preferences over time without manually configuring anything
- **You need to record corrections**: Every time you say "don't do that," Claude should remember — `feedback` memories handle this automatically
- **You're tracking external resources**: Slack channels, Linear projects, Grafana dashboards, and other pointers that don't fit in a project config file
- **You're onboarding to a new codebase** and want Claude to accumulate your growing understanding of the system — what's fragile, what's deprecated, who owns what
- **Context is personal**: Your role, your expertise level, your communication preferences

Memory shines for the kind of context that makes Claude feel like it knows you. After a few sessions, Claude remembers that you're a data scientist investigating logging, that you prefer single bundled PRs for refactors, and that you want explanations framed in terms of backend analogues when discussing frontend code. No config file can replicate this.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary context mechanism when:

- **You're working on a team** and need consistent AI behavior across all contributors
- **You have non-negotiable rules**: Build gates, test requirements, deployment procedures — anything that must be enforced every session
- **Your project has known gotchas** that would bite any developer, not just you: fragile modules, forbidden import patterns, CJK word-counting requirements
- **You want auditability**: Changes to AI behavior should go through code review
- **You need hierarchical scoping**: Global rules in `~/.claude/CLAUDE.md`, project rules in the repo root, path-specific rules in subdirectories

CLAUDE.md is the backbone. It's the first thing to set up when adopting Claude Code on a project, and the most impactful single file you can write. As detailed in our guide on [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code), a well-structured CLAUDE.md eliminates entire categories of AI mistakes by stating constraints up front rather than correcting them after the fact.

## Using Both Together

The strongest setup uses both systems in their respective strengths. Here's a concrete workflow:

1. **Start with CLAUDE.md**: Define your build commands, test requirements, coding standards, and known gotchas. Commit it to the repo. This is your team's shared contract with the AI.

2. **Let memory accumulate naturally**: As you work, Claude learns your preferences. You'll notice it adapting to your style — shorter responses, different explanation depth, awareness of your role.

3. **Promote memories to CLAUDE.md when they're universal**: If you keep correcting Claude about the same thing, and the correction applies to everyone, move that rule from your personal memory to the shared CLAUDE.md. A correction that you needed is likely one your teammates need too.

4. **Keep memory for the personal layer**: Your expertise level, your preferred verbosity, your awareness of external resources — these stay in memory because they're yours.

This layered approach is what the [Claude Code skills guide](/blog/5-claude-code-skills-i-use-every-single-day) recommends: CLAUDE.md for the project baseline, skills for task-specific workflows, and memory for personal calibration. Each layer has a different scope, different authorship, and different maintenance cadence.

## Verdict

**CLAUDE.md and Claude Memory are complementary, not competing.** If you're only going to set up one, **start with CLAUDE.md** — it has the highest immediate impact because it prevents the most common AI mistakes from the first session. Memory adds value over time but requires no upfront investment.

The decision framework is simple: **shared rules go in CLAUDE.md, personal context goes in memory.** If you catch yourself putting team conventions in memory or personal preferences in CLAUDE.md, you're creating friction — either for your teammates or for yourself. Get the boundary right, and both systems work better.

For teams scaling Claude Code across multiple repositories, standardize your CLAUDE.md structure first, then let individual memory accumulate. The combination of deterministic project rules and adaptive personal context is what makes Claude Code feel like a capable teammate rather than a stateless tool. ## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md instructions are loaded as system-level rules and always apply. Claude Memory provides additional context that informs Claude's behavior, but it cannot contradict or override explicit CLAUDE.md rules. If your CLAUDE.md says "always run tests before committing," a memory preference for skipping tests won't take effect.

### Does Claude Memory sync across machines?

Claude Memory files are stored locally in the `.claude/` directory within your project path. They do not automatically sync across machines. If you work from multiple devices, your memory context will differ between them. CLAUDE.md, being committed to git, syncs wherever you clone the repository.

### How much can I store in Claude Memory before it degrades?

The `MEMORY.md` index file is loaded into every conversation, and lines after 200 are truncated. Individual memory files are loaded on-demand based on relevance. Practically, keeping the index under 150 entries with concise one-line descriptions ensures Claude can efficiently find relevant context. Overstuffing memory with redundant or low-value entries dilutes retrieval quality.

### Should I commit my Claude Memory files to the repository?

Generally no. Memory files contain personal preferences, role-specific context, and individual corrections that are scoped to you. Committing them would impose your personal context on the entire team. Keep memory local and use CLAUDE.md for anything the team should share.

### Can I convert a Claude Memory entry into a CLAUDE.md rule?

Yes, and you should do this when a personal correction turns out to be a universal rule. If you've saved a feedback memory like "always use `pnpm` instead of `npm` in this repo," and that applies to everyone, move it to CLAUDE.md and delete the memory file. This promotes tested, validated rules into the shared configuration.
