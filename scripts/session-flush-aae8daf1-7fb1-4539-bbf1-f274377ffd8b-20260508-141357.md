**CLAUDE.md is a team resource.** When you add a rule to CLAUDE.md, every developer who clones the repo and runs Claude Code inherits that rule. This makes it the right place for anything that should be consistent across the team: build commands, testing requirements, architectural constraints, style guidelines, and known gotchas. A new team member's first Claude Code session behaves identically to a veteran's — at least for project-level instructions.

This shared nature also means CLAUDE.md goes through code review. Adding "never use `any` types in TypeScript" to CLAUDE.md is a team decision, not a personal preference. It gets discussed in the PR, approved by reviewers, and versioned alongside the code it governs. For teams adopting [Claude Code at scale](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify), this review process is critical for maintaining consistent AI behavior.

**Memory is a personal resource.** Your memories are yours alone. They reflect your expertise level ("senior Go developer, new to React"), your communication preferences ("no trailing summaries"), your working context ("investigating the auth migration for compliance reasons"), and your corrections ("don't mock databases in integration tests — we got burned last quarter"). These are things that make Claude Code more effective for you specifically, but that wouldn't make sense as rules imposed on the whole team.

This personal nature means Memory can store information that would be inappropriate or too noisy for CLAUDE.md. Your team doesn't need to know that you prefer verbose error messages during debugging, or that you're the one responsible for the SEO pipeline this sprint. But Claude knowing these things makes every session more productive.

**Decision rule:** If removing the information would hurt a teammate's Claude Code experience, it belongs in CLAUDE.md. If it only affects your own experience, it belongs in Memory.

## Content Types: What Goes Where

Choosing between Memory and CLAUDE.md comes down to the nature of the information. Here's a practical breakdown.

### Belongs in CLAUDE.md

- **Build and test commands**: `npm run build`, `npm test`, `npm run lint` — anything the CI pipeline also runs
- **Quality gates**: "All tests must pass before commit," "Run validate-pipeline.ts for pipeline changes"
- **Coding standards**: Naming conventions, import rules, forbidden patterns
- **Architecture constraints**: "Don't import Next.js modules in pipeline scripts," "Use CJK word count for Chinese content"
- **Known gotchas**: Technical pitfalls specific to the codebase that any developer would hit
- **Documentation rules**: When and how to update docs alongside code changes
- **Git workflow requirements**: Commit message format, branching strategy, review requirements

These are all things that are true regardless of who's working on the project. They're stable, reviewable, and shared.

### Belongs in Memory

- **Your role and expertise**: "Data scientist investigating logging" or "Backend engineer, first time touching the React frontend"
- **Communication preferences**: Response length, explanation depth, summary style
- **Feedback corrections**: Approaches Claude tried that you rejected, with the reason why
- **Confirmed approaches**: Non-obvious choices Claude made that you validated — so it doesn't second-guess them later
- **Project state**: Who's working on what, upcoming deadlines, ongoing incidents — information that changes weekly
- **External references**: Links to Linear projects, Slack channels, Grafana dashboards relevant to your work
- **Working context**: Why you're making certain changes, what you're investigating, what you've already ruled out

These are all personal, temporal, or contextual. They'd clutter CLAUDE.md and wouldn't apply to other team members.

### The Gray Zone

Some information could reasonably go in either place. The tiebreaker is always: **does the team need this, or just you?**

- "We use Vitest, not Jest" → CLAUDE.md (team standard)
- "I prefer seeing test output in verbose mode" → Memory (personal preference)
- "The auth migration is driven by legal compliance" → Memory (project context that's temporary)
- "Auth module must never store session tokens in cookies" → CLAUDE.md (permanent architectural constraint)

## Creation and Maintenance: How Each System Evolves

**CLAUDE.md requires deliberate authorship.** Someone on the team writes the initial file — either manually or via Claude Code's `/init` command — and maintains it as the codebase evolves. It's a living document that should be updated alongside code changes. The best CLAUDE.md files are treated like code: they go through review, they're kept current, and they're deleted when they become irrelevant.

Writing an effective CLAUDE.md is a [skill in itself](/blog/how-to-effectively-prompt-a-claude-code). The best ones are specific and actionable — "run `npm run build` before committing" beats "make sure the code compiles." They focus on what Claude can't infer from the code alone: the why behind conventions, the history behind constraints, the priorities behind tradeoffs.

**Memory is conversational and automatic.** You don't sit down and write memory files. Instead, Claude saves memories as a natural byproduct of your conversations. When you correct Claude's behavior, it saves feedback memory. When you mention your role or responsibilities, it saves user memory. When you discuss project deadlines or decisions, it saves project memory. You can also explicitly ask Claude to remember something, and it will categorize and store it appropriately.

Memory maintenance is lighter but still important. Memories can become stale — a project context memory about "the auth migration" is useless six months after the migration shipped. Claude is instructed to verify memories against current state before acting on them, but periodic cleanup prevents context bloat. You can ask Claude to review and prune outdated memories, or manually edit the files in `~/.claude/projects/`.

**SKILL.md files** — which sit alongside CLAUDE.md in the repo — are a related but distinct mechanism. They encode reusable task-specific instructions (how to write tests, how to generate content) rather than project-wide rules. ## When to Invest in CLAUDE.md

Invest in a thorough CLAUDE.md when:

- **Multiple developers use Claude Code on the same repo.** Without shared instructions, each person's Claude Code session may follow different conventions, produce inconsistent code style, or miss critical quality gates. CLAUDE.md is the single source of truth.
- **Your codebase has non-obvious constraints.** If a new developer would step on a landmine without tribal knowledge — "don't use `any` types," "this legacy module can't be refactored yet," "the build requires Node 20" — encode it in CLAUDE.md.
- **You want reproducible AI behavior.** CLAUDE.md makes Claude Code's behavior deterministic for a given project. Two developers running the same task get the same quality gates, the same conventions, the same guardrails. This matters for [enterprise adoption](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify).
- **You're onboarding new team members.** A well-maintained CLAUDE.md dramatically reduces the ramp-up time for new developers using Claude Code. They get the team's collective wisdom on day one.
- **You want AI behavior to be reviewable.** Since CLAUDE.md is version-controlled, changes to AI instructions go through the same review process as code changes. You can see exactly when a rule was added, why, and by whom.

## When to Invest in Memory

Invest in building up Memory when:

- **You work on the same project repeatedly.** Memory accumulates context across sessions, so your tenth conversation about the auth module starts with everything Claude learned in the first nine. This eliminates repetitive setup.
- **You have strong personal preferences.** If you consistently correct Claude's output style, explanation depth, or approach to certain tasks, Memory captures those corrections so you only give feedback once.
- **You're tracking evolving project state.** Sprint goals, incident investigations, migration progress, team responsibilities — these change too frequently for CLAUDE.md but are valuable context for your sessions.
- **You work across multiple projects.** User-type memories (your role, expertise, communication style) apply globally, making Claude Code effective across every repo you touch.
- **You use external tools that Claude should know about.** Reference memories that point to your team's Linear board, Slack channels, or monitoring dashboards help Claude give contextually relevant suggestions.

## Using Both Systems Together

The most effective Claude Code setup uses both systems strategically. Here's a practical workflow:

1. **Start with CLAUDE.md** when setting up a new project. Run `/init` to generate a baseline, then customize with your team's build commands, quality gates, and conventions. This is your foundation.

2. **Let Memory build naturally** as you work. Don't try to pre-populate Memory — let it emerge from real conversations. When you correct Claude, it saves feedback. When you share context about your role or the project, it saves that too.

3. **Promote stable memories to CLAUDE.md** when you notice patterns. If you keep giving the same feedback in Memory ("always run integration tests, never mock the DB"), that's a signal it should be a project-wide rule in CLAUDE.md instead.

4. **Keep Memory lean** by periodically reviewing and pruning stale entries. A memory about last quarter's incident is noise once the fix has shipped.

5. **Use CLAUDE.md hierarchically** for subsystem-specific rules. A `scripts/CLAUDE.md` with pipeline-specific constraints keeps the root file clean while still providing targeted instructions.

## Common Mistakes

**Putting personal preferences in CLAUDE.md.** "I like verbose output" doesn't belong in a shared project file. Use Memory for individual preferences.

**Putting critical team rules only in Memory.** If "always run `npm test` before commit" is only in your Memory, your teammates' Claude Code sessions won't follow that rule. Shared rules go in CLAUDE.md.

**Duplicating information across both systems.** Memory is explicitly designed not to store information already in CLAUDE.md. If your CLAUDE.md says "use Tailwind v4," don't also tell Claude to remember that.

**Letting CLAUDE.md grow unchecked.** A 500-line CLAUDE.md with outdated rules is worse than no CLAUDE.md. Keep it maintained. Remove rules that no longer apply. Update commands when tooling changes.

**Never reviewing Memory.** Stale memories can cause Claude to apply outdated context. Review periodically — ask Claude "what do you remember about this project?" and correct anything that's no longer true.

## Verdict

**CLAUDE.md and Claude Memory are complementary, not competing.** CLAUDE.md is your project's constitution — shared rules that every Claude Code session must follow, version-controlled and reviewable. Memory is your personal assistant's notebook — accumulated context that makes each session smarter without imposing on the team.

**Start with CLAUDE.md.** It has the highest leverage: one file improves Claude Code for every developer on the project. Get your build commands, quality gates, and critical constraints documented there first.

**Let Memory grow organically.** Don't pre-populate it. Just use Claude Code naturally, give corrections when needed, and let the system accumulate context over time. The value compounds with each conversation.

If you're working solo, both systems matter equally. If you're on a team, CLAUDE.md matters more — it's the shared foundation that Memory builds on top of. ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions are treated as directives that override default behavior. Memory provides supplemental context but cannot contradict explicit CLAUDE.md rules. If CLAUDE.md says "use Jest for testing" and a memory says "user prefers Vitest," CLAUDE.md wins for that project.

### Does Claude Memory sync across machines?

No. Memory files are stored locally in `~/.claude/projects/` and do not sync between machines or between team members. If you use Claude Code on multiple machines, each maintains its own separate memory. CLAUDE.md, being version-controlled, syncs automatically through git.

### How much should I put in CLAUDE.md?

Keep it focused on actionable instructions. A good CLAUDE.md is 50-150 lines covering build commands, quality gates, key conventions, and critical gotchas. If it exceeds 200 lines, consider splitting subsystem-specific rules into directory-level CLAUDE.md files instead of bloating the root file.

### Can I export or share my Claude Memory with teammates?

Memory files are plain markdown, so you could technically copy them. But this defeats the purpose — Memory is personal context that reflects your specific expertise, preferences, and corrections. If information in your Memory would benefit the team, promote it to CLAUDE.md instead.

### What happens if I delete CLAUDE.md?

Claude Code continues to work but loses all project-specific instructions. It falls back to default behavior — which means no custom build commands, no quality gates, no convention enforcement. For projects that relied on CLAUDE.md for consistency, this effectively resets Claude Code to a generic state.
