**Codex uses a cloud-first, container-per-task model.** When you assign a task to a Codex agent, the platform provisions an isolated environment with your repository, installs dependencies from your lock files, and executes the agent's work. The agent cannot access your local filesystem, other running tasks, or any state outside its container. This is a deliberate security and reliability choice: tasks cannot interfere with each other, and a failing agent cannot corrupt your working directory.

The tradeoff is latency and context loss. Each Codex task starts cold — cloning the repo, installing packages, building the project. There is no shared memory between tasks. If Task A discovers that a function signature changed and Task B needs to know, there is no built-in mechanism for that communication. You manage coordination through the repository itself (merging Task A's PR before Task B starts) or through your own orchestration.

**Claude Code uses a local-first, shared-context model.** Sub-agents run as child processes within your active Claude Code session. They inherit the conversation context you provide in their prompt, access your local filesystem directly, and return results that the parent session can act on immediately. When you spawn an Explore agent to find all usages of a function, the results flow back into your working context within seconds — no PR workflow, no async wait.

Claude Code's [worktree isolation](/blog/anatomy-of-git-worktree-add) provides an optional middle ground: agents work on a separate git branch in a linked working tree, so their file changes don't affect your main checkout until you choose to merge. This gives you container-like isolation without the cold-start penalty.

For teams evaluating these tools, the architecture choice maps directly to workflow style. Codex suits organizations that want to queue up dozens of independent tasks — bug fixes, test generation, documentation updates — and review the results as PRs. Claude Code suits developers who want to break a complex task into parallel subtasks within a single interactive session and synthesize the results on the spot.

## Custom Agent Configuration: Detailed Analysis

How you define and customize agent behavior differs significantly between the two platforms, affecting both the setup cost and the precision of control you get.

**Codex custom agents** are configured primarily through markdown instruction files in your repository. You define the agent's role, constraints, and conventions in a structured format that Codex reads before starting each task. Environment configuration — which dependencies to install, which build commands to run, which directories to focus on — is set through the Codex platform UI or API. The system is straightforward: write the instructions, and every task assigned to that agent profile follows them.

The limitation is granularity. Codex agents operate at the task level — you configure what kind of work an agent does, but you have limited control over how it decomposes that work internally. If a Codex agent decides to split a refactoring task into multiple steps, you see the final result, not the intermediate reasoning.

**Claude Code's configuration is layered and composable.** At the project level, `CLAUDE.md` sets global constraints — coding standards, forbidden patterns, quality gates. At the task level, [SKILL.md files](/blog/5-claude-code-skills-i-use-every-single-day) define specific workflows: how to write a newsletter, how to review a PR, how to generate tests. At the agent level, you choose from typed sub-agents (Explore, Plan, general-purpose) and pass them custom prompts that combine project context with task-specific instructions.

This layered system means you can create highly specialized agent compositions. For example, you might spawn a Plan agent to design a refactoring strategy, then launch three general-purpose agents in parallel to implement different parts of that plan, each working in its own worktree. The [9 principles for writing effective skills](/blog/9-principles-writing-claude-code-skills-i-use-every-single-day) apply directly to crafting sub-agent prompts — they are, effectively, the same mechanism.

The tradeoff is complexity. Codex's "write instructions, submit task" model has a lower learning curve. Claude Code's layered system offers more power but requires understanding how CLAUDE.md, SKILL.md, agent types, and prompts interact. Teams that invest in building a [robust extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) see significant returns; teams that just want to throw tasks at an AI may find Codex's simplicity more appealing.

## Execution Model and Parallelism: Detailed Analysis

Both platforms support parallel execution, but the mechanics and practical limits differ in ways that matter for real workloads.

**Codex parallelism is horizontal and cloud-scaled.** You can submit dozens of tasks simultaneously, each running in its own container. This is genuine parallelism — no shared compute, no resource contention between tasks. For organizations processing a backlog of issues or running Codex across multiple repositories, this scales cleanly. The [Codex for open source](/blog/codex-for-open-source) program demonstrates this at scale, where maintainers use Codex to process issue backlogs in parallel.

**Claude Code parallelism is vertical and session-scoped.** Within a single Claude Code session, you can spawn multiple sub-agents that run concurrently. Independent agents execute in parallel (multiple tool calls in one response), while dependent agents run sequentially. Background agents run without blocking your main conversation — you get a notification when they complete.

The practical difference: Codex parallelism is better for breadth (many independent tasks across a project), while Claude Code parallelism is better for depth (decomposing one complex task into coordinated subtasks). A Codex workflow might be "fix these 15 issues simultaneously." A Claude Code workflow might be "research this bug from three angles in parallel, then synthesize the findings and implement the fix."

## When to Choose OpenAI Codex

Codex is the right choice when your workflow matches its async, cloud-native model:

- **Team-scale task processing**: You have a queue of well-defined tasks — bug fixes, test additions, documentation updates — and want to process them in parallel without tying up developer machines. Codex's GitHub integration means tasks flow naturally from issues to PRs.
- **Fire-and-forget jobs**: Tasks where you do not need to interact with the agent mid-execution. Submit the task, review the result. The async model works well when the tasks are independent and self-contained.
- **Strong isolation requirements**: Regulated environments where agent work must be sandboxed and auditable. Each Codex task produces a clean diff from a known starting state — no local filesystem side effects, no ambient environment dependencies.
- **Non-technical stakeholders**: Product managers or designers who want to submit coding tasks through GitHub issues without touching a terminal. Codex's web and [VS Code interfaces](/blog/codex-vscode) lower the barrier.

Codex is less ideal for interactive debugging, exploratory refactoring, or tasks that require real-time context sharing between multiple agents. If you find yourself submitting a task, waiting for the result, then submitting a follow-up task based on what you learned, the async round-trip adds friction that Claude Code's interactive model avoids.

## When to Choose Claude Code

Claude Code is the right choice when you need tight control and interactive orchestration:

- **Complex multi-step tasks**: Refactoring a module where you need to research the current implementation, plan the new architecture, implement changes across multiple files, and update tests — all within one session. Sub-agents handle the research and implementation in parallel while you orchestrate.
- **Exploratory work**: You are not sure what the right approach is yet. Spawn an Explore agent to search the codebase, a Plan agent to propose strategies, and iterate based on what they find. This interactive loop is not possible with Codex's async model.
- **Context-heavy workflows**: Tasks where the agent needs deep understanding of recent conversation context, in-progress changes, or uncommitted local state. Claude Code sub-agents inherit this context; Codex agents start from the committed repository state only.
- **Skill-intensive pipelines**: If you have invested in building [SKILL.md files](/blog/do-skills-actually-improve-your-agents-output) and want sub-agents to follow the same conventions, Claude Code's layered configuration gives you that control. Each sub-agent can be briefed with specific skills and constraints.

Claude Code is less ideal for teams that want to distribute many independent tasks without a developer actively orchestrating them. Its local-first model means someone is at the terminal, directing the work. For a deeper look at how this fits into long-running workflows, see our coverage of [effective harnesses for long-running agents](/blog/effective-harnesses-for-long-running-agents).

## Pricing and Access

Pricing models reflect the architectural differences.

**Codex** is included in ChatGPT Pro ($200/month), Team ($30/user/month + usage), and Enterprise plans. Pro users get a generous allocation; Team and Enterprise pricing scales with usage. The cloud compute — container provisioning, dependency installation, execution — is included. For [students](/blog/codex-for-students), OpenAI offers $100 in free credits, though with caveats about usage limits.

**Claude Code** uses usage-based API billing — you pay per input and output token, with no fixed monthly fee for the tool itself (though a Max subscription or API key is required). Sub-agent calls consume additional tokens: each agent processes its own prompt and context. The cost scales with how many agents you spawn and how much context each one processes.

**Cost comparison by workflow type:**

- **10 independent bug fixes**: Codex is likely cheaper — one plan covers all tasks, parallel execution at no incremental compute cost to you. Claude Code would require 10 separate agent sessions with their own token budgets.
- **1 complex refactoring with 5 sub-agents**: Claude Code may be cheaper — sub-agents share context efficiently and avoid the overhead of full environment provisioning. Codex would require sequential tasks with cold-start costs.
- **Daily team usage across 10 developers**: Codex Team plan provides predictable per-seat pricing. Claude Code costs vary with individual usage patterns, making budgeting harder but potentially cheaper for light users.

## Combining Both Tools

The comparison does not have to be either-or. Several teams use both tools in complementary roles:

1. **Claude Code for design and orchestration** — use sub-agents to research, plan, and prototype locally
2. **Codex for execution at scale** — once the approach is validated, submit well-defined tasks to Codex for parallel processing
3. **Claude Code for review** — use sub-agents to review the PRs that Codex generates, checking for consistency and adherence to project conventions

Claude Code can even delegate directly to Codex through its [codex-rescue agent type](/blog/claude-code-subagents-examples), creating a hybrid workflow where the terminal agent orchestrates cloud-based execution.

## Verdict

**For interactive, context-rich multi-agent workflows, choose Claude Code.** Its sub-agent system gives you real-time orchestration, typed agents for different tasks, and layered configuration through CLAUDE.md and SKILL.md. If you are a developer who wants to decompose complex problems into parallel subtasks and synthesize results on the spot, Claude Code's architecture is purpose-built for that workflow.

**For team-scale, async task processing, choose Codex.** Its cloud-native model handles many independent tasks in parallel with strong isolation and GitHub-native integration. If your workflow is "assign tasks, review PRs," Codex fits cleanly.

Most teams doing serious [agentic coding](/glossary/agentic-coding) will benefit from understanding both models. The right tool depends on whether your bottleneck is task orchestration (Claude Code) or task throughput (Codex). ## Frequently Asked Questions

### Can Codex custom agents call other agents?

Codex tasks run independently in isolated containers and cannot spawn sub-agents or communicate with other running tasks. Coordination happens at the repository level — merge one task's PR before starting a dependent task — or through external orchestration that submits tasks sequentially via the Codex API.

### How many sub-agents can Claude Code run in parallel?

Claude Code can spawn multiple sub-agents in a single response, and they execute concurrently. There is no hard-coded limit on agent count, but practical limits come from token budgets and system resources. Most workflows use 2-5 parallel agents for tasks like simultaneous codebase research, parallel implementation across modules, or concurrent testing.

### Do Codex agents have access to my local files?

No. Codex agents run in cloud sandboxes with a clone of your repository at its latest committed state. They cannot access uncommitted local changes, local environment variables, or files outside the repository. This is a security feature but means you must commit and push changes before Codex can work with them.

### Can I use Claude Code sub-agents with the Codex CLI?

Yes. Claude Code includes a `codex-rescue` agent type that can delegate tasks to the Codex CLI if it is installed locally. This creates a hybrid workflow where Claude Code handles orchestration and context management while Codex handles specific execution tasks in its sandboxed environment.
