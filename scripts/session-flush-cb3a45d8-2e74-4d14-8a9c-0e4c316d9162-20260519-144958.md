# Codex Subagents vs Custom Agents：哪种多智能体模式更适合你？

**TL;DR：** Codex 的 **Subagents** 是系统自动拆分并行子任务的内置能力，适合大规模重构和跨文件变更；**Custom Agents** 是用户自定义的智能体配置，通过 agents.json 定义专属角色和工具链，适合团队协作和特定领域工作流。**如果你需要开箱即用的并行加速，选 Subagents；如果你需要可复用、可定制的工程角色，选 Custom Agents。** 两者并非互斥——最佳实践是组合使用。

## 概述：Codex Subagents

Codex Subagents 是 [OpenAI Codex](/zh/blog/codex-complete-guide) 平台内置的任务拆分与并行执行机制。当你向 Codex 提交一个复杂任务时，系统会自动将其分解为多个独立的子任务，每个子任务由一个独立的 Subagent 在隔离的沙箱环境中执行。

Subagents 的核心价值在于**自动并行化**。你不需要手动拆分工作——系统判断哪些文件变更可以独立进行，自动派发给不同的 Subagent，最后合并结果。这种模式在处理「给 20 个文件加上类型标注」或「批量迁移 API 调用」这类高并行度任务时效率显著。每个 Subagent 运行在独立的 Docker 容器中，拥有完整的代码库副本和工具链访问权限，但彼此隔离，不会产生冲突。

这是 Codex 区别于本地编码工具的关键能力——本地 CLI 工具受限于单机资源，而 Codex 的云端架构天然支持弹性扩展。

## 概述：Codex Custom Agents

Custom Agents 是 Codex 提供的用户自定义智能体系统。通过在项目根目录创建 `agents.json` 或 `.codex/agents/` 配置文件，你可以定义具有特定角色、工具权限和行为约束的专属 Agent。

Custom Agents 的核心价值在于**可定制性和可复用性**。你可以创建一个「安全审计 Agent」，限定它只能读取代码但不能执行写操作，并在 system prompt 中注入安全审计清单；也可以创建一个「测试生成 Agent」，预配置好测试框架和覆盖率目标。这些 Agent 定义跟随代码仓库，团队成员共享同一套 Agent 配置，确保一致性。

Custom Agents 的实质是将[工程规范编码为可执行的智能体角色](/zh/blog/9-principles-writing-claude-code-skills)——从「口头约定」变为「机器执行」。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **触发方式** | 系统自动拆分 | 用户显式配置和调用 | 各有所长 |
| **并行执行** | 原生支持，自动调度 | 需手动编排多 Agent 协作 | Subagents |
| **定制程度** | 低——系统决定任务拆分策略 | 高——自定义角色、权限、工具链 | Custom Agents |
| **隔离级别** | 独立沙箱容器 | 共享项目上下文 | Subagents |
| **团队协作** | 透明——无需额外配置 | 配置文件可版本控制和共享 | Custom Agents |
| **适用范围** | 批量变更、大规模重构 | 特定领域工作流、角色分工 | 各有所长 |
| **学习成本** | 低——自动生效 | 中——需要理解配置语法 | Subagents |
| **可预测性** | 较低——系统决策 | 高——用户完全控制 | Custom Agents |

## 并行执行能力：详细分析

Subagents 在并行执行方面具有结构性优势。当你提交一个涉及多文件的任务时，Codex 的任务规划器会分析文件依赖关系，将无冲突的变更分配给不同的 Subagent 并行处理。整个过程对用户透明——你提交一个 prompt，系统返回一个合并后的 PR，中间的拆分和合并逻辑完全自动化。

这种[多智能体工作流模式](/zh/blog/con-u-pour-des-workflows-multi-agents)在以下场景中表现突出：批量类型迁移（JavaScript 到 TypeScript）、全局 API 升级（v2 接口替换 v1）、代码风格统一（ESLint 规则批量修复）。实际使用中，涉及 10+ 文件的任务，Subagents 的并行执行通常能将完成时间压缩到串行执行的 30%-50%。

Custom Agents 也可以实现并行，但需要用户手动编排。你可以同时启动多个 Custom Agent 处理不同的任务分支，但协调和合并需要人工介入。这种显式控制在需要精确把控每步操作的场景（如安全敏感变更）中反而是优势。

关键区别：Subagents 的并行是**系统驱动**的——它决定怎么拆、拆多少；Custom Agents 的并行是**用户驱动**的——你决定谁做什么。

## 定制与工程规范：详细分析

Custom Agents 在定制化方面拥有压倒性优势。通过配置文件，你可以精确控制：

- **角色定义**：Agent 的身份、专长、行为边界
- **工具权限**：哪些 Agent 可以执行 shell 命令、哪些只能读取文件
- **上下文注入**：将团队编码规范、架构决策文档自动注入 Agent 的系统提示
- **输出约束**：要求特定格式的 commit message、PR 描述或代码注释

这种能力在团队场景中价值巨大。一个 10 人的工程团队可以定义统一的 Agent 角色集——代码审查 Agent、测试 Agent、文档 Agent——每个角色的行为标准化且可审计。新成员入职时，无需口头传授「我们团队怎么写代码」，Agent 配置本身就是活文档。

Subagents 在这方面能力有限。它的行为主要由 Codex 平台决定，用户只能通过 prompt 间接影响任务拆分策略。你可以在 [Codex VS Code 扩展](/zh/blog/codex-vscode)中引导 Subagent 的行为，但无法像 Custom Agents 那样做到精确的角色定义和权限隔离。

想深入了解如何将工程规范编码为 Agent 指令，可以参考[Agent 长时间运行的工程实践](/zh/blog/effective-harnesses-for-long-running-agents)。

## 什么时候选 Subagents

选择 Subagents 的典型场景：

- **大规模批量变更**：需要对数十个文件做相似的机械性修改，如类型迁移、API 升级、依赖替换
- **快速原型验证**：不想花时间配置 Agent，直接提交任务让系统自动处理
- **独立开发者**：个人项目中不需要复杂的角色分工，关注的是执行效率
- **探索性任务**：不确定最优的任务拆分方式，让系统智能决策

Subagents 的核心优势是**零配置成本**。对于[刚开始使用 Codex 的开发者](/zh/blog/first-few-days-with-codex-cli)，Subagents 提供了一个无门槛的并行执行体验——提交任务，等待结果，审查 PR。

## 什么时候选 Custom Agents

选择 Custom Agents 的典型场景：

- **团队协作**：需要统一的编码标准和 Agent 行为规范，确保不同成员使用一致的 AI 辅助流程
- **安全敏感项目**：需要精确控制 Agent 的权限边界，如限制对生产配置的访问
- **特定领域工作流**：有固定的工作模式（如每次 PR 必须包含测试、文档更新、changelog 条目），通过 Agent 自动执行
- **合规要求**：需要审计 Agent 的行为记录，确保 AI 操作可追溯

Custom Agents 的核心优势是**可预测性和可控性**。当你定义了一个 Agent 的角色和约束，它的行为模式是确定的——这在生产环境和团队协作中至关重要。

## 最佳实践：组合使用

实际工程中，Subagents 和 Custom Agents 并非二选一。推荐的模式是：

1. **用 Custom Agents 定义角色**——创建符合团队标准的专属 Agent 配置
2. **在 Custom Agent 内利用 Subagents 并行**——让自定义 Agent 处理需要并行的批量任务时，系统自动调用 Subagent 能力
3. **用 Custom Agents 做质量门禁**——在 Subagent 完成批量变更后，用专门的 Review Agent 审查结果

这种分层架构在[编码 Agent 重塑产品工程](/zh/blog/coding-agents-reshaping-epd)的趋势下尤其重要——AI 不只是写代码，更是执行团队定义的工程流程。

## 结论

**Subagents 是 Codex 的执行引擎，Custom Agents 是你的工程规范编码器。** 对于个人开发者和快速原型场景，Subagents 的自动并行能力开箱即用，足以覆盖大多数需求。对于团队协作和生产环境，Custom Agents 的定制能力和可预测性不可或缺。成熟的 Codex 使用者会将两者组合——用 Custom Agents 定义「做什么、怎么做」，让 Subagents 负责「快速做完」。

## 常见问题

### Subagents 需要额外付费吗？

Subagents 是 Codex 平台的内置能力，包含在标准 Codex 使用额度内。每个 Subagent 的执行会消耗独立的计算资源和 token 额度，但不需要单独购买。并行 Subagent 的数量取决于你的 Codex 计划等级。

### Custom Agents 配置文件的格式是什么？

Custom Agents 通过项目根目录的配置文件定义，支持 JSON 格式。配置包括角色名称、system prompt、工具权限列表和行为约束。配置文件可以纳入版本控制，团队成员克隆仓库后自动获得相同的 Agent 配置。

### 能否在本地 Codex CLI 中使用 Subagents？

Codex CLI 本地模式主要面向单任务执行，Subagents 的并行能力依赖云端基础设施。不过 CLI 支持 Custom Agents 配置的加载，你可以在本地使用自定义 Agent 角色处理任务。完整的 Subagent 并行体验需要通过 Codex 云端平台或 VS Code 扩展。

### Subagents 和 Custom Agents 可以同时使用吗？

可以。推荐的做法是用 Custom Agents 定义工作角色和行为规范，在需要并行处理批量任务时由系统自动调度 Subagents。两者在架构上是互补的，而非互斥的。
