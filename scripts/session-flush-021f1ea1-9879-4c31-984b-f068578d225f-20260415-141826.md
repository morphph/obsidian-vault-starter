# Claude Memory vs CLAUDE.md：两套持久化机制，到底该用哪个？

**TL;DR：** **CLAUDE.md** 是项目级的规则手册——团队共享、版本控制、确定性执行。**Claude Memory** 是个人级的自动笔记——跨会话记忆、自动积累、按需召回。两者不是替代关系，而是互补关系：**CLAUDE.md 管"怎么做"，Memory 管"记住什么"**。理解这个区别，是高效使用 [Claude Code](/zh/glossary/agentic-coding) 的关键。

## 概览：Claude Memory

Claude Memory 是 Claude Code 的自动记忆系统，存储在 `.claude/projects/` 目录下的 markdown 文件中。它的核心设计理念是：**让 AI 在不同会话之间保持上下文连续性**，而不需要用户每次都重复说明背景。

Memory 支持四种类型：用户记忆（user）、反馈记忆（feedback）、项目记忆（project）和引用记忆（reference）。每种类型有不同的触发条件和使用场景。例如，当你纠正 Claude 的某个行为时（"不要在测试里 mock 数据库"），它会自动保存为 feedback 类型的记忆，下次遇到类似场景时主动应用。

Memory 的存储结构是一个索引文件 `MEMORY.md` 加上若干独立的记忆文件，每个文件包含 frontmatter 元数据和具体内容。这套系统完全是**个人级别**的——它不进入版本控制，不与团队共享，只属于当前用户。

关于 Memory 系统的完整机制，可以参考我们的深度解析：[Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概览：CLAUDE.md

CLAUDE.md 是 Claude Code 的项目级指令文件，放在项目根目录（或 `.claude/` 目录）下，随代码一起提交到 Git 仓库。它的定位非常明确：**告诉 Claude 这个项目的规则、约束和工作方式**。

一个典型的 CLAUDE.md 包含：构建命令（`npm run build`）、测试命令（`npm test`）、代码风格约定、提交规范、架构决策、已知陷阱等。Claude Code 在每次会话开始时自动加载这些指令，确保所有操作都遵循项目标准。

CLAUDE.md 的核心优势是**确定性和可共享性**。因为它是普通的 markdown 文件，可以 code review、可以 diff、可以在团队成员之间保持一致。当你写下"提交前必须跑 `npm test`"，每个使用 Claude Code 的团队成员都会遵循同一条规则。

更进一步，Claude Code 还支持 `skills/*/SKILL.md` 文件来编码特定任务的执行标准，这是 CLAUDE.md 理念的延伸。详见我们的 Skills 最佳实践指南：[Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **作用范围** | 个人级，跨项目 | 项目级，跨团队 | 各有所长 |
| **存储位置** | `~/.claude/projects/*/memory/` | 项目根目录 | — |
| **版本控制** | 不进 Git | 随代码提交 | CLAUDE.md |
| **团队共享** | 仅限当前用户 | 所有团队成员 | CLAUDE.md |
| **内容类型** | 用户偏好、反馈、项目上下文 | 规则、命令、约束、风格 | — |
| **更新方式** | Claude 自动写入或用户指示 | 手动编辑 | Memory |
| **加载时机** | 按相关性召回 | 每次会话开始自动加载 | CLAUDE.md |
| **持久性** | 可能过时，需要清理 | 与代码同步演进 | CLAUDE.md |

## 核心差异：确定性 vs 适应性

这是理解两者区别最重要的维度。

**CLAUDE.md 是确定性的。** 你写什么，Claude 就执行什么。"提交前必须通过所有测试"——这条规则没有模糊空间，每次都会被执行。这种确定性让它适合编码团队规范、质量门禁、架构约束等"硬规则"。

**Memory 是适应性的。** 它记录的是上下文和偏好，Claude 会根据当前场景判断是否需要应用。比如记忆了"用户是数据科学家，目前在研究日志系统"，Claude 不会机械地在每次对话中提及这一点，而是在相关场景中调整自己的解释方式和建议角度。

这种区别在实际使用中影响很大。如果你把应该写在 CLAUDE.md 里的规则放进 Memory（比如"不要用 var，用 const"），它可能在某些会话中被召回，某些会话中被忽略——因为 Memory 的召回是基于相关性判断的，不是强制加载的。反过来，如果你把个人偏好写进 CLAUDE.md（比如"我喜欢简洁的回复"），它会成为整个团队的规则——这可能不是你的本意。

## 核心差异：写入机制

CLAUDE.md 完全由人类控制。你手动编辑文件、提交到 Git、通过 code review。这意味着每一条规则都经过了有意识的决策。

Memory 的写入则是混合模式：Claude 可以在对话中自动检测到值得记住的信息并主动保存，用户也可以明确说"记住这个"。这种自动化带来了便利，但也意味着你需要定期审查 Memory 内容，清理过时或不准确的记录。

一个常见的陷阱是：Memory 中记录了某个文件路径或函数名，但代码后来重构了。Memory 不会自动更新——它记录的是写入时的状态，不是实时状态。CLAUDE.md 通常不会有这个问题，因为它描述的是规则和约束，而不是具体的代码细节。

## 什么时候用 CLAUDE.md

选择 CLAUDE.md 的场景很明确：

- **团队共享的工程标准**：构建命令、测试要求、lint 规则、提交规范。这些是所有人都必须遵循的"法律"
- **项目架构约束**：不要在 pipeline 脚本里 import Next.js 模块、中文内容必须用 CJK 分词计数。这些是技术上的硬限制
- **质量门禁**：提交前必须通过 build、test、validate。没有例外
- **已知陷阱**：某个函数必须传三个参数、某个 API 有特定的调用顺序。这些是所有开发者都需要知道的

简单来说，如果一条规则是**项目级的、确定性的、需要强制执行的**，它属于 CLAUDE.md。

## 什么时候用 Memory

Memory 的最佳使用场景是：

- **个人工作偏好**：喜欢简洁回复、不要在每次操作后总结、偏好某种代码风格。这些是个人级的，不应该强加给团队
- **项目背景上下文**：当前在做什么、为什么做、截止日期是什么。这些信息帮助 Claude 做出更好的判断，但不是硬性规则
- **用户画像**：你的角色、专业领域、技术栈熟悉度。Claude 会据此调整解释的深度和角度
- **外部资源引用**：Bug 追踪在 Linear 的哪个 project、监控看板在哪个 URL。这些信息在代码库里找不到

简单来说，如果信息是**个人级的、上下文相关的、辅助判断的**，它属于 Memory。

## 协同使用：最佳实践

高效使用 Claude Code 的关键不是二选一，而是让两套系统各司其职：

1. **CLAUDE.md 定义"什么必须做"**——质量标准、架构约束、工作流程。这是团队的工程宪法
2. **Memory 记录"怎么做更好"**——个人偏好、项目背景、历史决策的原因。这是个人的工作助手
3. **不要重复**——如果某条信息已经在 CLAUDE.md 里，不需要再存到 Memory
4. **定期清理 Memory**——代码在演进，三个月前记录的文件路径可能已经不存在了

一个实际例子：CLAUDE.md 里写"提交前必须跑 `npm test`"（规则），Memory 里记录"上次测试失败是因为忘了 mock 环境变量"（经验）。前者确保每次都执行测试，后者帮助 Claude 在测试失败时更快定位问题。

## 结论

**CLAUDE.md 和 Claude Memory 解决的是不同层次的问题。** CLAUDE.md 是项目的规则手册，确定性强、团队共享、版本可控——适合编码所有"必须遵循"的工程标准。Memory 是个人的上下文笔记，自动积累、按需召回、跨会话持久——适合记录偏好、背景和经验。

最常见的错误是把所有东西都塞进 CLAUDE.md，或者完全忽略 Memory。正确的做法是：**硬规则进 CLAUDE.md，软上下文进 Memory**。两者配合使用，Claude Code 才能既遵守团队标准，又理解你的个人工作方式。

## 常见问题

### Claude Memory 的内容会被团队其他人看到吗？

不会。Memory 存储在用户本地的 `~/.claude/projects/` 目录下，不进入 Git 版本控制，完全是个人级别的。团队共享的信息应该写在 CLAUDE.md 或 SKILL.md 中。

### CLAUDE.md 支持多层级配置吗？

支持。Claude Code 会加载三层指令：全局级（`~/.claude/CLAUDE.md`）、项目级（项目根目录的 `CLAUDE.md`）、以及任务级（`skills/*/SKILL.md`）。优先级从具体到通用，任务级最高。

### Memory 记录了错误信息怎么办？

可以直接告诉 Claude"忘记关于 X 的记忆"，它会找到并删除相关记录。也可以手动编辑 `.claude/projects/` 下的 markdown 文件。建议定期审查 Memory 内容，清理过时的记录。

### 两者可以完全互相替代吗？

不能。CLAUDE.md 的确定性加载机制是 Memory 无法替代的——Memory 基于相关性召回，可能遗漏关键规则。反过来，Memory 的自动积累和个人化特性也不适合放进团队共享的 CLAUDE.md。它们是互补关系，不是竞争关系。
