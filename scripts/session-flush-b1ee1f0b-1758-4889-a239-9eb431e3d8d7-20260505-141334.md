**[CLAUDE.md](/glossary/claude-md)** is a markdown instruction file that defines how Claude Code should behave within a specific project. It's checked into your repository's root directory, version-controlled alongside your code, and shared with every team member who uses Claude Code on that project. When Claude Code starts a conversation, it reads CLAUDE.md first — before any user input — making it the highest-priority source of project context.

CLAUDE.md typically contains build commands, coding standards, architectural constraints, workflow rules, and quality gates. It's deterministic: every developer on the team gets the same instructions, every time. Changes to CLAUDE.md go through code review like any other file, creating an auditable history of how your AI assistant's behavior evolves.

There's also a user-level `~/.claude/CLAUDE.md` for personal instructions that apply across all projects — things like commit message preferences or communication style. But the project-level file is what most teams care about.

## Feature Comparison

| Feature | Claude Memory | CLAUDE.md |
|---------|--------------|-----------|
| **Location** | `.claude/projects/<id>/memory/` | Project root (repo) or `~/.claude/` |
| **Version controlled** | No (gitignored) | Yes (checked into repo) |
| **Shared with team** | No (personal) | Yes (via git) |
| **Who writes it** | Claude (automatically) | Developers (manually) |
| **When it's read** | When contextually relevant | Every conversation start |
| **Content type** | Observations, preferences, context | Instructions, rules, commands |
| **Editability** | Claude manages; user can request changes | Direct file editing |
| **Scope** | Per-user, per-project | Per-project (or per-user global) |
| **Determinism** | Emergent, evolving | Explicit, versioned |
| **Priority** | Lower (contextual guidance) | Higher (project instructions) |

## How They Work Together: The Context Stack

Claude Code's context system isn't an either/or choice — it's a layered stack where CLAUDE.md and Memory serve different roles. Understanding this stack is critical to using both effectively.

When Claude Code starts a session, it loads context in this order:

1. **System prompt** (Anthropic's base instructions)
2. **User-level CLAUDE.md** (`~/.claude/CLAUDE.md` — personal global preferences)
3. **Project-level CLAUDE.md** (repo root — shared team rules)
4. **Memory index** (`MEMORY.md` — always loaded for relevance scanning)
5. **Individual memory files** (loaded when contextually relevant)

This means CLAUDE.md acts as the **constitution** — hard rules that never bend. Memory acts as **institutional knowledge** — soft context that informs judgment calls. A CLAUDE.md rule saying "always run tests before committing" is absolute. A Memory noting "this user prefers verbose commit messages" is a preference that guides behavior without overriding explicit instructions.

As described in our [guide to Claude Code's seven programmable layers](/blog/claude-code-seven-programmable-layers), these context mechanisms form part of a broader extension architecture that includes skills, hooks, and MCP servers. CLAUDE.md and Memory are the foundational layers — they shape how all other layers behave.

## Determinism vs Adaptation: The Core Tradeoff

The fundamental difference between these systems is **determinism vs adaptation**. This tradeoff drives every decision about what belongs where.

CLAUDE.md is deterministic. If you write "use Vitest, not Jest" in CLAUDE.md, Claude will use Vitest every single time, for every team member, in every conversation. There's no ambiguity, no drift, no "sometimes it works differently." This predictability is essential for team consistency — you don't want Claude using different testing frameworks depending on who's asking.

Memory is adaptive. It lets Claude learn that you personally prefer to see TypeScript type annotations spelled out explicitly rather than inferred, or that your project just entered a code freeze, or that the last time Claude suggested a particular library you rejected it. This adaptation makes Claude increasingly effective for individual developers without requiring everyone to agree on personal preferences.

The tradeoff: **determinism gives you reliability but requires maintenance; adaptation gives you personalization but introduces variability.** A team of five developers will have five different Memory states, but one shared CLAUDE.md. This is by design.

## What Belongs in CLAUDE.md

Put information in CLAUDE.md when it meets these criteria:

**Shared rules everyone must follow:**
- Build commands (`npm run build`, `npm test`)
- Quality gates (tests must pass, linting required)
- Architectural constraints (no direct DB imports in frontend)
- Coding style rules (naming conventions, file organization)
- Workflow requirements (discuss design before coding, no force-push to main)

**Context that's code-adjacent:**
- How to run the dev server
- Known gotchas and sharp edges
- Tech stack decisions and their rationale
- Deployment procedures

**Instructions that should survive team changes:**
If a new developer joins tomorrow, they should get the same Claude behavior without any setup. CLAUDE.md ensures this. Everything in it is discoverable, reviewable, and changeable through normal pull request workflows.

Our [complete guide to Claude Code](/blog/claude-code-complete-guide) covers CLAUDE.md setup in detail, including how to structure it for different team sizes and project complexities.

## What Belongs in Memory

Put information in Memory when it meets these criteria:

**Personal context that varies by developer:**
- "I'm a data scientist, not a frontend engineer — explain React in terms I know"
- "I prefer terse responses without trailing summaries"
- "I'm new to this codebase but experienced with Go"

**Feedback and corrections:**
- "Don't mock the database in integration tests"
- "The single bundled PR approach worked well for this area"
- "Stop suggesting Redux — we committed to Zustand six months ago"

**Ephemeral project state:**
- "Merge freeze begins March 5 for mobile release"
- "The auth rewrite is driven by legal compliance, not tech debt"
- "Bugs are tracked in Linear project INGEST"

**Reference pointers:**
- "The oncall dashboard is at grafana.internal/d/api-latency"
- "Design mockups live in Figma project X"

Memory excels at context that's true right now but might change, that's specific to one person's workflow, or that Claude learned through interaction rather than explicit documentation.

## Team Dynamics: Shared vs Personal

For teams, the distinction between CLAUDE.md and Memory creates a clean separation of concerns:

**CLAUDE.md = team contract.** Everyone agrees to these rules. Changes go through PR review. New hires inherit them immediately. This is where you encode your engineering culture — the "how we do things here" that shouldn't vary by individual.

**Memory = personal assistant context.** Each developer's Claude instance learns their individual preferences, expertise level, and working style. A junior developer's Memory might note that they need more explanation of architectural patterns. A senior developer's Memory might note that they want code-only responses with no exposition.

This separation means you never have to compromise between team consistency and personal effectiveness. The team-level rules stay rigid; the personal experience stays flexible.

For teams evaluating how [Claude Code reshapes engineering workflows](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify), this dual system is a key enabler — it lets organizations standardize AI behavior at the project level while preserving individual developer productivity gains.

## Common Mistakes

### Mistake 1: Putting personal preferences in CLAUDE.md

If you add "I prefer verbose commit messages" to your project's CLAUDE.md, every team member's Claude now writes verbose commit messages — even those who prefer concise ones. Personal preferences belong in Memory (or user-level `~/.claude/CLAUDE.md`).

### Mistake 2: Putting project rules in Memory

If Claude learns through Memory that "tests must pass before commit," that rule only applies to you. A new team member's Claude won't know about it. Critical project rules belong in CLAUDE.md where they're universal and version-controlled.

### Mistake 3: Duplicating information across both

If CLAUDE.md says "use Vitest" and Memory also records "this project uses Vitest," you're creating maintenance burden with no benefit. Memory should never duplicate what's already in CLAUDE.md — the [Claude Code memory system](/blog/claude-code-memory) is designed to avoid exactly this redundancy.

### Mistake 4: Using Memory for code patterns

Don't ask Claude to remember that "the DB helper lives at `scripts/lib/db.ts`" — that's discoverable by reading the code. Memory should store things that **can't be derived** from the current codebase state: decisions, preferences, external references, and context about why things are the way they are.

### Mistake 5: Never updating CLAUDE.md

Some teams write CLAUDE.md once and forget it. As your project evolves — new tools, new conventions, deprecated patterns — CLAUDE.md should evolve too. Treat it like living documentation, updated in the same PR that introduces the change.

## Migration Paths

### Starting from scratch

If you're new to [Claude Code](/glossary/claude-code), start with CLAUDE.md. Write your build commands, coding standards, and key constraints. Memory will accumulate naturally as you work — you don't need to seed it.

### Moving from Memory to CLAUDE.md

If you've been using Claude Code solo and your Memory has accumulated project rules that should apply to everyone, extract those into CLAUDE.md before onboarding teammates. Look for memory entries that describe project-wide conventions rather than personal preferences.

### Moving from CLAUDE.md to Memory

If your CLAUDE.md has grown bloated with personal notes, development journal entries, or context that only applies to one person, move those to Memory (or user-level CLAUDE.md) to keep the project file focused.

## When to Choose Claude Memory

- You're a solo developer who wants Claude to improve over time without manual config
- You want Claude to remember corrections ("don't do X again")
- You need to store context that changes frequently (current sprint focus, active incidents)
- You want Claude to adapt to your expertise level and communication preferences
- You have external system references that only you need (your specific dashboard URLs, your team's Linear project)

## When to Choose CLAUDE.md

- You work on a team and need consistent Claude behavior across all members
- You have hard rules that must never be violated (quality gates, security constraints)
- You want new team members to get full context immediately without interaction history
- You need auditable, reviewable AI configuration that goes through code review
- You want to define build commands, deployment procedures, or workflow requirements

## When to Use Both

Most production setups use both. The pattern:

1. **CLAUDE.md** defines the project's rules, commands, and constraints
2. **Memory** captures individual developer context, corrections, and evolving project state
3. **User-level CLAUDE.md** (`~/.claude/CLAUDE.md`) holds personal cross-project preferences

This three-layer approach gives you team consistency (project CLAUDE.md), personal effectiveness (Memory), and cross-project preferences (user CLAUDE.md) — each scoped appropriately.

For a deeper look at how these layers interact with skills, hooks, and other extension mechanisms, see our analysis of [Claude Code's extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp).

## Verdict

**Use CLAUDE.md as your foundation** — it's where project rules, build commands, and team conventions live. It's deterministic, shared, and version-controlled. Every Claude Code project should have one.

**Use Memory as your accelerator** — it makes Claude increasingly effective with you personally, without requiring manual configuration or team consensus. It handles the soft context that makes the difference between a generic AI assistant and one that knows your codebase, your preferences, and your working style.

The two systems are complementary, not competing. CLAUDE.md answers "how should Claude behave in this project?" Memory answers "what has Claude learned about working with this specific developer on this project?" Teams that use both effectively report the highest satisfaction with Claude Code's output quality over time.

If you're just getting started, write a CLAUDE.md first. Memory will build itself. If you want to understand more about how to [effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code), the interplay between these context systems is the foundation.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions take priority as they represent explicit project rules. Memory provides supplementary context that informs Claude's judgment within the boundaries set by CLAUDE.md. If Memory conflicts with CLAUDE.md, the CLAUDE.md rule wins.

### Does Claude Memory sync across devices?

Memory lives in the local filesystem at `.claude/projects/<id>/memory/`. It doesn't automatically sync across machines. If you work from multiple devices, you'd need to manually sync the memory directory or accept that each device develops its own context over time.

### How do I see what Claude has stored in Memory?

Memory files are plain markdown stored in `.claude/projects/<project>/memory/`. You can read them directly, ask Claude to recall specific memories, or browse the `MEMORY.md` index file that lists all stored memories with one-line descriptions.

### Should I check Memory files into version control?

No. Memory is personal and gitignored by default. It contains individual preferences and context that shouldn't be imposed on other team members. Project-wide knowledge belongs in CLAUDE.md, which is designed for version control.

### How often should I update CLAUDE.md?

Update CLAUDE.md in the same PR that introduces the change it documents. New build step? Update CLAUDE.md. New coding convention? Update CLAUDE.md. Treat it as living documentation, not a one-time setup file.
