# Claude Memory vs CLAUDE.md：两套记忆系统怎么选？

**TL;DR：** **CLAUDE.md** 是团队共享的规则手册，写进仓库、版本控制、所有人遵守同一套标准。**Claude Memory**（Auto Memory）是个人自动笔记本，跨会话记住你的偏好和上下文。两者不是替代关系——**CLAUDE.md 管规则，Memory 管偏好，最佳实践是同时使用**。

## 概述：Claude Memory（Auto Memory）

**Claude Memory** 是 Claude Code 内置的自动记忆系统，它在对话过程中自动学习并持久化你的工作偏好、技术背景和项目上下文。记忆文件存储在 `~/.claude/projects/<project>/memory/` 目录下，以 Markdown 格式组织，包含 `user`（用户画像）、`feedback`（行为反馈）、`project`（项目状态）和 `reference`（外部资源指引）四种类型。

核心特点：**自动触发、个人专属、跨会话持久**。你不需要手动维护——当你纠正 Claude 的行为（"不要 mock 数据库"）或透露背景信息（"我是数据工程师"），系统会自动捕捉并保存。下次开启新会话时，Claude 会读取这些记忆来调整自己的行为模式。

Anthropic 在 2026 年初对[记忆系统进行了大幅升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，免费用户也能使用，并支持从 ChatGPT 导入历史数据。

## 概述：CLAUDE.md

**CLAUDE.md** 是一个手动维护的项目配置文件，放在仓库根目录下，随代码一起提交到 Git。它定义了 Claude Code 在这个项目中必须遵守的规则：构建命令、代码风格、质量门禁、工作流约定、已知陷阱等。

核心特点：**手动编写、团队共享、版本控制**。CLAUDE.md 像一份工程规范——不管谁打开这个项目的 Claude Code，都会读到同一套指令。它还支持层级结构：全局 `~/.claude/CLAUDE.md` 定义个人偏好，项目级 `CLAUDE.md` 定义团队标准，子目录下的 `CLAUDE.md` 可以覆盖上层规则。

在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中，CLAUDE.md 是最基础的一层，决定了 AI 对整个项目的理解框架。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **维护方式** | 自动捕捉 + 手动补充 | 纯手动编写 | Memory |
| **作用范围** | 个人专属 | 团队共享 | CLAUDE.md |
| **版本控制** | 不进 Git | 随仓库提交 | CLAUDE.md |
| **内容类型** | 偏好、反馈、上下文 | 规则、命令、约束 | 各有侧重 |
| **生效时机** | 每次会话自动加载 | 每次会话自动加载 | 平局 |
| **可审计性** | 低（个人文件） | 高（代码审查可见） | CLAUDE.md |
| **学习成本** | 近乎零 | 需要编写和维护 | Memory |
| **一致性保障** | 因人而异 | 全团队统一 | CLAUDE.md |

## 深度对比一：规则执行 vs 偏好适应

这是两者最根本的区别。CLAUDE.md 定义的是**硬规则**——"提交前必须通过 `npm run build` 和 `npm test`"、"不要在 pipeline 脚本里 import Next.js 模块"。这些规则不因人而异，不可商量，违反就是 bug。

Claude Memory 捕捉的是**软偏好**——"这个用户喜欢简短的回复"、"不要在回复结尾加总结"、"集成测试用真实数据库而不是 mock"。这些偏好因人而异，同一个团队里不同的开发者可能有完全不同的记忆库。

实际案例：假设你在团队项目中工作。CLAUDE.md 写明 "commit message 用英文、遵循 Conventional Commits 格式"——这对所有人生效。而你的 Memory 记住了 "这个用户是后端工程师、熟悉 Go 但不熟悉 React"——这只影响 Claude 跟你的沟通方式。

**判断标准：如果一条规则应该所有团队成员都遵守，放 CLAUDE.md；如果只是个人习惯，让 Memory 自动记录。**

## 深度对比二：可维护性与生命周期

CLAUDE.md 是代码的一部分，享受代码的所有治理优势：Pull Request 审查、变更历史、分支隔离。当项目架构调整时，团队可以像修改代码一样修改 CLAUDE.md，有完整的审计链条。

但这也是它的负担——**CLAUDE.md 需要持续维护**。规则过时了没人删、新模块加了没人补、约束写得太泛导致 Claude 行为诡异。根据我们在[扩展栈实践](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中的观察，很多团队的 CLAUDE.md 在写完第一版后就很少更新，逐渐与实际代码脱节。

Claude Memory 的维护成本几乎为零——它自动学习、自动更新。但这也意味着**不可预测性**：你不知道 Memory 到底记住了什么，也无法保证它记住了正确的东西。过时的记忆可能导致 Claude 在新上下文中做出错误判断。好在你可以随时查看和清理记忆文件，系统也会在检测到冲突时优先信任当前代码状态。

## 深度对比三：团队协作场景

在团队环境中，两者的角色分工最为清晰：

**CLAUDE.md 是团队的工程标准层。** 它确保不管是新入职的实习生还是资深工程师，打开 Claude Code 后看到的规则完全一致。CI/CD 约定、测试策略、安全要求——这些都必须是确定性的。配合 [Skills 系统](/zh/blog/9-principles-writing-claude-code-skills)使用时，CLAUDE.md 定义全局约束，SKILL.md 定义任务级指令，形成完整的指令层级。

**Claude Memory 是个人的效率加速层。** 它记住你常用的调试方式、你对代码风格的微妙偏好、你上周在排查的那个 bug 的上下文。这些信息对你有价值，但对团队没有标准化意义。

一个常见的反模式是把个人偏好写进 CLAUDE.md——比如 "回复要简短" 或 "用中文解释"。这些应该留给 Memory 系统处理，或放在个人级别的 `~/.claude/CLAUDE.md` 中，不要污染团队的项目配置。

## 最佳搭配策略

理解了区别之后，最佳实践不是二选一，而是分层使用：

**第一层：项目 CLAUDE.md（团队标准）**
- 构建和测试命令
- 代码规范和质量门禁
- 架构约束和已知陷阱
- 工作流规则（分支策略、commit 格式）

**第二层：个人 ~/.claude/CLAUDE.md（个人规则）**
- Git 工作流偏好（如 "每次改动后立即 commit + push"）
- 语言偏好（如 "用中文沟通"）
- 通用工作习惯

**第三层：Claude Memory（自动适应）**
- 技术背景和专长领域
- 沟通风格偏好
- 项目上下文和进度
- 反馈纠正记录

这三层的加载顺序是确定的：全局 CLAUDE.md → 项目 CLAUDE.md → Memory。冲突时项目级规则优先于个人规则，Memory 作为最后的上下文补充。

## 结论

**CLAUDE.md 和 Claude Memory 解决的是不同层次的问题。** CLAUDE.md 是写给团队的工程规范，确保一致性和可审计性——如果你在团队项目中工作，它是必须配置的。Claude Memory 是个人效率工具，减少重复沟通成本——它会自动工作，你几乎不需要操心。把规则放 CLAUDE.md，把偏好交给 Memory，这就是最简单也最有效的策略。

如果你还没有为项目创建 CLAUDE.md，现在就开始——从构建命令和测试命令写起，不需要一次写完。Memory 系统会在你使用过程中自然积累。关于如何写好 CLAUDE.md，我们在 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)中有完整的实践指南。

## 常见问题

### Claude Memory 会自动读取 CLAUDE.md 的内容吗？

不会。两者是独立的系统。CLAUDE.md 在每次会话开始时作为系统指令加载，Memory 文件从个人存储目录加载。它们在 Claude 的上下文中同时生效，但数据源完全分离，互不干扰。

### 个人偏好应该写在 CLAUDE.md 还是让 Memory 记住？

个人偏好推荐让 Memory 自动记录，或写在全局 `~/.claude/CLAUDE.md` 中。项目级 CLAUDE.md 只放团队标准——如果一条规则只对你有意义而不是所有人，它就不属于项目 CLAUDE.md。

### 如果 CLAUDE.md 的规则和 Memory 的记忆冲突了怎么办？

CLAUDE.md 的规则优先级更高。如果 Memory 记住了 "不写测试"，但 CLAUDE.md 要求 "提交前必须通过所有测试"，Claude 会遵守 CLAUDE.md 的规则。Memory 系统在检测到与当前代码状态或配置冲突时，也会优先信任当前状态。
