**CLAUDE.md as team alignment tool.** When a team adopts Claude Code, CLAUDE.md becomes the single source of truth for how the AI should behave in the project. Senior engineers encode hard-won knowledge: "never import Next.js modules in pipeline scripts," "always run tests before committing," "Chinese content must use CJK word count." New team members — and their Claude Code sessions — inherit this knowledge immediately. No onboarding document to read, no tribal knowledge to absorb. The instructions execute automatically.

This makes CLAUDE.md especially powerful for enforcing quality gates. A well-written CLAUDE.md can require that every commit passes build, tests, and linting. Claude Code reads these constraints and follows them, reducing the "works on my machine" problem and the "I forgot to run tests" problem simultaneously.

**Claude Memory as personal context.** Memory fills a different gap: it remembers *you*. After a few conversations, Claude Code knows that you're a data scientist investigating logging (so it frames code explanations differently), that you prefer bundled PRs over many small ones (so it doesn't split refactors unnecessarily), and that pipeline bugs are tracked in Linear project "INGEST" (so it knows where to look). None of this belongs in CLAUDE.md — it's personal context that would be noise for other contributors.

**The conflict scenario.** Problems arise when information that should be in CLAUDE.md ends up only in someone's Memory, or vice versa. If your senior engineer's Claude Memory knows "never use mocks in integration tests" but the CLAUDE.md doesn't mention it, new team members will generate mocked tests that pass locally but fail in production. If your CLAUDE.md contains personal preferences ("use terse responses"), it annoys teammates who prefer detailed explanations.

**Decision rule for teams:** After every conversation where Claude learns something useful, ask: "Would this help other contributors?" If yes, add it to CLAUDE.md. If it's just about your working style, let Memory keep it. For a practical guide to structuring these files, see [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code).

## Content and Control: Detailed Analysis

What you can store in each system — and how much control you have over it — differs significantly.

**CLAUDE.md content is explicit and structured.** You write exactly what you want Claude Code to know. Common content includes:

- Build and test commands (`npm run build`, `npm test`)
- Quality gates ("all tests must pass before commit")
- Architecture constraints ("don't import server-only modules in client code")
- Coding style rules ("default to writing no comments")
- Workflow rules ("new feature → discuss design first, get approval before coding")
- Known gotchas ("ZH content must use CJK word count, not English space tokenization")

You control every word. If a rule isn't working, you edit the file. If a rule is outdated, you remove it. Changes are visible in git history.

**Claude Memory content is observational and categorized.** Memory files are organized into four types:

- **User memories**: Your role, expertise, preferences ("senior engineer, new to React")
- **Feedback memories**: Corrections and confirmed approaches ("don't mock the database — prior incident with migration failures")
- **Project memories**: Ongoing initiatives, deadlines, decisions ("merge freeze begins March 5 for mobile release")
- **Reference memories**: Pointers to external resources ("pipeline bugs tracked in Linear project INGEST")

You have less direct control over Memory. Claude decides what to save based on conversation signals — corrections, confirmations, explicit "remember this" requests. You can ask Claude to save or delete specific memories, but the day-to-day accumulation is automatic. This means Memory can drift or accumulate stale entries. A project memory about a merge freeze from three months ago is noise, not signal.

**Staleness risk.** CLAUDE.md gets stale too, but the staleness is visible — you see the file every time you open the project root. Memory staleness is hidden. Memory files sit in dotfile directories that most developers never browse. Claude Code mitigates this by treating memories as "claims to verify" rather than absolute truth — it checks whether a remembered file path still exists before recommending it — but the risk remains.

**Control recommendation:** Treat CLAUDE.md as law and Memory as context. CLAUDE.md rules should be hard constraints that Claude Code must follow. Memory should be soft context that informs Claude Code's approach but doesn't override explicit instructions. If there's a conflict between a CLAUDE.md rule and a Memory entry, CLAUDE.md wins.

## How They Work Together

In practice, **Claude Memory** and **CLAUDE.md** are not competing systems — they're complementary layers that serve different purposes in the same workflow. Understanding their interaction is the key to getting the most out of Claude Code.

**Session startup flow.** When you launch Claude Code, it loads CLAUDE.md files first (project root, then subdirectories, then your personal `~/.claude/CLAUDE.md`). These establish the rules of engagement. Then it loads relevant Memory files from `.claude/projects/<hash>/memory/MEMORY.md`, which provide personal context. The result: Claude Code knows both the project's rules (from CLAUDE.md) and your personal working style (from Memory).

**Practical example.** Your project's CLAUDE.md says "commit message must accurately describe changes." Your Memory says "this user prefers conventional commit format (feat:, fix:, chore:)." Claude Code combines both: it writes accurate commit messages *in conventional commit format*. Neither system alone would produce this result.

**Migration path.** When a Memory entry proves universally useful, promote it to CLAUDE.md. If you notice Claude Code consistently applying a Memory-learned pattern that helps your whole team, codify it. "This user prefers running tests before build" might start as personal Memory, but if it catches bugs earlier for everyone, it belongs in the project's quality gates section of CLAUDE.md.

For teams building sophisticated Claude Code workflows, the [complete guide to Claude Code](/blog/claude-code-complete-guide) covers how CLAUDE.md, Memory, Skills, Hooks, and MCP servers compose together into a full [agentic coding](/glossary/agentic-coding) platform.

## When to Choose CLAUDE.md

Use CLAUDE.md when the information is **project-specific, team-shared, and stable**:

- **Build and deployment instructions** — every contributor needs to know how to build, test, and deploy. These commands rarely change and must be consistent across the team.
- **Architecture constraints** — "don't import X in Y," "always use library Z for database access." These are hard rules that prevent bugs if violated.
- **Quality gates** — "all tests must pass before commit," "run linter before push." Non-negotiable requirements that Claude Code should enforce every time.
- **Known gotchas** — project-specific footguns that cost hours to debug. "The `upsertKeyword()` function requires three arguments, not two" saves every future developer from the same trap.
- **Coding style** — comment policy, naming conventions, abstraction philosophy. These should be consistent across the team, not personal preference.

CLAUDE.md is also the right choice when you need **auditability**. Because it's version-controlled, you can trace when a rule was added, who added it, and why. This matters for compliance-sensitive projects and for onboarding new team members who want to understand the "why" behind constraints. Teams that have invested in well-structured CLAUDE.md files report in community discussions that new engineers — and their AI sessions — become productive faster because the project's conventions are explicit rather than tribal.

## When to Choose Claude Memory

Use Claude Memory when the information is **user-specific, evolving, or contextual**:

- **Personal working style** — response verbosity, preferred diff format, whether you want explanations or just code. These vary between team members and shouldn't be imposed project-wide.
- **Your expertise profile** — "senior backend engineer, first time touching React." This helps Claude Code calibrate explanations to your knowledge level without affecting how it explains things to your frontend colleague.
- **Feedback patterns** — "don't suggest mocks for this project's tests" (after you corrected Claude once). Memory captures these corrections so you don't repeat yourself across sessions.
- **Active project context** — current sprint goals, ongoing incidents, temporary constraints. These change frequently and would create excessive CLAUDE.md churn if tracked there.
- **External resource pointers** — "bugs are in Linear project X," "the oncall dashboard is at URL Y." These are useful navigation shortcuts that don't affect code behavior.

Claude Memory is also the right choice when you want **zero maintenance overhead**. You don't write Memory files — Claude writes them based on conversation signals. For solo developers or small teams where the overhead of maintaining a detailed CLAUDE.md feels excessive, Memory provides a low-friction alternative that captures useful context automatically. However, be aware that Memory's automatic nature means it can accumulate stale or incorrect entries over time. Periodically reviewing your Memory directory (`.claude/projects/<hash>/memory/`) is good hygiene.

## Verdict

**Use both — they solve different problems.** CLAUDE.md is your project's constitution: shared rules, build commands, architecture constraints, quality gates. Every team should have one, and it should be maintained with the same rigor as any other code artifact. Claude Memory is your personal assistant's notebook: it remembers your preferences, your corrections, and your context so you don't repeat yourself across sessions.

**If you can only invest in one**, start with CLAUDE.md. A well-written CLAUDE.md immediately improves every Claude Code session for every team member. Memory accumulates naturally without effort. But the real power comes from using both: CLAUDE.md sets the rules, Memory personalizes the experience within those rules.

**The most common mistake** is putting personal preferences in CLAUDE.md (annoying your teammates) or leaving team-critical knowledge only in Memory (invisible to everyone else). The decision rule is simple: if removing this information would hurt the next developer who joins the project, it belongs in CLAUDE.md. If it would only affect your personal workflow, let Memory handle it. ## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md instructions take priority over Memory entries. If your CLAUDE.md says "always run tests before commit" but your Memory contains a note about skipping tests for draft PRs, Claude Code follows the CLAUDE.md rule. Memory provides soft context; CLAUDE.md provides hard constraints. In cases of conflict, CLAUDE.md wins by design.

### Does Claude Memory sync across devices?

No. Claude Memory files are stored locally in `.claude/projects/` directories on your machine. If you switch to a new laptop, your Memory does not transfer automatically. You could manually copy the `.claude/` directory, but Memory files are tied to project path hashes, so they may not resolve correctly. CLAUDE.md, by contrast, syncs automatically through git because it's committed to the repository.

### Should solo developers bother with CLAUDE.md?

Yes. Even solo developers benefit from CLAUDE.md for build commands, quality gates, and architecture constraints. Without it, you rely on Claude Code re-discovering your project's conventions every session. A minimal CLAUDE.md with build commands, test commands, and two or three key constraints takes five minutes to write and saves time on every subsequent session. Memory handles the rest — your preferences, your context, your working style.

### How often should I clean up Claude Memory files?

Review your Memory directory every few weeks or when Claude Code starts making outdated assumptions. Memory files about completed initiatives, resolved incidents, or past deadlines become noise over time. You can ask Claude Code to forget specific memories during a conversation, or manually delete stale `.md` files from `.claude/projects/<hash>/memory/`. Keep the MEMORY.md index in sync when you remove files.

### Can I commit Claude Memory files to version control?

Technically yes, but it's not recommended. Memory files contain per-user preferences and observations that may not apply to other team members. Committing them could create conflicts and confusion. If a Memory entry is valuable enough to share, promote it to CLAUDE.md or a SKILL.md file where it benefits the whole team through the proper shared configuration path.
