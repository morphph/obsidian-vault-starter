**CLAUDE.md is the deterministic layer.** Every session loads it. Every team member gets the same instructions. If your project requires running tests before commits, that rule belongs in CLAUDE.md because it must fire reliably, without exception. The [extension stack that makes Claude Code programmable](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) — skills, hooks, agents, MCP — all builds on this deterministic foundation.

**Memory is the adaptive layer.** It captures context that emerges from working together over time. Your debugging preferences, the fact that you're a data scientist not a frontend engineer, the feedback that Claude should stop summarizing diffs you can read yourself — these are personal signals that make Claude more effective for you specifically.

The two systems complement rather than compete. CLAUDE.md says "use Vitest for testing." Memory says "this user prefers integration tests over unit tests and got burned by mocks last quarter." CLAUDE.md provides the shared floor; Memory personalizes the experience above it.

Where teams run into trouble is when they put personal preferences in CLAUDE.md (cluttering the shared config with one person's opinions) or project standards in Memory (where they won't apply to other team members). The rule is simple: **if everyone on the team needs it, it goes in CLAUDE.md. If only you need it, let Memory handle it.**

## Content Types: What Goes Where

Choosing the right system for each piece of context is the most practical decision you'll make. Here's a detailed breakdown.

### What belongs in CLAUDE.md

- **Build and test commands**: `npm run build`, `npm test`, lint commands — anything the CI/CD pipeline also enforces
- **Coding standards**: naming conventions, import ordering, error-handling patterns
- **Architecture constraints**: "don't import Next.js modules in pipeline scripts," "use CJK word count for Chinese content"
- **Workflow rules**: "discuss design before coding new features," "run validation before committing pipeline changes"
- **Known gotchas**: project-specific pitfalls that any contributor could hit
- **Documentation rules**: what to update when certain files change

These are the kinds of instructions you see in production CLAUDE.md files. Our coverage of [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) shows how well-structured CLAUDE.md files dramatically reduce prompt repetition.

### What belongs in Memory

- **User profile**: your role, expertise level, areas of focus ("senior backend engineer, new to the React parts of this repo")
- **Feedback on Claude's approach**: corrections ("don't mock databases") and confirmations ("yes, the single bundled PR was right")
- **Project status**: who's working on what, deadlines, ongoing incidents — context that changes frequently and isn't in git
- **External references**: where to find bugs in Linear, which Grafana dashboard to check, Slack channels for specific topics
- **Collaboration preferences**: verbosity level, whether you want summaries, how you like commit messages structured

Memory shines for the kind of context that would be inappropriate in a shared config file. Your preference for terse responses doesn't need to be a team-wide rule.

## Lifecycle and Maintenance

**CLAUDE.md maintenance is manual but predictable.** You write it, review it in PRs, and update it when the project changes. The overhead is proportional to how fast your project conventions evolve. For stable projects, a CLAUDE.md might go months without changes. For rapidly iterating codebases, it needs regular updates — and the best practice is to update docs alongside code changes, not as a separate task.

**Memory maintenance is mostly automatic but requires occasional pruning.** Claude creates and updates memory files as it learns. Over time, memories can become stale — a project status from three months ago, a preference you've since changed, a reference to a system that's been decommissioned. Claude is instructed to verify memories against current state before acting on them, but periodic review of your `~/.claude/projects/` directory keeps things clean.

The portability story differs significantly. CLAUDE.md travels with the repo — clone the project on a new machine, and all instructions are already there. Memory is local. If you switch machines, your memory files don't follow unless you manually copy them. Anthropic's [memory importing features](/blog/anthropic-claude-memory-upgrades-importing) are beginning to address this gap, but as of early 2026, Memory remains fundamentally tied to the local filesystem.

## Team Dynamics: Shared vs Personal Context

The shared-vs-personal distinction becomes critical on teams. Consider a team of five engineers working on the same repository.

**With CLAUDE.md alone**, every engineer gets the same Claude behavior. Build commands, test requirements, and coding standards are uniform. But Claude treats every engineer identically — it doesn't know that Alice is a principal engineer who wants concise responses while Bob is a junior who benefits from detailed explanations. It doesn't remember that Carol told it last week to stop suggesting TypeScript migrations.

**With Memory alone**, each engineer gets a personalized experience. Claude remembers their expertise, preferences, and prior feedback. But there's no shared foundation — if one engineer teaches Claude that the project uses Vitest, that knowledge doesn't transfer to other team members. Each person spends time re-teaching project basics.

**With both**, the team gets deterministic shared standards from CLAUDE.md plus personalized experiences from Memory. Alice gets terse, expert-level responses within the same project constraints as Bob, who gets more detailed explanations. This is the [agentic coding](/glossary/agentic-coding) model that scales — shared rules with individual adaptation.

## Configuration Hierarchy

Claude Code reads context in a specific priority order, and understanding this hierarchy prevents confusion when CLAUDE.md and Memory seem to conflict.

1. **Global CLAUDE.md** (`~/.claude/CLAUDE.md`) — user-wide defaults, lowest project-specific priority
2. **Project CLAUDE.md** (`./CLAUDE.md`) — repository-level instructions
3. **Subdirectory CLAUDE.md** (`./src/CLAUDE.md`) — module-specific overrides
4. **Memory files** — loaded from `~/.claude/projects/` based on the current working directory
5. **Conversation context** — what you've said in the current session

In practice, conflicts are rare because the systems hold different types of information. CLAUDE.md holds rules ("always run tests before committing"); Memory holds context ("this user is investigating logging infrastructure"). When there is genuine overlap, the more specific context typically wins — a Memory feedback entry saying "this user wants no trailing summaries" overrides a CLAUDE.md style guideline suggesting summaries, because Memory captures a direct user correction.

## Common Anti-Patterns

### Putting everything in CLAUDE.md

Some teams dump every possible instruction into CLAUDE.md — personal preferences, temporary project status, debugging notes. The file grows to hundreds of lines, becomes hard to maintain, and contains noise that doesn't apply to every team member. Keep CLAUDE.md focused on durable, team-wide conventions.

### Ignoring Memory entirely

Developers who carefully maintain their CLAUDE.md but never let Memory work for them miss the personalization benefits. If you find yourself repeating the same correction to Claude across sessions ("I told you, don't use mocks"), that's a signal Memory should be capturing that feedback.

### Duplicating content across both

Writing your coding standards in both CLAUDE.md and Memory creates maintenance burden and potential conflicts. The rule of thumb: if it's in CLAUDE.md, Memory doesn't need it. Memory should hold what CLAUDE.md cannot — personal context, transient project status, and adaptive feedback.

### Never cleaning up Memory

Memory files accumulate. A project status memory from January saying "we're in a code freeze for the mobile release" is actively misleading in April. Periodically review your memory index and remove or update stale entries.

## When to Choose Claude Memory

Use Memory as your primary context mechanism when:

- **You work solo** and don't need to share instructions with a team — Memory captures everything organically as you work
- **You want zero-maintenance context** — Claude automatically identifies and persists useful information without you writing config files
- **Your preferences differ from team defaults** — Memory lets you personalize Claude's behavior without modifying shared configuration
- **Project status changes frequently** — deadlines, incidents, and who-is-working-on-what are better captured in adaptive Memory than in static CLAUDE.md
- **You're onboarding to a new codebase** — Memory accumulates your learnings about the project naturally through conversations

Memory is particularly effective for the [kind of product management workflows](/blog/claude-code-for-product-managers) where context is conversational and evolving rather than codified and static.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary context mechanism when:

- **You're on a team** and need every contributor to get the same Claude behavior — CLAUDE.md is the only mechanism that's version-controlled and shared
- **You need deterministic enforcement** — build gates, test requirements, and quality checks that must fire every time without exception
- **You're encoding architecture decisions** — constraints like "don't import server modules in client code" belong in the shared project record
- **You want auditability** — CLAUDE.md changes show up in git history, so you can track when and why instructions evolved
- **You maintain [skills](/blog/5-claude-code-skills-i-use-every-single-day)** — SKILL.md files extend the CLAUDE.md pattern for task-specific instructions, and both are version-controlled

The [principles for writing effective Claude Code skills](/blog/9-principles-writing-claude-code-skills) apply equally to CLAUDE.md — specificity, examples, and clear constraints produce better results than vague instructions.

## Verdict

**Use both.** CLAUDE.md and Claude Memory are complementary systems, not alternatives. **CLAUDE.md is your team's shared operating manual** — deterministic, version-controlled, reviewed in PRs. **Memory is your personal working relationship with Claude** — adaptive, automatic, and unique to how you work. Start with a focused CLAUDE.md covering build commands, coding standards, and architecture constraints. Let Memory accumulate naturally from there. The developers getting the most out of [Claude Code](/blog/claude-code-complete-guide) are the ones who treat CLAUDE.md as the floor and Memory as the ceiling — a reliable shared foundation with personalized intelligence on top.

## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

Memory captures personal feedback and preferences, while CLAUDE.md holds project-wide rules. In practice, Memory personalizes within CLAUDE.md's constraints rather than overriding them. If your CLAUDE.md says "run tests before commits," Memory won't skip that — but it might remember that you prefer verbose test output.

### Does Claude Memory work without a CLAUDE.md file?

Yes. Memory functions independently — Claude saves and retrieves learned context whether or not a CLAUDE.md exists. However, without CLAUDE.md, Claude has no deterministic project instructions and relies entirely on what it has learned through conversations, which can be inconsistent across sessions.

### How do I see what Claude has saved in Memory?

Memory files are stored as markdown in `~/.claude/projects/` organized by project directory. Each file has frontmatter with a name, description, and type. The `MEMORY.md` index file lists all active memories. You can read, edit, or delete these files directly — they're plain text.

### Should I check Memory files into version control?

No. Memory files are personal to your machine and workflow. Checking them in would impose one person's preferences on the entire team, which is exactly what CLAUDE.md handles. Keep Memory local and CLAUDE.md shared.

### How often should I clean up stale Memory entries?

Review your memory index monthly or when you notice Claude acting on outdated information. Project status memories decay fastest — a "merge freeze this week" entry from a month ago is actively misleading. User profile and feedback memories tend to stay relevant longer.
