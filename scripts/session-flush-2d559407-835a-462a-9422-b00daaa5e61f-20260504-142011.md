# Claude Memory vs CLAUDE.md：哪个才是你需要的持久化方案？

**TL;DR：** **Claude Memory**（Auto Memory）是跨会话自动笔记系统，记录用户偏好和项目发现；**CLAUDE.md** 是手动编写的规则手册，定义项目约束和编码标准。两者不是竞争关系——Memory 记住「你是谁」，CLAUDE.md 定义「怎么干活」。小项目用 CLAUDE.md 就够，团队协作两者都要用。

## 概述：Claude Memory

Claude Memory 是 Claude Code 的自动持久化记忆系统，解决的核心问题是「每次新对话都要重新解释上下文」。它将跨会话发现的信息——用户角色、技术偏好、项目背景、反馈修正——自动保存为结构化的 markdown 文件，存储在 `~/.claude/projects/` 目录下。

Memory 的关键特征是**自动触发**：当 Claude Code 发现值得记住的信息时，它会主动写入记忆文件，无需用户手动操作。记忆分为四种类型：user（用户画像）、feedback（行为修正）、project（项目状态）、reference（外部资源指针）。每条记忆带有 frontmatter 元数据，通过 MEMORY.md 索引文件统一管理。

这套系统的设计哲学是「渐进式学习」——随着对话积累，Claude Code 对你和你的项目理解越来越深，最终像一个真正了解你的同事一样工作。详细机制可参考我们的[Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

CLAUDE.md 是 Claude Code 的项目级指令文件，本质上是一份「AI 行为规范手册」。它解决的问题是「如何让 AI 遵守团队约定」——编码风格、构建流程、架构约束、禁止事项，全部写在一个文件里，随代码库版本控制。

CLAUDE.md 的关键特征是**显式声明**：每条规则都由人类手动编写，Claude Code 在每次会话开始时读取并严格遵守。它支持多层级——全局 `~/.claude/CLAUDE.md` 定义个人偏好，项目根目录的 `CLAUDE.md` 定义团队规范，子目录可以有更细粒度的覆盖。

与 Memory 不同，CLAUDE.md 是确定性的：同一份文件，不同团队成员使用 Claude Code 时得到一致的行为。这使它成为团队协作的基础设施，而非个人工具。想深入了解如何编写高效的 CLAUDE.md，可以阅读我们关于 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)的分析。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **触发方式** | 自动（AI 主动记录） | 手动（人类编写） | 各有所长 |
| **存储位置** | `~/.claude/projects/*/memory/` | 项目根目录（可 git 管理） | CLAUDE.md |
| **版本控制** | 不跟随 git | 随代码库提交 | CLAUDE.md |
| **团队共享** | 仅限个人 | 整个团队共享 | CLAUDE.md |
| **内容类型** | 用户偏好、项目发现、反馈 | 规则、约束、流程、标准 | 各有所长 |
| **维护成本** | 接近零（自动维护） | 需要手动更新 | Memory |
| **可预测性** | 渐进式，可能过时 | 确定性，始终生效 | CLAUDE.md |
| **适用范围** | 跨项目、跨会话个人上下文 | 单项目团队规范 | 各有所长 |

## 持久化机制：深度对比

Claude Memory 和 CLAUDE.md 的底层机制完全不同，这决定了它们各自的优势和局限。

**Memory 的工作方式**：Claude Code 在对话过程中识别「值得记住的信息」——比如用户说「我是数据科学家」或「不要在测试里 mock 数据库」——然后将其写入独立的 markdown 文件，并在 MEMORY.md 索引中添加一行指针。下次新对话开始时，MEMORY.md 索引被加载到上下文中，Claude Code 根据当前任务的相关性决定是否读取具体记忆文件。

这意味着 Memory 有一个固有的**新鲜度问题**：记忆可能过时。如果你三个月前记录了「项目使用 PostgreSQL」，但后来迁移到了 SQLite，旧记忆可能误导 Claude Code。系统通过「验证后再使用」的原则缓解这个问题——在基于记忆采取行动前，先检查当前代码状态。

**CLAUDE.md 的工作方式**：每次 Claude Code 启动时，它会沿目录树向上搜索所有 CLAUDE.md 文件并全部加载。这些文件的内容直接注入到系统提示中，作为不可违反的指令。没有「相关性判断」这一步——所有规则始终生效。

这意味着 CLAUDE.md 有一个**信息密度问题**：规则太多会挤占上下文窗口，太少则覆盖不够。最佳实践是保持 CLAUDE.md 精简，聚焦于「AI 最容易犯错的行为约束」，而非试图文档化所有内容。

关于 Claude Code 如何组合这些持久化层来实现高效工作流，我们在[Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中有详细分析。

## 适用场景：深度对比

两者的适用场景几乎没有重叠，这也是为什么它们是互补而非竞争关系。

**Memory 最擅长的场景**：

- **个人偏好累积**：你喜欢 commit message 用中文、代码注释用英文、变量命名用 camelCase——这些不需要团队统一的个人习惯，Memory 自动学习后就不需要重复说明
- **项目状态追踪**：「上周决定把认证模块迁移到 OAuth2」「周四有代码冻结」——这类时效性信息不适合写入 CLAUDE.md
- **反馈循环**：「不要在 response 末尾加总结」「PR 用单个大 PR 而不是拆小」——对行为的修正，一次纠正后永久生效
- **外部资源指针**：「Bug 跟踪在 Linear 的 INGEST 项目」「oncall 看 Grafana 的 api-latency 面板」

**CLAUDE.md 最擅长的场景**：

- **构建流程**：`npm run build` 前必须跑 lint、测试必须通过才能 commit——这些硬性约束所有团队成员都要遵守
- **架构决策**：不要在 pipeline 脚本里 import Next.js 模块、API 路由必须返回标准错误格式——防止 AI 犯常见架构错误
- **编码风格**：中文内容用 CJK 字数统计、commit message 准确描述改动内容——需要确定性执行的规范
- **安全边界**：不要提交 .env 文件、不要 force push 到 main——不可违反的安全规则

如果你不确定一条信息该放哪里，问自己：「这是我的个人偏好，还是团队规范？」个人偏好 → Memory。团队规范 → CLAUDE.md。

## 什么时候选 Claude Memory

选择 Memory 作为主要持久化手段的场景：

- **个人项目、频繁切换上下文**：你同时维护多个项目，每个项目的细节不想重复解释
- **探索阶段**：项目还没定型，规则频繁变化，手动维护 CLAUDE.md 成本太高
- **非代码类工作**：写文档、做研究、整理资料——没有「编码规范」需要强制执行
- **长期合作关系**：你希望 Claude Code 随时间越来越了解你的工作方式

Memory 的核心价值是**降低重复沟通成本**。如果你发现每次新对话都要花前 5 分钟解释背景，Memory 能帮你省掉这一步。关于 Memory 的最新升级和导入功能，参见我们的[Claude 记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)报道。

## 什么时候选 CLAUDE.md

选择 CLAUDE.md 作为主要持久化手段的场景：

- **团队项目**：多人协作，需要一致的 AI 行为——不能因为每个人的 Memory 不同而产出不同风格的代码
- **关键约束不可违反**：「测试不能跳过」「不能 import server-only 模块」——这类规则必须 100% 执行率
- **需要审计和版本控制**：规则变更通过 PR review，有历史记录，可以回滚
- **新成员 onboarding**：clone 代码库就自动获得所有 AI 行为规范，零配置

CLAUDE.md 的核心价值是**确定性和可复现性**。同样的规则文件，不同的人在不同时间使用 Claude Code，得到一致的行为。这在工程团队中不可替代。

想了解如何将 CLAUDE.md 和 Skills 系统结合使用以获得最佳效果，推荐阅读[我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

## 最佳实践：两者协同

实际工程中，Memory 和 CLAUDE.md 应该协同工作，各司其职：

1. **CLAUDE.md 定义硬规则**：构建流程、代码风格、安全约束、架构决策
2. **Memory 记录软上下文**：用户角色、项目背景、行为反馈、外部资源
3. **Skills 处理具体任务**：特定类型工作的详细指令（写 newsletter、做 code review）
4. **不要在两边重复信息**：如果一条规则在 CLAUDE.md 里已经写了，Memory 不需要再记一遍

**决策规则**：如果你删掉这条信息，下次 Claude Code 会犯错吗？会 → CLAUDE.md。不会犯错但效率降低 → Memory。无所谓 → 不要记录。

## 结论

**Claude Memory 和 CLAUDE.md 解决的是两个不同维度的问题**。Memory 是「个人助手记忆」——它记住你是谁、你的偏好、你的项目历史。CLAUDE.md 是「团队行为规范」——它定义 AI 在这个项目里必须遵守什么规则。

对于个人开发者：从 CLAUDE.md 开始，写下 3-5 条最重要的规则，让 Memory 自然积累处理剩余部分。对于团队：CLAUDE.md 是必须的基础设施，Memory 是每个成员的个人增强。不要试图用一个替代另一个——它们是[Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中各自独立的层。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。CLAUDE.md 提供确定性的项目规则，Memory 补充个人上下文和时效性信息。两者在 Claude Code 启动时都会加载，互不冲突。

### Claude Memory 的内容会被团队其他人看到吗？

不会。Memory 存储在本地 `~/.claude/projects/` 目录下，属于个人私有数据，不会随代码库提交或共享。团队共享的持久化信息应该放在 CLAUDE.md 中。

### CLAUDE.md 写太长会有什么问题？

会挤占上下文窗口。CLAUDE.md 的内容在每次会话开始时全量注入，过长会减少可用于实际任务的 token 数。建议控制在 200 行以内，聚焦于 AI 最容易犯错的约束，避免写成完整的项目文档。

### 如果 Memory 记录了过时信息怎么办？

Claude Code 的 Memory 系统内置了「验证后再使用」原则——在基于记忆采取行动前会检查当前状态。如果发现过时信息，你可以要求 Claude Code 更新或删除对应的记忆文件。
