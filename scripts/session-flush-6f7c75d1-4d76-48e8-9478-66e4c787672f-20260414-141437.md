**Claude Memory, by contrast, is deliberately per-user.** Your memories about your role ("I'm new to the React side of this repo"), your preferences ("I prefer terse responses"), and your workflow ("I always review diffs manually") stay in your local `.claude/` directory. They don't pollute other team members' contexts.

This scoping is a feature, not a limitation. Not everything should be shared. Your debugging approach, your communication preferences, your current focus area — these are personal context that makes Claude Code more effective for *you* without imposing on others.

The collaboration pattern that works best: **CLAUDE.md carries project-level authority** (build commands, test requirements, coding standards, architectural decisions), while **Memory carries individual-level context** (user role, preferences, accumulated feedback). When a developer joins the team, CLAUDE.md gives Claude Code everything it needs to follow project rules from day one. Memory fills in over subsequent conversations as Claude Code learns that developer's specific working style.

Some teams have tried using Memory for team-level rules by sharing memory files. This is fragile — Memory files aren't designed for manual editing, they lack version control semantics, and there's no merge strategy when two people's memories conflict. If it needs to be a team rule, put it in CLAUDE.md.

## Context Window Impact: What Gets Loaded and When

Both systems consume tokens from Claude Code's context window, but they load differently.

**CLAUDE.md loads entirely at conversation start.** Every line of your CLAUDE.md file (project-level and global) is injected as system-level context before your first message. This is deterministic — the same CLAUDE.md produces the same starting context every time. The tradeoff: a bloated CLAUDE.md wastes context window space on every conversation, even when most rules aren't relevant to the current task.

**Memory loads via the MEMORY.md index.** At conversation start, Claude Code reads the MEMORY.md file — a concise index of one-line pointers to individual memory files. When a memory seems relevant (or you explicitly ask Claude to recall something), it reads the full memory file. This is a lazy-loading pattern: only relevant memories consume significant context.

This architectural difference means:

- **CLAUDE.md should be concise and essential.** Every line pays context-window rent on every conversation. Include only rules that apply broadly. If a rule only applies to one subsystem, consider using a nested CLAUDE.md in that directory instead.
- **Memory can be more verbose.** Individual memory files only load when needed. You can accumulate hundreds of memories over months without bloating every conversation. The MEMORY.md index acts as a filter — its one-line descriptions help Claude Code decide what's worth loading.

For teams concerned about context efficiency, the guideline from Claude Code's [extension stack architecture](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) is clear: keep CLAUDE.md under a few hundred lines of high-signal instructions, and let Memory handle the long tail of accumulated context.

As noted in the [long-running agent guide](/blog/effective-harnesses-for-long-running-agents), context management becomes increasingly critical over extended sessions. CLAUDE.md's fixed cost is predictable, while Memory's variable loading can help preserve context budget for the actual work.

## Staleness and Maintenance: Which System Ages Better?

**CLAUDE.md ages well when maintained alongside the code.** Because it lives in your repo, it follows the same review and update cycle as your source code. When you change a build command, you update CLAUDE.md in the same commit. When you add a new quality gate, it goes into CLAUDE.md alongside the CI config change. The instructions stay synchronized with reality because they share the same change process.

The risk: CLAUDE.md that *isn't* maintained becomes actively harmful. Outdated build commands, deprecated linter rules, or references to removed modules don't just fail to help — they cause Claude Code to do the wrong thing confidently. Stale CLAUDE.md is worse than no CLAUDE.md.

**Claude Memory has a built-in staleness problem.** Memories are written at a point in time and may not be updated when the underlying reality changes. A project memory that says "we're using Jest for testing" persists even after you migrate to Vitest. A user memory that says "new to the React side of this repo" lingers months after you've become proficient.

Claude Code partially mitigates this with a verification principle: before acting on a memory that references specific files, functions, or flags, it's supposed to check that those things still exist. But this verification isn't foolproof, and Claude Code sometimes acts on stale memories without checking.

The maintenance pattern that works: **treat CLAUDE.md as infrastructure code** (update it in the same PR that changes the thing it describes) and **periodically review MEMORY.md** to prune outdated entries. Some developers add a quarterly "memory cleanup" to their workflow — scan MEMORY.md, delete anything that's no longer true, update anything that's partially correct.

## Content Boundaries: What Belongs Where

One of the most common mistakes is putting the wrong type of information in the wrong system. Here's a clear boundary:

**CLAUDE.md is for imperative rules and project facts:**
- Build and test commands
- Quality gates (what must pass before committing)
- Coding style rules (naming conventions, file organization)
- Architectural constraints ("never import Next.js modules in pipeline scripts")
- Workflow requirements ("discuss design first for new features")
- Known gotchas that affect every developer

**Memory is for accumulated context and personal knowledge:**
- Your role and expertise level
- Corrections you've made to Claude's behavior ("don't summarize at the end of responses")
- Project decisions and their motivation ("auth rewrite is driven by compliance, not tech debt")
- Pointers to external resources ("pipeline bugs tracked in Linear project INGEST")
- Confirmed approaches ("single bundled PR was right for this refactor")

**Neither system should store:**
- Code patterns derivable from reading the codebase
- Git history or recent changes (use `git log`)
- Debugging solutions (the fix is in the code)
- Ephemeral task details (use TodoWrite for in-conversation tracking)

A useful heuristic: if the information answers "what should Claude Code always do?", it belongs in CLAUDE.md. If it answers "what does Claude Code need to know about me or this situation?", it belongs in Memory. If it answers "what did the code look like yesterday?", it belongs in neither — use git.

## When to Choose Claude Memory

Choose Claude Memory as your primary context mechanism when:

- **You're a solo developer** who doesn't need team-shared rules. Memory accumulates your preferences and project knowledge naturally without the overhead of maintaining a CLAUDE.md file.
- **You want zero-setup context building.** Memory starts working from your first conversation. No upfront writing required. Just use Claude Code normally, and it learns your patterns.
- **Your context is personal and evolving.** Your role changes, your expertise grows, your preferences shift. Memory adapts through ongoing conversations rather than requiring manual file edits.
- **You're tracking project decisions and their rationale.** Memory's project type with "Why" and "How to apply" structure captures decision context that's too detailed or transient for CLAUDE.md.
- **You rely on external systems** (issue trackers, dashboards, Slack channels). Memory's reference type creates durable pointers that Claude Code can recall when relevant.

Memory is particularly valuable for developers who use Claude Code across multiple projects. Your global user memories ("I prefer concise responses", "I'm a senior backend engineer") transfer to every project, while project-specific memories stay scoped.

The [memory importing feature](/blog/anthropic-claude-memory-upgrades-importing) also makes Memory the right choice when onboarding Claude Code into an existing workflow — you can import context from previous tools rather than rewriting it as CLAUDE.md rules.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary context mechanism when:

- **You're on a team** and need consistent Claude Code behavior across all developers. CLAUDE.md is the only way to ensure everyone's Claude Code follows the same rules.
- **You have firm, stable project rules** — build commands, test requirements, commit conventions, architectural constraints. These don't change often and should be enforced reliably.
- **You want version-controlled, reviewable instructions.** CLAUDE.md changes go through code review like any other repo change. You can blame, diff, and revert instruction changes.
- **You need deterministic behavior.** CLAUDE.md loads the same way every time. There's no ambiguity about what rules are active. This matters for CI/CD integrations where Claude Code runs automated tasks.
- **Your project has known gotchas** that trip up every new developer (and every new Claude Code session). Encoding these in CLAUDE.md prevents repeated mistakes.

CLAUDE.md is also the right choice when you're using Claude Code's [skills system](/blog/5-claude-code-skills-i-use-every-single-day). Skills reference CLAUDE.md for project context, creating a layered instruction system where CLAUDE.md provides the base rules and skill files provide task-specific guidance.

For teams adopting [agentic coding](/glossary/agentic-coding) practices at scale — where Claude Code handles significant portions of implementation, testing, and review — a well-maintained CLAUDE.md is non-negotiable. It's the difference between an agent that follows your standards and one that improvises.

## Verdict

**Use both.** The question isn't which system to choose — it's understanding what belongs in each. **CLAUDE.md is your project's rulebook**: deterministic, version-controlled, team-shared, and manually maintained. **Claude Memory is your personal notebook**: automatic, evolving, per-user, and conversation-driven. Teams that treat CLAUDE.md as infrastructure and let Memory handle the human context get the best of both systems.

If forced to start with just one: **start with CLAUDE.md** if you're on a team, because shared rules prevent more problems than personal context solves. **Start with Memory** if you're solo, because it delivers value immediately with zero setup. Then add the other system as your usage matures. ## Frequently Asked Questions

### Can Claude Memory and CLAUDE.md conflict with each other?

Yes. If CLAUDE.md says "use Jest for testing" but a stale memory says "this project migrated to Vitest," Claude Code may produce inconsistent behavior. CLAUDE.md instructions generally take priority because they load as system-level directives, but ambiguity causes problems. The fix: keep Memory pruned and ensure CLAUDE.md stays current with the codebase. When in doubt, Claude Code should trust what it observes in the code now over what either system claims.

### Does CLAUDE.md replace the need for Claude Memory entirely?

No. CLAUDE.md can't capture personal preferences, accumulated feedback, or evolving project context without becoming a bloated catch-all file. Memory handles the long tail of context that doesn't fit in a concise project rulebook — your role, your working style, corrections you've made, and pointers to external resources. Trying to put everything in CLAUDE.md wastes context window tokens on every conversation and creates a maintenance burden.

### How do I migrate important memories into CLAUDE.md?

If a memory has proven universally useful — not just for you, but for any developer on the project — promote it to CLAUDE.md. Copy the core rule (not the full memory frontmatter), write it as a clear directive, add it to the appropriate section of CLAUDE.md, and then delete the original memory file and its MEMORY.md index entry. This happens most often with feedback memories that capture genuine project rules disguised as personal preferences.

### Is there a size limit for CLAUDE.md or Memory files?

CLAUDE.md has no hard size limit, but every line consumes context window tokens on every conversation. Best practice is to keep it under a few hundred lines of high-signal content. For Memory, the MEMORY.md index is truncated after 200 lines, so keep index entries concise. Individual memory files have no practical limit, but shorter memories load faster and consume less context when accessed.

### Should I check Memory files into version control?

Generally no. Memory files live in `.claude/projects/*/memory/`, which is typically gitignored. They contain per-user context (preferences, role, feedback) that shouldn't be imposed on other team members. If you find yourself wanting to share a memory, it probably belongs in CLAUDE.md instead. The exception: teams that want to bootstrap a shared knowledge base sometimes commit a starter MEMORY.md, but this pattern is fragile and not officially supported.
