# Claude Memory vs CLAUDE.md：两套记忆系统，该用哪个？

**TL;DR:** **Claude Memory** 是 Anthropic 的跨会话持久记忆功能，自动记住用户偏好和上下文；**CLAUDE.md** 是 Claude Code 的项目级指令文件，随代码仓库分发，定义 AI 在特定项目中的行为规则。两者解决不同问题：Memory 让 AI 记住「你是谁」，CLAUDE.md 告诉 AI「这个项目怎么做」。团队协作选 CLAUDE.md，个人偏好用 Memory，最佳实践是两者配合使用。

## 概述：Claude Memory

**Claude Memory** 是 Anthropic 为 Claude 产品线引入的持久化记忆功能，让 AI 在多次对话之间保持对用户的了解。它解决的核心问题是：每次新对话都要重新解释背景信息。

Memory 的运作方式是自动从对话中提取关键信息——你的角色、偏好、项目背景、工作习惯——存储为结构化的记忆条目。下次对话时，Claude 会自动加载这些记忆，无需手动提示。用户可以通过「记住这个」或「忘掉这个」来主动管理记忆内容。

Anthropic 已将 Memory 功能[开放给免费用户](/zh/blog/anthropic-claude-memory-upgrades-importing)，并支持从 ChatGPT 导入历史数据，显示出将其作为核心差异化功能的战略定位。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级配置文件，作用类似于代码仓库的 `.editorconfig` 或 `.eslintrc`——但它配置的不是编辑器，而是 AI 助手的行为。CLAUDE.md 文件 checked into 代码仓库，团队成员共享同一套规则。

CLAUDE.md 解决的问题是：如何让 AI 理解一个项目的具体约束，而不需要每次都口头说明。它包含编码规范、架构决策、测试要求、部署流程等项目特有的指令。Claude Code 在每次会话开始时自动读取这些文件，确保行为一致。

配合 [SKILL.md 技能文件](/zh/blog/9-principles-writing-claude-code-skills)使用时，CLAUDE.md 构成了 Claude Code [七层架构](/zh/blog/claude-code-seven-programmable-layers)的核心配置层。

## 核心差异：记忆范围与生命周期

两套系统的根本区别在于它们「记住什么」以及「为谁记住」。

**Claude Memory** 是以用户为中心的。它记住的是关于你这个人的信息：你是后端工程师、偏好函数式编程、不喜欢过度抽象、项目截止日期是下周五。这些信息跟随你的账户，无论你在哪个对话中都生效。它的优势是零配置——对话中自然表达的偏好会被自动捕获。

**CLAUDE.md** 是以项目为中心的。它定义的是这个代码仓库的规则：用 TypeScript 不用 JavaScript、测试必须覆盖 80%、commit message 用中文、部署前必须跑 lint。这些规则属于项目，不属于个人。新成员 clone 仓库后立即获得相同的 AI 行为。

实际使用中，Memory 更像是「AI 对你的了解」，CLAUDE.md 更像是「AI 的工作手册」。两者服务于不同层次的上下文需求，详见我们对 [Claude Code 记忆系统的完整解析](/zh/blog/claude-code-memory)。

## 核心差异：协作与一致性

在团队场景下，两套系统的差异更加明显。

CLAUDE.md 的最大优势是**团队一致性**。当团队约定「所有 API 路由必须有错误处理中间件」这条规则写入 CLAUDE.md 后，无论哪位开发者使用 Claude Code，AI 都会遵守这条规则。这消除了「AI 对不同人表现不一致」的问题。更重要的是，规则变更通过 Pull Request 流程管理——可以讨论、审查、追溯。

Claude Memory 在协作方面存在天然局限。它是个人的、不可共享的。你的 Memory 里可能存着「这个项目用 Vitest 不用 Jest」，但新同事的 Memory 里没有这条信息。他们需要通过对话重新「教会」AI，或者踩一次坑才能积累这条记忆。

对于[企业级 Claude Code 落地](/zh/blog/claude-code-enterprise-engineering-ramp-shopify-spotify)，CLAUDE.md 是标准化 AI 行为的基础设施，Memory 则是个人效率的增量优化。

## 核心差异：灵活性与维护成本

**Claude Memory** 的优势在于零维护。你不需要编写、更新、或版本管理任何文件。AI 从对话中自主学习，自动淘汰过时信息。缺点是不够精确——你无法控制 AI 到底「记住了什么」，也可能记错或遗漏关键信息。

**CLAUDE.md** 需要主动维护。项目架构变了、规范更新了、新工具引入了，都需要同步更新文件。好处是精确可控——你写什么 AI 就执行什么，没有歧义。配合 [Hooks 系统](/zh/blog/claude-code-hooks-mastery)还可以实现自动化验证，确保 AI 行为符合预期。

维护成本的取舍是：Memory 省前期投入但控制力弱，CLAUDE.md 投入更大但回报确定。对于严肃的工程项目，CLAUDE.md 的确定性通常更有价值。

## 什么时候选 Claude Memory

以下场景优先使用 Claude Memory：

- **个人项目和探索**：快速原型、学习新技术时，不值得专门写配置文件
- **跨项目通用偏好**：编码风格、沟通语言、输出格式等不随项目变化的偏好
- **非 Claude Code 场景**：在 Claude.ai 网页端或移动端使用时，CLAUDE.md 不可用
- **渐进式学习**：希望 AI 逐渐了解你的工作方式，而不是一次性定义所有规则
- **低成本启动**：不想在项目早期就花时间写详细指令

Memory 的理想用户画像是：独立开发者、非工程场景（写作、研究、咨询），以及需要 AI 记住人际/商业上下文的用户。

## 什么时候选 CLAUDE.md

以下场景优先使用 CLAUDE.md：

- **团队项目**：需要所有成员获得一致的 AI 行为
- **关键质量门禁**：测试覆盖率、代码审查流程、部署检查清单等不可协商的规则
- **复杂项目约束**：特殊架构决策、已知陷阱、技术债务的上下文
- **合规要求**：需要审计 AI 行为规则的变更历史
- **持续集成**：将 AI 行为标准纳入 CI/CD 流程

CLAUDE.md 的核心价值在于[将 AI 行为代码化](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。它不是「建议」，而是「指令」——AI 必须遵守，违反即为 bug。

## 最佳实践：两者配合使用

成熟的工作流不是二选一，而是分层使用：

1. **CLAUDE.md**（项目层）：定义技术规范、架构约束、质量门禁
2. **SKILL.md**（任务层）：定义特定任务的执行方式（写测试、做 Code Review）
3. **Claude Memory**（个人层）：存储个人偏好、角色信息、通用习惯

这种分层架构确保：团队规则通过文件强制执行，个人效率通过 Memory 持续提升。当个人偏好与项目规则冲突时，CLAUDE.md 的优先级更高——这是正确的，因为项目规则是团队共识。

想了解如何从零搭建这套体系，参考我们的 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)。

## 判决

**如果你在团队中工作、项目有明确规范，CLAUDE.md 是必选项**——它是工程纪律的延伸，不是可选的便利工具。Claude Memory 则是锦上添花，让日常交互更顺畅。独立开发者可以只依赖 Memory 起步，但一旦项目复杂度上升或有协作需求，CLAUDE.md 的投入回报比远高于记忆系统。

最务实的建议：先花 15 分钟写好 CLAUDE.md（项目约束 + 关键规则），让 Memory 自然积累个人偏好。两者协同时效果最佳。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，且推荐同时使用。CLAUDE.md 管项目级规则（团队共享），Memory 管个人偏好（账户绑定）。两者作用于不同层级，互不冲突。当内容冲突时，CLAUDE.md 的指令优先级更高。

### CLAUDE.md 放在仓库哪个位置？

放在项目根目录。Claude Code 启动时自动读取根目录的 `CLAUDE.md`。子目录也可以有自己的 CLAUDE.md，用于覆盖或补充根级规则。全局级的个人偏好则放在 `~/.claude/CLAUDE.md`。

### Claude Memory 能否替代 CLAUDE.md？

不能。Memory 是个人的、模糊的、不可审计的——不适合作为团队规范的载体。CLAUDE.md 是精确的、共享的、可版本控制的。两者解决不同层次的问题，没有替代关系。

### 如何迁移 ChatGPT 的记忆到 Claude？

Anthropic 已支持从 ChatGPT 导入记忆数据。在 Claude 设置中选择导入选项，上传从 OpenAI 导出的数据文件即可。详见我们的[记忆升级详解](/zh/blog/anthropic-claude-memory-upgrades-importing)。
