**Where they overlap — and where conflicts arise:** If CLAUDE.md says "use conventional commits" and your Memory says "commit messages should be terse one-liners," Claude Code will follow CLAUDE.md. Project instructions take precedence over personal memory for behavioral rules. Memory wins for personal preferences that don't contradict project rules — like your preferred level of explanation detail or which terminal commands you want Claude Code to run without asking.

## Content Types: What Goes Where

Choosing the right persistence layer comes down to one question: **should this apply to everyone on the project, or just to me?**

### What belongs in CLAUDE.md

CLAUDE.md is for **project truths** — things that are true regardless of who's working:

- **Build and test commands**: `npm run build`, `npm test`, `cargo check` — the commands that validate correctness
- **Quality gates**: "All tests must pass before commit," "run linting before push"
- **Architecture constraints**: "Don't import Next.js modules in pipeline scripts," "use snake_case for database columns"
- **Style guides**: "Commit messages must describe what changed," "no TODO comments without issue links"
- **Known gotchas**: "CJK content must use character-based word count, not space-delimited"
- **Workflow rules**: "New features require design discussion before coding," "pipeline changes require validation"

These are project-level invariants. They belong in version control where they can be reviewed, debated, and evolved alongside the code. Our guide on [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) covers how to write CLAUDE.md instructions that produce consistent results.

### What belongs in Memory

Memory is for **personal and temporal context** — things specific to you or to the current state of work:

- **User profile**: "Senior backend engineer, new to React" — so Claude Code calibrates explanations appropriately
- **Feedback corrections**: "Don't mock the database in integration tests" — a lesson learned that should persist
- **Project state**: "Merge freeze starts May 15 for mobile release" — temporary but important context
- **External references**: "Deploy logs are in the #eng-deploys Slack channel" — a pointer you shouldn't have to repeat
- **Workflow preferences**: "I prefer terse responses with no trailing summaries" — personal style that shouldn't be imposed on the team

Memory excels at capturing the kind of context that emerges organically from working sessions. When you correct Claude Code — "no, don't use that testing pattern" — it saves a feedback memory so the correction sticks across future conversations. When you mention your role or expertise level, it saves a user memory to calibrate future interactions.

### The gray zone

Some information could plausibly go in either place. A useful heuristic: **if removing it would break the build or violate team standards, it belongs in CLAUDE.md. If removing it would just make Claude Code slightly less helpful for you personally, it belongs in Memory.**

For example, "we use Vitest, not Jest" is a CLAUDE.md instruction — using the wrong test framework breaks things. But "I prefer seeing test output in verbose mode" is a Memory preference — it's personal and non-breaking.

## Durability and Maintenance

CLAUDE.md and Memory have fundamentally different maintenance profiles, and understanding this prevents common mistakes.

**CLAUDE.md ages like code.** It's tied to the repo's evolution. When you refactor from Express to Hono, you update the CLAUDE.md build commands. When you add a new linting rule, you document it. Because CLAUDE.md is committed to git, it goes through code review — teammates can flag stale instructions, and the git history shows when and why instructions changed. The maintenance burden is explicit and shared.

**Memory ages like notes.** It can become stale without anyone noticing. A project memory from March saying "the auth rewrite is blocked on legal review" might be irrelevant by May. Claude Code includes guardrails — it's instructed to verify memory claims against current code before acting on them — but stale memories still waste context and can mislead. Regular pruning matters.

The practical implication: **put durable, structural information in CLAUDE.md** and **temporal, situational information in Memory**. "We use SQLite for the content database" is a CLAUDE.md fact — it'll be true until a migration happens. "The SQLite migration to Postgres is planned for Q3" is a project memory — it's useful context with a natural expiration date.

## Team Dynamics: Shared vs Personal

The shared-vs-personal distinction has real consequences for team workflows.

**CLAUDE.md creates team alignment.** When every developer's Claude Code instance follows the same instructions, you get consistent behavior: same commit message format, same test requirements, same architectural constraints. This is especially valuable for onboarding — a new developer gets Claude Code working correctly for your project just by cloning the repo. No setup, no "ask Sarah how she configured her AI tool."

**Memory creates individual effectiveness.** A senior developer's Memory might contain deep project context and terse-response preferences. A junior developer's Memory might note that they're learning TypeScript and prefer detailed explanations. Both work in the same project with the same CLAUDE.md, but Claude Code adapts its communication style to each person.

**The tension:** some teams try to put personal preferences in CLAUDE.md ("always explain your reasoning step by step"), which creates friction for experienced developers who find verbose explanations annoying. Conversely, some developers put project-critical instructions in Memory instead of CLAUDE.md, which means the knowledge doesn't propagate to teammates. ## When to Choose Claude Memory

Use Memory as your primary context mechanism when:

- **You're a solo developer.** The shared-vs-personal distinction matters less when there's no team. Memory's automatic capture is more convenient than manually maintaining CLAUDE.md for personal preferences.
- **Context is temporal.** Sprint goals, incident status, upcoming deadlines — these have natural expiration dates. Memory handles ephemeral context better than a version-controlled file.
- **You want automatic capture.** Memory builds itself from your conversations. If you correct Claude Code's approach, that correction persists without you writing documentation.
- **Different team members need different behavior.** A mixed-experience team benefits from Memory's per-user calibration. The senior engineer gets concise responses; the intern gets detailed explanations.
- **You're tracking external systems.** "Bugs are in Linear project INGEST" and "the oncall dashboard is at grafana.internal/d/api-latency" are reference memories that don't belong in a committed file.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary context mechanism when:

- **You're on a team.** Any instruction that should apply to all developers belongs in CLAUDE.md. Build commands, test requirements, architectural constraints, and coding standards are shared concerns.
- **Consistency matters more than personalization.** If you need every Claude Code instance to produce the same commit message format or follow the same PR template, CLAUDE.md is the only reliable mechanism.
- **Instructions need review and accountability.** CLAUDE.md goes through pull requests. Memory is edited by Claude Code itself, with no review gate. For critical instructions — security constraints, deployment procedures, data handling rules — the review process matters.
- **You want nested, directory-level instructions.** CLAUDE.md supports hierarchical overrides: a CLAUDE.md in `packages/api/` can add API-specific instructions on top of the root CLAUDE.md. Memory doesn't have this spatial awareness.
- **The project has known gotchas.** Every project has landmines — "don't import server-only modules in scripts," "CJK word count uses characters not spaces." These belong in CLAUDE.md where every contributor (human and AI) sees them. ## Verdict

**Use both — they solve different problems.** CLAUDE.md is your project's constitution: the shared rules, constraints, and commands that every Claude Code session must follow. Memory is your personal assistant's notebook: your preferences, your corrections, your situational context. Trying to use only one creates gaps.

The practical setup for most teams: **invest heavily in CLAUDE.md** for project instructions, build commands, quality gates, and architectural constraints. **Let Memory accumulate naturally** from your working sessions — correct Claude Code when it does something wrong, and those corrections persist. Periodically prune stale memories, especially project-type memories with time-sensitive information.

If you're starting from zero, **write CLAUDE.md first**. A good CLAUDE.md immediately makes Claude Code useful for your project. Memory builds itself over time. ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions take precedence over Memory for project-level behavior. If CLAUDE.md says "use conventional commits" and your Memory says "use terse commit messages," Claude Code follows CLAUDE.md. Memory only overrides for personal preferences that don't conflict with project rules — like response verbosity or explanation depth.

### Does Claude Memory sync across devices?

Claude Memory is stored locally in `.claude/projects/` and is not synced across devices or shared with teammates. Each machine maintains its own Memory files. If you switch computers, you start with a fresh Memory that builds up again from your conversations. CLAUDE.md, being committed to git, syncs automatically when you pull the repo.

### Should I commit my Memory files to git?

No. Memory files are personal and often contain user-specific preferences, corrections, and temporal context that would create noise in version control. The `.claude/` directory is typically gitignored. Any instruction important enough to share with the team should be moved from Memory into CLAUDE.md through a pull request.

### How do I migrate useful Memory into CLAUDE.md?

Review your Memory files in `.claude/projects/<hash>/memory/` periodically. Any feedback or project memory that applies to all developers — not just you — should be rewritten as a CLAUDE.md instruction and submitted as a PR. Delete the Memory entry afterward to avoid duplication. Common candidates: testing patterns, architectural constraints, and recurring gotchas.

### What happens when Memory gets too large?

Claude Code loads the `MEMORY.md` index into every conversation, with lines after 200 truncated. Individual memory files are only loaded when deemed relevant. If your Memory grows large, prune stale entries — especially project memories with expired deadlines and feedback memories for patterns you've since changed. Quality matters more than quantity; a focused set of 20 memories outperforms 100 outdated ones.
