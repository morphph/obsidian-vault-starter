# Claude Memory vs CLAUDE.md：哪种记忆机制更适合你的工作流？

**TL;DR：** **CLAUDE.md** 是你写给 AI 的规则手册——团队共享、版本可控、确定性强；**Claude Memory** 是 AI 自己记的笔记——跨会话自动积累，个人化且无需手动维护。两者不是替代关系，而是互补的双层记忆架构。规则放 CLAUDE.md，个人偏好和项目动态交给 Memory。

## 概述：Claude Memory（自动记忆）

**Claude Memory** 是 Claude Code 的自动笔记系统，它在对话过程中主动识别值得记住的信息，并将其写入 `.claude/projects/` 目录下的 markdown 文件。这些记忆在后续会话中自动加载，让 Claude 能「记住」你的偏好、项目背景和工作习惯。

Memory 的核心价值在于**零维护成本**。你不需要手动编写任何文件——Claude 会自己判断哪些信息值得保存。它特别擅长记录那些难以文档化的隐性知识：你喜欢的代码风格、项目的历史决策、团队成员的分工等。根据 [Anthropic 的记忆系统升级公告](/zh/blog/anthropic-claude-memory-upgrades-importing)，Memory 支持从其他 AI 工具导入已有上下文，进一步降低了冷启动成本。

## 概述：CLAUDE.md（规则手册）

**CLAUDE.md** 是 Claude Code 的项目级配置文件，放在仓库根目录，随代码一起版本管理。它定义了 Claude 在该项目中必须遵守的规则：构建命令、代码风格、架构约束、工作流要求等。

CLAUDE.md 的核心价值在于**确定性和可共享性**。团队中每个人拉取代码后，Claude 的行为都一致——因为规则文件是代码的一部分。它解决的是「AI 要怎么做事」的问题，而不是「AI 记得什么」。关于 CLAUDE.md 的完整工程实践，可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 胜出 |
|------|--------------|-----------|------|
| **存储位置** | `.claude/projects/` 本地目录 | 项目根目录（进入 git） | 取决于需求 |
| **创建方式** | AI 自动生成 | 人工手写 | Memory |
| **版本管理** | 不进入 git | 随代码版本管理 | CLAUDE.md |
| **团队共享** | 个人私有 | 团队共享 | CLAUDE.md |
| **内容类型** | 偏好、上下文、动态信息 | 规则、约束、命令 | 平手 |
| **维护成本** | 接近零 | 需要手动更新 | Memory |
| **确定性** | 可能过时或冲突 | 始终反映当前规则 | CLAUDE.md |
| **生效范围** | 当前用户 + 当前项目 | 所有使用该仓库的人 | CLAUDE.md |

## 工作机制对比：谁在什么时候生效

Claude Code 启动时会同时加载两种记忆源。**CLAUDE.md** 作为系统级指令注入，优先级最高——它定义了「必须做」和「绝不做」的硬性规则。**Memory** 作为上下文参考注入，提供背景信息和软性偏好。

具体加载顺序：

1. 全局 `~/.claude/CLAUDE.md`（用户级规则）
2. 项目 `./CLAUDE.md`（项目级规则）
3. `.claude/projects/` 下的 Memory 文件（自动记忆）

当两者冲突时，CLAUDE.md 的显式规则优先。例如，如果 Memory 记录了「用户喜欢用 tab 缩进」，但 CLAUDE.md 写了「本项目统一用 2 空格缩进」，Claude 会遵守 CLAUDE.md。

这种分层设计意味着你可以放心让 Memory 自由积累，不用担心它覆盖团队规范。了解完整的七层架构设计，参见 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 适用场景分析：什么信息该放哪里

选择存储位置的核心判断标准：**这条信息是给「这个项目的所有人」还是给「我自己」？**

### 该放 CLAUDE.md 的信息

- 构建和测试命令（`npm run build`、`npm test`）
- 代码风格强制规则（命名约定、文件组织）
- 架构约束（「不要在 pipeline 脚本里 import Next.js 模块」）
- Git 工作流要求（commit message 格式、分支策略）
- 质量门禁（提交前必须通过的检查）

### 该交给 Memory 的信息

- 用户角色和技术背景（「我是后端工程师，前端不太熟」）
- 个人工作偏好（「我喜欢先看方案再动手」）
- 项目动态信息（「本周在做 auth 模块重构」）
- 反馈和修正（「不要在回复末尾加总结」）
- 外部资源指引（「bug 追踪在 Linear 的 INGEST 项目」）

## 实战建议：最佳组合策略

单独使用任何一种都不够。最有效的做法是建立**双层记忆架构**：

**第一层：CLAUDE.md 定义边界。** 把不可协商的规则写进去——构建流程、测试要求、代码约束。这是给团队所有人（和所有 AI 会话）的硬性指令。参考我们的 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)，同样的原则适用于 CLAUDE.md 的编写。

**第二层：Memory 积累上下文。** 让 Claude 自动记录你的偏好和项目进展。随着会话积累，Claude 会越来越了解你的工作方式，减少重复沟通。

**维护节奏：**

- CLAUDE.md：代码发生架构变化时更新，作为 PR 的一部分提交
- Memory：每月检查一次，删除过时记录，修正不准确的内容

关于如何让 Claude Code 真正成为你的工程伙伴而不只是工具，推荐阅读 [Claude Code 不是一个编程工具](/zh/blog/claude-code-is-not-a-coding-tool)——它的观点正好契合双层记忆的设计哲学。

## 判断决策树

如果你不确定某条信息该放哪里，用这个决策流程：

1. **这条信息会过期吗？** → 是 → Memory（项目动态、临时决策）
2. **团队其他人需要知道吗？** → 是 → CLAUDE.md（规则、约束）
3. **这是个人偏好还是项目要求？** → 个人偏好 → Memory
4. **违反这条规则会导致 bug 或构建失败吗？** → 是 → CLAUDE.md
5. **这条信息只对 AI 有用，人读了没意义？** → 是 → Memory

## 结论

**CLAUDE.md 是法律，Memory 是经验。** 法律写在明处、全员遵守、不会自己变化；经验由个人积累、因人而异、随时间进化。一个成熟的 Claude Code 工作流需要两者配合——用 CLAUDE.md 保证底线，用 Memory 提升效率。

如果你刚开始用 Claude Code，先写好 CLAUDE.md（参考 [我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)），Memory 会自己慢慢长出来。

## 常见问题

### Claude Memory 会不会覆盖 CLAUDE.md 的规则？

不会。CLAUDE.md 的指令优先级高于 Memory。当两者冲突时，Claude 始终遵守 CLAUDE.md 中的显式规则。Memory 提供的是补充上下文，不是覆盖规则。

### Memory 文件可以手动编辑吗？

可以。Memory 存储为普通 markdown 文件，你可以直接编辑、删除或重组。但通常不需要——让 Claude 自动管理即可，只在发现错误记忆时手动修正。

### 一个项目的 CLAUDE.md 应该多长？

没有硬性限制，但建议控制在 200 行以内。过长的 CLAUDE.md 会占用上下文窗口，降低 Claude 处理实际任务的能力。把详细指令拆分到 `skills/*/SKILL.md` 文件中，CLAUDE.md 只保留全局规则。

### 如果我换电脑，Memory 会丢失吗？

会。Memory 存储在本地 `.claude/` 目录，不进入 git。如果需要跨设备同步，可以手动备份 `.claude/projects/` 目录，或者依靠 Anthropic 的云端记忆功能（如已启用）。
