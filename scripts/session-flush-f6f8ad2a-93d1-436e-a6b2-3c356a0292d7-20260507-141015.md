# Claude Memory vs CLAUDE.md: Which Context System Should You Use?

**TL;DR:** **CLAUDE.md** is your project's explicit instruction manual — checked into git, shared with your team, loaded every session. **Claude Memory** is your personal, auto-generated knowledge base that grows across conversations. Use CLAUDE.md for project rules and standards; use Memory for personal preferences and learned context. Most teams need both.

## Overview: Claude Memory

Claude Memory is Claude Code's persistent, file-based knowledge system that automatically accumulates information across conversations. It stores user preferences, project context, feedback on working style, and references to external resources in individual markdown files under `.claude/projects/*/memory/`. Memory is relevance-based — Claude reads its memory index at session start and accesses specific memories when they seem relevant to the current task.

The key characteristic of Memory is that it's **auto-generated and personal**. You don't manually write memory files (though you can ask Claude to remember something). It learns from your corrections, confirmations, and the context you share. Memory is local to your machine — it doesn't live in the repo and isn't shared with teammates.

## Overview: CLAUDE.md

CLAUDE.md is a markdown instruction file that lives in your repository root (and optionally in subdirectories). It provides explicit, deterministic instructions that Claude Code reads at the start of every session — build commands, coding standards, architecture decisions, workflow rules, and project-specific constraints. Think of it as a README specifically for your AI coding agent.

The key characteristic of CLAUDE.md is that it's **manually authored and shared**. You write it deliberately, check it into git, and every team member's Claude Code session loads the same instructions. It's version-controlled, reviewable in PRs, and evolves with your codebase. CLAUDE.md is part of Claude Code's [seven programmable layers](/blog/claude-code-seven-programmable-layers) — the project-level configuration layer.

## Context Loading: How They Work Together

Claude Code's context system is hierarchical. CLAUDE.md files form the deterministic base layer — they're always loaded, always authoritative. Memory sits on top as a supplementary layer that personalizes behavior without overriding project rules.

When Claude Code starts a session, it reads CLAUDE.md first (project instructions), then scans the memory index for relevant entries. If a memory conflicts with a CLAUDE.md instruction, the CLAUDE.md instruction wins. This hierarchy means teams can enforce standards via CLAUDE.md while individual developers still benefit from personalized memory.

For example, your CLAUDE.md might specify "use Vitest for all tests" and "commit messages must follow Conventional Commits." Your memory might store "this user prefers terse responses" and "they're a senior engineer comfortable with advanced TypeScript patterns." Both inform Claude's behavior, but at different levels. Read more about how these layers interact in our [complete guide to Claude Code](/blog/claude-code-complete-guide).

## Team Workflows: Detailed Analysis

The team dimension is where these two systems diverge most sharply. CLAUDE.md solves the problem of **consistency** — every engineer on your team gets the same AI behavior for project-specific tasks. When you add a build command, a lint rule, or an architectural constraint to CLAUDE.md, it immediately applies to everyone who pulls that change.

Memory solves the problem of **personalization**. A junior developer and a staff engineer working on the same repo have different needs. Memory lets Claude adapt its communication style, explanation depth, and workflow suggestions to each individual without polluting the shared project instructions.

The practical implication: put anything that should be universal in CLAUDE.md. Put anything that's about *you* — your role, your preferences, your working style — in memory (or let Claude learn it naturally). Teams that mix these up end up with bloated CLAUDE.md files full of one person's preferences, or with memory entries that should be shared standards.

For teams adopting Claude Code at scale, this separation is critical. Our coverage of [how enterprises like Shopify and Spotify use Claude Code](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify) shows that CLAUDE.md standardization is typically the first step in team rollouts.

## Content Types: Detailed Analysis

What belongs in each system follows a clear rule: **if removing it would break the project for another developer, it goes in CLAUDE.md. If removing it would only affect your personal experience, it belongs in memory.**

**CLAUDE.md content** (shared, explicit):
- Build and test commands (`npm run build`, `npm test`)
- Code style rules and linting configuration
- Architecture decisions and constraints
- Workflow requirements (e.g., "run tests before commit")
- Project-specific gotchas and known issues
- Documentation update rules

**Memory content** (personal, learned):
- Your role and expertise level
- Communication preferences (verbose vs. terse)
- Feedback on past interactions ("don't summarize at the end")
- Project context not derivable from code (who's working on what, deadlines)
- References to external systems (Linear projects, Slack channels)

The [Claude Code memory system documentation](/blog/claude-code-memory) covers the full taxonomy of memory types: user, feedback, project, and reference memories each serve distinct purposes.

## Maintenance and Evolution

CLAUDE.md requires deliberate maintenance. As your project evolves — new build tools, changed conventions, updated dependencies — someone needs to update the file. Stale CLAUDE.md instructions actively harm productivity because Claude follows them faithfully even when they're outdated. Treat CLAUDE.md like infrastructure code: review changes in PRs, keep it accurate.

Memory is largely self-maintaining. Claude updates memories when it learns new information and can remove stale entries. However, memories can become outdated if project context shifts significantly. The system includes freshness checks — Claude verifies memory claims against current code state before acting on them.

The maintenance tradeoff: CLAUDE.md is more work to maintain but more reliable (you control exactly what it says). Memory is zero-maintenance but occasionally needs correction (ask Claude to "forget" something outdated).

## When to Choose Claude Memory

Memory is the right system when:

- You want Claude to **adapt to your personal style** without configuring anything explicitly
- You're capturing **cross-session context** that's specific to you (your role, your current focus area)
- You need to store **feedback** about how Claude should work with you specifically
- You're tracking **project state** that changes too frequently for CLAUDE.md (current sprint goals, active incidents)
- You want **zero-maintenance persistence** — just work normally and let Claude learn

Memory shines for solo developers and for personalizing the experience on team projects where CLAUDE.md handles the shared rules.

## When to Choose CLAUDE.md

CLAUDE.md is the right system when:

- You need **team-wide consistency** — everyone's Claude should follow the same rules
- You're encoding **project standards** that are non-negotiable (test requirements, commit conventions)
- You want **version-controlled, reviewable** AI instructions that evolve with your codebase
- You're defining **build commands and workflows** that are project-specific
- You need **deterministic behavior** — the same instructions, every session, for every developer

CLAUDE.md is essential for any project with more than one contributor. Even solo developers benefit from it for complex projects where you'd otherwise repeat the same context every session. See our guide on [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) for practical CLAUDE.md writing patterns.

## Verdict

**Use both.** CLAUDE.md and Claude Memory aren't competing systems — they're complementary layers. **CLAUDE.md is your project's constitution**: the shared, explicit, version-controlled rules that ensure consistent AI behavior across your team. **Memory is your personal context layer**: the learned preferences and accumulated knowledge that make Claude more effective for you specifically over time.

Start with CLAUDE.md — it delivers immediate value by eliminating repeated instructions. Memory will build itself naturally as you work. The developers getting the most out of Claude Code treat CLAUDE.md as team infrastructure (reviewed in PRs, kept current) and let memory handle everything personal. ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions take priority over memory entries. If your CLAUDE.md says "always use TypeScript strict mode" but a memory entry suggests otherwise, Claude follows the CLAUDE.md rule. Memory supplements project instructions — it never overrides them.

### Does Claude Memory sync across machines?

No. Memory is stored locally on your machine under `.claude/projects/`. If you work on multiple machines, each will develop its own memory independently. CLAUDE.md, being checked into git, syncs everywhere you pull the repo.

### How big should CLAUDE.md be?

Keep it focused — typically 50-200 lines for most projects. Include only what Claude needs to know that isn't obvious from the code itself. If your CLAUDE.md exceeds 300 lines, consider whether some content belongs in skill files (`skills/*/SKILL.md`) instead, which are loaded on-demand rather than every session.

### Can I see what Claude has memorized?

Yes. Memory files are plain markdown stored in `.claude/projects/*/memory/`. You can read, edit, or delete them directly. The `MEMORY.md` index file lists all stored memories with one-line descriptions.

### Should I check memory files into git?

No. Memory files are personal and should stay in `.claude/` (which is typically gitignored). If you find yourself wanting to share a memory entry with your team, that's a signal it belongs in CLAUDE.md instead.
