# Claude Memory vs CLAUDE.md：哪个才是你该用的上下文管理方式？

**TL;DR：** **CLAUDE.md** 是团队共享的确定性指令文件，适合编码规范、架构约束和工作流规则；**Claude Memory** 是个人自动记忆系统，适合积累用户偏好、项目背景和反馈修正。两者不是替代关系——**成熟的 Claude Code 工作流应该同时使用两者**，CLAUDE.md 管"规则"，Memory 管"认知"。

## 概述：Claude Memory

Claude Memory 是 Claude Code 的自动持久化记忆系统，数据存储在 `.claude/projects/` 目录下的 markdown 文件中。它的核心设计理念是：**让 AI 跨对话积累对你和你项目的理解**，而不需要你每次重新解释上下文。

Memory 分为四种类型：用户记忆（你的角色和偏好）、反馈记忆（你对 AI 行为的修正）、项目记忆（进行中的工作和决策背景）、引用记忆（外部系统的指向）。Claude Code 会根据对话内容自动判断哪些信息值得保存——你纠正它的错误、告诉它你的技术栈偏好、提到一个截止日期，这些都可能被写入记忆。

关键特性：Memory 是**个人级别**的，不会被 git 追踪，也不会与团队成员共享。它更像是 AI 给自己记的"工作笔记"。详细机制可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是放在项目根目录的指令文件，本质上是一份给 Claude Code 的"规则手册"。它会在每次对话开始时被自动加载到上下文中，确保 AI 的行为符合项目约定。

CLAUDE.md 的内容通常包括：构建命令（`npm run build`、`npm test`）、代码风格约定、架构决策、已知陷阱（gotchas）、提交规范等。它被设计为**签入代码仓库**的文件——意味着团队每个成员用 Claude Code 时都会读取同一份指令。

CLAUDE.md 还支持层级结构：项目根目录的 CLAUDE.md 提供全局规则，子目录的 CLAUDE.md 可以覆盖或补充特定模块的约定。这种设计让大型 monorepo 也能精细化管理 AI 行为。更多关于这套分层架构的解析，见 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **持久化方式** | 自动写入 `.claude/projects/` | 手动编写，签入 git | 各有所长 |
| **共享范围** | 仅限个人 | 整个团队 | CLAUDE.md |
| **内容类型** | 偏好、反馈、项目背景 | 规则、约束、命令 | 各有所长 |
| **可控性** | AI 自主决定存储 | 完全由人类控制 | CLAUDE.md |
| **跨对话生效** | 是 | 是 | 平手 |
| **适应性** | 随对话动态演化 | 需要手动更新 | Memory |
| **可审计性** | 较弱（分散的 markdown 文件） | 强（git history 可追踪） | CLAUDE.md |
| **上手成本** | 零（自动运行） | 需要编写和维护 | Memory |

## 核心差异一：确定性 vs 自适应

这是两者最根本的区别。

CLAUDE.md 是**确定性的**——你写什么，AI 就遵守什么。"commit message 用中文"、"测试必须通过才能提交"、"不要在 pipeline 脚本里直接 import Next.js 模块"——这些规则每次都会被执行，没有歧义，没有遗忘。对于团队协作来说，这种确定性至关重要：所有人的 Claude Code 行为一致，代码风格统一，质量门禁可靠。

Memory 是**自适应的**——它从交互中学习，逐步建立对你的理解。第一次告诉 Claude Code 你是数据科学家，它记住了；第二次你纠正它"不要在这类测试里 mock 数据库"，它也记住了。下次对话开始时，它已经知道该怎么和你协作。这种渐进式的知识积累无法用静态文件预先定义。

**决策规则：如果一条信息对团队所有成员都适用且不应有歧义，放 CLAUDE.md；如果它只对你个人有意义或会随时间变化，交给 Memory。**

## 核心差异二：团队共享 vs 个人私有

CLAUDE.md 签入 git，走 PR review 流程，所有团队成员共享同一份规则。这意味着新成员 clone 仓库后，Claude Code 立刻就知道项目的编码规范和架构约定。这是一种**可版本控制的 AI 配置**。

Memory 存储在本地的 `.claude/` 目录，不进入版本控制。你的 Memory 记录了"这个用户喜欢简洁的回复"、"上次他让我用 Vitest 而不是 Jest"，这些是你个人的协作偏好，不需要也不应该强加给团队其他人。

这种分离设计很合理：团队规则和个人偏好本就不该混在一起。但它也带来一个实际问题——当你换了一台机器，Memory 不会自动跟过去。CLAUDE.md 会，因为它在 git 里。

## 核心差异三：维护成本与演化路径

CLAUDE.md 需要主动维护。添加了新的构建步骤？改了数据库 schema？引入了新的 lint 规则？你需要手动更新 CLAUDE.md，否则 AI 会按照过时的指令行事。在快速迭代的项目中，CLAUDE.md 的维护成本不可忽视。

Memory 的维护几乎是零成本——Claude Code 自动决定什么值得记住，什么需要更新。当它发现一条记忆与当前代码状态不符时，它应该主动验证并更新。但"自动"也意味着你对存储内容的控制力有限：Claude Code 可能记住了一些你不在意的信息，也可能遗漏了你觉得重要的细节。

**实际建议：每两周 review 一次 CLAUDE.md，确保它与项目现状同步。Memory 则可以偶尔检查 `.claude/projects/` 下的文件，删除过时或不相关的记录。**

Anthropic 在 2026 年初对 [Claude 记忆系统做了重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，包括支持从 ChatGPT 导入数据、免费用户开放记忆功能，这进一步降低了 Memory 的使用门槛。

## 什么时候用 Claude Memory

Memory 在以下场景中价值最高：

- **长期个人助手模式**：你每天都用 Claude Code 写代码，希望它逐渐理解你的编码习惯、技术栈偏好和沟通风格
- **项目背景积累**：正在进行中的 initiative、已知的技术债务、关键决策的背景——这些信息太琐碎不适合放在 CLAUDE.md，但对 AI 理解上下文很有帮助
- **反馈闭环**：你纠正了 AI 的某个行为（"不要在 PR 末尾加 summary"），Memory 确保这个修正在未来所有对话中生效
- **外部系统指向**：Linear 项目在哪、Grafana 看板的 URL、谁负责哪个模块——这些"指路牌"信息用 Memory 存储最合适

## 什么时候用 CLAUDE.md

CLAUDE.md 在以下场景中不可替代：

- **团队规范执行**：所有成员必须遵守的编码风格、提交规范、质量门禁（构建必须通过、测试不能跳过）
- **架构约束**：已知陷阱（"不要在 pipeline 脚本里直接 import Next.js 模块"）、技术决策（为什么选了 SQLite 而不是 Postgres）
- **新成员 onboarding**：一份好的 CLAUDE.md 就是最好的项目指南，新人 clone 仓库后 Claude Code 立刻就能正确工作
- **可审计的规则变更**：通过 git history 追踪规则的演化——谁在什么时候加了什么约束，为什么

想了解如何构建一套完整的 Claude Code 扩展体系（包括 CLAUDE.md、Skills、Hooks、MCP），参考 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 最佳实践：两者协同使用

成熟的 Claude Code 工作流不是在 Memory 和 CLAUDE.md 之间二选一，而是让它们各司其职：

1. **CLAUDE.md 管硬规则**：构建命令、lint 配置、提交规范、架构红线——这些写死在 CLAUDE.md 里，不留解释空间
2. **Memory 管软认知**：你的角色背景、协作偏好、进行中的工作上下文——这些交给 Memory 自动积累
3. **Skills 管任务模板**：如果某类任务有固定的执行流程（写 newsletter、做 code review），用 [SKILL.md 文件](/zh/blog/9-principles-writing-claude-code-skills)定义，不要塞进 CLAUDE.md
4. **定期清理**：CLAUDE.md 随代码一起 review，Memory 偶尔手动检查，删除过时记录

了解 Claude Code 不仅仅是一个编码工具的更多视角，推荐阅读[《Claude Code 不是一个编程工具》](/zh/blog/claude-code-is-not-a-coding-tool)。

## 结论

**CLAUDE.md 是"法律"，Memory 是"经验"。** 法律必须明确、一致、可审计；经验可以模糊、个人化、不断演化。一个没有 CLAUDE.md 的项目，Claude Code 每次都像新人一样摸索；一个没有 Memory 的工作流，Claude Code 永远无法真正理解你的工作方式。两者结合，才能释放 Claude Code 作为[工程伙伴](/zh/blog/whats-so-special-about-the-claude-code)的全部潜力。

## 常见问题

### Claude Memory 的数据存在哪里？
Claude Memory 存储在本地 `.claude/projects/` 目录下的 markdown 文件中，按项目路径组织。这些文件不会被 git 追踪，也不会上传到 Anthropic 的服务器。每个记忆是一个独立的 markdown 文件，包含 frontmatter 元数据和正文内容。

### CLAUDE.md 支持子目录覆盖吗？
支持。Claude Code 会沿目录树向上查找所有 CLAUDE.md 文件，子目录的规则可以补充或覆盖父目录的规则。这对 monorepo 特别有用——根目录定义全局规范，各模块目录定义特定约定。

### 能不能让 Claude Code 忽略 Memory？
可以。在对话中明确告诉 Claude Code "ignore memory" 或 "don't use memory"，它就不会引用、对比或提及记忆内容。但 CLAUDE.md 的指令始终生效，无法在对话中临时禁用。
