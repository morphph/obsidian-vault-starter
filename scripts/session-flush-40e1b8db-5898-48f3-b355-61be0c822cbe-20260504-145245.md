# Codex Subagents vs Custom Agents：哪种多智能体模式更适合你？

**TL;DR:** **Subagents** 是 Codex 内置的任务分拆机制，适合将一个大任务自动拆解为多个并行子任务；**Custom Agents** 是用户自定义的智能体配置，适合为特定领域或团队规范创建可复用的专属 Agent。两者不互斥——Subagents 解决的是「怎么拆」，Custom Agents 解决的是「拆给谁」。大多数生产级工作流需要两者配合使用。

## 概述：Codex Subagents

**Subagents** 是 [OpenAI Codex](/zh/blog/codex-complete-guide) 平台内置的子任务派发能力。当一个编码任务涉及多个文件、多个模块或多个独立步骤时，Codex 可以自动或手动将其拆分为多个 Subagent，每个 Subagent 在独立的沙箱环境中并行执行。

这种设计的核心价值在于吞吐量。传统的单线程 Agent 模式下，一个涉及前端、后端、测试三个模块的重构任务需要串行完成；而 Subagents 可以让三个子任务同时进行，主 Agent 负责协调和合并结果。对于大型代码库的批量操作——比如跨 50 个文件的 API 迁移——Subagents 可以将执行时间从小时级压缩到分钟级。

Subagents 的典型使用场景包括：并行测试生成、多模块同步重构、批量代码审查等。它们继承主任务的上下文和权限设置，开发者无需为每个子任务单独配置环境。

## 概述：Codex Custom Agents

**Custom Agents** 是 Codex 提供的自定义智能体机制，允许开发者通过配置文件定义具有特定指令、工具集和行为规范的专属 Agent。你可以把它理解为「带人格和技能的 Agent 模板」。

Custom Agents 的核心价值在于一致性和复用。当团队有特定的代码规范、安全要求或领域知识时，Custom Agent 可以将这些约束编码为持久化配置，确保每次任务执行都遵循相同标准。比如，你可以创建一个专门的「安全审计 Agent」，它默认启用严格的依赖检查、遵循 OWASP 规范、并在发现漏洞时自动生成修复建议。

Custom Agents 通过 `AGENTS.md` 或项目配置文件定义，支持自定义系统指令、可用工具范围、输出格式等参数。这与 [Claude Code 的 Skills 系统](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)在理念上相似——都是将工程最佳实践编码为可版本控制的配置。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **核心定位** | 任务并行拆解 | 领域专属配置 | 各有所长 |
| **配置复杂度** | 低——平台自动调度 | 中——需编写配置文件 | Subagents |
| **执行模式** | 并行，沙箱隔离 | 按配置独立运行 | Subagents |
| **上下文管理** | 继承主任务上下文 | 自带独立指令和约束 | Custom Agents |
| **复用性** | 单次任务内有效 | 跨项目、跨团队复用 | Custom Agents |
| **团队协作** | 自动分配，无需协调 | 统一团队规范和流程 | Custom Agents |
| **学习曲线** | 几乎为零 | 需理解配置体系 | Subagents |
| **精细控制** | 有限——依赖平台调度 | 高——自定义工具和行为 | Custom Agents |

## 任务编排能力：深度分析

Subagents 的核心优势在于自动化的任务编排。当你向 Codex 提交一个复杂任务时，平台会分析任务依赖关系，识别可并行执行的子任务，并自动派发给多个 Subagent。这个过程对开发者基本透明——你描述目标，Codex 决定怎么拆。

这种模式在处理「广度型」任务时效果尤其明显。比如「为这个项目的所有 API 端点添加输入验证」，Codex 可以为每个端点创建一个 Subagent，它们同时工作，互不干扰。根据 [OpenAI 对 Codex 多智能体工作流的设计理念](/zh/blog/con-u-pour-des-workflows-multi-agents)，这种并行模式可以显著提升大规模代码操作的效率。

但 Subagents 的局限也很明显：它们缺乏领域特化能力。每个 Subagent 使用相同的基础模型和默认行为，无法针对不同类型的子任务定制专属策略。一个负责前端样式调整的 Subagent 和一个负责数据库迁移的 Subagent，行为模式完全相同。

相比之下，Custom Agents 在编排层面需要更多手动配置，但它允许你为不同类型的任务指定完全不同的行为。你可以定义一个「前端 Agent」只关注 React 组件和 CSS，另一个「后端 Agent」专注于 API 设计和数据库操作——每个 Agent 都有自己的指令集和工具权限。

## 团队规范与一致性：深度分析

Custom Agents 真正发光的场景是团队协作。在一个 20 人的工程团队中，每个成员使用 Codex 的方式可能差异巨大——有人偏好详细注释，有人偏好极简代码；有人习惯写集成测试，有人倾向单元测试。Custom Agents 通过标准化配置消除这种不一致。

将 Custom Agent 配置纳入版本控制后，团队规范的传递就变得自动化了。新成员 clone 代码库，Custom Agents 配置随项目一起加载，无需额外培训即可遵循团队标准。这与 [Claude Code 的 Skills 和 Hooks 体系](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)的设计哲学一脉相承——将工程规范从「人记住」转变为「配置保证」。

Custom Agents 还支持权限隔离。安全敏感的操作可以限定只有特定 Custom Agent 有权执行，普通 Agent 无法触及生产环境的数据库或部署管道。这对于合规要求严格的企业团队至关重要。

Subagents 在团队规范方面的能力相对有限。它们继承主任务的上下文，但无法携带独立的行为约束。如果主任务的提示词没有明确指定规范，Subagents 也不会自动遵循。

## 何时选择 Subagents

Subagents 是「扩展执行带宽」的最佳选择：

- **批量操作**：跨多个文件或模块的统一修改，如 API 版本迁移、依赖升级、格式统一
- **并行测试生成**：为多个模块同时生成测试用例，参见 [Claude Code 中类似的 Agent Teams 实践](/zh/blog/claude-code-agent-teams)
- **大规模重构**：涉及数十个文件的重命名、接口变更、架构调整
- **代码审查分片**：将大型 PR 拆分给多个 Subagent 分别审查不同模块

如果你的任务是「一件大事拆成多件小事并行做」，Subagents 是天然的选择。它们不需要预先配置，即开即用，平台自动处理调度和合并。

## 何时选择 Custom Agents

Custom Agents 是「定制化执行策略」的最佳选择：

- **领域特化**：前端/后端/DevOps/安全等不同领域需要不同的 Agent 行为和工具集
- **团队标准化**：统一代码风格、测试覆盖率要求、文档规范等，参考 [Claude Code Skills 的最佳实践](/zh/blog/9-principles-writing-claude-code-skills)
- **合规审计**：需要特定 Agent 执行安全扫描、许可证检查、敏感信息检测
- **客户定制**：面向不同客户或项目需要不同的交付标准和流程

如果你的需求是「不同的事要用不同的方式做」，Custom Agents 提供了必要的灵活性。它们需要前期投入来设计和维护配置，但长期回报是显著的一致性和效率提升。

## 结论

**Subagents 和 Custom Agents 不是二选一的关系，而是互补的两层能力。** 最强大的 Codex 工作流同时使用两者：Custom Agents 定义「由谁来做」和「怎么做」，Subagents 解决「如何并行」和「如何扩展」。

对于刚开始使用 [Codex](/zh/blog/codex-complete-guide) 的开发者，建议先从 Subagents 入手——它零配置、即开即用，能立刻感受到并行执行的效率提升。当团队规模增长或项目复杂度提高时，再逐步引入 Custom Agents 来编码领域知识和团队规范。对于已经在使用 [Claude Code Agent Teams](/zh/blog/claude-code-agent-teams) 等类似功能的团队，迁移到 Codex Custom Agents 的概念门槛很低，核心理念是相通的。

## 常见问题

### Subagents 和 Custom Agents 能同时使用吗？

可以。Custom Agents 定义特化的行为和约束，Subagents 负责在执行层面并行拆解任务。一个 Custom Agent 可以在执行过程中自动派发 Subagents 来并行处理子任务，两者协同工作。

### Custom Agents 的配置是否需要编程能力？

不需要传统编程能力，但需要理解 Agent 配置的基本语法。Custom Agents 通过 Markdown 或 YAML 格式的配置文件定义，主要涉及自然语言指令和简单的参数设置，类似于编写详细的任务说明书。

### Subagents 的并行数量有上限吗？

Codex 平台会根据任务复杂度和资源可用性动态调整并行 Subagent 的数量。具体上限取决于账户类型和当前平台负载，建议参考 OpenAI 官方文档获取最新的配额信息。

### 从 Claude Code 迁移到 Codex 的多智能体工作流难度大吗？

核心概念高度相似。Claude Code 的 [Agent Teams](/zh/blog/claude-code-agent-teams) 对应 Codex 的 Subagents，Claude Code 的 [Skills 系统](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 对应 Codex 的 Custom Agents。主要差异在于配置格式和平台 API，逻辑层面的迁移成本较低。
