# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR:** **CLAUDE.md** 是手动编写的项目规则手册，签入仓库，团队共享，每次会话自动加载——适合编码规范、架构约束、工作流指令。**Claude Memory（Auto Memory）** 是 Claude Code 自动维护的持久化笔记，按用户存储，跨会话保留——适合个人偏好、项目上下文、反馈记录。两者不是替代关系，而是互补的双层记忆架构。选哪个取决于信息的受众：团队共享的写 CLAUDE.md，个人积累的交给 Memory。

## 概览：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Claude Code 内置的文件级持久化记忆系统，自动在 `~/.claude/projects/` 目录下维护一组 markdown 文件。它的核心设计理念是：让 Claude 在多次对话之间记住关于你、你的项目和你的工作方式的重要信息，而不需要你每次重复。

Memory 支持四种记忆类型：**user**（用户画像）、**feedback**（行为修正）、**project**（项目动态）、**reference**（外部资源指针）。每条记忆以独立 markdown 文件存储，带有 frontmatter 元数据，并通过 `MEMORY.md` 索引文件统一管理。Claude 会在对话开始时自动加载相关记忆，也会在对话过程中判断是否需要新增或更新记忆。

关键特征：Memory 是**用户级别**的，不签入 Git 仓库，不会被团队其他成员看到。它是你和 Claude 之间的私人笔记本。更多技术细节可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概览：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件，放在仓库根目录，签入版本控制，团队所有成员共享。它的作用类似于 `.editorconfig` 或 `CONTRIBUTING.md`——定义"这个项目应该怎么做"的共识规则。

CLAUDE.md 的内容通常包括：构建和测试命令、代码风格约定、架构约束、工作流要求（比如"提交前必须通过所有测试"）、已知陷阱和注意事项。Claude Code 在每次会话启动时会自动读取项目根目录和用户全局的 CLAUDE.md 文件，将其作为系统级指令注入上下文。

关键特征：CLAUDE.md 是**团队级别**的，签入仓库，所有使用 Claude Code 的团队成员都会自动获得相同的指令。它是项目的公共规则手册。CLAUDE.md 在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中处于最基础的配置层，是整个扩展栈的根基。

## 信息归属：哪些该写 CLAUDE.md，哪些交给 Memory

这是最核心的决策维度。判断标准很简单：**这条信息是属于项目还是属于你个人？**

### 属于 CLAUDE.md 的信息

团队层面的共识和约束，任何使用这个仓库的开发者都应该遵守的规则：

- 构建、测试、部署命令（`npm run build`、`npm test`）
- 代码风格和架构约束（"不要在 pipeline 脚本里直接 import Next.js 模块"）
- 质量门禁（"提交前必须通过 build + test + validate"）
- 已知陷阱（"ZH 内容必须用 CJK word count"）
- 文档更新规则（"修改 scripts/ 下的文件后同步更新 PIPELINE.md"）

这些信息的共同特征：跟**项目**绑定，不跟**人**绑定。换一个开发者来，规则不变。

### 属于 Memory 的信息

个人层面的上下文和偏好，只对你自己的工作方式有意义：

- **user 类型**：你的角色、技能背景、关注领域（"用户是数据科学家，主要关注日志系统"）
- **feedback 类型**：你对 Claude 行为的修正（"不要在每个回答末尾总结刚做了什么"）
- **project 类型**：当前项目动态和临时状态（"3月5日后冻结非关键合并"）
- **reference 类型**：外部资源指针（"pipeline bug 在 Linear 的 INGEST 项目里追踪"）

这些信息的共同特征：跟**你**绑定，不跟**项目**绑定。换一个项目，你的偏好依然有效。

## 实战场景对比

### 场景一：新成员入职

新成员 clone 仓库后，CLAUDE.md 立即生效——编码规范、构建命令、架构约束全部自动加载。但 Memory 是空的，因为它是用户级别的。随着新成员和 Claude 交互，Memory 会逐渐积累：Claude 会记住这位成员是前端新手但有深厚的后端经验，会记住他不喜欢冗长的解释，会记住他习惯用 vim 键位。

**结论**：CLAUDE.md 提供即刻生效的团队基线，Memory 提供渐进式的个人适配。

### 场景二：调试一个复杂的 pipeline bug

你花了两天调试一个 newsletter 去重问题。过程中你告诉 Claude "这个项目的去重分两层：Claude 语义去重 + DB 标记过滤"。这条信息应该进 Memory 的 project 类型——它是你在调试过程中积累的项目洞察，对你未来的对话有价值，但不适合写进 CLAUDE.md（太具体、太临时）。

但如果你发现了一个通用的陷阱——比如"重新运行 newsletter 测试前必须先重置 DB 标记"——这条应该写进 CLAUDE.md 的 Known Gotchas 部分，因为任何开发者都可能踩这个坑。

**结论**：调试过程中的个人洞察进 Memory，提炼出的通用陷阱进 CLAUDE.md。

### 场景三：个人工作风格偏好

你告诉 Claude "不要 mock 数据库，上次 mock 测试通过了但生产环境迁移失败"。这是典型的 feedback 类型 Memory。它记录了你的工作偏好和背后的原因，帮助 Claude 在未来的对话中避免同样的错误。

但如果这是团队级别的决策——"本项目所有集成测试必须连接真实数据库"——那就应该写进 CLAUDE.md。

**结论**：个人偏好进 Memory，团队决策进 CLAUDE.md。

## 最佳搭配策略

Memory 和 CLAUDE.md 不是二选一的关系。最高效的用法是双层协作：

1. **CLAUDE.md 定义基线**：项目规则、代码规范、架构约束。所有团队成员共享，Git 版本控制，变更可追溯。
2. **Memory 提供个性化**：用户背景、行为偏好、项目动态。自动维护，跨会话持久化，渐进式积累。
3. **不要重复**：如果信息已经在 CLAUDE.md 里，不要再存进 Memory。Memory 的设计原则明确规定：不存储可以从代码或 CLAUDE.md 中推导出的信息。
4. **定期清理**：Memory 会随时间积累过时信息。project 类型的记忆尤其容易过期——"3月5日冻结合并"在3月6日就失效了。定期检查并清理过时记忆。
5. **升级路径**：当 Memory 中的某条信息被反复触发，说明它可能应该升级为 CLAUDE.md 中的正式规则。

## 判定

**CLAUDE.md 是你的团队规则手册，Memory 是你的私人助理笔记本。** 两者解决不同层面的问题：CLAUDE.md 确保团队一致性和项目规范，Memory 确保个人效率和上下文连续性。

如果你是独立开发者，CLAUDE.md 依然是第一优先级——它提供确定性的、可复现的行为基线。Memory 是锦上添花，让 Claude 越用越懂你。

如果你在团队协作，两者缺一不可。CLAUDE.md 是入职第一天就需要的公共知识，Memory 是随着协作深入自然积累的私人上下文。

**核心决策规则：问自己"换一个人来，这条信息还有用吗？"有用就写 CLAUDE.md，没用就交给 Memory。**

## 常见问题

### Claude Memory 会被团队其他成员看到吗？

不会。Memory 存储在用户本地的 `~/.claude/projects/` 目录下，不签入 Git 仓库，完全是用户私有的。团队成员之间的 Memory 互相独立，不会交叉影响。如果需要团队共享的信息，应该写入 CLAUDE.md。

### CLAUDE.md 文件太长会影响性能吗？

会。CLAUDE.md 在每次会话启动时全量加载到上下文窗口中，过长的文件会占用宝贵的上下文空间。建议保持 CLAUDE.md 简洁精炼，将详细的规范文档放在 `docs/` 目录下，CLAUDE.md 中只保留关键规则和指向详细文档的引用。

### Memory 能否手动编辑？

可以。Memory 本质上就是 markdown 文件，你可以直接用编辑器打开修改。每条记忆是一个独立的 `.md` 文件，带有 frontmatter 元数据（name、description、type），通过 `MEMORY.md` 索引文件统一管理。你也可以让 Claude 帮你更新或删除特定记忆。

### 两者信息冲突时以谁为准？

CLAUDE.md 的优先级更高。CLAUDE.md 中的指令被视为系统级约束，Claude 必须严格遵守。如果 Memory 中的某条记忆与 CLAUDE.md 中的规则矛盾，Claude 会以 CLAUDE.md 为准，并可能自动更新或删除过时的 Memory 条目。
