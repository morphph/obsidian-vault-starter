# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **Claude Memory** 是跨会话的个人记忆——它记住你的偏好、角色和工作习惯，下次对话自动带入。**CLAUDE.md** 是项目级的指令文件——它定义代码规范、构建命令和工程约束，跟着仓库走。两者不是替代关系，而是互补：Memory 记住「你是谁」，CLAUDE.md 告诉 Claude「这个项目怎么做」。大多数认真使用 Claude Code 的开发者应该同时用好两者。

## 概述：Claude Memory

**Claude Memory** 是 Anthropic 为 Claude 提供的持久化记忆功能，让 AI 在多次对话之间保留关于用户的关键信息。它解决的核心问题是：每次新对话都要从头解释自己的背景、偏好和工作方式。

Memory 的运作方式是自动提取或用户主动要求保存的信息片段。这些片段存储在用户账户级别，跨所有对话生效。比如你告诉 Claude「我是后端工程师，主要写 Go」，下次聊天时它不会再建议你用 Python 方案。

2026 年 5 月，Anthropic [大幅升级了记忆系统](/zh/blog/anthropic-claude-memory-upgrades-importing)：免费用户开放使用，支持从 ChatGPT 导入历史数据，记忆容量也显著提升。这意味着 Memory 不再是 Pro 用户的专属功能。

在 Claude Code 场景下，Memory 以 `~/.claude/` 目录下的文件形式存在，包括自动记忆（auto memory）和用户主动保存的偏好。它的作用范围是个人级别——跟着你的账户走，不跟着项目走。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件，放在仓库根目录，作用类似于一份「AI 工程师入职手册」。它告诉 Claude 这个项目用什么技术栈、怎么构建、有哪些约束、遵循什么编码规范。

CLAUDE.md 的核心价值在于可版本控制、可团队共享。它 check in 到 Git 仓库，团队每个成员使用 Claude Code 时都会自动加载同一份指令。这保证了 AI 在不同人的终端里表现一致——不会因为张三没提过「我们用 Tailwind v4」而生成错误的 CSS 方案。

CLAUDE.md 支持多层级结构：根目录的全局指令、子目录的局部指令，以及 `skills/*/SKILL.md` 形式的可复用任务模板。这套[七层架构](/zh/blog/claude-code-seven-programmable-layers)让开发者从 Claude Code 的「使用者」变成「系统设计者」。

与 Memory 不同，CLAUDE.md 的作用范围严格限定在项目内。换一个仓库，就是另一套规则。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **作用范围** | 用户级，跨所有对话和项目 | 项目级，仅限当前仓库 | 各有所长 |
| **持久化方式** | 云端账户存储 / 本地 `~/.claude/` | Git 仓库文件 | CLAUDE.md |
| **团队共享** | 不可共享（个人隐私） | 可共享（check in 到 Git） | CLAUDE.md |
| **版本控制** | 无版本历史 | 完整 Git 历史 | CLAUDE.md |
| **自动性** | 自动提取 + 手动保存 | 手动编写，自动加载 | Memory |
| **内容类型** | 用户偏好、角色、习惯 | 构建命令、代码规范、架构约束 | 各有所长 |
| **可审计性** | 难以审计，内容分散 | 完全透明，团队可 review | CLAUDE.md |
| **迁移成本** | 支持从 ChatGPT 导入 | 复制文件即可 | 平手 |

## 记忆机制：深度对比

Claude Memory 和 CLAUDE.md 在底层机制上完全不同，理解这些差异是正确使用它们的前提。

**Claude Memory 是隐式的。** 它在对话中自动捕捉信息，用户也可以说「记住这个」来主动触发保存。保存的内容以结构化片段形式存储，每次新对话时注入到系统提示词中。用户无法精确控制哪些记忆会在什么时候被调用——这由模型自行判断相关性。

在 Claude Code 中，auto memory 将信息写入 `~/.claude/projects/` 目录下的 markdown 文件，并维护一个 `MEMORY.md` 索引。这些文件按类型分类：用户信息（user）、工作反馈（feedback）、项目状态（project）、外部引用（reference）。详细的[记忆系统机制](/zh/blog/claude-code-memory)我们在之前的文章中有完整分析。

**CLAUDE.md 是显式的。** 它是开发者主动编写的指令文档，内容完全透明可控。Claude Code 启动时会按照固定优先级加载：用户全局 `~/.claude/CLAUDE.md` → 项目根目录 `CLAUDE.md` → 子目录 `CLAUDE.md` → `skills/*/SKILL.md`。每一层都是确定性的——你写什么它就读什么，没有模糊的「相关性判断」。

关键区别在于**可预测性**。CLAUDE.md 的行为是确定性的：同一份文件，不同人执行，结果一致。Memory 的行为是概率性的：模型可能调用某条记忆，也可能忽略它，取决于对话上下文。这意味着对于团队协作和生产环境，CLAUDE.md 是更可靠的选择。

## 适用场景：深度对比

选择 Memory 还是 CLAUDE.md，本质上取决于信息的归属——它属于「你个人」还是「这个项目」。

**个人偏好用 Memory。** 你喜欢简洁的 commit message、你习惯用 TypeScript 而不是 JavaScript、你是高级工程师不需要基础解释——这些信息跟着人走，不应该写进项目的 CLAUDE.md 里。Memory 的「用户类型」记忆就是为此设计的。

**工程约束用 CLAUDE.md。** 这个项目用 Next.js 16 + Tailwind v4、构建前必须跑 `npm test`、不允许在 pipeline 脚本里 import Next.js 模块——这些信息跟着项目走，团队每个人都需要遵守。把它们写进 CLAUDE.md，确保 AI 不会因为某个新同事没有配置 Memory 而违反规范。

**流程反馈两者都行，但优先 Memory。** 如果你发现 Claude 总是在回复末尾加多余的总结，告诉它「别这样」并让它记住。这种反馈是个人交互偏好，Memory 是更合适的载体。但如果是项目级的 AI 行为约束（比如「生成代码时不要加注释」），写进 CLAUDE.md。

## 什么时候选 Claude Memory

Memory 在以下场景发挥最大价值：

- **跨项目的个人偏好**：你的编码风格、沟通偏好、技术栈熟悉度，这些在所有项目中通用
- **角色与背景信息**：你的工作职责、技术水平、关注领域——让 Claude 不需要每次重新了解你
- **交互反馈的积累**：纠正过的错误不再重犯，确认过的好做法持续执行
- **快速上手新项目**：即使没有 CLAUDE.md，Memory 中的个人偏好仍然能让 Claude 提供个性化的帮助

Memory 的弱点在于不可审计、不可共享。你无法确保团队其他人的 Memory 里有相同的信息，也无法通过 code review 来检查 Memory 的内容是否正确。

## 什么时候选 CLAUDE.md

CLAUDE.md 在以下场景不可替代：

- **团队协作项目**：所有人必须遵守相同的构建流程、代码规范和质量门槛
- **生产环境约束**：数据库命名规范、API 设计标准、安全要求——不能靠个人记忆
- **可复用的 AI 工作流**：通过 [Skills 系统](/zh/blog/9-principles-writing-claude-code-skills)定义的标准化任务模板，比如代码审查、内容生成、测试编写
- **新成员上手**：新人 clone 仓库后，Claude Code 立刻知道这个项目的所有规则，零配置成本
- **CI/CD 集成**：自动化流水线中运行 Claude Code 时，CLAUDE.md 是唯一的指令来源——没有人类的 Memory 可调用

如果你只能选一个，选 CLAUDE.md。它覆盖了工程协作中最关键的需求：一致性和可追溯性。

## 结论

**Claude Memory 和 CLAUDE.md 不是竞争关系，而是分层设计。** Memory 是个人层——记住你是谁、你怎么工作。CLAUDE.md 是项目层——定义这个项目怎么做、有什么规则。最佳实践是两者配合使用：把个人偏好交给 Memory，把工程约束写进 CLAUDE.md。

对于独立开发者，Memory 能快速提升 Claude 的个性化程度。对于团队，CLAUDE.md 是不可协商的基础设施——没有它，AI 的行为在不同成员之间无法保持一致。理解这个分层，才算真正掌握了 Claude Code 的[记忆体系](/zh/blog/claude-code-memory)。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。Memory 存储个人偏好和交互习惯，CLAUDE.md 存储项目级的工程规范和构建指令。Claude Code 启动时会同时加载两者，Memory 中的个人偏好与 CLAUDE.md 中的项目规则共同作用，不会冲突。

### 如果 Memory 和 CLAUDE.md 的指令矛盾，Claude 听谁的？

CLAUDE.md 优先级更高。作为显式写入的项目指令，CLAUDE.md 的规则会覆盖 Memory 中可能存在的冲突偏好。这是合理的设计——项目规范应该高于个人习惯。

### 新团队成员没有 Memory 记录怎么办？

这正是 CLAUDE.md 的价值所在。新成员 clone 仓库后，CLAUDE.md 立刻生效，无需任何额外配置。个人 Memory 会在使用过程中自然积累，但项目级的关键信息不依赖于此。

### CLAUDE.md 需要写多长？

没有固定长度要求，但建议精简实用。核心内容包括：技术栈声明、构建命令、质量门槛、关键约束。避免写成面面俱到的文档——Claude 能从代码中推断的信息不需要重复写入。参考[Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)的分层设计，把详细的任务指令放到 SKILL.md 中。
