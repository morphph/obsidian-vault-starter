# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **CLAUDE.md** 是你写给 AI 的项目规则手册——签入 Git、团队共享、每次对话自动加载。**Claude Memory** 是 AI 自己做的笔记——跨会话持久化、按用户隔离、自动积累上下文。两者不是竞争关系，而是互补的两层持久化机制。规则放 CLAUDE.md，经验放 Memory。

## 概述：Claude Memory

Claude Memory 是 Claude Code 内置的自动记忆系统，存储在 `~/.claude/projects/` 目录下。它的核心机制是：在对话过程中，Claude 会自动识别值得记住的信息——用户偏好、项目背景、工作反馈——并写入独立的 Markdown 文件，通过 `MEMORY.md` 索引管理。

Memory 按类型分为四类：**user**（用户画像）、**feedback**（行为纠偏）、**project**（项目动态）、**reference**（外部资源指针）。每条记忆都有 frontmatter 元数据，支持按相关性检索。关键特征是它属于个人——你的记忆不会出现在同事的 Claude Code 里。

详细机制可参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是放在项目根目录的指令文件，Claude Code 每次启动时自动读取。它本质上是一份「AI 协作契约」——告诉 Claude 这个项目用什么技术栈、遵循什么规范、有哪些禁区。

CLAUDE.md 签入 Git 仓库，团队所有成员共享同一套规则。它的作用域是确定性的：写什么就执行什么，不存在「AI 自行判断要不要记住」的环节。除了项目级 CLAUDE.md，还支持用户级 `~/.claude/CLAUDE.md`（个人全局偏好）和目录级 CLAUDE.md（子模块特殊规则）。

这是 Claude Code [七层可编程架构](/zh/blog/claude-code-seven-programmable-layers)中最基础的一层。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 胜出 |
|------|--------------|-----------|------|
| **写入方式** | AI 自动生成 + 用户触发 | 人工手写 | 各有优势 |
| **作用域** | 按用户、按项目隔离 | 按仓库，团队共享 | CLAUDE.md |
| **版本控制** | 不签入 Git | 签入 Git | CLAUDE.md |
| **持久化位置** | `~/.claude/projects/*/memory/` | 项目根目录 | 各有优势 |
| **加载时机** | 每次对话自动加载索引 | 每次对话自动加载全文 | 平手 |
| **内容类型** | 用户偏好、项目动态、行为反馈 | 构建命令、代码规范、架构约束 | 各有优势 |
| **可预测性** | 中等——AI 判断什么值得记 | 高——写什么就是什么 | CLAUDE.md |
| **维护成本** | 低——自动积累 | 中——需要手动更新 | Memory |

## 核心区别：确定性 vs 自适应

这是理解两者差异的关键维度。

CLAUDE.md 是**确定性规则**。你写「commit 前必须跑 `npm test`」，Claude 就会每次都跑。你写「不要在 pipeline 脚本里 import Next.js 模块」，它就不会犯这个错。规则不会因为对话内容变化而变化，也不会因为 Claude「忘了」而失效。这种确定性是团队协作的基础——新成员 clone 仓库，立刻获得和老成员一样的 AI 行为。

Memory 是**自适应知识**。它记录的是对话中涌现的信息：「用户是数据科学家，目前在做日志系统调研」「这个用户不喜欢每次回复后加总结」「mobile 团队 3 月 5 日后冻结非关键合并」。这些信息无法提前写入 CLAUDE.md——它们是在协作过程中逐步浮现的。

打个比方：CLAUDE.md 是公司的员工手册，Memory 是你和这位同事共事几个月后积累的默契。两者都重要，但解决不同问题。

## 数据流与生命周期

理解数据如何流动，有助于选择正确的持久化层。

**CLAUDE.md 数据流：** 开发者编辑 → Git commit → 团队 pull → Claude Code 启动时加载 → 影响当前会话所有行为。更新频率通常是项目级事件驱动：新增构建步骤、变更部署流程、引入新规范。

**Memory 数据流：** 对话中 AI 识别有价值信息 → 写入 memory 文件 → 更新 MEMORY.md 索引 → 下次对话加载索引 → 按相关性检索具体记忆。更新频率是对话驱动的：每次有意义的交互都可能产生新记忆。

一个容易踩的坑：**不要用 Memory 存代码模式、架构决策或文件路径**。这些信息会随代码变化而过时，应该直接读代码或 Git 历史。Memory 存的应该是「为什么」而非「是什么」——比如某个架构决策背后的业务原因，而非架构本身。

## 实战场景：什么放哪里

### 放 CLAUDE.md 的典型内容

```markdown
# 构建命令
npm run build    # 生产构建
npm test         # 必须通过再提交

# 代码规范
- 中文内容用 CJK 字符计数，不用空格分词
- 不要在 pipeline 脚本里直接 import Next.js 模块
```

这些是**团队共享的确定性规则**。任何人 clone 仓库都需要遵守。

### 放 Memory 的典型内容

- 「这位用户偏好简洁回复，不要结尾总结」（feedback 类型）
- 「项目 Q2 目标是把 SEO 页面从 200 扩展到 500」（project 类型）
- 「pipeline bug 在 Linear 的 INGEST 项目里追踪」（reference 类型）

这些是**个人化的、随时间演进的知识**。写进 CLAUDE.md 会污染团队共享的规则集。

### 两者协作的场景

一个真实的工作流：你在 CLAUDE.md 里定义了「[Skills 文件](/zh/blog/9-principles-writing-claude-code-skills)结构必须包含 frontmatter」，这是规则。Memory 里存的是「上次修改 newsletter skill 时用户要求保留中文语气的独特性，不要翻译腔」，这是经验。Claude Code 同时读取两者，既遵守规则又保持个性化。

如果你还在探索 Claude Code 的完整[扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)，会发现 CLAUDE.md 和 Memory 只是四层架构中的两层——Skills 和 Hooks 处理的是另外的问题域。

## 什么时候用 Claude Memory

选择 Memory 存储当：

- 信息是**个人化的**——你的沟通偏好、你的角色背景、你关注的领域
- 信息是**对话中涌现的**——无法提前预知，在协作过程中自然产生
- 信息会**随时间演进**——项目阶段变化、优先级调整、团队动态
- 信息**不适合团队共享**——比如「我是前端新手，解释时多用后端类比」

Memory 的优势在于低维护成本。你不需要主动管理，Claude 会自动判断什么值得记住。但这也是它的风险——AI 的判断不总是对的，偶尔需要手动清理过时的记忆。

## 什么时候用 CLAUDE.md

选择 CLAUDE.md 当：

- 规则必须**团队统一执行**——构建流程、代码规范、禁止的操作
- 规则是**确定性的**——不希望 AI 有任何自由裁量空间
- 规则需要**版本控制**——谁改了什么、什么时候改的、为什么改
- 规则和**代码紧耦合**——比如「这个项目用 Tailwind v4」「SQLite 不要用 ORM」

CLAUDE.md 的维护成本高于 Memory——你需要在项目变化时手动更新它。但这个成本换来的是可预测性和团队一致性。对于日常使用 Claude Code 的[实用技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)，一个维护良好的 CLAUDE.md 是效率的基石。

## 判定

**不要二选一——两者是互补的持久化层。** 把 CLAUDE.md 当作项目的「宪法」，把 Memory 当作你和 AI 的「工作日志」。团队规范、构建命令、架构约束写进 CLAUDE.md；个人偏好、项目动态、行为反馈交给 Memory 自动管理。

实际操作建议：先投入时间写好 CLAUDE.md——它的 ROI 最高，因为每次对话都会加载。Memory 则让它自然积累，定期检查 `MEMORY.md` 索引清理过时条目即可。如果你发现自己反复在对话中纠正 Claude 的同一个行为，那就是信号：**这条规则该从 Memory 升级到 CLAUDE.md**。

想深入了解 Claude Code 的完整架构设计，推荐阅读[Claude Code 不是一个编程工具](/zh/blog/claude-code-is-not-a-coding-tool)——它会改变你对这两套系统的理解框架。

## 常见问题

### Claude Memory 的内容会被其他团队成员看到吗？

不会。Memory 存储在用户本地的 `~/.claude/projects/` 目录下，按用户和项目隔离。它不会签入 Git，也不会上传到 Anthropic 的服务器。团队成员之间的 Memory 完全独立，互不影响。

### CLAUDE.md 支持多层级配置吗？

支持三个层级：用户全局级（`~/.claude/CLAUDE.md`）、项目根目录级（`./CLAUDE.md`）、子目录级（`./src/CLAUDE.md`）。Claude Code 启动时会合并所有层级，子目录级规则可以覆盖或补充上层规则。

### Memory 记错了怎么办？

直接告诉 Claude「忘记 XXX」或手动编辑 `~/.claude/projects/` 下的 memory 文件。Memory 文件是标准 Markdown，可以用任何编辑器修改或删除。建议定期审查 `MEMORY.md` 索引，清理过时或不准确的条目。

### 两者有大小限制吗？

CLAUDE.md 会被完整加载到上下文窗口，过长会占用宝贵的 token 预算——建议控制在 500 行以内。Memory 只加载索引文件（MEMORY.md），具体记忆按需检索，对上下文压力较小，但索引超过 200 行会被截断。
