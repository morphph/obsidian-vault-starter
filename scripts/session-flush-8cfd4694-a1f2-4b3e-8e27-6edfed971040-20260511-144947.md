# Codex Subagents vs Custom Agents：哪种多智能体方案更适合你？

**TL;DR：** 如果你需要在单次任务中动态拆解子任务、并行处理，**Subagents 是首选**。如果你需要预定义角色和行为规范、跨任务复用，**Custom Agents 更合适**。两者不互斥——成熟团队通常将 Custom Agents 作为基础框架，在其中按需调度 Subagents 执行具体工作。

## 概览：Codex Subagents

Subagents 是 [OpenAI Codex](/zh/blog/codex-complete-guide) 在任务执行过程中动态生成的子智能体。当主 Agent 遇到可拆解的复杂任务时，它会自动或按指令将工作分配给多个 Subagents 并行处理，每个 Subagent 在独立的沙箱环境中运行，拥有自己的上下文和工具权限。

Subagents 的核心价值在于**运行时编排**：你不需要提前定义子任务的结构，Codex 会根据当前任务的需求自动规划分工。这种机制特别适合探索性任务——重构一个大型模块、在多个文件中查找并修复同类 bug、或者同时生成测试和文档。Subagents 完成后，结果会汇总回主 Agent，由它做最终整合。

## 概览：Codex Custom Agents

Custom Agents 是用户在 Codex 平台上预先配置的专用智能体。你可以为每个 Custom Agent 定义系统指令、可用工具集、代码访问范围和行为约束，使其成为某个特定领域的"专家"。

与 Subagents 的动态生成不同，Custom Agents 是**设计时定义**的。你在使用前就明确了它的角色——比如"安全审计 Agent"、"测试生成 Agent"或"代码风格检查 Agent"。配置完成后可跨多个任务、多个团队成员复用。对于有标准化工程流程的团队，Custom Agents 相当于把最佳实践编码为可执行的 Agent 配置。这与 Claude Code 中的 [Skills 系统](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)有相似的设计理念。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **创建方式** | 运行时由主 Agent 动态生成 | 用户提前配置，保存为模板 | 各有所长 |
| **上下文隔离** | 每个 Subagent 独立沙箱 | 按配置的访问范围隔离 | Subagents |
| **并行执行** | 原生支持多任务并行 | 需要手动编排或搭配 Subagents | Subagents |
| **可复用性** | 一次性，任务结束即销毁 | 持久化配置，跨任务复用 | Custom Agents |
| **行为一致性** | 取决于主 Agent 的指令质量 | 由预设系统指令保证 | Custom Agents |
| **团队协作** | 个人使用为主 | 可在团队间共享配置 | Custom Agents |
| **学习门槛** | 低——自然语言描述任务即可 | 中——需要理解配置结构 | Subagents |
| **调试可见性** | 可查看每个 Subagent 的执行日志 | 可查看 Agent 配置 + 执行日志 | 平手 |

## 并行执行与任务编排：深度分析

Subagents 最大的优势在于**零配置的并行能力**。当你给 Codex 一个包含多个独立子任务的指令时——比如"重构 auth 模块、更新相关测试、同步 API 文档"——它可以自动将这三个任务分配给不同的 Subagent 同时执行。每个 Subagent 在自己的沙箱环境中工作，互不干扰，完成后由主 Agent 整合结果。

这种模式的实际效果是**显著缩短复杂任务的端到端时间**。在 [Claude Code 的 Agent Teams 实践](/zh/blog/claude-code-agent-teams)中，我们看到了类似的架构思路——主 Agent 负责规划和协调，子 Agent 负责执行。Codex 的 Subagents 遵循同样的分治原则，区别在于 Codex 的沙箱隔离更严格，每个 Subagent 的文件系统访问是独立的。

但 Subagents 的局限也明显：**缺乏跨任务记忆**。每次运行都是从零开始，主 Agent 必须在指令中明确传递所有必要上下文。如果你反复执行同类任务（比如每次 PR 都要做安全审计），每次都需要重新描述审计标准和关注点。

## 角色定义与复用性：深度分析

Custom Agents 的核心竞争力在于**将工程最佳实践固化为可复用的 Agent 配置**。一个典型场景：团队要求所有 PR 必须通过安全扫描。你可以创建一个 Custom Agent，预设它的系统指令（"检查 OWASP Top 10 漏洞模式"）、工具权限（只读代码访问 + 安全扫描工具）和输出格式（标准化的审计报告模板）。

配置完成后，团队任何成员都可以调用这个 Agent，得到一致的审计质量。这解决了 Subagents 的**一致性问题**——你不必依赖每个开发者都能写出精确的自然语言指令来获得相同质量的输出。

Custom Agents 的另一个优势是**权限精细控制**。你可以限制 Agent 只能访问特定目录、只能使用特定工具、只能执行只读操作。这在企业环境中至关重要——尤其当 Agent 需要接触生产数据或敏感配置时。关于 [Agent 编码范式](/zh/glossary/agentic-coding)的更多讨论可参考我们的术语表。

不过，Custom Agents 需要前期投入配置时间。如果你只是做一次性的探索性任务，创建 Custom Agent 的成本超过了收益。

## 什么时候用 Subagents

选择 Subagents 的场景通常具有这些特征：

- **一次性复杂任务**：大规模重构、代码库迁移、多文件批量修改——这些任务结构每次不同，不值得预配置
- **探索性工作**：你还不确定任务的最佳拆解方式，让 Codex 自动规划比手动定义更高效
- **快速原型**：验证一个想法时，Subagents 的零配置启动比设置 Custom Agent 快得多
- **个人开发者工作流**：如果你是独立开发者，Subagents 的灵活性比 Custom Agents 的规范性更实用

关于 Subagents 在实际项目中的更多用法，可以参考我们的[实战案例分析](/zh/blog/claude-code-subagents-examples)。

## 什么时候用 Custom Agents

Custom Agents 适合这些场景：

- **标准化流程**：代码审查、安全扫描、测试生成——每次执行的标准和流程相同
- **团队协作**：需要多人调用同一个 Agent 且期望输出一致时，Custom Agent 的预设配置是唯一可靠方案
- **合规与审计**：需要确保 Agent 的行为可预测、可追溯、权限可控
- **高频重复任务**：如果某个 Agent 工作流每周执行超过 3 次，将其固化为 Custom Agent 能节省大量重复指令编写时间

对于希望深入了解 Codex 在多智能体工作流中全面能力的读者，我们的 [Codex 完全指南](/zh/blog/codex-complete-guide)覆盖了从基础设置到高级编排的完整路径。

## 结论

**Subagents 和 Custom Agents 不是二选一的关系，而是不同层次的工具。** Subagents 是运行时的执行引擎——灵活、自动、适合动态任务拆解。Custom Agents 是设计时的规范框架——可控、可复用、适合标准化流程。

最成熟的做法是**将两者结合**：用 Custom Agents 定义团队的标准化 Agent（安全审计、代码风格检查、测试生成），在每个 Custom Agent 内部允许它按需调度 Subagents 处理并行子任务。这样既保证了行为一致性，又不牺牲执行效率。

如果你刚开始接触 Codex 的多智能体能力，建议从 Subagents 开始——直接描述任务，让 Codex 自动编排。等你积累了足够的 Agent 使用模式后，再将高频模式固化为 Custom Agents。关于类似的渐进式策略，我们在 [Claude Code 多 Agent 协作](/zh/blog/claude-code-agent-teams)和[扩展栈架构](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中有更详细的实践分享。

## 常见问题

### Subagents 和 Custom Agents 能同时使用吗？

可以。Custom Agents 在执行过程中可以调度 Subagents 来并行处理子任务。典型做法是用 Custom Agent 定义任务标准和约束，用 Subagents 处理实际的并行执行——两者是互补关系，不是互斥选择。

### 使用 Subagents 需要额外付费吗？

Subagents 的计费基于实际消耗的计算资源和 token 用量。每个 Subagent 都是独立的 Agent 实例，会产生各自的 API 调用成本。并行执行多个 Subagents 会在短时间内消耗更多额度，但总时间缩短通常意味着更高的性价比。

### Custom Agents 的配置可以版本控制吗？

Custom Agents 的配置可以导出为 JSON 或 YAML 格式，纳入团队的版本控制系统。这意味着 Agent 配置的变更可以像代码一样经过 review、测试和回滚，这对于需要审计追踪的企业团队尤其重要。
