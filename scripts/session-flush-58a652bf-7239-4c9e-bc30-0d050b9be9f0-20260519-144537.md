**Parallel research and implementation.** A parent agent can spawn an Explore agent to search the codebase for all usages of a deprecated API while simultaneously spawning another agent to draft the replacement implementation. The parent waits for both results, synthesizes them, and proceeds.

**Specialized roles.** Claude Code ships with built-in agent types — Explore (fast read-only search), Plan (architecture design), and general-purpose agents. Each has different tool access and behavioral constraints. An Explore agent cannot edit files, which makes it safe for broad codebase scanning without risk of unintended changes.

**Delegated subtasks.** The parent agent can break a large task into independent pieces, spawn agents for each piece, collect their outputs, and integrate the results. For a codebase-wide refactor, this means parallel investigation of different modules followed by coordinated implementation.

**Inter-agent messaging.** Named agents can receive follow-up instructions via SendMessage, allowing the parent to refine a subagent's work without restarting from scratch. This creates a conversational delegation model — closer to how a tech lead delegates to team members than how a task queue dispatches jobs.

The tradeoff is that Claude Code runs locally. Your machine provides the compute, and all agents share your terminal session. For long-running tasks, you need to keep the session alive — though background agents and the remote session feature help mitigate this.

## Custom Agent Definitions: Detailed Analysis

The ability to define custom agents tailored to your project is one of the most significant architectural differences between these two platforms.

### Codex: No Custom Agent Framework

Codex does not offer a mechanism for defining custom agent types. Every Codex task uses the same underlying agent behavior — it reads the task description, works in its sandbox, and produces a diff. You can influence behavior through the task prompt and through repository-level configuration (like a `AGENTS.md` or similar instruction file that Codex reads), but you cannot create distinct agent personas with different tool access, behavioral constraints, or specialized instructions that persist across tasks.

If you want "a security review agent" and "a test-writing agent" in Codex, the differentiation happens entirely in the prompt you write for each task. There is no reusable definition file that locks in the agent's role, tools, or constraints. This means consistency depends on your prompt discipline, not on system-level configuration.

### Claude Code: Project-Level Agent Definitions

Claude Code supports custom agent definitions through `.claude/agents/*.md` files in your repository. Each file defines a named agent type with its own system instructions, available tools, and behavioral guidelines. These definitions are version-controlled alongside your code, which means they evolve with your project and are consistent across all team members.

A custom agent definition might specify:

- **A security reviewer agent** that only has read access to files and grep, cannot edit code, and follows your team's security checklist
- **A test generator agent** that has access to your test framework's conventions and writes tests matching your project's patterns
- **A documentation agent** that reads code changes and updates corresponding documentation files

When the primary Claude Code agent spawns a custom agent, it inherits the defined constraints automatically. The agent's behavior is determined by its definition file, not by the parent's prompt — which makes multi-agent workflows more predictable and less dependent on getting the delegation prompt exactly right.

This is where Claude Code's [skills and agents architecture](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) creates a compounding advantage: custom agents can reference SKILL.md files for task-specific instructions, read CLAUDE.md for project context, and use MCP servers for external tool access. The customization is deep and composable.

## Workflow Comparison: Real-World Scenarios

### Scenario 1: Codebase-Wide Refactor

**With Codex:** You break the refactor into independent tasks manually — "update module A," "update module B," "update shared types." Each runs in its own container. If module B's changes depend on the shared types update, you must wait for the types task to complete and merge before launching the module B task. You are the orchestration layer, monitoring task completion and sequencing dependent work.

**With Claude Code:** You describe the full refactor to the primary agent. It spawns an Explore agent to map all affected files, spawns parallel agents to update independent modules simultaneously using worktree isolation, and handles dependent changes sequentially. The parent agent coordinates the full workflow in a single session. You review the final result rather than managing intermediate steps.

### Scenario 2: Bug Investigation and Fix

**With Codex:** You assign a single task: "investigate and fix the intermittent timeout in the payment service." Codex works in its container, searches the code, proposes a fix. If the investigation reveals the bug spans multiple services, you may need to assign additional tasks for each service, each working from a fresh repo clone.

**With Claude Code:** The primary agent spawns an Explore agent to search for timeout-related code across the codebase, identifies the root cause, then spawns a focused agent to implement the fix while another agent writes regression tests. Results flow back to the parent, which integrates everything into a coherent commit.

### Scenario 3: Async Batch Processing

**With Codex:** This is where Codex excels. You assign 20 independent tasks — backfill tests, fix lint warnings, update docstrings — and walk away. Codex runs all 20 in parallel in the cloud. You come back to 20 completed diffs ready for review. No local machine required, no terminal session to maintain.

**With Claude Code:** You can run background agents, but they require an active terminal session. For truly async, fire-and-forget batch work, Codex's cloud model is more convenient. Claude Code's strength is coordinated multi-step work, not independent batch processing.

## When to Choose OpenAI Codex

**Choose Codex when you need async, independent task execution.** Codex's cloud containers are ideal for:

- **Batch task processing**: Assign many independent tasks and review results later. No local machine overhead, no session management.
- **Low-setup environments**: Codex runs in ChatGPT — no CLI installation, no API key management, no local environment configuration. Ideal for quick tasks or team members who aren't command-line users.
- **Isolated, safe execution**: Each task runs in a sandboxed container that cannot affect your local environment. For teams with strict security requirements around AI tool access, this isolation model is compelling.
- **PR-centric workflows**: Codex naturally produces pull requests from cloud diffs, fitting well into teams that review all AI-generated changes through standard PR review processes.

Codex is the right choice when your tasks are independent and don't require cross-task coordination. If you can describe each unit of work as a self-contained assignment, Codex's parallel container model is efficient and predictable. See the [Codex VS Code extension guide](/blog/codex-vscode) for how this integrates into editor workflows.

## When to Choose Claude Code

**Choose Claude Code when your tasks require coordination, specialization, or deep context.** Claude Code's subagent system excels at:

- **Complex, multi-step workflows**: Refactors, investigations, and feature implementations that span multiple files and require intermediate decisions. The parent-child agent model handles sequencing and integration automatically.
- **Custom agent roles**: Teams that want reusable, version-controlled agent definitions for specific tasks — security review, documentation updates, test generation — benefit from `.claude/agents/` definitions.
- **Deep codebase context**: Claude Code's agents share the live working directory and all project configuration. For tasks that require understanding file relationships, recent changes, or project conventions, this shared context is critical.
- **Interactive workflows**: When you need to guide the agent mid-task — redirecting, refining, or adding constraints — Claude Code's terminal-based interaction model and inter-agent messaging support real-time collaboration.

Claude Code is the right choice when the work is complex enough to benefit from delegation and specialization. ## Verdict

**For subagents and custom agents, Claude Code is the decisive winner.** Codex does not have a subagent system — its parallelism comes from independent task containers with no coordination, no custom agent types, and no inter-agent communication. If "use subagents and custom agents" is your requirement, Claude Code is the only platform that delivers it natively.

That said, **Codex has real strengths in async batch execution and zero-setup cloud environments.** If your workflow is "assign 20 independent tasks and review later," Codex's container model is simpler and requires no local resources.

The practical recommendation: **use Claude Code for coordinated, multi-agent workflows** where tasks depend on each other or require specialized agent roles. **Use Codex for independent, async task batches** where each task is self-contained. Many teams will find value in using both — Claude Code for complex engineering work, Codex for high-volume independent tasks. ## Frequently Asked Questions

### Can you create custom agents in OpenAI Codex?

Codex does not support custom agent definitions. Every task uses the same underlying agent behavior, and differentiation happens through task prompts and repository-level instruction files. There is no equivalent to Claude Code's `.claude/agents/*.md` system for defining reusable agent types with distinct tool access and behavioral constraints.

### How do Claude Code subagents share context?

Claude Code subagents operate within the same repository working directory as the parent agent. They inherit project context from CLAUDE.md files, can read the same files, and return results to the parent agent via the Agent tool response. Named agents can also receive follow-up instructions through SendMessage, enabling iterative refinement without restarting.

### Can Codex tasks communicate with each other?

No. Each Codex task runs in an isolated sandbox container with its own repository clone. Tasks cannot share state, send messages, or access each other's outputs. If Task B depends on Task A's changes, you must wait for Task A to complete and merge before launching Task B.

### Is Claude Code or Codex better for large teams?

It depends on the workflow. Codex's cloud model requires no local setup and works through the ChatGPT interface, making it accessible to team members without terminal experience. Claude Code's custom agent definitions and CLAUDE.md files are version-controlled and consistent across the team, which benefits organizations that need standardized AI-assisted workflows. Large teams often use both.

### Do Claude Code subagents cost extra?

Claude Code subagents consume tokens from the same API billing as the parent agent. Each subagent uses its own context window and generates its own token usage, so spawning multiple agents increases total token consumption. There is no separate per-agent fee — costs scale with total tokens processed across all agents in the session.
