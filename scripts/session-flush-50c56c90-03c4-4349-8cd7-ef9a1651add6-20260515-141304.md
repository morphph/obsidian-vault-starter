**CLAUDE.md loading** is deterministic and hierarchical. When Claude Code starts a session, it reads CLAUDE.md files from multiple locations in a fixed order: the user's home directory (`~/.claude/CLAUDE.md`), any parent directories above the project, the project root, and the `.claude/` subdirectory. Instructions from all levels merge together, with more specific files taking precedence. This means a team can define organization-wide rules in a parent directory and project-specific rules in the repo — both apply simultaneously.

Because CLAUDE.md is loaded into the system prompt at the start of every conversation, its instructions have strong behavioral weight. Claude Code treats CLAUDE.md content as authoritative directives — "run `npm test` before committing" is followed as a rule, not a suggestion. The tradeoff is context window cost: every line in CLAUDE.md consumes tokens in every conversation, whether relevant or not.

**Claude Memory loading** is selective and index-based. The `MEMORY.md` index file is always loaded (capped at roughly 200 lines), but individual memory files are only retrieved when Claude Code judges them relevant to the current conversation. This makes memory more context-efficient for large amounts of stored information — you might have dozens of memory files, but only two or three get loaded per session.

Memory files use structured frontmatter with a `description` field that Claude Code uses for relevance matching. The four memory types serve distinct purposes: `user` memories capture who you are and how you work; `feedback` memories record corrections and validated approaches; `project` memories track ongoing initiatives and decisions; `reference` memories point to external resources. Each type has different persistence characteristics — user and feedback memories tend to be long-lived, while project memories decay as initiatives complete.

The key architectural difference: CLAUDE.md is loaded unconditionally and treated as instructions. Memory is loaded conditionally and treated as context. This distinction matters when you're deciding where to put information.

## Team Workflows: Shared Rules vs Personal Context

The most important practical difference between Claude Memory and CLAUDE.md is audience. CLAUDE.md is a team artifact; memory is a personal one. This distinction drives most "where should I put this?" decisions.

**CLAUDE.md excels at encoding team agreements.** Coding standards, naming conventions, architectural boundaries, required test patterns, deployment procedures — anything that every developer (human or AI) should follow belongs in CLAUDE.md. When a new team member clones the repo and runs Claude Code, they get the full set of project conventions without any setup. This is especially valuable for [agentic coding](/glossary/agentic-coding) workflows where Claude Code operates with significant autonomy — CLAUDE.md constraints keep that autonomy within safe bounds.

Consider a real example: your team decides that all database queries must go through a repository pattern, never direct SQL in route handlers. In CLAUDE.md, this becomes a rule that Claude Code follows every time it generates code, for every developer. In memory, it would only apply to whoever happened to be in the conversation where it was discussed — and it might drift or be forgotten over time.

**Claude Memory excels at personal adaptation.** Your role on the team, your experience level with specific technologies, the way you prefer explanations structured, the mistakes Claude Code has made with you before — these are personal context that shouldn't be imposed on the whole team. A senior backend engineer and a junior frontend developer working on the same repo need different things from Claude Code, and memory handles that differentiation.

Memory also handles temporal project context that doesn't belong in a version-controlled file. "We're in a code freeze until Thursday for the mobile release" is critical context for the next three days but noise after that. Memory can capture it, and Claude Code can later recognize it's stale. CLAUDE.md would require someone to add it and then remember to remove it.

For teams adopting Claude Code's full [extension stack — skills, hooks, agents, and MCP](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) — CLAUDE.md typically serves as the entry point that references these other systems. Memory complements by tracking how individual developers interact with those extensions.

## Content Strategy: What Goes Where

Knowing the architecture and team dynamics, here's a practical decision framework for what content belongs in each system.

**Put it in CLAUDE.md when:**

- It applies to every developer on the project
- It defines a rule or constraint, not a preference
- It needs to be version-controlled and reviewed in PRs
- It's stable — changes infrequently (monthly or less)
- It's actionable — Claude Code should follow it as an instruction
- Examples: build commands, test requirements, style rules, architecture boundaries, known gotchas, deployment procedures

**Put it in Claude Memory when:**

- It applies only to you
- It describes context, not rules — your role, your preferences, your workflow
- It was learned during a conversation — a correction, a validated approach, a discovered constraint
- It may change frequently or become stale
- It's informational — helps Claude Code understand context, not follow a directive
- Examples: your expertise level, preferred explanation style, feedback on Claude Code's behavior, ongoing project initiatives with deadlines, pointers to external tracking systems

**Common mistakes:**

Putting personal preferences in CLAUDE.md — "explain code changes before making them" works for you but annoys your teammate who prefers terse output. This belongs in memory.

Putting critical project rules in memory only — "never commit to main directly" gets forgotten if it's only in one person's memory. This belongs in CLAUDE.md with your branch protection documentation.

Duplicating information across both systems — if CLAUDE.md already says "run `npm test` before committing," a memory that says the same thing wastes context tokens and creates a maintenance burden. For tips on [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) without redundancy, focus each system on its unique role.

## Maintenance and Lifecycle

CLAUDE.md and Claude Memory have fundamentally different maintenance models, and understanding this helps you keep both systems healthy over time.

**CLAUDE.md maintenance is manual and deliberate.** Someone on the team needs to update it when conventions change, and outdated instructions cause active harm — Claude Code will follow stale CLAUDE.md rules even when they conflict with the current codebase state. The best practice is to treat CLAUDE.md like production code: update it in the same PR that changes the convention it documents. Many teams add a CI check or hook that reminds contributors to review CLAUDE.md when certain files change.

The risk with CLAUDE.md is staleness through neglect. A file that was accurate six months ago may now reference deprecated commands, outdated architecture patterns, or removed tools. Regular audits — quarterly at minimum — keep it useful. Some teams use their [Claude Code skills](/blog/5-claude-code-skills-i-use-every-single-day) to include a periodic CLAUDE.md review in their workflow.

**Claude Memory maintenance is semi-automatic.** Claude Code is designed to update and prune its own memories — marking stale project memories as outdated, refining user memories as it learns more about you, and consolidating duplicate entries. In practice, this self-maintenance is imperfect. Memories can accumulate, become contradictory, or persist past their relevance.

The recommended hygiene practice for memory is periodic review. Ask Claude Code "what do you remember about me?" or "show me your project memories" and prune anything outdated. You can also ask it to forget specific items. Unlike CLAUDE.md, a stale memory is usually low-harm — Claude Code treats memories as context rather than directives, so an outdated memory is more likely to be ignored than to cause incorrect behavior.

**Lifecycle comparison:**

| Aspect | Claude Memory | CLAUDE.md |
|--------|-------------|-----------|
| Creation | Automatic during conversations | Manual, authored by developer |
| Updates | Semi-automatic, Claude-initiated | Manual, PR-driven |
| Deletion | On request or self-pruning | Manual removal |
| Staleness risk | Medium — context may be ignored | High — rules will be followed |
| Audit frequency | Monthly review recommended | Quarterly review recommended |
| Recovery | Rebuild through conversation | Restore from git history |

## When to Choose Claude Memory

Choose Claude Memory as your primary persistence mechanism when:

- **You're a solo developer** and the distinction between "personal" and "team" doesn't apply. Memory captures everything — your conventions, your preferences, your project context — without the overhead of maintaining a separate file.
- **You want zero-maintenance persistence.** Memory builds itself as you work. Corrections become feedback memories, your role becomes a user memory, and project context accumulates naturally. You don't need to stop coding to write documentation.
- **Your context is temporal.** Sprint goals, release deadlines, ongoing investigations, and active incidents are better suited to memory's natural lifecycle than CLAUDE.md's version-controlled permanence.
- **You need personal adaptation.** If you work across multiple projects with different teams, memory lets Claude Code adapt to you specifically, regardless of which project's CLAUDE.md is loaded.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence mechanism when:

- **You're on a team** and need consistent AI behavior across all contributors. CLAUDE.md is the only mechanism that guarantees every developer's Claude Code session starts with the same project rules.
- **You need deterministic, auditable rules.** CLAUDE.md instructions are loaded every time, version-controlled, and reviewable in PRs. For compliance-sensitive environments or projects with strict coding standards, this predictability is essential.
- **Your instructions are stable and critical.** Build commands, test requirements, architecture boundaries, and security constraints should survive across team members and time. CLAUDE.md's git-backed persistence ensures they do.
- **You're setting up Claude Code for a project for the first time.** Start with CLAUDE.md — it's the foundation. Add memory naturally as you work. Teams that use [Claude Code hooks](/blog/claude-code-hooks-mastery) and skills alongside CLAUDE.md build a comprehensive automation layer that memory alone cannot provide.

## Verdict

**Use both — but start with CLAUDE.md.** CLAUDE.md is the foundation of Claude Code's project awareness: it defines what Claude Code should and shouldn't do in your codebase, and it's shared with your whole team. Every project using Claude Code should have a well-maintained CLAUDE.md file. Claude Memory builds on top of that foundation by adding personal context, learned preferences, and temporal information that makes Claude Code increasingly effective for you individually over time.

The practical rule: **if removing the information would break Claude Code's behavior for the team, it belongs in CLAUDE.md. If removing it would only affect your personal experience, it belongs in memory.** For the full picture of how these persistence layers fit into Claude Code's broader architecture, read our guide on [Claude Code's extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp).

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions are loaded as authoritative directives in Claude Code's system prompt and take precedence over memory-stored context. If a memory conflicts with a CLAUDE.md rule, Claude Code follows CLAUDE.md. Memory provides supplementary context — it informs Claude Code's decisions but doesn't override explicit project rules.

### Does Claude Memory sync across devices?

Claude Memory is stored locally in the `.claude/projects/` directory on your machine. It does not automatically sync across devices or between team members. If you work on multiple machines, each will develop its own memory. CLAUDE.md, being git-tracked, syncs naturally through your repository across any machine that clones it.

### How much should I put in CLAUDE.md before it hurts performance?

CLAUDE.md content is loaded into every conversation's context window, consuming tokens regardless of relevance. Aim for under 500 lines of focused, actionable instructions. If your CLAUDE.md exceeds that, consider moving reference material into skill files (`skills/*/SKILL.md`) that are loaded on-demand, keeping CLAUDE.md for rules and commands that apply to every session.

### Can I convert memory entries into CLAUDE.md rules?

Yes, and this is a recommended practice. When you notice Claude Memory has captured a correction or pattern that applies to the whole team — not just you — promote it to CLAUDE.md. Ask Claude Code to show you its feedback memories, identify the team-relevant ones, and add them as explicit rules in your project's CLAUDE.md file. Then remove the memory entry to avoid duplication.

### Should I check Claude Memory files into git?

No. Memory files are personal, user-specific context that would conflict across team members. The `.claude/` directory containing memory files should be in your `.gitignore`. CLAUDE.md is the git-tracked persistence layer; memory is the local, personal one. Mixing these roles defeats the purpose of having separate systems.
