# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR:** **Claude Memory**（Auto Memory）是个人化的跨会话自动记忆，Claude Code 在对话中自动学习你的偏好并写入本地文件。**CLAUDE.md** 是项目级的显式指令文件，签入 Git 仓库，团队共享。两者不是竞争关系——**CLAUDE.md 定义项目规则，Memory 记录个人上下文**，搭配使用效果最佳。

## 概述：Claude Memory（Auto Memory）

**Claude Memory** 是 [Claude Code](/glossary/claude-code) 的跨会话持久记忆系统，解决的核心问题是：每次开新对话都要重新解释「我是谁、我喜欢什么风格、上次聊到哪了」。Memory 以 Markdown 文件形式存储在 `~/.claude/projects/` 目录下，包含用户画像、工作反馈、项目状态和外部资源引用四种类型。

Memory 的关键特征是**自动化**。Claude Code 在对话过程中识别值得记住的信息（比如你纠正了它的某个行为、提到了自己的角色），自动写入记忆文件。下次对话开始时，这些记忆被加载进上下文，让 Claude Code 从「陌生人」变成「了解你的同事」。

Memory 是**个人私有**的，不签入代码仓库。它跟随用户，不跟随项目。详细的记忆系统设计可参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是放在项目根目录的 Markdown 指令文件，签入 Git、团队共享。它告诉 Claude Code 这个项目是什么、怎么构建、有什么规矩。类似于 `.editorconfig` 或 `CONTRIBUTING.md`——但读者不是人，是 AI。

CLAUDE.md 解决的问题是**项目一致性**。无论哪个团队成员启动 Claude Code，读到的规则都一样：构建命令、代码风格、架构约束、禁止事项。它是显式的、版本控制的、团队级别的。

CLAUDE.md 还支持层级结构：全局 `~/.claude/CLAUDE.md` 定义个人通用偏好，项目根目录的 `CLAUDE.md` 定义项目规则，子目录的 `CLAUDE.md` 可以覆盖或补充上层规则。更深入的实践可参考 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)。

|------|--------------|-----------|-----------|

## 核心差异：显式规则 vs 隐式学习

这是两套系统最根本的区别。

**CLAUDE.md 是显式规则系统。** 你写下「commit 前必须跑 `npm test`」，Claude Code 就会遵守。规则清晰、可预测、可审计。团队新成员加入时不需要额外配置，克隆仓库即刻生效。但维护成本在你——规则过时了不会自动更新，写得太细会变成负担。

**Memory 是隐式学习系统。** 你在对话中说「别在测试里 mock 数据库」，Claude Code 记住了，下次自动避免。你不需要主动维护，AI 替你管理。但这也意味着你看不到全貌——记忆分散在多个文件里，可能包含过时信息，也可能丢失微妙的上下文。

实际工程中，这两种模式覆盖不同的需求层级：

- **项目级硬规则** → CLAUDE.md（比如：构建命令、禁止的模式、代码风格）
- **个人软偏好** → Memory（比如：你喜欢简洁回复、你是后端工程师、你上次在调某个 bug）
- **团队协作约定** → CLAUDE.md（比如：PR 模板、分支策略、部署流程）
- **个人工作上下文** → Memory（比如：你负责哪个模块、你本周的任务优先级）

想了解 Claude Code 完整的分层架构设计，推荐阅读 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 实际场景：该用哪个？

### 场景一：新项目初始化

**用 CLAUDE.md。** 在项目根目录创建 CLAUDE.md，写明技术栈、构建命令、代码规范和架构决策。这是团队共识的固化，每个使用 Claude Code 的人都会遵守。Memory 在这个阶段没什么用——还没有对话历史可以学习。

### 场景二：个人开发习惯

**用 Memory。** 你喜欢 commit message 用中文、喜欢简短回复、讨厌 Claude Code 在每次操作后总结一遍——这些是个人偏好，不该强加给团队。纠正 Claude Code 一次，它自动记住，下次不再犯。

### 场景三：跨项目的通用偏好

**用全局 CLAUDE.md（`~/.claude/CLAUDE.md`）。** 这是 CLAUDE.md 的个人层级，不签入任何仓库，但每个项目都会加载。适合放「永远先 git pull」「不要过度设计」这类你在所有项目中都想遵守的原则。

### 场景四：长期项目的上下文积累

**两个都用。** CLAUDE.md 存稳定的项目架构信息，Memory 存演进中的项目状态。比如 CLAUDE.md 写「数据库用 SQLite」，Memory 记「当前正在做从 SQLite 迁移到 PostgreSQL 的调研」。[Anthropic Claude 记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)后，Memory 的跨会话连续性更强了。

## 常见误区

**误区一：「有了 Memory 就不需要 CLAUDE.md 了」**

错。Memory 是个人的、隐式的、不可审计的。团队项目中，你不能指望每个人的 Memory 都恰好记住了相同的规则。CLAUDE.md 是 single source of truth。

**误区二：「CLAUDE.md 越详细越好」**

也错。CLAUDE.md 应该只包含硬性规则和关键上下文。写太多，Claude Code 的注意力被稀释。日常偏好交给 Memory，可复用的任务流程交给 [SKILL.md](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

**误区三：「Memory 会自己保持最新」**

部分正确。Memory 会自动写入新信息，但不会自动清理过时信息。如果项目架构发生了大变更，旧的 Memory 记录可能误导 Claude Code。定期审查 `~/.claude/projects/` 下的记忆文件是个好习惯。

## 最佳搭配策略

| 信息类型 | 存放位置 | 原因 |
|---------|---------|------|
| 构建 / 测试 / 部署命令 | CLAUDE.md | 团队共享，版本控制 |
| 代码风格和禁止模式 | CLAUDE.md | 需要可审计、可 review |
| 架构决策和设计约束 | CLAUDE.md | 稳定信息，变化慢 |
| 个人回复风格偏好 | Memory | 个人私有，不影响团队 |
| 当前任务上下文 | Memory | 临时性，跨会话有用 |
| 用户角色和技术背景 | Memory | 个性化交互 |
| 可复用的任务流程 | SKILL.md | 专用系统，比 CLAUDE.md 更结构化 |

最有效的配置是三层协同：**CLAUDE.md 定义规则，SKILL.md 定义流程，Memory 记录上下文**。Claude Code 在每次会话开始时同时加载这三层信息，形成完整的工作语境。关于扩展栈的完整架构，参见 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 结论

**如果你是独立开发者**，Memory 的自动化特性可能足够应付大部分场景，但仍然建议维护一个轻量的 CLAUDE.md 记录构建命令和核心约束。**如果你在团队中工作**，CLAUDE.md 是不可替代的——它是 AI 行为一致性的唯一保障。Memory 作为个人层补充，让每个人都能获得个性化的交互体验。

两者不是二选一的关系。把它们想象成「公司规章制度」和「个人备忘录」——前者所有人遵守，后者只对自己有用，缺了任何一个工作效率都会打折扣。

## 常见问题

### Claude Memory 存储在哪里？
Claude Memory 以 Markdown 文件形式存储在本地 `~/.claude/projects/` 目录下，按项目路径组织。这些文件不会被上传到云端，也不会签入 Git 仓库，完全属于个人私有数据。

### CLAUDE.md 支持多层级配置吗？
支持。Claude Code 会按优先级加载三个层级的 CLAUDE.md：全局（`~/.claude/CLAUDE.md`）、项目根目录、当前工作子目录。子目录的规则可以补充或覆盖上层规则，适合 monorepo 中不同模块有不同约定的场景。

### Memory 记错了怎么办？
直接告诉 Claude Code「忘掉 XXX」，它会找到并删除对应的记忆文件。你也可以手动编辑或删除 `~/.claude/projects/` 下的 Markdown 文件。定期审查记忆内容，清理过时条目，是保持 Memory 系统有效的关键。

### 团队中应该只用 CLAUDE.md 吗？
不建议。CLAUDE.md 解决团队一致性，Memory 解决个人效率。两者协同才是最优解。团队的硬性规则写进 CLAUDE.md 并签入仓库，个人偏好让 Memory 自动管理。
