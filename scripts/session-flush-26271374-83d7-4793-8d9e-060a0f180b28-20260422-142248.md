# Claude Memory vs CLAUDE.md：两套记忆系统怎么选？

**TL;DR：** **Claude Memory**（自动记忆）负责跨会话积累用户偏好和项目上下文，属于"学习型"记忆；**CLAUDE.md** 是手动编写的项目级指令文件，属于"规则型"配置。两者不是竞争关系——Memory 记住你是谁，CLAUDE.md 规定项目怎么做。最佳实践是同时使用：硬性规则写进 CLAUDE.md，软性偏好交给 Memory 自动积累。

## 概述：Claude Memory

Claude Memory 是 Claude Code 的持久化记忆系统，存储在 `.claude/projects/` 目录下，以 Markdown 文件形式组织。它的核心设计理念是**让 AI 在多次对话中逐步了解你**——你的角色、偏好、项目背景、工作习惯，以及你给过的反馈和纠正。

Memory 分为四种类型：用户记忆（你是谁）、反馈记忆（你希望我怎么做）、项目记忆（当前在做什么）和引用记忆（外部资源在哪里）。每种类型都有独立的 Markdown 文件，通过 `MEMORY.md` 索引串联。详细机制可参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

关键特征：Memory 是**自动触发**的。当用户纠正 Claude 的行为（"别这样做"）或确认某个做法（"对，就这样"），Claude 会主动将这些信息写入记忆文件，无需手动操作。

## 概述：CLAUDE.md

CLAUDE.md 是 Claude Code 的项目级指令文件，放在仓库根目录或 `.claude/` 目录下，随代码一起提交到 Git。它的定位是**确定性的规则手册**——构建命令、代码风格、提交规范、架构约束，所有团队成员共享同一份配置。

CLAUDE.md 支持多层级覆盖：全局级（`~/.claude/CLAUDE.md`）、项目级（仓库根目录）、目录级（子目录内的 CLAUDE.md）。每一层都可以添加或覆盖上层的规则。这套分层机制是 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中的基础层，决定了 AI 如何理解和操作你的代码库。

关键特征：CLAUDE.md 是**手动维护**的。内容不会自动更新——你写什么，Claude 就执行什么，具有高度的可预测性和可审计性。

## 记忆机制：深度对比

Claude Memory 和 CLAUDE.md 在底层机制上有本质区别。理解这个区别是正确使用它们的前提。

**CLAUDE.md 是确定性注入。** 每次 Claude Code 启动会话时，它会自动读取所有层级的 CLAUDE.md 文件并注入到系统提示词中。这意味着 CLAUDE.md 中的每一条规则都会被执行——不存在"Claude 忘了看 CLAUDE.md"的情况。这使它成为质量门禁（"提交前必须跑测试"）和硬性约束（"不要在 pipeline 脚本里 import Next.js 模块"）的理想载体。

**Memory 是概率性召回。** Claude 在会话开始时加载 `MEMORY.md` 索引，但具体的记忆文件只在 Claude 判断相关时才会读取。这意味着 Memory 更适合存储"有用但不强制"的信息——比如你喜欢简洁的回复风格，或者你是后端工程师第一次碰前端代码。如果某条规则被遗忘会导致严重后果（构建失败、安全问题），它不应该只存在 Memory 里。

从 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)的视角看，CLAUDE.md 处于最底层的"配置层"，Memory 处于更上层的"学习层"。配置层提供确定性基础，学习层提供个性化增强。

## 团队协作：谁该用什么

这是选择 Memory 还是 CLAUDE.md 时最关键的决策维度。

**选 CLAUDE.md** 的场景：任何需要团队一致性的规则。代码风格、提交规范、构建命令、测试要求、架构约束——这些写进 CLAUDE.md，提交到 Git，团队所有人共享同一份配置。新成员 clone 仓库后，Claude Code 立即知道该遵循什么规则。参考 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)中的分层设计思路，CLAUDE.md 的内容应该是"对这个项目永远成立"的规则。

**选 Memory** 的场景：个人偏好和个人上下文。你习惯用中文交流、你是数据科学家背景、你讨厌冗长的总结——这些是属于你个人的信息，不需要也不应该强加给团队其他成员。Memory 文件不会被提交到 Git，所以每个团队成员可以有自己的记忆配置。

一个实际的例子：假设团队约定"commit message 用英文，遵循 Conventional Commits 格式"。这条规则写进 CLAUDE.md。但你个人偏好 Claude 在讨论时用中文、只在最终 commit message 切换英文——这个偏好存入 Memory。

## 最佳搭配策略

Memory 和 CLAUDE.md 的关系不是"选一个"，而是"怎么配合"。

**规则写 CLAUDE.md，偏好交 Memory。** 这是最简洁的判断标准。如果某条信息的缺失会导致构建失败、代码质量下降、或者违反团队规范——写进 CLAUDE.md。如果缺失只会让体验不够个性化——交给 Memory。

**CLAUDE.md 不要写太长。** 它在每次会话都会被完整注入，过长的 CLAUDE.md 会占用上下文窗口。保持精简，只写真正的规则和约束。背景信息、设计决策的详细解释、历史上下文——这些更适合放在 Memory 或者文档中。

**定期清理 Memory。** Memory 文件会随时间积累，可能出现过时或矛盾的信息。Claude 会尽量在发现冲突时更新记忆，但定期人工审查 `.claude/projects/*/memory/` 目录是个好习惯。Anthropic 近期也在 [升级记忆系统](/zh/blog/anthropic-claude-memory-upgrades-importing)，支持从其他平台导入记忆数据。

**善用分层。** 全局 CLAUDE.md 放通用规则（Git 工作流、通用编码风格），项目 CLAUDE.md 放项目特定规则（构建命令、技术栈约束），目录级 CLAUDE.md 放模块特定规则。Memory 层面也类似：全局记忆放个人身份信息，项目记忆放项目特定上下文。

## 判断结论

**如果你只能选一个**，选 CLAUDE.md——它提供确定性的规则执行，是团队协作的基础。**如果你想要最佳体验**，两个都用：CLAUDE.md 定义"项目的规则"，Memory 记住"你的偏好"。对于 [agentic coding](/zh/glossary/agentic-coding) 工作流来说，这两层配合才能让 AI 既遵循项目规范，又适应个人风格。

## 常见问题

### Claude Memory 会自动写入 CLAUDE.md 吗？

不会。两套系统完全独立。Memory 写入 `.claude/projects/*/memory/` 目录下的 Markdown 文件，CLAUDE.md 是你手动维护的项目配置文件。Claude 不会自动把学到的偏好写进 CLAUDE.md，也不会把 CLAUDE.md 中的规则复制到 Memory。

### CLAUDE.md 太长会有什么影响？

CLAUDE.md 在每次会话启动时被完整注入到系统提示词中，过长的内容会占用宝贵的上下文窗口空间，可能导致后续对话中可用的上下文减少。建议保持 CLAUDE.md 精简——只写硬性规则和关键约束，背景说明放到文档或 Memory 中。

### 团队成员之间的 Memory 会冲突吗？

不会。Memory 是本地文件，不提交到 Git，每个开发者有自己独立的记忆空间。团队一致性通过 CLAUDE.md 保证，个人偏好通过 Memory 各自维护，两者互不干扰。

### 如果同一条信息在 Memory 和 CLAUDE.md 中冲突了，Claude 听谁的？

CLAUDE.md 优先。因为 CLAUDE.md 作为系统提示词的一部分被确定性加载，而 Memory 是辅助性的上下文补充。在实践中应避免这种冲突——硬性规则只写在 CLAUDE.md 中，不要在 Memory 里存储可能与项目规则矛盾的偏好。
