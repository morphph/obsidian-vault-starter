**CLAUDE.md wins conflicts.** If a memory contradicts a CLAUDE.md instruction, CLAUDE.md takes precedence. This is by design — the team's shared configuration shouldn't be overridden by one developer's personal context. Memory is advisory; CLAUDE.md is authoritative.

This two-layer model means you get consistency where it matters (build commands, quality gates, architectural constraints) and personalization where it helps (communication style, role-specific context, workflow preferences). ## What Goes in CLAUDE.md vs Memory: Decision Rules

This is where most developers get confused. Here are concrete rules for what belongs where.

### Put it in CLAUDE.md if:

- **Multiple team members need it.** Build commands, test commands, lint rules, deployment steps — anything where inconsistency across developers would cause problems.
- **It's a hard constraint.** "Never skip failing tests." "Don't import server-only modules in pipeline scripts." "Chinese content must use CJK word count." These are rules, not preferences.
- **It changes with the code.** If a new migration changes the database schema, the corresponding instructions should update in the same PR. CLAUDE.md travels with the codebase because it *describes* the codebase.
- **It's verifiable.** You can check whether Claude Code followed a CLAUDE.md instruction by looking at the output. "Run `npm test` before commit" is verifiable. "Remember that I prefer concise responses" is not.

### Put it in Memory if:

- **It's about you, not the project.** Your role, your expertise level, your communication preferences. "I'm a senior engineer" doesn't belong in a shared config file.
- **It's a correction or confirmed approach.** When you tell Claude Code "don't mock the database in tests" and explain why, that's feedback memory — a learned behavior specific to your working relationship.
- **It's about project state that changes frequently.** "Merge freeze starts April 5 for mobile release" is useful context but would clutter CLAUDE.md and go stale fast.
- **It points to an external resource.** "Pipeline bugs are tracked in Linear project INGEST" — this is a reference memory that helps Claude Code find information, not a rule it needs to enforce.

### Gray Areas

Some information genuinely fits in either place. A useful heuristic: **if you'd put it in a PR description, it belongs in CLAUDE.md. If you'd say it in a 1:1, it belongs in Memory.**

For example, "we're ripping out the old auth middleware because legal flagged session token storage" could go in either place. If the migration is an active project that multiple developers are working on, add it to CLAUDE.md so everyone's Claude Code sessions understand the context. If it's background context that only you need to know for your specific task, let Memory capture it naturally.

## Content Lifecycle: How Each One Evolves

### CLAUDE.md Lifecycle

CLAUDE.md changes through deliberate human action. A developer edits the file, commits it, and pushes. The change goes through code review like any other configuration update. This means CLAUDE.md is always intentional, always reviewed, and always auditable through git history.

The tradeoff is maintenance overhead. CLAUDE.md files can go stale if developers forget to update them when the codebase changes. A common pattern is to include a "Documentation Rules" section in CLAUDE.md itself — instructing Claude Code to update relevant docs when it modifies certain files. This creates a self-maintaining loop where the AI helps keep its own instructions current.

Stale CLAUDE.md instructions are worse than missing ones. A missing instruction means Claude Code falls back to its defaults. A stale instruction means Claude Code confidently follows wrong rules. Regular audits — even quarterly — prevent this.

### Memory Lifecycle

Memory changes through organic conversation. Claude Code writes memories when it learns something worth retaining: a user correction, a role detail, a project decision. It also updates or removes memories when they become outdated. The system is designed to be low-maintenance — you shouldn't need to manually manage your memory files, though you can.

Memory has a built-in staleness awareness. Before acting on a recalled memory, Claude Code is instructed to verify that the information is still current by checking the actual code or system state. A memory that says "the API uses REST" gets verified by checking whether the API still uses REST. If reality has changed, the memory gets updated.

The tradeoff is less predictability. You can't review your teammate's memories in a PR. You can't guarantee that two developers have the same memory state. This is acceptable because memories are personal context, not shared rules — but it means Memory alone isn't sufficient for team alignment.

## Common Mistakes

### Mistake 1: Putting Team Rules in Memory Instead of CLAUDE.md

"Remember that we always run tests before committing." If you say this in conversation, Claude Code might save it as a feedback memory — but only for you. Your teammate's Claude Code won't know about this rule. Put it in CLAUDE.md where it's enforced for everyone.

### Mistake 2: Overloading CLAUDE.md with Personal Preferences

"I prefer short responses without trailing summaries." This is a personal communication preference. Adding it to CLAUDE.md forces everyone on the team into your preferred style. Let Memory handle personal preferences so each developer gets their own experience.

### Mistake 3: Duplicating Information Across Both

If your CLAUDE.md says "deployment target is Vercel" and your memory also says "deployment target is Vercel," you've created a maintenance burden. When the deployment target changes, you need to update both. Worse, if only one gets updated, Claude Code sees conflicting information. Rule of thumb: each fact should live in exactly one place.

### Mistake 4: Using Memory for Architecture Decisions

"We chose PostgreSQL over SQLite for the user service." If this is an architectural decision that affects how Claude Code writes code, it belongs in CLAUDE.md or in the codebase itself (where Claude Code can read it from the code). Memory is for context that helps Claude Code understand *why* decisions were made, not the decisions themselves.

### Mistake 5: Ignoring Memory Entirely

Some developers disable or ignore Memory because they want full control. This works but sacrifices personalization. Claude Code will ask you the same clarifying questions every session, won't remember your corrections, and won't adapt to your expertise level. Memory exists to reduce friction — the cost of ignoring it is repeating yourself.

## Advanced Patterns

### Pattern 1: CLAUDE.md References Memory Categories

A sophisticated CLAUDE.md can instruct Claude Code on what *types* of information to prioritize in memory. For example:

```markdown
## Memory Priorities
When learning new context, prioritize saving:
- External system locations (Linear projects, Grafana dashboards)
- Team member roles and responsibilities
- Active incidents and their status
```

This doesn't put the actual information in CLAUDE.md — it tells Claude Code what to pay attention to and retain.

### Pattern 2: Layered CLAUDE.md Files

Claude Code supports CLAUDE.md files at multiple directory levels. A monorepo might have:

- Root `CLAUDE.md`: Global build commands, shared constraints
- `packages/api/CLAUDE.md`: API-specific conventions, endpoint patterns
- `packages/frontend/CLAUDE.md`: Component standards, styling rules

Each directory's CLAUDE.md adds to (or overrides) the parent's instructions. This creates a hierarchy that mirrors the codebase structure. Combined with personal Memory, each developer working on a specific package gets precisely the right context: shared package rules from CLAUDE.md plus personal workflow preferences from Memory.

### Pattern 3: Memory as Onboarding Accelerator

New team members benefit disproportionately from Memory. As they ask Claude Code questions about the codebase, their memory builds up a personalized knowledge base: what they've learned, what confused them, what patterns they're still getting used to. After a few sessions, their Claude Code "knows" their learning trajectory and can provide appropriately detailed explanations — more detail for unfamiliar areas, less for areas they've already mastered.

CLAUDE.md, by contrast, gives the same level of detail to everyone. It doesn't know that the new developer already understands the database layer but is still learning the deployment pipeline.

### Pattern 4: Skills Bridge Both Layers

[SKILL.md files](https://loreai.dev/blog/5-claude-code-skills-i-use-every-single-day) represent a middle ground between CLAUDE.md and Memory. Like CLAUDE.md, they're checked into the repo and shared with the team. But they're invoked on-demand for specific tasks rather than loaded every session. A well-designed skill system reduces what needs to go in CLAUDE.md (keeping it focused on universal rules) while providing task-specific instructions that Memory alone can't reliably reproduce.

For guidance on writing effective skills, see our [9 principles for Claude Code skills](https://loreai.dev/blog/9-principles-writing-claude-code-skills).

## When to Choose Claude Memory

Choose Memory as your primary persistence layer when:

- **You're a solo developer.** Without team alignment concerns, Memory's automatic context-building is pure upside. You still want a minimal CLAUDE.md for build commands and key constraints, but Memory handles the rest.
- **Your workflow is exploratory.** If you're investigating a codebase, debugging intermittent issues, or doing research-heavy work, Memory captures the evolving context naturally. You don't need to stop and update a config file every time you learn something new.
- **You value personalized AI interaction.** If you want Claude Code to adapt to your communication style, remember your expertise level, and learn from your corrections, Memory is the mechanism that enables this.
- **You work across many repos.** Global memory (`~/.claude/CLAUDE.md` handles some of this, but Memory's per-project storage means each repo gets its own learned context without manual configuration.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence layer when:

- **You're on a team.** Any rule that affects code quality, build processes, or architecture must be in CLAUDE.md so every developer's Claude Code enforces it consistently.
- **Reproducibility matters.** If a new developer clones the repo and runs Claude Code, CLAUDE.md ensures they get the right instructions immediately. Memory starts empty.
- **You need auditability.** CLAUDE.md changes are tracked in git history. You can see who changed what instruction, when, and why. Memory changes are invisible to teammates.
- **You want deterministic behavior.** CLAUDE.md instructions are followed every session, exactly as written. Memory is advisory — Claude Code verifies and may override stale memories. For critical rules, CLAUDE.md's determinism is what you want.

## Verdict

**Use both.** CLAUDE.md and Claude Memory solve different problems, and trying to use one for the other's job creates friction. **CLAUDE.md is your team's shared configuration** — build commands, quality gates, architectural rules, workflow conventions. It's version-controlled, reviewed in PRs, and enforced deterministically. **Claude Memory is your personal context layer** — your role, your preferences, your corrections, your project knowledge. It's automatic, adaptive, and private.

The practical setup for most teams: invest 30 minutes writing a solid CLAUDE.md with your build commands, test requirements, and top 5 architectural constraints. Then let Memory handle everything else naturally. Review CLAUDE.md quarterly to remove stale instructions. Don't manage Memory manually unless something is clearly wrong.

If you're choosing between investing time in one or the other, **start with CLAUDE.md**. A good CLAUDE.md delivers value from the first session. Memory needs a few conversations to build up useful context. But don't skip Memory — the compounding returns of an AI that remembers your working style are substantial over weeks and months. ## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md instructions take precedence over memory in all cases. Memory provides advisory context — your role, preferences, and learned corrections — but it cannot override explicit rules defined in CLAUDE.md. If a memory contradicts a CLAUDE.md instruction, Claude Code follows CLAUDE.md and may update the stale memory.

### Does Claude Memory sync across machines?

No. Claude Memory is stored locally in `~/.claude/projects/` and does not sync between machines or team members. If you work on multiple computers, each machine builds its own memory independently. CLAUDE.md, by contrast, travels with the repo via git and is identical on every machine that clones it.

### How much should I put in CLAUDE.md?

Keep CLAUDE.md focused on what matters for code correctness and team consistency. Build commands, test requirements, key architectural constraints, and critical workflow rules. A typical effective CLAUDE.md is 30-80 lines. If yours exceeds 200 lines, you're likely including information that belongs in documentation, skills files, or Memory instead.

### Can I see what Claude Memory has stored about me?

Yes. Memory files are plain markdown stored in `~/.claude/projects/` directories, organized by project. Each memory has a frontmatter block with a name, description, and type. The `MEMORY.md` index file lists all active memories for a project. You can read, edit, or delete any memory file directly.

### Do I need CLAUDE.md if I'm working solo?

A minimal CLAUDE.md is still valuable for solo projects. At minimum, include your build and test commands so Claude Code runs the right things automatically. Skip the team-oriented rules and let Memory handle the rest. Even a 10-line CLAUDE.md saves you from repeating "run npm test before committing" every session.
