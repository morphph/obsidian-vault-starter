# Claude Memory vs CLAUDE.md：两套记忆系统到底该用哪个？

**TL;DR：** **Claude Memory** 是 Anthropic 的跨会话自动记忆功能，记住你的个人偏好和事实信息；**CLAUDE.md** 是项目级指令文件，定义代码规范、工作流约束和项目上下文。两者不是竞争关系——Memory 记住"你是谁"，CLAUDE.md 定义"项目怎么做"。**大多数开发者应该同时使用两者**，但如果只能选一个，团队协作场景优先 CLAUDE.md，个人效率场景优先 Memory。

## 概述：Claude Memory

**Claude Memory** 是 Anthropic 在 Claude 产品线中引入的持久化记忆功能，让 Claude 能够跨对话记住关于你的信息。它解决的核心问题是：每次开新对话都要重复说明自己的背景、偏好和工作方式。

Memory 有两种工作模式。第一种是显式记忆——你直接告诉 Claude"记住我是前端工程师"或"我偏好 TypeScript"，Claude 将这些信息存入持久化存储。第二种是自动记忆——Claude 在对话过程中自动识别值得保留的信息并存储，无需你主动触发。

2026 年初，Anthropic 对 Memory 进行了[重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，向免费用户开放了记忆功能，并支持从 ChatGPT 导入历史数据。这意味着即使你是从其他 AI 助手迁移过来，也不需要从零开始建立 Claude 对你的了解。

在 Claude Code 中，Memory 以 `~/.claude/` 目录下的文件形式存在，包含用户偏好、反馈记录、项目上下文和外部资源引用等类型。每次新对话开始时，相关记忆会自动加载到上下文中。

## 概述：CLAUDE.md

**CLAUDE.md** 是一个 Markdown 格式的项目指令文件，放在代码仓库的根目录，随代码一起版本控制。它告诉 Claude Code 这个项目是什么、怎么构建、有哪些规范——本质上是一份"项目说明书"。

CLAUDE.md 解决的问题完全不同：不是"Claude 不了解我"，而是"Claude 不了解这个项目"。一个典型的 CLAUDE.md 文件包含技术栈说明、构建命令、代码规范、已知陷阱和工作流约束。

这个文件的关键特征是**团队共享**。它被 `git commit` 进代码仓库，团队里每个人用 Claude Code 时都会自动加载相同的指令。这保证了 AI 在不同开发者之间行为一致——不会因为某个人忘了说明"我们用 Tailwind v4"就生成错误的 CSS。

CLAUDE.md 是 [Claude Code 七层可编程架构](/zh/blog/claude-code-seven-programmable-layers)的基础层，也是整个 [Skills、Hooks、Agents、MCP 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)的入口点。没有 CLAUDE.md，Claude Code 对你的项目几乎是"盲人摸象"。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **作用域** | 个人级，跨所有项目 | 项目级，跨所有团队成员 | 各有所长 |
| **存储位置** | `~/.claude/` 用户目录 | 项目根目录，随 Git 管理 | CLAUDE.md |
| **版本控制** | 不受 Git 管理 | 受 Git 管理，可追溯变更 | CLAUDE.md |
| **团队共享** | 仅个人可见 | 团队全员自动生效 | CLAUDE.md |
| **内容类型** | 个人偏好、角色、反馈 | 项目规范、命令、约束 | 各有所长 |
| **更新方式** | 对话中自动/手动更新 | 手动编辑文件 | Memory |
| **跨项目生效** | 是 | 否，仅当前项目 | Memory |
| **可审计性** | 低，个人文件 | 高，Git 历史可查 | CLAUDE.md |

## 核心差异：持久化机制

这两套系统最根本的区别在于它们持久化信息的方式和生命周期。

Claude Memory 是**用户绑定**的。无论你打开哪个项目、启动哪次对话，Memory 中的信息都会自动加载。它的更新是动态的——Claude 在对话中发现你纠正了它的行为（比如"不要在这些测试里 mock 数据库"），就会自动记录这条反馈，下次遇到类似场景时主动遵守。这种机制特别适合积累个人工作风格：你的技术栈偏好、代码审查习惯、沟通方式等。

CLAUDE.md 是**项目绑定**的。它只在 Claude Code 打开对应项目时加载，但对该项目的所有使用者生效。它的更新是显式的——有人手动编辑文件、提交到 Git。这种机制适合编码标准、构建流程、已知 bug 等项目级知识。当新成员加入团队，不需要任何额外配置，Claude Code 就会按照团队规范工作。

一个实际的例子：如果你告诉 Memory"我偏好函数式编程风格"，那你在所有项目里都会得到函数式的代码建议。但如果团队的 CLAUDE.md 里写了"本项目使用 class-based 组件"，Claude Code 在这个项目里会遵守 CLAUDE.md 的约束，而不是你的个人偏好。**项目指令优先于个人记忆**——这是一个重要的优先级规则。

## 实际应用：谁该用什么

理解这两者的区别后，关键问题是：什么信息放 Memory，什么信息放 CLAUDE.md？

**放 Memory 的信息：**
- 你的技术背景和角色（"我是全栈工程师，Go 写了十年，React 刚上手"）
- 你对 Claude 行为的反馈（"不要在每次回复末尾总结你做了什么"）
- 你的工作习惯偏好（"commit message 用英文"）
- 外部资源引用（"bug 追踪在 Linear 的 INGEST 项目里"）

**放 CLAUDE.md 的信息：**
- 技术栈和依赖版本（"Next.js 16 + TypeScript + Tailwind v4"）
- 构建和测试命令（"npm run build, npm test"）
- 代码规范和约束（"中文内容必须用 CJK 字数统计"）
- 已知陷阱（"不要在 pipeline 脚本里 import Next.js 模块"）
- 质量门禁（"提交前必须通过 build + test + lint"）

简单的判断标准：**如果这条信息换个项目还有用，放 Memory；如果换个人也要遵守，放 CLAUDE.md。**

## 进阶用法：CLAUDE.md 的层级系统

很多开发者不知道 CLAUDE.md 不只是根目录的一个文件。Claude Code 实际上支持多层级的指令文件：

1. **`~/.claude/CLAUDE.md`**——全局个人指令，对你的所有项目生效
2. **项目根目录 `CLAUDE.md`**——项目级指令，随代码提交，团队共享
3. **子目录 `CLAUDE.md`**——模块级指令，只在 Claude Code 操作该目录时加载

这种层级设计让你可以精细控制不同范围的行为。全局 CLAUDE.md 可以定义你的 Git 工作流（比如"每次改动后立即 commit + push"），项目 CLAUDE.md 定义技术栈约束，子目录 CLAUDE.md 定义特定模块的规范。

结合 Memory 使用时，加载优先级是：项目 CLAUDE.md > 全局 CLAUDE.md > Memory。也就是说，最具体的项目指令永远优先。

## 选择建议

如果你是**独立开发者**，两者都值得配置，但 Memory 的投入产出比更高。花 5 分钟告诉 Claude 你的背景和偏好，就能在所有项目中获得更个性化的体验。CLAUDE.md 在项目复杂到你自己也记不清所有约束时才变得必要。

如果你在**团队中工作**，CLAUDE.md 是必选项。没有它，团队成员各自给 Claude 不同的指令，生成的代码风格、测试策略、提交习惯都不一致。这是 [Claude Code 不仅仅是编码工具](/zh/blog/claude-code-is-not-a-coding-tool)的一个具体体现——它是团队工程规范的载体。

最佳实践是**两者配合**：团队规范写进 CLAUDE.md 并提交到 Git，个人偏好和工作习惯让 Memory 自动积累。当两者冲突时，CLAUDE.md 优先——这保证了团队一致性。

如果你想深入了解 Claude Code 的完整记忆体系（包括 Memory、CLAUDE.md、SKILL.md 和 Auto Memory 四层），推荐阅读[Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。两者服务于不同目的——Memory 存储个人偏好和跨项目知识，CLAUDE.md 存储项目级规范和约束。Claude Code 会同时加载两者，当内容冲突时以 CLAUDE.md 为准。

### CLAUDE.md 文件会暴露敏感信息吗？

CLAUDE.md 随代码提交到 Git 仓库，因此不应包含密钥、token 或凭证等敏感信息。环境变量和密钥应通过 `.env` 文件或密钥管理服务处理，CLAUDE.md 中只需说明需要哪些环境变量以及如何配置。

### 如果我不写 CLAUDE.md，Claude Code 还能正常工作吗？

能工作，但效果大打折扣。没有 CLAUDE.md，Claude Code 只能通过读取文件结构和代码来推断项目信息，无法知道你的构建命令、测试要求、已知陷阱等关键上下文。这就像给新同事安排任务但不告诉他项目背景——能做，但大概率需要反复修正。

### Claude Memory 的内容会被其他人看到吗？

不会。Memory 存储在你个人的 `~/.claude/` 目录下，不受 Git 管理，不会上传到任何共享空间。它完全是个人私有的，只有你自己的 Claude Code 会话可以访问。
