**CLAUDE.md excels at declarative rules.** It's structured like a configuration file: here are the build commands, here are the style rules, here are the gotchas. A well-written CLAUDE.md reads like an onboarding document for a new team member — everything they need to know to contribute correctly on day one. It handles constraints ("never import server-only modules in pipeline scripts"), workflow sequences ("run build, then test, then validate before committing"), and quality gates ("all vitest tests must pass").

**Memory excels at accumulated context.** It captures things that emerge over multiple conversations: the user is a data scientist focusing on observability, the user prefers terse responses without trailing summaries, the team has a merge freeze starting Thursday for a mobile release. These are context signals that inform *how* Claude Code works with you, not *what rules* it follows.

The structural difference matters too. CLAUDE.md is a single file (or a small hierarchy of files) that you edit directly. Memory is a directory of individual markdown files, each with typed frontmatter (`user`, `feedback`, `project`, `reference`), indexed by MEMORY.md. Claude Code manages the Memory files — creating, updating, and linking them — while you manage CLAUDE.md manually or through pull requests.

A practical test: if you delete the information, would another team member need it? If yes, it belongs in CLAUDE.md. If it's specific to your workflow, it belongs in Memory.

## Team Collaboration: Detailed Analysis

For teams, the distinction between these systems maps directly to **shared vs personal** context.

**CLAUDE.md is the team's single source of truth for AI behavior.** When a senior engineer adds a rule like "ZH content must use CJK word count, not English space-based tokenization," every developer's Claude Code session immediately follows that rule. This eliminates the problem of one teammate getting different AI behavior because they prompted Claude differently. Code review catches bad CLAUDE.md changes the same way it catches bad code.

**Memory creates individual expertise profiles.** A frontend developer's Memory might note they're experienced with React but new to the backend Go services. A DevOps engineer's Memory might store that they prefer infrastructure-as-code explanations in Terraform syntax. These personal contexts improve Claude Code's responses without cluttering the shared CLAUDE.md with individual preferences.

The team collaboration pattern that works best: CLAUDE.md handles **what** (rules, standards, commands), while Memory handles **who** (roles, preferences, feedback). A team of five developers shares one CLAUDE.md but maintains five separate Memory directories, each reflecting how that developer works with the codebase.

One limitation: Memory doesn't sync across machines. If you use Claude Code on both a laptop and a cloud dev environment, your Memory exists independently on each. CLAUDE.md, being git-tracked, stays consistent everywhere you clone the repo.

## Maintenance and Evolution: Detailed Analysis

Both systems require maintenance, but the maintenance models differ significantly.

**CLAUDE.md requires deliberate upkeep.** When you add a new build step, change a testing framework, or restructure your project, someone needs to update CLAUDE.md. Outdated CLAUDE.md instructions cause real problems — Claude Code might run a build command that no longer exists, or follow a coding convention that the team abandoned. The best teams treat CLAUDE.md updates as part of the PR process: if you change the build system, update CLAUDE.md in the same commit.

**Memory is self-maintaining but can accumulate stale entries.** Claude Code creates and updates memories automatically, but it doesn't always know when information has expired. A project memory about a merge freeze from three months ago is noise, not signal. Memory files include frontmatter and content that Claude Code uses to judge relevance, but periodic manual cleanup of the `memory/` directory keeps things sharp.

The staleness risk differs too. A stale CLAUDE.md instruction produces *wrong behavior* — Claude Code follows an outdated rule confidently. A stale Memory entry produces *slightly off context* — Claude Code might reference an old project detail but usually self-corrects when it reads the current code. This asymmetry means CLAUDE.md accuracy matters more and deserves more maintenance attention.

For teams using Claude Code's [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) with skills and hooks, CLAUDE.md also serves as the integration point — it can reference skill files, define hook behavior expectations, and document MCP server configurations. Memory doesn't play this role.

## When to Choose Claude Memory

Use Claude Memory as your primary context system when:

- **You're a solo developer** and don't need team-shared rules — Memory accumulates your project knowledge automatically without manual file editing
- **You want Claude Code to adapt to your style** — your preferred response length, commit message format, explanation depth, and workflow patterns
- **You're onboarding to a new codebase** — Memory captures your evolving understanding and the questions you've asked, building a personal knowledge base
- **You work across many small projects** — maintaining a CLAUDE.md for each is overhead; Memory builds up naturally as you work
- **You're tracking external systems** — reference memories store pointers to Linear projects, Slack channels, Grafana dashboards, and other tools that don't belong in a repo-level config file

Memory is also the right choice for information that changes frequently. Sprint goals, current incident status, and in-progress decisions are better as project memories than CLAUDE.md entries because Claude Code can update them conversationally rather than requiring a file edit and commit.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary context system when:

- **You're on a team** and need consistent AI behavior across all developers — CLAUDE.md ensures everyone's Claude Code follows the same rules
- **You have hard constraints** that must never be violated — build commands, test requirements, security rules, and deployment procedures belong in a deterministic, reviewable file
- **You're establishing project conventions** — coding style, architecture decisions, naming patterns, and forbidden anti-patterns are CLAUDE.md material
- **You want reviewable AI configuration** — CLAUDE.md changes go through pull requests, creating an audit trail of how your team's AI behavior evolves
- **You're setting up [agentic coding](/glossary/agentic-coding) workflows** — skill references, hook expectations, and pipeline documentation live in CLAUDE.md because they're part of the project's infrastructure

CLAUDE.md is also the right choice for anything that a new team member would need on day one. If the information is essential for correct contributions to the codebase, it belongs in a file that's guaranteed to be present when someone clones the repo. ## The Combined Approach: How to Use Both

The most effective Claude Code configuration uses both systems together, each handling what it does best.

**CLAUDE.md handles the project layer:**
- Build, test, and lint commands
- Coding style rules and forbidden patterns
- Architecture constraints and gotchas
- Workflow sequences (what to run before committing)
- Skill file and hook documentation

**Memory handles the personal layer:**
- Your role and expertise level

This separation mirrors how teams already work: the repo has a contributing guide (CLAUDE.md), and each developer has their own mental model of the project (Memory). Claude Code just makes both layers explicit and persistent.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions are treated as authoritative project rules that Claude Code must follow. Memory provides supplementary context — preferences, feedback, and project awareness — but cannot override explicit CLAUDE.md constraints. If CLAUDE.md says "run tests before every commit," a memory saying "skip tests sometimes" will be ignored in favor of the CLAUDE.md rule.

### Does Claude Memory sync across devices?

No. Memory is stored locally in `~/.claude/projects/<project>/memory/` and does not sync between machines. If you use Claude Code on multiple devices, each maintains its own Memory. CLAUDE.md, being part of the git repository, stays consistent everywhere you clone the project. For critical context you need on all devices, put it in CLAUDE.md.

### How do I migrate useful Memory entries into CLAUDE.md?

Review your `memory/` directory for feedback and project memories that apply to the whole team, not just you. If a memory says "integration tests must hit a real database, not mocks," that's a team rule — move it to CLAUDE.md. If it says "user prefers concise responses," that's personal — keep it in Memory. Run `ls ~/.claude/projects/<project>/memory/` to see all stored memories and evaluate each one.

### Should I check Memory files into git?

No. Memory files are designed to be personal and local. Checking them into git would mix individual preferences with team configuration, creating conflicts when different developers have different feedback memories. Keep CLAUDE.md in git for shared rules and Memory in its local directory for personal context.

### How much does CLAUDE.md affect Claude Code's behavior compared to Memory?

CLAUDE.md has a stronger behavioral impact because it loads as explicit instructions that Claude Code treats as mandatory. Memory influences tone, depth, and approach but doesn't override rules. A well-written CLAUDE.md with build commands, style rules, and constraints will shape 80% of Claude Code's project behavior. Memory fine-tunes the remaining 20% — how Claude Code communicates with you specifically.
