**Claude Memory uses a push model.** During a conversation, Claude Code identifies information worth retaining and writes it to a memory file. The `MEMORY.md` index loads automatically in future sessions, but individual memory files are accessed on demand — Claude Code reads them when it judges they're relevant to the current task. This makes memory ideal for context that accumulates and changes: your role shifted from backend to full-stack, a project deadline moved, you told Claude Code to stop summarizing its work at the end of every response.

The practical consequence: CLAUDE.md is authoritative and immediate (every session sees every rule), while Memory is advisory and selective (Claude Code decides which memories to surface). If something must always be followed, put it in CLAUDE.md. If something should inform Claude Code's judgment when relevant, let it live in memory.

## Scope and Sharing: Team vs Individual Context

The most important architectural difference between these two systems is who sees the content. This determines what belongs in each.

**CLAUDE.md is team infrastructure.** Because it's committed to the repository, every developer who clones the repo gets the same instructions. This is powerful for consistency — a new team member's Claude Code session follows the same build commands, the same quality gates, the same style rules as a senior engineer's session. Changes go through pull requests, so the team can review and discuss instruction updates. For organizations using [agentic coding](/glossary/agentic-coding) workflows at scale, CLAUDE.md is the primary mechanism for encoding engineering standards.

Consider what this means in practice. If your team decides that all API routes must include rate limiting, you add that rule to CLAUDE.md once, and every future Claude Code session across the team enforces it. No need to tell each developer, no need to hope they remember — the instruction is part of the repo.

**Claude Memory is personal context.** Memory files live outside the repository in `.claude/projects/`, which is never committed. Your memories are invisible to your teammates. This is appropriate because memory stores subjective information: your communication preferences, your role, your past corrections to Claude Code's behavior. A senior engineer who tells Claude Code "don't explain basic TypeScript concepts to me" shouldn't have that preference applied to a junior developer on the same team.

Memory also stores ephemeral project context that would clutter CLAUDE.md — "the auth rewrite is driven by legal compliance, not tech debt" or "merge freeze starts May 20 for mobile release cut." These facts matter for a few weeks, then become irrelevant. They don't belong in a version-controlled file that persists indefinitely.

There's a useful heuristic here: **if removing the information would break a build or violate a team rule, it belongs in CLAUDE.md. If removing it would just make Claude Code slightly less informed, it belongs in memory.**

## Content Types: What Goes Where

Knowing which system to use comes down to the nature of the information. Here's a detailed breakdown with real examples from the [Claude Code memory system](/blog/claude-code-memory).

### CLAUDE.md Content

**Build and validation commands.** The most critical CLAUDE.md content is the commands Claude Code must run. `npm run build`, `npm test`, `npm run lint` — these are non-negotiable quality gates. Claude Code treats them as hard requirements, not suggestions.

**Architectural constraints.** Rules like "never import Next.js modules in pipeline scripts" or "always use the `upsertKeyword()` function with all three parameters" prevent classes of bugs. These constraints are derived from past incidents and encode team knowledge that would otherwise live only in someone's head.

**Code style and conventions.** Commit message formats, naming conventions, file organization rules. CLAUDE.md is the right place because these must be consistent across all contributors.

**Workflow rules.** "New feature → discuss design first, get human approval before coding" or "pipeline changes → run validate-pipeline.ts before commit." These define how Claude Code should interact with the developer, not what it should build.

### Claude Memory Content

**User profile.** "Senior backend engineer, new to React." "Data scientist focused on observability." Claude Code uses this to calibrate its explanations — a senior engineer gets terse technical answers, a newcomer gets more context.

**Behavioral feedback.** "Don't summarize what you just did at the end of every response." "Use single bundled PRs for refactors in this area." These are corrections and confirmations from past sessions that shape future behavior.

**Project context.** "The auth middleware rewrite is driven by legal/compliance, not tech debt." "Merge freeze begins May 20 for mobile release." These facts inform Claude Code's suggestions without being hard rules.

**External references.** "Pipeline bugs are tracked in Linear project INGEST." "The oncall latency dashboard is at grafana.internal/d/api-latency." These help Claude Code point you to the right resources.

### Gray Areas

Some information could reasonably go in either system. The deciding factors are: does the team need it (CLAUDE.md) or just you (Memory)? Will it change frequently (Memory) or rarely (CLAUDE.md)? Must it be enforced every session (CLAUDE.md) or just inform judgment (Memory)?

For example, "we're using Tailwind v4" is a CLAUDE.md fact — everyone needs it, it rarely changes. But "we chose Tailwind v4 because the design team prefers utility-first CSS and we needed v4's dynamic theming for the rebrand" is Memory — it's context that helps Claude Code make better suggestions, but it's not a rule to enforce.

## Lifecycle and Maintenance: How Each System Evolves

**CLAUDE.md evolves through code review.** When you add a new pipeline script, you update CLAUDE.md to document it. When a convention changes, you modify the relevant rule. Because it's version-controlled, you have a full history of what changed and why. The file tends to grow over time as the project accumulates conventions, but should be pruned periodically — rules for deleted features, outdated build commands, and deprecated workflows should be removed.

**Claude Memory evolves through conversation.** Claude Code creates, updates, and deletes memories as it learns. A memory saved in March might be updated in April when the project context shifts, or deleted in May when the initiative it describes is complete. The `MEMORY.md` index has a practical limit — lines past 200 are truncated — so stale memories should be cleaned up regularly.

The maintenance burden differs accordingly. CLAUDE.md requires deliberate upkeep — if you change a build command and don't update CLAUDE.md, Claude Code will run the wrong command. Memory is self-maintaining to a degree — Claude Code can detect when a memory conflicts with current code and update it — but it's not infallible. Periodic manual review of both systems keeps them accurate.

For teams that want to understand how CLAUDE.md fits into the broader [Claude Code extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp), it's one of several programmable layers — alongside skills, hooks, agents, and MCP servers — that turn a CLI tool into a customized development platform.

## Practical Setup: Getting the Most from Both Systems

### Start with CLAUDE.md

Every project should have a CLAUDE.md before using memory. Start with the essentials:

```markdown
# CLAUDE.md

## Commands
npm run dev          # Local dev server
npm run build        # Production build
npm test             # Tests (must pass before commit)
npm run lint         # Linting

## Rules
- Run build + test before every commit
- Never skip failing tests
```

This alone dramatically improves Claude Code's behavior. It stops guessing at your build commands, stops committing without testing, and follows your project's conventions. The [complete guide to Claude Code](/blog/claude-code-complete-guide) covers the full setup process.

### Let Memory Accumulate Naturally

Don't try to pre-populate memory. Instead, correct Claude Code during normal work:

- "I'm a frontend engineer, not a backend engineer — adjust your suggestions accordingly"
- "Stop adding JSDoc comments to every function, we don't do that here"
- "The deploy pipeline is documented in Notion at [URL]"

Claude Code saves these as typed memories — user, feedback, reference — and recalls them in future sessions. Over a few weeks, you build up a personal context layer that makes Claude Code increasingly effective for your specific workflow.

### Use Skills for Reusable Prompts

Neither CLAUDE.md nor Memory is the right place for detailed, multi-paragraph instructions for specific tasks. That's what [SKILL.md files](/blog/5-claude-code-skills-i-use-every-single-day) are for — reusable prompt templates stored in `skills/*/SKILL.md` that Claude Code loads when invoked. Skills complement both persistence layers: CLAUDE.md says "what rules to follow," Memory says "what context to remember," and skills say "how to approach this specific task."

## Common Mistakes

**Putting personal preferences in CLAUDE.md.** "I prefer terse responses" doesn't belong in a team-shared file. Put it in memory so it applies only to your sessions.

**Putting build commands in memory.** "npm run build" must be in CLAUDE.md where it's loaded authoritatively every session. Memory is advisory — Claude Code might not surface a build command memory at the right moment.

**Duplicating information across both systems.** If CLAUDE.md says "use Vitest for testing" and a memory also says "the project uses Vitest," you've created a maintenance burden with no benefit. Let CLAUDE.md be the source of truth for project facts; use memory only for information that can't go in CLAUDE.md.

**Never pruning either system.** CLAUDE.md instructions for deleted features confuse Claude Code. Stale memories about completed initiatives waste context window space. Review both periodically.

**Treating CLAUDE.md as documentation.** CLAUDE.md is instructions for Claude Code, not a README for humans. Keep it focused on actionable rules: commands to run, constraints to follow, patterns to use. Background context and explanations belong in actual documentation or in memory.

## When to Choose Claude Memory

Choose Memory when the information is:

- **Personal**: your role, your preferences, your communication style
- **Evolving**: project decisions that change week to week
- **Contextual**: background that informs judgment rather than enforcing rules
- **External**: pointers to tools, dashboards, and resources outside the repo
- **Temporary**: deadlines, freezes, and initiatives with expiration dates

Memory is the right tool when you want Claude Code to be smarter about your specific situation without imposing your context on the rest of the team.

## When to Choose CLAUDE.md

Choose CLAUDE.md when the information is:

- **Team-wide**: rules that apply to every developer on the project
- **Deterministic**: commands, constraints, and conventions that must be followed every time
- **Stable**: facts about the project that change infrequently
- **Enforceable**: quality gates that should never be skipped
- **Reviewable**: instructions that benefit from team discussion and version history

CLAUDE.md is the right tool when you want Claude Code to behave consistently across all team members and all sessions, with changes tracked through normal code review. For more on building effective project instructions, see [what makes Claude Code special](/blog/whats-so-special-about-the-claude-code).

## Verdict

**Use both — they solve different problems.** CLAUDE.md is your project's constitution: explicit, shared, version-controlled rules that every Claude Code session must follow. Claude Memory is your personal context layer: evolving, private, and advisory information that makes Claude Code more effective for you specifically.

If you're just getting started, **begin with CLAUDE.md** — it delivers the highest immediate impact by ensuring Claude Code runs the right commands, follows your conventions, and respects your quality gates. Let memory build up naturally through conversation over the following weeks. The combination of explicit rules (CLAUDE.md) and accumulated context (Memory) is what makes Claude Code's persistence system more powerful than either layer alone.

For teams scaling [agentic coding](/glossary/agentic-coding) across multiple developers, CLAUDE.md is non-negotiable — it's the foundation of consistent AI behavior. Memory adds a personalization layer on top that makes each developer's experience better without fragmenting the team's standards.

## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md instructions are loaded authoritatively at the start of every session and treated as hard rules. Memory is advisory — Claude Code uses it to inform decisions, but CLAUDE.md takes precedence when they conflict. If you want to override a project rule for your personal sessions, you'd need to use the user-level `~/.claude/CLAUDE.md` file, not memory.

### Does Claude Memory persist if I switch machines?

No. Memory files are stored locally in `.claude/projects/` on your machine. They are not synced across devices or backed up to any cloud service. If you switch machines, you start with a blank memory slate. CLAUDE.md, being committed to the repo, is available on any machine that clones the repository.

### How many memories can Claude Code store?

There's no hard limit on the number of memory files, but the `MEMORY.md` index is truncated at 200 lines in the conversation context. This means Claude Code may not be aware of memories beyond that threshold unless it searches for them. In practice, keeping 30-50 well-organized memories is more effective than accumulating hundreds of granular entries.

### Should I commit the .claude/memory/ directory?

No. The `.claude/` directory is meant to be local and personal. Committing memory files would expose individual preferences and potentially sensitive project context to the entire team. Keep `.claude/` in your `.gitignore` and use CLAUDE.md for anything that should be shared.

### Can I manually create or edit memory files?

Yes. Memory files are plain markdown with YAML frontmatter. You can create, edit, or delete them directly with any text editor. Just make sure to update the `MEMORY.md` index file to reflect your changes, and follow the expected frontmatter schema (name, description, metadata.type) so Claude Code can categorize and retrieve them correctly.
