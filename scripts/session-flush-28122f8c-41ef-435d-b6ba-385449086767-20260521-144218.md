**Codex custom agents** are configured through a combination of the Codex web dashboard and repository-level `codex.md` files. In the dashboard, you define an agent profile with a name, description, environment setup script (which runs before each task), and behavioral instructions. The setup script can install dependencies, configure tools, and prepare the sandbox environment. Instructions follow the same prompt-engineering patterns as system prompts — you tell the agent what it specializes in, what conventions to follow, and what constraints to respect.

The `codex.md` file in your repository root acts as project-level context, similar to Claude Code's `CLAUDE.md`. When a Codex agent picks up a task from your repo, it reads `codex.md` for project-specific guidance. However, `codex.md` applies to all agents working on that repo — you cannot define per-agent instructions at the repository level without using the dashboard.

**Claude Code subagents** take a different approach: agent specialization is defined at spawn time through the prompt parameter of the `Agent` tool. The parent agent writes a detailed brief — what the subagent should do, what context it needs, and what form the response should take. This means every subagent invocation can be uniquely tailored to the specific task at hand.

For reusable specialization, Claude Code uses [SKILL.md files](/blog/9-principles-writing-claude-code-skills-that-actually-work) stored in the repository's `skills/` directory. These files encode domain-specific instructions — how to write tests, generate content, review security — that persist across sessions and team members. When spawning a subagent, the parent can reference a skill to provide consistent behavioral instructions. The `subagent_type` parameter further specializes agents: `Explore` for read-only codebase search, `Plan` for architecture design, `pipeline-reviewer` for pipeline script validation.

**The practical difference**: Codex agents are pre-built profiles you select from a menu; Claude Code subagents are dynamically composed from prompts, skills, and type parameters. Codex's approach is simpler for teams with well-defined workflows. Claude Code's approach is more powerful for complex, context-dependent tasks where the right agent configuration depends on what the parent agent has already discovered.

If your team has 5-10 recurring task types and wants non-technical team members to submit work, Codex custom agents are more accessible. If your agents need to adapt their behavior based on runtime context — what files exist, what tests failed, what the codebase looks like today — Claude Code subagents offer more flexibility.

## Parallel Execution: Detailed Analysis

Multi-agent value comes from parallelism — doing multiple things at once instead of sequentially. Both platforms support parallel work, but the coordination models differ.

**Codex** treats each task as an independent unit of work. You submit multiple tasks to different custom agents, and they execute in separate cloud sandboxes simultaneously. Each agent produces its own output — typically a PR branch with changes and a summary. There is no built-in mechanism for agents to coordinate with each other during execution. If Agent A's work depends on Agent B's output, you need to orchestrate that dependency manually by waiting for one task to complete before submitting the next.

This model works well for embarrassingly parallel work: "write tests for module A" and "update docs for module B" and "refactor module C" can all run independently. It breaks down when tasks have data dependencies or when the right approach for one task depends on discoveries made during another.

**Claude Code** enables parallel subagent execution within a single session. The parent agent spawns multiple subagents in a single message, and they run concurrently. When all subagents complete, the parent agent receives their results and synthesizes them — resolving conflicts, combining findings, and making decisions that account for all agents' discoveries.

This coordination layer is the key differentiator. In Claude Code, a parent agent can spawn three subagents to investigate a bug from different angles, compare their findings, and then take a single coherent action. The parent maintains full context about what each subagent was asked and what it found. Claude Code also supports [agent teams](/blog/claude-code-agent-teams) — a pattern where specialized subagents handle different aspects of a large task while the parent orchestrates the overall workflow.

For real-world multi-agent coding scenarios, see our [Claude Code subagents examples](/blog/claude-code-subagents-examples) for patterns that work in production.

**Codex's async model** means you can submit work and walk away — check back in 10 minutes, review the PR, merge or request changes. This is valuable for workflows where a developer wants to parallelize their own time, not just the agent's compute.

**Claude Code's sync model** means results are available immediately within the session. This is valuable for iterative work where the next step depends on what the agents found. The tradeoff: you are present during execution, and the session's context window is shared across all subagents.

## Environment and Security

**Codex custom agents** run in isolated cloud containers. Each agent gets a fresh environment spun up from your repository's default branch (or a specified branch). The setup script installs dependencies, and the agent works within that sandbox. It cannot access your local filesystem, your credentials, or your running services. Output is delivered as a git branch with changes and a summary.

This isolation model has clear security benefits: a misbehaving agent cannot access sensitive local files, environment variables, or running processes. It also means Codex agents cannot interact with local development servers, databases, or tools that are not committed to the repository.

**Claude Code subagents** share the parent agent's local environment. They can read and write the same files, run the same shell commands, and access the same tools. Claude Code's permission system requires user approval for potentially dangerous operations, but subagents operate within the same security boundary as the parent session.

For teams with strict security requirements — especially enterprises handling sensitive data — Codex's container isolation provides stronger guarantees. For individual developers who want their agents to interact with their full local development environment, Claude Code's shared environment is more practical.

## Workflow Integration

How agents fit into your existing development workflow determines whether you will actually use them daily.

**Codex** integrates through GitHub. Each completed task produces a PR branch. You review the changes in GitHub's interface, request modifications by commenting on the PR, and merge when satisfied. This fits naturally into teams already using PR-based workflows. Codex also integrates with GitHub Issues — you can assign an issue to a Codex agent and it will attempt to resolve it.

**Claude Code** integrates through the terminal. Subagent results feed directly into the parent session, which can then stage, commit, and push changes. For PR workflows, Claude Code can create PRs using the `gh` CLI, but the primary interaction model is direct terminal work rather than async PR submission. Claude Code's [hooks system](/blog/claude-code-hooks-a-complete-guide-to-automating-your-ai-coding-workflow) allows automated pre- and post-actions around agent work — running linters after edits, validating schemas before commits.

If your team's workflow centers on GitHub PRs and async code review, Codex custom agents slot in more naturally. If your workflow is terminal-centric with rapid iteration cycles, Claude Code subagents integrate more tightly.

## When to Choose Codex Custom Agents

Codex custom agents are the right choice when:

- **Your team needs async task delegation.** Submit a task, go do other work, come back to a PR. Non-technical team members or managers can submit tasks through the web interface without terminal access.
- **You want strong environment isolation.** Each agent runs in its own container. No risk of agents accessing local secrets, modifying local state, or interfering with each other.
- **You have well-defined, repeatable task types.** "Write tests for this module," "update the changelog," "fix this linted error" — tasks with clear inputs and outputs that do not require mid-execution judgment calls.
- **Your team is on GitHub-centric workflows.** Codex's PR-based output model fits naturally into review, CI, and merge pipelines.

Codex custom agents are less suitable for exploratory work, tasks requiring local environment access, or workflows where the right approach depends on interactive discovery.

## When to Choose Claude Code Subagents

Claude Code subagents are the right choice when:

- **You need real-time coordination between agents.** A parent agent spawns subagents, receives their findings, and makes decisions that account for all results — critical for bug investigation, codebase-wide refactoring, and architecture exploration.
- **Your tasks require local environment access.** Building and testing locally, interacting with running services, accessing local databases or APIs, or using tools not in the repository.
- **You want dynamic agent specialization.** Instead of pre-defined profiles, compose the right agent for each task on the fly based on what the parent agent has already discovered.
- **You work in the terminal.** Claude Code subagents integrate into your existing terminal workflow without context-switching to a web interface. See our [complete Claude Code guide](/blog/claude-code-complete-guide) for the full capability set.

Claude Code subagents are less suitable for fire-and-forget delegation, teams that need non-technical members to submit tasks, or workflows requiring strict sandboxing between agents.

## Verdict

**For async, team-oriented task delegation with strong isolation, choose Codex custom agents.** They excel when you have well-defined task types, want PR-based output, and need non-technical team members to submit work. The cloud sandbox model provides security guarantees that matter for enterprise environments.

**For interactive, developer-driven multi-agent workflows with real-time coordination, choose Claude Code subagents.** They excel when tasks have dependencies, require local environment access, or need dynamic specialization based on runtime context. The parent-subagent coordination model handles complex engineering work that async agents cannot.

Many teams will benefit from using both: Codex custom agents for routine, well-defined tasks that can run in the background, and Claude Code subagents for complex, interactive work that requires developer oversight and cross-agent synthesis. The platforms are not mutually exclusive — they optimize for different parts of the development workflow. ## Frequently Asked Questions

### Can Codex custom agents communicate with each other during execution?

No. Each Codex custom agent runs in an isolated sandbox with no inter-agent communication channel. If tasks have dependencies, you must orchestrate them sequentially — wait for one agent's PR to complete before submitting the dependent task. Claude Code subagents, by contrast, report back to a parent agent that coordinates across all results within a single session.

### How many Claude Code subagents can run in parallel?

Claude Code does not impose a hard limit on parallel subagents, but practical limits exist. Each subagent consumes tokens from the session's context window, and spawning too many simultaneously can degrade response quality as the parent agent juggles multiple result sets. In practice, 3-5 parallel subagents per spawn is the effective ceiling for complex tasks.

### Do Codex custom agents support local tool access?

No. Codex agents run in cloud containers and can only access tools installed during the environment setup script. They cannot reach your local filesystem, running services, or credentials. If your workflow requires local tool access — interacting with a development database, running a local build pipeline, or testing against a local server — Claude Code subagents operating in your local environment are the better fit.

### Which platform is cheaper for multi-agent workloads?

It depends on task volume and complexity. Codex charges compute credits per task execution on Pro ($200/month) and Team plans, with credits consumed based on runtime duration and model usage. Claude Code bills per token across the entire session, including all subagent work. For many small, quick tasks, Codex's per-task model can be more predictable. For fewer, larger tasks with complex coordination, Claude Code's per-token model may be more cost-effective. Both platforms' pricing is subject to change — verify current rates on their respective pricing pages.

### Can I use SKILL.md files with Codex custom agents?

Not directly. SKILL.md is a Claude Code convention — Codex uses `codex.md` for repository-level instructions and dashboard-configured agent profiles for agent-specific behavior. However, the underlying concept is similar: both platforms support defining reusable instructions that shape how agents approach tasks. If migrating between platforms, you will need to translate SKILL.md content into Codex's instruction format.
