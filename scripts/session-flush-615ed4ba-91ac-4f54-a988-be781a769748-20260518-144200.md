**Claude Code uses [SKILL.md files](/blog/claude-code-extension-stack-skills-hooks-agents-mcp)** — markdown documents stored in your repository under a `skills/` directory that define how the agent approaches specific task types. A skill file might encode your team's testing conventions, your PR review checklist, or your content generation standards. When a subagent is spawned with a specific skill, it follows those instructions as project-level constraints.

The power of this system lies in its **composability with version control**. Skills live in your repo alongside your code. They go through code review. They evolve with your codebase. When a senior engineer encodes a testing standard into a SKILL.md, every team member's Claude Code sessions — and every subagent those sessions spawn — follow the same standard automatically. According to analysis in [how skills improve agent output](/blog/do-skills-actually-improve-your-agents-output), skill files measurably increase consistency across agent-generated code.

Claude Code subagents also inherit context from `CLAUDE.md` — the project-level configuration file that describes architecture decisions, coding conventions, and workflow constraints. This means subagents understand the broader project context even when focused on a narrow task. A subagent assigned to refactor a database module knows about the project's migration strategy, ORM conventions, and test requirements because those are documented in `CLAUDE.md`.

For practical examples of how teams structure these multi-agent workflows, our [Claude Code subagent examples](/blog/claude-code-subagents-examples) guide walks through real configurations including parallel test generation, codebase-wide refactoring, and coordinated feature development.

**Codex custom agents use configuration objects** defined through the Codex interface or API. These configurations specify the agent's system instructions, environment setup commands, verification steps, and behavioral constraints. For example, a "migration agent" configuration might include instructions for how to approach database schema changes, a setup command to install migration tools, and a verification step to run the migration dry-run.

The trade-off: Codex configurations are more portable across repositories (they're not repo-specific files) but less integrated with your development workflow. They live in the Codex platform, not in your codebase. This means they don't go through code review, don't version with your code, and can drift from your actual project conventions without anyone noticing.

Where Codex's approach shines is **team-wide standardization without repo changes**. An engineering manager can define a set of custom agents — code review, test writing, documentation — and make them available to the entire team through the Codex dashboard. Team members don't need to set up skill files or understand the configuration; they just pick an agent type and assign a task. For organizations onboarding developers who aren't comfortable with terminal-based workflows, this is a significant accessibility advantage.

## Real-Time Control vs Asynchronous Delegation

This is the fundamental workflow difference, and it determines which tool fits which engineering culture.

**Claude Code subagents operate in a supervised, interactive loop.** The parent agent proposes a plan, you approve it, subagents execute, and you review results before the parent synthesizes. At any point, you can intervene — rejecting a subagent's approach, providing additional context, or redirecting the overall strategy. This maps to **pair programming culture**: you're actively working with the AI, maintaining awareness of every decision.

The interactive model excels when:
- The task has hidden dependencies that only become visible during implementation
- You need to make judgment calls about trade-offs (performance vs. readability, backward compatibility vs. clean design)
- The codebase has undocumented conventions that the agent might miss
- You want to learn from the agent's approach, not just consume its output

**Codex custom agents operate in a delegation loop.** You define the task clearly, assign it, and review the output. The agent works independently, without the ability to ask you clarifying questions mid-task. This maps to **ticket-based engineering culture**: you write a clear spec, hand it off, and review the deliverable.

The asynchronous model excels when:
- Tasks are well-defined with clear acceptance criteria
- Multiple independent tasks need to run simultaneously
- The team lead wants to distribute work without being available for real-time feedback
- Tasks are routine enough that clarifying questions shouldn't be necessary

For teams that mix both cultures — interactive for complex features, async for routine maintenance — the choice isn't either/or. Many engineering teams use Claude Code subagents for complex, high-context work (architecture refactors, feature development, debugging) and Codex custom agents for batch operations (dependency updates, boilerplate generation, documentation).

## Isolation and Safety

Both platforms take isolation seriously, but their approaches reflect their architectural differences.

**Claude Code subagents** use git worktrees for code isolation and the Claude Code permission system for action isolation. Subagents can be spawned in different permission modes — from requiring approval for every action to running autonomously within defined boundaries. The worktree model means subagents can't accidentally overwrite each other's changes, but they do share the same local machine resources (filesystem, network, environment variables).

The [Agent SDK](/glossary/agent-sdk) provides additional isolation controls for teams building custom agent workflows on top of Claude Code. You can define which tools each subagent type has access to, restrict file system access to specific directories, and require human approval for specific action categories.

**Codex custom agents** use full container isolation — each agent runs in its own sandbox with no access to the host filesystem, no network access by default, and no ability to affect other running agents. This is stronger isolation than worktrees provide. A malicious or confused agent can't exfiltrate code, install backdoors, or interfere with other processes.

The security trade-off: Codex's stronger isolation comes at the cost of capability. No network access means agents can't run integration tests that hit external services, can't install packages not already in the repo, and can't access internal APIs. Claude Code's lighter isolation gives agents more capability — full shell access, network access, tool integration via MCP — but requires trust in the permission system and the user's oversight.

For security-sensitive codebases, Codex's sandbox model provides stronger guarantees by default. For codebases that require full environment access (microservices with external dependencies, projects with complex build toolchains), Claude Code's local execution model is more practical.

## When to Choose Claude Code Subagents

**Choose Claude Code subagents when your work requires deep project context and real-time coordination.**

The ideal use cases for Claude Code subagents include:

- **Complex refactoring** across multiple modules where changes in one area affect others — the parent agent coordinates dependencies between subagents in real time
- **Feature development** that spans multiple files and requires architectural decisions — you steer the high-level approach while subagents handle implementation
- **Debugging sessions** where the root cause isn't clear — spawn explorer subagents to investigate different hypotheses in parallel, then focus on the most promising lead
- **Codebase-wide migrations** where each file needs context-aware changes (not just find-and-replace) — subagents handle individual files while the parent ensures consistency

Claude Code subagents are also the better choice when your project relies heavily on **local tooling** — custom build systems, local databases, Docker environments, MCP servers — that wouldn't be available in a cloud sandbox.

The developer profile for Claude Code subagents: senior engineers comfortable in the terminal who want to maintain oversight of AI-generated changes while delegating execution. Teams that already use Claude Code's [skill system](/blog/9-principles-writing-claude-code-skills-that-actually-work) and CLAUDE.md conventions get the most value, since subagents inherit all that context automatically.

## When to Choose Codex Custom Agents

**Choose Codex custom agents when your workflow benefits from asynchronous task delegation and team-wide access.**

The ideal use cases for Codex custom agents include:

- **Batch task processing** where you have 10-50 well-defined tasks (dependency updates, API migration, test coverage gaps) that can run independently
- **Team-wide delegation** where engineering leads assign tasks to agents from a dashboard without requiring each developer to run local sessions
- **Routine maintenance** where the task is well-defined enough that real-time oversight isn't needed — fix this lint error, add this missing type, update this deprecated API call
- **Onboarding and accessibility** where team members who aren't comfortable with terminal workflows can still leverage AI coding agents through a GUI

Codex custom agents also make sense when **security isolation is paramount** — the sandboxed execution model provides stronger guarantees than local execution for codebases handling sensitive data.

The developer profile for Codex custom agents: teams that operate with ticket-based workflows, engineering managers who want to scale AI-assisted development across the organization, and developers who prefer GUI interfaces over terminal-based tools. The [Codex VS Code extension](/blog/codex-vscode) bridges the gap for developers who want IDE integration without full terminal workflows.

## Cost Considerations

Pricing models differ significantly and can determine the right choice for budget-conscious teams.

**Claude Code subagents** use Anthropic's token-based API billing. Each subagent consumes tokens independently — a parent agent spawning five subagents for a refactoring task might use 3-5x the tokens of a single-agent session, depending on task complexity and context size. There's no per-seat or subscription cost; you pay for what you use. This makes Claude Code subagents cost-effective for teams with variable workloads but potentially expensive for sustained, high-volume usage.

**Codex custom agents** are included in ChatGPT Pro ($200/month) and Team plans. This flat-rate model is more predictable — you know the monthly cost regardless of usage volume. For teams running many agents daily, the subscription model can be significantly cheaper than per-token billing. However, the higher base cost means occasional users pay more than they would with usage-based pricing.

As of mid-2026, pricing structures for both platforms continue to evolve. Check current pricing on each vendor's official pages before making procurement decisions.

## Verdict

The choice between Claude Code subagents and Codex custom agents comes down to your workflow model: **interactive orchestration vs. asynchronous delegation**.

**Choose Claude Code subagents** if you're a developer or small team doing complex, high-context engineering work where real-time control and deep project understanding matter. The local execution model, skill-based customization, and parent-child coordination enable workflows that cloud sandboxes can't replicate — particularly for [agentic coding](/glossary/agentic-coding) tasks that require iterative decision-making.

**Choose Codex custom agents** if you're managing engineering work at team scale, need strong isolation guarantees, or prefer a GUI-driven delegation model. The cloud sandbox architecture and pre-configured agent types make it easier to distribute AI-assisted development across an organization without requiring every team member to master terminal workflows.

For teams with the budget and flexibility, **using both is the pragmatic answer**. Claude Code subagents for the hard problems that need a human in the loop; Codex custom agents for the routine tasks that just need to get done. The tools aren't competing for the same workflow — they're solving different coordination problems in software engineering.

## Frequently Asked Questions

### Can Claude Code subagents and Codex agents work on the same codebase?

Yes. Claude Code subagents run locally on your machine and push changes through git, while Codex agents work in cloud sandboxes and produce pull requests. Both integrate with standard git workflows, so a team can use Claude Code for complex local development and Codex for batch tasks on the same repository without conflicts.

### How many subagents can Claude Code run in parallel?

Claude Code can spawn multiple subagents simultaneously, limited primarily by your local machine's resources and API rate limits. In practice, teams commonly run 3-7 subagents in parallel for large refactoring tasks. Each subagent operates in its own git worktree, so parallelism doesn't create file conflicts.

### Do Codex custom agents support MCP servers or external tool integrations?

Codex agents run in sandboxed containers without network access by default, which means they cannot connect to external MCP servers or APIs during execution. This is a deliberate security constraint. Claude Code subagents, running locally, can connect to any MCP server or external tool configured in the project.

### Which platform is better for open-source contributors?

Codex offers dedicated free access for [open-source maintainers](/blog/codex-for-open-source), making it accessible for community projects. Claude Code subagents require API credits, though Anthropic periodically offers credits for open-source work. For well-defined contribution tasks (issue triage, documentation, test coverage), Codex's async model fits the open-source workflow of reviewing external contributions.

### Is there a learning curve difference between the two approaches?

Claude Code subagents require familiarity with terminal workflows, git worktrees, and the CLAUDE.md/SKILL.md configuration system. Codex custom agents have a lower barrier to entry — the web dashboard and VS Code extension provide GUI-based task assignment. However, getting the most from either platform's customization features (skills vs. agent configurations) requires similar investment in prompt engineering and workflow design.
