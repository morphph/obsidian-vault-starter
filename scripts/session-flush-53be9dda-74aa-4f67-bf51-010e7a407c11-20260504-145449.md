# Codex Subagents vs Custom Agents：哪种多智能体方案更适合你？

**TL;DR：** **Subagents** 是 Codex 内置的轻量任务分发机制，适合将大任务自动拆解为并行子任务；**Custom Agents** 是用户自定义的专用智能体，通过预设指令和工具配置处理特定领域工作。如果你需要快速并行处理多文件重构，选 Subagents；如果你需要针对特定代码规范或业务逻辑定制 Agent 行为，选 Custom Agents。两者可以组合使用。

## 概述：Codex Subagents

Codex Subagents 是 [OpenAI Codex](/zh/blog/codex-complete-guide) 平台内置的子智能体机制，允许主 Agent 在执行复杂任务时自动生成多个子 Agent 并行工作。这种机制的核心设计理念是"分而治之"——当一个编程任务涉及多个文件、多个模块或多个独立步骤时，主 Agent 可以将任务拆解并分配给不同的 Subagent，每个 Subagent 在独立的沙箱环境中运行。

Subagents 的关键优势在于自动化程度高。开发者不需要手动配置每个子 Agent 的行为——主 Agent 根据任务语义自动决定如何拆分工作、分配上下文、合并结果。这使得处理大规模代码库操作时效率显著提升，尤其是测试生成、批量重构和跨模块依赖更新等场景。

从架构上看，每个 Subagent 运行在独立的容器化环境中，拥有完整的文件系统访问权限和工具调用能力，但彼此隔离。这种设计保证了并行执行的安全性，也避免了子任务之间的状态污染。

## 概述：Custom Agents

Custom Agents 是一种用户定义的专用智能体配置方案，允许开发者通过预设系统指令、工具集合和行为约束来创建针对特定任务优化的 Agent。与 Subagents 的"自动拆解"不同，Custom Agents 强调的是"精确控制"。

Custom Agents 的核心价值在于可复用性和一致性。通过定义 Agent 的角色、可用工具、代码规范和输出格式，团队可以确保不同成员在使用 Codex 时获得一致的代码质量和风格。这与 [Claude Code 的 Skills 系统](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)理念相似——将工程标准编码为可执行的指令，而非依赖口头约定。

典型的 Custom Agent 配置包括：系统提示词（定义 Agent 的角色和行为边界）、工具白名单（限制 Agent 可调用的 API 和命令）、输出模板（规范生成代码的格式和注释风格），以及上下文注入规则（指定 Agent 启动时自动加载的文件和文档）。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **启动方式** | 主 Agent 自动生成 | 用户预定义配置 | 取决于场景 |
| **配置复杂度** | 零配置，开箱即用 | 需要编写 Agent 定义文件 | Subagents |
| **任务粒度** | 自动拆分，粒度由模型决定 | 用户精确控制任务边界 | Custom Agents |
| **行为一致性** | 每次拆分策略可能不同 | 严格遵循预设指令 | Custom Agents |
| **并行能力** | 原生支持多 Agent 并行 | 需要额外编排层 | Subagents |
| **工具控制** | 继承主 Agent 全部工具 | 可按需限制工具集 | Custom Agents |
| **可复用性** | 一次性，随任务消亡 | 跨项目、跨团队复用 | Custom Agents |
| **调试难度** | 较高，子任务链路长 | 较低，行为可预测 | Custom Agents |

## 并行任务处理：Subagents 的主场

Subagents 最大的技术优势是原生并行执行能力。当你向 Codex 提交一个涉及多文件修改的任务时——比如"为 src/services/ 下所有模块添加单元测试"——主 Agent 会自动分析模块依赖关系，将独立模块的测试生成任务分配给不同的 Subagent。

这种并行机制在大规模重构场景中效果尤为明显。以一个典型的 API 迁移任务为例：将项目从 REST 迁移到 GraphQL，涉及数十个端点的改写。Subagents 可以同时处理多个独立端点的转换，每个 Subagent 负责一个端点的 schema 定义、resolver 编写和测试更新，最终由主 Agent 合并所有变更并验证整体一致性。

值得注意的是，Subagents 的拆分质量高度依赖主 Agent 对任务的理解。对于依赖关系复杂、执行顺序敏感的任务，自动拆分可能导致冲突或遗漏。这种情况下，[Claude Code 的 Agent Teams 方案](/zh/blog/claude-code-agent-teams)提供了另一种思路——通过更精细的 Agent 角色定义来控制协作流程。

## 定制化与可控性：Custom Agents 的主场

Custom Agents 的核心竞争力在于行为可预测性。在企业级开发中，"AI 写的代码质量不一致"是最常见的痛点之一。Custom Agents 通过将代码规范、架构约束和业务逻辑编码为 Agent 配置，从根本上解决了这个问题。

一个实际的应用场景：团队定义一个"数据库迁移 Agent"，其配置包含以下约束——只使用经过审批的 ORM 方法、所有迁移必须包含回滚脚本、字段命名遵循团队的 snake_case 规范、禁止使用 raw SQL。这样，无论哪位开发者使用这个 Agent，生成的迁移脚本都符合团队标准。

Custom Agents 的另一个关键优势是安全边界控制。通过工具白名单机制，你可以限制 Agent 只能读取特定目录、只能执行特定命令，避免意外修改敏感文件或执行危险操作。这在处理生产环境配置、密钥管理等敏感场景时尤为重要。

与之对比，[Codex 在 VS Code 中的集成](/zh/blog/codex-vscode)也提供了一定程度的行为定制能力，但 Custom Agents 的配置粒度更细、复用性更强。

## 适合选择 Subagents 的场景

选择 Subagents 的判断标准很简单：**任务可并行、子任务之间低耦合、不需要精确控制每个子任务的行为**。

具体场景包括：

- **批量测试生成**：为多个独立模块同时生成单元测试，每个模块的测试逻辑互不依赖
- **大规模格式化和代码风格统一**：对数十个文件执行相同的重构规则（如统一错误处理模式）
- **文档批量生成**：为多个 API 端点同时生成文档，每个端点的文档独立成文
- **依赖更新**：升级一个底层库后，自动修复所有受影响模块的 breaking changes

如果你的日常工作涉及大量重复性的跨文件操作，Subagents 能够将线性执行时间压缩为并行时间，效率提升与子任务数量成正比。关于更多 Subagent 的实际用例，可以参考我们整理的 [Claude Code Subagents 实战案例](/zh/blog/claude-code-subagents-examples)——虽然是 Claude Code 生态的案例，但多智能体编排的核心思路是相通的。

## 适合选择 Custom Agents 的场景

选择 Custom Agents 的判断标准同样明确：**任务需要领域知识、输出需要高度一致、或安全约束要求精确控制 Agent 行为**。

具体场景包括：

- **代码审查 Agent**：配置审查标准（安全漏洞检查、性能反模式检测、命名规范验证），确保每次审查覆盖相同维度
- **特定框架的代码生成**：预设框架版本、组件模式和最佳实践，避免 Agent 使用过时的 API 或反模式
- **合规性检查 Agent**：针对 GDPR、SOC 2 等合规要求定制检查逻辑，确保生成的代码满足审计标准
- **技术文档写作 Agent**：预设文档模板、术语表和风格指南，保证文档输出的一致性

Custom Agents 的投入产出比在团队规模上升时急剧提高——配置一次，全团队受益。这与 [agentic coding](/zh/glossary/agentic-coding) 的发展趋势一致：从个人生产力工具走向团队级的 AI 工程基础设施。

## 总结

Subagents 和 Custom Agents 不是二选一的关系，而是解决不同层面问题的互补方案。**Subagents 解决的是执行效率问题**——如何让多个任务同时跑起来；**Custom Agents 解决的是质量一致性问题**——如何确保每次输出都符合标准。

最佳实践是组合使用：用 Custom Agent 定义任务规范（"按照这些规则写测试"），然后让 Subagents 并行执行该规范（"同时为 20 个模块按规则写测试"）。这种"Custom Agent 定义标准 + Subagents 并行执行"的模式，是当前 [Codex 多智能体工作流](/zh/blog/con-u-pour-des-workflows-multi-agents)的推荐架构。

选择的关键问题只有两个：你的瓶颈是速度还是一致性？如果是速度，先用 Subagents；如果是一致性，先配 Custom Agents。两者都有，就组合起来用。

## 常见问题

### Subagents 和 Custom Agents 可以同时使用吗？

可以。推荐的模式是用 Custom Agent 定义任务的行为规范和约束条件，然后通过 Subagents 机制将任务并行分发。这样既保证了输出一致性，又获得了并行执行的效率提升。主 Agent 负责编排，Subagents 负责执行，Custom Agent 配置负责质量控制。

### 使用 Subagents 时如何避免子任务之间的冲突？

Subagents 运行在隔离的沙箱环境中，文件系统操作互不影响。但如果多个 Subagent 修改同一个文件的不同部分，合并阶段可能产生冲突。最佳实践是确保子任务的文件修改范围不重叠，或者在任务描述中明确指定每个子任务的文件边界。

### Custom Agents 的配置文件如何在团队间共享？

Custom Agent 配置通常以 JSON 或 YAML 文件的形式存储在项目仓库中，与代码一起版本控制。团队成员 clone 仓库后即可使用相同的 Agent 配置。这种方式类似于 [Agent SDK](/zh/glossary/agent-sdk) 的设计理念——将 AI 行为定义为可版本化、可审计的工程资产。
