# Claude Memory vs CLAUDE.md：哪个才是你真正需要的上下文管理方案？

**TL;DR：** **CLAUDE.md** 是写给项目的规则手册——团队共享、版本可控、确定性输出。**Claude Memory（Auto Memory）** 是写给个人的自动笔记——跨会话记忆用户偏好和反馈。两者不是替代关系，而是互补层：CLAUDE.md 管"这个项目怎么做"，Memory 管"这个人怎么合作"。大多数认真使用 Claude Code 的开发者应该同时用好两者。

## 概述：Claude Memory（自动记忆）

**Claude Memory** 是 Claude Code 的跨会话持久化记忆系统。它通过 `.claude/projects/` 目录下的 markdown 文件，自动积累用户在对话中透露的偏好、反馈、项目背景和外部资源引用。

Memory 解决的核心问题是"重复交代上下文"。没有 Memory 之前，每次新开一个 Claude Code 会话，你都得重新说一遍"我是做数据工程的""测试不要用 mock""commit message 用中文"。Memory 把这些信息写入文件，下次自动加载。

Memory 分四种类型：**user**（用户画像）、**feedback**（行为纠正）、**project**（项目动态）、**reference**（外部资源指针）。它由 Claude Code 自动管理——检测到值得记住的信息时主动保存，但用户也可以显式要求"记住这个"或"忘掉那个"。详细机制可参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是放在项目根目录的指令文件，Claude Code 在每次启动时自动读取。它的定位是项目级规则手册：编码规范、构建命令、架构约束、工作流要求——所有你希望 AI 助手在这个项目里遵守的规则。

CLAUDE.md 的核心优势是**确定性和可共享性**。它是一个普通的 markdown 文件，跟代码一起进 git，团队每个成员看到的规则一模一样。修改它就像修改代码——提 PR、review、merge，有完整的变更历史。

在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中，CLAUDE.md 是最基础的一层。它决定了 Claude Code 对整个项目的理解框架，影响从代码风格到提交规范的方方面面。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **作用范围** | 个人级，跨项目 | 项目级，跨团队 | 各有所长 |
| **创建方式** | AI 自动提取 + 用户指令 | 人工编写和维护 | CLAUDE.md |
| **版本控制** | 不进 git，个人本地文件 | 进 git，团队共享 | CLAUDE.md |
| **内容类型** | 偏好、反馈、背景、资源引用 | 规则、命令、约束、架构 | 各有所长 |
| **持久化位置** | `~/.claude/projects/` | 项目根目录 `CLAUDE.md` | 平手 |
| **一致性保证** | 因人而异，同一团队不同人不同记忆 | 团队统一，所有人看到相同规则 | CLAUDE.md |
| **维护成本** | 低，自动积累 | 中，需要手动更新 | Memory |
| **可审计性** | 低，难以 review 全部记忆 | 高，标准 code review 流程 | CLAUDE.md |

## 深度分析：确定性 vs 自适应性

这是 CLAUDE.md 和 Memory 最根本的设计差异。

**CLAUDE.md 追求确定性。** 同一个 CLAUDE.md 文件，不管是谁在用、第几次打开，Claude Code 收到的指令完全一致。这对团队协作至关重要——你不希望同一个项目里，张三的 AI 用 tabs 缩进，李四的用 spaces。规则一致，输出才一致。

**Memory 追求自适应。** 它根据你的行为模式自动调整。如果你三次纠正"不要在 response 末尾加总结"，Memory 会记住这个偏好，以后自动遵守。你不需要把这种个人习惯写进项目的 CLAUDE.md（那对其他团队成员没意义）。

实际使用中，最常见的错误是把该放 CLAUDE.md 的内容说成 Memory，或者反过来。一个判断标准：**如果新加入团队的人也需要知道这件事，放 CLAUDE.md；如果只有你自己在乎，留给 Memory。**

以 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)的视角来看，CLAUDE.md 是基础设施层——所有上层能力（Skills、Hooks、Agents）都建立在它提供的项目上下文之上。Memory 则是个性化层——在同一套基础设施上叠加个人偏好。

## 深度分析：维护模式与生命周期

**CLAUDE.md 的维护是显式的。** 你写什么它就是什么，不会自动变化。这意味着你需要在项目演进时主动更新它——新增了构建命令、改了数据库 schema、调整了部署流程，都要同步到 CLAUDE.md。好处是你完全可控；代价是如果忘了更新，CLAUDE.md 和实际项目会脱节。

**Memory 的维护是隐式的。** Claude Code 在对话中自动积累记忆，也会在发现记忆过时时更新或删除。但这种自动化并非万能——Memory 可能记住已经不再正确的信息（比如"用户负责 auth 模块"，但三个月后你换了方向）。过期的 Memory 会导致 Claude Code 基于错误假设行动。

实践建议：每隔几周检查一下 `~/.claude/projects/` 下的记忆文件，清理不再准确的内容。CLAUDE.md 则应该在每次项目架构变更时更新——最好在 code review 流程中加一条检查项。

## 什么时候用 Claude Memory

Memory 最适合记录**个人工作方式**和**短期项目动态**：

- **个人偏好**：commit message 用中文还是英文、response 要不要加总结、代码注释的风格
- **行为纠正**：你纠正过的错误不应该再犯——"不要 mock 数据库""不要自动 push"
- **角色上下文**：你是后端工程师、数据科学家、还是产品经理，决定了 Claude Code 应该怎么跟你沟通
- **外部资源指针**：bug 在哪个 Linear 项目里追踪、oncall dashboard 的 URL 是什么

如果你正在探索 Claude Code 的深度用法，[我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)里有一些关于如何让 Memory 为你工作的实际例子。

## 什么时候用 CLAUDE.md

CLAUDE.md 最适合记录**团队共识**和**项目规则**：

- **构建和验证命令**：`npm run build`、`npm test`、`npm run lint` 的用法和前置条件
- **代码风格约束**：命名规范、文件组织方式、错误处理模式
- **架构决策**：为什么用 SQLite 而不是 Postgres、为什么 pipeline 分七个 stage
- **安全红线**：哪些操作绝对不能做——不要跳过测试、不要 commit .env 文件
- **工作流规则**：新功能先讨论设计、bug fix 不能随机试错、pipeline 改动必须跑验证

## 结论

**CLAUDE.md 和 Claude Memory 不是二选一的关系——它们是同一个系统的两个层次。** CLAUDE.md 是地基：项目规则、团队共识、确定性保障。Memory 是个性化层：个人偏好、行为校正、动态上下文。

如果你只能选一个先做好，**先写 CLAUDE.md**。一个好的 CLAUDE.md 能让 Claude Code 从"通用 AI 助手"变成"了解你项目的工程师"。Memory 会在你使用过程中自然积累，但没有 CLAUDE.md 提供的项目框架，Memory 记住的碎片信息也难以发挥最大价值。

对于团队用户，CLAUDE.md 是必须的——它是保证团队成员获得一致 AI 行为的唯一手段。Memory 是锦上添花：让每个人的 Claude Code 适应自己的工作节奏，而不影响团队规范。如果你想更系统地掌握 Claude Code 的配置层次，[Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)提供了从 CLAUDE.md 到 SKILL.md 的完整指南。

## 常见问题

### Claude Memory 和 CLAUDE.md 会冲突吗？

不会直接冲突，但可能出现不一致。如果 CLAUDE.md 规定"commit message 用英文"，而 Memory 记录了"用户偏好中文 commit message"，Claude Code 会优先遵守 CLAUDE.md 的规则。CLAUDE.md 是显式指令，优先级高于 Memory 中的隐式偏好。

### 我可以把 Memory 的内容手动复制到 CLAUDE.md 吗？

可以，但要区分适用范围。如果某条 Memory 对整个团队都有用（比如"这个项目的测试必须用真实数据库"），就应该提炼成规则写进 CLAUDE.md。如果只是个人偏好（比如"我喜欢简短的 response"），留在 Memory 里更合适。

### Memory 文件可以在团队间共享吗？

技术上可以复制，但不建议这样做。Memory 的设计初衷是个人化——每个人的工作方式不同，强行共享会导致规则混乱。团队级别的共识应该放在 CLAUDE.md 和 SKILL.md 里，通过 git 管理。
