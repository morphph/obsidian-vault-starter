# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR:** **CLAUDE.md** 是团队共享的项目规则手册，签入 Git、所有人生效；**Claude Memory** 是个人自动笔记本，跨会话记住你的偏好和上下文。两者不是替代关系——**CLAUDE.md 管"项目怎么做"，Memory 管"你是谁、你习惯怎么做"**。大多数团队应该同时使用：CLAUDE.md 统一工程规范，Memory 让 AI 适应个人工作风格。

## 概览：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Claude Code 的自动持久化记忆系统，它在对话中学习你的偏好、角色、反馈和项目上下文，然后以 Markdown 文件的形式存储在 `~/.claude/projects/` 目录下。下次开启新会话时，这些记忆会自动加载，让 AI 不需要你重复说明"我是谁"和"我喜欢怎么工作"。

Memory 的核心价值在于**个性化**。它记录四类信息：用户画像（你的角色和技术背景）、行为反馈（"不要在测试里 mock 数据库"）、项目动态（"周四之后冻结合并"）、外部引用（"Bug 追踪在 Linear 的 INGEST 项目里"）。这些记忆是私有的，不进 Git，不与团队共享。详细机制可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概览：CLAUDE.md

**CLAUDE.md** 是放在项目根目录（或 `~/.claude/` 全局目录）的 Markdown 指令文件，它告诉 Claude Code 这个项目的规则、约定和工作流程。它是手动编写、签入版本控制的——本质上是一份给 AI 读的工程规范文档。

CLAUDE.md 解决的问题是**团队一致性**。当五个工程师在同一个 repo 里用 Claude Code，CLAUDE.md 确保 AI 对每个人都遵循相同的编码规范、测试策略和 Git 工作流。它支持层级结构：全局 `~/.claude/CLAUDE.md` → 项目根目录 `CLAUDE.md` → 子目录 `CLAUDE.md`，越具体的规则优先级越高。想深入了解 Claude Code 的分层架构，可以看 [七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 核心对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **维护方式** | AI 自动写入 | 人工手动编写 | Memory |
| **作用域** | 个人（当前用户） | 项目（所有协作者） | CLAUDE.md |
| **版本控制** | 不进 Git | 签入 Git | CLAUDE.md |
| **内容类型** | 偏好、反馈、上下文 | 规则、约定、工作流 | 平手 |
| **生效时机** | 每次会话自动加载 | 每次会话自动加载 | 平手 |
| **可审查性** | 仅本人可见 | 团队可 review | CLAUDE.md |
| **更新频率** | 对话中实时更新 | 需要手动修改并提交 | Memory |
| **存储位置** | `~/.claude/projects/*/memory/` | 项目根目录或子目录 | 平手 |

## 维护成本与自动化：谁更省心？

**Claude Memory 几乎零维护成本**，这是它最大的优势。你在对话中说"以后不要在 commit message 末尾加 emoji"，AI 自动把这条反馈写入记忆文件，下次会话直接生效。你不需要打开任何配置文件，不需要记住语法格式，也不需要提交代码。

CLAUDE.md 则需要刻意维护。你要手动编写规则、确保格式正确、提交到 Git、在团队中达成共识。当项目规范变化时（比如从 Jest 迁移到 Vitest），你需要手动更新 CLAUDE.md，否则 AI 会继续按旧规范工作。

但这个"缺点"其实也是优点——CLAUDE.md 的每一次修改都经过人类审查，都有 Git 历史可追溯。Memory 的自动写入虽然方便，但有时会记录不准确或过时的信息，需要定期清理。

实际工作中，Memory 适合那些"说一次就不想再说第二次"的个人习惯；CLAUDE.md 适合那些"必须每个人都遵守"的项目规范。

## 协作与团队场景：谁更适合多人项目？

**CLAUDE.md 在团队协作中占绝对优势。** 它签入 Git，走 PR 审查流程，所有人看到的是同一份规则。当新成员加入团队，他们不需要手动配置 Claude Code——克隆 repo 后，AI 自动读取 CLAUDE.md 并按团队规范工作。

Memory 是完全私有的。你的记忆不会影响同事的 Claude Code 行为，同事的记忆也不会影响你。这意味着即使两个人在同一个 repo 工作，AI 对他们的响应风格可能完全不同——一个人的 AI 习惯用中文回复，另一个人的 AI 默认英文。

这在大多数场景下是合理的分工：项目级规则用 CLAUDE.md 统一，个人偏好用 Memory 各自保留。但如果你的团队有很强的"AI 使用规范"（比如"所有人的 Claude Code 都必须在 commit 前跑测试"），这条规则应该写在 CLAUDE.md 里，而不是依赖每个人的 Memory。

关于团队如何系统化地管理 Claude Code 的扩展机制（包括 Skills、Hooks 等），可以参考 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 内容边界：什么该放哪里？

这是最容易搞混的部分。一条简单的判断规则：

**放 CLAUDE.md 的内容：**
- 编码规范（命名风格、缩进、import 排序）
- 构建和测试命令（`npm run build`、`npm test`）
- Git 工作流（commit 前必须通过哪些检查）
- 架构约束（"不要在 pipeline 脚本里 import Next.js 模块"）
- 文档更新规则（改了哪个目录就更新哪个文档）

**放 Memory 的内容：**
- 你的角色和技术背景（"我是数据科学家，主要看日志"）
- 行为反馈（"不要在每次回复末尾总结你做了什么"）
- 项目动态（"周四之后冻结非关键合并"）
- 外部资源指向（"oncall 看板在 grafana.internal/d/api-latency"）

**两者都不该放的内容：**
- 代码结构和文件路径（直接读 repo 就行）
- Git 历史（`git log` 是权威来源）
- 调试方案（修复已经在代码里了）

如果你发现自己在 CLAUDE.md 里写"我喜欢简短的回复"，那应该移到 Memory。如果你在 Memory 里写"所有 API 路由必须有错误处理中间件"，那应该移到 CLAUDE.md。[Anthropic 的记忆系统升级](/zh/blog/anthropic-claude-memory-upgrades-importing)也在持续优化这两个系统之间的边界。

## 什么时候选 Claude Memory

Memory 最适合**个人开发者**和**重复使用 Claude Code 的高频用户**。如果你每天都在用 Claude Code，而且受够了每次新会话都要重复说明"我用 Vim 键位"、"commit message 用中文"、"不要加多余的注释"——Memory 就是为你设计的。

具体场景：
- 你在多个项目间切换，但个人偏好保持一致
- 你的工作习惯在不断演化，不想每次都手动更新配置
- 你希望 AI 记住之前的对话上下文（比如"上次讨论的那个 bug"）
- 你是独立开发者，不需要和团队同步 AI 行为

## 什么时候选 CLAUDE.md

CLAUDE.md 最适合**团队项目**和**需要可审查、可复现的 AI 行为**的场景。如果你的项目有多个贡献者，或者你需要确保 AI 在 CI/CD 环境中的行为和本地一致——CLAUDE.md 是唯一选择。

具体场景：
- 团队需要统一的编码规范和工作流
- 新成员入职时需要 AI 自动遵循项目约定
- 你需要 AI 行为的变更历史（Git blame）
- CI/CD 流水线中使用 Claude Code（没有个人 Memory）
- 项目有复杂的构建、测试、部署规则

了解如何从工具使用者进阶到系统设计者，可以看看 [Claude Code 不是一个编程工具](/zh/blog/claude-code-is-not-a-coding-tool)这篇文章。

## 结论

**大多数认真使用 Claude Code 的开发者应该同时使用两个系统。** CLAUDE.md 是项目的"宪法"——团队共识、工程规范、质量门禁，写一次、所有人受益。Memory 是你的"个人助理笔记"——你的身份、偏好、反馈，自动积累、跨会话保持。

实操建议：先从 CLAUDE.md 开始，把项目最重要的 3-5 条规则写进去。然后正常使用 Claude Code，让 Memory 自然积累你的个人偏好。定期（每月一次）检查 Memory 文件，清理过时或不准确的记忆。当团队中有人发现某条个人 Memory 对所有人都有价值时，把它"提升"到 CLAUDE.md。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。Claude Code 在每次会话启动时会同时加载 CLAUDE.md 和 Memory 文件。CLAUDE.md 提供项目级规则，Memory 提供个人偏好，两者互补。如果出现冲突，CLAUDE.md 的规则通常优先，因为它代表的是项目约定而非个人习惯。

### Claude Memory 的内容会被同事看到吗？

不会。Memory 文件存储在你本地的 `~/.claude/projects/` 目录下，不会进入 Git 仓库，也不会上传到任何共享服务。你的记忆完全私有，只影响你自己的 Claude Code 会话。

### CLAUDE.md 写太长会影响性能吗？

会。CLAUDE.md 的内容会被注入到每次对话的上下文中，过长的文件会占用 token 额度并可能降低 AI 对关键规则的注意力。建议控制在 500 行以内，将详细文档放在 `docs/` 目录下，CLAUDE.md 只放最核心的规则和指向详细文档的引用。

### 如何清理过时的 Memory？

Memory 文件是普通的 Markdown 文件，存放在 `~/.claude/projects/` 对应目录的 `memory/` 文件夹下。你可以直接编辑或删除这些文件。也可以在对话中告诉 Claude Code"忘记关于 X 的记忆"，它会自动找到并移除相关记忆条目。
