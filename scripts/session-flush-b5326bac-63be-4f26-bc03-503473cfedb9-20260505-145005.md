**Codex custom agents** execute in ephemeral cloud containers. When you submit a task, Codex clones your repository into a fresh sandbox, applies your custom agent's configuration (system prompt, allowed tools, file access patterns), and lets it work. The agent can install dependencies, run builds, execute tests — all within its isolated environment. When done, it produces a diff or PR that you review.

This isolation model has clear advantages for safety: a misbehaving agent can't corrupt your local state, can't accidentally push to production, can't interfere with another agent's work. It also enables true parallelism — ten agents can work simultaneously on different parts of your codebase without conflicts.

The tradeoff is latency and context. Each agent starts cold — it must re-read the repository, rebuild its understanding of the codebase, and work without access to your current editor state, uncommitted changes, or in-progress work. For tasks that depend on nuanced local context ("fix the bug I'm currently debugging"), this creates friction.

**Claude Code subagents** operate within your active terminal session. The parent Claude Code instance decides when to spawn a subagent, writes a detailed prompt briefing it on the task (since subagents don't inherit conversation history), and either waits for results or continues working in parallel.

Subagents access the same filesystem, the same git state, and the same development environment as the parent. An Explore-type subagent can grep your codebase and report findings. A general-purpose subagent can edit files, run tests, and commit changes. A Plan-type subagent can analyze architecture and propose implementation strategies — all using the exact state of your working directory right now.

The worktree isolation option (`isolation: "worktree"`) provides a middle ground: the subagent gets its own git worktree to work in, preventing conflicts with the parent's changes while still sharing the local environment.

## Configuration and Customization: Detailed Analysis

How you define and specialize agents differs substantially between the platforms, affecting flexibility and reusability.

**Codex custom agents** are configured as named profiles. You define an agent with a specific system prompt that shapes its behavior — "You are a test-writing specialist. Given a module, write comprehensive unit tests covering edge cases" — along with tool permissions and file access rules. These profiles persist and can be reused across tasks.

The configuration model favors teams with repeatable workflows. You build a library of specialized agents — your "test writer," your "docs updater," your "security reviewer" — and route tasks to the appropriate agent. Configuration happens outside the coding flow, in Codex's management interface or via CLI setup commands. Changes to agent behavior require updating the profile, not the task prompt.

**Claude Code subagents** are configured per-invocation through the prompt parameter. Rather than pre-defined profiles, you write a briefing each time you spawn an agent. The built-in agent types (Explore, Plan, general-purpose, code-reviewer) provide baseline tool access and behavioral patterns, but the specific instructions are crafted for each task.

This means Claude Code subagents are more adaptive — you can tailor instructions to the exact situation without maintaining a library of agent profiles. But it also means you're writing more detailed prompts for each delegation. The [SKILL.md system](/blog/5-claude-code-skills-i-use-every-single-day) partially addresses this: reusable instruction files that Claude Code loads as context, giving subagents consistent behavioral guidance without rigid profile management.

For teams, Claude Code's approach through [skills and hooks](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) means agent behavior is version-controlled alongside your code — agent instructions evolve with the codebase rather than living in a separate configuration system.

## Orchestration and Coordination: Detailed Analysis

Multi-agent systems live or die by how well agents coordinate. A collection of brilliant but uncoordinated agents produces conflicting diffs and wasted work.

**Codex's coordination model** is deliberately minimal. Custom agents run independently — they don't communicate with each other during execution. Coordination happens at the human layer: you review outputs, resolve conflicts, and decide which agent's work to merge. This matches Codex's async mental model — agents are like team members working on separate branches. You merge their work through standard code review.

For tasks with natural boundaries (different files, different modules, different concerns), this works well. Ten agents can generate ten independent PRs with no coordination overhead. For tasks with dependencies ("refactor the API, then update all callers"), you must sequence them manually — submit the API refactoring first, wait for the result, then submit the caller updates with the new API shape as context.

**Claude Code's coordination model** is orchestrated by the parent agent. The parent maintains full context of the overall task, decides what to delegate and what to handle directly, waits for subagent results, and synthesizes findings. This enables sophisticated workflows:

- Spawn three Explore agents in parallel to search different aspects of the codebase
- Use their findings to inform a Plan agent's architecture proposal
- Execute the plan with a general-purpose agent working in a worktree
- Review the result with a code-reviewer agent before integrating

The parent acts as project manager — it holds the big picture and routes work to specialists. This works well for complex tasks where sub-results inform later steps. The limitation is throughput: everything flows through a single parent session, which bounds total parallelism and context capacity.

## Practical Workflows

### Workflow 1: Comprehensive Test Generation

**With Codex custom agents:** Configure a "test writer" agent with instructions about your testing framework, coverage expectations, and style. Submit each module separately — "Write tests for src/auth/login.ts", "Write tests for src/api/users.ts" — and let agents work in parallel. Review all generated test PRs, resolve any import conflicts, merge.

**With Claude Code subagents:** In a single session, the parent analyzes which modules need tests, spawns background subagents for each module (using worktree isolation to avoid file conflicts), collects results, and presents a unified diff. You review once and commit.

**Verdict:** Codex is simpler for independent modules. Claude Code handles interdependent test suites better because the parent can coordinate shared fixtures and avoid duplicate helper functions.

### Workflow 2: Codebase-Wide Refactoring

**With Codex custom agents:** Submit the refactoring task to a single agent (multi-agent coordination for coupled changes is complex). The agent works in its sandbox, produces a large diff. For very large refactors, you might split by directory and submit multiple agents, then manually reconcile.

**With Claude Code subagents:** The parent agent plans the refactoring strategy, uses Explore agents to map all affected files, delegates file-level changes to subagents, and integrates results — handling conflicts as they arise within the session.

**Verdict:** Claude Code's orchestrated approach handles coupled refactors more naturally. Codex is safer for embarrassingly parallel changes (rename a symbol across independent packages).

### Workflow 3: Code Review

**With Codex custom agents:** Submit a PR for review to a "security reviewer" agent and a "performance reviewer" agent simultaneously. Each produces independent feedback.

**With Claude Code subagents:** Spawn a code-reviewer subagent that analyzes the diff. For deeper review, the parent can spawn multiple specialized agents (security focus, performance focus) and synthesize their findings into unified feedback.

**Verdict:** Similar capability. Codex's named profiles make it easier to maintain consistent review standards across a team. Claude Code's synthesis step produces more coherent feedback.

## When to Choose Codex Custom Agents

Codex custom agents fit teams and workflows with these characteristics:

- **Batch-oriented development**: You accumulate tasks and dispatch them in batches, reviewing results later. The async model matches how you think about work.
- **Strong isolation requirements**: You need guarantees that agent work can't affect production, can't see secrets outside its scope, or can't interact with other agents' work.
- **Team standardization**: You want consistent agent behavior across team members without relying on individual prompt craft. Named profiles enforce organizational standards.
- **Independent task parallelism**: Your work decomposes into clearly independent units — separate services, separate modules, separate concerns — with minimal coordination needs.
- **CI/CD integration**: You want agents as part of your automated pipeline — triggered by events, producing PRs for review, running on schedule.

Read our [complete Codex guide](/blog/codex-complete-guide) for setup instructions and [Codex VS Code integration](/blog/codex-vscode) for IDE-based workflows.

## When to Choose Claude Code Subagents

Claude Code subagents fit developers and workflows with these characteristics:

- **Interactive development**: You're actively working on a task and want AI to handle sub-tasks in real-time, feeding results back into your current flow.
- **Context-dependent work**: Your task requires deep local context — uncommitted changes, current debugging state, environment-specific configuration — that would be lost in a cloud sandbox.
- **Coupled sub-tasks**: The work has dependencies between parts. One agent's output informs another agent's input. The parent's orchestration handles this naturally.
- **Rapid iteration**: You want to see results immediately, adjust course, and re-delegate — not wait for async completion.
- **Exploration and research**: You need agents to search, analyze, and report findings that inform your next decision. Explore-type agents excel at this.

See our [subagent examples](/blog/claude-code-subagents-examples) for practical patterns and our [agent teams deep-dive](/blog/claude-code-agent-teams) for architecture details.

## Combining Both Approaches

These tools aren't mutually exclusive. A practical hybrid workflow:

1. Use **Claude Code subagents** during active development — exploring code, planning changes, executing coupled edits with real-time orchestration
2. Use **Codex custom agents** for batch operations — generating tests across a monorepo, running standardized reviews on all open PRs, producing documentation updates overnight
3. Use Claude Code's [skills system](/blog/9-principles-writing-claude-code-skills) to codify agent instructions that work in both contexts

The decision boundary is synchronous vs. asynchronous: if you need results now to continue working, use Claude Code subagents. If you can dispatch work and review later, use Codex custom agents.

## Verdict

**For interactive, context-rich, orchestrated multi-agent coding — choose Claude Code subagents.** The parent-child architecture, shared local environment, and real-time coordination handle complex development tasks where sub-results inform later steps. You stay in flow and get synthesized results without leaving your terminal.

**For batch-parallel, isolated, team-standardized agent work — choose Codex custom agents.** The async cloud model, named profiles, and strong isolation make it ideal for dispatching independent tasks at scale and reviewing results through standard code review workflows.

Most teams will benefit from both: Claude Code for the developer's active session, Codex for background automation and team-wide standardization. The [agentic coding](/glossary/agentic-coding) landscape is converging on multi-agent patterns regardless of platform — the question is whether your workflow is synchronous or asynchronous, coupled or independent.

## Frequently Asked Questions

### Can Codex custom agents communicate with each other during execution?

No. Codex custom agents run in isolated sandboxes without cross-agent communication. Coordination happens at the human or CI layer — you review outputs and resolve conflicts between agent results manually or through automated merge checks.

### How many subagents can Claude Code run in parallel?

Claude Code can spawn multiple subagents simultaneously using parallel tool calls. Practical limits depend on your context window and token budget rather than a hard agent count cap. Background agents run without blocking the parent, while foreground agents return results before the parent continues.

### Do I need to configure anything to use Claude Code subagents?

No setup is required. Subagents are built into Claude Code and available immediately. You can enhance their behavior through [SKILL.md files](/blog/5-claude-code-skills-i-use-every-single-day) and project-level CLAUDE.md instructions, but the base capability works out of the box.

### Can Codex custom agents access my local development environment?

No. Codex agents run in cloud sandboxes with a snapshot of your repository. They cannot access local files, running processes, environment variables, or uncommitted changes outside what's in the repo. This is a deliberate safety choice that trades local context for execution isolation.

### Which approach is better for large monorepos?

Both handle monorepos but differently. Codex custom agents can parallelize across packages with zero coordination overhead — ideal for independent services. Claude Code subagents with worktree isolation handle coupled cross-package changes better because the parent orchestrates dependencies. For monorepos with tight coupling, Claude Code's orchestration wins; for loosely coupled monorepos, Codex's parallelism wins.
