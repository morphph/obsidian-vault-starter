**Claude Memory** is Claude Code's automatic persistence system that saves context across conversations without manual file editing. When you tell Claude something about yourself, correct its approach, or share project context that isn't derivable from code, Memory captures it in structured markdown files stored in `~/.claude/projects/<project>/memory/`. It operates across four types: user memories (your role and expertise), feedback memories (corrections and confirmed approaches), project memories (ongoing initiatives and deadlines), and reference memories (pointers to external systems).

The key distinction is that Memory is **per-user and automatic**. Claude decides what to save based on conversation signals — a correction triggers a feedback memory, a role mention triggers a user memory. The system maintains a `MEMORY.md` index file that gets loaded into every future conversation, giving Claude persistent awareness of who you are and how you work. For a deeper look at how both persistence layers function together, see our [Claude Code Memory System guide](/blog/claude-code-memory).

## Overview: CLAUDE.md

**CLAUDE.md** is a manually maintained markdown file checked into your repository's root that provides project-level instructions to Claude Code. Every session reads it before doing any work. It contains build commands, quality gates, coding conventions, workflow rules, and project-specific constraints — anything that should be consistent across all team members and all sessions.

Because CLAUDE.md lives in your repo, it travels with the code. New team members, CI environments, and fresh sessions all inherit the same instructions. There's also a hierarchy: a global `~/.claude/CLAUDE.md` for cross-project preferences, a project-level `CLAUDE.md` in the repo root, and optionally per-directory CLAUDE.md files for subsystem-specific rules. This layered system means you can set organization-wide standards while allowing project-specific overrides. Our [complete Claude Code guide](/blog/claude-code-complete-guide) covers how CLAUDE.md fits into the broader tool architecture.

## Persistence Architecture: How They Work Together

Claude Code's persistence is a two-layer system, not an either-or choice. Understanding the architecture explains when each layer activates and why both exist.

**CLAUDE.md loads first.** When you start a Claude Code session, the tool reads the global CLAUDE.md, then the project CLAUDE.md, then any directory-level CLAUDE.md files. These form the base instruction set — the rules that apply regardless of who is running the session. Build commands, lint configurations, commit message formats, forbidden patterns, and architecture constraints all belong here. This layer is deterministic: same repo, same instructions, every time.

**Memory loads second.** After CLAUDE.md, Claude loads the `MEMORY.md` index from `~/.claude/projects/<project>/memory/`. This adds your personal context layer — your role, expertise level, past corrections, and ongoing project context. Memory is probabilistic: Claude decides what to save and when to recall it, based on relevance signals.

The separation matters for a specific reason. CLAUDE.md answers "how should this project be worked on?" Memory answers "how should this project be worked on *by you*?" A senior backend engineer and a junior frontend developer working on the same repo get the same CLAUDE.md instructions but different Memory contexts — Claude adjusts its explanations, suggestions, and assumptions accordingly.

## Content Boundaries: What Goes Where

The most common mistake teams make is putting the wrong content in the wrong layer. Here's a concrete decision framework.

### Belongs in CLAUDE.md

- **Build and test commands**: `npm run build`, `npm test`, `npm run lint` — anything a new contributor needs
- **Quality gates**: "Before ANY commit, all tests must pass" — non-negotiable project rules
- **Coding conventions**: "Use snake_case for database columns" — standards that affect code review
- **Architecture constraints**: "Never import Next.js modules in pipeline scripts" — guardrails against known pitfalls
- **Workflow rules**: "New feature → discuss design first, get approval before coding" — process that applies to everyone
- **Forbidden patterns**: "Do NOT skip failing tests. Do NOT comment out lint rules." — explicit prohibitions

These are all deterministic, team-wide, and derive their authority from being checked into the repo alongside the code they govern.

### Belongs in Memory

- **Your role and expertise**: "Senior data engineer, new to React" — so Claude calibrates its explanations
- **Behavioral corrections**: "Don't summarize at the end of responses" — personal preference that shouldn't be imposed on teammates
- **Confirmed approaches**: "User prefers bundled PRs over many small ones for refactors" — validated judgment calls
- **Project context not in code**: "Merge freeze starts March 5 for mobile release" — temporal facts that affect decision-making
- **External system references**: "Pipeline bugs tracked in Linear project INGEST" — pointers Claude can't derive from the repo

These are personal, adaptive, and would be inappropriate to enforce on the entire team through a checked-in file.

### The Gray Zone

Some information could reasonably go in either layer. The deciding question is: **should a new team member who has never talked to Claude inherit this instruction?**

If yes → CLAUDE.md. If no → Memory.

For example, "commit messages should follow conventional commits format" affects everyone and belongs in CLAUDE.md. But "this user prefers terse commit messages with no emoji" is personal and belongs in Memory.

## Team Dynamics: Shared vs Personal

The team implications of each layer are significant and often underappreciated.

**CLAUDE.md creates consistency.** When five engineers work on the same repo, CLAUDE.md ensures Claude follows the same rules for all of them. Code style, test requirements, architectural boundaries — these stay uniform. Pull requests from Claude-assisted sessions follow the same conventions regardless of who initiated them. This is especially valuable for [agentic coding](/glossary/agentic-coding) workflows where Claude operates with significant autonomy.

**Memory creates personalization.** Each engineer's Memory layer captures their individual working style. One developer might prefer detailed explanations of database changes. Another might want Claude to skip explanations and just make the change. These preferences coexist without conflict because Memory is per-user.

**The failure mode of CLAUDE.md-only teams** is that Claude treats every developer identically. A senior architect gets the same level of explanation as an intern. This wastes time for experts and may under-explain for newcomers.

**The failure mode of Memory-only teams** is inconsistency. Without shared project rules, each developer's Claude sessions may produce code with different conventions, skip different quality checks, or follow different commit workflows. The [skills system](/blog/5-claude-code-skills-i-use-every-single-day) can mitigate some of this by encoding reusable task instructions, but CLAUDE.md remains the foundation for project-wide standards.

## Maintenance and Lifecycle

How each system ages and requires upkeep is a practical concern that affects your daily workflow.

**CLAUDE.md requires deliberate maintenance.** When your build system changes, when you adopt a new testing framework, when you restructure your project — someone needs to update CLAUDE.md. The upside is predictability: you know exactly what instructions Claude is following because you wrote them. The downside is that stale instructions actively mislead Claude. A CLAUDE.md that references a removed script or outdated architecture is worse than no CLAUDE.md at all.

**Memory is self-maintaining but not self-cleaning.** Claude adds memories as it learns from your interactions. But memories can become stale — a project deadline passes, a team member's role changes, a workflow gets replaced. Claude Code has built-in guidance to verify memories before acting on them, but outdated memories can still influence behavior subtly. Periodic review of your `MEMORY.md` index and the individual memory files is good hygiene.

**Practical maintenance workflow:**

1. Review CLAUDE.md during sprint retrospectives or after significant infrastructure changes
2. Scan `~/.claude/projects/<project>/memory/` quarterly for stale memories
3. When you notice Claude doing something wrong, check both layers — the cause might be an outdated instruction in either one
4. When onboarding a new team member, walk through CLAUDE.md as part of repo orientation

## When to Choose Claude Memory

Choose Memory as your primary persistence mechanism when:

- **You're a solo developer** and don't need shared team conventions — Memory captures everything organically
- **Your project context changes frequently** — deadlines, priorities, and collaborator roles that you don't want to manually update in a file
- **You want Claude to learn from corrections** — tell it once not to do something, and it remembers across sessions
- **You're using Claude across multiple projects** — global user memories carry your personal preferences everywhere
- **You want adaptive behavior** — Claude that gets better at working with you specifically over time

Memory is particularly valuable for the [effective prompting](/blog/how-to-effectively-prompt-a-claude-code) workflow. Instead of repeating context in every prompt, Memory lets Claude build a persistent understanding of your working style.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence mechanism when:

- **You're working in a team** and need consistent AI behavior across all members
- **Your project has strict quality gates** — build checks, test requirements, and lint rules that must never be skipped
- **You need auditability** — CLAUDE.md is version-controlled, so you can track what instructions changed and when
- **You're onboarding new developers** — CLAUDE.md serves as both AI configuration and project documentation
- **You want deterministic behavior** — the same session setup every time, no variation based on past conversations
- **You're setting up CI/CD with Claude** — automated environments need checked-in instructions, not per-user memory

For teams building structured AI workflows, combining CLAUDE.md with [skills files](/blog/9-principles-writing-claude-code-skills) creates a powerful, reproducible system.

## Verdict

**Use both.** CLAUDE.md and Claude Memory are complementary layers that solve different problems. **CLAUDE.md is your project's constitution** — the shared rules that every session and every team member follows. **Memory is your personal context** — the adaptive layer that makes Claude work better with you specifically over time.

If forced to choose one: **pick CLAUDE.md for team projects** (consistency matters more than personalization) and **Memory for solo projects** (adaptiveness matters more than formality). But the optimal setup is CLAUDE.md for shared project rules plus Memory for personal working style — and being disciplined about putting the right content in the right layer.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions take priority as the project-level authority. If CLAUDE.md says "always run tests before committing" and a memory says "user prefers to skip tests," the CLAUDE.md rule wins. Memory provides additional context and preferences, but it cannot contradict checked-in project instructions.

### Does Claude Memory sync across machines?

No. Memory files are stored locally in `~/.claude/projects/<project>/memory/` and do not sync automatically. If you work on multiple machines, each machine builds its own memory independently. CLAUDE.md, by contrast, syncs through git — clone the repo and you have the instructions.

### How do I see what Claude has memorized about me?

Check `~/.claude/projects/<project>/memory/MEMORY.md` for the index, then read individual memory files in the same directory. Each file has frontmatter with a name, description, and type. You can edit or delete any memory file directly — Claude will respect the changes in your next session.

### Should I commit CLAUDE.md to version control?

Yes. CLAUDE.md is designed to be checked into your repository alongside your code. It serves the same purpose as `.editorconfig` or `.eslintrc` — project configuration that should be shared. The only exception is `~/.claude/CLAUDE.md`, which is your global personal config and lives outside any repo.

### How much does each system add to Claude's context window?

CLAUDE.md loads the full file contents into every session. Keep it concise — a 200-line CLAUDE.md is workable, but a 2,000-line one wastes context. Memory loads only the `MEMORY.md` index (one line per memory) plus any individual memory files Claude deems relevant. The index is capped at 200 lines, so the memory footprint stays bounded even as you accumulate memories over time.
