# Claude Memory vs CLAUDE.md：两套持久化机制怎么选？

**TL;DR:** **CLAUDE.md** 是项目级的规则手册，定义 Claude Code 的行为约束和编码规范，跟随代码库版本控制；**Claude Memory** 是跨会话的自动笔记系统，记录用户偏好、项目上下文和反馈历史。两者互补而非替代——CLAUDE.md 管「怎么做」，Memory 管「记住什么」。团队协作场景优先用 CLAUDE.md，个人工作流优先用 Memory。

## 概述：Claude Memory

Claude Memory 是 Claude Code 内置的文件级持久化记忆系统，存储在 `.claude/projects/` 目录下。它的核心设计是让 Claude 在不同会话之间保持对用户、项目和工作上下文的理解——不需要每次重新解释你是谁、项目处于什么阶段、你偏好什么工作方式。

Memory 的存储结构是 Markdown 文件加 frontmatter 元数据，包含四种类型：用户信息（user）、行为反馈（feedback）、项目状态（project）和外部引用（reference）。系统通过 `MEMORY.md` 索引文件管理所有记忆条目，每次会话开始时自动加载。

Memory 的核心价值在于**累积学习**——Claude 可以记住你上周说「不要在测试里 mock 数据库」，下次自动遵守，无需重复指令。详细机制见我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是放在项目根目录（或 `~/.claude/CLAUDE.md` 全局路径）的 Markdown 指令文件，本质上是给 Claude Code 的「操作手册」。它随代码库版本控制，团队成员共享同一份规则，确保 AI 行为的一致性。

CLAUDE.md 的内容通常包括：构建命令、代码风格约束、提交规范、架构决策、已知陷阱和质量门槛。Claude Code 每次启动时会读取当前项目的 CLAUDE.md，将其作为系统级上下文注入对话。

CLAUDE.md 的核心价值在于**规范执行**——它定义了项目的「法律」，所有会话、所有用户都遵守同一套规则。关于如何高效编写 CLAUDE.md 规则，可参考 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)。

## 核心差异：定位与设计哲学

两者最根本的区别是**谁写、给谁用、用多久**。

CLAUDE.md 是人类工程师写给 AI 的项目规则书。它解决的是「团队里每个人用 Claude Code 时，AI 的行为是否一致」这个问题。一个人设定好编码规范、构建流程、架构约束，团队所有成员（以及 CI 中的自动化 Claude 实例）都遵守。这是一种**声明式的行为约束**。

Claude Memory 是 AI 写给自己的笔记本。它解决的是「我昨天跟用户聊了什么、用户偏好什么、项目现在到哪个阶段了」这个问题。Memory 是**累积式的上下文管理**——随着交互增多，Claude 对你和项目的理解越来越深。

打个比方：CLAUDE.md 像公司的员工手册，Memory 像你私人助理的工作笔记。前者对所有人生效，后者只对你和你的助理之间有意义。

想了解 Claude Code 完整的扩展体系，可以阅读 [Claude Code 扩展栈拆解：Skills、Hooks、Agents、MCP 四层架构](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 实际使用场景对比

### 场景一：编码规范

- **用 CLAUDE.md**：`## Style\n- Use Tailwind v4 utility classes\n- Never use inline styles`
- **不用 Memory**：编码规范是团队共享的规则，不是个人偏好

### 场景二：用户偏好

- **用 Memory**：「用户是高级后端工程师，前端解释需要用后端类比」
- **不用 CLAUDE.md**：这是针对个人的行为调整，不该强加给团队

### 场景三：项目当前状态

- **用 Memory**：「3月5日后代码冻结，mobile 团队切 release branch」
- **不用 CLAUDE.md**：这是临时状态，不是持久规则。放 CLAUDE.md 里会变成永久指令

### 场景四：构建和测试命令

- **用 CLAUDE.md**：`npm run build && npm test` 必须在提交前通过
- **不用 Memory**：这是项目的质量门槛，所有人都要遵守

### 场景五：过去的错误教训

- **用 Memory（feedback 类型）**：「不要在集成测试里 mock 数据库——上次 mock 测试通过但生产迁移失败」
- **也可以用 CLAUDE.md**：如果这个教训适用于所有团队成员，放 CLAUDE.md 的 Known Gotchas 部分

## 协同使用的最佳实践

真正高效的 Claude Code 工作流是两者协同使用，而非二选一。

**CLAUDE.md 负责的层面：**
- 项目架构和技术栈声明
- 构建、测试、部署命令
- 代码风格和提交规范
- 已知陷阱和避坑指南
- 质量门槛（所有提交必须通过的检查）

**Memory 负责的层面：**
- 用户的技术背景和知识水平
- 用户对 Claude 行为的反馈和修正
- 项目的当前阶段和近期目标
- 外部资源的位置（Linear board、Grafana dashboard）
- 团队成员的分工和时间线

一个典型的工作流：新团队成员 clone 项目后，CLAUDE.md 让 Claude 立即知道怎么构建、测试、遵循什么规范。随着这位成员使用 Claude Code，Memory 逐渐学习他的个人偏好——他喜欢简短回复、不要代码注释、习惯一次性大 PR 而非拆分小 PR。

关于 Anthropic 最近对记忆系统的升级（包括从 ChatGPT 导入数据），详见 [Claude 记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)。

## 常见误区

**误区一：「Memory 可以替代 CLAUDE.md」**

不能。Memory 是用户私有的，不入版本控制，不在团队间共享。如果你的规则需要所有团队成员和 CI 环境都遵守，必须放 CLAUDE.md。

**误区二：「把所有东西都放 CLAUDE.md」**

CLAUDE.md 应该精简。临时的项目状态、个人偏好、一次性的上下文不该污染项目规则文件。这些信息放 Memory 更合适。

**误区三：「Memory 会自动记住所有对话内容」**

Memory 是选择性存储——只记录跨会话有价值的信息。当前对话的任务细节、调试过程、中间状态不会自动进入 Memory。

## 结论

**选择 CLAUDE.md** 当你需要：团队共享的规则、版本控制的约束、所有环境一致的行为。

**选择 Claude Memory** 当你需要：个人偏好的累积、项目状态的跟踪、跨会话的上下文延续。

最佳实践是两者配合：CLAUDE.md 定义项目的「宪法」，Memory 记录个人的「工作日志」。对于想深入了解 Claude Code 完整能力的开发者，推荐阅读 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)，理解持久化机制在整体架构中的位置。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，两者是互补关系。CLAUDE.md 在会话开始时作为系统上下文加载，Memory 提供跨会话的个人化补充。Claude Code 会同时读取两者，不会冲突。

### CLAUDE.md 的修改会立即生效吗？

是的。Claude Code 在每次会话启动时读取 CLAUDE.md，修改后下次对话即生效。由于文件入 Git，团队成员 pull 最新代码后自动获得更新的规则。

### 如何删除或修改已有的 Memory 条目？

直接告诉 Claude「忘记 X」或「更新关于 Y 的记忆」，它会找到并修改对应的 Memory 文件。你也可以直接编辑 `.claude/projects/` 下的 Markdown 文件。

### Memory 的内容会被发送给 Anthropic 吗？

Memory 存储在本地文件系统中。它作为会话上下文的一部分发送给 API（和你的代码、对话内容一样），但不会被额外收集或用于模型训练。
