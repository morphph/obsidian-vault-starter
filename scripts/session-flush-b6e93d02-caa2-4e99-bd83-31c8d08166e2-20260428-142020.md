# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **CLAUDE.md** 是写给项目的规则手册——版本控制、团队共享、确定性执行；**Claude Memory** 是写给个人的自动笔记——跨会话、自动积累、因人而异。两者不是替代关系，而是互补关系。项目级规范用 CLAUDE.md，个人偏好和上下文用 Memory。搞反了，团队协作会出问题。

## Overview：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Anthropic 为 Claude 设计的持久化记忆系统，核心目标是让 AI 在多轮对话之间保持上下文连续性。在 Claude Code 中，Memory 以 Markdown 文件的形式存储在 `~/.claude/projects/<project>/memory/` 目录下，每条记忆都有独立文件和 frontmatter 元数据。

Memory 的关键特征是**自动性**和**个人性**。Claude Code 会在对话过程中自动识别值得记住的信息——你的角色、技术偏好、项目背景、反馈纠正——然后主动写入记忆文件。这些记忆绑定到你个人的 `~/.claude/` 目录，不会进入版本控制，也不会与团队成员共享。

根据 [Claude 记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing) 的报道，Anthropic 在 2026 年初大幅增强了 Memory 功能，包括免费用户开放、支持从 ChatGPT 导入历史数据。在 Claude Code 中，Memory 分为 user、feedback、project、reference 四种类型，覆盖从用户画像到外部资源索引的完整场景。

## Overview：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件，放在仓库根目录，随代码一起提交到 Git。它的角色是项目的「规则手册」——告诉 Claude Code 这个项目的构建命令是什么、代码风格有哪些约定、哪些操作是禁止的、工作流程是什么。

CLAUDE.md 的核心特征是**确定性**和**团队性**。同一份文件，不管谁打开 Claude Code，看到的指令完全一致。它是版本控制的，可以 code review，可以追溯变更历史。这意味着项目级的质量门槛、编码规范、部署流程都可以编码在 CLAUDE.md 中，确保团队每个成员使用 Claude Code 时行为一致。

如 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers) 所述，CLAUDE.md 是 Claude Code 可编程架构中的基础层——它定义了项目上下文，所有上层能力（Skills、Hooks、Agent Teams）都建立在这个基础之上。

## 记忆机制：详细分析

Claude Memory 和 CLAUDE.md 在「记忆」这件事上采取了截然不同的策略，理解这个差异是做出正确选型的关键。

**CLAUDE.md 是声明式的。** 你提前写好所有规则，Claude Code 每次启动时读取并严格执行。这类似于写一份 `.eslintrc`——规则是明确的、完整的、可审计的。缺点是你必须预见所有需要告诉 Claude 的事情，而且随着项目演化，CLAUDE.md 需要手动维护。

**Memory 是归纳式的。** Claude Code 在对话过程中观察你的行为和反馈，自动提取值得记住的信息。你不需要提前规划，系统会随着交互逐步建立对你的理解。缺点是记忆内容不可预测——不同的对话路径会产生不同的记忆集合，这在团队场景下是问题。

实际使用中，两者的交互模式也不同。CLAUDE.md 在每次会话开始时就被加载到上下文中，是「前置知识」。Memory 则通过 `MEMORY.md` 索引文件按需加载——Claude Code 会根据当前任务的相关性决定读取哪些记忆文件。这意味着 Memory 的影响力是概率性的，而 CLAUDE.md 的影响力是确定性的。

根据 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory) 的分析，Memory 的四种类型（user、feedback、project、reference）各自服务于不同的场景。其中 feedback 类型尤其重要——它记录你对 Claude Code 行为的纠正（「不要用 mock 测数据库」「commit message 用英文」），确保同样的错误不会在下一次对话中重复。这是 CLAUDE.md 不擅长的领域，因为这类偏好往往是个人化的、渐进发现的。

## 团队协作：详细分析

团队场景是两套系统差异最显著的地方。

**CLAUDE.md 天然适合团队。** 它是代码的一部分，PR review 时可以审查对 CLAUDE.md 的修改，新成员 clone 仓库后自动获得全部项目规则。构建命令、质量门槛、禁止操作、提交规范——这些团队级约束只应该放在 CLAUDE.md 中。

**Memory 天然适合个人。** 高级工程师可能让 Claude Code 记住「我偏好函数式风格」，初级工程师可能让它记住「给我更详细的解释」。这种个性化不应该、也不能强制统一。

问题出在灰色地带。比如「这个项目的部署流程需要先跑 staging 再上 production」——这算项目规则（CLAUDE.md）还是项目背景（Memory）？答案是：如果这是所有人都必须遵守的流程，放 CLAUDE.md；如果这只是你个人的工作备忘，放 Memory。

[Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 中提到的四层架构（Skills、Hooks、Agents、MCP）提供了一个更细粒度的框架。CLAUDE.md 是这个架构的基础层，定义「项目是什么」；Skills 定义「如何做特定任务」；Hooks 定义「什么时候触发什么」；Memory 则横跨所有层，提供个人化的上下文补充。

## 什么时候用 Claude Memory

Memory 在以下场景下发挥最大价值：

- **个人偏好沉淀**：代码风格偏好、解释详细程度、commit message 语言——这些因人而异的偏好不应该写进项目文件
- **渐进式学习**：你不确定需要告诉 Claude 什么，让它在交互中自己学习。比如你纠正了三次「不要用 any 类型」，Memory 会自动记住
- **跨项目上下文**：你的角色、技术栈偏好、常用工具——这些信息在全局 Memory 中只需记一次，所有项目受益
- **外部资源索引**：「Bug 在 Linear 的 INGEST 项目里追踪」「oncall 看 Grafana 的 api-latency 面板」——这类信息放在 Memory 的 reference 类型中最合适

如果你是独立开发者，Memory 的自动化优势更明显——不需要维护 CLAUDE.md，让系统自己积累对项目的理解。

## 什么时候用 CLAUDE.md

CLAUDE.md 在以下场景下不可替代：

- **团队规范执行**：代码风格、构建命令、质量门槛、禁止操作——任何需要团队统一执行的规则
- **可审计性要求**：你需要追溯「谁在什么时候修改了什么规则」，这只有 Git 管理的文件能做到
- **确定性行为**：你需要 Claude Code 的行为完全可预测，不因对话历史不同而产生差异
- **新成员 onboarding**：新人 clone 仓库后立即获得完整的项目上下文，不需要「和 Claude 聊几天让它了解项目」
- **CI/CD 集成**：自动化流程中运行 Claude Code 时，只有 CLAUDE.md 能提供上下文——Memory 绑定到具体用户，CI 环境中不存在

如 [Claude Code Hooks 深度掌握](/zh/blog/claude-code-hooks-mastery) 所述，CLAUDE.md 还与 Hooks 系统配合使用——你可以在 CLAUDE.md 中定义规则，然后用 Hooks 自动执行检查。这种组合在团队项目中尤其强大。

## 结论

**Claude Memory 和 CLAUDE.md 不是竞争关系，而是分层互补。** 最佳实践是：用 CLAUDE.md 定义项目的「法律」——所有人必须遵守的规则和约定；用 Memory 积累个人的「经验」——你的偏好、纠正、和工作上下文。

决策规则很简单：**如果这条信息需要团队统一执行，放 CLAUDE.md；如果只对你个人有意义，放 Memory。** 不确定的时候，默认放 CLAUDE.md——确定性和可审计性的价值在团队项目中几乎总是更高。

独立开发者可以更多依赖 Memory 的自动化能力，但即便如此，核心构建命令和质量门槛仍建议写进 CLAUDE.md——这是你给未来自己的文档。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。CLAUDE.md 负责项目级规则（构建命令、代码规范、质量门槛），Memory 负责个人级上下文（偏好、反馈纠正、外部资源引用）。两者在不同层面工作，互不冲突。Claude Code 每次启动时会同时加载两者。

### Memory 的内容会覆盖 CLAUDE.md 中的规则吗？

不会。CLAUDE.md 中的指令具有更高的优先级，因为它是显式的项目级指令。如果 Memory 中记录的偏好与 CLAUDE.md 中的规则冲突，Claude Code 会遵循 CLAUDE.md。这也是为什么团队规范应该放在 CLAUDE.md 而非依赖 Memory 的原因。

### 独立开发者还需要写 CLAUDE.md 吗？

需要。即使只有你一个人开发，CLAUDE.md 的确定性和可追溯性仍有价值。核心构建命令、测试流程、关键约束应该写在 CLAUDE.md 中。Memory 更适合处理偏好、上下文、反馈等渐进积累的信息。两者配合使用效果最好。

### 如何迁移 Memory 中的内容到 CLAUDE.md？

如果你发现 Memory 中积累了某些应该团队共享的规则，手动将其提取到 CLAUDE.md 中即可。检查 `~/.claude/projects/<project>/memory/` 目录下的文件，识别出项目级规则（而非个人偏好），然后添加到 CLAUDE.md 的相应章节。迁移后可以删除对应的 Memory 文件。
