**Claude Memory** uses a file-per-memory design. Each piece of saved context becomes its own markdown file with structured frontmatter — a `name`, `description`, `type` classification, and the memory content itself. These files live in `.claude/projects/{project-hash}/memory/` on your local machine. A central `MEMORY.md` index file contains one-line pointers to each memory file, and Claude loads this index at conversation start to decide which memories are relevant.

This architecture optimizes for incremental accumulation. Adding a new memory means creating a single file and appending one line to the index. Memories can be updated or deleted individually without affecting others. The tradeoff is discoverability — with dozens of small files, it's harder for a human to audit what Claude "knows" about them without reading through the index.

**CLAUDE.md** is a single document (or a small set of documents at different directory levels). It uses standard markdown with optional sections and follows whatever organizational structure the author chooses. Claude Code reads the entire file into its context at conversation start, treating every instruction as active for the session.

This architecture optimizes for clarity and control. You read one file and know exactly what Claude has been told. The tradeoff is maintenance — as projects grow, CLAUDE.md files can become long and unwieldy, and keeping them current requires manual discipline. Unlike Memory, nothing updates automatically when your project evolves.

The practical implication: Memory handles the long tail of context that would be impractical to manually maintain, while CLAUDE.md handles the critical instructions that must be precise and team-consistent. Claude Code's [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) layers both systems together at runtime so they complement rather than compete.

## What Each System Is Good At: Practical Analysis

The most important distinction between these systems isn't technical — it's about what kind of information belongs where. Getting this wrong leads to either instructions that drift between team members or personal context that gets lost between sessions.

### Where CLAUDE.md Excels

CLAUDE.md is the right choice for anything that should be **consistent across all developers and all sessions**. Concrete examples:

**Build and test commands.** Your `npm run build`, `npm test`, and deployment commands belong in CLAUDE.md because they're project facts, not personal preferences. When Claude needs to validate its changes, it reads CLAUDE.md for the exact commands to run — and every team member's Claude runs the same validation.

**Code style constraints.** Rules like "never use `any` in TypeScript," "all API routes must validate input with Zod," or "Chinese content uses CJK word counting" are project-level decisions. If these lived in Memory, each developer's Claude would learn them independently (if at all), and new team members would start with zero context.

**Architectural guardrails.** Instructions like "don't import Next.js modules in pipeline scripts" or "always pass three arguments to `upsertKeyword()`" prevent recurring bugs. These belong in CLAUDE.md because they protect the entire team, not just one developer.

**Workflow gates.** Quality gates — "run tests before committing," "validate pipeline changes with `validate-pipeline.ts`" — must be universal. If they lived only in Memory, a developer who hasn't triggered that memory yet would skip the gate entirely.

### Where Claude Memory Excels

Memory is the right choice for anything that's **personal, evolving, or conversation-derived**. Concrete examples:

**Your role and expertise level.** If you're a senior backend engineer who's new to the frontend, Memory stores this so Claude adjusts its explanations — more detail on React patterns, less on Go idioms. This is personal context that has no place in a shared project file.

**Approach corrections.** When you tell Claude "don't mock the database in integration tests" or "stop summarizing at the end of every response," Memory saves the correction so you never repeat it. These are personal working-style preferences that may differ between team members.

**Project context that changes frequently.** Deadlines, merge freezes, who's working on what, which features are blocked — this information changes weekly or daily. It would rot in CLAUDE.md but stays current in Memory because Claude updates it as you mention changes.

**External system pointers.** "Bugs are tracked in Linear project INGEST" or "the oncall dashboard is at grafana.internal/d/api-latency" — these are reference memories that help Claude find information but aren't prescriptive instructions.

## Team Workflows: Where the Distinction Matters Most

For solo developers, the line between Memory and CLAUDE.md is mostly about convenience — you could put everything in CLAUDE.md and it would work. But for teams, the distinction becomes critical.

**CLAUDE.md creates team alignment.** When five engineers work on the same repo, CLAUDE.md ensures everyone's Claude follows the same rules. Code style, testing requirements, architectural constraints, and deployment procedures are defined once and shared automatically through version control. When the team decides to change a convention, one PR updates CLAUDE.md and every developer's next session reflects the change.

**Memory creates individual effectiveness.** Each developer's Memory captures their personal workflow optimizations. The frontend specialist's Claude remembers they prefer Tailwind utility classes over CSS modules. The backend lead's Claude remembers they want database queries reviewed for N+1 patterns. The new hire's Claude remembers they need extra context on the authentication system because they haven't worked on it before.

**The failure mode of mixing them up** is instructive. If team conventions go in Memory instead of CLAUDE.md, they fragment — each developer's Claude learns slightly different rules based on which conversations they've had. If personal preferences go in CLAUDE.md, they clutter the shared file with instructions that only apply to one person and may contradict another team member's workflow.

A practical team setup looks like this:

1. **CLAUDE.md in repo root**: Build commands, test gates, coding standards, architectural constraints, documentation rules — everything the team agreed on
2. **Personal `~/.claude/CLAUDE.md`**: Your global defaults — response style preferences, git workflow habits, common tool configurations
3. **Auto Memory**: Accumulates naturally — your role, corrections to Claude's behavior, project timeline context, references to external tools

## Using Both Systems Together

The strongest Claude Code setup uses CLAUDE.md and Memory as complementary layers. Here's how they interact in practice.

**CLAUDE.md sets the rules; Memory refines the application.** CLAUDE.md might say "write tests before committing." Memory might record that you prefer integration tests over unit tests for database code, based on a correction you made three sessions ago. Claude follows the rule from CLAUDE.md and applies it according to your preference from Memory.

**CLAUDE.md defines what to do; Memory captures why.** CLAUDE.md might list `npm run build` as a required check. Memory might note that the build broke last week due to a specific ESM import issue, so Claude should watch for that pattern. The instruction is shared; the situational awareness is personal.

**CLAUDE.md is the floor; Memory raises the ceiling.** Every developer starts with the same baseline from CLAUDE.md. Over time, each developer's Memory accumulates context that makes Claude increasingly effective for their specific workflow. A new team member's Claude on day one follows all the right conventions from CLAUDE.md. By month three, their Memory has captured enough context to make Claude a genuinely personalized assistant.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary configuration tool when:

- **You work on a team** and need consistent AI behavior across all developers. CLAUDE.md is the only persistence mechanism that travels with the repository and is version-controlled.
- **You have non-negotiable quality gates** — test requirements, build checks, lint rules — that must run every time, for every developer, without exception.
- **You're defining project architecture** and want Claude to respect constraints like "don't import server-only modules in client code" or "all API endpoints must use the shared validation middleware."
- **You're onboarding new team members** and want their Claude Code sessions to immediately understand the project's conventions, structure, and workflow requirements.
- **You need auditability** — CLAUDE.md changes show up in git history, making it clear who changed what instruction and when.

The investment is manual authoring and maintenance. You need to write CLAUDE.md, keep it updated as the project evolves, and review it periodically for stale instructions. For many teams, this investment pays for itself immediately in reduced "why did Claude do that?" conversations.

## When to Choose Claude Memory

Choose Memory as your primary context mechanism when:

- **You're a solo developer** and want Claude to learn your preferences organically without manually configuring a file. Memory accumulates from normal conversation.
- **Your context is personal** — your expertise areas, your preferred coding patterns, your working style, tools you use outside the codebase.
- **Information changes frequently** — project timelines, who's working on what, current blockers, temporary decisions. Memory updates naturally as you mention changes in conversation.
- **You want zero-maintenance persistence** — Memory requires no manual file editing. Claude writes and updates memories automatically based on what you tell it.
- **You've given Claude specific corrections** and want them to stick across sessions. "Don't use mocks for database tests" or "always suggest the simplest approach first" are classic Memory candidates.

The tradeoff is that Memory is per-user and not shared. If you discover an important project pattern through Memory, consider promoting it to CLAUDE.md so the whole team benefits.

**Use both.** CLAUDE.md and Claude Memory aren't competing solutions — they're complementary layers in Claude Code's persistence architecture. **CLAUDE.md is essential for teams** and for any project where consistency, quality gates, and shared conventions matter. **Memory is essential for individual productivity** and for the personal context that makes Claude feel like a teammate who knows your working style rather than a fresh assistant every session.

If you must choose one to invest in first, **start with CLAUDE.md**. It has the highest immediate impact — write your build commands, coding standards, and key constraints, and every session immediately improves. Memory will accumulate naturally as you work. Over time, the combination of deliberate project instructions and organic personal context creates an AI coding experience that improves with every session.

## Frequently Asked Questions

### Can CLAUDE.md and Claude Memory conflict with each other?

In practice, they rarely conflict because they store different types of information. CLAUDE.md holds project instructions (build commands, coding standards), while Memory holds personal context (your role, preferences, corrections). If a conflict arises — say Memory records a preference that contradicts a CLAUDE.md rule — CLAUDE.md takes precedence because it represents the team's agreed-upon standards.

### Does Claude Memory sync across devices?

No. Claude Memory is stored locally in your `.claude/projects/` directory on the machine where you run Claude Code. If you work from multiple machines, each one accumulates its own memory independently. CLAUDE.md, by contrast, syncs automatically through git — another reason to put critical project instructions there rather than relying on Memory.

### How do I see what Claude has saved in Memory?

Check the `.claude/projects/{project-hash}/memory/` directory on your local machine. The `MEMORY.md` index file lists all saved memories with one-line descriptions. Each memory is a separate markdown file you can read, edit, or delete. You can also ask Claude directly: "What do you remember about me?" or "What memories do you have for this project?"

### Should I put CLAUDE.md instructions in Memory instead to avoid cluttering the repo?

No. CLAUDE.md belongs in the repository because it needs to be version-controlled and shared. A CLAUDE.md file in your repo root is no more "clutter" than a `.eslintrc` or `tsconfig.json` — it's project configuration that defines how an important tool behaves. Keeping it in version control means changes are tracked, reviewable, and consistent across the team.

### How often should I update CLAUDE.md?

Update CLAUDE.md whenever project conventions, build processes, or architectural constraints change. A good rule of thumb: if you find yourself correcting Claude about the same thing twice, that correction belongs in CLAUDE.md rather than just Memory. Many teams review CLAUDE.md quarterly or when onboarding new team members to ensure instructions are current and complete.
