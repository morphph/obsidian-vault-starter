# Claude Memory vs CLAUDE.md：两套记忆系统，到底该用哪个？

**TL;DR：** **Claude Memory** 是跨会话的自动记忆系统，记住你是谁、你喜欢什么；**CLAUDE.md** 是项目级的规则手册，告诉 Claude 这个项目怎么做。两者解决的问题完全不同——Memory 管"人"，CLAUDE.md 管"事"。大多数团队应该**同时使用两者**，但如果只能选一个，工程团队优先 CLAUDE.md，个人用户优先 Memory。

## 概览：Claude Memory

**Claude Memory** 是 Anthropic 为 Claude 打造的持久化记忆功能，让 AI 在多次对话之间记住关于你的信息。它解决的核心问题是：每次开新对话都要重新自我介绍。Memory 会自动从对话中提取关键信息——你的角色、偏好、技术栈、工作习惯——并在后续对话中主动应用。

在 [Claude Code](/zh/glossary/agentic-coding) 场景下，Memory 存储在 `~/.claude/` 目录中，以 markdown 文件形式组织。它包含用户画像（user memory）、行为反馈（feedback memory）、项目状态（project memory）和外部资源引用（reference memory）四种类型。Anthropic 在 2026 年初对 Memory 系统做了[重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，开放了免费用户使用，并支持从 ChatGPT 等平台导入记忆数据。

Memory 的定位是**个人助理**——它了解你，而不是了解你的项目。

## 概览：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级配置文件，放在仓库根目录，跟代码一起提交到 Git。它的作用是为 Claude 提供项目上下文：技术栈是什么、怎么跑测试、代码风格有什么要求、哪些坑要避开。

CLAUDE.md 解决的问题和 Memory 截然不同：它不关心"你是谁"，只关心"这个项目怎么做"。当团队成员共享同一个仓库时，每个人的 Claude 实例都会读取同一份 CLAUDE.md，确保 AI 的行为在团队内保持一致。这是 Memory 做不到的——Memory 是个人的，CLAUDE.md 是团队的。

配合 [SKILL.md 技能文件](/zh/blog/9-principles-writing-claude-code-skills)，CLAUDE.md 构成了一套完整的 AI 行为控制体系。详细的工作机制可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **持久化范围** | 跨项目、跨会话 | 单项目内 | Memory |
| **存储内容** | 用户偏好、反馈、项目状态 | 构建命令、代码规范、架构决策 | 各有所长 |
| **团队共享** | 不支持（个人专属） | 支持（Git 提交） | CLAUDE.md |
| **自动学习** | 自动从对话中提取 | 需要手动编写和维护 | Memory |
| **版本控制** | 无版本历史 | 完整 Git 历史 | CLAUDE.md |
| **覆盖粒度** | 全局 + 项目级 | 项目级 + 目录级 | CLAUDE.md |
| **内容可审计** | 个人可查看和删除 | 团队可 review 和 PR | CLAUDE.md |
| **上手成本** | 零配置，自动工作 | 需要编写和迭代 | Memory |

## 记忆机制：深度对比

Claude Memory 和 CLAUDE.md 的底层运作方式完全不同，理解这一点是选对工具的关键。

**Claude Memory 是隐式的。** 它在对话过程中悄悄工作——当你说"我是后端工程师，主要写 Go"，Memory 会自动提取并保存这条信息。下次你问一个架构问题，Claude 会自动用 Go 的视角来回答，而不是给你一个 Python 示例。这种"无感"体验很好，但也意味着你无法精确控制 Claude 记住了什么、遗忘了什么。

**CLAUDE.md 是显式的。** 每一条规则都是你（或团队）刻意写下的。`npm run build` 必须在 commit 前通过、中文内容必须用 CJK 字符计数、不要在 pipeline 脚本里 import Next.js 模块——这些规则是经过讨论、验证、甚至踩坑之后沉淀下来的。它们不会被 AI "遗忘"，因为每次会话开始时 Claude Code 都会重新读取。

从[七层架构](/zh/blog/claude-code-seven-programmable-layers)的角度看，CLAUDE.md 是最底层的"系统指令"层，而 Memory 工作在更上层的"用户上下文"层。两者的优先级也不同：当 Memory 中的偏好与 CLAUDE.md 中的规则冲突时，CLAUDE.md 优先。

## 团队协作：谁更适合多人场景

这是两者差异最大的地方，也是选型的关键决策点。

**CLAUDE.md 天生为团队设计。** 它跟代码一起进 Git，走 PR review 流程，有完整的变更历史。新成员 clone 仓库的那一刻，就自动获得了团队积累的所有 AI 使用规范。这意味着不管是谁在用 Claude Code，AI 的行为都是一致的——同样的构建命令、同样的代码风格、同样的质量门禁。

**Claude Memory 是私人的。** 这不是缺点——它就应该是私人的。你喜欢 terse 的回答风格、你对 React 不熟但精通 Go、你上周被一个数据库迁移坑了——这些信息帮助 Claude 更好地服务你个人，但不应该强加给团队其他成员。

实际工程中的最佳实践是：**用 CLAUDE.md 管项目规范，用 Memory 管个人偏好**。一个团队的 CLAUDE.md 通常包含构建命令、测试流程、代码风格、架构约束等"项目知识"；而每个成员的 Memory 则记录各自的技术背景、沟通偏好和工作习惯。

详细的 Claude Code 扩展体系可以参考[扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 实战搭配：如何同时用好两者

理解了区别之后，关键问题是：怎么搭配使用？

**第一步：写好 CLAUDE.md。** 这是基础。把你的项目最关键的信息写进去——技术栈、构建命令、质量门禁、已知的坑。不需要写得很长，但要准确。参考我们整理的 [Claude Code 实用技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)来建立初始配置。

**第二步：让 Memory 自然积累。** 不需要刻意"训练" Memory。正常使用 Claude Code 的过程中，它会自动学习你的习惯。当你纠正 Claude 的行为（"不要在这类测试里 mock 数据库"），Memory 会记住这条反馈。当你确认某个方案可行（"对，就这样做"），Memory 也会记录这个正向信号。

**第三步：定期审查和清理。** Memory 会积累过时信息——三个月前你可能主要写 Python，现在已经切到 TypeScript 了。定期检查 `~/.claude/` 下的 memory 文件，删除不再准确的条目。CLAUDE.md 也需要维护：当项目架构变化时，及时更新对应的规则。

**第四步：用 Skills 补充结构化知识。** 对于特定任务（写测试、做 code review、生成内容），用 [SKILL.md 文件](/zh/blog/9-principles-writing-claude-code-skills)来编码最佳实践。Skills 比 CLAUDE.md 更细粒度，比 Memory 更可控。

## 什么时候选 Claude Memory

Memory 在以下场景中价值最大：

- **个人开发者跨项目工作**：你在多个项目间切换，但技术偏好和沟通风格是一致的。Memory 让你不需要在每个项目的 CLAUDE.md 里重复写个人偏好。
- **快速原型和探索**：在新项目初期，还没有成型的规范需要写进 CLAUDE.md，让 Memory 先记住你的决策偏好。
- **非编码场景**：在 Claude.ai 上做研究、写文档、头脑风暴——这些场景没有 CLAUDE.md，Memory 是唯一的上下文持久化手段。
- **从其他工具迁移**：Anthropic 支持从 ChatGPT [导入记忆数据](/zh/blog/anthropic-claude-memory-upgrades-importing)，降低了切换成本。

## 什么时候选 CLAUDE.md

CLAUDE.md 在以下场景中不可替代：

- **团队项目**：任何超过一个人的项目，CLAUDE.md 是确保 AI 行为一致性的唯一可靠方式。
- **质量门禁**：构建必须通过、测试不能跳过、lint 规则必须遵守——这些硬性约束必须写在 CLAUDE.md 里，不能依赖 Memory 的"软记忆"。
- **新人入职**：CLAUDE.md 是最好的项目上下文文档之一。新成员不需要问"怎么跑测试"、"commit 前要做什么"——CLAUDE.md 已经回答了。
- **可重现性**：当你需要确保 Claude 在任何时候、对任何人都表现一致时，CLAUDE.md 的显式声明远比 Memory 的隐式学习可靠。

## 结论

**Claude Memory 和 CLAUDE.md 不是竞争关系，而是互补关系。** Memory 负责"Claude 了解你"，CLAUDE.md 负责"Claude 了解项目"。工程团队的正确做法是两者并用：把项目规范、质量门禁和架构决策写进 CLAUDE.md，让 Memory 自动学习每个成员的个人偏好和工作习惯。

如果你还没有开始用 CLAUDE.md，从今天开始——这是投入产出比最高的 Claude Code 配置。如果你已经在用 CLAUDE.md 但还没关注 Memory，它已经在默默工作了，偶尔检查和清理一下就好。想要更深入地了解整个体系，推荐阅读我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 常见问题

### Claude Memory 的数据存在哪里？

Claude Memory 在 Claude Code 中以 markdown 文件形式存储在用户本地的 `~/.claude/` 目录下。在 Claude.ai 网页端，记忆数据存储在 Anthropic 的服务器上。两者的记忆是独立的，不会自动同步。用户可以随时查看、编辑或删除自己的记忆条目。

### CLAUDE.md 支持子目录级别的配置吗？

支持。Claude Code 会读取项目根目录的 CLAUDE.md，也会读取当前工作目录路径上的所有 CLAUDE.md 文件。这意味着你可以在 `src/frontend/CLAUDE.md` 中写前端特定的规则，在 `src/backend/CLAUDE.md` 中写后端特定的规则，它们会与根目录的全局规则合并生效。

### 两者冲突时以谁为准？

CLAUDE.md 优先。CLAUDE.md 中的规则被视为项目的"硬约束"，而 Memory 中的偏好被视为"软偏好"。例如，如果你的 Memory 记录了"我喜欢简短的 commit message"，但 CLAUDE.md 要求"commit message 必须包含改动原因"，Claude 会遵循 CLAUDE.md 的要求。
