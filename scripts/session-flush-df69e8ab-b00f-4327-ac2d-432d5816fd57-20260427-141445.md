**CLAUDE.md loads first and loads fully.** When Claude Code starts a session, it reads every CLAUDE.md file in the hierarchy — user-level (`~/.claude/CLAUDE.md`), project-level (`./CLAUDE.md`), and any directory-level variants. These contents become part of the system prompt, meaning Claude Code treats them as authoritative instructions with the highest priority. If your CLAUDE.md says "run `npm test` before every commit," Claude Code will do that every single time, in every conversation, for every team member. There's no ambiguity, no drift, no interpretation.

**Claude Memory loads second and loads selectively.** After CLAUDE.md, Claude Code reads the `MEMORY.md` index file and loads relevant memories based on their descriptions. The agent decides which memories are relevant to the current conversation — a memory about your database preferences might not load during a frontend styling task. This selective loading is efficient but introduces a layer of agent judgment that CLAUDE.md doesn't have.

The practical implication: **put anything that must always be followed in CLAUDE.md, and anything that's useful-but-contextual in Memory.** Build commands, quality gates, and architectural constraints belong in CLAUDE.md because skipping them even once causes problems. Your role ("I'm a backend engineer new to React"), your preferences ("I prefer explicit error types over string errors"), and project context ("we're in a code freeze until March 5") belong in Memory because they're personal, temporal, or situational.

This layered architecture is part of what makes Claude Code's [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) powerful — each layer serves a different persistence and scope need.

## Team Collaboration: Shared Standards vs Personal Context

The sharpest difference between Claude Memory and CLAUDE.md emerges in team settings. This is where choosing the wrong system creates real friction.

**CLAUDE.md is the team's single source of truth for AI behavior.** When you commit a CLAUDE.md file to your repository, every developer who uses Claude Code on that project gets identical instructions. This solves a critical problem in AI-assisted development: consistency. Without shared instructions, one developer's Claude Code might use `console.log` for debugging while another's uses a structured logger. One might commit directly to main while another creates feature branches. CLAUDE.md eliminates this variance.

Effective team CLAUDE.md files typically include:

- **Build and test commands** — so Claude Code never guesses how to validate changes
- **Code style rules** — naming conventions, import ordering, error handling patterns
- **Architecture boundaries** — which modules can import from which, where business logic lives
- **Workflow requirements** — branch naming, commit message format, required checks before push
- **Known gotchas** — project-specific pitfalls that aren't obvious from the code

**Claude Memory is inherently personal.** Your memories about your role, your expertise gaps, and your feedback to Claude Code should not be shared with the team. A senior backend engineer and a junior frontend developer working on the same repo need different explanations, different levels of hand-holding, and different assumptions about prior knowledge. Memory handles this naturally — each developer's memory store reflects their individual interactions.

The anti-pattern to avoid: **putting personal preferences in CLAUDE.md or team standards in Memory.** If you put "I prefer verbose commit messages" in CLAUDE.md, your entire team gets verbose commit messages whether they want them or not. If you put "always run `npm test` before committing" only in your Memory, new team members' Claude Code sessions won't know to run tests. Match the scope of the instruction to the scope of the system.

For teams adopting Claude Code, our [complete guide](/blog/claude-code-complete-guide) covers how to structure CLAUDE.md for maximum team benefit, including patterns for monorepos and multi-team projects.

## Maintenance and Staleness: The Long-Term Tradeoff

Both systems can become outdated, but they rot in different ways and at different rates. Understanding this helps you decide where to invest maintenance effort.

**CLAUDE.md gets stale when the code changes but the instructions don't.** If you add a new build step but forget to update CLAUDE.md, Claude Code will skip it. If you rename a module but leave the old name in your architecture constraints, Claude Code gets confused. The fix is straightforward: treat CLAUDE.md like documentation that's part of your code review process. When someone changes a build command or an architecture boundary, the CLAUDE.md update should be part of the same PR.

Well-maintained CLAUDE.md files include a documentation update checklist — "when modifying pipeline scripts, update PIPELINE.md; when changing database schema, update SYSTEM-OVERVIEW.md." This makes staleness a reviewable failure rather than a silent one.

**Claude Memory gets stale when project state changes between conversations.** A memory that says "we're in a code freeze until March 5" is useful on March 3 and actively misleading on March 10. A memory that says "the auth service uses JWT tokens" becomes wrong after a migration to session-based auth. Claude Code is designed to verify memories against current state before acting on them, but this verification isn't foolproof.

Memory staleness is harder to detect because memories are distributed across individual files without a review process. The mitigation: Claude Code's memory system includes guidelines to avoid saving information that's easily derived from code or git history. Architectural facts belong in CLAUDE.md (where they're reviewed); only non-obvious context — the *why* behind decisions, personal preferences, external system references — belongs in Memory.

The practical rule: **if the information has a shelf life shorter than a sprint, think twice before saving it as a memory. If it has a shelf life longer than a quarter, put it in CLAUDE.md.**

## Real-World Patterns: How Experienced Teams Use Both

Teams that get the most out of Claude Code's context systems follow a consistent pattern. Here's what the division typically looks like in practice.

**In CLAUDE.md (shared, deterministic, high-stakes):**

```markdown
# Build & Validate
npm run build          # Must pass before any commit
npm test               # All tests must pass
npm run lint           # No lint violations

# Architecture Rules
- Never import server-only modules in client components
- All API routes must validate input with zod schemas
- Database queries go through the repository layer, never direct SQL

# Commit Standards
- Conventional commits: feat|fix|refactor|docs|test(scope): description
- Every PR needs at least one test covering the change
```

**In Claude Memory (personal, contextual, low-stakes):**

```markdown
# User Memory: Senior backend engineer, new to React
# Feedback Memory: Prefers single bundled PR for refactors, not many small ones
# Project Memory: Auth migration to session-based planned for Q2, don't invest in JWT tooling
# Reference Memory: Pipeline bugs tracked in Linear project "INGEST"
```

This split follows a clear principle: **CLAUDE.md answers "what are the rules?" and Memory answers "who am I working with and what's the current situation?"**

The [skills system](/blog/5-claude-code-skills-i-use-every-single-day) adds a third layer — reusable instruction sets for specific tasks like writing tests, generating content, or reviewing PRs. Skills complement both CLAUDE.md and Memory by providing task-specific context that's too detailed for project-level instructions but too structured for conversational memory. If you're investing in Claude Code's context stack, [learning to write effective skills](/blog/9-principles-writing-claude-code-skills) is the natural next step after setting up CLAUDE.md and Memory.

## When to Invest in CLAUDE.md

Choose CLAUDE.md as your primary context investment when:

- **You're working on a team** and need consistent AI behavior across developers. CLAUDE.md is the only context system that travels with the repo and applies to everyone.
- **You have non-negotiable quality gates** — build commands, test requirements, lint rules — that must run every time, without exception. CLAUDE.md's deterministic loading guarantees this.
- **Your project has architectural constraints** that aren't obvious from the code structure. "Don't import X from Y" rules prevent subtle bugs that Claude Code can't infer from reading the codebase.
- **You're onboarding new team members** who will use Claude Code. A well-written CLAUDE.md means their first Claude Code session follows the same standards as a veteran's.
- **You want reviewable AI instructions.** CLAUDE.md changes show up in PRs, which means team leads can review and approve changes to how AI behaves on the project.

The investment is writing and maintaining the file. Most teams find that a 50-100 line CLAUDE.md covers 90% of their needs, with periodic updates during architecture changes.

## When to Invest in Claude Memory

Choose Claude Memory as your primary context investment when:

- **You're a solo developer** and don't need team-shared instructions. Memory captures your preferences and context automatically, reducing the need to maintain a separate instruction file.
- **Your workflow involves repeated corrections.** If you keep telling Claude Code "don't use mocks in integration tests" or "use explicit return types," saving that as a feedback memory means you only say it once.
- **You work across multiple projects** with different contexts. Memory is project-scoped, so your context for Project A doesn't leak into Project B.
- **You need to track non-code context** — who's working on what, upcoming deadlines, external system locations — that doesn't belong in a committed file.
- **You prefer low-maintenance personalization.** Memory accumulates naturally from conversations. You don't have to stop and write an instruction file.

The investment is minimal — you just work normally and occasionally tell Claude Code to remember something important. The tradeoff is less control over exactly what gets saved and how it's applied.

## Verdict

**Use both — they solve different problems.** CLAUDE.md is your project's constitution: explicit rules that every Claude Code session must follow, shared across the team, reviewed in PRs. Claude Memory is your personal assistant's notebook: accumulated knowledge about you, your preferences, and the current state of work that makes each session more efficient.

If you're forced to choose one, **start with CLAUDE.md**. The deterministic, shared instructions have higher impact per line written, especially on teams. You can [prompt Claude Code effectively](/blog/how-to-effectively-prompt-a-claude-code) without Memory, but you'll waste significant time without CLAUDE.md repeating build commands and coding standards every session.

Add Memory once your CLAUDE.md is solid and you want to reduce repetitive corrections. The two systems together create a context stack where CLAUDE.md handles the "what" and Memory handles the "who" and "why" — the combination that makes [agentic coding](/glossary/agentic-coding) feel like working with a colleague who knows both the project and you.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions are loaded as part of the system prompt and treated as authoritative rules. Claude Memory provides supplementary context that informs how Claude Code works within those rules. If CLAUDE.md says "always run tests before committing," a memory preference cannot override that. Memory might influence *how* Claude Code explains test failures to you, but it won't skip the tests.

### Do I need to manually write Claude Memory files?

No. Claude Code creates and maintains memory files automatically based on your conversations. When you correct its approach, share your role or preferences, or mention project context, Claude Code decides whether to save that as a memory. You can also explicitly ask it to remember something. The memory files are plain markdown, so you *can* edit them manually, but it's not required.

### How do I migrate from Memory-only to CLAUDE.md?

Review your memory files in `~/.claude/projects/<project>/memory/` and identify anything that should apply to all team members — coding standards, build commands, architecture rules, workflow requirements. Move those into a CLAUDE.md file at your project root and commit it. Keep personal preferences, role context, and temporal project state in Memory. This typically takes 15-30 minutes and immediately improves consistency across the team.

### Does CLAUDE.md work without Claude Memory enabled?

Yes. CLAUDE.md is completely independent of the memory system. It's loaded from the filesystem at session start regardless of whether memory is active. Many developers use CLAUDE.md as their only context mechanism and get strong results, especially for well-defined projects with clear standards.

### What happens when Claude Memory contradicts current code state?

Claude Code is designed to verify memories against the current codebase before acting on them. If a memory says "the auth service uses JWT" but the code shows session-based auth, Claude Code should trust the code. In practice, you should periodically review and clean up outdated memories — especially project-type memories that reference specific timelines or technical decisions that may have changed.
