**Claude Memory stays on your machine.** It's stored outside the repository in `~/.claude/projects/`, tied to your local environment. This makes it inherently personal — your feedback corrections, role context, and workflow preferences don't leak to teammates. This is intentional: your preference for terse responses shouldn't override a teammate's preference for detailed explanations.

The scoping difference creates a natural division of responsibility. CLAUDE.md handles **project truth** — facts and rules that are objectively correct for the codebase. Memory handles **personal context** — subjective preferences and accumulated knowledge that help Claude work better with *you specifically*. A senior backend engineer and a junior frontend developer working on the same repo should have identical CLAUDE.md files but very different Memory stores, because Claude should explain React hooks differently to each of them.

One practical consequence: if you discover something that should be a team-wide rule, don't just tell Claude to remember it. Add it to CLAUDE.md so everyone benefits. Memory is the wrong place for team standards because it doesn't propagate. The [seven programmable layers](/blog/claude-code-seven-programmable-layers) article maps out exactly where each type of configuration belongs in the stack.

## Content Structure: Rules vs Rich Context

**CLAUDE.md is flat and imperative.** It reads like a configuration file with markdown formatting — build commands, bullet-pointed rules, short explanations of constraints. The format favors brevity because Claude Code reads the entire file every session, and longer files consume context window. A well-written CLAUDE.md is 50–200 lines covering the essentials: what to run, what to avoid, how to validate.

```markdown
## Commands
npm run build    # Production build
npm test         # Must pass before commit

## Gotchas
- Never import Next.js modules in pipeline scripts
- ZH content uses CJK word count, not English tokenization
```

**Claude Memory is structured and explanatory.** Each memory entry is a standalone markdown file with typed frontmatter (`user`, `feedback`, `project`, `reference`) and a body that captures not just the rule but the reasoning. Feedback memories, for example, follow a `rule → Why → How to apply` structure:

```markdown
---
name: no-db-mocks
description: Integration tests must hit a real database, not mocks
metadata:
  type: feedback
---
Integration tests must hit a real database, not mocks.

**Why:** Prior incident where mock/prod divergence masked a broken migration.
**How to apply:** When generating or modifying tests for database operations, always use the real SQLite connection, never jest.mock() the db module.
```

This structural difference reflects their purposes. CLAUDE.md needs to be **scannable and authoritative** — Claude Code should parse rules quickly without ambiguity. Memory needs to be **contextual and judgment-enabling** — Claude Code should understand *why* a preference exists so it can apply it intelligently to novel situations. The "no database mocks" rule in CLAUDE.md would be a one-liner. In Memory, the incident context helps Claude decide whether the rule applies to a new edge case.

Memory entries also link to each other using `[[name]]` references, building a knowledge graph over time. CLAUDE.md has no linking mechanism — it's a standalone document. This means complex, interconnected context (like the relationship between a team's testing philosophy, their CI pipeline, and a past production incident) is better captured in Memory's linked entries than in CLAUDE.md's flat rules.

## Maintenance and Lifecycle

**CLAUDE.md requires active maintenance.** When your build commands change, your testing strategy evolves, or your team adopts new conventions, someone needs to update CLAUDE.md. Stale instructions are worse than no instructions — they actively mislead the AI. Teams that treat CLAUDE.md as a living document (updating it in the same PR that changes the behavior it describes) get the most value. Teams that write it once and forget it accumulate drift.

**Claude Memory is self-maintaining but needs occasional pruning.** Claude Code creates, updates, and removes memory entries as conversations reveal new information or invalidate old knowledge. If you change roles from backend to full-stack, telling Claude updates the relevant user memory. If a project deadline passes, the project memory becomes stale — Claude Code is supposed to detect this, but in practice, old memories can linger.

The maintenance tradeoff is real. CLAUDE.md demands discipline but gives you full control — you see exactly what instructions the AI follows, and you can review changes in pull requests. Memory is low-effort but less transparent — you might not realize what Claude "knows" about you until it surfaces in an unexpected way. Periodically reviewing your `MEMORY.md` index and pruning outdated entries is good practice. Anthropic's [memory upgrades](/blog/anthropic-claude-memory-upgrades-importing) have improved the system's ability to handle stale context, but human review remains valuable.

**Staleness risk differs by type.** CLAUDE.md rules about build commands and file paths go stale when code changes — but the code change is usually obvious. Memory entries about project timelines and team decisions go stale silently — there's no diff to trigger an update. This is why Memory includes a guideline to verify remembered facts against current state before acting on them.

## Use Case Deep Dive: When Each System Shines

### CLAUDE.md excels at guardrails and standards

The strongest use case for CLAUDE.md is **preventing known mistakes**. If your team has learned (perhaps painfully) that certain patterns cause bugs, CLAUDE.md encodes those lessons as hard rules:

- "Never use `git add -A` — always stage specific files"
- "Run `validate-pipeline.ts` before committing any pipeline changes"
- "Do not skip failing tests or comment out lint rules"

These rules are non-negotiable and apply to everyone. They don't need context or reasoning in the file — they need to be followed every time. CLAUDE.md's deterministic, team-wide nature makes it perfect for this. You can read more about how teams like Shopify and Spotify operationalize these patterns in our [enterprise engineering coverage](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify).

### Memory excels at personalization and accumulated context

Memory shines when Claude needs to adapt to **how you specifically work**. A developer who says "stop summarizing what you did at the end of every response — I can read the diff" shouldn't have to repeat that instruction in every session. A team lead who mentions "we're freezing non-critical merges after Thursday for the mobile release" needs Claude to factor that into suggestions for the rest of the week.

These are inherently personal or temporal — they don't belong in a team-wide config file. Memory captures them once and applies them going forward, making each subsequent conversation more efficient.

### Combined: the full context stack

The most effective setup uses both systems together. CLAUDE.md provides the **baseline** — project structure, build commands, coding standards, quality gates. Memory provides the **personalization layer** — your role, preferences, recent corrections, and awareness of ongoing work.

When Claude Code starts a session, it loads both: CLAUDE.md tells it *what the project requires*, and Memory tells it *who you are and how you've asked it to work*. A new team member gets the full CLAUDE.md baseline immediately and builds up Memory over their first few sessions. This is the model described in our guide to [what makes Claude Code different](/blog/whats-so-special-about-the-claude-code) from simpler AI coding tools.

## When to Choose CLAUDE.md

**Use CLAUDE.md when the information is:**

- **Team-relevant**: Build commands, testing requirements, architecture decisions
- **Stable**: Rules that change infrequently (quarterly, not daily)
- **Authoritative**: Things that should be true for every developer, every session
- **Reviewable**: Instructions you want visible in pull request diffs
- **Prescriptive**: Clear rules with no ambiguity ("always run X before Y")

**Concrete scenarios:**
1. You're setting up a new project and want Claude Code to know the stack, commands, and conventions from the first session
2. A team member keeps getting Claude to use a deprecated pattern — add a rule to CLAUDE.md to prevent it for everyone
3. You need Claude to follow specific editorial guidelines (like the SEO writing standards or newsletter tone in this project's CLAUDE.md)

## When to Choose Claude Memory

**Use Memory when the information is:**

- **Personal**: Your role, expertise level, communication preferences
- **Temporal**: Project deadlines, ongoing incidents, current sprint context
- **Corrective**: Feedback on Claude's behavior that you don't want to repeat
- **External**: Pointers to tools and dashboards outside the codebase
- **Nuanced**: Context that requires reasoning about *why*, not just *what*

**Concrete scenarios:**
1. You've corrected Claude three times about your testing preferences — let Memory capture it so you don't need a fourth time
2. You're tracking a production incident and want Claude to be aware of the ongoing context across multiple debugging sessions
3. You use a specific Grafana dashboard for latency monitoring — Memory stores the URL so Claude can reference it when you're editing request-handling code

## Common Mistakes

**Mistake 1: Putting personal preferences in CLAUDE.md.** Your preference for terse responses or your role as a data scientist doesn't belong in a team-wide config file. That's Memory territory.

**Mistake 2: Relying on Memory for team rules.** If "always run tests before commit" only exists in your Memory, your teammates' Claude Code sessions won't know about it. Put it in CLAUDE.md.

**Mistake 3: Duplicating content across both systems.** If a rule is in CLAUDE.md, Memory shouldn't also store it. Memory explicitly excludes "anything already documented in CLAUDE.md files" to avoid contradictions.

**Mistake 4: Never reviewing Memory.** Auto-generated content drifts. Check your `MEMORY.md` index periodically and remove entries that reference deleted files, completed projects, or outdated deadlines.

**Mistake 5: Writing a 500-line CLAUDE.md.** Every line consumes context window. If your CLAUDE.md is longer than 200 lines, move detailed guidelines into `skills/` files (which are loaded on-demand) and keep CLAUDE.md focused on universal rules and commands. The [skills guide](/blog/5-claude-code-skills-i-use-every-single-day) covers this pattern in detail.

## Verdict

**CLAUDE.md is the foundation; Claude Memory is the personalization layer.** Every project using Claude Code should have a CLAUDE.md — it's the minimum viable configuration that makes AI assistance consistent and reliable across your team. Memory builds on top of that foundation, making Claude progressively better at working with you specifically.

If you're starting from zero, **write CLAUDE.md first**. Add your build commands, test requirements, and top three "don't do this" rules. That alone dramatically improves Claude Code's output. Memory will accumulate naturally as you work — you don't need to set it up or think about it.

If you're already using CLAUDE.md but find yourself repeating the same corrections across sessions, **that's the signal to let Memory work**. The correction you give once should stick. If it doesn't, check that Memory is enabled and that the feedback is being captured.

For teams evaluating [agentic coding](/glossary/agentic-coding) tools, the CLAUDE.md + Memory architecture is a meaningful differentiator. It separates team configuration from individual adaptation in a way that scales — you get consistent project behavior without forcing every developer into the same interaction style. ## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md instructions take precedence because they represent deliberate, team-approved configuration. If a Memory entry conflicts with a CLAUDE.md rule, Claude Code follows CLAUDE.md. Memory is designed to supplement, not override — it fills gaps that CLAUDE.md doesn't address, like personal preferences and temporal project context.

### Does Claude Memory sync across devices?

Claude Memory is stored locally in `~/.claude/projects/` and does not sync across machines. If you work on multiple devices, each builds its own Memory independently. CLAUDE.md, by contrast, syncs automatically through git. Anthropic has been working on [memory portability features](/blog/anthropic-claude-memory-upgrades-importing), but cross-device sync for Claude Code's auto memory is not yet available.

### How do I see what Claude has saved in Memory?

Check the `MEMORY.md` index file in your project's `.claude/projects/` directory. It lists all memory entries with one-line descriptions. Each entry links to a standalone markdown file you can read, edit, or delete. You can also ask Claude Code directly: "what do you remember about my preferences?" and it will surface relevant memories.

### Should I check CLAUDE.md into version control?

Yes — that's the primary design intent. CLAUDE.md belongs in your repository root so every team member and CI environment gets the same instructions. Treat it like `.editorconfig` or `tsconfig.json`: project configuration that travels with the code. Some teams also add a global `~/.claude/CLAUDE.md` for personal cross-project preferences, but that file stays local.

### How often should I update CLAUDE.md?

Update CLAUDE.md in the same pull request that changes the behavior it documents. If you change your build command, update CLAUDE.md in that commit. If you add a new quality gate, add the rule immediately. Stale CLAUDE.md instructions actively mislead Claude Code, so treating it as a living document is essential. Most active projects update it a few times per month.
