# Claude Memory vs CLAUDE.md：两套持久化系统到底该怎么选？

**TL;DR：** **Claude Memory**（Auto Memory）是 Claude Code 的自动笔记本，跨会话积累你的个人偏好、项目背景和反馈修正，存储在 `.claude/projects/` 下，不进版本控制。**CLAUDE.md** 是手动编写的项目指令文件，定义编码规范、工作流约束和质量门禁，随代码库一起提交，团队共享。两者不是竞争关系，而是互补层——CLAUDE.md 管「规则」，Memory 管「经验」。

## 概述：Claude Memory

**Claude Memory** 是 Claude Code 内置的自动持久化记忆系统，解决的核心问题是：每次开启新会话，AI 都从零开始。Memory 让 Claude Code 在多次对话间保留对你的了解——你的角色、技术偏好、项目背景、过去的反馈修正。

Memory 存储为 `.claude/projects/<project-hash>/memory/` 目录下的 Markdown 文件，每条记忆包含 frontmatter（名称、描述、类型）和正文内容。一个 `MEMORY.md` 索引文件汇总所有条目。Memory 分为四种类型：

- **user**：用户角色和偏好（如「资深后端工程师，React 新手」）
- **feedback**：行为修正（如「不要在这类测试里 mock 数据库」）
- **project**：项目动态（如「周四后冻结非关键合并」）
- **reference**：外部资源指引（如「Bug 追踪在 Linear 的 INGEST 项目」）

Memory 的关键特征是**自动积累**。你不需要刻意维护它——Claude Code 在对话中观察到有价值的信息时会主动保存。详细机制可参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是放置在项目根目录（或 `~/.claude/CLAUDE.md` 全局路径）的指令文件，相当于给 Claude Code 写的「工程手册」。每次会话启动时，Claude Code 自动加载这些文件作为系统级上下文。

CLAUDE.md 的内容通常包括：

- 构建和测试命令（`npm run build`、`npm test`）
- 质量门禁规则（提交前必须通过哪些检查）
- 编码风格约定（命名规范、注释策略）
- 项目架构说明（技术栈、关键目录结构）
- 已知陷阱和注意事项

CLAUDE.md 的核心特征是**手动编写、版本控制、团队共享**。它不是 AI 自己写的笔记，而是工程师为 AI 制定的规则手册。它和 [SKILL.md 文件](/zh/blog/9-principles-writing-claude-code-skills)一起，构成了 Claude Code 七层架构中最关键的配置层。更多架构细节见 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 核心差异详解：控制权与自动化的取舍

两者最本质的区别在于**谁拥有控制权**。

CLAUDE.md 是一份契约。你写下「提交前必须通过 `npm test` 和 `npm run build`」，Claude Code 每次都会遵守，不会因为对话上下文变化而忽略这条规则。它的确定性是最大优势——团队里每个人启动 Claude Code，看到的都是同一套规则。

Memory 是一份笔记。Claude Code 根据对话内容判断什么值得记住，什么该更新，什么已过时。它的价值在于个性化——同一个项目，后端工程师和前端工程师的 Memory 会自然分化，各自积累对自己最有用的上下文。

这意味着 CLAUDE.md 适合存放**不可协商的硬性规则**：测试必须通过、不能跳过 lint、commit message 必须描述改了什么。Memory 适合存放**软性偏好和背景知识**：用户是数据科学家（解释代码时侧重数据流）、上次调试时发现某个 API 有隐藏限制、项目正在准备 Q2 发布所以不要引入大规模重构。

一个常见误区是把所有信息都塞进 CLAUDE.md。如果你发现自己在 CLAUDE.md 里写「我喜欢简洁的回复」或「我用 Vim 键位」，这些应该是 Memory 的内容——它们是个人偏好，不是项目规则。反过来，如果某条 Memory 反复影响代码质量（比如「必须用 CJK 字符计数而不是空格分词」），那就应该提升为 CLAUDE.md 中的正式规则。

## 实际应用：何时选择 Claude Memory

选择 Memory 的场景：

- **个人工作流优化**：你习惯的代码风格、回复长度偏好、解释详细程度——这些因人而异，不应该强制写进团队的 CLAUDE.md
- **项目动态追踪**：「周四后冻结合并」「当前在修 auth 模块的 bug」——这类时效性信息放在 CLAUDE.md 会快速过时，Memory 可以自然淘汰
- **跨会话学习**：你纠正了 Claude Code 的一个行为（「不要在测试里 mock 数据库」），Memory 会记住，下次不再犯同样的错误
- **外部资源索引**：Slack 频道、Grafana 看板、Linear 项目——这些指引对个人有用，但不适合作为代码库的一部分

如果你想深入了解 Claude Code 作为系统级工具的全貌，推荐阅读 [Claude Code 不是一个编程工具](/zh/blog/claude-code-is-not-a-coding-tool)。

## 实际应用：何时选择 CLAUDE.md

选择 CLAUDE.md 的场景：

- **团队规范统一**：编码风格、测试策略、提交规则——所有成员必须遵守的标准
- **质量门禁**：构建检查、lint 规则、pipeline 验证——硬性约束不能靠 Memory 的「软记忆」
- **新成员上手**：新人 clone 代码库后，Claude Code 自动加载 CLAUDE.md，立即了解项目规范，无需口口相传
- **架构文档**：技术栈选型、关键目录结构、已知 gotcha——这些是项目的「法律文件」，需要版本控制和代码评审
- **构建命令**：`npm run dev`、`npm run build`、`npm test`——写一次，所有会话都能用

想了解如何设计更精细的指令层，包括 Skills、Hooks 和 MCP 的配合，可以参考 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 最佳搭配策略

实际项目中，两者应该协同工作，各司其职：

**CLAUDE.md 负责「法律」——不可违反的规则：**
- 提交前必须通过的质量检查
- 代码风格和命名约定
- 禁止使用的模式或依赖
- 项目架构的关键约束

**Memory 负责「经验」——随时间积累的软知识：**
- 团队成员的角色和技术背景
- 过去犯过的错误和修正方式
- 当前阶段的项目重点和临时约束
- 外部工具和资源的位置

**升级路径**：当一条 Memory 被反复验证（比如「这个项目的中文内容必须用 CJK 字符计数」），就应该正式写入 CLAUDE.md。当一条 CLAUDE.md 规则只对特定成员有意义，就应该降级为个人 Memory。

这种分层设计让 Claude Code 既有团队一致性，又有个人适应性——这正是理解 [Claude Code Hooks](/zh/blog/claude-code-hooks-mastery) 和整个扩展体系的关键思路。

## 判断结论

**如果你在团队中工作，CLAUDE.md 是第一优先级。** 先把构建命令、测试规则、编码规范写进 CLAUDE.md，确保每个团队成员的 Claude Code 行为一致。然后让 Memory 自然积累个人层面的偏好和上下文。

**如果你是独立开发者，两者同等重要。** CLAUDE.md 定义你的项目规则（即使只有你一个人，明确的规则也能防止 AI 偏离），Memory 积累你的工作习惯和项目知识。

**不要试图只用一个。** CLAUDE.md 没有 Memory 的自动学习能力，Memory 没有 CLAUDE.md 的确定性保障。两者的交集几乎为零，重叠的部分恰好是你需要做出「升级或降级」决策的信号。

## 常见问题

### Claude Memory 的内容会被其他团队成员看到吗？

不会。Memory 存储在个人的 `.claude/projects/` 目录下，不进入版本控制，完全是个人私有的。团队共享的持久化信息应该放在 CLAUDE.md 中。

### CLAUDE.md 支持多层级配置吗？

支持。你可以在 `~/.claude/CLAUDE.md` 放全局配置（适用于所有项目），在项目根目录放项目级配置，甚至在子目录放模块级配置。Claude Code 会按层级合并加载。

### Memory 会无限增长吗？需要手动清理吗？

Memory 有内置的管理机制。MEMORY.md 索引文件超过 200 行会被截断，Claude Code 也会在发现过时记忆时主动更新或删除。但如果你发现无关记忆积累过多，可以手动清理 memory 目录下的文件。

### 我能让 Claude Code 把 Memory 的内容自动同步到 CLAUDE.md 吗？

目前不支持自动同步。这是有意为之——Memory 的个人偏好和 CLAUDE.md 的团队规则需要人工判断哪些值得「升级」。强制自动同步会把个人习惯污染进团队规范。
