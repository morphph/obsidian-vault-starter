**CLAUDE.md** is a plain text file. Claude Code reads it at the beginning of every conversation, injecting its contents into the system context. There's no indexing, no selective retrieval, no relevance filtering — the entire file loads every time. This means CLAUDE.md works best when it's concise and universally relevant. A 50-line file with build commands and key conventions loads fast and stays useful. A 500-line file with every decision ever made wastes context window space on information that's irrelevant to most tasks.

The hierarchy system helps manage scope. Your global `~/.claude/CLAUDE.md` carries personal defaults (git workflow preferences, commit message style). The project-level file carries repo-specific rules. Subdirectory files carry module-specific constraints. Claude Code merges these layers, so you get the right instructions at the right scope without bloating any single file.

**Claude Memory** uses a file-per-memory architecture with an index. Each memory is a separate markdown file with structured frontmatter (name, description, type). The `MEMORY.md` index file lists all memories with one-line descriptions. Claude Code loads the index at conversation start and can read individual memory files when they seem relevant. This selective retrieval means memory scales better — you can have dozens of memories without overwhelming the context window, because only the relevant ones get fully loaded.

The tradeoff is discoverability. Because memories are loaded selectively, Claude Code might miss a relevant memory if its description doesn't match the current task. CLAUDE.md instructions, by contrast, are always present — you never have to worry about Claude Code forgetting a build command or coding convention.

## Collaboration and Sharing: Team vs Individual Context

This is the sharpest distinction between the two systems, and it's where most of the decision logic lives.

**CLAUDE.md is team infrastructure.** When you write "always run `npm test` before committing" in CLAUDE.md, every developer on the team gets that instruction. New hires who clone the repo and start using Claude Code immediately inherit your team's conventions. Changes go through pull requests, so the team can discuss and approve updates to AI behavior. This makes CLAUDE.md the right place for anything that should be consistent across the team: coding standards, architectural decisions, deployment procedures, and quality gates.

**Claude Memory is individual context.** When Claude Code learns that you prefer terse responses without trailing summaries, that memory stays on your machine. Your teammate who likes detailed explanations keeps their own preference. When Claude Code remembers that you're investigating a specific bug or that you have a meeting-free Thursday for deep work, that's personal context that would be noise in a shared file.

The practical rule: **if removing the information would hurt a teammate's Claude Code session, it belongs in CLAUDE.md. If it would only hurt yours, it belongs in memory.**

This distinction also applies to project knowledge. "We use Vitest, not Jest" belongs in CLAUDE.md — every developer needs to know this. "The auth module rewrite is driven by legal compliance requirements, not tech debt" might belong in memory if it's context for your current work, or in CLAUDE.md if it's a lasting architectural decision the team should understand.

## Content Lifecycle: Creation, Updates, and Staleness

Each system handles the content lifecycle differently, with implications for accuracy and maintenance burden.

**CLAUDE.md follows a code lifecycle.** You create it deliberately, update it through commits, and review changes in pull requests. This means CLAUDE.md content tends to be high-quality and intentional — someone thought about what to write and a reviewer approved it. The downside is maintenance friction. When your build process changes, someone needs to remember to update CLAUDE.md. When a convention evolves, the file can lag behind actual practice. Teams that treat CLAUDE.md as living documentation keep it accurate; teams that write it once and forget it end up with stale instructions that actively mislead Claude Code.

**Claude Memory follows a conversation lifecycle.** Memories are created during work sessions, often triggered by corrections ("don't mock the database in these tests") or discoveries ("the user is a data scientist focused on observability"). Updates happen organically when Claude Code encounters contradicting information. The advantage is low friction — useful context accumulates without deliberate maintenance effort. The disadvantage is that memories can become stale without anyone noticing. A project memory about a merge freeze that passed three weeks ago still sits in your memory directory until Claude Code or you clean it up.

Memory staleness is mitigated by a verification step — Claude Code is instructed to verify memory claims against current project state before acting on them. If a memory says "function X is in file Y," Claude Code should check that the file and function still exist before recommending changes. But this verification adds latency and isn't foolproof.

**Practical maintenance recommendation:** Review CLAUDE.md quarterly or whenever your stack changes significantly. For memory, let Claude Code manage creation but periodically scan `MEMORY.md` for outdated entries — especially project-type memories, which decay fastest.

## Scope and Hierarchy: Where Each System Reaches

Both systems support hierarchical scope, but the hierarchies work differently.

**CLAUDE.md hierarchy** is directory-based:
1. `~/.claude/CLAUDE.md` — global defaults (your personal preferences across all projects)
2. `<project>/CLAUDE.md` — project-level rules (shared with team)
3. `<project>/src/CLAUDE.md` — subdirectory rules (module-specific constraints)

Claude Code merges these top-down, with more specific files taking precedence. This lets you set global preferences ("always use conventional commits") while allowing projects to override them ("this repo uses a custom commit format").

**Claude Memory hierarchy** is type-based, not directory-based:
1. **User memories** — persist across all projects (your role, expertise, preferences)
2. **Feedback memories** — typically project-scoped (corrections within a specific codebase)
3. **Project memories** — project-scoped and time-sensitive (current initiatives, deadlines)
4. **Reference memories** — project-scoped (pointers to external systems)

Memory doesn't have subdirectory scoping. A memory about your React preferences applies whenever you're working on React code, regardless of which directory you're in. This is both a strength (no need to repeat yourself) and a limitation (you can't give different memory context to different modules).

## When to Choose CLAUDE.md

Use CLAUDE.md when the information meets any of these criteria:

- **Team-shared knowledge**: Build commands, test procedures, deployment steps, coding standards. If a new developer needs this on day one, it belongs in CLAUDE.md.
- **Deterministic instructions**: "Run `npm run build` before committing." "Never use `any` type." "Use snake_case for database columns." These are rules, not suggestions, and they should be consistent across all sessions.
- **Architectural constraints**: "Don't import Next.js modules in pipeline scripts." "All API routes go through the middleware layer." These are load-bearing decisions that Claude Code must always respect.
- **Known gotchas**: "Chinese content uses CJK word count, not English space tokenization." Project-specific traps that would waste debugging time if forgotten.

CLAUDE.md is also the right choice when **auditability matters**. Because it's version-controlled, you can trace when an instruction was added, who approved it, and why. This matters for teams with compliance requirements or for understanding why Claude Code behaves a certain way.

Teams building serious [agentic coding](/glossary/agentic-coding) workflows should treat CLAUDE.md as a core part of their developer infrastructure — as important as their linter config or CI pipeline. Our guide on [Claude Code's seven programmable layers](/blog/claude-code-seven-programmable-layers) explains how CLAUDE.md fits into the broader configuration stack, from skills to hooks to agent teams.

## When to Choose Claude Memory

Use Claude Memory when the information meets any of these criteria:

- **Personal preferences**: Communication style, response length, whether you want explanations or just code. These vary between developers and don't belong in a shared file.
- **Your individual context**: Your role, expertise level, what you're currently working on, your learning goals. "User is a senior backend engineer, new to React" helps Claude Code calibrate explanations without affecting teammates.
- **Workflow corrections**: When you tell Claude Code "don't summarize what you just did" or "always check tests before committing," that feedback should persist across sessions but doesn't need to be a team-wide rule.
- **Ephemeral project context**: "We're in a merge freeze until March 5th." "The auth rewrite is blocked on legal review." These are time-sensitive facts that would clutter CLAUDE.md and go stale quickly.
- **External system pointers**: "Pipeline bugs are tracked in the Linear INGEST project." "The oncall dashboard is at grafana.internal/d/api-latency." These reference links help Claude Code navigate your tooling without bloating the project README.

Memory also shines for **cross-project context**. Your user memories — role, expertise, preferences — carry across every project you work on. CLAUDE.md only applies within its repository. If you want Claude Code to remember that you prefer functional components in React regardless of which project you're in, that's a memory, not a CLAUDE.md instruction.

## Using Both Together

The strongest Claude Code setups use both systems deliberately, with clear boundaries between them.

**CLAUDE.md carries the "what" and "how"** — project rules, build commands, conventions, constraints. It's the constitution of your project's AI-assisted workflow.

**Memory carries the "who" and "why"** — who you are, why certain decisions were made, what's happening right now, and how you personally want to work.

A concrete example: CLAUDE.md says "run `npm test` before every commit." Memory says "this user prefers to see test output inline rather than in a summary" and "the test suite has been flaky in the auth module since the migration — check twice if auth tests fail."

Another example: CLAUDE.md says "commit messages follow conventional commit format." Memory says "this user was corrected last session for writing 'update' when the change was a bug fix — use 'fix' for bug fixes."

The interaction between systems also matters. Memory can reference CLAUDE.md context ("the CLAUDE.md says to run validate-pipeline.ts, but the user mentioned this is temporarily broken and to skip it until the migration is done"). CLAUDE.md should never reference memory content, since memory is personal and not visible to the team.

**Anti-pattern to avoid**: Don't duplicate information across both systems. If your build command is in CLAUDE.md, don't also save a memory about it. If your role is in memory, don't also put it in CLAUDE.md. Duplication creates drift — when one copy updates and the other doesn't, Claude Code gets contradictory instructions.

## Verdict

**CLAUDE.md and Claude Memory are complementary layers, not competitors.** Treating them as alternatives is a category error — they store different types of information for different audiences with different lifecycles.

**Start with CLAUDE.md** if you're setting up Claude Code for the first time. Get your build commands, coding standards, and key constraints into a project-level file. This gives you immediate, consistent value across every session.

**Let Memory accumulate naturally** as you work. Don't try to pre-populate it — the system works best when it captures real corrections and context from actual work sessions. Over time, your memory builds a profile that makes Claude Code increasingly effective at working with you specifically.

**The decision rule is simple**: shared, stable, auditable knowledge goes in CLAUDE.md. Personal, evolving, individual context goes in memory. When in doubt, ask: "Would my teammate need this?" If yes, CLAUDE.md. If no, memory.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

Memory and CLAUDE.md occupy the same context window, and Claude Code treats both as authoritative input. In practice, more specific instructions tend to win — a memory correction like "the user said to skip linting for this sprint" can override a CLAUDE.md rule about always running the linter. To avoid conflicts, keep CLAUDE.md for stable rules and use memory for temporary exceptions.

### Does Claude Memory sync across machines?

No. Claude Memory is stored locally in `~/.claude/projects/` and is not synced, version-controlled, or backed up by default. If you switch machines, your memories don't follow. CLAUDE.md, because it's committed to git, automatically syncs wherever the repo is cloned. For critical personal preferences, consider also adding them to your global `~/.claude/CLAUDE.md`, which you can manually copy between machines.

### How many memories is too many?

The `MEMORY.md` index truncates after 200 lines, so aim to stay well under that limit. In practice, 20-40 focused memories cover most developers' needs. Prune aggressively — delete project memories after the project phase ends, remove feedback memories that have become obvious habits, and consolidate related memories into single entries. Quality beats quantity; a few precise memories outperform dozens of vague ones.
