# Codex vs Claude Code：子代理与自定义 Agent 能力对比

**TL;DR：** 在多 Agent 编程场景中，**Claude Code 的 Agent Teams 更成熟**——支持会话内动态生成子代理、worktree 隔离和类型化的专用代理。**OpenAI Codex** 的子代理能力目前更偏向云端任务并行，适合批量处理独立编码任务，但自定义程度和灵活性不及 Claude Code。选择取决于你的工作流：需要精细编排选 Claude Code，需要大规模云端并行选 Codex。

## 概述：OpenAI Codex 的 Agent 架构

[OpenAI Codex](/zh/blog/codex-complete-guide) 是一个云端编程智能体，每个任务在独立的沙箱容器中运行。Codex 的核心理念是"一个任务一个 Agent"——你提交一个编码请求，Codex 在云端环境中克隆仓库、读取代码、执行修改，最终生成 PR 或补丁。

Codex 的"子代理"概念与传统的多 Agent 编排不同。它更接近任务级并行：你可以同时提交多个独立任务，每个任务由一个 Codex 实例处理。这种模式适合批量修复 bug、生成测试、或对多个模块执行相同操作。但每个实例之间没有通信机制，无法在运行时动态协调。

在自定义方面，Codex 支持通过 `AGENTS.md` 文件（类似 Claude Code 的 `CLAUDE.md`）注入项目级指令，以及通过 [VS Code 扩展](/zh/blog/codex-vscode)在 IDE 内发起任务。但目前缺少可编程的 Agent 类型系统——你无法定义"这是一个专门做代码审查的 Agent"并在工作流中复用它。

## 概述：Claude Code 的 Agent Teams

[Claude Code Agent Teams](/zh/blog/claude-code-agent-teams) 是 Anthropic 在 2026 年推出的多代理协作系统。与 Codex 的任务并行不同，Claude Code 的子代理是会话内的——主 Agent 可以在对话过程中动态生成多个子代理，每个子代理有独立的上下文和工具权限。

Claude Code 提供了类型化的子代理系统。开发者可以通过 `subagent_type` 指定代理类型——`Explore` 用于代码搜索、`Plan` 用于架构设计、`code-reviewer` 用于代码审查。每种类型有预定义的工具集和行为模式。更强大的是，通过 [Skills 系统](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)和 `SKILL.md` 文件，团队可以编码标准化的 Agent 行为，让子代理在不同项目中保持一致的工作方式。

子代理还支持 `worktree` 隔离模式——每个子代理在独立的 git worktree 中工作，避免并行编辑时的文件冲突。这是 Codex 沙箱隔离思路的本地化版本，但粒度更细。

## 功能对比

| 功能维度 | OpenAI Codex | Claude Code | 优势方 |
|---------|-------------|-------------|--------|
| **Agent 运行环境** | 云端沙箱容器 | 本地终端 + 可选远程 | 各有优势 |
| **子代理通信** | 无（独立实例） | 会话内消息传递 | Claude Code |
| **类型化 Agent** | 不支持 | 支持（Explore, Plan 等） | Claude Code |
| **自定义 Agent 行为** | AGENTS.md 项目指令 | CLAUDE.md + SKILL.md | Claude Code |
| **并行隔离** | 容器级隔离 | git worktree 隔离 | 平手 |
| **批量任务** | 原生支持，云端并行 | 通过子代理实现 | Codex |
| **IDE 集成** | VS Code 扩展 | VS Code + JetBrains | Claude Code |
| **定价** | 按任务计费 | 按 token 使用量 | 视用量而定 |

## 子代理编排能力：详细分析

这是两者差距最大的维度。Claude Code 的子代理系统本质上是一个运行时编排框架——主 Agent 可以在对话过程中根据任务需要动态决定是否生成子代理、生成什么类型、分配什么任务。

举个实际场景：你要求重构一个认证模块。Claude Code 的主 Agent 可能会：

1. 生成一个 `Explore` 类型子代理，搜索所有引用该模块的文件
2. 生成一个 `Plan` 类型子代理，设计重构方案
3. 确认方案后，生成多个通用子代理，分别在独立 worktree 中修改不同文件
4. 最后用 `code-reviewer` 子代理审查所有变更

这个编排过程是动态的，主 Agent 根据中间结果调整策略。子代理完成后，结果回传给主 Agent 进行汇总。

Codex 的做法更直接：你提交多个任务，每个任务独立执行。如果你想实现类似的编排，需要在外部（比如 CI/CD 脚本或自定义工具链）完成任务分解和结果汇总。Codex 本身不提供 Agent 间的协调原语。

对于需要多步骤推理和动态决策的复杂工程任务，Claude Code 的方案明显更强。对于可以预先分解为独立子任务的批量操作，[Codex 的云端并行](/zh/blog/con-u-pour-des-workflows-multi-agents)更高效。

## 自定义与可扩展性：详细分析

自定义 Agent 行为的深度决定了工具的天花板。Claude Code 通过多层配置实现深度自定义：

- **CLAUDE.md**：项目级指令，定义代码风格、架构约束、工作流规则
- **SKILL.md**：任务级指令，编码特定操作的标准化流程（如"写测试时必须覆盖边界情况"）
- **Hooks**：事件驱动的自动化——文件保存时自动 lint、提交前自动测试
- **MCP Server**：连接外部工具和数据源，扩展 Agent 能力边界

这套 [四层扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 让团队可以把工程实践编码为可复用的 Agent 配置，新成员加入时自动继承团队标准。

Codex 的自定义目前集中在 `AGENTS.md` 和环境配置。你可以指定依赖安装命令、测试运行方式、代码风格要求，但无法定义可复用的 Agent 类型或编程化的事件钩子。OpenAI 已经暗示了更丰富的自定义能力在路线图上，但截至目前尚未发布。

## 什么时候选 Codex

如果你的场景符合以下特征，Codex 是更好的选择：

- **批量独立任务**：需要对 20 个文件执行类似的修改，每个修改互不依赖。Codex 的云端并行能在几分钟内完成。
- **CI/CD 集成**：Codex 天然运行在云端，适合嵌入 CI pipeline。GitHub bot 触发 Codex 修复 lint 错误、生成缺失测试——这是它的甜区。
- **资源敏感场景**：本地机器性能有限时，Codex 的云端沙箱不占用本地资源。对于[学生用户](/zh/blog/codex-for-students)和[开源项目](/zh/blog/codex-for-open-source)，OpenAI 还提供了免费额度。
- **团队异步协作**：提交任务后无需等待，Codex 后台执行，完成后通知。适合跨时区团队的异步工作模式。

## 什么时候选 Claude Code

如果你的场景符合以下特征，Claude Code 更合适：

- **复杂编排任务**：重构涉及多个模块、需要理解上下文关系和动态决策的工程任务。Agent Teams 的运行时编排能力在这类场景中优势明显。
- **精细化 Agent 定制**：团队有成熟的工程规范，需要把最佳实践编码为 [Skills](/zh/blog/9-principles-writing-claude-code-skills) 并在 Agent 行为中强制执行。
- **交互式开发**：需要在对话中逐步推进、根据中间结果调整方向。Claude Code 的会话式子代理支持人机协作的编排模式。
- **本地优先**：代码不能离开本地环境，或需要访问本地工具链、数据库、自定义脚本。Claude Code 运行在终端，完全本地化。
- **代码审查自动化**：Claude Code 的 [Review Agent](/zh/blog/claude-code-review-agents) 是专门设计的子代理类型，提供结构化的多维度代码审查。

## 结论

**如果你需要灵活的多 Agent 编排和深度自定义，选 Claude Code。如果你需要大规模云端并行和异步任务处理，选 Codex。**

两者并不互斥。一些团队已经在实践中组合使用：Claude Code 在本地处理需要深度理解和多步推理的复杂任务，Codex 在云端批量处理可预定义的重复性工作。选择的关键不是"哪个更好"，而是你的工作流更需要编排深度还是并行广度。

随着两个平台持续迭代，差距可能缩小。OpenAI 有可能为 Codex 增加更丰富的 Agent 类型系统，Anthropic 也在探索 Claude Code 的云端运行模式。关注 [LoreAI 的每日简报](/zh/blog/claude-code-subagents-examples)获取最新动态。

## 常见问题

### Codex 支持自定义 Agent 类型吗？

截至目前，Codex 不支持开发者定义可复用的 Agent 类型。每个 Codex 实例本质上是同一类通用编码 Agent，通过 `AGENTS.md` 接收项目级指令，但无法像 Claude Code 那样指定 `Explore`、`Plan` 等专用类型。

### Claude Code 的子代理可以并行运行吗？

可以。Claude Code 支持在同一会话中生成多个子代理并行执行，每个子代理可以运行在独立的 git worktree 中以避免文件冲突。主 Agent 等待所有子代理完成后汇总结果。

### 两者可以一起使用吗？

完全可以。实际工作流中，Claude Code 适合处理需要深度推理和动态编排的本地任务，Codex 适合处理可批量分解的云端任务。两者通过 Git 仓库共享代码，互不冲突。
