# Claude Memory vs CLAUDE.md：两套持久化机制，到底该用哪个？

**TL;DR：** **CLAUDE.md** 是你写给 Claude 的规则手册，适合团队共享的编码规范、项目约束和工作流程——它是确定性的、版本可控的。**Claude Memory** 是 Claude 自己写给自己的笔记本，自动记录你的偏好、反馈和项目上下文——它是渐进式的、个人化的。两者不是竞争关系，而是互补的持久化层。**大多数成熟项目应该同时使用两者**：CLAUDE.md 定义"必须做什么"，Memory 记住"你喜欢怎么做"。

## 概述：Claude Memory

Claude Memory 是 Claude Code 的自动记忆系统，存储在 `.claude/projects/` 目录下。它在对话过程中自动捕捉并持久化用户的偏好、项目背景和工作反馈，使得跨会话的上下文不会丢失。

Memory 的核心价值在于**隐式学习**。你不需要主动编写任何配置文件——当你纠正 Claude 的做法（"不要在这些测试里 mock 数据库"）或确认某个非显而易见的方案有效时，Claude 会自动将这些信息写入记忆文件。下次对话时，它会读取这些记忆并调整行为。

Memory 分为四种类型：`user`（用户角色和偏好）、`feedback`（行为纠正和确认）、`project`（项目动态和决策）、`reference`（外部资源指向）。每条记忆都是独立的 Markdown 文件，通过 `MEMORY.md` 索引管理。详细机制可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

CLAUDE.md 是 Claude Code 的项目级指令文件，通常放在仓库根目录并 commit 到版本控制中。它定义了项目的编码规范、构建命令、架构约束和工作流规则——本质上是一份**显式的、结构化的规则手册**。

CLAUDE.md 的设计哲学是**确定性和可共享**。团队中每个人的 Claude Code 都会读取同一份 CLAUDE.md，确保 AI 行为在不同开发者之间保持一致。无论谁打开项目，Claude 都知道"先跑 `npm run build`，再跑 `npm test`，commit 之前必须通过所有检查"。

CLAUDE.md 支持多层级：全局的 `~/.claude/CLAUDE.md` 定义个人偏好，项目根目录的 `CLAUDE.md` 定义团队规范，子目录的 `CLAUDE.md` 可以覆盖特定模块的规则。这种层级结构在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中处于最基础的配置层。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **创建方式** | 自动生成，Claude 主动写入 | 手动编写，开发者定义 | 各有所长 |
| **存储位置** | `.claude/projects/` 目录 | 项目根目录（可嵌套） | CLAUDE.md |
| **版本控制** | 通常 .gitignore，不入库 | commit 到仓库，团队共享 | CLAUDE.md |
| **内容类型** | 偏好、反馈、项目动态、外部引用 | 规则、命令、约束、架构说明 | 各有所长 |
| **更新频率** | 每次对话可能更新 | 项目里程碑时手动更新 | Memory |
| **个人化程度** | 高——每个开发者不同 | 低——团队统一 | Memory |
| **可预测性** | 较低——Claude 自主判断何时记忆 | 高——写什么就执行什么 | CLAUDE.md |
| **跨会话持久** | 是 | 是 | 平局 |
| **适合新成员** | 需要积累，无法继承他人记忆 | 立刻生效，新人也能用 | CLAUDE.md |

## 持久化机制：深度分析

这是两者最根本的区别。CLAUDE.md 是**声明式配置**——你明确告诉 Claude"做什么"和"不做什么"，它每次都会严格遵循。Claude Memory 是**经验式积累**——通过多次对话逐渐建立起对你工作方式的理解。

CLAUDE.md 的优势在于**可审计性**。因为它是纯文本文件且纳入版本控制，你可以 review 每一次修改、追溯规则的来源、在 PR 中讨论规则变更。当团队对"AI 应该怎么做"产生分歧时，CLAUDE.md 是唯一的事实来源。

Memory 的优势在于**低摩擦**。很多偏好和上下文是你在工作中自然流露的，不值得专门写进配置文件。比如"这个用户是数据科学家，解释代码时多用统计类比"——这种信息放在 CLAUDE.md 里显得格格不入，但作为 Memory 条目恰到好处。

一个常见的误区是把 Memory 当作 CLAUDE.md 的替代品。实际上，Memory 明确声明了**不应该保存的内容**：代码模式、文件路径、项目结构、Git 历史——这些要么可以从代码中直接获取，要么应该写在 CLAUDE.md 里。Memory 的定位是保存那些"代码和配置文件都无法表达"的隐性知识。

关于这两套系统如何融入更大的 Claude Code 架构，可以参考 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 团队协作：深度分析

在多人协作场景下，CLAUDE.md 和 Memory 的角色差异更加明显。

**CLAUDE.md 是团队的共识层。** 假设你的团队约定"所有 API 路由必须用 Zod 做输入验证"——这条规则写在 CLAUDE.md 里，每个团队成员的 Claude 都会遵守。新人加入时不需要口头传递这些规范，Claude 会自动执行。

**Memory 是个人的适配层。** 同一个团队里，后端工程师可能偏好简洁的终端输出，前端工程师可能希望 Claude 多解释 CSS 布局逻辑。这些个人偏好不适合写进团队共享的 CLAUDE.md，但 Memory 会自然地为每个人建立不同的交互模式。

一个实际的例子：团队的 CLAUDE.md 规定"commit message 用英文，遵循 Conventional Commits 格式"。通过几次对话后，Memory 可能额外记住"这个用户习惯在 commit 之前先看 diff，不要自动 commit"。前者是团队规则，后者是个人习惯——两者互不冲突。

写好 CLAUDE.md 是一项需要刻意练习的技能。[Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)中的很多原则同样适用于 CLAUDE.md 的编写。

## 适合使用 Claude Memory 的场景

当你的需求是**个性化**和**渐进式学习**时，Memory 是更好的选择：

- **个人偏好沉淀**：你喜欢 terse 的回复风格、不想看到 trailing summary、偏好特定的测试框架——这些通过自然对话就能积累
- **项目动态追踪**：谁在负责什么、下周有 code freeze、某个 bug 的背景故事——这类时效性信息不适合写死在配置文件里
- **外部资源索引**：Linear 项目链接、Grafana 看板地址、Slack 频道用途——Memory 的 `reference` 类型专为此设计
- **行为微调**：你纠正了 Claude 一次（"不要用 `any` 类型"），Memory 确保它不会犯同样的错误

如果你还没有体验过 Memory 的实际效果，[Claude 记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)这篇文章详细介绍了最新的记忆导入功能。

## 适合使用 CLAUDE.md 的场景

当你的需求是**确定性**和**团队一致性**时，CLAUDE.md 是更好的选择：

- **构建和测试命令**：`npm run build`、`npm test`、`npm run lint`——这些是每次 commit 前必须执行的质量门禁
- **编码规范**：命名约定、错误处理模式、禁止使用的 API——团队所有人必须遵守的硬性规则
- **架构约束**：不要在 pipeline 脚本里导入 Next.js 模块、中文内容必须用 CJK 字符计数——这类项目特有的陷阱
- **工作流程**：新功能先讨论设计再编码、pipeline 变更前跑验证脚本——流程性的规则

CLAUDE.md 最大的优势是**立即生效且可复现**。不需要几轮对话来"训练" Claude——规则写下来就会被执行。关于如何将 CLAUDE.md 与 Skills 系统结合使用，可以参考[我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

## 结论

**不要在 Claude Memory 和 CLAUDE.md 之间二选一——它们解决的是不同层面的问题。**

CLAUDE.md 是你项目的"宪法"：明确、共享、版本可控。它定义了不可协商的规则和约束，确保 AI 行为在团队内一致。Memory 是你和 Claude 之间的"工作默契"：隐式、个人化、持续演进。它让 Claude 越用越懂你，而不需要你反复解释同样的偏好。

**实践建议：**

- **先写 CLAUDE.md**——把构建命令、编码规范和架构约束写清楚，这是任何项目的基础
- **让 Memory 自然积累**——不要试图手动管理每一条记忆，让它在日常对话中有机生长
- **定期审查 Memory**——Memory 会过时，定期检查 `.claude/projects/` 下的记忆文件，删除不再准确的条目
- **团队规则进 CLAUDE.md，个人偏好留 Memory**——这是最清晰的分界线

如果你想更深入地理解 Claude Code 为什么需要这种多层持久化设计，[Claude Code 不是一个编程工具](/zh/blog/claude-code-is-not-a-coding-tool)这篇文章从更高的视角解释了 Claude Code 作为系统（而非工具）的设计哲学。

## 常见问题

### Claude Memory 的内容会被 commit 到仓库吗？

不会。Memory 存储在 `.claude/projects/` 目录下，通常被 `.gitignore` 排除。它是个人化的持久数据，不适合也不需要纳入版本控制。如果你希望某条信息被团队共享，应该写进 CLAUDE.md。

### CLAUDE.md 支持多层级配置吗？

支持。Claude Code 会读取三个层级的 CLAUDE.md：全局配置（`~/.claude/CLAUDE.md`）、项目根目录配置、以及子目录配置。优先级从具体到通用——子目录规则可以覆盖项目级规则，项目级规则可以覆盖全局规则。

### 如果 Memory 记录了错误的信息怎么办？

你可以直接告诉 Claude"忘记这个"，它会找到并删除对应的记忆条目。也可以手动编辑 `.claude/projects/` 下的 Markdown 文件。Memory 设计上就是可修正的——它是建议而非强制规则。

### 两者发生冲突时，哪个优先？

CLAUDE.md 的规则具有更高的优先级。如果 CLAUDE.md 规定"commit message 用英文"，即使 Memory 记录了"用户偏好中文 commit message"，Claude 也会遵循 CLAUDE.md 的规则。这是设计意图：显式规则优先于隐式偏好。
