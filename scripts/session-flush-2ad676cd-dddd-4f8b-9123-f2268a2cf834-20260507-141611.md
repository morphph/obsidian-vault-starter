# Claude Memory vs CLAUDE.md：两套持久化机制怎么选？

**TL;DR：** **CLAUDE.md** 是写给 Claude Code 的规则手册——编码规范、项目约束、工作流要求，随代码库版本控制，所有人共享。**Claude Memory（Auto Memory）** 是 Claude Code 的自动笔记本——跨会话记住你的偏好、项目上下文和反馈，存储在本地 `.claude/` 目录，仅对当前用户生效。两者不是替代关系，而是互补的两层持久化。**团队规则放 CLAUDE.md，个人偏好交给 Memory。**

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 在每个会话开始时自动读取的项目级指令文件，放在代码仓库根目录，随 Git 版本控制。它的核心作用是为 Claude Code 设定"行为边界"——哪些命令必须在提交前运行、代码风格遵循什么标准、哪些操作绝不允许执行。

CLAUDE.md 本质上是一份静态的、人工维护的规则文件。你写什么，Claude 就遵循什么。它不会自动更新，也不会随着对话积累新内容。这种确定性正是它的优势：团队中每个成员拉取代码后，Claude Code 的行为一致。

CLAUDE.md 支持层级覆盖：全局级（`~/.claude/CLAUDE.md`）覆盖所有项目，项目级（仓库根目录）针对特定代码库，子目录级可进一步细化。详细用法可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：Claude Memory（Auto Memory）

**Claude Memory** 是 Claude Code 内置的自动持久化记忆系统，存储在 `.claude/projects/<project-hash>/memory/` 目录下。它不需要你手动编辑配置文件——Claude Code 在对话过程中自动识别值得记住的信息，写入结构化的 Markdown 文件。

Memory 分为四种类型：**user**（用户角色和偏好）、**feedback**（工作方式纠正）、**project**（项目动态和决策）、**reference**（外部资源指向）。每种类型解决不同的上下文问题。比如你告诉 Claude "我是后端工程师，第一次接触 React"，它会保存为 user 类型记忆，后续在解释前端概念时自动调整表达方式。

Memory 的关键特征是**增量积累**：每次对话可能新增、更新或删除记忆条目，逐渐构建出对你和项目的完整理解。这与 CLAUDE.md 的"写一次，手动维护"模式形成鲜明对比。关于记忆系统的最新升级，可以阅读 [Claude 记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)。

## 功能对比

| 维度 | CLAUDE.md | Claude Memory | 优势方 |
|------|-----------|---------------|--------|
| **存储位置** | 仓库根目录，Git 版本控制 | `.claude/` 本地目录，不进 Git | 视需求 |
| **维护方式** | 手动编辑 | Claude 自动写入 | Memory |
| **作用范围** | 全团队共享 | 仅当前用户 | CLAUDE.md |
| **内容类型** | 规则、约束、工作流 | 偏好、反馈、项目动态 | 视需求 |
| **生命周期** | 与代码库同步 | 跨会话持久 | 平手 |
| **确定性** | 100% 可预测 | 有一定自主判断 | CLAUDE.md |
| **上下文负担** | 每次全量加载 | 按相关性选择加载 | Memory |
| **可审计性** | Git diff 可追踪 | 需手动检查文件 | CLAUDE.md |

## 核心差异：控制模型

CLAUDE.md 和 Memory 最本质的区别在于**谁决定内容**。

CLAUDE.md 是**显式规则注入**。你写下"提交前必须运行 `npm test` 和 `npm run build`"，Claude Code 就严格执行。这是一种声明式的控制——你定义行为边界，Claude 在边界内操作。规则的优先级、措辞和范围完全由你决定，没有任何"自动推断"的空间。

Memory 是**隐式上下文积累**。Claude Code 在对话中观察到你的工作模式、偏好和反馈，自主判断哪些信息值得持久化。你说"别在回复末尾加总结"，它会记住这个偏好并在后续会话中应用。但什么该记、怎么记、记多详细，这些判断由 Claude Code 自己做出。

这意味着两件事：第一，**关键规则必须放 CLAUDE.md**，因为 Memory 的记录依赖模型判断，可能遗漏或误解你的意图；第二，**个性化调整适合交给 Memory**，因为手动维护一份"我喜欢简洁回复、我熟悉 Go 但不熟悉 React"的配置文件既繁琐又容易过时。

## 团队协作场景

在团队环境中，CLAUDE.md 的版本控制特性让它成为**工程规范的载体**。一个典型的 CLAUDE.md 可能包含：

- 提交前必须通过的质量门禁（lint、test、build）
- 项目特有的代码风格约束
- 禁止的操作（如不允许 `--force push`）
- 架构决策记录（如"数据库查询走 ORM，不写原生 SQL"）

这些规则对团队每个成员一致生效，新人 clone 仓库后 Claude Code 立刻了解项目规范，零配置成本。

Memory 在团队场景中的角色不同——它处理**个人维度的差异**。同一个团队中，后端工程师和前端工程师需要不同的解释方式；Tech Lead 关注架构决策，Junior 关注实现细节。这些差异不适合写进共享的 CLAUDE.md，而 Memory 自动处理。

想了解如何用 Skills 系统进一步标准化团队中 Claude Code 的行为，可以阅读 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)。

## 什么时候用哪个

### 用 CLAUDE.md 的场景

- 代码质量门禁和 CI 流程要求
- 项目级编码规范和架构约束
- 团队统一的 Git 工作流
- 安全边界（禁止的操作、敏感文件保护）
- 任何需要版本控制和代码审查的规则

### 用 Memory 的场景

- 个人技术背景和沟通偏好
- 对 Claude Code 工作方式的纠正和反馈
- 项目中的临时决策和动态上下文
- 外部资源的指向（Slack 频道、监控面板等）
- 任何"说一次就够了，别让我重复"的信息

### 两者结合的最佳实践

最有效的用法是让两个系统各司其职：CLAUDE.md 定义"做什么"和"不做什么"，Memory 记录"怎么做对这个人最有帮助"。在 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 中，我们讨论了 Skills、Hooks、Memory 和 CLAUDE.md 四层系统如何协同工作。

一个常见的反模式是把个人偏好写进 CLAUDE.md（污染团队配置），或者把关键规则依赖 Memory（可能在新会话中丢失上下文）。保持边界清晰：**共享规则 → CLAUDE.md，个人上下文 → Memory**。

## 结论

**CLAUDE.md 和 Memory 不是竞争关系，而是 Claude Code 持久化系统的两个互补层。** CLAUDE.md 是你的项目宪法——明确、共享、可审计，适合编码规范和工程流程。Memory 是你的个人助理笔记——自动、私有、渐进积累，适合偏好和上下文。

对于个人开发者，先把核心规则写进 CLAUDE.md，其余交给 Memory 自动处理。对于团队，CLAUDE.md 是必须认真维护的工程资产，Memory 则让每个成员获得个性化的协作体验。两者结合使用，Claude Code 才能真正从"每次都要重新解释"进化到"一个越用越懂你的编程搭档"。

## 常见问题

### Claude Memory 会不会记错东西？

Memory 依赖模型判断，确实可能记录不准确或过时的信息。Claude Code 在使用记忆前会尝试验证其时效性，但关键规则不应依赖 Memory——放进 CLAUDE.md 更可靠。你也可以随时手动检查和编辑 `.claude/` 目录下的记忆文件。

### CLAUDE.md 太长会不会影响性能？

CLAUDE.md 在每次会话开始时全量加载到上下文窗口中。过长的文件会占用宝贵的上下文空间，建议控制在合理篇幅内，把详细的领域规则拆分到 `skills/*/SKILL.md` 文件中按需加载。

### 能不能只用一个，不用另一个？

可以，但不推荐。只用 CLAUDE.md 意味着每次新会话都要重复说明你的偏好；只用 Memory 意味着关键规则缺乏确定性保障。两者配合使用的效果远大于单独使用。
