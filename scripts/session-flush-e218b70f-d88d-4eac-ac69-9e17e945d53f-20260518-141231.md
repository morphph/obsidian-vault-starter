**Put in CLAUDE.md:**
- Build commands and development setup
- Coding standards and lint rules
- Architectural constraints and known gotchas
- Workflow requirements (testing gates, deployment steps)
- Project-specific terminology or naming conventions

**Put in Claude Memory:**
- Your role, expertise level, and areas of focus
- Your interaction preferences (verbosity, formatting, explanation depth)
- Ongoing project context that changes weekly (current sprint focus, active incidents)
- Pointers to external systems you personally reference often
- Corrections to Claude Code's approach that reflect your personal workflow

Some teams try to put everything in CLAUDE.md, including personal preferences. This creates bloat and conflicts — one developer wants verbose explanations while another wants terse output. Memory solves this by keeping personal context personal, while CLAUDE.md stays focused on universal project truths.

## Maintenance and Lifecycle

CLAUDE.md maintenance follows software development patterns. You update it when the codebase changes: new build commands, new architectural decisions, deprecated patterns. Teams that treat CLAUDE.md as a living document (updating it alongside code changes) get the most value. Teams that write it once and forget about it see diminishing returns as the codebase drifts from the documented state.

A good practice from teams using Claude Code effectively is to include CLAUDE.md updates in their definition of done. Changed the test framework? Update CLAUDE.md. Added a new pipeline stage? Document the command. This is why the [skills and extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) documentation recommends co-locating instructions with the code they describe.

Claude Memory maintenance is lighter but less structured. The memory system includes built-in decay signals — memories record when they were written, and Claude Code is instructed to verify stale-looking memories against current state before acting on them. You can also explicitly tell Claude Code to forget something, or manually delete memory files from the `.claude/projects/` directory.

The risk with memory is accumulation without curation. After months of daily use, you might have dozens of memory files, some contradicting each other, some referencing files or features that no longer exist. Periodic cleanup helps — scanning through MEMORY.md every few weeks and removing entries that are clearly outdated. But this is optional maintenance that most developers skip, which is fine because the verification mechanisms catch most stale entries at query time.

## Migration and Portability

CLAUDE.md is fully portable. Clone the repo on a new machine, and all project instructions are there. Onboard a new team member, and they get the same Claude Code behavior as everyone else. Move to a different CI environment, and CLAUDE.md travels with the code. This portability is one of CLAUDE.md's strongest properties — it's just a file in your repo.

Claude Memory is tied to a specific machine and user profile. If you switch laptops, your memory doesn't automatically follow. If you work on the same project from two different machines, you'll build up two separate memory profiles. Anthropic has been working on [memory import and export capabilities](/blog/anthropic-claude-memory-upgrades-importing), but as of mid-2026, memory is still fundamentally local.

This portability gap matters most during transitions: new team members, new machines, CI/CD environments. CLAUDE.md works everywhere Claude Code runs. Memory only works where it's been built up. For teams that rely heavily on Claude Code across multiple environments (local development, CI agents, cloud workstations), CLAUDE.md is the only reliable persistence layer.

## Practical Setup: Getting Started With Each

### Setting Up CLAUDE.md

Create a `CLAUDE.md` file in your project root. Start minimal — you can always add more:

```markdown
# Project Name

## Commands
npm run dev          # Local dev server
npm run build        # Production build
npm test             # Run tests

## Rules
- All new code must include tests
- Use TypeScript strict mode
- Never commit directly to main
```

That's it. Claude Code reads this file automatically on every session. No configuration, no initialization, no setup command. For personal global instructions (applying across all projects), create `~/.claude/CLAUDE.md`.

### Setting Up Claude Memory

Claude Memory requires no setup. It activates automatically when Claude Code learns something worth remembering. You can accelerate it by explicitly telling Claude Code important context:

- "I'm a backend engineer, new to the frontend code in this repo"
- "We track bugs in Linear project PLATFORM"
- "Don't add trailing summaries to your responses, I read the diffs"

Claude Code writes these to memory files and indexes them automatically. You can check what's been saved by reading the MEMORY.md index file in your project's memory directory.

### Using Both Together

The most effective setup uses both layers with clear separation. [Effective prompting practices](/blog/how-to-effectively-prompt-a-claude-code) for Claude Code include maintaining both a strong CLAUDE.md foundation and letting memory accumulate naturally over time.

**CLAUDE.md handles the constants**: build commands, coding standards, architectural rules, workflow gates. These change infrequently and apply to everyone.

**Memory handles the variables**: your current focus area, your preferences, corrections to Claude Code's approach, pointers to resources you reference often. These change frequently and are personal.

The result is a Claude Code instance that follows team rules (from CLAUDE.md) while adapting to your individual working style (from memory). Neither layer alone achieves this — CLAUDE.md without memory gives you a generic team tool, and memory without CLAUDE.md gives you a personalized tool that might violate team conventions.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence layer when:

- **You're working on a team** and need consistent Claude Code behavior across all members. CLAUDE.md is the only persistence mechanism that travels with the repo and applies uniformly.
- **You're setting up a new project** and want Claude Code to follow specific conventions from the first session. CLAUDE.md works immediately with no warm-up period.
- **Your constraints are non-negotiable** — build commands, security rules, testing requirements. These belong in the instruction layer, not the learning layer.
- **You want reviewable, auditable AI instructions** that go through the same PR process as your code. This matters for teams that need to understand and control how AI tools behave in their codebase.

CLAUDE.md is also the right choice for CI/CD and automated environments. If you run Claude Code in a headless pipeline (code review bots, automated refactoring), there's no conversation for memory to learn from. CLAUDE.md is the only way to provide project context in these scenarios. Teams building on the [skills system](/blog/5-claude-code-skills-i-use-every-single-day) use CLAUDE.md as the base layer that skills reference and extend.

## When to Choose Claude Memory

Choose Claude Memory as your primary persistence layer when:

- **You're a solo developer** and the distinction between "team instructions" and "personal preferences" doesn't apply. Memory captures everything you need without the overhead of maintaining a separate file.
- **Your context is conversational** — you learn things during debugging sessions, discover undocumented behaviors, build up tribal knowledge through working with the codebase. Memory captures this naturally.
- **Your preferences are personal** and shouldn't affect teammates — explanation verbosity, commit message style, whether you want Claude Code to ask before committing or just do it.
- **You work across many projects** and want Claude Code to remember your general working style regardless of which repo you're in. Global memory (user-type memories) carries across all projects.

Memory particularly shines for long-running engagements with a codebase. After weeks of daily sessions, Claude Code builds a nuanced understanding of how you work, what you care about, and what mistakes it should avoid with you specifically. This accumulated context is difficult to replicate in a static instruction file because much of it is implicit — the kind of thing you'd explain to a new pair programmer over the first week of working together, not the kind of thing you'd write in a README.

## Verdict

**Start with CLAUDE.md.** It's the higher-leverage tool: write it once, and every team member's Claude Code sessions immediately improve. It's version-controlled, reviewable, and portable across machines and environments. If you're only going to invest in one persistence layer, CLAUDE.md gives you the most return with the least ongoing effort.

**Then let memory build naturally.** Don't try to front-load your memory profile — it's designed to accumulate over time as you work. After a few weeks of regular Claude Code use, you'll have a memory profile that captures your role, preferences, and working patterns without any deliberate maintenance.

**The combination is where the real value lives.** CLAUDE.md keeps Claude Code aligned with team standards. Memory keeps Claude Code aligned with you. Together, they give you an AI coding assistant that follows the rules and knows your style — which is what makes the difference between a generic tool and a genuinely useful one. For a deeper look at how both layers fit into the full context architecture, read our [Claude Code memory system breakdown](/blog/claude-code-memory).

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md is loaded as authoritative system-level instructions, while memory is loaded as contextual knowledge. If a memory entry contradicts a CLAUDE.md rule, the CLAUDE.md rule wins. This is by design — team-level instructions should always take precedence over individually accumulated context.

### Does Claude Memory work in CI/CD environments?

Not in a meaningful way. Memory requires interactive conversations to build up, and CI/CD environments typically run Claude Code in headless, single-session mode. For automated pipelines, rely on CLAUDE.md and skill files for all project context. Memory is designed for developer workstations with ongoing interactive sessions.

### How do I see what Claude Memory has saved?

Check the `MEMORY.md` file in your project's memory directory (located at `~/.claude/projects/<project-hash>/memory/MEMORY.md`). This index lists all saved memories with one-line descriptions. You can also read individual memory files in the same directory for full details, or ask Claude Code directly: "What do you remember about me?"

### Should I put coding standards in CLAUDE.md or rely on memory to learn them?

Always put coding standards in CLAUDE.md. Standards are team-wide rules that should apply deterministically from the first session, not personal observations that accumulate over time. Memory might eventually learn your standards through repeated corrections, but that means every new team member (and every new machine) starts from zero. CLAUDE.md makes standards immediate and universal.

### Can I share my Claude Memory profile with teammates?

Not directly through built-in mechanisms. Memory files are local markdown files, so you could technically copy them, but they're designed to be personal. If you find yourself wanting to share a memory entry, that's usually a signal it belongs in CLAUDE.md instead — shared knowledge should live in the shared instruction layer.
