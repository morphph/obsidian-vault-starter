**Codex runs each agent in a cloud sandbox.** When you submit a task, Codex clones your repository into an isolated environment, installs dependencies via your setup script, and lets the agent work without risk of corrupting your local state. This is powerful for batch operations — submit 20 refactoring tasks and let them all run simultaneously in the cloud. Each agent has no awareness of what other agents are doing, which prevents coordination but guarantees isolation.

The tradeoff is latency and context. Each Codex agent starts cold: it must clone the repo, install dependencies, and build context from scratch. There is no shared memory between tasks. If task B depends on the output of task A, you must wait for A to complete, merge its changes, and then submit B against the updated repository.

**Claude Code runs agents locally with explicit coordination primitives.** The parent agent can spawn a sub-agent, pass it a detailed prompt with full context about what has been discovered so far, and receive structured results back. Sub-agents can be typed — an Explore agent is optimized for read-only codebase search, a Plan agent designs implementation strategies, and custom agents can be configured for any specialized role.

```typescript
// Claude Code: Spawn a background research agent while continuing work
Agent({
  description: "Investigate auth module",
  subagent_type: "Explore",
  prompt: "Find all files that import from src/auth/. List each file and which auth exports it uses.",
  run_in_background: true
});
// Parent agent continues working immediately
```

This coordination model means Claude Code can handle dependent tasks within a single session — agent A researches, parent agent synthesizes, agent B implements based on combined findings. See practical [subagent workflow examples](/blog/claude-code-subagents-examples) for patterns that take advantage of this.

The tradeoff is resource contention. All agents share your local machine's CPU, memory, and filesystem. Running too many concurrent agents degrades performance, and without worktree isolation, file-level conflicts are possible.

## Custom Agent Configuration: Detailed Analysis

Defining "how an agent should behave" is where these platforms diverge most in developer experience.

**Codex custom agents** are configured at three levels. First, the system prompt sets the agent's persona and constraints — you control this through the Codex interface or API. Second, an `AGENTS.md` file (or equivalent) in your repository provides project-specific instructions that every agent instance reads on startup. Third, a setup script (`setup.sh` or similar) installs tools, sets environment variables, and configures the sandbox before the agent begins work.

This layered approach is straightforward but static. Each agent instance reads the same configuration files. You cannot easily tell one agent instance to behave as a "security reviewer" while another acts as a "test writer" within the same batch submission — differentiation happens at the prompt level, not through a structured agent-type system.

**Claude Code custom agents** use a richer configuration stack. The [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) provides multiple layers of customization:

- **CLAUDE.md**: Project-level context and constraints that all agents inherit
- **SKILL.md files**: Task-specific instruction sets (e.g., `skills/security-review/SKILL.md`) that can be loaded per agent
- **Hooks**: Deterministic shell commands that execute on specific events (pre-edit, post-commit) — these apply to all agents, enforcing guardrails
- **MCP servers**: External tool integrations that extend what agents can do — database access, API calls, browser automation
- **Agent type parameter**: Built-in specializations (Explore, Plan) with pre-configured tool access and behavioral constraints

```yaml
# Claude Code: Custom agent via SKILL.md
# skills/api-reviewer/SKILL.md
---
name: API Review Agent
description: Reviews API endpoint changes for backwards compatibility
---
Check every modified API route for:
1. Breaking changes to request/response schemas
2. Missing versioning headers
3. Undocumented query parameters
```

This composable system means you can create highly specialized agents that combine project context, task instructions, tool access, and behavioral constraints — all defined in version-controlled files that travel with your repository. For practical patterns, our coverage of [Claude Code skills in daily use](/blog/5-claude-code-skills-i-use-every-single-day) shows how teams build these agent configurations.

## Parallel Workflow Patterns: Detailed Analysis

How you structure parallel work differs significantly between the two platforms, and the right choice depends on your task decomposition style.

**Codex parallel pattern: Task farming.** You decompose work upfront into independent units, submit them all, and collect results. This works exceptionally well for:

- Running the same refactoring across multiple modules
- Generating tests for independent components
- Applying a style migration (e.g., converting callback patterns to async/await) across files
- Reviewing multiple PRs simultaneously

Each task runs to completion independently. The [Codex VS Code extension](/blog/codex-vscode) provides a task queue view where you can monitor progress across all running agents. The cloud-native model means you are not limited by local compute — submitting 50 parallel tasks is architecturally no different from submitting one.

**Claude Code parallel pattern: Coordinated agent teams.** You spawn sub-agents from a parent session, with the parent acting as an orchestrator that can synthesize results and adapt the plan in real time. This works better for:

- Research-then-implement workflows (explore first, code second)
- Tasks with dependencies between sub-steps
- Interactive development where you want to steer the agents mid-flight
- Complex refactoring where changes in one area inform changes in another

```
// Claude Code: Coordinated parallel research
// Parent sends both agents simultaneously
Agent({ description: "Find all API routes", subagent_type: "Explore", ... })
Agent({ description: "Find all test files", subagent_type: "Explore", ... })
// Results return → parent synthesizes → spawns implementation agents
```

The [multi-agent workflow patterns](/blog/con-u-pour-des-workflows-multi-agents) emerging in 2026 increasingly favor this coordinated approach for complex tasks, while the task-farming model remains superior for high-volume, independent operations.

## Frequently Asked Questions

### Can Codex agents communicate with each other during execution?

No. Each Codex task runs in an isolated sandbox with no inter-agent communication channel. Tasks are independent by design. If you need coordination between agents, you must handle it at the submission layer — waiting for one task to complete before submitting dependent follow-ups.

### How many sub-agents can Claude Code run simultaneously?

Claude Code does not enforce a hard limit on concurrent sub-agents, but practical performance depends on your local machine's resources. Most developers find 3-7 concurrent agents workable, with diminishing returns beyond that as CPU, memory, and API rate limits become bottlenecks. Using worktree isolation helps prevent filesystem conflicts.

### Can I define custom agent types in Codex like Claude Code's Explore or Plan agents?

Codex does not have a built-in typed agent system. You achieve similar specialization through prompt engineering — writing different system prompts for "reviewer," "implementer," or "researcher" roles. The behavior differentiation is in the prompt, not in a structural agent-type parameter with pre-configured tool access restrictions.

### Is one platform cheaper for multi-agent workflows?

Both use token-based billing, so cost scales with how much each agent reads and writes. Codex may be more expensive per task due to cold-start overhead — each sandbox must re-read the repository and build context from scratch. Claude Code sub-agents share the parent's context, potentially reducing redundant token usage. Actual costs vary significantly by task complexity and repository size.

### Can I use both Codex and Claude Code on the same project?

Yes. They operate independently — Codex tasks run in the cloud against your repository, while Claude Code runs locally in your terminal. Many teams use Claude Code for interactive development and Codex for batch operations, reviewing all outputs through their standard PR process.
