**CLAUDE.md** defines project-level context: coding standards, architecture decisions, deployment constraints, and any instructions that should apply to every Claude Code session in this repo. It's checked into git and travels with the code.

**SKILL.md files** define task-specific agent behaviors. Each skill is a markdown file in the `skills/` directory that encodes how Claude Code should approach a particular type of work — writing newsletters, generating tests, reviewing security, or producing SEO content. Skills are composable: Claude Code loads the relevant skill based on the task type and combines it with the project-level CLAUDE.md context.

**Hooks** add deterministic automation: shell commands that execute at specific points in Claude Code's lifecycle (before/after tool calls, on session start, etc.). Hooks aren't AI — they're conventional scripts that enforce constraints the AI shouldn't be trusted to remember on its own, like running a linter before every commit or validating schema changes against a migration checklist. See our [guide to Claude Code hooks](/blog/claude-code-hooks-a-complete-guide-to-automating-your-ai-coding-workflow) for implementation patterns.

The result is a custom agent system that's **version-controlled, team-shared, and composable**. New team members get the same agent behaviors by cloning the repo. Skills evolve through normal code review. Hooks enforce invariants that the AI alone might miss. For organizations that treat [AI coding skills as engineering assets](/blog/5-claude-code-skills-i-use-every-single-day), this is a significant operational advantage.

## Multi-Agent Workflows: Practical Patterns

Understanding the architecture is one thing — knowing which patterns work in practice is another. Here's how real multi-agent workflows play out on each platform.

### Codex: Parallel Tasks via API

Since Codex doesn't support intra-task subagents, multi-agent workflows happen at the **task level** — you submit multiple independent tasks through the API or web interface. Each task gets its own container, runs independently, and produces its own output (typically a PR or diff).

This pattern works well for embarrassingly parallel work: "fix the 12 failing tests in these 12 test files" becomes 12 separate Codex tasks. Each runs in isolation, produces a targeted fix, and you review the results. There's no coordination overhead because there's no coordination needed.

Where it breaks down is **sequential dependency**. If task B depends on task A's output — "first redesign the data model, then update all queries to match" — you can't chain them within Codex. You'd need external orchestration: a CI pipeline, a script that polls for task completion and submits the next task, or manual sequencing.

### Claude Code: Orchestrated Agent Teams

Claude Code's subagent system handles both parallel and sequential workflows natively. The parent session acts as an orchestrator:

**Parallel pattern**: Launch multiple agents simultaneously for independent work. Example: three Explore agents search for usages of different deprecated APIs while a Plan agent designs the migration. Results feed back to the parent session, which synthesizes a unified implementation plan.

**Sequential pattern**: One agent's output feeds the next. An Explore agent maps the codebase, a Plan agent uses those findings to design the approach, and general-purpose agents implement the plan in phases. The parent session maintains full context across the chain.

**Worktree isolation pattern**: Each implementation agent works in its own git worktree, making changes on separate branches. The parent session reviews diffs, resolves conflicts, and merges results. This prevents agents from stepping on each other's changes — a real problem in large refactoring tasks.

For a detailed breakdown of these patterns with code examples, see our [Claude Code subagent examples guide](/blog/claude-code-subagents-examples). The [Claude Code extension stack overview](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) explains how skills, hooks, and agents compose into complete workflows.

## Developer Experience and Onboarding

### Codex: Minimal Friction, Shallow Customization

Getting started with Codex takes minutes. Sign in through ChatGPT or the API, connect a repository, and submit a task. The return comes through compounding: every skill you write, every hook you configure, and every CLAUDE.md refinement makes all future sessions more effective. The [Agent SDK](/glossary/agent-sdk) enables programmatic agent orchestration beyond the interactive CLI, opening the door to CI/CD integration and automated pipeline agents. Teams that treat these configurations as engineering assets — reviewed, tested, and iterated — report significant productivity gains over time.

## Cost Considerations

Both platforms use usage-based pricing, but the cost profiles differ based on how you structure work.

**Codex** charges per task through OpenAI's API tier system. Each task spins up a container, which means compute costs include environment provisioning time. Submitting 10 parallel tasks costs roughly 10x a single task. OpenAI offers [free Codex access for open source maintainers](/blog/codex-for-open-source) and [$100 in credits for students](/blog/codex-for-students), which can offset initial experimentation costs.

**Claude Code** charges per token through Anthropic's API. Subagents consume additional tokens, but they share the local environment — no container provisioning overhead. Launching an Explore subagent for a quick codebase search is significantly cheaper than a full Codex task container. However, complex multi-agent sessions with extended thinking can accumulate substantial token costs.

**Decision rule**: For infrequent, self-contained tasks (bug fixes, one-off refactors), Codex's per-task pricing is straightforward and predictable. For continuous, complex workflows with frequent subagent use, Claude Code's token-based model typically costs less because lightweight operations (search, planning) are cheap.

**Choose Claude Code** if your workflow demands multi-agent orchestration, custom agent behaviors, or deep codebase integration. Claude Code is the right fit when:

Claude Code rewards investment in configuration. Teams that build out their skill libraries and hook automation report that the platform becomes more effective over time — a compounding return that Codex's simpler model doesn't offer.

## Verdict

The choice between Codex and Claude Code for subagent and custom agent workflows comes down to **orchestration depth vs. operational simplicity**.

**If you need native multi-agent orchestration** — parallel subagents, typed agent roles, worktree isolation, and composable skill configurations — **Claude Code is the clear choice**. Its agent team system is purpose-built for complex, multi-step engineering workflows where a single agent hitting a single task isn't enough.

**If you need sandboxed simplicity** — submit a task, get a PR, no local setup required — **Codex delivers with minimal friction**. Its cloud-based model trades orchestration depth for operational ease, and for teams doing independent, self-contained tasks, that tradeoff is worth it.

For many teams, the best approach is using both: Codex for quick, independent tasks submitted through the API, and Claude Code for complex orchestration work that benefits from subagents, skills, and hooks. The platforms don't compete — they serve different points on the complexity spectrum.

Codex custom agents are configured through task-level natural language instructions and setup scripts — simple and effective, but not version-controlled or composable. Yes. A practical pattern is using Claude Code locally for complex orchestration — multi-file refactoring with parallel subagents, skill-driven workflows, hook-enforced quality gates — and Codex for independent, fire-and-forget tasks submitted through the API. The platforms address different workflow needs and complement each other well.
