**Use both. They solve different problems.**

CLAUDE.md is your project's governance layer — deterministic, team-shared, version-controlled rules that define how Claude Code behaves for everyone. Memory is your personal adaptation layer — accumulated context that makes Claude Code increasingly effective for *you* across sessions. Trying to replace one with the other creates friction: a CLAUDE.md stuffed with personal preferences clutters the team's instruction set, and Memory entries that try to encode build commands are fragile and unauditable.

**The decision rule is simple: if it's about the project, write it in CLAUDE.md. If it's about you or the current state of work, let Memory handle it.** Start every new project by writing a lean CLAUDE.md with build commands, quality gates, and known gotchas. Then work normally — Memory will learn your preferences, track your project context, and adapt to your corrections automatically.

For teams, CLAUDE.md is non-negotiable. For individuals, Memory alone can be sufficient. For anyone serious about getting the most out of Claude Code, both systems working together deliver the best results. Read our guide on [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) to see how CLAUDE.md and Memory fit into a broader prompting strategy.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md rules?

No. CLAUDE.md instructions take precedence over Memory in all cases. If CLAUDE.md says "run tests before every commit" and a Memory entry suggests skipping tests for speed, Claude Code follows CLAUDE.md. Memory provides contextual suggestions; CLAUDE.md provides hard rules. The hierarchy is explicit: CLAUDE.md is the governance layer, Memory is the adaptation layer.

### Do I need to manually create Memory entries?

Not usually. Claude Code creates Memory entries automatically when it detects information worth persisting — role descriptions, corrections, project decisions, external references. You can trigger memory creation explicitly by saying "remember this" or "save this for next time," but most useful memories emerge naturally from conversations. Periodically reviewing your `MEMORY.md` index helps prune stale entries.

### Is CLAUDE.md shared with my team automatically?

Only if you check it into your git repository, which is the recommended practice. A `CLAUDE.md` file in your repo root travels with the code — every developer who clones the repo gets the same AI behavior rules. The separate `~/.claude/CLAUDE.md` (global) and `.claude/CLAUDE.md` (project-private) files are not shared and not version-controlled.

### Can I use CLAUDE.md without Claude Code?

CLAUDE.md is specifically designed for Claude Code's runtime — it's loaded and interpreted by the Claude Code agent at conversation start. Other AI coding tools don't read CLAUDE.md files. However, the practice of maintaining a project instruction file is broadly useful, and some teams keep CLAUDE.md as general project documentation even if not all members use Claude Code.

### How do I migrate from Memory-only to using both systems?

Review your `MEMORY.md` index and identify entries that are project-intrinsic rather than person-specific. Build commands, coding standards, known gotchas, and architectural constraints should move to CLAUDE.md. Personal preferences, role context, and feedback corrections should stay in Memory. After migrating, delete the moved Memory entries to avoid duplication.
