# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** 这不是二选一的问题。**Claude Memory** 是自动的、个人的、跨会话持久化的记忆系统——它记住你是谁、你喜欢什么。**CLAUDE.md** 是手动的、项目级的、版本控制的规则手册——它告诉 Claude 这个项目该怎么做。前者解决「我不想重复自我介绍」的问题，后者解决「团队里每个人用 Claude 的方式不一致」的问题。大多数专业开发者两套都在用。

## Overview：Claude Memory

**Claude Memory** 是 Claude 的自动持久化记忆系统，能在多次对话之间保留用户偏好、项目背景和工作习惯等上下文信息。它解决的核心问题是：每次开新对话都要从零开始解释自己是谁、在做什么。

在 Claude Code 中，Memory 以文件形式存储在 `~/.claude/projects/` 目录下，按项目分隔。系统会自动将相关记忆注入到新对话的上下文中。根据 [Anthropic 的记忆升级公告](/zh/blog/anthropic-claude-memory-upgrades-importing)，2026 年初 Memory 功能已向免费用户开放，并支持从 ChatGPT 导入数据。

Memory 分为几种类型：用户记忆（你的角色和偏好）、反馈记忆（你对 Claude 工作方式的修正）、项目记忆（正在进行的工作上下文）和引用记忆（外部资源的指针）。这套分类设计让 Claude 能在不同维度上「认识」你。

## Overview：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级配置文件，放在仓库根目录下，随代码一起版本控制。它的作用是给 Claude 一份「项目手册」——编码规范、架构决策、构建命令、质量门禁等，所有参与项目的人（和 AI）都遵循同一套规则。

CLAUDE.md 是显式的、确定性的。你写什么，Claude 就遵循什么。没有推断，没有自动学习。它更接近传统的 `.editorconfig` 或 `CONTRIBUTING.md`——只不过读者是 AI。详细的 CLAUDE.md 实践可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

CLAUDE.md 支持分层：全局的 `~/.claude/CLAUDE.md` 定义个人全局偏好，项目根目录的 `CLAUDE.md` 定义项目规则，子目录还可以放更细粒度的配置。这种分层设计让你能在不同粒度上控制 Claude 的行为。

| **覆盖范围** | 跨项目持久化 | 单项目内生效 | Memory |

## 核心差异：显式规则 vs 隐式学习

这是两套系统最本质的区别。CLAUDE.md 是**声明式**的——你写下规则，Claude 每次都遵循。「commit 前必须跑 `npm test`」「不要在 pipeline 脚本里 import Next.js 模块」这些硬性约束，CLAUDE.md 是唯一可靠的方式。

Memory 是**归纳式**的——Claude 从交互中观察、总结、记住。「这个用户是高级工程师，偏好简洁回答」「上次他纠正我说不要在回复末尾加总结」这些软性偏好，Memory 处理得更自然。

实际开发中，混淆这两者会导致问题。把团队规范放进个人 Memory？其他成员看不到，行为不一致。把个人偏好写进 CLAUDE.md？每个人的习惯不同，代码审查时会起冲突。

关于 Claude Code 的完整扩展架构（包括 Skills、Hooks、Agents、MCP 四层），可以参考 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 实际工作流：两套系统如何协作

理解了核心差异，来看一个典型的日常开发场景。

假设你是一个全栈工程师，同时维护三个项目。你的 Memory 记住了：你偏好 TypeScript、commit message 用中文、不喜欢过度设计。这些偏好跨项目生效。

每个项目各有一份 CLAUDE.md：项目 A 要求用 Vitest 跑测试，项目 B 用 Jest，项目 C 根本没有测试（是个原型）。这些规则只在各自项目内生效。

当你在项目 A 里让 Claude 帮你重构一个模块时，Claude 同时读取了：
1. 你的全局 Memory（知道你偏好简洁方案）
2. 项目 A 的 CLAUDE.md（知道重构后必须跑 Vitest）
3. 历史反馈记忆（知道你上次说过不要在回复末尾加总结）

三层上下文叠加，Claude 的行为才真正符合你的期望。如果你想深入了解这种分层记忆的工作机制，推荐阅读 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 常见误区与最佳实践

**误区一：用 CLAUDE.md 替代 Memory。** 有人把所有偏好都写进 CLAUDE.md，包括「我是后端工程师」「我喜欢用函数式风格」。问题是这些信息跟项目无关，不应该进版本控制，也不该强加给团队其他成员。

**误区二：完全依赖 Memory，不写 CLAUDE.md。** Memory 是概率性的——Claude 判断某条记忆「相关」才会调用。关键的项目规则（构建命令、质量门禁、架构约束）必须放在 CLAUDE.md 里，确保每次会话都加载。

**误区三：把代码模式和架构写进 Memory。** Memory 文件自己就说了——代码模式、项目架构、文件路径这些能从代码中直接读取的信息，不应该存进 Memory。它们会过时，而代码本身是「活的」。

**最佳实践：**

- **CLAUDE.md 放规则**：构建命令、测试要求、代码规范、已知陷阱、文档更新规则
- **Memory 放偏好**：沟通风格、技术背景、工作习惯、常用工具链
- **Skills 放流程**：重复性任务的标准化指令（比如写 newsletter、做代码审查），详见 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)

## 该选哪个？决策指南

| 你的场景 | 推荐方案 |
|---------|---------|
| 个人项目，只有你一个人用 Claude | Memory 为主，轻量 CLAUDE.md 放构建命令即可 |
| 团队项目，多人用 Claude Code | CLAUDE.md 必须写，Memory 自然积累 |
| 刚开始用 Claude Code | 先写 CLAUDE.md（5 分钟搞定），Memory 会自动积累 |
| 跨多个项目工作 | 全局 Memory 管偏好，每个项目各写 CLAUDE.md |
| 想让 Claude 的行为 100% 可预测 | 只依赖 CLAUDE.md，Memory 作为补充 |

核心判断标准只有一个：**这条信息是属于「你」还是属于「项目」？** 属于你的放 Memory，属于项目的放 CLAUDE.md。

## Verdict

**Claude Memory 和 CLAUDE.md 不是竞争关系，是互补关系。** 如果只能选一个，先写 CLAUDE.md——它是确定性的、可审计的、团队共享的，对代码质量的保障更直接。Memory 则会在你使用过程中自然积累，逐渐让 Claude 更了解你的工作方式。

对于认真使用 Claude Code 的开发者，两套系统都不可少。CLAUDE.md 是底线（没有它 Claude 就是在盲人摸象），Memory 是上限（有了它 Claude 从「工具」变成「搭档」）。如果你还没有开始使用这两套系统，从 CLAUDE.md 开始——打开终端，在项目根目录创建一个 `CLAUDE.md`，写入构建命令和三条最重要的规则。五分钟，你会立刻感受到区别。

## Frequently Asked Questions

### Claude Memory 会不会记错东西？
Memory 是概率性的——Claude 根据对话推断应该记什么。偶尔会记录不准确或过时的信息。你可以通过 `~/.claude/projects/` 目录手动查看和编辑 Memory 文件，删除不需要的记忆。关键项目规则不要依赖 Memory，放进 CLAUDE.md 更可靠。

### CLAUDE.md 太长会影响性能吗？
会。CLAUDE.md 的内容每次会话都会注入到上下文窗口中，过长的文件会占用宝贵的 token 额度。建议控制在 500 行以内，把详细的技能指令拆分到 `skills/*/SKILL.md` 文件中，按需加载而不是全量注入。

### 团队成员的 Memory 会互相干扰吗？
不会。Memory 是严格 per-user 的，存储在各自的 `~/.claude/` 目录下。这也是为什么团队规范必须放在 CLAUDE.md 里——它是唯一所有人共享的上下文来源。

### 能不能把 ChatGPT 的记忆导入到 Claude Memory？
可以。Anthropic 在 2026 年初推出了记忆导入功能，支持从 ChatGPT 导入历史对话中积累的上下文。具体操作方式见 [Claude 记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)。
