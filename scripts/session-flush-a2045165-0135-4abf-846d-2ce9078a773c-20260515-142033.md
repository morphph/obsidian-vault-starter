# Claude Memory vs CLAUDE.md：两套记忆系统，到底该用哪个？

**TL;DR：** **CLAUDE.md** 是你主动编写的项目规则手册，适合团队协作和工程规范的统一执行；**Claude Memory** 是 AI 自动积累的上下文笔记，适合个人偏好和跨会话知识的沉淀。两者不是竞争关系，而是互补——规则放 CLAUDE.md，经验放 Memory。搞反了，两个都用不好。

## 概述：Claude Memory

**Claude Memory** 是 Anthropic 为 Claude 产品线推出的持久化记忆功能，让 AI 在多轮对话之间保留对用户的了解。在 [Claude Code](/zh/blog/claude-code-memory) 中，这套机制被称为 **Auto Memory**——Claude Code 在工作过程中自动将你的偏好、反馈、项目背景写入 `~/.claude/projects/<project>/memory/` 目录下的 Markdown 文件。

核心特征：AI 主导写入。你不需要手动编辑这些文件（虽然可以），Claude Code 会根据对话内容自动判断哪些信息值得记住。比如你说「不要在测试里 mock 数据库」，它会把这条反馈存下来，下次自动遵守。

Memory 的数据类型包括四类：用户画像（user）、行为反馈（feedback）、项目上下文（project）、外部引用（reference）。每条记忆都有 frontmatter 元数据，支持交叉引用。Anthropic 在 2026 年初对 Memory 做了 [重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，免费用户也可以使用，并且支持从 ChatGPT 导入历史数据。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件，放在仓库根目录，随代码一起提交到 Git。它的作用是告诉 Claude Code：这个项目的规则是什么、构建命令怎么跑、代码风格有什么要求、哪些坑不能踩。

核心特征：人类主导编写。CLAUDE.md 是你写的，不是 AI 写的。它是确定性的规则手册，团队里每个人拉下代码就能共享同一套 AI 行为标准。你也可以在 `~/.claude/CLAUDE.md` 放全局指令，覆盖所有项目。

CLAUDE.md 的进阶形态是 [SKILL.md](/zh/blog/9-principles-writing-claude-code-skills)——针对特定任务（写测试、生成内容、做 code review）的结构化提示词文件，放在 `skills/` 目录下。两者配合，构成了 Claude Code 的 [七层架构](/zh/blog/claude-code-seven-programmable-layers) 中最核心的两层。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|---------------|-----------|--------|
| **写入方式** | AI 自动写入 | 人类手动编写 | 各有适用场景 |
| **存储位置** | `~/.claude/projects/*/memory/` | 项目根目录（Git 跟踪） | CLAUDE.md |
| **团队共享** | 仅本地，不进 Git | 随代码提交，团队共享 | CLAUDE.md |
| **内容类型** | 偏好、反馈、项目背景、外部引用 | 构建命令、代码规范、架构约束 | 平手 |
| **持久性** | 跨会话保留 | 跨会话保留 | 平手 |
| **可控性** | AI 判断什么值得记 | 你决定写什么 | CLAUDE.md |
| **维护成本** | 低（自动） | 中（需要人工更新） | Memory |
| **适用范围** | 个人工作流 | 团队工程规范 | 各有适用场景 |

## 机制差异：确定性 vs 概率性

这是两套系统最本质的区别。

**CLAUDE.md 是确定性的。** 每次 Claude Code 启动，它都会读取 CLAUDE.md 的全部内容并注入到系统提示词中。你写「commit 前必须跑 `npm test`」，它每次都会执行。规则就是规则，没有模糊空间。这种确定性让它天然适合团队场景——你不想让 AI 对不同成员执行不同的代码规范。

**Memory 是概率性的。** Claude Code 根据当前对话的语义相关性决定是否调取某条记忆。如果你存了「用户偏好 Vim 键位」，在讨论编辑器配置时它大概率会想起来，但在讨论数据库迁移时可能不会。记忆的召回不是 100% 的，而且随着记忆条目增多，相关性排序可能出现偏差。

实际影响：关键的工程规范（构建流程、测试要求、安全约束）必须放 CLAUDE.md，因为你需要它们被 100% 执行。用户偏好和历史上下文放 Memory 更合适，因为它们不需要每次都加载，按需召回反而更高效。

## 团队协作：谁能看到什么

这是选型时最容易被忽略的维度。

**CLAUDE.md 是团队工具。** 它在 Git 仓库里，新成员 clone 代码就自动继承了所有 AI 行为规范。代码审查时，团队可以一起迭代 CLAUDE.md 的内容，就像迭代 `.eslintrc` 一样。Boris Cherny（Claude Code 创建者）在他的 [最佳实践](/zh/blog/how-to-effectively-prompt-a-claude-code) 中反复强调：把 CLAUDE.md 当成你的 AI 编码标准文档来维护。

**Memory 是个人工具。** 它存在本地文件系统，不进 Git，不跨机器同步。你积累的记忆只属于你自己的 Claude Code 实例。这意味着同一个项目里，两个开发者的 Claude Code 可能因为各自的 Memory 积累而表现出不同的行为——这在个人项目里无所谓，但在团队里可能导致一致性问题。

## 实战场景：什么放哪里

### 放 CLAUDE.md 的内容

```markdown
# 构建命令
npm run build    # 生产构建
npm test         # 必须在 commit 前通过

# 代码规范
- 使用 TypeScript strict mode
- 禁止使用 any 类型
- 测试覆盖率不低于 80%

# 已知陷阱
- 不要在 pipeline 脚本里直接 import Next.js 模块
```

这些是**规则**——团队共识，不容商量，每次都要执行。

### 放 Memory 的内容

- 用户 A 偏好简洁的 commit message，不要 emoji
- 用户 B 习惯先写测试再写实现
- 项目正在进行 auth 模块重构，预计下周完成
- bug 追踪在 Linear 的 INGEST 项目里

这些是**上下文**——个人化的、随时间变化的、不需要团队统一的信息。

### 不应该放在任何一个里的内容

- 代码片段（直接放在代码里）
- Git 历史摘要（用 `git log` 查）
- 临时调试记录（当前会话结束就没用了）

## 进阶用法：两者配合的最佳实践

最高效的做法不是二选一，而是让两者各司其职。

**第一层：CLAUDE.md 定义「法律」。** 构建流程、质量门禁、代码规范、安全约束——这些写进 CLAUDE.md，确保 AI 每次都遵守。配合 [SKILL.md](/zh/blog/5-claude-code-skills-i-use-every-single-day) 进一步细化特定任务的行为标准。

**第二层：Memory 积累「判例」。** 你在工作中的反馈（「不要 mock 数据库」「PR 合并前先跑 e2e 测试」）被 Memory 自动记录，成为 AI 理解你个人偏好的依据。这些反馈如果被验证为团队共识，可以升级到 CLAUDE.md。

**第三层：定期清理 Memory。** Memory 会随时间积累大量条目，其中一些可能已经过时（项目背景变了、技术选型改了）。建议每月检查一次 `memory/` 目录，删除不再准确的记忆。Claude Code 在调取记忆时会尝试验证其时效性，但人工审查仍然必要。

这种分层模式在 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 的框架下效果最好——CLAUDE.md 和 Skills 处理确定性指令，Memory 和 Hooks 处理动态上下文。

## 选型建议

**选 CLAUDE.md，如果你需要：**
- 团队统一的 AI 行为标准
- 100% 确定性的规则执行
- 可版本控制、可代码审查的指令
- 新成员开箱即用的 AI 配置

**选 Memory，如果你需要：**
- 个人偏好的自动记录
- 跨会话的项目上下文沉淀
- 低维护成本的上下文积累
- AI 自主学习你的工作习惯

**两个都用（推荐），如果你在做：**
- 长期维护的团队项目——CLAUDE.md 管规范，Memory 管个人偏好
- 复杂的多模块项目——CLAUDE.md 定义架构约束，Memory 记录各模块的历史决策背景

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。两者作用于不同层面：CLAUDE.md 提供确定性的项目规则，Memory 提供个人化的上下文补充。Claude Code 在每次会话中会同时读取两者，不存在冲突。

### Memory 里的内容会覆盖 CLAUDE.md 的规则吗？

不会。CLAUDE.md 的内容作为系统提示词的一部分，优先级高于 Memory。如果 Memory 中的某条记忆与 CLAUDE.md 的规则矛盾，Claude Code 会遵循 CLAUDE.md。

### CLAUDE.md 需要多久更新一次？

每次修改构建流程、代码规范或架构约束时就应该更新。把它当成 `.eslintrc` 或 `tsconfig.json` 一样对待——代码变了，配置也要跟着变。Claude Code 的文档规则建议在修改相关代码时同步更新 CLAUDE.md。

### 如何查看 Claude Code 记住了什么？

直接查看 `~/.claude/projects/<project-path>/memory/` 目录下的 Markdown 文件。每个文件都有 frontmatter 标注类型和描述，`MEMORY.md` 是索引文件。你可以随时编辑或删除任何记忆条目。
