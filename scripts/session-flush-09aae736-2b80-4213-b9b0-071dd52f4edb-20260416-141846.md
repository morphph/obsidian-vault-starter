# Claude Memory vs CLAUDE.md：两套记忆系统，到底该用哪个？

**TL;DR：** **CLAUDE.md** 是手动编写的项目规则手册，签入 Git，团队共享，适合定义编码规范和架构约束。**Claude Memory（Auto Memory）** 是跨会话自动记录的个人笔记本，存储用户偏好和反馈，不进版本控制。两者不是竞争关系——**CLAUDE.md 管"项目该怎么做"，Memory 管"你喜欢怎么做"**。大多数团队应该同时使用两者。

## 概述：Claude Memory

Claude Memory（也叫 Auto Memory）是 Claude Code 的自动持久化记忆系统。它在对话过程中观察你的工作习惯、偏好和反馈，自动将有价值的信息写入 `.claude/projects/` 目录下的 Markdown 文件。

Memory 存储四种类型的信息：**用户画像**（你的角色和技术背景）、**反馈记录**（你对 AI 行为的纠正和确认）、**项目动态**（正在进行的工作和截止日期）、**外部引用**（外部系统的指针，比如 Linear 项目或 Grafana 面板地址）。这些记忆跨会话持久存在，让 Claude Code 在新对话中能延续之前的上下文。

核心特征：自动触发、个人专属、不进 Git。关于记忆系统的完整机制，参见我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是放在项目根目录（或 `~/.claude/` 全局目录）的 Markdown 文件，手动编写，用于向 Claude Code 传达项目级别的指令和约束。它是确定性的——每次会话都会被加载，内容由你完全控制。

CLAUDE.md 的典型内容包括：构建命令（`npm run build`、`npm test`）、代码风格规范、架构约束（"不要在 pipeline 脚本里直接 import Next.js 模块"）、工作流规则（"commit 前必须通过所有测试"）。它签入版本控制，团队所有成员共享同一份规则。

在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中，CLAUDE.md 属于"项目上下文层"，是整个指令体系的基石。

## 核心差异一：确定性 vs 自适应

这是两套系统最根本的区别。

**CLAUDE.md 是确定性的**。你写下"commit 前必须运行 `npm test`"，Claude Code 每次都会遵守。规则不会因为对话内容而改变，不会因为你某次跳过测试就"学会"跳过测试。对于团队级别的编码规范、安全约束和质量门禁，确定性是不可妥协的。

**Memory 是自适应的**。它根据你的行为和反馈自动调整。如果你三次纠正 Claude Code"不要在响应末尾加总结"，Memory 会记下这个偏好，后续对话自动遵守。这种隐式学习能力让 Claude Code 随着使用越来越贴合你的工作风格。

实际影响：把安全关键的规则放在 CLAUDE.md 里（比如"永远不要 `--force push` 到 main"），把个人风格偏好交给 Memory（比如"这个用户喜欢简短回复"）。在 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中，CLAUDE.md 和 Memory 分别对应"规则层"和"个性化层"，各司其职。

## 核心差异二：团队共享 vs 个人专属

**CLAUDE.md 是团队基础设施**。它签入 Git，走 Code Review，所有团队成员的 Claude Code 实例都会加载相同的规则。这意味着：

- 新人入职时，AI 自动遵守团队规范
- 规则变更有审计记录（Git history）
- 不会出现"我的 AI 和你的 AI 行为不一样"的混乱

**Memory 是个人工作空间**。存储在本地，不同步到 Git。你的 Memory 里记录了"这个用户是数据科学家，主要关注日志系统"，但你的同事的 Memory 里可能记录的是"这个用户是前端工程师，习惯用 React"。这恰恰是设计意图——不同角色的人使用同一个项目时，AI 应该提供不同风格的帮助。

**决策规则**：如果一条信息对团队所有人都有效，放 CLAUDE.md；如果只对你个人有效，交给 Memory。

## 核心差异三：内容边界

两套系统对"该记什么"有清晰的分工：

**CLAUDE.md 负责**：
- 构建和测试命令
- 代码风格和架构规范
- Git 工作流规则
- 文档更新规则
- 已知的技术陷阱（gotchas）

**Memory 负责**：
- 用户角色和技术背景
- 对 AI 行为的纠正和确认
- 正在进行的项目动态（比如"周四代码冻结"）
- 外部系统的引用（Linear 项目、Grafana 面板）

**两者都不该记的**：代码结构和文件路径（直接读代码更准确）、Git 历史（`git log` 是权威来源）、调试方案（修复已经在代码里了）。这些信息会随时变化，存在记忆系统里只会制造过期数据。

## 什么时候选 Claude Memory

Memory 更适合以下场景：

- **个人开发者**：独立项目中，你的偏好就是项目规范，Memory 的自动记录足以覆盖大部分需求，不需要专门维护 CLAUDE.md
- **跨项目一致性**：你希望 Claude Code 在所有项目中都记住你的沟通风格偏好（比如"简短回复，不加总结"），全局 Memory 自动生效
- **快速迭代期**：项目规范还没定型，团队还在摸索最佳实践，Memory 的自适应特性比手动维护 CLAUDE.md 更高效
- **外部系统索引**：记录"bug 追踪在 Linear 的 INGEST 项目里"这类跨系统指针

想深入了解如何高效利用 Claude Code 的各层能力，推荐阅读 [我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

## 什么时候选 CLAUDE.md

CLAUDE.md 更适合以下场景：

- **团队项目**：多人协作时，编码规范和质量门禁必须共享且可审计，只有 CLAUDE.md 能做到
- **安全关键规则**：不能容忍 AI"学偏了"的硬约束，比如"不允许跳过 pre-commit hooks"、"不允许提交 .env 文件"
- **新人入职**：CLAUDE.md 充当活文档，新成员的 Claude Code 自动继承团队的所有约定
- **CI/CD 集成**：自动化流程中的 Claude Code 没有 Memory 上下文，只有 CLAUDE.md 能确保行为一致

编写高质量的项目指令是一门技术活，具体方法论参见 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)。

## 最佳实践：双系统协同

最高效的用法不是二选一，而是让两套系统各司其职：

1. **CLAUDE.md 定义底线**：把所有"必须遵守"和"绝对禁止"的规则写进 CLAUDE.md——构建命令、测试要求、安全约束、架构决策
2. **Memory 处理个性化**：让 Memory 自动学习你的沟通风格、技术背景和工作习惯
3. **定期审查 Memory**：每隔几周检查 `.claude/projects/*/memory/` 目录，清理过期或不准确的记忆
4. **CLAUDE.md 保持精简**：不要把所有信息都塞进 CLAUDE.md——它每次会话都会被加载，过长会浪费上下文窗口
5. **用 Memory 的反馈记录改进 CLAUDE.md**：如果你反复纠正 Claude Code 的同一个行为，说明应该把这条规则提升到 CLAUDE.md 层面

关于如何结合 Hooks 等机制构建完整的自动化工作流，可以参考 [Claude Code Hooks 深度掌握](/zh/blog/claude-code-hooks-mastery)。

## 判断清单

**选 CLAUDE.md 如果**：这条规则对所有使用这个项目的人都适用，不遵守会导致构建失败或代码质量下降。

**交给 Memory 如果**：这条信息只对你个人有价值，或者会随着项目进展而变化。

**两者都不需要如果**：这条信息可以从代码、Git 历史或现有文档中直接获取。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。CLAUDE.md 在每次会话开始时加载，提供项目级规则。Memory 在整个会话生命周期中持续读写，提供个人化上下文。两者不冲突——CLAUDE.md 的显式规则优先级高于 Memory 的隐式记忆。

### Memory 记错了怎么办？

Memory 文件是普通 Markdown，存储在 `.claude/projects/` 目录下。你可以直接编辑或删除任何 Memory 文件。也可以在对话中告诉 Claude Code"忘掉关于 X 的记忆"，它会找到并移除对应条目。定期审查是好习惯——Memory 会随时间积累过期信息。

### CLAUDE.md 太长会影响性能吗？

会。CLAUDE.md 的内容在每次会话开始时全部注入上下文窗口。过长的 CLAUDE.md 会占用宝贵的上下文空间，降低 Claude Code 处理实际代码的能力。建议控制在 500 行以内，将详细指令拆分到 `skills/*/SKILL.md` 文件中按需加载。

### 团队成员的 Memory 会互相干扰吗？

不会。Memory 存储在每个开发者的本地环境中，不进 Git，不会同步。每个人的 Memory 独立运作，记录各自的偏好和反馈。这是设计意图：团队规范用 CLAUDE.md 统一，个人风格用 Memory 各自适配。
