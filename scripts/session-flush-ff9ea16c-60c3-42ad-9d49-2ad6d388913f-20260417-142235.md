# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **CLAUDE.md** 是写给项目的规则手册——团队共享、Git 版本控制、每次会话自动加载。**Claude Memory** 是写给个人的自动笔记——跨会话持久化、按用户隔离、由 Claude 自主维护。两者不是竞争关系，而是互补：CLAUDE.md 管「项目该怎么做」，Memory 管「这个人习惯怎么做」。大多数认真使用 Claude Code 的开发者应该同时用好这两套系统。

## 概览：Claude Memory

Claude Memory 是 [Claude Code](/glossary/claude-code) 内置的自动记忆系统，文件存储在 `~/.claude/projects/<project>/memory/` 目录下。它的核心设计思路是：让 Claude 在多次对话中逐步积累对用户和项目的理解，而不需要用户每次重复说明。

Memory 分为四种类型：**user**（用户画像）、**feedback**（行为反馈）、**project**（项目动态）、**reference**（外部资源指针）。Claude 会在对话中自动判断哪些信息值得记住，写入独立的 Markdown 文件，并在 `MEMORY.md` 索引中维护指针。下次新会话启动时，这些记忆会被自动加载到上下文中。

关键特征：Memory 是**个人的、自动的、非版本控制的**。它不进 Git，不被队友看到，也不需要手动维护。

## 概览：CLAUDE.md

CLAUDE.md 是 Claude Code 的项目级配置文件，放在项目根目录，随代码一起提交到 Git。每次 Claude Code 启动会话时，会自动读取当前项目的 CLAUDE.md 和用户全局的 `~/.claude/CLAUDE.md`，将其中的指令注入到系统上下文中。

CLAUDE.md 本质上是一份写给 AI 的「项目操作手册」：构建命令、代码风格、架构约束、质量门禁、已知坑点——任何你希望 Claude 在每次交互中都遵守的规则。它的内容由人类编写和维护，团队所有成员共享同一份。

关键特征：CLAUDE.md 是**共享的、手动的、版本控制的**。它进 Git，所有人看到同一份规则，修改需要有意识地提交。更多架构细节可以参考 [Claude Code 七层架构解析](/zh/blog/claude-code-seven-programmable-layers)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **存储位置** | `~/.claude/projects/*/memory/` | 项目根目录 `CLAUDE.md` | — |
| **版本控制** | 不进 Git | 随项目提交到 Git | CLAUDE.md |
| **共享范围** | 仅当前用户 | 团队所有成员 | CLAUDE.md |
| **维护方式** | Claude 自动写入 | 人类手动编写 | Memory |
| **内容类型** | 用户偏好、反馈、项目动态 | 项目规则、构建命令、架构约束 | 各有侧重 |
| **加载时机** | 每次会话自动加载 | 每次会话自动加载 | 平手 |
| **可靠性** | 可能过时，需验证 | 人工维护，通常准确 | CLAUDE.md |
| **学习曲线** | 零配置，自动工作 | 需要主动编写和维护 | Memory |

## 核心差异：谁写、写什么、给谁看

这是理解两套系统最关键的一个维度。

**CLAUDE.md 是人写给 AI 看的。** 你告诉 Claude：「这个项目用 Vitest 跑测试」「commit 前必须 build 通过」「中文内容必须用 CJK 字数统计」。这些是确定性的、不会因人而异的项目规则。写一次，所有队友的 Claude Code 会话都会遵守。

**Memory 是 AI 写给自己看的。** Claude 在对话中发现你是高级工程师、你讨厌冗长的总结、你偏好一个大 PR 而不是拆成小的——它把这些观察记录下来，下次对话时调整自己的行为。这些信息高度个人化，同一个项目里不同开发者的 Memory 可以完全不同。

这意味着一个重要的实践原则：**不要把项目规则放到 Memory 里，也不要把个人偏好写进 CLAUDE.md。** 混淆两者会导致规则要么无法共享（本该是 CLAUDE.md 的内容放到了 Memory），要么强加于人（本该是个人偏好的内容放到了 CLAUDE.md）。

详细了解 Memory 的四种类型和写入机制，可以阅读 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 可靠性与时效性：两套系统的信任边界

CLAUDE.md 的内容由人类维护，通常可以直接信任。如果 CLAUDE.md 说「用 `npm run build` 构建」，那基本就是对的——除非有人忘了更新。

Memory 的可靠性则需要打折扣。Claude 自动记录的信息可能会过时：一个月前记住的「项目在用 React 17」，可能早已升级到 19。Memory 系统本身也意识到这一点，因此在使用记忆时会先验证——比如先检查文件是否还存在，再推荐一个记忆中的路径。

实际使用中的建议：

1. **CLAUDE.md 当作 source of truth**——项目的构建、测试、部署规则以它为准
2. **Memory 当作辅助上下文**——提供有用的背景信息，但不能替代实际检查
3. **定期清理过时 Memory**——如果发现 Claude 基于错误记忆给出建议，删除对应的记忆文件

## 实战场景：什么时候用哪个

### 场景一：新人加入团队

新同事 clone 项目，打开 Claude Code。CLAUDE.md 自动生效——构建命令、代码规范、已知坑点全部就位。不需要任何额外操作。

随着新同事和 Claude Code 交互，Memory 逐渐积累：Claude 了解到这位同事是前端专精但不熟悉后端、偏好详细的解释而不是简洁的指令。这些个人化调整不会影响其他人。

### 场景二：建立编码规范

团队决定所有 API 路由必须有错误处理中间件。这条规则应该写进 CLAUDE.md——它是项目级的、确定性的、所有人都必须遵守的。如果你把它放到 Memory 里，只有你的 Claude Code 会遵守，其他人的不会。

### 场景三：个人工作习惯

你喜欢 Claude 在完成任务后不要长篇总结，直接给 diff。这是个人偏好，应该由 Memory 系统记录。写进 CLAUDE.md 会强制所有队友都接受这种风格——他们可能并不喜欢。

## 协同使用：推荐的配置策略

最有效的使用方式是让两套系统各司其职，参考 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 了解它们在整体架构中的位置：

**写进 CLAUDE.md 的内容：**
- 构建、测试、部署命令
- 代码风格和架构约束
- 质量门禁（commit 前必须通过的检查）
- 已知坑点和技术债务说明
- 环境变量和配置说明

**交给 Memory 自动管理的内容：**
- 用户角色和技术背景
- 沟通风格偏好（简洁 vs 详细）
- 已验证有效的工作方式
- 外部资源指针（Linear 项目、Grafana 面板等）
- 近期项目动态和优先级

**两者都不应该存的内容：**
- 代码结构和文件路径（直接读代码更准确）
- Git 历史和变更记录（用 `git log` 查）
- 一次性调试信息（修完就没用了）

## 判断标准

**选 CLAUDE.md** 如果这条信息满足以下任意一条：所有人都需要遵守、会影响代码质量、需要版本控制、是项目级规则而非个人偏好。

**让 Memory 处理** 如果这条信息满足以下任意一条：因人而异、会自然过时、是观察而非规则、不适合强加给团队。

**核心原则是：CLAUDE.md 管规则，Memory 管上下文。** 规则需要明确、共享、可审计；上下文需要灵活、个人、可演化。把两者混在一起用，就像把公司制度写在个人笔记本上，或者把个人日记贴在公司公告栏上——技术上可行，但注定会出问题。

## 常见问题

### Claude Memory 会自动同步到其他设备吗？

不会。Memory 文件存储在本地 `~/.claude/` 目录下，不通过云端同步。如果你在多台机器上使用 Claude Code，每台机器会独立积累自己的 Memory。如果需要迁移，可以手动复制 `memory/` 目录。

### CLAUDE.md 支持多层级配置吗？

支持。Claude Code 会同时读取三个层级的配置：全局 `~/.claude/CLAUDE.md`、项目根目录 `CLAUDE.md`、以及项目中的 `skills/*/SKILL.md` 文件。全局配置适合放个人通用规则（如 Git 工作流偏好），项目配置放项目特定规则，Skill 文件放任务特定指令。

### 可以手动编辑 Memory 文件吗？

可以。Memory 文件就是普通的 Markdown 文件，存储在 `~/.claude/projects/<project>/memory/` 目录下。你可以直接编辑、删除或新增文件。`MEMORY.md` 是索引文件，修改后 Claude 下次会话会读取更新后的内容。

### 如果 CLAUDE.md 和 Memory 冲突了怎么办？

CLAUDE.md 的指令标注为「OVERRIDE any default behavior」，优先级高于 Memory。如果 CLAUDE.md 规定「commit message 用英文」，而 Memory 记录了「用户偏好中文 commit message」，Claude 会遵守 CLAUDE.md 的规则。这是设计如此——项目规则优先于个人偏好。
