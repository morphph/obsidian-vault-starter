**Claude Code's parent-child model** means a single session acts as the orchestrator. When you ask Claude Code to tackle a complex task, the parent agent can spawn multiple sub-agents — each with a specific role — and synthesize their outputs before proceeding. For example, a parent agent might launch an Explore agent to find all files referencing a deprecated API, a Plan agent to design the migration strategy, and then a general-purpose agent to implement the changes. The parent sees all results and makes decisions about what to do next.

This creates a tight feedback loop. If the Explore agent discovers unexpected complexity — say, the deprecated API is used in 47 files instead of the expected 5 — the parent can adjust the plan immediately. The agents share conversation context with the parent, so the orchestration is coherent. You can run independent sub-agents in parallel by making multiple agent calls in a single message, or run them sequentially when outputs depend on each other.

The typed agent system provides guardrails. An Explore agent has read-only access — it can search and read but cannot edit files. A Plan agent designs implementation strategies without executing them. This separation of concerns prevents accidental side effects during the research phase. As documented in our [Claude Code subagents examples](/blog/claude-code-subagents-examples), this pattern is particularly effective for large-scale refactoring where you want to understand the blast radius before making changes.

**Codex's task-dispatch model** takes a different approach. Each custom agent runs as an independent unit of work in its own cloud container. There is no parent agent coordinating between them in real time. You define what each agent does through its configuration — system prompt, environment setup, tool access — and then assign it a task. The agent works asynchronously and returns results when complete.

This design excels at parallelism without coordination overhead. You can dispatch ten tasks to ten different custom agents simultaneously, and they all run in isolated environments without interfering with each other. The trade-off is that these agents cannot dynamically adjust based on what other agents discover. If Agent A finds that a migration is more complex than expected, Agent B (working on a related task) will not know about it until both complete and you review the results.

Codex compensates for this with its configuration flexibility. You can create highly specialized custom agents — a "Python test writer" that knows your pytest conventions, a "Go linter" that enforces your style guide, a "security auditor" that checks for OWASP vulnerabilities — and reuse them across tasks. This reusability is a significant productivity multiplier for teams with established workflows.

**Verdict for orchestration:** Claude Code subagents win for tasks requiring real-time coordination and adaptive planning. Codex custom agents win for batch processing and well-scoped independent tasks.

## Customization and Configuration: Detailed Analysis

How you configure agent behavior differs substantially between the two platforms, and this affects how teams standardize their AI-assisted workflows.

**Claude Code's customization** operates through a layered file system. Project-level instructions live in `CLAUDE.md` — coding standards, architecture decisions, constraints that apply to every agent in the session. Task-specific instructions go into `SKILL.md` files under the `skills/` directory. When a sub-agent is spawned, it inherits the project context but receives a task-specific prompt from the parent. You can also specify the sub-agent type (Explore, Plan, general-purpose, code-reviewer) and set parameters like isolation mode (worktree for git-isolated changes) and permission mode.

This layered approach means your customization travels with the repository. Any developer who clones the repo and runs Claude Code gets the same agent behavior — the [CLAUDE.md and SKILL.md files](/blog/claude-code-complete-guide) are checked into version control. For teams, this is a significant advantage: AI behavior is code-reviewed just like any other configuration. Our guide on [writing effective skills](/blog/9-principles-writing-claude-code-skills-i-use-every-single-day) covers the principles for making these files work reliably.

However, Claude Code's sub-agent types are predefined. You get Explore, Plan, general-purpose, and code-reviewer out of the box. You cannot create entirely new agent types with custom tool access — you work within the framework Anthropic provides. The parent agent's prompt controls what each sub-agent does, but the sub-agent's capabilities are bounded by its type.

**Codex's customization** is more freeform. Each custom agent is defined by its configuration: a system prompt that sets behavior, environment setup commands that install dependencies and configure the workspace, and a list of approved tools. You can create arbitrary agent roles — "database migration specialist," "API documentation writer," "performance benchmarker" — each with its own tailored instructions and environment.

The environment setup is particularly powerful. Because each Codex agent runs in its own container, you can install specific tool versions, configure databases, set up test fixtures, and create custom workflows that would be impractical in a shared local environment. This containerized approach also means agents cannot accidentally break each other's environments.

The trade-off is discoverability and standardization. Codex custom agent configurations live in the platform, not in the repository. Team members need to know which custom agents exist and when to use them. There is no built-in mechanism (at the time of writing) for version-controlling custom agent configurations alongside your code, though teams can manage this through external tooling.

**Verdict for customization:** Codex custom agents offer more flexibility in defining agent roles and environments. Claude Code's file-based system is more portable and version-controllable. Teams that prioritize reproducibility across developers will prefer Claude Code's approach. Teams that need specialized environments per agent will prefer Codex.

## Practical Workflow: Multi-Agent Refactoring Example

To make the differences concrete, here is how each system handles the same task: refactoring a legacy authentication module across a monorepo.

### Claude Code Subagent Workflow

1. You describe the task to Claude Code in your terminal
2. The parent agent spawns an **Explore agent** to search for all auth-related files, identify dependencies, and map the call graph
3. The Explore agent reports back: 23 files reference the old auth module, 4 services depend on it directly
4. The parent spawns a **Plan agent** to design the migration strategy — which files to change first, what the new API surface looks like, how to maintain backward compatibility during migration
5. You review and approve the plan
6. The parent spawns **multiple general-purpose agents in parallel** (using worktree isolation) to implement changes in different services simultaneously
7. Each agent reports back; the parent synthesizes results, checks for conflicts, and runs tests
8. You review the final diff and approve the commit

The entire workflow happens in one interactive session. You can intervene at any step, adjust the plan, or redirect agents. For a deeper walkthrough of this pattern, see our [Claude Code subagents examples](/blog/claude-code-subagents-examples).

### Codex Custom Agent Workflow

1. You create (or reuse) custom agents: "auth-refactorer" with system instructions about your auth patterns, "test-updater" for fixing test suites, "doc-updater" for documentation
2. You submit a task to the "auth-refactorer" agent: "Refactor the legacy auth module to use the new token-based API"
3. The agent works asynchronously in its cloud container, producing a set of changes
4. You review the output, then submit follow-up tasks to "test-updater" and "doc-updater" to handle the downstream effects
5. Each agent works independently and returns results
6. You review and merge

The workflow is more sequential because agents cannot coordinate in real time. You trade interactivity for isolation and reusability — the same custom agents can be dispatched against the same task pattern across multiple repositories.

## When to Choose Claude Code Subagents

Choose Claude Code subagents when your workflow requires **adaptive, coordinated multi-agent execution**:

- **Complex refactoring** where the scope is not fully known upfront — Explore agents discover the blast radius, Plan agents adapt the strategy, and implementation agents execute with full context of what was discovered
- **Interactive development sessions** where you want to stay in the loop and redirect agents based on intermediate results
- **Cross-file tasks with dependencies** where the output of one agent directly informs what another agent should do
- **Teams that version-control AI configuration** — CLAUDE.md and SKILL.md files live in the repo and are reviewed like code

Claude Code subagents are ideal for senior engineers who think in terms of orchestration and want fine-grained control over how AI agents collaborate. The [agent teams architecture](/blog/claude-code-agent-teams) is designed for workflows where the whole is greater than the sum of the parts. If you are already using Claude Code's [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) — skills, hooks, MCP servers — subagents are a natural extension of that system.

## When to Choose Codex Custom Agents

Choose Codex custom agents when your workflow benefits from **specialized, independent, fire-and-forget execution**:

- **Batch processing** across multiple repositories or modules where tasks are well-scoped and independent
- **Standardized operations** like test generation, documentation updates, or linting fixes where you have established patterns that can be encoded in agent configurations
- **Environment-sensitive tasks** that need specific tooling, database access, or dependency versions — each agent gets its own isolated container
- **Teams already invested in the OpenAI ecosystem** using GPT models, the [Codex VS Code extension](/blog/codex-vscode), and OpenAI's platform tooling

Codex custom agents shine when you can clearly define the task boundary upfront. If you know exactly what needs to happen and do not need real-time coordination between agents, the async dispatch model is cleaner and scales better horizontally. The [complete Codex guide](/blog/codex-complete-guide) covers how to set up effective custom agent configurations.

## Verdict

The choice between Claude Code subagents and Codex custom agents comes down to your workflow model: **interactive orchestration vs. async dispatch**.

**Choose Claude Code subagents** if you need agents that coordinate in real time, adapt to discoveries during execution, and integrate into a version-controlled configuration system. This is the better choice for complex, exploratory engineering tasks where the scope evolves as you work.

**Choose Codex custom agents** if you need specialized, reusable agent profiles that run independently in isolated environments. This is the better choice for well-defined, repeatable tasks that benefit from containerized execution and batch parallelism.

For teams that can invest in both ecosystems, the two approaches are complementary rather than competing. Use Claude Code subagents for your daily interactive development — debugging, refactoring, feature implementation — and Codex custom agents for standardized batch operations like cross-repo test generation or documentation sweeps.

## Frequently Asked Questions

### Can I use Claude Code subagents and Codex custom agents together?

Yes. They run on different platforms (Anthropic and OpenAI respectively) and solve different parts of the multi-agent problem. Use Claude Code subagents for interactive, coordinated tasks in your terminal and Codex custom agents for async batch work dispatched through the web UI or VS Code extension. There is no technical conflict between using both.

### How many subagents can Claude Code run in parallel?

Claude Code can spawn multiple sub-agents in a single message, and they execute concurrently. The practical limit depends on your API usage tier and the complexity of each sub-agent's task. For most workflows, 2-5 parallel sub-agents is the effective sweet spot — enough to parallelize independent work without overwhelming context management.

### Do Codex custom agents share state between tasks?

No. Each Codex custom agent task runs in an isolated container. The agent starts fresh with the codebase and its configured environment for every task. This isolation prevents side effects but means you cannot build incremental context across tasks within a single agent — each task is self-contained.

### Which approach is better for code review?

Claude Code has a built-in code-reviewer sub-agent type designed specifically for this purpose, and it can coordinate with other agents (e.g., running tests after identifying issues). Codex custom agents can be configured for code review through their system prompts, but the review runs independently without the ability to trigger follow-up actions automatically. For integrated review workflows, Claude Code's approach is more cohesive.

### What are the cost differences?

Claude Code uses Anthropic's usage-based API billing — you pay per token across all agents in the session. Codex pricing depends on your OpenAI subscription tier and per-task compute usage. Direct cost comparison is difficult because the billing models differ, but for high-volume batch tasks, Codex's subscription model may offer more predictable costs. For interactive sessions with moderate agent usage, Claude Code's per-token billing avoids paying for idle capacity.
