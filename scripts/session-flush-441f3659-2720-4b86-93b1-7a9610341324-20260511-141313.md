**CLAUDE.md loads eagerly.** Every conversation begins by injecting the full contents of all applicable CLAUDE.md files — global, project-level, and any directory-scoped variants — directly into Claude's system context. This means CLAUDE.md instructions are always present, always authoritative, and never forgotten mid-session. The tradeoff: longer CLAUDE.md files consume more of the context window on every single turn, whether the instructions are relevant to the current task or not.

**Memory loads selectively.** The `MEMORY.md` index file loads into every conversation, but individual memory files are only accessed when Claude determines they're relevant. This means memory scales better — you can accumulate dozens of memories without bloating every conversation's context. But it also means a memory might not surface when you expect it to, especially if the index description doesn't clearly signal relevance.

This architectural difference drives the core tradeoff: **CLAUDE.md guarantees visibility at the cost of context space; memory optimizes context space at the cost of guaranteed visibility.** For rules that must never be missed (quality gates, forbidden patterns, deployment procedures), CLAUDE.md is the only safe choice. For context that's useful but not critical on every turn (your preferred commit message style, a reference to the team's Slack channel), memory is more efficient.

The [Claude Code extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) positions CLAUDE.md as the base layer and memory as an overlay — a design that mirrors how human teams work. The employee handbook (CLAUDE.md) covers universal rules; tribal knowledge (memory) fills in the gaps.

## Collaboration and Team Dynamics: Shared vs Personal Context

The sharpest divide between these two systems is how they handle teams. This distinction alone determines which system to use for most decisions.

**CLAUDE.md is a team artifact.** Because it lives in the repository, every developer on the team gets the same instructions. When a senior engineer adds "always run `npm test` before committing" to CLAUDE.md, every team member's Claude Code session enforces that rule immediately after pulling the change. Changes go through pull requests, which means the team can review, discuss, and approve modifications to AI behavior the same way they review code changes. This is critical for engineering organizations where consistency matters — you don't want one developer's Claude writing Jest tests while another's writes Vitest tests.

**Memory is strictly personal.** Your memories live in your local `.claude/` directory, tied to your machine and your user profile. When you tell Claude "I'm a backend engineer, explain frontend concepts in terms of API design," that context improves *your* experience but doesn't transfer to teammates. This is by design — personal preferences shouldn't leak into shared configuration. But it means teams can't use memory to share project context like "the auth rewrite is driven by compliance requirements, not tech debt."

The practical recommendation: **encode anything the team needs to agree on in CLAUDE.md, and let memory handle everything personal.** If you find yourself writing the same correction to Claude across multiple conversations ("no, we use pnpm here, not npm"), that's a signal to move it from memory into CLAUDE.md where the whole team benefits. Our [complete guide to Claude Code](/blog/claude-code-complete-guide) covers how to structure CLAUDE.md files for team use.

## Maintenance and Staleness: The Long-Term Cost of Each System

Both systems require maintenance, but the failure modes differ — and understanding these failure modes helps you decide what goes where.

**CLAUDE.md rots silently.** Because it's manually maintained, CLAUDE.md instructions can fall out of sync with the actual codebase. A build command that worked six months ago might reference a deprecated script. A style rule might reference a linter configuration that's been replaced. The file itself never warns you — Claude will follow outdated instructions faithfully, producing errors or suboptimal output. The fix is discipline: treat CLAUDE.md as living documentation that gets updated alongside the code it describes.

**Memory drifts noisily.** Memory entries can become stale when project state changes — a deadline passes, a decision gets reversed, a team member switches roles. Claude is designed to verify memories against current state before acting on them, but this verification isn't perfect. The advantage: when memory causes a wrong action, the error is usually visible in the conversation, and you can correct it immediately. Claude will then update or remove the stale memory.

The maintenance burden breaks down roughly like this: CLAUDE.md requires **periodic review** (monthly or per-milestone is reasonable for most teams), while memory requires **in-conversation correction** (just tell Claude when something is wrong). For teams practicing [agentic coding](/glossary/agentic-coding) at scale, the CLAUDE.md review cadence should match your sprint or release cycle.

A practical pattern that works well: keep CLAUDE.md lean and focused on things that change slowly (build commands, quality gates, architecture principles), and let memory handle things that change often (current sprint goals, temporary workarounds, in-progress decisions). This minimizes maintenance for both systems.

## Scope and Content: What Belongs Where

Not everything fits neatly into one system. Here's a decision framework based on the content type.

### Put in CLAUDE.md

- **Build and test commands**: `npm run build`, `npm test`, quality gates
- **Coding standards**: naming conventions, file structure rules, import ordering
- **Architecture decisions**: "we use SQLite, not Postgres" or "no direct Next.js imports in pipeline scripts"
- **Forbidden patterns**: "never skip failing tests" or "don't use `any` type"
- **Deployment procedures**: Vercel configuration, environment variable requirements
- **Documentation rules**: what to update when code changes

### Put in Memory

- **Your role and expertise**: "I'm a data scientist" or "I'm new to React"
- **Behavioral feedback**: "don't summarize at the end of responses" or "always explain your reasoning"
- **Project state**: "merge freeze starts May 15 for mobile release"
- **External references**: "bugs tracked in Linear project INGEST" or "oncall dashboard at grafana.internal/d/api-latency"
- **Relationship context**: "the auth rewrite is driven by legal compliance, not tech debt"

### Could go in either (use judgment)

- **Workflow preferences**: "commit after every change" could be CLAUDE.md (team standard) or memory (personal habit). If the team agrees, put it in CLAUDE.md. If it's just you, let memory handle it.
- **Tool-specific context**: "use pnpm not npm" is CLAUDE.md if it's a project requirement, memory if it's a personal preference.

Teams using [Claude Code skills](/blog/5-claude-code-skills-i-use-every-single-day) add a third option: skill files for task-specific instructions that are too detailed for CLAUDE.md but too universal for personal memory.

## Interaction Between the Two Systems

CLAUDE.md and memory don't operate in isolation — they form a layered context system where CLAUDE.md instructions take precedence over memory when they conflict. Understanding this hierarchy prevents confusion.

**CLAUDE.md always wins.** If CLAUDE.md says "use Vitest for all tests" and a memory entry says "the user prefers Jest," Claude follows CLAUDE.md. This is by design — team-wide explicit instructions override personal learned context. The only exception: if the user explicitly tells Claude to ignore CLAUDE.md instructions in a specific conversation, which overrides both.

**Memory fills gaps CLAUDE.md doesn't cover.** A typical CLAUDE.md file runs 50-200 lines covering essential rules. Memory fills in everything else: your communication preferences, project history, recurring patterns Claude has learned from your corrections. Together, they give Claude a comprehensive picture — the explicit rulebook plus the tribal knowledge.

**Memory can signal CLAUDE.md improvements.** If Claude keeps storing the same feedback memory across multiple users ("user corrected: we use snake_case for database columns"), that's a sign the convention should be promoted to CLAUDE.md. Treat recurring memory patterns as CLAUDE.md candidates — if everyone needs to know it, write it down once in the shared file.

## When to Choose Claude Memory

Choose memory as your primary persistence mechanism when:

- **You work solo** and don't need to share AI configuration with a team. Memory captures your preferences naturally without requiring you to write and maintain a configuration file.
- **Your context changes frequently.** Sprint goals, project priorities, and team dynamics shift week to week. Memory adapts through conversation; CLAUDE.md would require constant manual edits.
- **You want zero-maintenance persistence.** If you correct Claude once ("don't mock the database in integration tests"), memory retains that correction for future sessions without any action on your part.
- **You're onboarding to a new codebase.** While ramping up, your questions and corrections teach Claude about your knowledge gaps and learning path. Memory tracks this naturally; putting "I don't understand the auth module yet" in CLAUDE.md would be odd.

Memory's strength is its low friction — it learns by watching, not by being configured. For solo developers or small teams where formal process is overkill, memory alone can handle most persistence needs.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence mechanism when:

- **You work on a team** and need everyone's Claude sessions to follow the same rules. CLAUDE.md is the only mechanism that travels with the repo and goes through code review.
- **You have non-negotiable quality gates.** Build must pass, tests must pass, linting must pass — these rules can't depend on Claude having "learned" them from a past conversation. CLAUDE.md enforces them deterministically.
- **You need auditability.** CLAUDE.md changes show up in git history. You can trace when a rule was added, who added it, and why. Memory has no audit trail.
- **You're setting up a new project.** Before any conversations happen, CLAUDE.md establishes the baseline. Memory has nothing to learn from until you start working. A well-written CLAUDE.md makes the first conversation productive immediately.
- **Reliability is non-negotiable.** For anything where "Claude forgot" would cause real damage (deploying to the wrong environment, skipping security checks), CLAUDE.md's eager loading guarantees the instruction is always present.

For engineering teams adopting Claude Code, CLAUDE.md should be the first file you create — before writing any code with Claude. See [what makes Claude Code special](/blog/whats-so-special-about-the-claude-code) for how CLAUDE.md fits into the broader workflow.

## Verdict

**Use both.** CLAUDE.md and Claude Memory aren't competing features — they're complementary layers designed for different types of context. **CLAUDE.md is your project's constitution**: deterministic, version-controlled, shared, and authoritative. **Memory is your personal relationship with Claude**: adaptive, learned, private, and low-maintenance.

The practical rule: **if the whole team needs to know it, put it in CLAUDE.md. If only Claude needs to remember it about you, let memory handle it.** Start with a lean CLAUDE.md covering build commands, quality gates, and coding standards. Let memory accumulate naturally from your conversations. Promote recurring memory patterns to CLAUDE.md when they prove universal. ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions always take precedence over memory when they conflict. Claude treats CLAUDE.md as the authoritative source — it loads first and applies unconditionally. Memory fills in context that CLAUDE.md doesn't cover but cannot contradict explicit CLAUDE.md rules.

### Does Claude Memory sync across machines?

No. Memory files are stored locally in your `.claude/` directory and are tied to your machine and project path. If you switch computers, your memories don't follow. CLAUDE.md, by contrast, syncs automatically through git. For critical personal preferences, some developers maintain a global `~/.claude/CLAUDE.md` and manually sync it across machines.

### How many CLAUDE.md files can a project have?

There's no hard limit. You can have a global `~/.claude/CLAUDE.md`, a project root `CLAUDE.md`, and directory-scoped CLAUDE.md files in subdirectories for path-specific rules. All applicable files load into every conversation. Keep them concise — every line consumes context window space on every turn.

### Should I check memory files into version control?

No. The `.claude/` directory is gitignored by default, and memory files are designed to be personal and machine-specific. Checking them in would leak individual preferences into the shared project configuration. If a memory entry is valuable enough to share, promote it to CLAUDE.md instead.

### How do I know if something is in memory vs CLAUDE.md?

Ask Claude directly: "What do you know about X?" or "Where did you learn that rule?" Claude can distinguish between instructions from CLAUDE.md (loaded at session start) and memories (loaded from `.claude/projects/*/memory/`). If Claude cites a memory that should be a team-wide rule, move it to CLAUDE.md.
