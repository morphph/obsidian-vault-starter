**Claude Code's explicit subagent model** works differently. The [Agent tool](/blog/claude-code-agent-teams) spawns typed subagents that run within the same session context. The orchestrating agent can:

- Launch multiple subagents simultaneously for independent work
- Wait for a subagent's result before spawning the next one
- Pass specific context and constraints to each subagent
- Route subagent output into subsequent decisions

This enables patterns like: spawn an Explore agent to find all database queries, spawn a Plan agent to design a migration strategy based on those findings, then spawn a general-purpose agent to execute the migration — all within a single interactive session.

The tradeoff is isolation. Claude Code subagents share your local environment. A subagent that runs a destructive command affects your actual filesystem. The `worktree` isolation option mitigates this by giving subagents their own git worktree, but it's opt-in rather than default.

## Custom Agent Configuration: Setup and Flexibility

Both platforms let you customize agent behavior, but the mechanisms and depth of customization differ significantly.

### Codex Custom Agents

Configuring custom agents in Codex revolves around three layers:

1. **Task-level instructions**: When submitting a task, you provide natural language instructions that shape how Codex approaches the work. These can specify coding standards, testing requirements, or architectural constraints.

2. **Repository-level configuration**: A setup script or configuration file in your repo can define environment setup, dependency installation, and project-specific context that every Codex task in that repo inherits.

3. **System-level prompts**: For teams using the API, system prompts define persistent agent behavior across all tasks — things like "always write tests," "follow our style guide," or "never modify files in /config without approval."

This layered approach is straightforward but relatively coarse-grained. You're configuring a single agent's behavior, not defining specialized agent types. Every Codex task is fundamentally the same kind of agent with different instructions.

The [Codex for open source](/blog/codex-for-open-source) program demonstrates this in practice — maintainers configure Codex with project-specific conventions, and contributors can submit Codex-assisted PRs that follow those conventions automatically.

### Claude Code Custom Agents

Claude Code's customization runs deeper because the subagent system is explicitly typed. Configuration happens at multiple levels:

1. **Built-in agent types**: Claude Code ships with specialized agents — Explore (fast read-only codebase search), Plan (architecture and implementation planning), code-reviewer, and general-purpose. Each type has different tool access and behavioral defaults.

2. **SKILL.md files**: Reusable instruction files that define how Claude Code approaches specific tasks. A `skills/testing/SKILL.md` might encode your team's testing philosophy, preferred assertion libraries, and coverage requirements. When invoked, the skill's instructions shape the agent's behavior for that specific workflow. Our guide on [writing effective skills](/blog/9-principles-writing-claude-code-skills-that-actually-work) covers the patterns that produce the best results.

3. **Agent spawn configuration**: When the orchestrating agent creates a subagent, it specifies the type, a detailed prompt, optional model override, permission mode, and isolation settings. This per-spawn configuration means different subagents in the same session can have completely different capabilities and constraints.

4. **CLAUDE.md project files**: Project-level and user-level instruction files that all agents inherit, providing global context about coding standards, architecture, and workflows.

This system enables genuinely specialized agents. A security-review agent that only has read access and runs against a checklist of OWASP patterns is a fundamentally different tool than a general-purpose agent with full shell access and write permissions. In Codex, both would be the same agent type with different text instructions.

## Parallel Execution and Task Orchestration

Parallel work is where the architectural differences create the starkest practical impact.

### Codex Parallel Workflows

Codex's parallelism is task-level. You submit five tasks, and five containers spin up independently. This scales cleanly — there's no coordination overhead, no shared state to manage, and each task gets its own isolated environment.

The limitation is coordination. If you're refactoring a large codebase and Task 3 depends on the interface changes from Task 1, you need to serialize those tasks or manage the dependency externally. Codex won't automatically detect that Task 3's imports broke because Task 1 renamed a module.

For independent work — fixing five unrelated bugs, writing tests for five separate modules, or reviewing five PRs — Codex's task-level parallelism is excellent. Submit them all and check back when they're done.

### Claude Code Parallel Workflows

Claude Code's parallelism is subagent-level within a session. The orchestrating agent can spawn multiple subagents simultaneously using independent tool calls, and they execute concurrently. The key advantage: the orchestrator sees all results and can make informed decisions about next steps.

A practical pattern from the [Claude Code agent teams feature](/blog/claude-code-agent-teams): spawn three Explore agents to search for all uses of a deprecated API across different parts of the codebase. When all three return, the orchestrator has a complete picture and can spawn a single agent to execute the migration with full context.

This intra-session coordination is something Codex can't replicate without external tooling. But it comes with a constraint — everything runs on your local machine. Spawning ten subagents means ten concurrent processes on your hardware, not ten cloud containers with dedicated resources.

## Isolation and Security

For teams evaluating where to run autonomous coding agents, isolation models matter.

**Codex provides strong isolation by default.** Every task runs in a sandboxed container with no access to your local machine, production systems, or other running tasks. The container has network access limited to specific endpoints. This makes Codex appealing for organizations with strict security requirements — the blast radius of a rogue task is contained to a disposable container.

**Claude Code runs locally with full shell access.** A subagent can execute any command your user account can execute. The permission system provides guardrails — users approve tool calls that modify files or run commands — but the underlying access is broad. The `worktree` isolation mode for subagents creates a separate git worktree, providing file-level isolation without the overhead of a container, but it doesn't restrict shell access.

For [security-conscious workflows](/blog/claude-code-security-vulnerability-scanning), Codex's container model is inherently safer. For workflows that need to interact with your actual development environment — running your test suite, accessing local services, using local credentials — Claude Code's local execution is necessary.

## IDE and Workflow Integration

How each tool fits into your existing development workflow determines day-to-day usability.

**Codex integrates through GitHub and VS Code.** The [VS Code extension](/blog/codex-vscode) lets you submit tasks from your editor, and completed work appears as PRs or branches in your GitHub repository. For teams already centered on GitHub-based workflows, this is natural. The async model means you can submit tasks during a meeting and review results afterward.

**Claude Code integrates through your terminal and supports IDE extensions.** It reads your local filesystem directly, runs your actual build tools, and commits to your local git repo. The [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) — Skills, Hooks, Agents, and MCP — turns it into a programmable platform. Hooks execute shell commands in response to agent events, MCP servers connect external tools, and Skills encode reusable workflows.

For teams building custom automation around their coding agent, Claude Code's programmable extension model offers more surface area. For teams that want a coding agent that fits into GitHub-centric workflows without custom infrastructure, Codex's PR-based output model is simpler.

## When to Choose Codex

Choose Codex for subagents and custom agents when your workflow matches these patterns:

- **Async batch processing**: You have a backlog of independent tasks — bug fixes, test additions, documentation updates — and want to submit them all and review results later. Codex's container-per-task model handles this cleanly without any orchestration overhead.

- **Strong isolation requirements**: Your organization requires that AI agents cannot access local machines, production credentials, or internal networks. Codex's sandboxed containers provide this by default.

- **GitHub-native teams**: Your entire workflow lives in GitHub — issues, PRs, code review, CI/CD. Codex outputs PRs that slot directly into this flow. No terminal sessions to manage, no local environment to configure.

- **Student and open-source use cases**: The [Codex for students](/blog/codex-for-students) program and [open source program](/blog/codex-for-open-source) provide free credits, making it accessible for learning and contributing to open source projects where local tooling setup might be a barrier.

- **Simple customization needs**: If custom agent behavior means "follow these coding conventions and always write tests," Codex's instruction-based configuration is sufficient. You don't need typed agent hierarchies for straightforward task execution.

## When to Choose Claude Code

Choose Claude Code for subagents and custom agents when your workflow requires:

- **Interactive multi-agent orchestration**: You need subagents that communicate within a session — one agent's findings inform another agent's actions. Claude Code's typed subagent system with real-time result passing enables workflows Codex can't replicate without external orchestration.

- **Deep codebase understanding**: The Explore agent type is purpose-built for fast read-only codebase search. When your task requires understanding code relationships before making changes, Claude Code's multi-agent approach — explore first, plan second, execute third — produces better results than submitting a task and hoping the agent finds what it needs.

- **Custom agent specialization**: You want fundamentally different agent types — a read-only security reviewer, a test-writing specialist with specific framework knowledge, an architecture planner that proposes but never edits. Claude Code's typed agent system with per-agent tool access and permissions makes this possible. See practical [subagent implementation patterns](/blog/claude-code-subagents-examples) for working examples.

- **Local environment integration**: Your workflow requires running actual build tools, local test suites, local databases, or services that can't be replicated in a cloud container. Claude Code's local execution means subagents interact with your real development environment.

- **Programmable automation**: You're building custom workflows around your coding agent using hooks, MCP servers, and skills. Claude Code's [seven programmable layers](/blog/claude-code-seven-programmable-layers) provide extension points Codex doesn't expose.

## Verdict

The choice between Codex and Claude Code for subagents and custom agents comes down to **async independence vs interactive orchestration**.

**Choose Codex** if you want to fire off independent coding tasks and review results later. Its container isolation, GitHub integration, and async execution model make it the better choice for teams processing batches of independent work without complex inter-task dependencies.

**Choose Claude Code** if you need subagents that collaborate within a session — exploring, planning, and executing as a coordinated team. Its typed agent system, SKILL.md customization, and real-time orchestration enable multi-agent workflows that Codex's task-per-container model cannot replicate without significant external tooling.

For many teams, the answer is both. Use Codex for your overnight task queue — the five bug fixes, the test coverage gaps, the documentation updates. Use Claude Code for the complex refactoring session where an Explore agent maps the dependency graph, a Plan agent designs the migration, and a general-purpose agent executes it step by step. The tools complement each other because they solve different shapes of the same problem.

## Frequently Asked Questions

### How do subagents work in Codex?

Codex treats each submitted task as an independent agent instance running in a sandboxed cloud container. There is no explicit subagent spawning within a Codex task — parallelism comes from submitting multiple tasks simultaneously, each executing in its own isolated environment against your repository.

### Can you create custom agents in Codex?

Yes. Custom agent behavior in Codex is configured through task-level instructions, repository-level setup files, and API-level system prompts. These control coding conventions, testing requirements, and workflow constraints. The customization is instruction-based rather than type-based — all Codex agents are the same fundamental type with different directives.

### How do Claude Code subagents differ from Codex tasks?

Claude Code subagents are explicitly typed (Explore, Plan, general-purpose, etc.) and run within a single interactive session with real-time result passing. Codex tasks are independent, async, and isolated in separate containers. Claude Code enables intra-session agent coordination; Codex requires external orchestration for dependent tasks.

### Which is better for team workflows?

Codex fits teams that want a hands-off async model integrated with GitHub. Claude Code fits teams that want programmable, interactive multi-agent workflows with deep customization through Skills and Hooks. Teams with both needs often use both tools for different workflow shapes.

### Is Codex or Claude Code more secure for running subagents?

Codex provides stronger default isolation — every task runs in a sandboxed container with limited network access. Claude Code subagents run locally with your user permissions, offering optional worktree isolation but no container-level sandboxing. For security-sensitive environments, Codex's isolation model requires less trust in the agent's behavior.
