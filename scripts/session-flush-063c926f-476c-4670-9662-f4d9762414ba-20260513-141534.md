# Claude Memory vs CLAUDE.md：哪个才是你需要的上下文管理方案？

**TL;DR：** **Claude Memory** 是跨会话的自动记忆系统，记住你的偏好和背景信息；**CLAUDE.md** 是项目级的手动规则文件，定义代码规范和工作流约束。两者解决不同问题——Memory 管「你是谁」，CLAUDE.md 管「这个项目怎么做」。大多数认真使用 Claude Code 的开发者两个都需要。

## Overview：Claude Memory

**Claude Memory** 是 Anthropic 为 Claude 设计的持久化记忆系统，解决的核心问题是：每次新对话都要从零开始交代背景。Memory 会自动从对话中提取关键信息——你的角色、技术栈偏好、常用工具链、工作习惯——并存储下来供后续会话使用。

在 Claude Code 中，Memory 以文件形式存储在 `~/.claude/projects/` 目录下，按项目隔离。系统会自动将记忆内容注入会话上下文，让 Claude 在下一次对话中「记住」你之前说过什么。Anthropic 在 2026 年初对 Memory 做了[重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，包括免费用户开放和支持从 ChatGPT 导入历史数据。

Memory 的关键特征是**自动化**：你不需要手动维护，Claude 自己判断什么值得记住。这降低了使用门槛，但也意味着你对记忆内容的控制力有限。

## Overview：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级配置文件，作用类似于团队的编码规范手册——但读者是 AI，不是人。它告诉 Claude 这个项目用什么技术栈、遵循什么代码风格、哪些操作必须做、哪些绝对不能做。

CLAUDE.md 的核心设计理念是**显式控制**。每一条规则都是开发者手动编写的，内容跟着代码仓库走，通过 Git 版本控制。团队成员共享同一份 CLAUDE.md，确保 Claude 在不同人的机器上表现一致。

在 Claude Code 的[七层架构](/zh/blog/claude-code-seven-programmable-layers)中，CLAUDE.md 位于项目上下文层，是最基础也最关键的配置机制。它和 `SKILL.md`、Hooks 等机制共同构成了 Claude Code 的[完整扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **存储位置** | `~/.claude/` 本地文件系统 | 项目根目录，Git 版本控制 | CLAUDE.md |
| **作用范围** | 用户级（跨项目可用） | 项目级（仅当前仓库） | 各有适用场景 |
| **维护方式** | 自动提取，Claude 决定记什么 | 手动编写，开发者完全控制 | 看需求 |
| **团队共享** | 不可共享，绑定个人 | 通过 Git 共享给整个团队 | CLAUDE.md |
| **内容类型** | 用户偏好、角色、反馈记录 | 编码规范、构建命令、工作流规则 | 各有适用场景 |
| **可审计性** | 有限，需手动检查记忆文件 | 完全可审计，Git diff 可追溯 | CLAUDE.md |
| **上手成本** | 零——自动生效 | 需要手动编写和维护 | Memory |

## 持久化机制与作用域：核心架构差异

这是两者最根本的区别。Claude Memory 采用**用户级持久化**——记忆绑定到你的 Claude 账号和本地环境，无论你切换到哪个项目，之前积累的偏好和背景信息都跟着你走。这意味着如果你在项目 A 中告诉 Claude「我更喜欢函数式风格」，在项目 B 中它也会记住。

CLAUDE.md 采用**项目级持久化**——配置文件绑定到代码仓库，跟代码一起 commit、push、merge。换一个仓库，CLAUDE.md 就不同了。这种设计的好处是**关注点分离**：前端项目的 CLAUDE.md 写 React 规范，后端项目的写 Go 规范，互不干扰。

实际场景中，这种架构差异决定了什么信息该放哪里。你的 API key 前缀、时区偏好、commit message 风格——这些跨项目通用的信息，适合让 Memory 自动记住。而「这个项目用 Tailwind v4 不用 v3」「测试必须通过才能 commit」「不要在 pipeline 脚本里 import Next.js 模块」——这些项目特定的规则，必须写在 CLAUDE.md 里。

如果你把项目规则放进 Memory，团队其他人不会受益；如果你把个人偏好写进 CLAUDE.md，每个队友 clone 下来都会受到你个人习惯的影响。

## 团队协作与版本控制：工程化维度

对于个人开发者，Memory 和 CLAUDE.md 的区别可能只是「方便」与「精确」的取舍。但对于团队，两者的差距被急剧放大。

CLAUDE.md 的最大优势是**团队一致性**。一旦 commit 到仓库，所有开发者的 Claude Code 都会遵循同样的规则。新成员加入团队，`git clone` 之后就自动获得了完整的 AI 行为配置，不需要额外的 onboarding 步骤。这在实际的[企业工程实践](/zh/blog/claude-code-enterprise-engineering-ramp-shopify-spotify)中被反复验证：CLAUDE.md 是团队 AI 协作的基石。

Memory 则完全是个人化的。你积累的 Memory 无法分享给队友，也无法通过 PR review 来审查。这不是缺陷——而是设计意图。Memory 存储的是「这个人喜欢怎样工作」，不是「这个项目应该怎样构建」。

版本控制带来的另一个好处是**可审计性**。CLAUDE.md 的每次修改都有 Git 历史，可以追溯谁在什么时候加了什么规则，以及为什么。Memory 的变更记录则不那么透明——虽然你可以手动查看 memory 文件，但没有结构化的变更历史。

从 [Claude Code 记忆系统的完整设计](/zh/blog/claude-code-memory)来看，CLAUDE.md 是「规则手册」，Memory 是「自动笔记」——两者的定位从一开始就不同。

## 什么时候选 Claude Memory

**选 Memory 的信号：**

- **你是个人开发者**，不需要和团队共享 AI 行为配置
- **你的偏好跨项目通用**：比如「始终用中文回复」「commit message 用英文」「不要加 emoji」
- **你不想手动维护配置文件**，希望 Claude 自动适应你的风格
- **你刚开始用 Claude Code**，还不确定需要什么规则——让 Memory 先帮你积累

Memory 最大的优势是零配置成本。你不需要提前想好所有规则，只要正常使用，Claude 就会逐渐「认识」你。对于快速原型开发、个人项目、探索性编码，这种自适应的方式足够好用。

## 什么时候选 CLAUDE.md

**选 CLAUDE.md 的信号：**

- **团队项目**，需要所有成员的 Claude 表现一致
- **有明确的编码规范**、构建流程、质量门禁需要 AI 遵守
- **项目有特殊约束**：比如不能使用某些依赖、必须遵循特定的安全策略
- **你需要可审计性**，能通过 Git 追踪 AI 行为配置的变更
- **你使用 [Skills](/zh/blog/do-skills-actually-improve-your-agents-output) 和 Hooks**，需要 CLAUDE.md 作为基础配置层

CLAUDE.md 的价值和代码质量的关系是正相关的——项目越复杂、团队越大、规范越严格，CLAUDE.md 带来的收益越高。一个精心维护的 CLAUDE.md 相当于给 AI 请了一个永远在线的 tech lead。

## 结论

**不要把 Claude Memory 和 CLAUDE.md 当作二选一的问题——它们是互补的两层。** Memory 是你的个人 AI 助手记忆，CLAUDE.md 是项目的 AI 行为规范。最佳实践是同时使用：让 Memory 自动记住你的个人偏好和工作习惯，用 CLAUDE.md 显式定义项目规则和团队标准。

如果你只能先做一件事：**个人项目先用 Memory**（零成本启动），**团队项目先写 CLAUDE.md**（确保一致性）。然后逐步把两者都用起来。更多 Claude Code 配置技巧，参考我们的 [5 个每天都在用的 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。Claude Code 会同时加载两者的内容：CLAUDE.md 提供项目级规则，Memory 提供用户级偏好。如果两者有冲突，CLAUDE.md 中的显式规则通常优先——因为它是开发者有意编写的指令，而 Memory 是自动推断的。

### CLAUDE.md 写错了会怎么样？

和代码 bug 一样——Claude 会遵循错误的规则，产出不符合预期的结果。但好处是 CLAUDE.md 受 Git 版本控制，发现问题后可以 revert。建议在团队中将 CLAUDE.md 的修改纳入 code review 流程，像审查代码一样审查 AI 配置。

### Claude Memory 记错了怎么办？

你可以手动查看和编辑 Memory 文件（存储在 `~/.claude/projects/` 下），删除不准确的记忆条目。也可以在对话中直接告诉 Claude「忘记 X」来触发记忆删除。定期检查 Memory 内容是个好习惯，尤其是在项目技术栈发生重大变化之后。
