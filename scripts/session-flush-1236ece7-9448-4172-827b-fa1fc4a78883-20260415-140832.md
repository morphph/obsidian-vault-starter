**CLAUDE.md is for the project.** It answers: "How should anyone working on this codebase behave?" Build commands, test requirements, coding style, architectural constraints, deployment procedures — these are project truths that apply regardless of who's typing. When a new team member clones the repo, CLAUDE.md is already there, and Claude Code immediately knows the project's rules.

A well-written CLAUDE.md typically includes:

- Build and test commands (`npm run build`, `npm test`)
- Quality gates ("all tests must pass before commit")
- Architectural constraints ("don't import Next.js modules in pipeline scripts")
- Style guidelines ("newsletter tone: sharp tech insider briefing a busy founder")
- Known gotchas ("ZH content must use CJK word count, not English space-based tokenization")

**Memory is for the developer.** It answers: "What has Claude learned about *this specific person* and their current work?" Your role, your expertise level, your correction history, your workflow preferences — these are personal truths that vary between team members. A senior backend engineer and a junior frontend developer working on the same repo should get different explanations, different levels of detail, and different assumptions about what they already know.

Memory files are organized into four types:

- **User memories**: Your role, expertise, and perspective ("deep Go expertise, new to React")
- **Feedback memories**: Corrections and confirmations ("integration tests must hit a real database, not mocks")
- **Project memories**: Current initiatives and context ("merge freeze begins 2026-03-05 for mobile release")
- **Reference memories**: Pointers to external systems ("pipeline bugs tracked in Linear project INGEST")

The practical rule: **if a new team member needs to know it, put it in CLAUDE.md. If it's about how Claude should interact with you specifically, let Memory handle it.**

## Static vs Dynamic: Detailed Analysis

The second critical distinction is **how each system updates**, which has significant implications for reliability and maintenance.

**CLAUDE.md is static by design.** It changes only when a developer deliberately edits and commits it. This makes it predictable and auditable — you can `git blame` every line, review changes in PRs, and roll back if instructions cause problems. The tradeoff: CLAUDE.md doesn't adapt. If your project's conventions evolve and nobody updates the file, Claude Code follows stale instructions.

Maintaining CLAUDE.md is a manual responsibility. Teams that treat it as living documentation — updating it alongside code changes — get the most value. Teams that write it once and forget it end up with a file that drifts from reality. The [9 principles for writing great Claude Code skills](/blog/9-principles-writing-claude-code-skills) apply equally here: instructions should be specific, testable, and maintained.

**Memory is dynamic by design.** Claude creates and updates memory files during normal conversations without explicit user action. When you say "don't summarize after every response," Claude saves a feedback memory and respects it in future sessions. When you mention you're a data scientist focused on observability, Claude saves a user memory and tailors explanations accordingly.

The tradeoff: Memory is less predictable. You don't review Memory changes in PRs. A misunderstood correction could become a persistent bad instruction. Claude Code mitigates this with structured frontmatter (making memories searchable and categorizable) and staleness checks (verifying memories against current code state before acting on them). But the responsibility for Memory accuracy is shared between you and Claude — if something feels off, you can ask Claude to check or update its memories.

**Decision rule: Use CLAUDE.md for instructions that need to be reviewed, shared, or rolled back. Use Memory for context that should evolve organically with your working relationship.**

## CLAUDE.md Hierarchy: Global, Project, and Directory Levels

CLAUDE.md supports a three-tier hierarchy, each serving a different scope:

**Global CLAUDE.md** (`~/.claude/CLAUDE.md`) applies to every project you work on. This is where personal cross-project preferences go — your git workflow rules, your preferred commit message style, your working habits. It's the equivalent of a global gitconfig but for Claude Code behavior.

**Project CLAUDE.md** (`./CLAUDE.md` in the repo root) applies to one specific repository. Build commands, architecture decisions, style guides, known gotchas — everything the team needs Claude to know about this particular codebase. This is the file most developers think of when they hear "CLAUDE.md."

**Directory CLAUDE.md** (e.g., `src/api/CLAUDE.md`) applies to a specific subdirectory. This is useful for monorepos or projects with distinct sub-components that have their own conventions. A `scripts/CLAUDE.md` might specify pipeline-specific rules while `src/components/CLAUDE.md` covers frontend conventions.

Memory doesn't have this hierarchy. It operates at one level: per-user, per-project. Your memories for the LoreAI project are separate from your memories for other projects, but there's no sub-project memory scoping. This is usually fine because Memory captures personal context (your preferences, your corrections) that applies across the entire project regardless of which directory you're working in.

## Memory Types and When They Trigger

Claude Memory's four types aren't just organizational labels — they determine when Claude writes a memory and how it uses the information later.

**User memories** trigger when Claude learns about your role, expertise, or perspective. These shape how Claude communicates with you. If Claude knows you're a senior engineer, it skips basic explanations. If it knows you're new to React but experienced in Go, it frames frontend concepts using backend analogies.

**Feedback memories** trigger when you correct Claude's approach or confirm an unusual choice. These are the highest-impact memory type because they directly prevent repeated mistakes. The key nuance: Claude should save both corrections ("don't do X") *and* confirmations ("yes, that approach was right"). Only saving corrections makes Claude overly cautious over time.

**Project memories** trigger when you share context about ongoing work, deadlines, or team decisions. These have the shortest shelf life — a merge freeze from last month is no longer relevant. Claude is expected to verify project memories against current state before acting on them.

**Reference memories** trigger when you mention external systems and their purpose — where bugs are tracked, which dashboard to check, where documentation lives. These are stable but narrow: they tell Claude where to look, not what to find.

## Practical Workflows: Using Both Systems Together

The most effective Claude Code setup uses CLAUDE.md and Memory in tandem. Here's how they complement each other in common workflows:

**Onboarding a new team member:**
The new developer clones the repo and gets CLAUDE.md immediately — build commands, conventions, quality gates. As they work with Claude Code over their first week, Memory builds up: their expertise areas, their preferred explanation depth, corrections they've given. By week two, Claude Code knows both the project rules (CLAUDE.md) and the person (Memory).

**Enforcing code review standards:**
CLAUDE.md defines the standards: "all tests must pass before commit," "no direct Next.js imports in pipeline scripts." Memory remembers individual reviewer preferences: "this developer wants terse PR descriptions," "this developer prefers single bundled PRs over many small ones for refactors."

**Debugging across sessions:**
CLAUDE.md provides the structural context: where scripts live, what the pipeline stages are, known gotchas. Memory provides the investigative context: "last session we identified the bug is in stage 3 filtering," "the user mentioned this started after the March deployment." Together, they let Claude pick up debugging work across multiple conversations without losing thread.

**Multi-project work:**
Your global CLAUDE.md keeps your personal workflow rules consistent ("always git pull before starting, always commit and push after changes"). Each project's CLAUDE.md provides repo-specific context. Memory adds per-project personal context — your role might differ between projects, or you might have different correction histories.

## Common Mistakes and How to Avoid Them

**Mistake 1: Putting personal preferences in CLAUDE.md.**
If you add "I prefer verbose explanations" to the project's CLAUDE.md, every team member's Claude Code sessions become verbose. Personal preferences belong in Memory (or your global `~/.claude/CLAUDE.md` if they apply everywhere).

**Mistake 2: Relying on Memory for team-critical instructions.**
Memory is per-user. If you tell Claude "always run the validation script before committing" and it saves this as feedback memory, only your sessions follow that rule. If it's a team requirement, it belongs in CLAUDE.md.

**Mistake 3: Never updating CLAUDE.md.**
A CLAUDE.md that says "use Jest for testing" when the project migrated to Vitest six months ago actively hurts. Treat CLAUDE.md as living documentation — update it when the project changes.

**Mistake 4: Duplicating CLAUDE.md content in Memory.**
Memory explicitly should not store code patterns, architecture, file paths, or project structure — these are derivable from CLAUDE.md and the codebase itself. Duplicating this information creates staleness risk without adding value.

**Mistake 5: Expecting Memory to be perfectly reliable.**
Memory records can become stale. Claude is designed to verify memories against current state before acting on them, but this isn't foolproof. If Claude makes a recommendation based on a memory ("the memory says function X exists"), verify before acting. The [Claude Code complete guide](/blog/claude-code-complete-guide) covers memory verification patterns in detail.

## When to Choose Claude Memory

Choose Claude Memory when the context is:

- **Personal**: Your expertise, your preferences, your communication style
- **Evolving**: Information that changes frequently and shouldn't require a commit to update
- **Conversational**: Corrections and confirmations that emerge during work, not planned in advance
- **Ephemeral-adjacent**: Current project status, ongoing investigations, temporary states that matter for weeks, not months
- **External**: Pointers to resources outside the codebase — Linear projects, Grafana dashboards, Slack channels

Memory shines for the unwritten knowledge that makes pair programming effective: your partner remembers that you hate mocks, that you're investigating a specific bug, that you prefer functional patterns. Claude Memory is the mechanism that gives Claude Code a similar adaptive quality across sessions.

## When to Choose CLAUDE.md

Choose CLAUDE.md when the context is:

- **Team-shared**: Rules, commands, and conventions that every developer must follow
- **Stable**: Information that changes infrequently and should be version-controlled when it does
- **Reviewable**: Instructions where getting it wrong has consequences — quality gates, deployment procedures, architectural constraints
- **Structural**: Build commands, file organization patterns, pipeline stages, known gotchas
- **Onboarding-critical**: Anything a new team member needs on day one without prior Claude Code interaction

CLAUDE.md is the deterministic foundation. It ensures that every Claude Code session — regardless of who's running it or how much Memory they've accumulated — starts with the same baseline understanding of the project. For teams using Claude Code across [agentic coding](/glossary/agentic-coding) workflows, this consistency is essential.

## Verdict

**Use both.** CLAUDE.md and Claude Memory are complementary layers, not alternatives. **CLAUDE.md is your project's constitution** — the shared rules and context that apply to everyone. **Claude Memory is your personal working relationship with Claude** — the preferences, corrections, and context that make each session more effective than the last.

If you're starting from scratch: **write your CLAUDE.md first.** It delivers immediate, team-wide value from the first session. Memory builds naturally over time as you work — there's nothing to set up. As your CLAUDE.md matures, you'll find Memory fills the gaps that a static file can't: adapting to your communication style, remembering your corrections, and tracking the project's evolving state between file updates.

## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

Both load into context at conversation start, but CLAUDE.md instructions are treated as project rules while Memory provides supplementary context. In practice, if CLAUDE.md says "always run tests before committing" and a feedback memory says "skip tests for docs-only changes," Claude weighs the explicit project rule more heavily. Memory informs Claude's approach; CLAUDE.md constrains it.

### Does Claude Memory sync across devices?

Memory files are stored locally in `.claude/projects/` directories. They do not automatically sync across machines. If you use Claude Code on both a laptop and a remote server, each environment maintains separate memory. CLAUDE.md, being checked into git, syncs wherever you pull the repository.

### How do I see what Claude has saved in Memory?

Memory files are plain markdown with YAML frontmatter stored in `.claude/projects/{project-hash}/memory/`. The `MEMORY.md` file in that directory serves as an index of all saved memories. You can read, edit, or delete any memory file directly — they're just text files on disk.

### Should I commit the .claude/ directory?

No. The `.claude/` directory contains per-user data including Memory files and is typically gitignored. CLAUDE.md lives in your project root and should be committed. This separation is intentional: project instructions are shared, personal context is private.

### How much Memory is too much?

The `MEMORY.md` index is truncated after 200 lines in context loading. More importantly, every memory file loaded adds to the conversation context, reducing space for actual work. Focus memories on genuinely useful, non-obvious information. Code patterns, file paths, and project structure should not be stored in Memory — Claude can derive these by reading the codebase directly.
