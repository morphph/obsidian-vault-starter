# Codex Subagents vs Custom Agents：哪种多智能体模式更适合你？

**TL;DR:** **Subagents** 适合在单次任务中拆分并行子任务，由系统自动调度，开箱即用；**Custom Agents** 适合定义可复用的专业化角色，需要前期配置但长期收益更高。小团队快速迭代选 Subagents，企业级标准化工作流选 Custom Agents，成熟团队两者结合效果最佳。

## 概述：Codex Subagents

**Subagents** 是 [OpenAI Codex](/zh/blog/codex-complete-guide) 内置的并行任务分解机制。当主 Agent 遇到可以拆分的复杂任务时，它会自动生成多个子 Agent，每个子 Agent 在独立的沙箱环境中处理一个子任务，完成后将结果汇总回主 Agent。

Subagents 的核心价值在于**自动化的任务并行**。你不需要预先定义 Agent 的行为模式或能力边界——只要给 Codex 一个足够复杂的任务，它会自行判断是否需要拆分，如何拆分，以及如何合并结果。这种模式降低了使用门槛，适合探索性开发和一次性的大规模操作。

典型场景包括：跨多个文件的重构、批量测试生成、多模块并行 code review。Subagents 在 Codex 的云端沙箱中运行，每个子 Agent 拥有独立的文件系统快照，互不干扰。

## 概述：Codex Custom Agents

**Custom Agents** 是 Codex 提供的可配置专业化 Agent 模式。开发者通过定义 Agent 的系统指令、可用工具集、行为约束和输出格式，创建面向特定任务类型的专用 Agent。

Custom Agents 解决的是**工程标准化**问题。在团队协作中，不同成员对"写好测试"或"做好 code review"的理解不一致。通过 Custom Agent，你可以将团队的最佳实践编码为可复用的 Agent 配置，确保每次执行都遵循相同的质量标准。

这一模式类似于 [Claude Code 的 Skills 系统](/zh/blog/9-principles-writing-claude-code-skills)，但运行在 Codex 的云端环境中。Custom Agents 可以指定使用的模型、可调用的工具、输出格式要求，甚至与外部 API 集成。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **启动方式** | 系统自动生成 | 开发者预先配置 | Subagents |
| **并行能力** | 原生支持多任务并行 | 需通过编排层实现 | Subagents |
| **可定制程度** | 低——由系统决定行为 | 高——完全自定义指令和工具 | Custom Agents |
| **复用性** | 一次性，任务结束即销毁 | 持久配置，跨任务复用 | Custom Agents |
| **团队协作** | 个人使用为主 | 可共享、可版本控制 | Custom Agents |
| **上手成本** | 零配置 | 需要编写 Agent 定义 | Subagents |
| **质量一致性** | 取决于主 Agent 的拆分策略 | 由预设规则保证 | Custom Agents |
| **调试难度** | 较高——自动生成的子任务链路不透明 | 较低——行为由配置决定，可预测 | Custom Agents |

## 任务并行与自动调度：深度分析

Subagents 的最大优势是**零配置的任务并行**。在实际使用中，当你向 Codex 提交一个涉及多个模块的重构任务时，系统会自动分析代码依赖关系，将无依赖的模块分配给不同的 Subagent 并行处理。这个过程完全由系统决策，开发者不需要手动拆分任务。

这种模式在[多智能体工作流](/zh/blog/con-u-pour-des-workflows-multi-agents)中表现出色。比如，一个"为所有 API 端点添加输入验证"的任务，Subagents 可以同时处理 10 个端点，每个子 Agent 独立修改、独立测试，最后合并结果。总耗时接近单个端点的处理时间，而非线性叠加。

但 Subagents 的局限也很明显：你无法控制拆分策略。如果系统判断某些子任务存在依赖关系，它会串行执行，而这个判断不一定准确。此外，Subagents 之间无法直接通信——如果子任务 B 的最优解取决于子任务 A 的中间结果，当前架构无法支持这种协作模式。

相比之下，Custom Agents 的并行需要通过外层编排实现，但你对每个 Agent 的行为拥有完全控制权。

## 可定制性与工程标准化：深度分析

Custom Agents 的核心竞争力在于**将隐性知识显性化**。以 code review 为例：一个团队可能要求检查安全漏洞、性能瓶颈、API 设计一致性和测试覆盖率。用 Subagents 执行 review，每次的检查维度和深度取决于模型的即时判断，结果不稳定。用 Custom Agent，你可以精确定义：

- 必须检查的安全规则清单
- 性能分析的具体指标和阈值
- 命名规范和 API 设计模式
- 测试覆盖率的最低要求

这种确定性在企业环境中至关重要。当团队有 20 个工程师同时使用 Codex，Custom Agents 确保每个人得到的 review 结果遵循相同标准。

Custom Agents 的配置还可以纳入版本控制，像代码一样进行 review 和迭代。这与 [Claude Code 的 SKILL.md 体系](/zh/blog/5-claude-code-skills-i-use-every-single-day)理念一致——把 AI 行为当作工程资产管理。相比之下，Codex 的实现方式更侧重云端执行和 API 集成能力。

不过，Custom Agents 的前期投入不可忽视。定义一个好用的 Agent 需要反复测试和调优，这个过程本身就是工程成本。对于一次性任务或探索性开发，这个成本往往不值得。

## 什么时候用 Subagents

选择 Subagents 的典型场景：

- **一次性大规模操作**：迁移数据库 schema、批量重命名、全仓库依赖升级。这些任务执行一次就结束，不需要复用 Agent 配置
- **探索性开发**：不确定最优方案时，让 Subagents 并行尝试多个方向，快速得到反馈
- **个人开发者**：如果你是独立开发者，没有团队标准化需求，Subagents 的零配置特性让你专注于任务本身
- **原型验证**：快速搭建 MVP 时，Subagents 的自动拆分能力可以显著加速多模块并行开发

如果你刚开始使用 Codex，[Codex 完全指南](/zh/blog/codex-complete-guide)是最好的入门资料，其中详细介绍了 Subagents 的触发条件和使用技巧。

## 什么时候用 Custom Agents

选择 Custom Agents 的典型场景：

- **重复性工程任务**：每个 PR 都需要的 code review、每个新模块都需要的测试生成、每次部署前的安全扫描——这些高频任务值得投入配置成本
- **团队协作**：5 人以上的工程团队，需要确保 AI 辅助的输出质量一致。Custom Agents 是强制标准化的最可靠手段
- **合规要求**：金融、医疗等行业对代码审查有明确的检查清单要求，Custom Agents 可以确保每次都覆盖所有必检项
- **CI/CD 集成**：将 Custom Agent 嵌入持续集成流水线，自动触发特定类型的代码分析或生成任务

对于正在构建多 Agent 工作流的团队，可以参考[多智能体工作流的工程范式](/zh/blog/con-u-pour-des-workflows-multi-agents)中的架构设计思路。[Codex VS Code 扩展](/zh/blog/codex-vscode)也支持在 IDE 中直接调用 Custom Agents。

## 结论

**如果你追求开箱即用的并行效率，从 Subagents 开始；如果你需要可复用、可预测、可协作的 AI 工程能力，投资 Custom Agents。** 两者不是互斥关系——成熟的工程实践是用 Custom Agents 定义标准化流程，在 Custom Agent 内部利用 Subagents 实现并行执行。先从 Subagents 入手理解 Codex 的多 Agent 机制，再根据团队实际需求逐步引入 Custom Agents，是最务实的路径。

与 [Claude Code 的 Agent Teams](/zh/blog/claude-code-agent-teams) 类似，Codex 的多 Agent 体系正在从"一个 AI 助手"演进为"一个 AI 工程团队"。选择哪种模式取决于你是需要一个灵活的临时工，还是一个训练有素的专职员工。

## 常见问题

### Subagents 会消耗额外的 API 额度吗？

会。每个 Subagent 独立运行在 Codex 的云端沙箱中，独立消耗 token 和计算资源。任务拆分为 5 个 Subagent 并行执行时，总 token 消耗通常高于单 Agent 串行执行，但总耗时更短。在使用前建议评估任务复杂度，避免不必要的拆分导致额度浪费。

### Custom Agents 的配置可以跨项目复用吗？

可以。Custom Agent 的定义本质上是一份结构化的指令文件，可以在不同项目间共享。建议将 Agent 配置纳入版本控制，像管理代码一样管理 Agent 行为。不过，跨项目复用时需要注意项目特定的约束（如命名规范、技术栈差异），通常需要做适当调整。

### 两种模式可以结合使用吗？

可以，这也是推荐的成熟用法。Custom Agent 定义整体任务的执行标准和工具集，在执行过程中由系统自动触发 Subagents 处理可并行的子任务。这样既保证了行为的可预测性，又充分利用了并行加速能力。
