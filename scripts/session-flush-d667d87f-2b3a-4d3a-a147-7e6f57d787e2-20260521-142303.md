# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **CLAUDE.md** 是写给整个团队的规则手册——确定性强、版本可控、适合编码规范和项目约束。**Claude Memory** 是写给你个人的自动笔记——自适应、跨会话、适合偏好和上下文积累。**大多数团队应该两者并用：CLAUDE.md 管规则，Memory 管个人上下文。** 只用其中一个，要么丢了团队一致性，要么丢了个性化效率。

## 概览：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Claude Code 内置的自动记忆系统，它在对话过程中识别值得记住的信息，自动写入 `~/.claude/projects/` 目录下的 Markdown 文件。这些记忆跨会话持久化——下次打开 Claude Code 时，它已经知道你是谁、你喜欢什么风格、项目当前在做什么。

Memory 的核心价值是**自适应**。你不需要手动维护配置文件，Claude 会从对话中提取四类信息：用户画像（你的角色和知识水平）、行为反馈（你纠正过的做法）、项目动态（正在进行的工作）、外部引用（Linear 看板、Grafana 仪表盘等）。根据 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory) 的分析，这套机制让 Claude 在多轮协作中越来越懂你的工作方式。

Anthropic 在 2026 年初对记忆系统做了重大升级，[支持从 ChatGPT 导入历史数据](/zh/blog/anthropic-claude-memory-upgrades-importing)，免费用户也能使用基础记忆功能。

## 概览：CLAUDE.md

**CLAUDE.md** 是放在项目根目录（或 `~/.claude/` 全局目录）的 Markdown 指令文件，Claude Code 每次启动时自动加载。它的本质是一份**显式的规则手册**——你写什么，Claude 就严格遵守什么。

CLAUDE.md 的核心价值是**确定性和可共享性**。它被提交到 Git 仓库，团队每个人拉下来都是同一份规则。典型内容包括：构建命令（`npm run build`、`npm test`）、代码风格约束、架构决策、禁止事项、工作流程。在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers) 中，CLAUDE.md 位于最基础的项目上下文层，是所有上层能力（Skills、Hooks、MCP）的根基。

CLAUDE.md 还支持层级覆盖：全局 `~/.claude/CLAUDE.md` 定义个人通用规则，项目级 `CLAUDE.md` 定义团队规范，两者合并生效。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **持久化方式** | 自动写入本地文件 | 手动编写，Git 版本控制 | 各有所长 |
| **共享范围** | 仅限个人 | 团队共享（通过 Git） | CLAUDE.md |
| **内容来源** | Claude 从对话中自动提取 | 开发者手动编写 | Memory |
| **确定性** | 概率性——Claude 决定记什么 | 确定性——写什么就执行什么 | CLAUDE.md |
| **维护成本** | 几乎为零 | 需要人工维护和更新 | Memory |
| **适用场景** | 个人偏好、项目动态、行为校正 | 编码规范、构建命令、架构约束 | 各有所长 |
| **可审计性** | 低——记忆文件分散且自动生成 | 高——Git history 完整追溯 | CLAUDE.md |
| **团队一致性** | 无保障——每人记忆不同 | 强保障——同一份规则 | CLAUDE.md |

## 记忆机制：详细对比

Claude Memory 和 CLAUDE.md 在底层机制上有本质区别，理解这一点才能用对。

**Claude Memory 是推断式的。** Claude 在对话中观察你的行为模式——你纠正了一次 commit message 风格，它就记住"这个用户偏好简洁的 commit message"。这种机制的优势是零配置、自进化；劣势是不可预测——你不确定 Claude 到底记住了什么、会不会在关键时刻忘记。Memory 文件虽然可读（存储在 `~/.claude/projects/` 下的 Markdown 文件中），但大多数开发者不会定期审查。

**CLAUDE.md 是声明式的。** 你写下 `npm test 必须在每次 commit 前通过`，Claude 就会在每次提交前运行测试。没有推断、没有遗忘、没有歧义。这种确定性在团队协作中至关重要——你不能指望每个团队成员的 Memory 都恰好记住了同一条规则。

根据 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 的架构分析，CLAUDE.md 和 Memory 分别工作在不同的层级：CLAUDE.md 是基础层的项目上下文，Memory 是会话层的个人上下文。两者互补而非替代。

## 协作场景：谁更适合团队？

**单人项目中，Memory 可能就够了。** 你的偏好、项目状态、常用命令——Claude 在几轮对话后就能自动学会。不需要花时间写 CLAUDE.md，Memory 的零维护成本是真正的优势。

**团队项目中，CLAUDE.md 是必需品。** 想象一个五人团队，每人的 Memory 记住了不同版本的代码风格规则。结果？同一个仓库里 Claude 给不同人生成的代码风格不一致。CLAUDE.md 通过 Git 共享，确保团队里每个 Claude Code 实例遵守同一套规范。这也是为什么 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills) 强调将关键规则固化到文件中而非依赖口头约定。

**最佳实践是两者并用：**

1. **CLAUDE.md 管「团队规则」**——构建命令、lint 配置、Git 工作流、架构约束、安全红线
2. **Memory 管「个人上下文」**——你的角色（前端/后端/全栈）、沟通偏好（简洁/详细）、当前任务进度、外部系统引用

这正是 [我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day) 中推荐的配置策略。

## 适合选择 Claude Memory 的场景

当你的需求是**个人化和自适应**时，Memory 是更好的选择：

- **个人开发者或自由职业者**：项目切换频繁，不想为每个项目维护 CLAUDE.md
- **行为校正**：告诉 Claude 一次"不要在函数末尾加多余的换行"，它就永远记住——不需要写进配置文件
- **项目动态追踪**：当前冲刺的目标、正在调查的 bug、即将到来的 deadline——这些信息变化太快，不适合写进 CLAUDE.md
- **外部系统映射**：Linear 看板在哪里、Grafana 仪表盘 URL、Slack 频道用途——这些是个人导航信息，不是团队规则

## 适合选择 CLAUDE.md 的场景

当你的需求是**确定性和团队一致性**时，CLAUDE.md 是唯一选择：

- **团队编码规范**：命名约定、测试策略、提交规则——必须所有人统一
- **构建和部署命令**：`npm run build`、`npm test`、部署流程——不能靠记忆
- **架构约束和安全红线**：禁止在 pipeline 脚本中 import Next.js 模块、禁止跳过失败的测试——这些是硬性规则
- **新成员 onboarding**：新人 clone 仓库就自动获得所有项目规则，不需要老成员口头传授
- **可审计场景**：需要追溯「为什么 Claude 会这样做」时，CLAUDE.md 的 Git history 提供了完整的变更记录

## 结论

**Claude Memory 和 CLAUDE.md 不是二选一的关系，而是互补的两层。** CLAUDE.md 解决的是「团队里每个人的 Claude 都应该遵守什么规则」的问题——它是确定性的、可共享的、可审计的。Memory 解决的是「我的 Claude 怎么越用越懂我」的问题——它是自适应的、零维护的、个人化的。

**推荐策略：** 任何超过一人的项目，先写好 CLAUDE.md（构建命令、代码规范、架构约束）。然后让 Memory 自然积累个人上下文。定期检查 Memory 文件（`~/.claude/projects/` 目录），清理过时的记忆。如果你还没有设置过这两个系统，可以从 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory) 开始了解完整的配置方法。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。CLAUDE.md 负责团队共享的硬性规则（构建命令、代码规范），Memory 负责个人化的软性上下文（偏好、当前任务进度）。两者在不同层级工作，不会冲突。

### 如果 Memory 和 CLAUDE.md 的内容矛盾了怎么办？

CLAUDE.md 的优先级更高。它是显式声明的规则，Claude Code 会优先遵守。但实际场景中矛盾很少发生，因为两者的内容类型本就不同——CLAUDE.md 管规则，Memory 管上下文。

### Memory 会不会记住错误的信息？

会。Memory 是从对话中自动提取的，可能记住过时的项目状态或误解你的意图。好消息是 Memory 文件是普通的 Markdown 文件，你可以直接编辑或删除。建议每隔几周检查一次 `~/.claude/projects/` 目录。

### 新团队成员需要手动配置 Memory 吗？

不需要。Memory 会在对话中自动积累。但 CLAUDE.md 会在第一次使用时就生效——这正是它的优势：新成员 clone 仓库后，Claude Code 立刻知道项目的所有规则和约束。
