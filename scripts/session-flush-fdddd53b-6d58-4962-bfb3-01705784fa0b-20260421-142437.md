# Claude Memory vs CLAUDE.md：两套记忆系统怎么选？

**TL;DR：** **CLAUDE.md** 是团队共享的项目规则手册——写一次，所有人（包括 AI）都遵守。**Claude Memory（Auto Memory）** 是个人化的自动笔记本——AI 在对话中观察到的偏好、习惯、上下文会自动沉淀。两者不是替代关系，而是互补：**CLAUDE.md 管"项目该怎么做"，Memory 管"你习惯怎么做"。** 搞混了放置位置，轻则信息冗余，重则团队成员收到不一致的 AI 行为。

## 概览：Claude Memory（Auto Memory）

Claude Memory 是 Claude Code 的自动持久化记忆系统，存储在 `.claude/projects/` 目录下的 `memory/` 文件夹中。它的核心机制是：AI 在对话过程中自动识别值得记住的信息——你的角色、偏好、项目背景、工作反馈——然后写入独立的 markdown 文件，并维护一个 `MEMORY.md` 索引。

Memory 的关键特征是**个人化**和**自动化**。你不需要手动维护，Claude 会在你纠正它的做法、提到项目背景、或表达偏好时主动记录。这些记忆在后续对话中自动加载，让 AI 不用每次从零开始理解你。Anthropic 近期还[升级了记忆系统](/zh/blog/anthropic-claude-memory-upgrades-importing)，支持从 ChatGPT 导入数据，进一步降低了迁移成本。

但 Memory 也有明显的局限：它是**本地的、个人的**。你的 Memory 不会同步到队友的环境，也不会进入版本控制。

## 概览：CLAUDE.md

CLAUDE.md 是 Claude Code 的项目级指令文件，放在仓库根目录，随代码一起提交到 Git。每次 Claude Code 启动会话时，它会自动读取当前项目的 CLAUDE.md，作为整个对话的基础上下文。

CLAUDE.md 的核心特征是**显式、确定性、团队共享**。同一个文件，所有团队成员看到的 AI 行为一致。它通常包含：构建命令、代码风格要求、架构约定、质量门禁规则、已知陷阱等。你可以把它理解为"给 AI 读的项目 README"。

在 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中，CLAUDE.md 位于最基础的上下文层——它定义了 AI 理解项目的起点。配合 [Skills 系统](/zh/blog/9-principles-writing-claude-code-skills)，CLAUDE.md 构成了可复现、可版本化的 AI 行为基础设施。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **存储位置** | `.claude/projects/*/memory/` | 仓库根目录 | 各有定位 |
| **版本控制** | 不进 Git | 提交到 Git | **CLAUDE.md** |
| **团队共享** | 仅个人可见 | 所有成员共享 | **CLAUDE.md** |
| **维护方式** | AI 自动写入 | 人工手动维护 | **Memory** |
| **内容类型** | 用户偏好、项目背景、工作反馈 | 构建命令、代码规范、架构约定 | 各有定位 |
| **生效范围** | 跨会话持久化 | 每次会话自动加载 | 平手 |
| **信息密度** | 分散在多个文件 | 集中在一个文件 | **CLAUDE.md** |
| **过时风险** | 可能积累陈旧记忆 | 随代码一起更新 | **CLAUDE.md** |

## 信息归属：什么该放哪里

这是实际使用中最容易搞混的问题。一条信息到底该写进 CLAUDE.md 还是让 Memory 自动记住？判断标准很简单：**这条信息是属于项目的，还是属于你个人的？**

**放进 CLAUDE.md 的信息：**
- 构建、测试、部署命令（`npm run build`、`npm test`）
- 代码风格和架构约定（"使用 Tailwind v4"、"API 路由放在 `src/app/api/`"）
- 质量门禁规则（"提交前必须通过所有测试"）
- 已知陷阱和 gotchas（"ZH 内容必须用 CJK 字符计数"）
- 团队工作流约定（"新功能先讨论设计，获批后再写代码"）

**让 Memory 自动记录的信息：**
- 你的角色和技术背景（"我是数据科学家，主要看日志系统"）
- 沟通偏好（"不要在回复末尾总结"、"用简短风格"）
- 工作习惯反馈（"集成测试必须跑真实数据库，不要 mock"）
- 项目动态（"本周四之后冻结非关键合并"）
- 外部资源指向（"pipeline bug 在 Linear 的 INGEST 项目里追踪"）

**决策规则：** 如果删掉这条信息，你的队友会不会踩坑？会→ CLAUDE.md。只有你自己在意？→ Memory。

## 协同机制：两层上下文如何叠加

Claude Code 在每次会话启动时，会同时加载 CLAUDE.md 和 Memory。理解它们的叠加逻辑很重要：

**CLAUDE.md 是基础层。** 它定义了项目的"宪法"——所有人必须遵守的规则。Claude 在生成代码、运行命令、做决策时，首先参考 CLAUDE.md 中的约束。

**Memory 是个性化层。** 它在 CLAUDE.md 的基础上叠加你的个人偏好。比如 CLAUDE.md 规定"commit message 用英文"，而你的 Memory 记录了"这个用户喜欢简短的 commit message"——两者不冲突，Claude 会同时遵守。

但如果两者矛盾，CLAUDE.md 优先。这是设计上的正确选择：团队规范大于个人偏好。

在实际的 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中，CLAUDE.md 和 Memory 分别对应不同的架构层级：CLAUDE.md 属于静态上下文注入层，Memory 属于动态上下文积累层。理解这个层级关系，有助于你设计更合理的 AI 工作流。

## 实战建议：从零开始的配置策略

**第一步：先写好 CLAUDE.md。** 这是投入产出比最高的动作。一个好的 CLAUDE.md 包含三部分：常用命令（构建/测试/部署）、代码规范（风格/架构/禁忌）、工作流规则（提交前检查、PR 流程）。参考[每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)中的实践案例。

**第二步：让 Memory 自然积累。** 不要试图预先填充 Memory。正常使用 Claude Code，当你纠正它的行为（"不要这样做"）或表达偏好（"我喜欢这种方式"）时，它会自动记住。几次对话之后，Memory 会自然形成你的个人画像。

**第三步：定期清理 Memory。** Memory 会过时。项目背景变了、技术栈升级了、团队流程调整了——旧的 Memory 记录可能变成误导。建议每月检查一次 `memory/MEMORY.md` 索引，删除不再准确的条目。

**第四步：CLAUDE.md 随代码迭代。** 每次修改架构、更换工具链、调整工作流时，同步更新 CLAUDE.md。它应该像 README 一样，始终反映项目的当前状态。

## 常见误区

**误区一："Memory 能替代 CLAUDE.md。"** 不能。Memory 是个人的、隐式的、可能过时的。团队规范必须写在 CLAUDE.md 里，否则新成员加入时 AI 行为不一致。

**误区二："把所有东西都塞进 CLAUDE.md。"** CLAUDE.md 不是百科全书。它应该精简、聚焦于 AI 需要知道的规则。个人偏好、临时项目状态、外部资源指向——这些交给 Memory。CLAUDE.md 膨胀到几千行时，Claude 的注意力分配反而会变差。

**误区三："Memory 会自动同步到团队。"** 不会。Memory 存储在本地 `.claude/` 目录下，不进版本控制。如果你发现一条 Memory 对整个团队都有价值，应该把它提炼后写进 CLAUDE.md。

## 结论

**CLAUDE.md 和 Claude Memory 解决的是不同层面的问题。** CLAUDE.md 是项目的规则手册，确保团队一致性和 AI 行为的可预测性。Memory 是个人的上下文积累，让 AI 越来越懂你的工作方式。最佳实践是：**CLAUDE.md 写少写精，Memory 让它自然生长，定期修剪。** 两者配合使用，才能发挥 Claude Code 的[完整七层架构](/zh/blog/claude-code-seven-programmable-layers)能力。想深入了解 Memory 的技术实现，推荐阅读我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 常见问题

### Claude Memory 的数据存储在哪里？

Claude Memory 存储在本地文件系统的 `.claude/projects/` 目录下，以 markdown 文件形式组织。每条记忆是一个独立文件，包含 frontmatter 元数据（名称、类型、描述）和正文内容。`MEMORY.md` 作为索引文件，在每次会话开始时自动加载到上下文中。

### CLAUDE.md 支持多层级配置吗？

支持。Claude Code 会按层级加载多个配置：全局 `~/.claude/CLAUDE.md` 对所有项目生效，项目根目录的 `CLAUDE.md` 对当前项目生效，子目录中的 `CLAUDE.md` 对特定模块生效。层级越深，优先级越高——子目录规则会覆盖项目级规则的同名配置。

### 两者冲突时以哪个为准？

CLAUDE.md 优先。这是 Claude Code 的设计原则：显式的、团队共享的规则优先于隐式的、个人的记忆。如果 Memory 中的某条记录与 CLAUDE.md 中的规则矛盾，Claude 会遵守 CLAUDE.md 的约定。发现冲突时，建议更新或删除过时的 Memory 条目。
