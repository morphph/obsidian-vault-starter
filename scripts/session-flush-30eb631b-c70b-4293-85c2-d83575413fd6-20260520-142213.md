# Claude Memory vs CLAUDE.md：哪个才是你真正需要的上下文管理方案？

**TL;DR：** **CLAUDE.md** 是写给整个团队的项目规则手册——构建命令、代码规范、架构约束，签入 Git，所有人共享同一套指令。**Claude Memory** 是 Claude Code 为每个用户自动积累的个人笔记——你的角色、偏好、反馈历史，跨会话持久化但不进版本控制。简单说：**CLAUDE.md 管项目规则，Memory 管个人上下文**。两者不是替代关系，而是互补的两层配置。

## 概述：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Claude Code 的跨会话记忆系统，解决的核心问题是：每次开新对话都要重复解释"我是谁、我在做什么、我喜欢怎样工作"。Memory 以 Markdown 文件的形式存储在 `.claude/projects/` 目录下，包含一个 `MEMORY.md` 索引文件和若干独立的记忆条目文件。

Memory 分为四种类型：**user**（用户角色和偏好）、**feedback**（行为纠正和确认）、**project**（项目动态和决策）、**reference**（外部资源指针）。Claude Code 在对话中自动判断何时写入记忆——当你纠正它的做法、透露自己的技术背景、或提到项目截止日期时，它会将这些信息持久化。下次对话开始时，这些记忆自动加载，Claude Code 就能"记住"你是谁。

关键特征：**个人的、隐式的、自动积累的**。Memory 不需要你手动维护，也不会出现在 Git 历史中。更多细节见我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件——一个 Markdown 文件，放在项目根目录，签入版本控制，团队所有成员共享。它告诉 Claude Code 这个项目的规则：用什么构建命令、遵循什么代码风格、有哪些已知的坑需要避开。

CLAUDE.md 的内容是开发者显式编写的。典型内容包括：构建和测试命令（`npm run build`、`npm test`）、提交前的质量门禁、代码风格偏好、架构约束、以及项目特有的注意事项。Claude Code 每次启动时都会读取这个文件，将其作为系统级上下文注入对话。

关键特征：**团队的、显式的、版本控制的**。CLAUDE.md 是你主动写下的规则，不是 AI 自动生成的。它和 [SKILL.md 技能文件](/zh/blog/9-principles-writing-claude-code-skills)一起，构成了 Claude Code [七层可编程架构](/zh/blog/claude-code-seven-programmable-layers)的基础配置层。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **存储位置** | `.claude/projects/*/memory/` | 项目根目录 | — |
| **版本控制** | 不进 Git | 签入 Git | CLAUDE.md |
| **共享范围** | 仅当前用户 | 整个团队 | CLAUDE.md |
| **生成方式** | AI 自动写入 | 开发者手动编写 | Memory |
| **内容类型** | 用户偏好、反馈、项目动态 | 构建命令、代码规范、架构规则 | 各有侧重 |
| **持久化范围** | 跨会话 | 跨会话 + 跨用户 | CLAUDE.md |
| **维护成本** | 几乎为零 | 需要定期更新 | Memory |
| **确定性** | 有机增长，可能过时 | 确定性输出，人工把关 | CLAUDE.md |

## 核心差异详解：谁控制上下文

这是两者最根本的区别。CLAUDE.md 是**你告诉 AI 的规则**，Memory 是 **AI 从你身上学到的模式**。

CLAUDE.md 里写的内容是确定性的——每个团队成员打开 Claude Code，看到的都是同一套指令。"提交前必须跑 `npm test`"、"中文内容用 CJK 字数统计"——这些规则不因人而异。它像一份团队共享的 `.editorconfig`，只不过对象是 AI 而不是编辑器。

Memory 则是个性化的。同一个项目里，前端工程师的 Memory 可能记录着"这个用户是 React 新手，解释组件时用后端类比"，而后端工程师的 Memory 里是"深度 Go 经验，偏好简洁回复不要总结"。Memory 随着对话自然积累，反映的是个人的工作习惯和沟通偏好。

实际工程中，这意味着：**项目级的约束写 CLAUDE.md，个人级的偏好让 Memory 自动处理**。如果你发现自己在 CLAUDE.md 里写"回复要简洁"——停下来，这应该是 Memory 的工作。如果你发现 Memory 里存了构建命令——那说明你的 CLAUDE.md 缺了关键信息。

## 核心差异详解：生命周期与可靠性

CLAUDE.md 走的是代码审查流程。有人改了构建命令，提 PR，团队审查，合并。它的变更历史清晰可追溯，和代码一样受版本控制保护。坏处是需要人工维护——如果构建命令改了但没人更新 CLAUDE.md，AI 就会用过时的指令。

Memory 的生命周期完全不同。它由 Claude Code 自动写入，没有审查流程，也没有版本历史。好处是零维护成本——你不需要"记得更新"任何东西。坏处是记忆可能过时或不准确：三个月前存的"项目用 Jest 测试"可能已经迁移到了 Vitest，但 Memory 不会自动感知这个变化。

Claude Code 对此有内置的缓解策略：在使用记忆前，它会验证记忆中提到的文件、函数或配置是否仍然存在。但这不是万无一失的——如果 Memory 里记录了一个高层决策（"这个项目优先性能而非可读性"），而团队方向已经转变，AI 没有足够的信号来发现这种偏移。

在我们的 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中，可以看到 CLAUDE.md 和 Memory 分别处于不同的架构层级——CLAUDE.md 是基础配置层，Memory 是运行时适应层。

## 什么时候该用 CLAUDE.md

把以下内容放进 CLAUDE.md：

- **构建和测试命令**：`npm run build`、`npm test`、`npm run lint`——团队每个人都需要的命令
- **提交前的质量门禁**：必须通过哪些检查才能提交代码
- **代码风格和架构约束**：命名规范、目录结构、禁止的做法
- **项目特有的坑**：容易踩的雷区，框架的已知限制
- **工作流规则**：分支策略、PR 要求、部署流程

一个好的判断标准：**如果新入职的工程师需要知道这件事，写进 CLAUDE.md**。如果只有你自己在意，让 Memory 处理。

想深入了解如何写好项目指令，可以参考 [如何有效提示 Claude Code](/zh/blog/how-to-effectively-prompt-a-claude-code)。

## 什么时候该依赖 Memory

以下场景让 Memory 自动工作：

- **个人角色和背景**："我是数据科学家"、"我是前端新手但后端经验丰富"
- **沟通偏好**："不要在回复末尾总结"、"解释时用具体例子"
- **行为纠正**：你说"别用 mock 测数据库"，Memory 会记住并在未来应用
- **项目动态**：截止日期、进行中的迁移、临时决策——那些不适合放进版本控制的短期上下文

Memory 最大的价值是**减少重复沟通**。如果你每次开新对话都要说同样的话，那就是 Memory 应该自动处理的场景。关于 Claude Code 的更多隐藏技巧，可以看 [我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

## 结论

**CLAUDE.md 和 Claude Memory 不是竞争关系，而是互补的两层上下文管理**。CLAUDE.md 是团队契约——确定性的、共享的、受版本控制保护的项目规则。Memory 是个人适应——自动的、渐进的、零维护成本的工作偏好积累。

最佳实践：**先写好 CLAUDE.md**，把团队需要统一的规则固定下来。然后让 Memory 在日常使用中自然积累你的个人偏好。两者配合，Claude Code 既能遵守项目规范，又能适应你的个人工作风格。

## 常见问题

### Claude Memory 会覆盖 CLAUDE.md 的规则吗？

不会。CLAUDE.md 的指令优先级高于 Memory。如果 CLAUDE.md 规定"提交前必须跑测试"，即使 Memory 里记录了"用户喜欢快速提交"，Claude Code 仍然会执行测试。Memory 补充的是 CLAUDE.md 未覆盖的个人偏好，而非覆盖已有规则。

### 我需要手动编辑 Memory 文件吗？

通常不需要。Claude Code 自动管理 Memory 的读写。但如果你发现某条记忆过时或错误，可以直接告诉 Claude Code "忘掉这个"，它会自动删除对应条目。你也可以直接编辑 `.claude/projects/` 下的 Markdown 文件。

### 团队里每个人的 Memory 都不一样，会不会导致 AI 行为不一致？

会有差异，但这恰恰是设计意图。Memory 存储的是个人偏好和沟通风格，本来就应该因人而异。团队一致性由 CLAUDE.md 保证——它定义的规则对所有人生效，不受个人 Memory 影响。
