**CLAUDE.md is loaded deterministically.** When Claude Code starts a session in your repo, it reads every CLAUDE.md file in the hierarchy — your global file at `~/.claude/CLAUDE.md`, the project file at the repo root, and any directory-scoped files relevant to the current working directory. This content goes directly into Claude's context window. There's no intelligence in the loading process — it's a straightforward file read. This means CLAUDE.md instructions are always present, always complete, and always the same for every team member.

**Claude Memory is loaded selectively.** Claude Code reads the `MEMORY.md` index at session start, which contains one-line pointers to individual memory files. The index acts as a table of contents — Claude uses the descriptions to decide which memories are relevant to the current task and reads those files. Memories that aren't relevant to the current conversation may not be loaded at all. This selective loading is necessary because memory accumulates over time and would eventually exceed useful context limits if loaded in full.

The practical implication: put mission-critical instructions in CLAUDE.md where they're guaranteed to be loaded every time. Use memory for context that's valuable but not essential — Claude Code will surface it when relevant and skip it when it's not.

CLAUDE.md also has a **staleness advantage**. Because it's manually maintained and version-controlled, you review changes through normal code review processes. Memory files can become stale — a project memory about a deadline that passed, a reference to a Linear project that was archived, or a feedback correction that no longer applies to the refactored code. Claude Code is instructed to verify memories against current state before acting on them, but this verification isn't guaranteed. For instructions that must always be followed, CLAUDE.md is the safer home.

## Scope and Ownership: Team vs Individual

The most important distinction between Claude Memory and CLAUDE.md is **who the information belongs to**.

CLAUDE.md belongs to the project. When you write "run `npm test` before every commit" in CLAUDE.md, that instruction applies to every developer on the team, every CI agent running Claude Code, and every future contributor who clones the repo. It's a form of **executable documentation** — not just describing how the project works, but actively enforcing it through the AI agent. Teams at companies like [Ramp, Shopify, and Spotify](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify) use CLAUDE.md to encode engineering standards that would otherwise live in wiki pages nobody reads.

Claude Memory belongs to the individual. When Claude Code learns that you're a data scientist who prefers pandas over polars, or that you've been burned by mocked database tests in the past, that's personal context that shouldn't be imposed on your teammates. A backend engineer and a frontend developer working on the same repo should get different Claude Code behavior based on their expertise and preferences — memory enables this personalization while CLAUDE.md keeps shared standards consistent.

This ownership distinction also affects how each system handles **conflicting information**. If your memory says "prefer single-file components" but CLAUDE.md says "split components into separate files for logic and presentation," CLAUDE.md wins — it's the authoritative project instruction. Memory informs Claude Code's approach within the boundaries CLAUDE.md sets, not outside them.

There's a middle ground worth noting: your global `~/.claude/CLAUDE.md` acts like persistent personal instructions, similar to memory but deterministic. Some developers use it for universal preferences ("always use TypeScript strict mode," "commit messages in conventional format") that they want applied across all projects. For a practical look at how developers structure these layers, see [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code).

## Content Types: What Belongs Where

Knowing the architectural difference is useful, but the daily question is simpler: **where should I put this specific piece of information?** Here's a decision framework with concrete examples.

### Put it in CLAUDE.md if:

- **It applies to everyone on the team.** Build commands, lint rules, test requirements, deployment steps, quality gates. "Run `npm run build` and `npm test` before any commit" is a CLAUDE.md instruction — every developer needs to follow it.
- **It's a project constraint or known gotcha.** "Don't import Next.js modules in pipeline scripts" or "The `upsertKeyword()` function requires three arguments — missing the third causes silent failures." These are codebase-specific traps that any developer (human or AI) needs to know about.
- **It defines coding style or architecture.** "Use Tailwind v4 utility classes, no custom CSS" or "Chinese content uses CJK word count, not English space-delimited counting." These are standards, not preferences.
- **It needs to survive team member changes.** If you leave the project, CLAUDE.md stays in the repo. Memory leaves with you.

### Put it in Claude Memory if:

- **It's about you, not the project.** Your role (senior backend engineer, data scientist, product manager), your expertise level with specific technologies, your communication preferences (terse responses, no trailing summaries).
- **It's feedback on Claude Code's behavior.** "Don't mock the database in integration tests — we got burned last quarter." "Stop summarizing what you just did." These are corrections and confirmations that shape how Claude works with you specifically.
- **It's about ongoing work context.** "The auth middleware rewrite is driven by legal compliance, not tech debt." "Merge freeze begins May 15 for mobile release." These are temporal facts about project state that would clutter CLAUDE.md and go stale quickly.
- **It's a pointer to external systems.** "Pipeline bugs are tracked in Linear project INGEST." "The oncall latency dashboard is at grafana.internal/d/api-latency." These references help Claude Code find information but aren't project instructions.

### Gray areas:

Some information could reasonably go in either place. A useful tiebreaker: **if removing it would break the project for a new team member, it belongs in CLAUDE.md.** If removing it would only degrade Claude Code's personalization for you, it belongs in memory.

For example, "we use vitest, not jest" belongs in CLAUDE.md — a new developer needs this. But "I prefer seeing test output in verbose mode" belongs in memory — that's a personal preference.

## Workflow Integration: How They Work Together

In practice, Claude Memory and CLAUDE.md aren't competing systems — they're complementary layers in [Claude Code's extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp). Here's how a typical workflow uses both.

**Session start:** Claude Code loads your project's CLAUDE.md hierarchy (global → project → directory) to understand the codebase rules. It loads your MEMORY.md index to understand who you are and what you've been working on. Both inform its behavior, but CLAUDE.md sets the boundaries and memory fills in the personal context.

**During work:** You ask Claude Code to refactor a module. CLAUDE.md tells it to run tests after changes and use the project's TypeScript strict configuration. Memory tells it you're a senior engineer who prefers minimal comments and that the last time it added excessive error handling, you corrected it. The result: a refactoring that follows project standards and matches your personal working style.

**Feedback loop:** You tell Claude Code "don't create separate utility files for one-off helpers — inline them." Claude Code saves this as a feedback memory. Next session, it remembers. Meanwhile, if the team decides this should be a project-wide rule, someone adds it to CLAUDE.md, and it becomes enforceable for everyone.

**Cross-session continuity:** You mention that the team is migrating from REST to GraphQL and the deadline is June 1st. Claude Code saves this as a project memory. In future sessions, when you're working on API endpoints, Claude Code has context about the migration without you re-explaining it. But the actual GraphQL schema conventions and migration steps go in CLAUDE.md once they're formalized.

This layered approach means CLAUDE.md stays clean and focused on durable project instructions, while memory handles the messy, evolving, personal context that makes Claude Code feel like a colleague who's been on your team for months rather than a fresh tool every session. To understand [what makes Claude Code special](/blog/whats-so-special-about-the-claude-code) compared to other AI coding tools, this dual-persistence architecture is a key differentiator.

## When to Choose Claude Memory

Use Claude Memory as your primary persistence mechanism when:

- **You're a solo developer** and the distinction between "team instructions" and "personal context" doesn't apply. Memory accumulates your preferences naturally without you maintaining a separate file.
- **You need cross-project context.** Memory about your role and working style follows you across repositories. CLAUDE.md is scoped to a single project.
- **The information is temporal.** Deadlines, ongoing incidents, current sprint goals — these change frequently and would create noisy CLAUDE.md diffs if tracked there.
- **You want zero-maintenance persistence.** Memory requires no manual editing. Claude Code creates, updates, and eventually removes stale memories based on conversation signals.

The tradeoff: memory is less reliable than CLAUDE.md for critical instructions. It's loaded selectively, can become stale, and isn't shared with teammates. Treat it as helpful context, not authoritative rules.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary persistence mechanism when:

- **You're on a team** and need consistent AI behavior across all developers. CLAUDE.md is the only way to ensure every Claude Code session follows the same rules.
- **Instructions are non-negotiable.** Quality gates, security constraints, architectural decisions — anything that must be followed every time, without exception.
- **You want version-controlled, reviewable changes.** CLAUDE.md changes go through pull requests. Memory changes are invisible to the team.
- **You're onboarding new team members.** A well-written CLAUDE.md gives new developers (and their Claude Code sessions) immediate context about project conventions. Memory starts empty for every new user.

The tradeoff: CLAUDE.md requires manual maintenance. It doesn't learn from your conversations or adapt to your personal style. It's a static document that you update deliberately, like any other piece of project documentation. ## Verdict

**Use both — they solve different problems.** CLAUDE.md is your project's source of truth: build commands, quality gates, coding standards, and architectural constraints that every developer and every Claude Code session must follow. Claude Memory is your personal context layer: your role, preferences, feedback corrections, and temporal project knowledge that makes Claude Code more effective for you specifically.

**The decision rule is straightforward: if the information should be enforced for the whole team and persist in the repo, put it in CLAUDE.md. If it's personal context that helps Claude Code work better with you, let memory handle it.** When in doubt, start with memory — if you find yourself re-correcting Claude Code on something that should be a project rule, promote it to CLAUDE.md. ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions take precedence over memory when they conflict. CLAUDE.md defines the project's authoritative rules — build commands, coding standards, architecture constraints. Memory provides personal context and preferences that operate within those boundaries. If your memory says "prefer single-file components" but CLAUDE.md says "separate logic and presentation files," CLAUDE.md wins.

### Does Claude Memory sync across devices?

No. Claude Memory is stored locally in `.claude/projects/` directories on your machine. It does not sync across devices, and it's typically gitignored. If you work on multiple machines, each will develop its own memory independently. CLAUDE.md, by contrast, syncs wherever your repo is cloned because it's checked into version control.

### How do I migrate information from memory to CLAUDE.md?

When you notice Claude Code repeatedly acting on a memory that should be a team-wide rule, manually add that instruction to your project's CLAUDE.md file. Then remove or update the memory file to avoid redundancy. Common candidates for promotion: coding style corrections that apply to the whole codebase, project-specific gotchas you've discovered, and workflow conventions the team has agreed on.

### Can I disable Claude Memory and only use CLAUDE.md?

Claude Code's memory system is built in and cannot be fully disabled through a single toggle. However, you can delete memory files from `.claude/projects/` and instruct Claude Code not to save memories during a session. For teams that want maximum control and consistency, relying primarily on CLAUDE.md with minimal memory usage is a valid approach — you trade personalization for determinism.

### How large should CLAUDE.md be?

Keep CLAUDE.md focused and scannable. The entire file loads into Claude Code's context window every session, so bloated files waste tokens and dilute important instructions. Most effective CLAUDE.md files are 50–200 lines covering build commands, quality gates, style rules, and critical gotchas. Move detailed guides, onboarding docs, and reference material to separate files that Claude Code can read on demand.
