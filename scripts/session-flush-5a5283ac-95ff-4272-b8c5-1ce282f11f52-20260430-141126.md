**Claude Memory is adaptive.** It captures context that emerges through conversation — things Claude could not know from reading the codebase alone. Your role as a data scientist investigating logging. Your preference for terse responses without trailing summaries. The fact that the team has a merge freeze starting Thursday. The location of the oncall dashboard in Grafana. These are personal, situational, and often ephemeral. Memory entries include a `type` field (user, feedback, project, or reference) and a one-line description used to judge relevance, so Claude does not load irrelevant memories into every session.

The architectural difference matters because it determines reliability. CLAUDE.md instructions are **always loaded and always followed** — they function as hard constraints. Memory entries are **selectively loaded and contextually applied** — they function as soft guidance. A CLAUDE.md rule saying "all tests must pass before commit" is a gate. A memory entry saying "this user prefers bundled PRs over split ones" is a preference. Conflating the two creates problems: project rules stored in Memory might not load when needed, and personal preferences stored in CLAUDE.md would affect the entire team.

For teams adopting [agentic coding](/glossary/agentic-coding) practices, getting this boundary right is one of the highest-leverage decisions you can make. Put the wrong thing in the wrong system, and you either get inconsistent AI behavior across the team or pollute shared configuration with individual quirks.

## Collaboration and Team Dynamics: Shared vs Personal

The most consequential difference between Claude Memory and CLAUDE.md is visibility. CLAUDE.md is a team artifact. Memory is a personal one. This distinction drives every practical decision about which system to use.

**CLAUDE.md enables team-wide consistency.** When your repository's CLAUDE.md specifies that commits require passing builds, tests, and validation — every Claude Code session for every developer on the team enforces those gates. Code review becomes simpler because the AI follows the same standards regardless of who invoked it. New team members get productive faster because Claude already knows the project conventions. Changes to standards go through pull requests, so the team can discuss and approve them. This is particularly valuable for organizations scaling [agentic coding](/glossary/agentic-coding) across dozens of engineers, as described in our coverage of [Claude Code in enterprise engineering teams](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify).

**Claude Memory enables personal adaptation.** Two developers on the same team can have completely different Memory profiles. A senior engineer's Memory might note their deep expertise in Go and preference for minimal explanations. A junior developer's Memory might note their need for more context when working in unfamiliar parts of the codebase. Both work in the same repo with the same CLAUDE.md, but Claude adapts its communication style, explanation depth, and suggestion complexity to each individual. This personalization happens without either developer needing to configure anything — Claude learns through natural conversation.

The interplay between these systems is where the real power lies. CLAUDE.md sets the floor: minimum standards, required workflows, architectural constraints. Memory raises the ceiling: Claude gets better at working with you specifically over time. A well-configured project has a thorough CLAUDE.md that any developer can rely on from day one, plus Memory that accumulates naturally as each developer works with Claude across sessions.

One practical consideration: Memory entries about project state (merge freezes, ongoing incidents, migration timelines) are inherently personal — only the developer who heard about the freeze gets that Memory entry. If this kind of context matters for the whole team, it belongs in CLAUDE.md or another shared document, not in one person's Memory.

## Content Strategy: What Goes Where

The boundary between Memory and CLAUDE.md is clear in theory but requires judgment in practice. Here is a decision framework based on the content types each system handles well.

### Put in CLAUDE.md

- **Build and test commands**: `npm run build`, `npm test`, `npm run lint` — anything the AI must run reliably
- **Quality gates**: "All tests must pass before commit," "run validate-pipeline.ts for pipeline changes"
- **Coding standards**: naming conventions, file organization, import rules, forbidden patterns
- **Architecture constraints**: "never import Next.js modules in pipeline scripts," "use CJK word count for Chinese content"
- **Workflow rules**: "new feature → discuss design first," "bug fix → systematic debug, not trial-and-error"
- **Documentation requirements**: which docs to update when modifying specific directories
- **Known gotchas**: project-specific pitfalls that every developer must know

These are all deterministic — they do not change based on who is coding, and they should be enforced consistently. Our [complete guide to Claude Code](/blog/claude-code-complete-guide) covers how to structure these files for maximum effectiveness.

### Put in Memory

- **Your role and expertise**: "senior backend engineer, new to React" — shapes how Claude explains things
- **Communication preferences**: "no trailing summaries," "prefer terse responses," "explain trade-offs before recommending"
- **Workflow feedback**: "don't mock the database in tests — we got burned by mock/prod divergence"
- **Project context not in code**: "merge freeze starts April 30 for mobile release," "auth rewrite is compliance-driven, not tech debt"
- **External resource pointers**: "pipeline bugs tracked in Linear project INGEST," "oncall dashboard at grafana.internal/d/api-latency"
- **Validated approaches**: "bundled PRs preferred for refactors in this area" — confirmed through successful collaboration

These are personal, contextual, or ephemeral. They enrich Claude's understanding of your specific situation without imposing on the rest of the team.

### Gray Areas

Some content could reasonably go in either system. A rule like "prefer integration tests over mocks" is a team standard (CLAUDE.md) if the whole team agrees, but a personal preference (Memory) if it is just your strong opinion. When in doubt, ask: "Would a new team member benefit from seeing this on day one?" If yes, CLAUDE.md. If no, Memory.

For teams building sophisticated [Claude Code skill files](/blog/5-claude-code-skills-i-use-every-single-day), the same principle applies: skills encode reusable, team-shared behavior — they are the CLAUDE.md of specific tasks, not the Memory.

## Lifecycle and Maintenance

CLAUDE.md and Memory have different maintenance characteristics, and understanding these prevents common problems.

**CLAUDE.md maintenance is explicit.** You write it, you update it, you review changes in PRs. The downside is that it can become stale — a build command changes but nobody updates CLAUDE.md, and suddenly Claude is running the wrong command. The upside is that staleness is visible and fixable. Good teams treat CLAUDE.md like any other code artifact: update it when the thing it describes changes. Some projects enforce this with documentation rules — "when modifying pipeline scripts, update PIPELINE.md" — which themselves live in CLAUDE.md.

**Memory maintenance is mostly automatic.** Claude creates, updates, and occasionally removes memories based on conversation. The downside is less visibility — you might not know what Claude remembers about you until it surfaces in a surprising recommendation. The upside is low maintenance burden: you do not need to manually update a file every time your preferences evolve. Claude is designed to verify memories against current state before acting on them, since a memory naming a specific file or function might reference something that has since been renamed or removed.

You can inspect and edit Memory files directly — they are plain markdown in `.claude/projects/*/memory/`. This is worth doing periodically, especially for long-running projects where project-state memories may have become outdated. A memory saying "merge freeze starts March 5" is actively harmful in April if Claude treats it as current. For strategies on managing Claude's context systems effectively, see [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code).

## When to Choose Claude Memory

Use Memory as your primary context mechanism when:

- **You work solo.** Without a team to share CLAUDE.md with, Memory captures everything Claude needs to know about your project and preferences. You still benefit from a minimal CLAUDE.md for build commands and key constraints, but Memory carries the bulk of the context.
- **Your working style varies by project.** Memory is scoped per project directory, so Claude can adapt its behavior for different codebases without you managing multiple configuration files.
- **You want Claude to improve over time.** Memory accumulates naturally through conversation. The more you work with Claude, the better it understands your preferences, your domain expertise, and your project's current state.
- **Context is personal or ephemeral.** Your role, communication preferences, current sprint priorities, and knowledge of external systems — none of this belongs in a shared configuration file.
- **You are exploring or prototyping.** When project conventions have not solidified yet, Memory is more forgiving than CLAUDE.md. Claude learns from what you do rather than requiring you to specify rules upfront.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary context mechanism when:

- **You work on a team.** Any rule that should apply to all developers — build commands, quality gates, coding standards — must go in CLAUDE.md. Memory is invisible to your teammates.
- **Consistency matters more than adaptation.** For production codebases with established conventions, you want Claude to follow rules, not learn preferences. CLAUDE.md provides deterministic, auditable behavior.
- **You need version-controlled AI behavior.** When AI coding standards change, you want that change tracked in git, reviewed in a PR, and applied to everyone simultaneously. Memory changes are invisible and individual.
- **You are onboarding new team members.** A thorough CLAUDE.md means a new developer's first Claude Code session already knows the project's conventions, architecture constraints, and workflow requirements. No ramp-up period.
- **You need to enforce hard constraints.** "Never skip failing tests," "always update docs when modifying pipeline scripts" — these are not preferences, they are rules. CLAUDE.md is the enforcement mechanism.
- **Your project has known gotchas.** Subtle bugs, counterintuitive requirements, or platform-specific constraints that every developer needs to know about belong in CLAUDE.md where they are guaranteed to load every session.

## Verdict

**Use both.** CLAUDE.md and Claude Memory are not competing alternatives — they are complementary layers in Claude Code's [programmable context system](/blog/claude-code-seven-programmable-layers). **CLAUDE.md is your project's constitution**: the shared, version-controlled rules that every session and every developer follows. **Memory is your personal relationship with Claude**: the accumulated understanding that makes each conversation more productive than the last.

Start with CLAUDE.md. Define your build commands, quality gates, and coding standards. This is the highest-leverage investment — it pays off from the first session. Memory will accumulate naturally as you work. Periodically review your Memory files to prune stale entries, but do not over-manage them. The system is designed to be low-maintenance.

The most common mistake is putting team rules in Memory (where teammates cannot see them) or personal preferences in CLAUDE.md (where they affect the whole team). Keep the boundary clear: **shared and deterministic → CLAUDE.md. Personal and adaptive → Memory.** Get this right, and Claude Code becomes dramatically more effective for both individual productivity and team consistency.

## Frequently Asked Questions

### Can Claude Memory and CLAUDE.md conflict with each other?

CLAUDE.md instructions take priority as they are treated as high-priority system directives. If a Memory entry contradicts a CLAUDE.md rule, the CLAUDE.md rule wins. Claude is also designed to verify Memory entries against current state before acting — if a remembered fact conflicts with what the codebase shows now, Claude trusts current evidence over stored memories.

### Should I check Memory files into version control?

No. Memory files are personal to each developer and stored in `.claude/projects/*/memory/`, which is outside the project repository. They contain individual preferences, role information, and ephemeral project context — none of which should be shared or version-controlled. Team-wide context belongs in CLAUDE.md, which is designed for version control.

### How do I migrate a Memory entry into CLAUDE.md?

If you notice Claude consistently applying a personal memory that the whole team should follow — like a testing approach or architectural constraint — add it manually to your project's CLAUDE.md and submit it as a pull request. Then remove or let the Memory entry become redundant. There is no automated migration tool; the manual step is intentional because it forces team review of new shared rules.

### Does CLAUDE.md work without Claude Memory enabled?

Yes. CLAUDE.md is completely independent of the Memory system. It loads at session start regardless of whether Memory is configured or populated. Many teams use CLAUDE.md effectively without relying on Memory at all, especially in CI/CD environments where Claude Code runs in ephemeral contexts with no persistent Memory state.

### How often should I review my Claude Memory entries?

Review Memory files every few weeks for active projects, or whenever you notice Claude making outdated assumptions. Project-state memories (merge freezes, sprint priorities, incident timelines) go stale fastest. User and feedback memories (your role, communication preferences) tend to stay relevant longer. Delete or update anything that no longer reflects reality.
