# Codex Subagents vs Custom Agents：如何选择正确的多智能体策略

**TL;DR：** **Subagents** 是 Codex 在执行任务时自动拆分出的子任务执行器，适合单次复杂任务的并行处理；**Custom Agents** 是你预先定义好的专用智能体，带有特定指令、工具和约束，适合团队反复使用的标准化工作流。选择标准很简单——**任务是临时的，用 Subagents；流程是固定的，用 Custom Agents**。两者可以组合使用，Custom Agent 内部可以调度 Subagents 来并行执行子任务。

## 概述：Codex Subagents

Subagents 是 [Codex](/zh/blog/codex-complete-guide) 多智能体架构的核心机制。当一个任务足够复杂——比如"重构这个模块并更新所有相关测试"——Codex 会自动将其拆解为多个子任务，每个子任务由一个独立的 Subagent 执行。这些 Subagent 在隔离的沙箱环境中并行运行，各自拥有独立的上下文窗口和工具调用能力。

Subagents 的关键特性在于**自动化**。开发者不需要手动定义子任务的边界或执行逻辑，Codex 的编排层会根据任务语义自动完成拆分、分发和结果汇总。这种模式在处理跨文件修改、批量代码审查、大规模测试生成等场景中表现突出。根据 OpenAI 的工程实践，Subagents 的并行执行可以将复杂任务的完成时间缩短 40-60%。

但自动化也意味着**控制粒度有限**。你无法精确指定每个 Subagent 应该使用哪些工具、遵循哪些约束，或按什么顺序执行。对于需要严格流程控制的场景，这是一个明显的局限。

## 概述：Codex Custom Agents

Custom Agents 是 Codex 提供的[可配置智能体框架](/zh/blog/con-u-pour-des-workflows-multi-agents)。你可以为特定任务类型创建专用的 Agent，预先定义它的系统指令、可用工具集、文件访问范围和行为约束。一个 Custom Agent 本质上是一个固化的工作流模板，团队成员可以反复调用。

典型的 Custom Agent 配置包括：指定代码审查标准的 Review Agent、遵循特定测试框架约定的 Test Agent、或者只允许读取特定目录的 Security Audit Agent。这些配置通过 Codex 的 Agent 定义文件（类似 Claude Code 中的 [SKILL.md](/zh/glossary/agent-sdk) 概念）持久化存储，与代码仓库一起版本管理。

Custom Agents 的优势在于**可预测性和可复用性**。每次调用都遵循相同的规则，输出质量一致，且团队中的每个人都能使用同一套标准化智能体。代价是前期配置成本——你需要花时间设计指令、测试行为、迭代优化。

## 功能对比

| 维度 | Subagents | Custom Agents | 胜出 |
|------|-----------|---------------|------|
| **启动方式** | Codex 自动拆分 | 开发者预先定义 | 看场景 |
| **配置成本** | 零配置 | 需要编写 Agent 定义文件 | Subagents |
| **执行控制** | 有限——由编排层决定 | 完全可控——指令、工具、范围均可自定义 | Custom Agents |
| **复用性** | 一次性——随任务生灭 | 持久化——可跨任务、跨团队复用 | Custom Agents |
| **并行能力** | 原生支持并行执行 | 单 Agent 串行，但内部可调度 Subagents | Subagents |
| **上下文隔离** | 每个 Subagent 独立沙箱 | 独立沙箱 + 自定义文件访问范围 | Custom Agents |
| **适合团队规模** | 个人开发者 | 团队/企业 | Custom Agents |
| **调试难度** | 较高——自动拆分逻辑不透明 | 较低——行为由显式指令决定 | Custom Agents |

## 任务编排：核心差异深度分析

Subagents 和 Custom Agents 最根本的区别在于**谁决定任务如何拆分和执行**。

使用 Subagents 时，你只需要描述最终目标。比如输入"将所有 REST 端点迁移到 GraphQL"，Codex 的编排层会自动分析代码库，识别所有需要修改的文件，将工作拆分为多个并行子任务，最后合并结果。这个过程对开发者是黑箱的——你看到的是输入一个需求、输出一组修改。

这种模式的优势是极低的认知负担。你不需要理解任务拆分的最优策略，不需要手动协调多个 Agent 之间的依赖关系。但当自动编排的结果不符合预期时，你能做的调整非常有限——基本只能修改输入的任务描述，期望 Codex 重新理解你的意图。

Custom Agents 将控制权交给开发者。你显式定义每个 Agent 的职责边界、可用工具和行为规范。比如，你可以创建一个只负责数据库迁移的 Agent，限制它只能访问 `migrations/` 目录，只能使用 SQL 相关工具，并且必须在执行任何 DDL 语句前生成回滚脚本。这种粒度的控制在 Subagents 模式下是不可能实现的。

对于[编码 Agent 的工程实践](/zh/blog/effective-harnesses-for-long-running-agents)而言，Custom Agents 更接近生产级需求——可审计、可复现、可持续优化。

## 实际工作流：场景化对比

**场景一：一次性大规模重构**

你需要将一个 monorepo 中的 200 个文件从 CommonJS 迁移到 ESM。这是一个典型的 Subagents 场景——任务目标明确、子任务之间相互独立、不需要复杂的执行逻辑。Codex 会自动将 200 个文件分配给多个 Subagents 并行处理，每个 Subagent 负责一批文件的转换和验证。

如果用 Custom Agent 处理同样的任务，你需要先定义一个 ESM 迁移 Agent，编写迁移规则，测试边界情况——前期投入显著更高。对于一次性任务，这个投入不划算。

**场景二：每日代码审查流水线**

团队每天有 15-20 个 PR 需要审查，审查标准包括安全性检查、性能模式识别、测试覆盖率验证。这是 Custom Agent 的典型场景。你创建一个 Review Agent，将团队的审查标准编码为系统指令，配置好需要的工具（静态分析器、测试覆盖率检查器），然后让每个 PR 自动触发这个 Agent。

审查标准随时间演进——发现新的安全模式就更新指令，调整性能阈值就修改配置。Custom Agent 的定义文件作为代码仓库的一部分被版本管理，变更历史完全可追溯。这是 [Claude Code Agent Teams](/zh/blog/claude-code-agent-teams) 也在解决的同类问题——如何将 AI 审查能力标准化。

**场景三：混合模式——Custom Agent 内调度 Subagents**

最强大的用法是组合两者。定义一个 Custom Agent 负责"发版前完整检查"，它的系统指令中包含检查清单（单元测试、集成测试、安全扫描、变更日志生成），但每个检查项由 Subagents 并行执行。Custom Agent 提供流程框架和质量标准，Subagents 提供执行效率。

## 什么时候选 Subagents

以下场景优先考虑 Subagents：

- **探索性任务**：你还不确定最终方案，需要 Codex 先分析代码库、提出建议——让 Subagents 并行调研不同模块比手动逐个查看高效得多
- **一次性批量操作**：大规模重命名、格式化、依赖升级等机械性任务，不值得为其创建持久化的 Agent 定义
- **个人项目**：只有你一个人使用，不需要跨团队复用，快速完成比流程规范更重要
- **原型验证**：在决定是否要创建 Custom Agent 之前，先用 Subagents 跑一次看看效果

Subagents 的核心优势是**零配置、即用即走**。如果你发现自己反复用 Subagents 做同样的事情，那就是创建 Custom Agent 的信号。

## 什么时候选 Custom Agents

以下场景优先考虑 Custom Agents：

- **团队标准化工作流**：代码审查、测试生成、文档更新等需要一致性的流程，参考 [Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)中类似的 Skills 复用模式
- **安全敏感操作**：需要严格限制 Agent 的文件访问范围和可用工具，避免意外修改不该碰的代码
- **可审计需求**：合规要求每次 AI 操作都有可追溯的配置定义和执行日志
- **迭代优化**：你会持续优化这个 Agent 的表现——调整指令、添加示例、修改约束——这些优化需要被持久化

Custom Agents 的核心优势是**可控、可复用、可演进**。前期配置成本通过长期复用摊薄。

## 结论

**Subagents 和 Custom Agents 不是替代关系，而是互补关系。** Subagents 解决的是"如何高效执行复杂任务"，Custom Agents 解决的是"如何让 AI 工作流标准化和可控"。对于[多智能体工作流](/zh/blog/con-u-pour-des-workflows-multi-agents)的工程实践，成熟团队的最终形态通常是：用 Custom Agents 定义关键工作流的骨架，在需要并行处理的环节调度 Subagents 提升效率。

如果你刚开始使用 Codex，从 Subagents 入手——零配置、即时反馈。当你发现某些任务模式反复出现，再将它们固化为 Custom Agents。这种渐进式的采用路径，比一上来就设计完整的 Agent 体系更务实。

## 常见问题

### Subagents 和 Custom Agents 可以同时使用吗？

可以。Custom Agent 内部可以调度 Subagents 来并行处理子任务。这是推荐的生产级用法——Custom Agent 提供流程控制和质量标准，Subagents 负责高效执行。两者的组合在大规模代码库操作中表现最优。

### 创建一个 Custom Agent 需要多长时间？

取决于复杂度。一个简单的代码审查 Agent（基础指令 + 工具配置）大约 30 分钟可以搭建并测试完成。复杂的多步骤工作流 Agent（如发版检查流水线）可能需要 2-4 小时的迭代调优。关键成本不在初始创建，而在后续根据实际表现持续优化。

### Codex 的 Subagents 和 Claude Code 的 Agent Teams 有什么区别？

两者解决类似问题但架构不同。Codex Subagents 在云端沙箱中运行，由 OpenAI 的编排层管理；[Claude Code Agent Teams](/zh/blog/claude-code-agent-teams) 在本地终端中运行，通过 Git Worktree 实现隔离。Codex 更适合异步的后台任务，Claude Code 更适合需要实时交互的开发场景。
