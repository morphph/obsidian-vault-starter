# Claude Memory vs CLAUDE.md：哪种记忆机制更适合你？

**TL;DR:** **Claude Memory** 是 Anthropic 的跨会话自动记忆系统，记住你的个人偏好和上下文；**CLAUDE.md** 是项目级的规则手册文件，定义编码规范和项目约束。两者不是竞争关系——Memory 解决「AI 记住你是谁」的问题，CLAUDE.md 解决「AI 理解这个项目怎么做」的问题。大多数专业开发者应该同时使用两者。

## 概述：Claude Memory

Claude Memory 是 Anthropic 推出的持久化记忆功能，让 Claude 在不同对话之间记住用户的关键信息。无论你是在 Claude.ai 网页端还是 Claude Code 终端中工作，Memory 都会自动捕捉你提到的偏好、角色、技术栈和工作习惯，并在后续对话中主动应用这些上下文。

Memory 的核心价值在于消除重复沟通。你不需要每次开新对话都说「我是后端工程师，用 Go 写微服务，偏好函数式风格」——说一次，Claude 就记住了。2026 年初的[记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)还支持从 ChatGPT 导入数据，进一步降低了迁移成本。

在 Claude Code 中，Memory 以 `~/.claude/` 目录下的文件形式存储，包括用户偏好、项目反馈和工作习惯等分类。它是个人级别的，跟随用户账户而非项目仓库。

## 概述：CLAUDE.md

**CLAUDE.md** 是一个 Markdown 文件，放在项目根目录（或 `~/.claude/` 全局目录），为 Claude Code 提供项目级的指令和约束。它本质上是一份「规则手册」——告诉 AI 这个项目用什么技术栈、遵循什么编码规范、有哪些已知陷阱需要避免。

CLAUDE.md 的关键特性是它跟随代码仓库。当你 `git clone` 一个项目，CLAUDE.md 也一起到位，团队中每个人的 Claude Code 都会读取相同的规则。这解决了团队协作中最头疼的问题：AI 在不同人手里表现不一致。关于 CLAUDE.md 在整个扩展体系中的定位，可以参考[Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

一个典型的 CLAUDE.md 包含构建命令、质量门禁、代码风格要求和已知 Gotcha。它不存储个人偏好，只存储项目事实。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 胜出 |
|------|--------------|-----------|------|
| **作用范围** | 用户级，跨所有项目 | 项目级，跟随仓库 | 各有所长 |
| **存储位置** | `~/.claude/` 目录 | 项目根目录 | CLAUDE.md |
| **团队共享** | 不共享，个人专属 | Git 提交，团队共享 | CLAUDE.md |
| **内容类型** | 偏好、角色、习惯 | 规则、命令、约束 | 各有所长 |
| **维护方式** | AI 自动写入 + 手动编辑 | 开发者手动维护 | Memory |
| **生效平台** | Claude.ai + Claude Code | 仅 Claude Code | Memory |
| **版本控制** | 无 | 随代码版本管理 | CLAUDE.md |
| **可审计性** | 较弱 | 强——可 code review | CLAUDE.md |

## 记忆机制详解：自动 vs 手动

Claude Memory 的最大优势是零配置。你正常对话，它在后台学习。当你说「我更喜欢用 Vitest 而不是 Jest」，下次它就自动用 Vitest 写测试。这种隐式学习降低了使用门槛，但也带来了一个问题：你不总是知道 AI 记住了什么、会怎么用。

CLAUDE.md 走的是完全相反的路线——显式声明。每条规则都是你亲手写的，清晰、可审计、可 diff。当团队新人问「为什么 AI 总是用 pnpm 而不是 npm」，答案就在 CLAUDE.md 里，白纸黑字。

两者在[Claude Code 记忆系统](/zh/blog/claude-code-memory)中协同工作：CLAUDE.md 定义项目级规则，Auto Memory 补充个人级偏好。规则冲突时，CLAUDE.md 优先——这是设计上的正确决策，因为项目规范应该覆盖个人习惯。

实际使用中，Memory 适合存储「我喜欢简洁的 commit message」「我是全栈工程师」这类个人上下文；CLAUDE.md 适合存储「这个项目必须先跑 `npm test` 再提交」「ZH 内容用 CJK 字数统计」这类项目硬性约束。

## 团队协作与可控性

对于个人开发者，Memory 的便利性几乎无可挑剔。但一旦进入团队场景，CLAUDE.md 的优势就变得压倒性。

**版本控制**是核心差异。CLAUDE.md 是代码的一部分，每次修改都经过 PR review。团队可以像审查代码一样审查 AI 规则——「为什么加了这条 lint 豁免？」「这个已知 bug 修复了吗？」Memory 没有这个能力，它存在你本地，别人看不到也改不了。

**一致性**是第二个关键点。团队中五个人的 Memory 各不相同，但 CLAUDE.md 对所有人生效。这意味着 AI 在处理同一个项目时，不会因为操作者不同而产出风格迥异的代码。想要进一步统一团队的 AI 工作流，可以结合[Claude Code Hooks](/zh/blog/claude-code-hooks-mastery) 实现自动化质量门禁。

**可审计性**是企业环境的刚需。CLAUDE.md 的变更历史清晰可查——谁在什么时候加了什么规则，一目了然。Memory 目前缺乏这种审计能力。

## 适用场景：什么时候用 Memory

选择依赖 Claude Memory 的场景：

- **个人项目或独立开发**：不需要团队共享，个人偏好就是项目规范
- **跨项目通用偏好**：编码风格、沟通语言、commit 格式等不随项目变化的习惯
- **快速原型阶段**：项目还没成型，不值得写正式的 CLAUDE.md
- **非代码场景**：在 Claude.ai 网页端写文档、做分析、头脑风暴——这些场景 CLAUDE.md 不生效

Memory 的典型用户画像：独立开发者、自由职业者、或者在探索阶段的工程师。你需要 AI 快速适应你的风格，但不需要团队级别的规则治理。

## 适用场景：什么时候用 CLAUDE.md

选择 CLAUDE.md 作为主要机制的场景：

- **团队项目**：多人协作，需要统一的 AI 行为规范
- **正式工程环境**：有 CI/CD、code review、质量门禁的项目
- **复杂项目**：有大量 Gotcha、特殊构建流程或非常规架构的代码库
- **合规需求**：需要审计 AI 行为规则的企业环境

CLAUDE.md 的典型用户画像：工程团队、开源项目维护者、企业开发环境。[如何有效地提示 Claude Code](/zh/blog/how-to-effectively-prompt-a-claude-code) 这篇文章详细介绍了 CLAUDE.md 的编写技巧和最佳实践。

## 最佳实践：两者配合使用

最成熟的做法不是二选一，而是分层配合：

1. **CLAUDE.md 定义项目事实**：技术栈、构建命令、代码规范、已知陷阱
2. **Memory 记录个人偏好**：沟通风格、角色背景、通用习惯
3. **Skills 文件处理特定任务**：[Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中的 SKILL.md 负责定义 newsletter 写作、代码审查等具体任务的规则

这三层形成了完整的 AI 上下文栈：项目规则 > 个人偏好 > 任务指令。冲突时自上而下覆盖。

## 结论

**如果你只能选一个，选 CLAUDE.md**——它可控、可审计、可团队共享，是工程实践的基础。但在实际工作中，两者配合使用才是最优解。CLAUDE.md 管项目规则，Memory 管个人上下文，各司其职。随着 Anthropic 持续迭代记忆系统，两者的边界会越来越清晰，协同也会越来越顺畅。

## 常见问题

### Claude Memory 和 CLAUDE.md 会冲突吗？

不会产生技术冲突。当两者对同一行为有不同指示时，CLAUDE.md 的项目级规则优先于 Memory 的个人偏好。例如你的 Memory 记录了「喜欢用 tabs 缩进」，但 CLAUDE.md 规定「必须用 spaces」，Claude Code 会遵循 CLAUDE.md。

### 没有 CLAUDE.md 的项目能用 Claude Memory 吗？

完全可以。Memory 独立于 CLAUDE.md 运行，在任何 Claude 对话中都生效。对于没有 CLAUDE.md 的项目，Memory 是唯一的持久化上下文来源，这时它的价值更加突出。

### CLAUDE.md 需要多久更新一次？

每当项目的技术栈、构建流程、编码规范或已知问题发生变化时就应该更新。好的实践是把 CLAUDE.md 的维护纳入 code review 流程——修改了构建脚本？顺手更新 CLAUDE.md。

### 企业环境应该禁用 Claude Memory 吗？

取决于安全策略。Memory 可能存储敏感上下文信息，企业应评估数据存储位置和生命周期。对于高安全要求的环境，可以只依赖 CLAUDE.md 和 Skills 文件，确保所有 AI 指令都在版本控制之下。
