# Claude Memory vs CLAUDE.md：两套持久化机制该怎么选？

**TL;DR：** **CLAUDE.md** 是你写给 AI 的规则手册——团队共享、版本控制、每次会话必定加载。**Claude Memory（Auto Memory）** 是 AI 自己记的笔记——个人专属、自动积累、按需召回。两者不是竞品关系，而是互补的持久化层。规则和约束放 CLAUDE.md，个人偏好和项目上下文交给 Memory。

## 概述：Claude Memory

**Claude Memory** 是 Claude Code 的自动记忆系统，它在对话过程中主动识别值得记住的信息，并将其写入 `~/.claude/projects/` 下的 markdown 文件。这些记忆分为四类：用户画像（user）、行为反馈（feedback）、项目上下文（project）和外部引用（reference）。

Memory 的核心设计理念是「让 AI 跨会话保持连贯性」。你在第一次对话中提到自己是数据科学家、偏好 Python、不喜欢冗长的解释，Memory 会把这些记下来。下次开新会话时，Claude 会自动加载相关记忆，调整回答风格和技术深度，不需要你重复说明。

Memory 的存储位置在用户目录下（`~/.claude/`），不进入代码仓库，属于个人私有数据。它的触发可以是自动的（AI 判断值得记住），也可以是显式的（用户说"记住这个"）。详细机制可参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

**CLAUDE.md** 是一个 markdown 文件，放在项目根目录（或 `~/.claude/` 全局目录），用于向 Claude Code 传达项目级别的指令和约束。它在每次会话启动时被完整加载到上下文中——不是按需召回，而是必定加载。

CLAUDE.md 的定位是「项目的规则手册」。典型内容包括：构建命令（`npm run build`）、代码风格约定、提交前必须通过的质量门禁、技术栈说明、已知陷阱等。因为它是代码仓库的一部分，所以通过 Git 进行版本控制，团队所有成员共享同一份规则。

这意味着 CLAUDE.md 的内容是确定性的——每个人、每次会话看到的都一样。它不会「忘记」，也不会因为 AI 的判断而选择性加载。想深入了解 CLAUDE.md 的层级结构和最佳实践，可以阅读 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 适用方 |
|------|--------------|-----------|--------|
| **存储位置** | `~/.claude/projects/*/memory/` | 项目根目录或 `~/.claude/` | 各有用途 |
| **版本控制** | 不进 Git | 提交到 Git | CLAUDE.md |
| **团队共享** | 个人私有 | 团队共享 | CLAUDE.md |
| **加载方式** | 按相关性召回 | 每次会话完整加载 | CLAUDE.md |
| **写入方式** | AI 自动 + 用户指令 | 人工手写 | Memory |
| **内容类型** | 偏好、上下文、反馈 | 规则、命令、约束 | 各有用途 |
| **可靠性** | 可能漏召回 | 确定性加载 | CLAUDE.md |
| **维护成本** | 几乎为零 | 需要人工维护 | Memory |

## 持久化机制：核心差异

CLAUDE.md 和 Memory 最根本的区别在于**确定性**。

CLAUDE.md 的内容在每次对话开始时被完整注入上下文窗口，这是框架层面的保证。无论你问什么问题、做什么任务，CLAUDE.md 里的规则都在那里。这使得它适合存放「必须遵守」的内容：构建命令、质量门禁、编码规范、已知陷阱。

Memory 则是按相关性召回的。系统会根据当前对话内容判断哪些记忆相关，然后加载它们。这意味着 Memory 可能漏召回某条信息，也可能加载了不再准确的旧记忆。Claude Code 的文档明确指出：「Memory records can become stale over time」，建议在执行操作前验证记忆的准确性。

这不是 Memory 的缺陷，而是设计取舍。如果把所有记忆都塞进上下文，会迅速耗尽有限的上下文窗口。按需召回让 Memory 可以积累大量信息而不占用固定空间。

## 协作与团队场景

团队场景下，两者的角色分工最为清晰。

**CLAUDE.md 是团队契约。** 当你在 CLAUDE.md 里写下「提交前必须通过 `npm test` 和 `npm run build`」，团队每个人的 Claude Code 都会遵守这个规则。新成员加入项目，不需要任何额外配置，CLAUDE.md 跟着代码仓库走。这也是为什么很多团队把 [CLAUDE.md 当作 AI 时代的编码规范](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)来维护。

**Memory 是个人助手。** 你偏好简洁回答、习惯用 vim 键位、上周修了一个复杂的 auth bug——这些信息对你有价值，但不需要（也不应该）共享给团队。Memory 存在用户目录下，天然隔离。

一个实际例子：团队的 CLAUDE.md 规定「commit message 必须用英文，遵循 Conventional Commits 格式」。而某个开发者的 Memory 里记着「这个用户喜欢在 commit message 后面加 emoji」。两者不冲突——Memory 在规则框架内个性化。

## 什么时候用 CLAUDE.md

选择 CLAUDE.md 的判断标准：**这条信息是否每次会话都需要、是否团队通用。**

具体场景：

- **构建和测试命令**：`npm run dev`、`npm run build`、`npm test`
- **质量门禁**：提交前必须通过的检查清单
- **编码规范**：命名约定、文件组织方式、禁止使用的模式
- **技术栈说明**：使用的框架版本、数据库类型、部署平台
- **已知陷阱**：容易踩的坑，比如「ZH 内容必须用 CJK word count」
- **工作流约定**：PR 流程、分支策略、部署步骤

CLAUDE.md 还支持多层级结构。根目录的 `CLAUDE.md` 设定全局规则，子目录的 `CLAUDE.md` 可以覆盖或补充特定模块的规则。用户级别的 `~/.claude/CLAUDE.md` 则定义跨项目的个人偏好。了解这种层级设计，可以参考 [如何有效地给 Claude Code 写 Prompt](/zh/blog/how-to-effectively-prompt-a-claude-code)。

## 什么时候用 Memory

选择 Memory 的判断标准：**这条信息是个人化的、会随时间积累的、不需要确定性保证的。**

具体场景：

- **用户画像**：你的角色、技术背景、沟通偏好
- **行为反馈**：「不要在回答末尾加总结」「测试用真实数据库，别用 mock」
- **项目动态**：「本周四之后冻结非关键合并」「auth 重写是因为合规要求」
- **外部资源**：「bug 跟踪在 Linear 的 INGEST 项目里」「oncall 看这个 Grafana 面板」

Memory 的最大优势是**零维护成本**。你不需要主动去写或更新它——AI 在对话中自动识别值得记住的信息。当然，你也可以显式要求「记住这个」或「忘掉那个」。Anthropic 最近还为 Claude 的记忆系统增加了[从 ChatGPT 导入的能力](/zh/blog/anthropic-claude-memory-upgrades-importing)，进一步降低了迁移成本。

## 结论

**CLAUDE.md 和 Memory 不是二选一，而是分层互补。** 把确定性规则、团队约定、质量门禁放进 CLAUDE.md——它是每次会话的硬性约束。把个人偏好、项目动态、行为反馈交给 Memory——它是跨会话的软性上下文。

判断一条信息该放哪里，问自己两个问题：「新同事第一天需要知道这个吗？」如果是，放 CLAUDE.md。「这只对我个人有用吗？」如果是，交给 Memory。

## 常见问题

### Claude Memory 会不会泄露个人信息给团队成员？

不会。Memory 存储在用户本地目录 `~/.claude/` 下，不进入 Git 仓库，团队其他成员无法访问。只有 CLAUDE.md 是通过代码仓库共享的。

### CLAUDE.md 的内容会占用上下文窗口吗？

会。CLAUDE.md 在每次会话启动时被完整加载到上下文中，因此内容应该精炼。建议控制在 500 行以内，避免放入大段示例代码或冗长说明。

### 如果 Memory 记住的内容和 CLAUDE.md 冲突了怎么办？

CLAUDE.md 的优先级更高，因为它是确定性加载的硬性规则。Memory 是软性上下文，当两者冲突时，Claude Code 会遵循 CLAUDE.md 的指令。

### 可以把 Memory 的内容迁移到 CLAUDE.md 里吗？

可以，但要判断是否合适。如果某条 Memory 反复被召回且对团队有价值（比如一个反复出现的技术决策），可以提炼后写入 CLAUDE.md。但个人偏好类的内容不适合放进团队共享的规则文件。
