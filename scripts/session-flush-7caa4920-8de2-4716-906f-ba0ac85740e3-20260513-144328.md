# Codex Subagents vs Claude Code Subagents：多 Agent 编码工作流怎么选？

**TL;DR：** OpenAI Codex 采用云端沙箱 + 任务分发模式，适合需要严格隔离和异步批量处理的团队；**Claude Code 的 subagent 系统在本地终端实时协作，适合需要深度上下文感知和即时反馈的开发者**。如果你的工作流偏向"甩任务给 AI 后台跑"，Codex 更合适；如果你习惯与 AI 实时对话式编程，Claude Code 的 agent teams 体验更好。

## 概述：Codex 的 Subagent 与自定义 Agent

Codex 是 OpenAI 推出的云端编程智能体，每个任务在独立的沙箱环境中执行。Codex 的多 agent 模式本质上是任务级别的并行——你可以同时发起多个 Codex 任务，每个任务独立运行在隔离的云端容器中，互不干扰。

Codex 的"自定义 agent"能力体现在两个层面：一是通过 `AGENTS.md`（类似 Claude Code 的 `CLAUDE.md`）定义项目级别的指令和约束；二是通过 VS Code 扩展或 CLI 接口发起带有特定上下文的任务。每个任务本质上是一个独立的 agent 实例，拥有完整的代码仓库快照和沙箱环境。这种架构的优势在于天然隔离——一个 agent 的操作不会影响另一个，适合批量处理场景，比如同时修复多个 issue 或在不同分支上并行开发。更多 Codex 基础能力可参考[《OpenAI Codex 完全指南》](/zh/blog/codex-complete-guide)。

## 概述：Claude Code 的 Subagent 系统

Claude Code 的 subagent 系统采用完全不同的架构思路。主 agent 运行在你的本地终端，可以按需生成（spawn）多个 subagent 来并行处理子任务。这些 subagent 共享同一个工作区上下文，通过 `Agent()` 工具调用启动，支持指定不同的 agent 类型（Explore、Plan、code-reviewer 等）。

Claude Code 的独特之处在于 [agent teams](/zh/blog/claude-code-agent-teams) 的概念——主 agent 扮演"项目经理"角色，将大任务拆解后分配给专门的 subagent，每个 subagent 可以使用 `isolation: "worktree"` 在独立的 git worktree 中工作，完成后汇报结果。这种模式保留了上下文连贯性的同时实现了任务隔离。实际使用案例可以参考[《Claude Code Subagents 实战》](/zh/blog/claude-code-subagents-examples)。

## 功能对比

| 功能维度 | Codex Subagents | Claude Code Subagents | 优势方 |
|---------|-----------------|----------------------|--------|
| **运行环境** | 云端沙箱容器 | 本地终端 + 可选 worktree | 各有千秋 |
| **任务隔离** | 天然隔离（独立容器） | 可选隔离（worktree 模式） | Codex |
| **上下文共享** | 任务间不共享 | 主 agent 可汇总 subagent 结果 | Claude Code |
| **自定义 Agent 类型** | 通过 AGENTS.md 配置 | 内置多种类型 + 自定义 prompt | Claude Code |
| **异步执行** | 原生支持，可后台批量运行 | 支持 `run_in_background` | Codex |
| **实时交互** | 提交后等待结果 | 实时对话、中途调整 | Claude Code |
| **代码审查集成** | PR 生成后人工审查 | 内置 review agent | Claude Code |
| **定价模式** | 按任务计费 | 按 token 计费 | 视使用量而定 |

## 任务编排：架构差异详解

这是两个系统最核心的区别。Codex 的任务编排是"fire-and-forget"模式——你提交一个任务描述，Codex 在云端完成所有工作，最终输出一个 PR 或代码变更。多个任务之间没有通信机制，无法在执行过程中协调。这种设计简化了系统复杂度，但也意味着你无法实现"agent A 的结果作为 agent B 的输入"这样的工作流。

Claude Code 的 subagent 系统支持更复杂的编排模式。主 agent 可以：

- 同时启动多个独立的 subagent 并行执行
- 等待某个 subagent 的结果后再决定下一步
- 将一个 subagent 的输出作为另一个的输入
- 在执行过程中根据中间结果动态调整策略

举个实际例子：你需要重构一个模块并确保所有测试通过。在 Claude Code 中，主 agent 可以先启动一个 Explore agent 分析代码依赖，拿到结果后启动重构 agent 和测试 agent 并行工作，最后用 review agent 检查变更质量。整个过程在一次对话中完成。关于多 agent 工作流的深入探讨，可以阅读[《为多智能体工作流而生》](/zh/blog/con-u-pour-des-workflows-multi-agents)。

在 Codex 中实现类似工作流，你需要手动分阶段提交任务：先提交分析任务，等结果出来后再提交重构任务，然后提交测试任务。每一步都需要人工介入衔接。

## 自定义能力：配置与扩展

Codex 通过 `AGENTS.md` 文件提供项目级自定义——你可以定义编码规范、测试要求、依赖约束等。这些指令对所有 Codex 任务全局生效。Codex 的 VS Code 扩展允许你在 IDE 中直接发起任务，提供了更便捷的交互界面。详情可参考[《Codex VS Code 扩展》](/zh/blog/codex-vscode)。

Claude Code 的自定义层次更丰富。除了项目级的 `CLAUDE.md`，还有：

- **Skill 文件**：`skills/*/SKILL.md` 定义特定任务的详细指令，比如"如何写测试"或"如何做代码审查"
- **自定义 subagent 类型**：通过 `subagent_type` 参数指定 agent 角色
- **Hooks 系统**：在特定事件触发时执行自定义脚本
- **MCP Server**：通过 Model Context Protocol 接入外部工具和数据源

这种多层扩展体系让 Claude Code 更像一个可编程的 [agentic coding](/zh/glossary/agentic-coding) 平台，而不仅仅是一个编码助手。想了解完整的扩展栈，可以阅读[《Claude Code 扩展栈拆解》](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 什么情况下选择 Codex

Codex 的核心优势在于云端隔离和异步批量处理。以下场景更适合 Codex：

- **批量 issue 修复**：一次性提交 10 个 bug 修复任务，让它们在云端并行执行，你去喝杯咖啡回来看结果
- **CI/CD 集成**：Codex 可以作为 CI 流程的一部分自动处理代码生成或修复任务
- **安全敏感项目**：代码在云端沙箱中处理，不需要在本地安装全部依赖
- **团队协作**：多个团队成员可以同时向同一个仓库提交 Codex 任务，互不冲突
- **开源项目维护**：[Codex for Open Source](/zh/blog/codex-for-open-source) 提供了专门的开源项目支持，适合处理大量外部贡献者的 PR

## 什么情况下选择 Claude Code Subagents

Claude Code 的 subagent 系统在以下场景更具优势：

- **复杂重构**：需要跨多个文件的协调性修改，subagent 之间的结果需要互相参考
- **探索性开发**：不确定最终方案，需要在过程中根据发现调整方向
- **代码审查**：内置的 [review agent](/zh/blog/claude-code-review-agents) 可以执行多维度的代码审查
- **实时调试**：与 AI 对话式排查问题，根据每一步结果决定下一步动作
- **知识密集型任务**：需要深度理解项目上下文、历史决策和架构约束的场景

如果你的日常工作是坐在终端前与 AI 协作编程，Claude Code 的 subagent 系统提供了更流畅的体验。[Agent SDK](/zh/glossary/agent-sdk) 的开放也意味着你可以在此基础上构建自己的 agent 工作流。

## 结论

**Codex subagents 和 Claude Code subagents 不是同一维度的竞争。** Codex 是"任务队列"模式——你提交任务，它在云端处理，你审查结果。Claude Code 是"实时协作"模式——你在终端与 AI 并肩工作，subagents 是你随时可以调用的专家团队。

如果你的团队已经形成了"写 ticket → 分配 → 审查 PR"的标准化工作流，Codex 的异步批处理模式可以无缝融入。如果你更看重开发过程中的灵活性和深度上下文理解，Claude Code 的 subagent 系统是更好的选择。

对于有能力的团队，最佳方案可能是两者结合：用 Codex 处理标准化、可并行的任务，用 Claude Code subagents 处理需要深度思考和实时协调的复杂工作。

## 常见问题

### Codex 的 subagent 可以互相通信吗？

目前 Codex 的任务之间是完全隔离的，不支持任务间直接通信。每个任务在独立容器中执行，结果通过 PR 或代码变更输出。如果需要任务间的协调，需要人工在中间环节衔接。

### Claude Code 最多可以同时运行多少个 subagent？

Claude Code 没有硬性的 subagent 数量限制，但实际受限于本地计算资源和 API 调用频率。通常建议同时运行 3-5 个 subagent 以保持响应速度和上下文质量。使用 worktree 隔离模式时，还需考虑磁盘空间。

### 两个平台的自定义 agent 配置可以互相迁移吗？

不能直接迁移。Codex 使用 `AGENTS.md`，Claude Code 使用 `CLAUDE.md` 和 `SKILL.md`，两者的语法和功能范围不同。但核心思路——用 markdown 文件定义项目约束和编码规范——是相通的，手动迁移的工作量不大。
