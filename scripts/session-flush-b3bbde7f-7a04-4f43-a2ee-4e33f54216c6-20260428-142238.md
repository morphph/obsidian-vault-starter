# Claude Memory vs CLAUDE.md：两套记忆系统该怎么选？

**TL;DR：** **CLAUDE.md** 是项目级的规则手册——团队共享、签入 Git、每次对话自动加载，适合编码规范和项目约束。**Claude Memory** 是个人化的自动笔记本——跨会话持久化、按相关性召回，适合记录用户偏好和项目动态。两者不是竞争关系，而是互补的两层持久化架构。选一个用是浪费，全用才是正确姿势。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件，放在仓库根目录，随代码一起签入版本控制。每次启动 Claude Code 会话时，它会自动读取这个文件作为系统级上下文。你可以把它理解为"给 AI 写的 README"——告诉 Claude 这个项目用什么技术栈、遵循什么编码规范、有哪些已知的坑要避开。

CLAUDE.md 的核心价值在于**团队一致性**。一个团队里十个人用 Claude Code，只要共享同一份 CLAUDE.md，AI 的行为就是一致的：相同的 commit message 风格、相同的测试要求、相同的架构约束。它解决的是"AI 助手不了解我们项目规矩"的问题。

除了项目根目录的 CLAUDE.md，还有全局级的 `~/.claude/CLAUDE.md`（个人偏好）和目录级的子 CLAUDE.md（子模块特定规则），形成一个三层指令体系。详细的[七层架构解析](/zh/blog/claude-code-seven-programmable-layers)中有完整说明。

## 概述：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Claude Code 的自动化知识管理系统，以文件形式存储在 `~/.claude/projects/<project>/memory/` 目录下。与 CLAUDE.md 的"你写给 AI"不同，Memory 更多是"AI 写给自己"——它在对话过程中自动识别值得记住的信息，保存为结构化的 Markdown 文件。

Memory 系统支持四种类型：**user**（用户画像）、**feedback**（行为反馈）、**project**（项目动态）、**reference**（外部资源指针）。每条记忆都有 frontmatter 元数据，通过 `MEMORY.md` 索引文件进行管理。Claude 在后续对话中会根据相关性自动召回这些记忆。

关于 Claude 记忆系统的[最新升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，包括免费用户开放和 ChatGPT 数据导入，Anthropic 在持续扩展这套系统的能力边界。

## 核心差异详解：定位与机制

CLAUDE.md 和 Claude Memory 解决的是持久化的两个完全不同的维度，理解这一点是正确使用它们的前提。

**CLAUDE.md 是声明式规则。** 它告诉 Claude"你必须做什么"和"你绝对不能做什么"。比如："commit 前必须跑 `npm test`"、"不要在 pipeline 脚本里 import Next.js 模块"、"中文内容必须用 CJK word count"。这些是硬性约束，不随对话变化，不因上下文不同而打折扣。每次会话启动时全量注入，确保 AI 从第一条消息就遵守所有规矩。

**Memory 是归纳式知识。** 它记录的是 Claude 从交互中学到的东西——"这个用户是数据科学家"、"上次他说不要在测试里 mock 数据库"、"这个项目正在做合规驱动的 auth 重写"。这些信息有生命周期，会过时，需要验证，但它们让 Claude 在跨会话场景中保持连贯性，不会每次都从零开始理解你。

从技术实现看，CLAUDE.md 走的是 system prompt 注入路径——内容直接拼入系统提示词，对 Claude 来说是"指令"级别的优先级。Memory 走的是上下文增强路径——内容作为参考信息提供，Claude 会判断相关性后决定是否使用。这个区别很重要：CLAUDE.md 里写"不允许 force push"，Claude 就会拒绝 force push；Memory 里记录"用户上次说不喜欢长摘要"，Claude 会倾向于写短摘要，但在特定场景下可能做出不同判断。

## 核心差异详解：协作与演进

**团队协作维度上，两者的差异最为显著。**

CLAUDE.md 天然支持团队协作。它签入 Git，走 PR 审查流程，团队成员可以讨论、修改、回滚。当新人加入团队，clone 仓库就自动获得了所有 AI 行为规范。这与代码风格指南（`.eslintrc`）或编辑器配置（`.editorconfig`）是同一个范式——用文件定义约定，用版本控制管理变更。

Memory 是彻底的个人空间。你的 Memory 记录了你的角色、你的偏好、你给过的反馈。同一个项目，后端工程师和前端工程师的 Memory 完全不同——Claude 会根据各自的技术背景调整解释方式和建议方向。这种个性化是 CLAUDE.md 做不到的，也不应该做的。

**演进模式也截然不同。** CLAUDE.md 的变更是低频且刻意的——新增一条构建规则、更新技术栈版本、记录一个新发现的 gotcha。每次变更都应该有明确的原因。Memory 的更新是高频且有机的——每次对话都可能产生新的记忆条目，旧的记忆会因过时被清理或更新。

实际使用中，最常见的错误是把动态信息写进 CLAUDE.md。比如"当前正在重构 auth 模块"——这类信息会过时，属于 Memory 的 project 类型。CLAUDE.md 应该只包含相对稳定的规则和约束。关于如何写好 CLAUDE.md，这篇[Claude Code 记忆系统详解](/zh/blog/claude-code-memory)有非常实用的指导。

## 什么时候用 CLAUDE.md

选择 CLAUDE.md 来存储以下内容：

- **构建和部署命令**：`npm run build`、`npm test`、部署流程——团队每个人都需要 AI 知道这些
- **编码规范和质量门禁**：commit 前必须通过的检查、禁止的模式、必须遵循的风格
- **已知陷阱（Gotchas）**：项目特有的技术坑，比如"ZH 内容必须用 CJK word count"
- **架构约束**：不允许在某些模块中引入特定依赖、API 设计规则
- **文档更新规则**：修改了哪类代码就必须更新哪个文档

CLAUDE.md 的黄金法则：**如果这条规则只对你个人有效，它不属于 CLAUDE.md；如果这条规则对每个使用 Claude Code 的团队成员都适用，它必须在 CLAUDE.md 里。**

## 什么时候用 Claude Memory

选择 Memory 来存储以下内容：

- **个人偏好**："不要在回复末尾加摘要"、"我喜欢简洁的 commit message"
- **用户画像**："我是后端工程师，不熟悉 React"——让 AI 调整解释方式
- **项目动态**："本周四代码冻结，mobile 团队要切分支"——有时效性的信息
- **行为反馈**："不要 mock 数据库，上次因此踩了坑"——从错误中学习
- **外部资源**："pipeline bug 在 Linear 的 INGEST 项目里追踪"——系统间的指针

Memory 的黄金法则：**如果这条信息有时效性、有个人属性、或者是从交互中归纳出来的，它属于 Memory。**

## 最佳实践：两者协同

最有效的使用方式不是二选一，而是让两套系统各司其职：

1. **CLAUDE.md 定义底线，Memory 优化体验**——CLAUDE.md 确保 AI 不会违反项目规则，Memory 确保 AI 理解你的工作方式
2. **CLAUDE.md 先行，Memory 补充**——新项目第一件事是写 CLAUDE.md，Memory 会随着使用自然积累
3. **定期审查 Memory**——Memory 会过时，特别是 project 类型的记忆。养成习惯检查 `MEMORY.md` 索引，清理不再准确的条目
4. **不要在 CLAUDE.md 里写个人偏好**——你喜欢什么样的代码风格是你的事，不要强加给整个团队
5. **不要在 Memory 里存项目规则**——如果是所有人都必须遵守的约束，它必须在 CLAUDE.md 里，否则只有你的 Claude 会遵守

## 总结

**CLAUDE.md 是法律，Memory 是经验。** 法律写在法典里，所有人必须遵守，变更需要审慎讨论；经验记在脑子里，因人而异，随时间更新。用 CLAUDE.md 管理项目的硬性规则和团队约定，用 Memory 管理个人偏好和动态上下文。两者结合，Claude Code 才能在保持一致性的同时提供个性化的辅助体验。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。CLAUDE.md 在每次会话启动时全量加载作为系统指令，Memory 按相关性选择性召回作为上下文补充。两者通过不同的注入路径协同工作，不会冲突。把 CLAUDE.md 当作项目级规则，Memory 当作个人级知识。

### 如果 CLAUDE.md 和 Memory 里的信息冲突了，Claude 会听哪个？

**CLAUDE.md 优先。** CLAUDE.md 的内容作为系统指令注入，优先级高于 Memory 的上下文增强。如果 CLAUDE.md 要求"commit message 必须用英文"，即使 Memory 记录了"用户偏好中文 commit message"，Claude 也会遵循 CLAUDE.md 的规则。发现冲突时应及时更新 Memory，移除过时的记录。

### 团队新成员需要手动配置 Memory 吗？

不需要。Memory 会在对话过程中自动积累。新成员 clone 仓库后立即获得 CLAUDE.md 的所有项目规则，而 Memory 会随着他们与 Claude 的交互逐渐建立个人化的知识库。前几次对话可能需要多解释一些背景，但 Claude 会把关键信息自动存入 Memory，后续对话就会越来越顺畅。
