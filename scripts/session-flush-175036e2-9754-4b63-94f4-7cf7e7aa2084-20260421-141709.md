**CLAUDE.md is human-authored.** You write every line. You decide what Claude Code needs to know about your project — the build system, the testing requirements, the architectural constraints. When those rules change, you update the file deliberately, through the same code review process as any other source file. This gives you complete control: nothing gets into CLAUDE.md that you did not put there, and nothing gets removed without your explicit action.

**Claude Memory is AI-authored.** Claude observes your interactions, detects useful context, and writes memory files on your behalf. You can trigger this explicitly ("remember that I prefer single-file PRs for this module") or let it happen organically as Claude picks up on corrections and preferences. You can also ask Claude to forget something, and it will remove the relevant memory file.

This distinction has practical consequences. CLAUDE.md is auditable — you can `git blame` any line and trace it to a specific commit and author. Memory is not auditable in the same way — it is a byproduct of conversation, and its contents depend on which conversations you have had and what Claude chose to extract from them.

For **team-wide rules** (coding standards, build commands, deployment procedures), CLAUDE.md is the only appropriate choice. You do not want AI-authored, per-developer memory files governing whether tests run before commits. For **individual workflow preferences** (preferred response verbosity, areas of expertise, role context), Memory is the natural fit because these are inherently per-person.

## Persistence and Sharing: Local vs Version-Controlled

CLAUDE.md lives in your repository and travels with your code. When a new developer clones the repo, they get the same CLAUDE.md as everyone else. When you push a change to CLAUDE.md, every team member's next Claude Code session picks it up automatically. This makes it the shared source of truth for project-level AI behavior.

Claude Memory lives on your local filesystem at `~/.claude/projects/`. It does not sync to any remote server, does not appear in your git history, and is invisible to your teammates. If you switch machines, your memories do not follow (unless you manually copy the directory). If a teammate has a different understanding of the project, their memory files reflect that — which is by design, since memories capture individual context.

This separation matters for **onboarding and consistency**. A well-written CLAUDE.md means that a new team member's first Claude Code session behaves identically to a veteran's — at least for the deterministic rules. Memory provides the personalization layer on top: over time, Claude learns that the new developer is less familiar with the auth module and adjusts its explanations accordingly, while the veteran gets more concise responses.

For solo developers, the distinction is less critical — you are the only author and the only consumer of both systems. But even solo developers benefit from putting stable rules in CLAUDE.md (so they survive machine migrations and are explicit) and letting Memory handle the evolving context.

## What Belongs Where: A Decision Framework

The most common mistake is putting the wrong type of information in the wrong system. Here is a concrete framework.

### Put in CLAUDE.md

- **Build and test commands**: `npm run build`, `npm test`, `pytest`
- **Quality gates**: "All tests must pass before commit"
- **Style rules**: "Use CJK word count for Chinese content"
- **Architectural constraints**: "Do not import Next.js modules in pipeline scripts"
- **Known gotchas**: Function signatures, parameter requirements, common pitfalls
- **Workflow rules**: "New feature → discuss design first, get approval before coding"
- **Documentation requirements**: What to update when specific files change

These are **stable, shared, and deterministic**. They do not depend on who is asking or what conversation happened yesterday.

### Put in Claude Memory

- **Your role and expertise**: "Senior backend engineer, new to the React side of this repo"
- **Communication preferences**: "Terse responses, no trailing summaries"
- **Validated approaches**: "For refactors in this area, user prefers one bundled PR"
- **Project state**: "Merge freeze begins March 5 for mobile release cut"
- **External references**: "Pipeline bugs tracked in Linear project INGEST"
- **Corrections**: "Integration tests must hit a real database, not mocks — prior incident with mock/prod divergence"

These are **personal, evolving, and context-dependent**. They improve over time and differ between team members.

### The Gray Zone

Some information sits at the boundary. A project decision like "we chose Postgres over SQLite for the auth service because of concurrent write requirements" could go in either place. The test: **will this matter to every developer, or just to you?** If every developer needs to know it to avoid making the wrong choice, put it in CLAUDE.md. If it is background context that helps Claude give you better-informed suggestions, Memory is fine.

## How They Work Together: The Two-Layer Architecture

Claude Memory and CLAUDE.md are not competing systems — they are complementary layers in Claude Code's [seven programmable layers](/blog/claude-code-seven-programmable-layers). Understanding how they compose is key to using Claude Code effectively.

**Layer 1: CLAUDE.md loads first.** At the start of every conversation, Claude Code reads your global `~/.claude/CLAUDE.md` and your project-level `CLAUDE.md`. These set the baseline: what tools to use, what quality gates to enforce, what style to follow. Every conversation starts from the same foundation.

**Layer 2: Memory loads selectively.** Claude Code's memory index (`MEMORY.md`) is also loaded at conversation start, but individual memory files are accessed only when relevant. If you ask about the deployment pipeline, Claude may recall a project memory about the merge freeze. If you ask about testing strategy, it may recall your feedback about using real databases instead of mocks.

**The result is personalized consistency.** The deterministic rules from CLAUDE.md ensure that Claude Code never skips your tests or commits without building. The adaptive context from Memory ensures that Claude Code remembers your role, your preferences, and the project context that has accumulated over weeks of collaboration.

This two-layer architecture is what makes Claude Code's [memory system](/blog/claude-code-memory) more sophisticated than a simple instruction file. CLAUDE.md alone gives you consistency without personalization. Memory alone gives you personalization without consistency. Together, they give you both.

## Practical Setup: Getting the Most From Both

### Setting Up CLAUDE.md

Start with the essentials: build command, test command, and one or two critical constraints. A CLAUDE.md that tries to document everything becomes a maintenance burden. Focus on what Claude Code cannot infer from the codebase itself.

A minimal effective CLAUDE.md:

```markdown
## Commands
npm run build    # Must pass before commit
npm test         # Must pass before commit

## Rules
- New features require design discussion before implementation
- Never import server-only modules in client code
```

Grow it incrementally. When Claude Code makes a mistake that a rule would have prevented, add the rule. When a team member encounters a gotcha, document it. This organic growth mirrors how engineering teams build their contributing guides.

### Setting Up Memory

Memory requires no explicit setup — Claude Code writes memories automatically as you work. But you can accelerate it:

1. **State your role early**: "I'm a frontend engineer focused on the dashboard module" gives Claude a user memory that shapes all future interactions
2. **Correct explicitly**: When Claude does something wrong, say why — "Don't use mocks here because we had a prod incident" gives Claude a feedback memory with rationale
3. **Confirm good approaches**: When Claude makes a non-obvious choice that works, acknowledge it — this creates a positive feedback memory that reinforces the behavior

Periodically review your memory index at `~/.claude/projects/<project>/memory/MEMORY.md`. Remove memories that have become stale (project decisions that were reversed, deadlines that passed, roles that changed). Memory is most useful when it is current.

## Common Pitfalls

### Putting volatile information in CLAUDE.md

CLAUDE.md should contain rules that change infrequently. If you find yourself updating it every sprint with current project state ("we are currently working on the auth migration"), that information belongs in Memory or in your project management tool. CLAUDE.md changes should go through code review and be meaningful enough to justify a commit.

### Relying on Memory for critical rules

Memory is non-deterministic — Claude decides when to recall it and may not always surface the right memory at the right time. If a rule is critical (quality gates, security constraints, architectural boundaries), it must be in CLAUDE.md where it is guaranteed to load every session. Never rely on Memory for rules where forgetting would cause damage.

### Duplicating information across both

If a rule exists in both CLAUDE.md and Memory, they can drift out of sync. CLAUDE.md gets updated in a PR, but the memory file still reflects the old rule. Claude Code is designed to trust CLAUDE.md over Memory when they conflict, but the confusion wastes context window space and can lead to inconsistent behavior. Pick one home for each piece of information.

### Overloading CLAUDE.md with personal preferences

Global CLAUDE.md (`~/.claude/CLAUDE.md`) is for your personal rules — commit message style, response verbosity, git workflow. Project CLAUDE.md is for project rules. If you put personal preferences in the project CLAUDE.md, you are imposing them on every team member. Use Memory for preferences that are truly individual.

## When to Choose Claude Memory

- You are a **solo developer** accumulating context over weeks of work on a project
- You want Claude Code to **adapt to your expertise level** and communication style
- You need to track **evolving project state** — current sprint goals, recent decisions, active incidents
- You work across **multiple projects** and want Claude to remember context about each
- You want to capture **corrections and validated approaches** so Claude does not repeat mistakes

## When to Choose CLAUDE.md

- You need **team-wide consistency** — every developer's Claude Code session follows the same rules
- You have **critical quality gates** that must never be skipped (build, test, lint)
- You want rules that are **version-controlled and auditable** through your normal code review process
- You need to **onboard new developers** quickly with a working AI configuration out of the box
- You have **architectural constraints** or known gotchas that are specific to the codebase

## Verdict

**Use both — CLAUDE.md for the shared rules, Memory for the personal context.** CLAUDE.md is your project's constitution: deterministic, version-controlled, and team-wide. Claude Memory is your personal notebook: adaptive, evolving, and unique to your working relationship with Claude Code. The mistake is treating them as alternatives when they are designed as complementary layers.

If you are just getting started, **begin with CLAUDE.md**. Add your build command, test command, and two or three critical rules. Let Memory accumulate naturally over your first week of use. After a few sessions, review your memory index and clean up anything stale. ## Frequently Asked Questions

### Can Claude Memory override rules in CLAUDE.md?

No. CLAUDE.md instructions are loaded at conversation start and treated as authoritative project rules. If a memory file contradicts a CLAUDE.md rule, Claude Code is designed to follow CLAUDE.md. Memory provides supplementary context, not overriding instructions. Critical rules should always live in CLAUDE.md precisely because of this precedence.

### Does Claude Memory sync across machines?

No. Claude Memory is stored locally at `~/.claude/projects/` on your filesystem. If you switch to a new machine, your memories do not transfer automatically. You would need to manually copy the memory directory. CLAUDE.md, by contrast, syncs automatically through git whenever you pull the repository.

### How many rules should I put in CLAUDE.md?

Start small — build command, test command, and two or three critical constraints. A CLAUDE.md file that tries to document every coding convention becomes a maintenance burden and wastes context window space. Add rules incrementally when Claude Code makes mistakes that a rule would have prevented. Most effective CLAUDE.md files are under 100 lines.

### Can I see what Claude has stored in Memory?

Yes. Memory files are plain markdown stored at `~/.claude/projects/<project-hash>/memory/`. The `MEMORY.md` file in that directory is an index of all stored memories. You can read, edit, or delete any memory file directly. You can also ask Claude to show, update, or forget specific memories during a conversation.

### Should I check CLAUDE.md into version control?

Yes — the project-level CLAUDE.md should always be checked into your repository. This ensures every team member and every CI environment gets the same rules. Your global `~/.claude/CLAUDE.md` is personal and should not be committed to any project repo, similar to how you would not commit your shell profile or editor settings.
