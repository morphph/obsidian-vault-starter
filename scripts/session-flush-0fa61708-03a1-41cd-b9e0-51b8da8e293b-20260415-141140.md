**CLAUDE.md handles the "what" and "how"** — what tools to use, how to run the build, what patterns to follow, how to structure commits. These are objective, team-wide instructions that should be consistent and deterministic.

**Claude Memory handles the "who" and "when"** — who you are, how you prefer to work, what's currently happening in the project, when certain constraints apply. These are subjective, individual, and time-bound.

A practical split:

| Put in CLAUDE.md | Put in Claude Memory |
|-----------------|---------------------|
| `npm run build` — Production build | "User prefers terse responses" |
| "Run tests before every commit" | "User is senior backend, new to React" |
| "Use Tailwind v4, not v3 syntax" | "Merge freeze starts April 5 for mobile release" |
| "Never skip failing tests" | "Pipeline bugs tracked in Linear project INGEST" |
| "ZH content uses CJK word count" | "User confirmed: bundled PRs preferred for refactors" |

When you find yourself repeating instructions to Claude across multiple conversations, that's a signal to move them into CLAUDE.md. When Claude keeps making the same mistake because it forgets your correction, that's a signal to save a memory (or to tell Claude to remember it explicitly).

The [principles for writing effective Claude Code skills](/blog/9-principles-writing-claude-code-skills) apply equally to writing good CLAUDE.md instructions: be specific, be actionable, and test that Claude actually follows them.

## Common Mistakes

**Mistake 1: Putting personal preferences in CLAUDE.md.** Your preferred commit message style or verbosity level doesn't belong in a file the whole team reads. Use memory for individual preferences, CLAUDE.md for team standards.

**Mistake 2: Relying on memory for critical instructions.** If forgetting an instruction causes bugs, broken builds, or security issues, it needs to be in CLAUDE.md where it's always loaded. Memory's contextual surfacing means critical rules might not always appear.

**Mistake 3: Never updating CLAUDE.md.** A stale CLAUDE.md is worse than no CLAUDE.md — it actively misleads Claude. If your build command changed six months ago and CLAUDE.md still shows the old one, every new Claude Code session starts with a broken build. Review CLAUDE.md quarterly at minimum.

**Mistake 4: Duplicating content across both systems.** If a rule is in CLAUDE.md, don't also save it as a memory. Duplication creates confusion when one version gets updated and the other doesn't. Choose one home for each piece of context.

**Mistake 5: Using CLAUDE.md for ephemeral context.** "We're in a code freeze until Friday" doesn't belong in a git-tracked file. By the time the PR is reviewed and merged, the freeze might be over. Use memory for time-bound context.

## Verdict

**Use CLAUDE.md for everything the team needs to agree on. Use Claude Memory for everything that's personal to you or temporary.** Both systems are essential for effective Claude Code usage — CLAUDE.md provides the stable foundation of shared instructions, and Claude Memory provides the adaptive layer that makes Claude feel like it knows you and your current context.

If you're just starting with Claude Code, **start with CLAUDE.md**. Write your build commands, your test commands, and your top three coding rules. That alone dramatically improves Claude Code's output. Add Claude Memory naturally as you work — correct Claude when it gets something wrong, mention your role and preferences, and let the memory system build up organically.

For teams, CLAUDE.md is non-negotiable. For individual developers, Claude Memory is what turns Claude Code from a generic AI assistant into one that genuinely understands how you work. For a complete walkthrough of both systems, read our [Claude Code Memory System guide](/blog/claude-code-memory).

## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

Claude Memory does not override CLAUDE.md instructions. CLAUDE.md is always loaded and treated as authoritative project-level guidance. If a memory conflicts with a CLAUDE.md rule, CLAUDE.md takes precedence. Memory supplements CLAUDE.md with personal context — it doesn't replace or contradict it.

### Is Claude Memory shared with my team?

No. Claude Memory files live in your local `.claude/projects/` directory and are not git-tracked. Each developer has their own independent memory store. If you want context shared across the team, put it in CLAUDE.md and commit it to the repository.

### How do I see what Claude has remembered about me?

Check your `.claude/projects/` directory for the relevant project path. The `MEMORY.md` file contains an index of all stored memories with one-line summaries. Individual memory files in the same directory contain the full content. You can read, edit, or delete any memory file directly.

### Does CLAUDE.md work in CI/CD environments?

Yes. Since CLAUDE.md is checked into your repository, any environment that clones the repo — including CI/CD pipelines running Claude Code — automatically gets the same instructions. Claude Memory does not persist in CI/CD environments since those typically start with a clean filesystem.

### How long should my CLAUDE.md be?

Keep it under 200 lines for the project-level file. Claude reads the entire file at the start of every conversation, so longer files consume context window space that could be used for your actual task. Focus on high-impact instructions: build commands, critical constraints, and known gotchas. Move detailed guidelines into skill files (`skills/*/SKILL.md`) instead.
