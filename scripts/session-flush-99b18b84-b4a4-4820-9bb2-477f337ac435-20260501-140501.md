**Claude Code runs locally.** It operates as a process on your machine, reading files from your actual working directory, executing commands in your real shell environment, and modifying your codebase in place. This means it has access to everything you have access to — your environment variables, your running services, your database, your build toolchain. When Claude Code runs `npm test`, it runs against your actual project configuration. When it edits a file, the change is immediately visible in your editor.

This local execution model has significant implications. Claude Code can interact with running services, hit local APIs, access private networks, and use tools that require specific system configurations. It also means you need to trust the agent with shell access — Claude Code uses a permission system where you approve or deny commands, but the blast radius of a mistake is your local environment.

**Codex runs in the cloud.** Each task gets its own microVM — a lightweight, sandboxed container with no network access. Codex clones your repository from GitHub, installs dependencies based on your setup scripts, and works in complete isolation. The sandbox cannot reach the internet, cannot access your local services, and cannot affect anything outside its container. When the task completes, Codex produces a diff or opens a pull request.

This isolation is both a strength and a constraint. On the security side, a Codex task cannot accidentally delete your production database or leak credentials to an external service. On the capability side, it cannot test against services that require network access, cannot use private package registries without pre-configuration, and cannot interact with tools that only exist on your local machine.

For teams with strict security requirements, Codex's sandbox model is attractive — every task runs in a disposable environment with no persistent side effects. For individual developers who need the agent to work with their full development stack, Claude Code's local model provides capabilities that cloud sandboxing cannot replicate.

## Developer Workflow and Integration: Detailed Analysis

The workflow difference between these tools shapes how you structure your day as a developer. Claude Code is a conversation partner; Codex is a task queue.

**Claude Code's interactive workflow** means you stay engaged. You describe a task, watch Claude Code plan its approach, see it read files and run commands in real time, and redirect it when it goes off track. You can interrupt mid-task ("actually, use the v2 API instead"), add context ("the auth module was refactored last week, check the new structure"), or ask it to explain its reasoning before it makes changes. This tight feedback loop reduces wasted work but requires your attention.

Claude Code's [agent teams](/blog/claude-code-agent-teams) feature adds a layer of parallelism — the primary agent can spawn sub-agents to handle independent subtasks, like researching a codebase while simultaneously drafting a solution. But the overall interaction remains synchronous: you are present and directing.

**Codex's asynchronous workflow** lets you offload tasks and context-switch. You open the ChatGPT interface, type "refactor the payment processing module to use the new Stripe API," and move on. Codex works in the background. You can queue five tasks, go to lunch, and come back to five pull requests waiting for review. This model excels when you have a backlog of well-defined, independent tasks.

The tradeoff is precision. Because Codex works asynchronously, you cannot course-correct during execution. If it misunderstands the task or takes a wrong approach, you discover this only when reviewing the output. For ambiguous or complex tasks that benefit from iterative refinement, this adds latency — you review, reject, re-describe, and wait again.

**IDE integration** differs meaningfully. Claude Code is terminal-native, with extensions for VS Code and JetBrains that embed the CLI experience into the editor. Codex offers a VS Code extension that lets you assign tasks from your editor and a web interface through ChatGPT. Developers already embedded in VS Code may find Codex's integration more natural; terminal-centric developers will prefer Claude Code's approach.

**Context systems** also diverge. Claude Code uses [CLAUDE.md for project memory](/blog/claude-code-memory) and SKILL.md for task-specific instructions — these are version-controlled files in your repo that any team member's Claude Code instance will pick up automatically. Codex uses an `AGENTS.md` convention that serves a similar purpose but with a different ecosystem of tooling around it. Both approaches solve the "teach the AI your codebase conventions" problem; Claude Code's system is more mature, with hooks and MCP integrations that extend the context pipeline.

## Model Capabilities and Task Performance: Detailed Analysis

Claude Code is powered by Anthropic's Claude model family — primarily Claude Sonnet for speed-sensitive tasks and Claude Opus for complex reasoning. You can switch models based on the task. The extended thinking feature lets Claude reason through multi-step problems before acting, which improves accuracy on architectural decisions and complex refactoring.

Codex uses a purpose-built model called **codex-1**, based on OpenAI's o3 architecture and fine-tuned specifically for software engineering tasks. OpenAI describes it as optimized for reading existing codebases, making targeted edits, and following coding conventions. The model runs in a reinforcement-learning-from-human-feedback loop that is specifically calibrated for code review and editing workflows.

Direct benchmark comparisons between the underlying models are difficult because they are deployed differently. Claude Code's models handle both the reasoning and the tool-use orchestration in a single agent loop. Codex-1 operates within a more constrained execution environment where the model's outputs are limited to file edits and shell commands within the sandbox. Both achieve strong results on standard coding benchmarks, but real-world performance depends heavily on the specific task, codebase, and how well the project context is configured.

Where Claude Code has a clear edge is in tasks requiring external context — accessing documentation sites, hitting APIs, querying databases, or interacting with running services. Because it runs locally with full shell access, it can gather information that a sandboxed model simply cannot reach. Codex compensates by allowing you to provide context upfront and through its AGENTS.md configuration, but it cannot dynamically discover information during execution the way Claude Code can.

Claude Code handles large codebases well because it reads your local file system directly and can use agent teams to parallelize exploration across directories. For monorepos with complex build configurations or local service dependencies, Claude Code's local execution is generally more practical.

### Does Codex have access to the internet during task execution?

No. Codex tasks run in sandboxed microVMs with no network access. The agent can only work with the repository contents and pre-installed dependencies. This is a deliberate security choice — it prevents the agent from leaking code or credentials to external services, but it also means Codex cannot fetch documentation, access APIs, or install packages not already configured in the project.

Code quality depends more on project context and task definition than on the underlying model. Claude Code's advantage is its rich context system — CLAUDE.md files, skills, and hooks that enforce coding standards automatically. Codex benefits from its AGENTS.md configuration and the focused codex-1 model. For well-configured projects, both tools produce professional-quality output. The difference is in the feedback loop: Claude Code lets you catch and correct quality issues in real time, while Codex requires post-hoc review.
