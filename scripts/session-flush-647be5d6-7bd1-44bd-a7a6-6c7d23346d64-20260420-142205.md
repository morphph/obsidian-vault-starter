# Claude Memory vs CLAUDE.md：两套记忆系统到底有什么区别？

**TL;DR：** **Claude Memory** 是 Anthropic 为 Claude 产品线提供的跨会话自动记忆功能，由模型自主决定记住什么；**CLAUDE.md** 是 Claude Code 的项目级指令文件，由开发者手动编写，随代码库版本控制。两者解决不同层面的上下文持久化问题：Memory 记住「你是谁」，CLAUDE.md 定义「项目怎么做」。大多数 Claude Code 用户应该同时使用两者。

## 概览：Claude Memory

**Claude Memory** 是 Anthropic 在 Claude 产品中内置的持久化记忆系统，能够在多次对话之间自动记住用户的偏好、背景和关键信息。用户无需每次重复自我介绍或项目背景——Claude 会在对话中主动识别值得记住的内容，并在后续对话中自动调用。

这套系统最早在 Claude.ai 网页端和桌面端上线，后来扩展到 Claude Code 的 auto memory 功能。2026 年初，Anthropic 对记忆系统做了[重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，向免费用户开放了记忆功能，并支持从 ChatGPT 导入历史数据。

Memory 的核心设计哲学是**零配置**：用户不需要手动管理，模型自动完成信息的提取、存储和召回。记忆内容包括用户角色、技术偏好、常用工具链、工作习惯等。在 Claude Code 中，auto memory 会将信息写入 `~/.claude/projects/` 目录下的 markdown 文件，按 user、feedback、project、reference 四种类型分类存储。

## 概览：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级配置文件，放在代码仓库根目录下，随 Git 一起版本控制。它不是模型自动生成的——而是开发者手动编写的一份「项目操作手册」，告诉 Claude Code 这个项目的技术栈、构建命令、代码规范、工作流约束等关键信息。

每次 Claude Code 启动时，会自动读取当前目录及上级目录的 CLAUDE.md 文件，将其内容注入到系统提示词中。这意味着文件里写的每一条规则都会直接影响 Claude 的行为。团队中每个成员使用 Claude Code 时，都会遵循同一份 CLAUDE.md 的约束——这是它最大的优势。

CLAUDE.md 通常包含：构建和测试命令、代码风格要求、Git 工作流规范、项目架构说明、已知问题和注意事项。更深入的了解可以参考我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 核心对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **生成方式** | 模型自动提取 | 开发者手动编写 | 各有所长 |
| **作用范围** | 跨项目、跨会话 | 单个项目内 | Memory 更广 |
| **版本控制** | 不在 Git 中 | 随代码库版本控制 | CLAUDE.md |
| **团队共享** | 仅个人可用 | 团队所有成员共享 | CLAUDE.md |
| **内容类型** | 用户偏好、反馈、项目背景 | 构建命令、代码规范、工作流 | 各有侧重 |
| **可编辑性** | 可手动编辑但通常自动管理 | 完全手动控制 | CLAUDE.md |
| **适用平台** | Claude.ai + Claude Code | 仅 Claude Code | Memory 更广 |
| **持久化位置** | `~/.claude/projects/` 目录 | 项目根目录 | 各有合理性 |

## 记忆机制的深层差异

两套系统虽然都解决「让 Claude 记住上下文」这个问题，但它们的技术实现和设计目标完全不同。

**Claude Memory** 基于模型的主动推理。在对话过程中，Claude 会判断哪些信息值得长期保存——比如你说「我是数据工程师，主要用 Python」，模型会自动将这条信息写入记忆文件。下次你开启新对话时，这些记忆被加载到上下文中，Claude 就知道不需要再问你的技术背景。这种机制的优点是自动化程度高，缺点是你无法精确控制模型记住什么。

**CLAUDE.md** 则是确定性的配置注入。文件内容在每次会话启动时被完整读取并注入系统提示词，没有任何模型推理参与。你写什么，Claude 就看到什么。这意味着你可以精确控制 Claude Code 在项目中的行为——从「commit 前必须跑测试」到「不要在 pipeline 脚本里 import Next.js 模块」，每条规则都是硬约束。

这种差异带来一个重要推论：**CLAUDE.md 适合团队规范，Memory 适合个人习惯**。一个前端团队可以在 CLAUDE.md 里规定「使用 Tailwind v4，不要引入 styled-components」，但每个开发者的个人偏好（比如「我喜欢简短的 commit message」）应该交给 Memory 系统。

## 实际工作流中的配合使用

理解两者的区别后，关键问题是：在实际开发中应该如何配合使用？

**CLAUDE.md 负责「项目法律」。** 把所有团队成员都必须遵守的规则写在这里：构建命令、测试要求、代码规范、Git 工作流、已知 bug 和注意事项。这些内容应该经过 code review，像代码一样维护。我们在 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)中详细讨论了 CLAUDE.md 在整个可编程层级中的定位。

**Memory 负责「个人助理」。** 你的角色背景、技术偏好、沟通风格、对 Claude 行为的反馈——这些因人而异的信息交给 Memory 自动管理。比如你纠正过一次「不要在每次回复末尾加总结」，Memory 会记住这个偏好并在后续对话中遵守。

**SKILL.md 是第三层。** 除了 CLAUDE.md 和 Memory，Claude Code 还有 [Skills 系统](/zh/blog/5-claude-code-skills-i-use-every-single-day)——放在 `skills/` 目录下的任务级指令文件。如果说 CLAUDE.md 是「项目宪法」，SKILL.md 就是「岗位操作手册」，而 Memory 是「个人笔记」。三者配合使用才能发挥 Claude Code 的全部潜力，具体实践可参考 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)。

## 常见误区

很多开发者在初次接触这两套系统时，容易犯以下错误：

**误区一：把个人偏好写进 CLAUDE.md。** CLAUDE.md 是团队共享的，不应该包含个人习惯。「我喜欢用 vim 键位」这种信息不属于项目配置。

**误区二：依赖 Memory 存储项目规范。** Memory 是个人的、非确定性的，Claude 可能忘记或误解。关键的项目约束必须写在 CLAUDE.md 里，确保每次都被准确注入。

**误区三：认为两者互相替代。** 它们解决的是不同层面的问题。删除 CLAUDE.md 不会让 Memory 自动补上项目规范；清空 Memory 也不会影响项目配置的加载。

## 选择 Claude Memory 的场景

当你需要跨项目的个人化体验时，Memory 是正确的选择：

- **多项目切换频繁**：你同时维护 5 个项目，Memory 记住你的通用偏好（编码风格、沟通方式），避免在每个项目的 CLAUDE.md 里重复配置
- **使用 Claude.ai 网页端**：CLAUDE.md 只在 Claude Code 中生效，如果你在网页端使用 Claude，Memory 是唯一的持久化机制
- **渐进式调教**：通过日常对话逐步让 Claude 理解你的工作方式，而不是一次性写完所有规则
- **迁移历史偏好**：Anthropic 已支持从 ChatGPT [导入记忆数据](/zh/blog/anthropic-claude-memory-upgrades-importing)，这对从其他平台迁移的用户尤其有价值

## 选择 CLAUDE.md 的场景

当你需要可靠、可审计、团队共享的项目约束时，CLAUDE.md 不可替代：

- **团队协作项目**：确保每个使用 Claude Code 的团队成员遵循相同的规范，从构建流程到代码风格
- **CI/CD 集成**：在自动化流程中调用 Claude Code 时，CLAUDE.md 确保行为一致性，不依赖任何人的个人记忆
- **关键约束执行**：「commit 前必须通过所有测试」这类硬性规则必须写在 CLAUDE.md 里——Memory 系统没有这种强制力
- **项目特定知识**：技术栈选型、已知 gotcha、架构决策等信息属于项目而非个人，适合版本控制

如果你正在考虑如何组织 Claude Code 的可编程层级，我们的 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp) 提供了完整的架构视角。

## 结论

**两者不是竞争关系，而是互补关系。** CLAUDE.md 是确定性的项目配置层，解决「这个项目怎么做」的问题；Claude Memory 是自适应的个人记忆层，解决「这个人怎么工作」的问题。最佳实践是：项目规范写 CLAUDE.md，个人偏好交给 Memory，任务级指令用 SKILL.md。三层配合，才是 Claude Code 记忆系统的完整形态。

对于刚开始使用 Claude Code 的开发者，建议先花 10 分钟写好 CLAUDE.md——把构建命令、测试要求、核心规范写进去。Memory 系统会在你日常使用中自动积累，不需要额外配置。

## 常见问题

### Claude Memory 的内容会被团队其他成员看到吗？

不会。Claude Memory 存储在用户本地的 `~/.claude/` 目录下，完全是个人数据，不会通过 Git 或任何方式与团队共享。团队共享的上下文应该写在 CLAUDE.md 中。

### CLAUDE.md 支持在子目录中覆盖配置吗？

支持。Claude Code 会从当前工作目录向上递归读取所有 CLAUDE.md 文件，子目录的配置会叠加到父目录的配置之上。这意味着你可以在 monorepo 的子包中放置特定的 CLAUDE.md 来覆盖或补充根目录的规则。

### 清空 Claude Memory 会影响 CLAUDE.md 的效果吗？

不会。两套系统完全独立运行。CLAUDE.md 是通过文件读取注入到系统提示词的，不依赖 Memory 系统。清空记忆只会让 Claude 忘记你的个人偏好，项目配置不受任何影响。

### 如果 Memory 和 CLAUDE.md 的内容冲突，哪个优先？

CLAUDE.md 的优先级更高。它作为系统提示词的一部分被直接注入，是硬约束。Memory 中的偏好如果与 CLAUDE.md 的规则冲突，Claude Code 应该遵循 CLAUDE.md 的要求。
