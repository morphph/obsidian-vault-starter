# Claude Memory vs CLAUDE.md: Which Context System Should You Use?

**TL;DR:** **Claude Memory** and **CLAUDE.md** are complementary, not competing. CLAUDE.md is your team's shared instruction manual — checked into git, explicit, deterministic. Claude Memory is your personal knowledge base — built automatically across conversations, private to you, and adaptive. **Use CLAUDE.md for project rules everyone must follow. Use Claude Memory for personal context that makes your sessions smarter over time.** Most teams need both.

## Overview: Claude Memory

**Claude Memory** is [Claude Code's](/glossary/claude-code) persistent, file-based memory system that accumulates context across conversations automatically. It stores what it learns about you — your role, preferences, feedback, project state, and external references — in structured markdown files under `.claude/projects/*/memory/`. Each memory has typed frontmatter (user, feedback, project, or reference) and a central `MEMORY.md` index that gets loaded into every conversation.

The system is designed to eliminate repetition. Instead of re-explaining your testing philosophy or team structure every session, Claude Memory retains these details and applies them automatically. It's personal — your memories don't affect your teammates' sessions — and it's adaptive, updating as your preferences evolve. Anthropic's [memory upgrades](/blog/anthropic-claude-memory-upgrades-importing) have expanded this system to support importing context from other tools, making it easier to onboard Claude Code into existing workflows.

## Overview: CLAUDE.md

**CLAUDE.md** is a project-level configuration file that lives in your repository root (or nested directories) and provides explicit instructions to Claude Code. It defines coding standards, build commands, architectural decisions, forbidden patterns, and workflow rules. Because it's checked into version control, every team member — and every Claude Code session — gets the same instructions.

Think of CLAUDE.md as your project's constitution. It answers: "How do we work here?" The file is read at the start of every session, providing deterministic context that doesn't depend on conversation history. You write it manually, review it in PRs, and evolve it alongside your codebase. It's the primary mechanism for ensuring Claude Code follows your team's conventions without being told each time.

## Context Delivery: How Each System Informs Claude Code

Claude Memory and CLAUDE.md solve the same fundamental problem — giving Claude Code enough context to be useful — but they deliver that context through entirely different mechanisms.

**CLAUDE.md loads at session start as a system-level instruction.** Claude Code reads the file before your first message and treats its contents as authoritative project rules. This means CLAUDE.md instructions override default behavior. If your CLAUDE.md says "never use `any` types in TypeScript," Claude Code will follow that rule in every file, every session, for every team member. The determinism is the point — you want build commands, lint rules, and architectural constraints to be identical regardless of who's running the session.

**Claude Memory loads from an index file and is consulted contextually.** The `MEMORY.md` index is always present in conversation context, but individual memory files are read on demand when relevant. If you previously told Claude Code that you prefer integration tests over unit tests, that feedback memory influences test-generation tasks — but it won't override an explicit CLAUDE.md rule saying "write unit tests for all utility functions."

The hierarchy matters: CLAUDE.md rules take precedence over memory-based preferences when they conflict. This is by design — team standards should override personal preferences in shared codebases. ## Use Cases: What Belongs Where

The biggest source of confusion is deciding what goes in CLAUDE.md versus what should be left to memory. The rule is simple: **if your team needs to agree on it, it goes in CLAUDE.md. If it's about how you personally work, let memory handle it.**

**Belongs in CLAUDE.md:**
- Build, test, and lint commands (`npm run build`, `npm test`)
- Coding style rules (naming conventions, forbidden patterns, import ordering)
- Architecture constraints ("never import server modules in client code")
- Deployment workflows and quality gates
- Project-specific gotchas the whole team hits

**Belongs in Claude Memory:**
- Your role and expertise level ("I'm a data scientist new to this frontend")
- Communication preferences ("don't summarize after edits, I read diffs")
- Personal workflow patterns ("I always want tests before implementation")
- External resource locations ("our bugs are tracked in Linear project X")
- Project context not derivable from code ("we're freezing merges Thursday for release")

The [Claude Code memory blog post](/blog/claude-code-memory) covers the full taxonomy of memory types — user, feedback, project, and reference — and when each applies.

## Team Dynamics: Shared vs Personal Context

For solo developers, the distinction between Claude Memory and CLAUDE.md is mostly organizational. You could put everything in CLAUDE.md and it would work fine. The separation matters when you add teammates.

**CLAUDE.md creates team alignment.** When a new engineer joins and runs Claude Code on your repo, they get the same instructions you do. No onboarding conversations needed — the rules are in the file. This is why companies like Shopify and Spotify standardize on CLAUDE.md for their engineering conventions, as explored in our coverage of [Claude Code in enterprise engineering](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify).

**Claude Memory creates personal efficiency.** Your senior backend engineer and your junior frontend developer have different needs from the same codebase. Memory lets Claude Code adapt its explanations, suggestions, and workflow to each person without polluting the shared configuration. The senior engineer's memory might say "deep Go expertise, skip basic explanations." The junior's might say "new to React, explain component lifecycle."

This separation also prevents conflicts. If you prefer tabs and your colleague prefers spaces, that's a CLAUDE.md decision (team standard wins). But if you like verbose commit messages and your colleague likes terse ones, memory handles that personal preference without forcing a team debate.

## Maintenance and Evolution

**CLAUDE.md requires active maintenance.** As your project evolves — new build tools, changed conventions, deprecated patterns — someone must update the file. Stale CLAUDE.md instructions actively harm productivity because Claude Code follows them faithfully even when they're wrong. Treat CLAUDE.md updates like code: review them in PRs, discuss changes, keep them current.

**Claude Memory is largely self-maintaining** but can go stale. Memories about project state ("we're in a code freeze") become outdated. Claude Code is designed to verify memories against current state before acting on them, but explicit cleanup helps. You can tell Claude Code to forget outdated information, and it will remove the relevant memory file.

For teams investing heavily in Claude Code, the recommended practice is: review your CLAUDE.md quarterly, and let memory accumulate organically with occasional pruning. The [skills system](/blog/5-claude-code-skills-i-use-every-single-day) provides a middle layer — reusable instruction files for specific tasks that are shared like CLAUDE.md but scoped like memory.

## When to Choose Claude Memory

Choose Claude Memory (or let it build naturally) when:

- **You're working solo** and want Claude Code to learn your preferences without writing a config file
- **Your context is personal** — role, expertise level, communication style, workflow habits
- **Information changes frequently** — project timelines, current initiatives, who's working on what
- **You want zero-effort context accumulation** — just work normally and let the system learn
- **You need to store references** to external systems (Linear projects, Slack channels, dashboards)

Claude Memory shines for the "soft" context that makes AI assistance feel personalized rather than generic. It's the difference between Claude Code that knows you're a staff engineer doing a security audit versus one that treats every session as a blank slate.

## When to Choose CLAUDE.md

Choose CLAUDE.md when:

- **Rules must be universal** — every team member and every session must follow them
- **Instructions are deterministic** — build commands, test requirements, lint rules don't vary by person
- **You need version control** — track when rules changed, who changed them, and why
- **Onboarding matters** — new team members should get full context immediately
- **Compliance requires it** — auditable, reviewable instructions for how AI operates in your codebase

CLAUDE.md is essential for any team larger than one person. Even solo developers benefit from writing one — it forces you to articulate your project's conventions explicitly, which makes Claude Code dramatically more effective. Our guide on [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) covers CLAUDE.md best practices in detail.

## Verdict

**Use both — they're designed to work together.** CLAUDE.md is your project's law; Claude Memory is your personal assistant's notebook. Start with a CLAUDE.md that covers build commands, style rules, and architecture constraints. Let Claude Memory accumulate naturally as you work — it will learn your preferences, remember your feedback, and adapt to your workflow.

The mistake teams make is trying to put everything in one system. Cramming personal preferences into CLAUDE.md creates noise for teammates. Relying solely on memory means new sessions start cold and different team members get inconsistent behavior. **CLAUDE.md for rules, memory for context** — that's the split.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions take precedence when they conflict with memory-based preferences. CLAUDE.md is treated as authoritative project configuration, while memory provides supplemental context. If your CLAUDE.md says "use Vitest" but your memory says "I prefer Jest," Claude Code will use Vitest.

### Does Claude Memory sync across machines?

Claude Memory is stored locally in your `.claude/` directory. It does not sync automatically across machines. If you work from multiple devices, each will build its own memory independently. CLAUDE.md, being in git, syncs everywhere your repo does.

### How much should I put in CLAUDE.md?

Keep CLAUDE.md focused on what Claude Code needs to know to work correctly in your project — build commands, style rules, architectural constraints, and known gotchas. A good CLAUDE.md is 50-200 lines. If it exceeds 500 lines, consider splitting specialized instructions into [SKILL.md files](/blog/9-principles-writing-claude-code-skills) for specific tasks.

### Can I see what Claude Memory has stored?

Yes. Memory files are plain markdown stored in `.claude/projects/*/memory/`. You can read, edit, or delete them directly. The `MEMORY.md` index file shows all stored memories at a glance. You can also ask Claude Code to recall or forget specific information during any conversation.

### Do I need both for a solo project?

CLAUDE.md is still valuable solo — it ensures consistent behavior across sessions without relying on memory accumulation. But for a small personal project, you could rely primarily on memory and add a minimal CLAUDE.md with just build commands and key constraints.
