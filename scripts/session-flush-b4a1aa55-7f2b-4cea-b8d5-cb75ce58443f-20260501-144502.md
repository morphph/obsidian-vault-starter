**Custom agents require explicit orchestration.** You need to decide when to invoke which agent, how to pass context between them, and how to handle conflicts when two agents propose contradictory changes. This orchestration can be manual (you trigger each agent in sequence) or automated (you build a pipeline that chains agents together).

The automated pipeline approach is where custom agents become genuinely powerful. A typical pipeline might look like:

1. **Planner agent** analyzes the task and produces a structured work plan
2. **Implementer agent** writes the code changes
3. **Test agent** generates and runs tests against the changes
4. **Reviewer agent** checks for style violations, security issues, and documentation gaps
5. **Committer agent** creates the PR with a structured description

Each agent in this pipeline has a single responsibility and clear inputs/outputs. The pipeline is deterministic — the same task produces the same sequence of agent invocations. This predictability is critical for teams that need to audit and explain their AI-assisted workflows.

## Use Case Analysis: Where Each Approach Excels

### Subagents Excel At:

**Large-scale refactoring.** When you need to rename a module, update every import, fix every test, and adjust documentation — all in one pass — subagents shine. The parent agent understands the full scope of changes needed and can parallelize the mechanical work across multiple subagents. Each subagent handles a subset of files, and the parent reconciles any conflicts.

**Codebase exploration and analysis.** Tasks like "find all API endpoints that don't have rate limiting" or "identify which tests are flaky and why" benefit from subagent parallelism. Multiple subagents can scan different parts of the codebase simultaneously, dramatically reducing analysis time on large repositories.

**One-off complex tasks.** When you have a task that's too big for a single agent pass but not recurring enough to justify building a custom agent, subagents provide the right level of automation. You get multi-agent benefits without any configuration overhead.

### Custom Agents Excel At:

**Recurring team workflows.** If your team runs the same type of code review, test generation, or documentation update every sprint, a custom agent that encodes your standards saves time on every invocation. The upfront configuration cost is amortized across dozens or hundreds of runs.

**Compliance and security reviews.** Custom agents with carefully scoped instructions can enforce specific security standards, check for common vulnerability patterns, or verify regulatory compliance. The deterministic nature of custom agent behavior — same instructions produce consistent review criteria — makes them suitable for workflows that need audit trails.

**Onboarding and knowledge transfer.** A custom agent that embodies your team's coding standards, architectural patterns, and review criteria acts as a always-available mentor for new team members. Instead of asking a senior engineer "how do we handle errors in the API layer?", a new developer can ask the custom agent and get an answer grounded in your codebase's actual patterns.

**Cross-tool integration.** Custom agents can be configured to work with specific external tools — databases, monitoring systems, deployment pipelines — through carefully scoped [MCP server](/glossary/agent-sdk) connections. This tool-specific configuration isn't possible with ad-hoc subagents.

## Cost and Token Efficiency

Token usage is a practical concern for teams running multi-agent workflows at scale, and the two approaches have different cost profiles.

**Subagents** tend to use more tokens because the system makes its own decisions about how many agents to spawn and how much context each one needs. A task that you might handle with two well-scoped custom agents could result in four or five subagents, each processing overlapping context. The automatic decomposition optimizes for task completion quality, not token efficiency.

**Custom agents** give you direct control over token usage. You decide how many agents run, what context each one receives, and how much output each one produces. A well-designed custom agent pipeline can be significantly more token-efficient than automatic subagent delegation, especially for tasks where you already know the optimal decomposition.

For teams on usage-based pricing — which includes both [Codex](/blog/codex-complete-guide) API access and comparable tools like Claude Code — this cost difference compounds over time. A team running 50 multi-agent tasks per week might see a 30-40% reduction in token costs by migrating from ad-hoc subagents to well-configured custom agents for their most common workflows.

That said, custom agents have their own hidden costs: the engineering time spent configuring, testing, and maintaining agent definitions. For infrequent tasks, the configuration overhead outweighs the token savings.

## Debugging and Observability

When multi-agent workflows produce unexpected results, your ability to diagnose the problem depends heavily on which approach you used.

**Subagent debugging is opaque.** The parent agent decides how to decompose the task, what instructions to give each subagent, and how to reconcile their outputs. If the final result is wrong, you need to reconstruct what happened: Which subagent handled which files? What instructions did each subagent receive? Where did the conflict or error originate? Codex provides some visibility into subagent execution, but the automatic nature of the decomposition means you're reverse-engineering decisions you didn't make.

**Custom agent debugging is transparent.** Each agent has explicit instructions, defined tool access, and a clear scope. When something goes wrong, you can trace the issue to a specific agent, review its instructions, and identify whether the problem is in the prompt, the tool configuration, or the orchestration logic. This transparency is especially valuable for teams that need to understand and explain their AI-assisted workflows — whether for internal review, compliance, or simply building trust with stakeholders.

The observability difference matters more as task complexity increases. For simple two-agent decompositions, subagent debugging is manageable. For workflows involving five or more agents with interdependent outputs, custom agent transparency becomes essential.

## Migration Path: Starting Simple and Scaling Up

Most teams follow a natural progression from subagents to custom agents as their multi-agent usage matures.

**Phase 1: Subagents only.** Start with Codex's built-in subagent delegation. Submit tasks, let the system decompose them, and observe how the work gets split. Pay attention to which decomposition patterns work well and which produce poor results.

**Phase 2: Identify recurring patterns.** After a few weeks of subagent usage, you'll notice patterns: certain types of tasks always benefit from a specific decomposition, certain agent roles keep appearing, and certain failure modes recur. Document these patterns — they become the basis for your first custom agents.

**Phase 3: Build custom agents for high-frequency tasks.** Start with your most common recurring task — usually code review or test generation. Build a custom agent that handles this task better than ad-hoc subagent delegation. Measure the improvement in output quality, consistency, and token cost.

**Phase 4: Hybrid workflows.** Use custom agents for well-understood, recurring work and subagents for ad-hoc, exploratory, or one-off tasks. This hybrid approach gives you the best of both: consistency where it matters, flexibility where it's needed.

This migration path is similar to what teams experience with other [agentic coding](/glossary/agentic-coding) tools. Our analysis of [5 skills used daily](/blog/5-claude-code-skills-i-use-every-single-day) shows the same progression from ad-hoc prompting to structured, reusable configurations.

## Frequently Asked Questions

### Can I use subagents and custom agents together in the same Codex session?

Yes. A custom agent can delegate subtasks to subagents when it encounters work that benefits from parallel decomposition. The custom agent defines the high-level strategy and behavioral constraints, while subagents handle the parallel execution of specific subtasks within that strategy.

### How many custom agents should a typical team maintain?

Most teams find 3-5 custom agents cover the majority of their recurring workflows: a code reviewer, a test writer, a documentation updater, and one or two domain-specific agents. More than 8-10 agents usually signals over-specialization — consider consolidating agents with overlapping responsibilities.

### Do subagents have access to the same tools as the parent agent?

Subagents inherit the parent agent's tool permissions by default. They can read and write files, run shell commands, and access any integrations the parent session has configured. Custom agents, by contrast, can have individually scoped tool access that restricts them to only the capabilities they need.

### Are custom agents portable between Codex and other AI coding tools?

The instructions and behavioral definitions are mostly portable — a well-written system prompt for a code review agent works across platforms. The tool configuration and orchestration layer is platform-specific. Teams that invest in high-quality agent instructions get value regardless of which platform they ultimately standardize on.

### What happens when two subagents make conflicting changes to the same file?

The parent agent handles conflict resolution by reviewing both subagents' changes and merging them. In practice, the parent agent usually re-applies one subagent's changes on top of the other's, similar to resolving a git merge conflict. If the conflict is too complex, the parent agent may re-run the task without subagent delegation.
