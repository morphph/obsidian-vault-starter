**CLAUDE.md is a team artifact.** When one engineer adds a rule to CLAUDE.md — say, "never import server-only modules in pipeline scripts" — every team member's Claude Code sessions immediately follow that rule after pulling the change. This makes CLAUDE.md the right place for institutional knowledge: architectural decisions, known gotchas, required quality gates, and coding conventions. It functions like a living style guide that Claude Code actually enforces.

Teams that use CLAUDE.md effectively treat it like infrastructure code. Changes go through pull requests. New rules get discussed before merging. The file grows organically as the team discovers patterns — a bug caused by a missing parameter becomes a documented constraint, a recurring review comment becomes an automated rule. Over time, CLAUDE.md encodes the team's collective engineering judgment.

**Claude Memory is a personal artifact.** It captures how one developer prefers to work, not how the team has agreed to work. This is an important distinction. Your memory might note that you're a backend specialist unfamiliar with the React portion of the codebase — Claude Code uses this to frame frontend explanations differently for you. Your colleague's memory might note that they prefer terse responses without summaries. These preferences would be inappropriate in CLAUDE.md because they're individual, not collective.

Memory also stores corrections that are personal in nature. When you tell Claude Code "stop suggesting TypeScript migrations for the legacy module," that's a workflow preference specific to your current focus, not a team-wide rule. If the entire team agreed to freeze the legacy module, that decision belongs in CLAUDE.md.

The failure mode teams should watch for: putting personal preferences in CLAUDE.md pollutes shared instructions with individual quirks. Putting team conventions in memory means they're invisible to other team members and will be inconsistently applied. Claude Code's [seven programmable layers](/blog/claude-code-seven-programmable-layers) are designed to handle this separation cleanly — CLAUDE.md operates at the project layer, memory at the user layer.

For organizations scaling Claude Code across engineering teams, CLAUDE.md is the primary lever. It ensures consistent AI behavior regardless of who's running the session. Memory is a secondary optimization that improves individual developer experience.

## What to Store Where: A Decision Framework

The most common confusion between Claude Memory and CLAUDE.md isn't about what each system *does* — it's about which system should hold a specific piece of information. Here's a concrete decision framework.

**Put it in CLAUDE.md if:**

- It's a project convention that every developer should follow (coding style, naming patterns, import rules)
- It's a build or test command that must run before commits (`npm run build`, `npm test`)
- It's an architectural constraint derived from a technical decision ("don't import Next.js modules in pipeline scripts")
- It's a known gotcha that has bitten someone before ("upsertKeyword() requires three parameters")
- It's a workflow requirement ("new features need design discussion before coding")
- It would need to survive if you left the team tomorrow

**Put it in Claude Memory if:**

- It's about your role or expertise level ("senior backend engineer, new to React")
- It's a behavioral correction specific to how you work ("don't summarize at the end of responses")
- It's a pointer to an external resource you personally reference ("bugs tracked in Linear project INGEST")
- It's about an ongoing project or initiative with a deadline ("merge freeze begins March 5")
- It's about your communication preferences ("I prefer concise explanations with code examples")
- It would be irrelevant or confusing to other team members

**Put it in neither if:**

- It can be derived by reading the code (file structure, function signatures, type definitions)
- It's in git history (`git log`, `git blame` are authoritative)
- It's a debugging solution (the fix is in the code; the commit message has context)
- It's ephemeral to the current conversation (use tasks or plans instead)

**Edge cases worth calling out:**

- *Team member roles and responsibilities* — memory, not CLAUDE.md. "Alice owns the auth module" is useful context but changes frequently and is personal to your understanding of the team.
- *Environment variables and secrets setup* — CLAUDE.md for the variable names and purpose, never for actual values. Memory for pointers to where secrets are stored ("API keys in 1Password vault 'Engineering'").
- *Skill files and their purpose* — CLAUDE.md should reference that skills exist and where to find them. Memory shouldn't duplicate skill content. If you find yourself saving a memory about how a skill works, the [skill file itself](/blog/5-claude-code-skills-i-use-every-single-day) should be improved instead.

## When to Rely on Claude Memory

Choose Claude Memory as your primary context mechanism when you're working solo or when the context is inherently personal. Specific scenarios:

**Solo projects and side work.** When you're the only developer, the distinction between personal and team context collapses. Memory is faster to build — Claude Code creates it automatically as you work — while CLAUDE.md requires manual authoring. For a personal project, memory handles most context needs without the overhead of maintaining a separate instruction file.

**Onboarding to a new codebase.** When you're ramping up on an unfamiliar project, memory captures your current understanding and knowledge gaps. "I don't understand the auth flow yet" is valuable context that helps Claude Code calibrate its explanations. As your understanding grows, these memories naturally get updated or removed.

**Cross-project knowledge.** Global memories (stored at the user level rather than the project level) carry context across repositories. Your role, communication preferences, and general technical background apply regardless of which repo you're working in. CLAUDE.md is strictly per-repository.

**Rapidly changing project context.** During incident response, sprint planning, or active development pushes, project context changes fast. "We're in a code freeze until Thursday" is a memory, not a CLAUDE.md rule — it expires in days and would clutter the permanent project instructions.

## When to Rely on CLAUDE.md

Choose CLAUDE.md as your primary context mechanism when consistency, auditability, and team sharing matter. Specific scenarios:

**Team projects with multiple Claude Code users.** The moment a second developer starts using Claude Code on your repo, CLAUDE.md becomes essential. Without it, their Claude Code sessions have zero context about project conventions, and any corrections they make are invisible to you. CLAUDE.md ensures everyone's agent follows the same rules.

**Open source repositories.** Contributors come and go. CLAUDE.md onboards every new contributor's Claude Code session automatically — they clone the repo, and their agent immediately knows the build commands, test requirements, and coding standards. Memory can't serve this function because it doesn't travel with the code.

**Codebases with strict quality gates.** If your project requires specific checks before commits — builds must pass, tests must pass, linting must pass — these rules belong in CLAUDE.md where they're enforceable and visible. Relying on individual memory to enforce quality gates means one developer's misconfigured memory could bypass critical checks. See how projects implement these gates in our [guide to Claude Code's extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp).

**Long-lived projects with accumulated institutional knowledge.** Every "don't do X because Y happened last quarter" rule belongs in CLAUDE.md. Institutional knowledge should be durable, shared, and version-controlled. It's too valuable to live in one person's memory.

**Monorepos with subdirectory-specific conventions.** CLAUDE.md's hierarchical scoping handles this naturally. A root CLAUDE.md sets global standards, and subdirectory CLAUDE.md files add context-specific rules. Memory has no equivalent scoping mechanism.

## How Claude Memory and CLAUDE.md Work Together

The most effective Claude Code setups use both systems in concert, with each handling what it's best at. Here's how the interaction works at runtime and how to design your workflow around it.

**At session start**, Claude Code loads three context layers in order: the system prompt (built-in behavior), CLAUDE.md files (project instructions), and memory entries (personal context). All three inform Claude Code's behavior, but CLAUDE.md instructions take priority over memory when they conflict — a deliberate design choice that ensures team conventions override personal preferences.

**During a session**, Claude Code reads CLAUDE.md but writes to memory. If you correct Claude Code's behavior and the correction is personal ("I prefer shorter explanations"), it saves a memory. If the correction reveals a project rule that should be shared ("always run the validator before pushing pipeline changes"), you should add it to CLAUDE.md manually. Claude Code sometimes suggests this distinction by noting when a correction seems project-scoped rather than personal.

**A practical workflow example:** You're working on a codebase with a CLAUDE.md that specifies "use Vitest for all tests, run `npm test` before commits." Your memory notes that you're a data scientist investigating logging infrastructure — you're not deeply familiar with the test setup. When you ask Claude Code to add logging to a module, it follows CLAUDE.md's test requirements (writes Vitest tests, runs them before committing) while calibrating its explanations to your memory context (explains the test structure rather than assuming you know it).

**Where the systems reinforce each other:** CLAUDE.md defines *what* Claude Code should do. Memory refines *how* it communicates and adapts to you. CLAUDE.md says "commit messages must describe what changed." Memory says "this developer prefers conventional commit format with scope prefixes." Both apply, and neither conflicts.

**Where friction can occur:** If your memory contains outdated project information that contradicts current CLAUDE.md instructions, Claude Code is designed to trust CLAUDE.md. But stale memories about project architecture or workflows can still cause confusion if they inform Claude Code's reasoning before it checks CLAUDE.md. Regular memory maintenance — reviewing and pruning outdated entries — prevents this.

## Verdict

**Use both.** CLAUDE.md is your project's source of truth — shared conventions, build commands, quality gates, and architectural constraints that every developer and every Claude Code session must follow. Claude Memory is your personal optimization layer — your role, your preferences, your corrections, and your contextual notes that make Claude Code work better for you specifically.

If you're forced to choose one: **start with CLAUDE.md**. It has higher leverage — one well-written CLAUDE.md file improves every Claude Code session for every developer on the project. Memory is valuable but secondary; Claude Code functions well without it, just with more repeated corrections. CLAUDE.md without memory is a consistent but slightly impersonal assistant. Memory without CLAUDE.md is a personalized assistant that might violate project conventions.

For teams adopting Claude Code, the recommended sequence is: write CLAUDE.md first, let memory accumulate naturally, and review both periodically. ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions are treated as high-priority project rules. When a memory conflicts with a CLAUDE.md instruction, Claude Code follows CLAUDE.md. This is by design — team conventions should not be overridden by individual preferences. Memory informs behavior in areas where CLAUDE.md is silent.

### Does Claude Memory sync across machines?

Claude Memory is stored locally in `.claude/` directories and does not automatically sync between machines. If you work on multiple computers, your memories will differ. CLAUDE.md, because it's checked into git, syncs naturally through your normal git workflow. Some developers manually copy their `.claude/` directory, but there is no official sync mechanism.

### Should I check `.claude/` memory files into git?

No. Memory files are personal and often contain context specific to one developer's role, preferences, and workflow. Checking them into git would impose one person's preferences on the team — exactly the problem CLAUDE.md solves through deliberate, reviewed project instructions. Keep `.claude/` in your `.gitignore`.

### How do I migrate a useful memory into CLAUDE.md?

When a personal memory turns out to be a team-relevant rule — for example, you discovered a gotcha that every developer should know — manually add it to CLAUDE.md and remove the memory entry. There is no automated migration. The manual step is intentional: it forces a conscious decision about whether context should be personal or shared.

### Do SKILL.md files replace either system?

No. SKILL.md files are task-specific instruction sets that define how Claude Code approaches particular types of work (writing tests, generating content, reviewing PRs). They complement both CLAUDE.md and memory. CLAUDE.md sets project-wide rules, SKILL.md sets task-specific rules, and memory sets personal context. All three can be active simultaneously. Learn more about skills in our [guide to daily Claude Code skills](/blog/5-claude-code-skills-i-use-every-single-day).
