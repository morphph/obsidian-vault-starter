# Codex Subagents vs Claude Code Subagents：多智能体编码工作流对比

**TL;DR：** OpenAI Codex 的 subagent 模式基于云端沙箱，每个任务在独立容器中运行，适合批量并行处理和异步工作流。**Claude Code 的 subagent 系统在本地终端运行**，通过 Agent 工具和自定义 agent 定义实现实时多 agent 协作，适合需要即时反馈和深度上下文共享的场景。选哪个取决于你的工作模式：批量派发选 Codex，交互式协作选 Claude Code。

## 概述：OpenAI Codex 的 Subagent 机制

OpenAI Codex 是一个云端编程智能体平台，每个任务在隔离的沙箱环境中执行。Codex 的"subagent"概念体现在任务拆分上——你可以将一个大需求拆成多个独立任务，每个任务获得自己的沙箱实例、文件系统快照和执行环境。

Codex 的核心设计哲学是**异步批处理**。你提交任务后不需要在线等待，Codex 在云端完成代码编写、测试运行，最终生成 PR 或 diff 供你审阅。这种模式下，多个"subagent"本质上是多个并行的 Codex 任务实例，彼此之间不共享运行时状态。关于 Codex 的完整能力，可参考我们的 [Codex 完全指南](/zh/blog/codex-complete-guide)。

Codex 目前通过 ChatGPT 界面和 [VS Code 扩展](/zh/blog/codex-vscode)两种方式触发任务。自定义 agent 的灵活性相对有限——你主要通过 AGENTS.md 文件定义项目级指令和约束，但无法在运行时动态指定 agent 的行为模式或工具集。

## 概述：Claude Code 的 Subagent 机制

Claude Code 的 subagent 系统直接内置于终端工具中，通过 Agent 工具实现。开发者可以在对话中随时派生子 agent，每个子 agent 拥有独立的上下文窗口，但共享同一个文件系统和 git 工作区。

Claude Code 的多 agent 架构支持多种专用 agent 类型：Explore（只读搜索）、Plan（架构设计）、codex-rescue（委托给 Codex 的混合模式）等。更关键的是，你可以通过 AGENTS.md 文件定义**自定义 agent**，指定它们的工具权限、提示模板和行为约束。详细的 agent 团队用法可参考 [Claude Code Agent Teams 详解](/zh/blog/claude-code-agent-teams)。

与 Codex 不同，Claude Code 的 subagent 在本地同步运行（也支持后台模式），主 agent 可以实时获取子 agent 的结果并据此调整后续步骤。这种**反应式编排**比 Codex 的批处理模式更适合需要中间决策的复杂任务。

## 功能对比

| 功能维度 | Codex Subagents | Claude Code Subagents | 优势方 |
|---------|----------------|----------------------|--------|
| **运行环境** | 云端沙箱容器 | 本地终端 + 可选 worktree | 各有优势 |
| **并行能力** | 原生多任务并行，无上限 | 支持并行派生，受本地资源限制 | Codex |
| **上下文共享** | 任务间隔离，不共享状态 | 共享文件系统，主 agent 汇总结果 | Claude Code |
| **自定义 Agent** | AGENTS.md 项目指令 | AGENTS.md + Agent 工具 + 类型系统 | Claude Code |
| **异步能力** | 原生异步，离线执行 | 同步为主，支持后台模式 | Codex |
| **工具访问** | 沙箱内预配置工具 | 完整 shell + MCP 服务器 | Claude Code |
| **审阅机制** | 执行完成后生成 PR | 实时交互，逐步确认 | 各有优势 |
| **定价模式** | ChatGPT Plus/Pro 订阅 | API 按 token 计费 | 视用量而定 |

## 多 Agent 编排：架构差异详解

这是两个平台最根本的分歧。Codex 采用的是**扇出-汇总（fan-out/fan-in）模式**——你把大任务拆成小任务，每个小任务独立执行，最后人工审阅合并结果。这种模式的优点是高度并行化，10 个任务可以同时跑；缺点是任务之间无法通信，如果任务 B 依赖任务 A 的结果，你需要手动串联。

Claude Code 采用的是**层级编排模式**——主 agent 作为协调者，根据需要派生子 agent，获取结果后决定下一步。子 agent 可以是预定义类型（Explore、Plan），也可以是你在 AGENTS.md 中定义的自定义 agent。主 agent 保持对整个工作流的控制权，能够根据中间结果动态调整策略。

实际使用中，Claude Code 的 subagent 实例展示了丰富的应用场景，具体案例可参考 [Claude Code Subagents 实战案例](/zh/blog/claude-code-subagents-examples)。

对于需要多步骤推理的重构任务——比如先分析依赖关系，再规划修改顺序，最后执行变更——Claude Code 的层级模式更自然。对于可以完全并行的独立任务——比如同时给 20 个模块写单元测试——Codex 的批处理模式效率更高。

## 自定义 Agent 的灵活性对比

在自定义能力上，两个平台的差距更为显著。

Codex 的 AGENTS.md 主要用于定义项目级约束：编码规范、测试要求、禁止使用的库等。它更像是一份"规则手册"，而不是一个 agent 定义系统。你无法通过 AGENTS.md 创建具有不同工具权限或行为模式的多种 agent 类型。

Claude Code 的自定义 agent 系统更为完整。通过 AGENTS.md，你可以定义多个 agent，每个 agent 有独立的：

- **工具权限**：指定该 agent 可以使用哪些工具（只读搜索、编辑文件、运行命令等）
- **提示模板**：定义 agent 的角色、行为约束和输出格式
- **模型选择**：为不同 agent 选择不同的模型（Opus 处理复杂任务，Haiku 处理简单查询）

这套系统配合 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中的 Skills 和 Hooks 机制，可以构建相当复杂的多 agent 工作流。而 Codex 在这方面仍处于较早期的阶段。

## 适合选择 Codex Subagents 的场景

如果你的团队工作模式偏向"派发任务-异步审阅"，Codex 是更自然的选择：

- **批量代码迁移**：同时启动数十个独立的文件转换任务，无需在线等待
- **开源项目维护**：[Codex for Open Source](/zh/blog/codex-for-open-source) 支持批量处理 issue 和 PR
- **CI/CD 集成**：Codex 的异步模式天然适配持续集成流水线
- **团队协作场景**：多人通过 ChatGPT 界面同时提交任务，共享项目上下文

Codex 的云端执行模式也意味着不受本地机器性能限制，这对资源受限的开发者来说是实际优势。

## 适合选择 Claude Code Subagents 的场景

如果你需要在编码过程中保持对 agent 工作流的实时控制，Claude Code 更合适：

- **复杂重构**：需要先分析再规划再执行的多阶段任务，[Agent Teams](/zh/blog/claude-code-agent-teams) 的层级编排是核心优势
- **探索性开发**：不确定最终方案时，让 Explore agent 先调研，Plan agent 出方案，再派生实现 agent
- **深度代码审查**：多个专用 agent 分别检查安全、性能、风格等维度
- **混合工作流**：通过 codex-rescue agent 类型，Claude Code 实际上可以委托任务给 Codex，实现两套系统的协同

Claude Code 的本地执行也意味着完整的 shell 访问权限——数据库查询、API 测试、日志分析都可以在 agent 工作流中完成。

## 结论

**如果你的核心需求是并行处理大量独立编码任务，选 Codex。如果你需要 agent 之间实时协作、共享上下文、动态调整策略，选 Claude Code。** 两者并不互斥——Claude Code 本身就提供了与 Codex 的集成路径。对于大多数日常开发场景，关键问题不是哪个平台更好，而是你的任务是"可并行的独立子任务"还是"需要协调的关联步骤"。前者是 Codex 的主场，后者是 Claude Code 的强项。

## 常见问题

### Codex 的 subagent 可以互相通信吗？

目前 Codex 的任务实例之间相互隔离，不支持运行时通信。如果任务之间有依赖关系，需要手动串联——先完成任务 A，再用 A 的结果启动任务 B。这是批处理架构的固有限制。

### Claude Code 的自定义 agent 需要额外付费吗？

不需要单独付费，但每个子 agent 消耗独立的 token 用量。派生一个 Opus 子 agent 处理复杂任务的成本会高于 Haiku 子 agent 做简单查询。合理选择 agent 的模型配置可以显著控制成本。

### 两个平台可以同时使用吗？

可以。Claude Code 内置了 codex-rescue 类型的 agent，可以将特定任务委托给 Codex 执行。这意味着你可以用 Claude Code 做主编排，把适合批量处理的子任务派发给 Codex 云端运行。
