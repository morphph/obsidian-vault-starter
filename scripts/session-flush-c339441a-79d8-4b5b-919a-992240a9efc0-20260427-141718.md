**CLAUDE.md loads deterministically at session start.** When you open Claude Code in a project directory, it reads every CLAUDE.md file in the hierarchy: `~/.claude/CLAUDE.md` (global), then the project root `CLAUDE.md`, then any nested CLAUDE.md files in subdirectories. This happens before you type anything. The content is injected into the system prompt verbatim — every word you write in CLAUDE.md becomes an instruction that Claude Code follows. There's no interpretation step, no relevance filtering. If your CLAUDE.md says "never use semicolons in JavaScript," that rule is active in every conversation.

**Memory loads selectively based on relevance.** Claude Code reads the `MEMORY.md` index at session start and has access to all memory files, but it uses them contextually. A memory about your testing preferences surfaces when you're working on tests. A memory about your role surfaces when Claude Code needs to calibrate its explanations. This selective loading is both a strength (less noise) and a weakness (important context might not surface when you expect it to).

The practical implication: put **critical rules** in CLAUDE.md where they're guaranteed to load. Let Memory handle **nice-to-have context** where occasional missed recall is acceptable.

## Team Collaboration: The Decisive Difference

The single biggest difference between these systems is how they handle teams, and this is where the choice becomes clear for most organizations.

**CLAUDE.md is a team artifact.** It lives in your repo. When a senior engineer adds "always use parameterized queries for SQL — we had an injection vulnerability in Q3 2025" to CLAUDE.md, every developer's Claude Code session enforces that rule immediately after pulling the change. New hires get the same guardrails as veterans. The rules evolve through pull requests with code review, just like any other project configuration. This makes CLAUDE.md the **single source of truth for how AI should behave in your codebase**.

**Memory is inherently personal.** Your memories about project architecture, your role, your preferred commit style — none of that transfers to teammates. If you've spent weeks training Claude Code through corrections and it finally handles your testing framework perfectly, a colleague starting fresh gets none of that benefit. There's no `git push` for Memory.

This isn't a design flaw — it's intentional. Memory captures *individual* working style, while CLAUDE.md captures *team* standards. The distinction matters because what works for one developer ("I prefer verbose error messages") might contradict another's preference ("keep error messages terse"). Memory accommodates both; CLAUDE.md enforces one standard.

For teams adopting Claude Code at scale, the recommendation from engineering organizations like those at [Ramp, Shopify, and Spotify](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify) has consistently been: invest heavily in CLAUDE.md as the shared foundation, and let Memory handle individual adaptation.

## Content Architecture: What Goes Where

Knowing which system to use for a specific piece of context is the most practical question. Here's the decision framework.

### Put in CLAUDE.md

- **Build and test commands**: `npm run build`, `npm test`, lint configurations
- **Coding standards**: naming conventions, file organization patterns, import ordering
- **Architectural constraints**: "never import server-only modules in client code," "all API routes must validate input with Zod"
- **Workflow rules**: "run tests before commit," "create a new branch for features," "always pull before starting work"
- **Known gotchas**: framework quirks, migration pitfalls, third-party API limitations specific to your project
- **Documentation requirements**: what needs updating when code changes, frontmatter standards
- **Security rules**: "never commit .env files," "use parameterized SQL," "validate user input at system boundaries"

### Put in Memory

- **Your role and expertise**: "senior backend engineer, new to React" — helps Claude Code calibrate explanations
- **Communication preferences**: "don't summarize at the end of responses," "keep answers terse"
- **Workflow corrections**: "when I say 'ship it,' I mean commit and push to the current branch, not create a PR"
- **Project context not in code**: "the auth rewrite is driven by a legal compliance deadline, not tech debt"
- **External references**: "bugs are tracked in Linear project INGEST," "the oncall dashboard is at grafana.internal/d/api-latency"
- **Teammate context**: "Alice owns the payment module, check with her before modifying `src/payments/`"

### The Gray Zone

Some context could live in either system. Decision rules:

- **If the whole team needs it** → CLAUDE.md
- **If it's about you personally** → Memory
- **If it changes with the code** → CLAUDE.md (version-controlled, updates in the same PR)
- **If it changes with organizational decisions** → Memory (not tied to a specific commit)
- **If getting it wrong causes a bug** → CLAUDE.md (deterministic loading, guaranteed visibility)
- **If getting it wrong causes a suboptimal explanation** → Memory (lower stakes, acceptable miss rate)

## Maintenance and Staleness

Both systems can become stale, but they rot in different ways and require different maintenance strategies.

**CLAUDE.md rots when the code changes but the docs don't.** If your CLAUDE.md says "run `yarn test`" but the project migrated to `pnpm` three months ago, every new Claude Code session starts with a broken instruction. The fix is straightforward: treat CLAUDE.md updates as part of your PR workflow. When you change a build command, update CLAUDE.md in the same commit. Some teams enforce this with CI checks.

**Memory rots when the world changes.** A memory that says "the deploy pipeline takes 20 minutes" might have been true when it was saved, but a recent infrastructure change cut it to 5 minutes. Unlike CLAUDE.md, stale Memory isn't caught by code review. Claude Code mitigates this by treating memories as claims to verify — it's instructed to check that referenced files still exist and functions haven't been renamed before acting on memory — but this verification is best-effort, not guaranteed.

The practical advice: **audit CLAUDE.md quarterly** as a team exercise. For Memory, trust Claude Code's built-in staleness checking, but be prepared to correct it when memories are clearly outdated.

## Layering Strategy: Using Both Together

The most effective Claude Code configurations use CLAUDE.md and Memory as complementary layers, not alternatives. Here's how to structure the stack.

**Layer 1 — Global CLAUDE.md (`~/.claude/CLAUDE.md`):** Your personal cross-project defaults. Git workflow preferences, commit message style, deployment habits. This is similar to Memory's "user" type but deterministic — you write it once and it applies everywhere.

**Layer 2 — Project CLAUDE.md (repo root):** Team-shared project rules. Build commands, coding standards, architectural constraints, known gotchas. This is the highest-value configuration file in any Claude Code project. Teams that write thorough [skill files](/blog/5-claude-code-skills-i-use-every-single-day) often complement them with a solid CLAUDE.md foundation.

**Layer 3 — Auto Memory:** Personal accumulated context that makes Claude Code progressively smarter about *you*. Your role, your feedback corrections, project decisions that aren't in the code, references to external systems. This layer is invisible to teammates and adapts automatically.

The layering matters because Claude Code processes all three. A project CLAUDE.md might say "use Vitest for all tests." Your Memory might add "this user prefers testing edge cases before happy paths." Both influence the output, but the CLAUDE.md rule is non-negotiable while the Memory preference is a suggestion.

## Skill Files: The Third Context System

It's worth addressing **SKILL.md** files briefly, since they're often confused with CLAUDE.md. Skill files are task-specific instruction sets that live in `skills/*/SKILL.md` — they define how Claude Code should approach a particular job (writing newsletters, generating SEO content, reviewing PRs). They're loaded on-demand, not at session start.

The hierarchy: CLAUDE.md sets project rules → Memory personalizes the experience → SKILL.md defines task-specific behavior. For a deeper exploration of how these layers interact, see [Claude Code's extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) and [how to write effective skills](/blog/9-principles-writing-claude-code-skills-that-actually-work).

## When to Choose Claude Memory

Choose Memory as your primary context mechanism when:

- **You're a solo developer** and team sharing isn't a concern — Memory's automatic accumulation saves you from manually maintaining CLAUDE.md
- **You're onboarding to a new project** and want Claude Code to learn your questions, misconceptions, and corrections over time without you explicitly documenting them
- **Your context is organizational, not code-level** — who owns which module, what's the deploy schedule, where are bugs tracked — information that doesn't belong in a repo file
- **You want low maintenance** — Memory builds itself through normal conversation; CLAUDE.md requires deliberate upkeep

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary context mechanism when:

- **You're on a team** and need every developer's Claude Code to follow the same rules — CLAUDE.md is the only option for shared, version-controlled AI instructions
- **Correctness is critical** — build commands, security rules, architectural constraints must load deterministically every time, not probabilistically based on relevance matching
- **You want reviewable AI configuration** — CLAUDE.md changes go through PRs, get reviewed by teammates, and have Git blame history
- **You're setting up a new project** and want Claude Code productive from the first session — a well-written CLAUDE.md provides immediate context without any learning period
- **You need to [prompt Claude Code effectively](/blog/how-to-effectively-prompt-a-claude-code)** with persistent, reproducible instructions

## Verdict

**Use both.** CLAUDE.md and Claude Memory are not competing systems — they're complementary layers that serve different purposes. **CLAUDE.md is your team's shared AI configuration**: deterministic, version-controlled, and reviewed. **Memory is your personal AI adaptation layer**: automatic, cumulative, and private.

If you're forced to choose one, **start with CLAUDE.md**. A well-written CLAUDE.md file delivers immediate, predictable value from the first session. Memory accumulates gradually and its benefits compound over weeks. But the most productive Claude Code users treat CLAUDE.md as the foundation and let Memory handle everything that doesn't belong in a shared repo file.

The one mistake to avoid: putting personal preferences in CLAUDE.md (where they become team rules) or putting critical project rules in Memory (where they might not surface). Match the context to the system that fits its sharing model and criticality level.

## Frequently Asked Questions

### Can Claude Memory override rules set in CLAUDE.md?

No. CLAUDE.md instructions are injected into the system prompt and treated as hard constraints. Memory provides supplementary context that influences Claude Code's behavior, but it cannot contradict explicit CLAUDE.md rules. If CLAUDE.md says "use Vitest" and a memory says "user prefers Jest," Claude Code follows the CLAUDE.md instruction.

### Does Claude Memory sync across devices?

Claude Memory is stored locally in the `.claude/projects/` directory structure. It does not automatically sync across machines. If you work on multiple devices, your Memory accumulates independently on each one. CLAUDE.md, by contrast, syncs through Git — another reason to put critical context there.

### How much should I write in CLAUDE.md?

Keep it focused and scannable. The most effective CLAUDE.md files are 50-200 lines covering build commands, key coding standards, workflow rules, and known gotchas. Avoid dumping entire architecture documents — link to them instead. If your CLAUDE.md exceeds 300 lines, consider splitting task-specific instructions into SKILL.md files.

### Can I disable Claude Memory entirely?

Yes. Claude Memory is opt-in behavior — you can delete the memory directory or instruct Claude Code to ignore memory in your CLAUDE.md. Some teams prefer this approach to ensure fully deterministic behavior controlled exclusively through version-controlled configuration files.

### What happens to Memory when I switch projects?

Memory is scoped per project directory. When you open Claude Code in a different project, it loads that project's memory index. Your global CLAUDE.md (`~/.claude/CLAUDE.md`) applies across all projects, but memory files are project-specific. This prevents context from one codebase from leaking into another.
