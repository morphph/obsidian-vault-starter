# Codex Subagents vs Custom Agents：哪种多智能体模式更适合你？

**TL;DR：** Codex 的 **Subagents** 是平台内置的轻量并行执行单元，适合将大任务拆解为独立子任务自动分配；**Custom Agents** 则是开发者通过 Agent SDK 自定义的智能体，拥有更灵活的工具链和行为控制。选择取决于你的需求：**快速分治用 Subagents，深度定制用 Custom Agents**。对于大多数工程团队，两者结合使用才能发挥 [Codex](/zh/blog/codex-complete-guide) 多智能体架构的全部价值。

## 概述：Codex Subagents

**Subagents** 是 OpenAI Codex 平台原生支持的并行任务执行机制。当你提交一个复杂的编程任务给 Codex 时，系统可以自动将其拆解为多个独立的子任务，每个子任务由一个 Subagent 在隔离的沙盒环境中执行。

Subagents 的核心价值在于**零配置的并行化**。你不需要手动定义任务边界或编排逻辑——Codex 的编排层自动处理任务分解、依赖分析和结果聚合。每个 Subagent 运行在独立的容器中，拥有自己的文件系统快照和执行环境，互不干扰。

这种模式特别适合大规模代码库的重构、多文件测试生成、以及跨模块的批量修改。根据 OpenAI 的文档，Subagents 可以显著缩短复杂任务的端到端完成时间，因为多个子任务可以真正并行执行而非串行排队。

## 概述：Codex Custom Agents

**Custom Agents** 是开发者通过 [Agent SDK](/zh/glossary/agent-sdk) 构建的自定义智能体，拥有完全可控的行为逻辑、工具集和交互模式。与 Subagents 的"开箱即用"不同，Custom Agents 需要你主动定义智能体的角色、可用工具、决策流程和输出格式。

Custom Agents 的杀手级特性是**工具链可编程**。你可以为一个 Custom Agent 配备特定的 MCP 工具、数据库连接、API 调用能力，甚至是自定义的代码执行环境。这意味着你的 Agent 可以做到 Subagents 做不到的事情——比如在代码生成后自动调用内部 CI 系统验证、或者从企业知识库中检索上下文信息。

对于正在构建[多智能体工作流](/zh/blog/con-u-pour-des-workflows-multi-agents)的工程团队来说，Custom Agents 是将 Codex 能力嵌入到现有开发流水线的关键接口。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **上手门槛** | 零配置，平台自动编排 | 需要定义 Agent 配置和工具链 | Subagents |
| **并行执行** | 原生支持，自动任务分解 | 需要手动编排或借助 SDK | Subagents |
| **工具自定义** | 受限于平台预设工具 | 完全可编程，支持 MCP 等协议 | Custom Agents |
| **执行环境** | 隔离沙盒，平台托管 | 可自定义沙盒配置和依赖 | Custom Agents |
| **行为控制** | 有限——依赖平台编排逻辑 | 完全可控——自定义决策流程 | Custom Agents |
| **适合团队规模** | 个人开发者到小团队 | 中大型工程团队 | 视情况而定 |
| **计费模型** | 按 token 使用量计费 | 按 token 使用量计费 | 平手 |

## 任务编排能力：深度对比

Subagents 和 Custom Agents 在任务编排层面的差异是决定选型的核心因素。

Subagents 的编排是**声明式**的。你告诉 Codex"重构这个模块并更新所有相关测试"，平台自动判断哪些文件需要修改、哪些测试需要重跑、哪些修改之间存在依赖关系。这种方式的优势在于简单——你不需要理解编排细节就能享受并行化带来的速度提升。但缺点同样明显：当平台的自动分解策略与你的预期不符时，你几乎没有调整的手段。

Custom Agents 的编排是**命令式**的。你通过 Agent SDK 显式定义每个 Agent 的职责边界、输入输出格式、以及 Agent 之间的通信协议。这给了你完全的控制力——比如你可以定义一个"架构评审 Agent"先分析代码结构，然后将分析结果传递给"实现 Agent"和"测试 Agent"并行执行。

从工程实践角度看，这类似于选择 PaaS 还是 IaaS 的决策。Subagents 像是托管服务——方便但不够灵活；Custom Agents 像是基础设施——强大但需要更多工程投入。参考 [Codex 完全指南](/zh/blog/codex-complete-guide) 了解两种模式的具体 API 用法。

## 工具与生态集成：深度对比

工具集成能力直接决定了 Agent 能"做什么"而不仅仅是"写什么代码"。

Subagents 使用 Codex 平台预置的工具集：文件读写、终端命令执行、代码搜索。这些工具足以覆盖大部分标准编程任务。但如果你需要调用企业内部 API、连接私有数据库、或者使用特定的构建工具链，Subagents 的能力就会遇到天花板。

Custom Agents 则可以通过 [Agent SDK](/zh/glossary/agent-sdk) 接入任意外部工具。你可以为 Agent 注册自定义 function tools，配置 MCP 服务器连接，甚至让 Agent 调用其他 Agent。这种可扩展性在企业场景中价值巨大——想象一个 Custom Agent 在生成代码后自动触发安全扫描、跑通集成测试、并将结果推送到 Slack 频道。

对比 Claude Code 的 Agent Teams 方案，Codex 的 Custom Agents 在工具链灵活性上更接近通用 Agent 框架的设计哲学，而 [Claude Code 的 Agent Teams](/zh/blog/claude-code-agent-teams) 则更专注于编程场景的开箱即用体验。

## 什么时候选 Subagents

Subagents 是以下场景的最优选择：

- **快速原型验证**：你有一个想法需要快速实现，不想花时间配置 Agent 架构。直接把需求甩给 Codex，让 Subagents 自动并行处理
- **大规模代码生成**：需要一次性生成或修改大量文件——比如为整个 API 层生成类型定义、或者批量迁移一个框架版本
- **个人开发者工作流**：你是一个人在干活，不需要复杂的团队协作流程，只需要一个能理解上下文并快速执行的 AI 编程助手
- **标准化任务**：测试生成、文档更新、代码格式化——这些不需要定制逻辑的重复性工作

如果你的需求不超出"写代码、改代码、跑测试"的范围，Subagents 的零配置特性就是最大的优势。参考 [Codex 初体验](/zh/blog/first-few-days-with-codex-cli) 了解 Subagents 的实际使用感受。

## 什么时候选 Custom Agents

Custom Agents 在以下场景中展现出不可替代的价值：

- **企业级 CI/CD 集成**：你需要 Agent 在代码生成后自动触发构建、运行测试套件、并根据结果决定下一步动作
- **多角色协作工作流**：不同 Agent 扮演不同角色——架构师 Agent、实现者 Agent、审查者 Agent——通过结构化消息传递协作
- **私有工具链接入**：你的开发环境中有自定义的 linter、内部包管理器、或者特定的部署工具，需要 Agent 能调用它们
- **合规与安全要求**：企业环境中可能需要对 Agent 的行为进行审计、限制某些操作、或者强制执行特定的代码审查流程

Custom Agents 的投入更大，但回报也更高。一个设计良好的 Custom Agent 工作流可以将重复性工程工作的效率提升一个量级。了解 [Agent SDK](/zh/glossary/agent-sdk) 的核心概念是上手 Custom Agents 的第一步。

## 结论

**如果你想快速上手、让 Codex 自动处理任务拆解和并行执行，选 Subagents。** 它是 Codex 多智能体能力的默认入口，零配置、低门槛，适合绝大多数标准编程任务。

**如果你需要深度定制 Agent 行为、接入企业工具链、或构建复杂的多 Agent 协作流程，选 Custom Agents。** 它的学习曲线更陡，但提供了 Subagents 无法实现的灵活性和控制力。

最务实的策略是**两者结合**：用 Subagents 处理日常编码任务保持开发节奏，用 Custom Agents 构建关键的自动化工作流提升团队效能。随着 Codex 生态的持续演进，掌握两种模式的工程实践将成为[敏捷编码](/zh/glossary/agentic-coding)团队的核心竞争力。

## 常见问题

### Subagents 和 Custom Agents 可以混合使用吗？

可以。在实际工程场景中，两者经常配合使用。比如你可以用一个 Custom Agent 来编排整体工作流——定义任务优先级和依赖关系——然后让具体的编码任务由 Subagents 并行执行。这种"Custom Agent 做大脑、Subagents 做手脚"的模式在大型项目中非常有效。

### Custom Agents 需要多少开发投入才能上手？

对于有 Agent SDK 经验的开发者，一个基础的 Custom Agent 工作流可以在几小时内搭建完成。核心工作量在于定义工具接口和编排逻辑。如果你的团队已经有完善的 CI/CD 流水线，将其接入 Custom Agent 通常只需要编写对应的 function tool 定义。

### Codex 的多智能体方案和 Claude Code Agent Teams 有什么区别？

Codex 的多智能体方案（Subagents + Custom Agents）更强调云端异步执行和工具链灵活性，适合需要与 CI/CD 深度集成的企业工作流。Claude Code Agent Teams 则更专注于终端内的实时编程协作，擅长在本地代码库上进行快速的多文件并行操作。选择取决于你的主要使用场景是异步自动化还是实时编码辅助。
