# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **CLAUDE.md** 是你手写的规则手册——告诉 Claude「怎么干活」；**Claude Memory**（Auto Memory）是 Claude 自己的笔记本——记住「发生过什么」。团队协作和项目规范用 CLAUDE.md，个人偏好和跨会话上下文用 Memory。大多数认真使用 Claude Code 的开发者两个都要用。

## 什么是 Claude Memory

Claude Memory 是 Claude Code 内置的自动记忆系统，存储在 `~/.claude/projects/` 目录下。当你在对话中提到个人偏好、项目背景或工作习惯时，Claude 会自动将这些信息写入本地的 markdown 文件，并在后续会话中读取。

Memory 的核心设计理念是**学习型持久化**。你不需要手动维护它——Claude 在对话过程中判断哪些信息值得跨会话保留，自动创建和更新记忆文件。这些记忆分为四类：用户画像（user）、行为反馈（feedback）、项目动态（project）和外部引用（reference）。

Anthropic 在 2026 年初对 [Claude 的记忆系统做了重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，免费用户也能使用，还支持从 ChatGPT 导入历史数据。但 Claude Code 中的 Auto Memory 和 Claude.ai 的 Memory 是两套独立实现——Claude Code 的版本更偏向工程场景，以文件系统为存储介质。

## 什么是 CLAUDE.md

CLAUDE.md 是放在项目根目录（或子目录）下的指令文件，相当于给 Claude Code 写的「员工手册」。它定义项目级别的规则：用什么编码规范、构建命令是什么、提交前必须跑哪些检查、哪些坑不能踩。

与 Memory 最大的区别在于：CLAUDE.md 是**声明式的、手动维护的、可版本控制的**。它跟着代码仓库走，`git push` 之后团队每个人的 Claude Code 都会读到同一份规则。这意味着 CLAUDE.md 本质上是一种团队级别的 AI 行为规范工具。

关于 CLAUDE.md 在 Claude Code 整体架构中的位置，可以参考我们的 [Claude Code 七层架构解析](/zh/blog/claude-code-seven-programmable-layers)——它是第一层「项目上下文」的核心组件。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **写入方式** | Claude 自动写入 | 开发者手动编写 | 各有所长 |
| **存储位置** | `~/.claude/projects/` | 项目根目录 | CLAUDE.md |
| **版本控制** | 不进 git | 进 git，团队共享 | CLAUDE.md |
| **内容类型** | 用户偏好、项目动态、反馈 | 规则、约束、命令、规范 | 各有所长 |
| **生效范围** | 当前用户 + 当前项目目录 | 所有使用该仓库的人 | CLAUDE.md |
| **维护成本** | 接近零 | 需要持续更新 | Memory |
| **确定性** | 模型判断，可能遗漏 | 显式声明，确定执行 | CLAUDE.md |
| **适合场景** | 个人工作流、上下文积累 | 团队规范、质量门禁 | 各有所长 |

## 核心差异深度分析：控制权与确定性

这两个系统最本质的区别在于**谁决定记什么、怎么用**。

CLAUDE.md 给你完全的控制权。你写下「提交前必须跑 `npm test` 和 `npm run build`」，Claude 每次都会照做。这是确定性行为——团队里不管谁用 Claude Code 打开这个项目，看到的规则一模一样。对于代码规范、安全约束、部署流程这类必须 100% 执行的事项，CLAUDE.md 是唯一正确的选择。

Memory 是 Claude 自己做的笔记，带有模型判断的不确定性。Claude 可能在你说「别在测试里 mock 数据库」之后记下这条反馈，也可能因为上下文太长而遗漏。记下来之后，它在未来的对话中被读取和应用的方式也取决于模型的理解。这不是缺陷，而是设计意图——Memory 处理的是那些「最好能记住，忘了也不会出大问题」的软性上下文。

实际工程中的对比场景：

- **「这个项目用 Vitest 不用 Jest」** → 写进 CLAUDE.md。这是硬性规则，不能靠模型记忆。
- **「我是后端出身，React 不太熟」** → 适合 Memory。Claude 会在解释前端概念时自动调整深度。
- **「PR 合并前必须跑 `validate-pipeline.ts`」** → CLAUDE.md。这是质量门禁。
- **「上次那个 bug 是因为 `upsertKeyword()` 少传了 clusterSlug」** → Memory 可以记，但修复方案应该体现在代码和 CLAUDE.md 的「已知坑」里。

更多关于这两套系统如何协作的细节，参考 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 组合使用：最佳实践

成熟的 Claude Code 工作流不是二选一，而是让两套系统各司其职。

**CLAUDE.md 负责「法律」**——不可违反的规则：

- 构建和测试命令
- 代码风格要求（缩进、命名、注释规范）
- 禁止事项（不要在 pipeline 脚本里 import Next.js 模块）
- 提交前的质量检查清单
- 已知陷阱和 workaround

**Memory 负责「经验」**——积累下来的软知识：

- 团队成员分工和角色
- 正在进行的项目背景
- 用户对 Claude 行为的反馈（「别在每次回复末尾总结」）
- 外部系统的指向（「bug 追踪在 Linear 的 INGEST 项目里」）

如果你正在搭建 Claude Code 的完整工作流，建议先从 CLAUDE.md 开始。把项目的核心规则写清楚，这能立刻提升 Claude 的输出质量。Memory 会在日常使用中自然积累，不需要刻意管理。

想了解 CLAUDE.md 之外的更多扩展机制（Skills、Hooks、MCP），参考 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 什么时候选 Claude Memory

- **个人项目**：规则不多，不需要团队共享，让 Memory 自动学习你的习惯就够了
- **新接手的项目**：在你还不确定该写什么规则的时候，先靠 Memory 积累上下文，之后再沉淀到 CLAUDE.md
- **跨项目的个人偏好**：比如你偏好 terse 风格的回复、喜欢用特定的 git commit 格式——这些跟项目无关，适合 Memory
- **快速迭代阶段**：需求变化快、规则还没定型，用 Memory 暂存灵活度更高

## 什么时候选 CLAUDE.md

- **团队协作**：多人共用一个仓库时，CLAUDE.md 是让所有人的 Claude Code 行为一致的唯一方式
- **关键规则**：测试必须过、lint 必须过、特定文件不能碰——这些不能靠模型记忆
- **新人上手**：新成员 clone 仓库后，CLAUDE.md 立刻生效，不需要「训练」Claude 一遍
- **合规和安全**：不允许跳过 pre-commit hooks、不允许 force push main——必须是显式规则

## 结论

**CLAUDE.md 是基础设施，Memory 是上层应用。** 如果你只能选一个，选 CLAUDE.md——它给你确定性和团队一致性。但如果你在认真使用 Claude Code 做日常开发，两个都用才能发挥最大价值：CLAUDE.md 确保底线，Memory 提供个性化。

这不是「记忆力好不好」的问题，而是「哪些知识该写成法律，哪些该当作经验」的工程判断。把这个判断做对了，Claude Code 就不只是一个会写代码的 AI，而是一个 [真正理解你项目上下文的工程伙伴](/zh/blog/claude-code-is-not-a-coding-tool)。

## 常见问题

### Claude Memory 的内容会上传到云端吗？

Claude Code 的 Auto Memory 以 markdown 文件形式存储在本地磁盘（`~/.claude/projects/` 目录下），不会自动上传到 Anthropic 的服务器。它在每次会话开始时被读入上下文窗口，仅在当次 API 调用中使用。

### CLAUDE.md 支持子目录级别的规则吗？

支持。Claude Code 会递归读取从项目根目录到当前工作目录路径上的所有 CLAUDE.md 文件。你可以在 `tests/CLAUDE.md` 里写测试专属规则，在 `scripts/CLAUDE.md` 里写脚本专属规则，它们会和根目录的规则合并生效。

### 可以手动编辑 Memory 文件吗？

可以。Memory 文件就是普通的 markdown，你可以直接编辑、删除或重新组织。如果发现 Claude 记住了过时或错误的信息，直接修改或删除对应文件即可。
