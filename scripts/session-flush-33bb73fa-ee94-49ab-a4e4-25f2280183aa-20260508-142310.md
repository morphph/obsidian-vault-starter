# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **CLAUDE.md** 是你写给 Claude Code 的"项目规则手册"，checked into 仓库，团队共享，每次会话自动加载。**Claude Memory**（Auto Memory）是 Claude Code 自己写给自己的"笔记本"，自动从对话中提取关键信息，存储在本地 `~/.claude/` 目录，跨会话持久化但不进入版本控制。两者不是竞争关系——**CLAUDE.md 管规则，Memory 管上下文**，搭配使用效果最好。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件，放在仓库根目录，作用类似于给 AI 写的"团队工程规范"。每次 Claude Code 启动会话时，会自动读取当前项目的 CLAUDE.md 和用户全局的 `~/.claude/CLAUDE.md`，将其中的规则注入到系统提示词中。

CLAUDE.md 的核心定位是**规则和约束**：代码风格、构建命令、质量门禁、禁止事项。它是确定性的——你写什么，Claude 就遵守什么。因为它 checked into Git，团队所有人用 Claude Code 时都会加载同一套规则，保证了一致性。

典型内容包括：构建和测试命令（`npm run build`、`npm test`）、代码规范（命名约定、commit message 格式）、架构约束（不要在 pipeline 脚本里 import Next.js 模块）、以及工作流规则（先讨论设计再写代码）。详细的 Claude Code 七层架构可参考我们的[深度解析文章](/zh/blog/claude-code-seven-programmable-layers)。

## 概述：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Claude Code 的自动记忆系统，它从你的对话中提取有价值的信息，以 Markdown 文件的形式保存在 `~/.claude/projects/<project>/memory/` 目录下。这些记忆跨会话持久化——下次你打开 Claude Code，它还记得你是谁、你在做什么、你之前纠正过它什么。

Memory 的核心定位是**上下文和理解**：用户角色、项目背景、反馈偏好、外部资源引用。它是演化的——随着你和 Claude Code 的互动不断积累和更新，不需要你手动维护。

Memory 分为四种类型：**user**（用户画像和偏好）、**feedback**（行为纠正和确认）、**project**（项目动态和决策）、**reference**（外部系统指针）。每条记忆都有 frontmatter 元数据，通过 `MEMORY.md` 索引文件管理。想了解完整的[记忆系统工作原理](/zh/blog/claude-code-memory)，可以阅读我们的专题文章。

## 功能对比

| 维度 | CLAUDE.md | Claude Memory | 说明 |
|------|-----------|---------------|------|
| **存储位置** | 仓库根目录（Git 管理） | `~/.claude/` 本地目录 | Memory 不进版本控制 |
| **创建方式** | 手动编写 | 自动提取 + 手动触发 | Memory 也支持"请记住这个" |
| **共享范围** | 团队共享（通过 Git） | 仅本地用户 | CLAUDE.md 是团队级，Memory 是个人级 |
| **内容类型** | 规则、约束、命令 | 上下文、偏好、背景 | 一个管"该怎么做"，一个管"我是谁" |
| **加载时机** | 每次会话自动加载 | 每次会话自动加载 | 两者都在会话启动时注入 |
| **更新频率** | 代码变更时手动更新 | 对话中自动演化 | Memory 更"活" |
| **层级结构** | 全局 + 项目 + 目录级 | 全局 + 项目级 | CLAUDE.md 支持更细粒度的目录级覆盖 |

## 定位差异：规则 vs 上下文

这是两者最本质的区别。

**CLAUDE.md 是声明式的规则引擎。** 你告诉 Claude Code："commit 前必须跑 `npm test`"、"不要用 `any` 类型"、"中文内容用 CJK 字数统计"。这些是硬性约束，每次会话都必须遵守，不会因为聊了几句就被覆盖。它类似于团队的 `.eslintrc` 或 `CONTRIBUTING.md`——是工程纪律的一部分。

**Memory 是归纳式的知识库。** Claude Code 从对话中学习："这个用户是数据科学家"、"他们上次说不要在测试里 mock 数据库"、"项目的 merge freeze 从 3 月 5 日开始"。这些是软性上下文，帮助 Claude Code 做出更贴合你的判断。它类似于一个同事的"备忘录"——记住你的偏好和项目背景，但不会强制执行。

实际效果的差异很明显：如果你在 CLAUDE.md 里写"永远不要删除测试文件"，Claude Code 就不会删。如果你只是在对话中说过"别删测试"，Memory 会记录这个反馈，但在边界情况下可能会权衡利弊做出不同判断。关于如何用好这套[扩展栈的分层设计](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)，我们有更详细的分析。

## 协作模式：怎么搭配用

理解了两者的定位，搭配使用的策略就很清晰了。

### 放进 CLAUDE.md 的内容

- 构建、测试、部署命令
- 代码风格和命名约定
- 架构边界和禁止事项
- Git 工作流规则
- 团队级的质量门禁

这些是**不变量**——不管谁来用 Claude Code，不管聊什么话题，都必须遵守。

### 交给 Memory 的内容

- 你的角色和技术背景
- 你对 Claude Code 行为的纠正（"别在回复结尾总结"）
- 项目当前阶段和近期目标
- 外部资源的位置（Linear 看板、Grafana 仪表盘）
- 你验证过的有效做法

这些是**上下文**——帮助 Claude Code 理解你和你的项目，但不需要强制团队其他人遵守。

### 不该放在任何一个里的内容

代码结构、文件路径、架构细节——这些应该通过读代码获取，而不是写进静态文档。Memory 和 CLAUDE.md 都会随时间变得过时，但代码本身是 source of truth。

## 实际场景对比

**场景一：新团队成员加入**

CLAUDE.md 起到了"入职手册"的作用——新人的 Claude Code 会话自动加载团队规范，不需要任何额外配置。但 Memory 从零开始，Claude Code 还不了解这个人的背景和偏好，需要几次对话才能建立起有用的记忆。

**场景二：长期个人项目**

CLAUDE.md 可能很简短（几个命令就够了），但 Memory 会变得非常丰富——你的技术偏好、之前踩过的坑、项目演化的决策历史。这时候 Memory 的价值远大于 CLAUDE.md。

**场景三：跨项目一致性**

全局 `~/.claude/CLAUDE.md` 和全局 Memory 都能跨项目生效。但 CLAUDE.md 更适合放通用规则（"commit message 要描述改了什么"），Memory 更适合放个人偏好（"我习惯用 Vim 键位"）。详细的[提示技巧](/zh/blog/how-to-effectively-prompt-a-claude-code)也能帮助你更高效地利用这两套系统。

## 常见误区

**误区一："有了 Memory 就不需要 CLAUDE.md 了"**

错。Memory 是个人的、软性的、可能过时的。CLAUDE.md 是团队的、硬性的、版本控制的。两者解决不同层面的问题。一个团队如果只靠 Memory 而没有 CLAUDE.md，每个人的 Claude Code 会话行为会完全不一致。

**误区二："把所有东西都塞进 CLAUDE.md"**

也不对。CLAUDE.md 应该精简，只放规则和约束。如果你发现自己在 CLAUDE.md 里写"用户是高级工程师"或"上次我们决定用 PostgreSQL 替换 SQLite"，这些属于 Memory 的领域。CLAUDE.md 膨胀会消耗宝贵的上下文窗口。

**误区三："Memory 会自动记住所有重要的事"**

不会。Memory 只记录 Claude Code 判断为"对未来会话有价值"的信息。临时的调试细节、一次性的实现方案不会被保存。如果有重要信息想确保被记住，明确说"请记住这个"。

## 结论

**CLAUDE.md 和 Claude Memory 不是二选一的关系，而是互补的双层架构。** CLAUDE.md 是团队共享的规则层——确定性强、版本可控、适合工程约束。Memory 是个人专属的上下文层——自动演化、跨会话持久、适合偏好和背景信息。

最佳实践很简单：**团队规则放 CLAUDE.md，个人上下文交给 Memory。** 如果你不确定某条信息该放哪里，问自己一个问题：这是每个团队成员都必须遵守的规则，还是只对我个人有用的上下文？答案自然就出来了。

## 常见问题

### CLAUDE.md 和 Memory 的内容冲突了怎么办？

CLAUDE.md 的优先级更高。它作为系统级指令注入，Claude Code 会将其视为硬性约束。如果 Memory 中记录的偏好与 CLAUDE.md 的规则矛盾，Claude Code 会遵循 CLAUDE.md。这也是为什么团队规则应该放在 CLAUDE.md 而不是依赖个人 Memory 的原因。

### Memory 文件可以手动编辑吗？

可以。Memory 文件就是普通的 Markdown 文件，存储在 `~/.claude/projects/<project>/memory/` 目录下。你可以直接编辑、删除或新建记忆文件。但要注意同步更新 `MEMORY.md` 索引文件，否则 Claude Code 可能找不到你手动添加的记忆。

### 团队能不能共享 Memory？

目前 Memory 设计为个人本地存储，不走 Git。如果团队想共享某些上下文信息，应该把它写进 CLAUDE.md 或项目文档。将个人 Memory 文件提交到仓库不是推荐做法——它包含个人偏好和反馈，对其他团队成员可能不适用甚至造成干扰。
