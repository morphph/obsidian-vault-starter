**CLAUDE.md loads first and forms the base layer.** When you start a Claude Code session, the system reads your global `~/.claude/CLAUDE.md`, then your project-level `CLAUDE.md`, then any subdirectory-level CLAUDE.md files relevant to your current working context. These are concatenated and injected as system instructions. Everything in CLAUDE.md functions as a hard rule — Claude treats these as constraints it must follow, not suggestions it might consider. If your CLAUDE.md says "never use `any` types in TypeScript," Claude will follow that instruction with the same reliability as its core safety guidelines.

**Claude Memory loads alongside CLAUDE.md but serves a different function.** The `MEMORY.md` index loads automatically, and Claude accesses individual memory files when they seem relevant to the current task. Memory doesn't override CLAUDE.md — it supplements it. If your CLAUDE.md says "use Vitest for testing" and your memory says "this user prefers verbose test descriptions," both apply. But if they conflict, CLAUDE.md wins because it's treated as an authoritative project instruction.

**The practical implication:** put rules in CLAUDE.md, put preferences in memory. A rule like "run `npm test` before every commit" belongs in CLAUDE.md because it applies to everyone and must be enforced deterministically. A preference like "this developer likes brief summaries instead of detailed explanations" belongs in memory because it's personal and adaptive.

This layered architecture means you can have a team of ten developers all sharing the same CLAUDE.md (consistent project behavior) while each developer's Claude Code instance remembers their individual communication style, areas of expertise, and workflow quirks.

## Authoring and Maintenance: Detailed Analysis

The authoring model is the most significant practical difference between the two systems, and it determines which one you should invest time in for different scenarios.

**CLAUDE.md requires intentional, upfront effort.** You write it like documentation — thinking through build commands, coding standards, known pitfalls, and workflow expectations before Claude ever sees them. A good CLAUDE.md typically includes: what the project is, how to build and test it, what style conventions to follow, what common mistakes to avoid, and what tools are in the stack. This is work, but it's one-time work that pays dividends across every conversation, every team member, and every CI run. The [skills system](/blog/5-claude-code-skills-i-use-every-single-day) extends this concept with task-specific instruction files.

**Claude Memory requires no upfront effort but accumulates over time.** The first conversation with a new project produces minimal memory. By the tenth conversation, Claude has built a rich picture of how you work — what tools you prefer, what patterns you've corrected, what your role involves. The tradeoff: memory is only as good as the conversations that produced it. If you spent three sessions debugging a Redis issue, Claude may over-index on Redis-related context in future conversations.

**Maintenance patterns differ accordingly.** CLAUDE.md is updated deliberately — when you add a new build step, change a linting rule, or discover a new gotcha, you edit the file. It's as reliable as any other checked-in documentation. Memory files, in contrast, can become stale. A project-type memory noting "we're in the middle of migrating to PostgreSQL" stops being useful once the migration is complete. Claude Code has built-in guardrails — it's instructed to verify memory-referenced files and functions still exist before acting on them — but stale memories can still waste context window space.

**A practical maintenance rule:** audit your CLAUDE.md when you'd audit a README (major releases, team changes, stack updates). Audit your memory files when something feels off — if Claude keeps bringing up a finished migration or a resolved bug, check `~/.claude/projects/*/memory/` for stale entries.

## Team Collaboration and Sharing

This dimension alone determines which system to prioritize for many organizations.

**CLAUDE.md is inherently collaborative.** Because it lives in the git repository, it goes through the same review process as code — pull requests, code review, merge approvals. When a senior engineer adds a gotcha to CLAUDE.md ("don't use `SELECT *` against the analytics table — it has 200 columns and will time out"), every team member's Claude Code instance immediately learns that constraint. New hires get the accumulated wisdom of the team from their first `git clone`. This is powerful for enforcing consistency: every AI-assisted coding session on the project follows the same rules.

**Claude Memory is inherently personal.** Your memories exist only on your machine (or, for Claude.ai memory, in your account). There's no mechanism to sync memories across team members, and for good reason — memories include personal preferences, role-specific context, and feedback that only makes sense for one person. A senior engineer's memory might include "this user prefers terse responses and understands complex type signatures without explanation." A junior developer's memory might include "this user is learning TypeScript and benefits from explanations of generic syntax." Both are correct and useful — for their respective users.

**The gap between these systems is intentional.** Shared instructions belong in CLAUDE.md. Personal context belongs in memory. When teams try to use one system for both purposes — stuffing personal preferences into CLAUDE.md or expecting memory to enforce team standards — the result is friction. A bloated CLAUDE.md full of individual preferences creates noise. A memory system expected to enforce linting rules will fail when a new team member joins with no memories.

## When to Choose Claude Memory

Claude Memory is the right system when the context is **personal, adaptive, or conversation-derived:**

- **Personal workflow preferences**: How verbose you want explanations, whether you prefer step-by-step or high-level summaries, your terminal setup quirks
- **Role and expertise context**: Your specialization (backend vs. frontend, data science vs. infrastructure), your experience level with the current stack, which parts of the codebase you own
- **Feedback accumulation**: Corrections you've made ("don't mock the database in integration tests"), approaches you've validated ("single bundled PRs work well for refactors in this area"), patterns you've rejected
- **External resource pointers**: Which Slack channel has deployment alerts, which Linear project tracks the current sprint, where the Grafana dashboard lives
- **Ongoing project context**: Current sprint goals, active incidents, upcoming deadlines — ephemeral enough that it shouldn't be in CLAUDE.md but persistent enough that Claude should know about it

Memory shines for solo developers and small teams where personal context has high leverage. If you're the only developer on a project, Claude Memory effectively becomes your project memory too — and the lack of team sharing is irrelevant.

## When to Choose CLAUDE.md

CLAUDE.md is the right system when the context is **shared, deterministic, or rule-based:**

- **Build and test commands**: `npm run build`, `npm test`, `cargo clippy` — every developer and CI agent needs these
- **Coding standards**: TypeScript strict mode, no `any` types, prefer `const` over `let`, import ordering rules
- **Architecture constraints**: "Don't import Next.js modules in pipeline scripts," "use the `db.ts` wrapper instead of raw SQLite calls"
- **Known gotchas**: "Chinese content must use CJK word count," "the analytics table has 200 columns — never `SELECT *`"
- **Workflow rules**: "Run tests before committing," "get human approval before starting new features," "always pull before starting work"
- **Tool and dependency rules**: Required Python versions, SDK constraints, which models to use for which tasks

CLAUDE.md is essential for any team larger than one person. Even for solo developers, it provides value by making instructions explicit and reproducible — if you set up a new machine or switch to a CI environment, CLAUDE.md carries over. Memory does not.

Teams building [skills](/blog/9-principles-writing-claude-code-skills) and [hooks](/blog/claude-code-hooks-a-complete-guide-to-automating-your-ai-coding-workflow) alongside CLAUDE.md get the most mileage: CLAUDE.md defines what rules to follow, skills define how to approach specific tasks, and hooks automate enforcement. Memory then fills in the personal layer that makes Claude feel like it knows you rather than just your project.

## Using Both Together: The Real Answer

For most developers and teams, the question isn't which system to choose — it's what goes where. Here's a decision framework:

**Put it in CLAUDE.md if:**
- Another developer would need this information
- It's a rule that should be enforced consistently
- It won't change based on who's asking
- It describes the project, not the person

**Put it in Memory if:**
- It's specific to one person's workflow or expertise
- Claude learned it from a conversation naturally
- It describes how to communicate with or help a specific user
- It's ephemeral enough that it shouldn't be version-controlled

**Concrete examples of the boundary:**

| Information | Where it belongs | Why |
|-------------|-----------------|-----|
| "Use Vitest, not Jest" | CLAUDE.md | Team-wide standard |
| "I prefer seeing test output inline" | Memory | Personal preference |
| "Run `npm run build` before committing" | CLAUDE.md | Enforced quality gate |
| "This user is a data scientist new to TypeScript" | Memory | Personal context |
| "Never use `any` in TypeScript" | CLAUDE.md | Coding standard |
| "Don't summarize after every response — I can read diffs" | Memory | Communication preference |
| "Pipeline cron: 4am collect, 5am newsletter" | CLAUDE.md | Shared operational context |
| "Sprint goal: finish auth migration by Friday" | Memory | Ephemeral project state |

The anti-pattern to avoid: duplicating information across both systems. If a rule is in CLAUDE.md, Claude doesn't also need a memory about it. If Claude learned something from a conversation that's actually a team-wide rule, promote it to CLAUDE.md and let the memory entry expire naturally.

## Verdict

**For team projects, invest in CLAUDE.md first** — it's the single highest-leverage file in your repository for AI-assisted development. A well-crafted CLAUDE.md eliminates entire categories of AI mistakes and ensures consistency across every developer's sessions. Claude Memory then develops naturally on top of this foundation, personalizing the experience without undermining shared standards.

**For solo projects, both systems matter equally**, but you'll find CLAUDE.md easier to maintain intentionally while memory builds itself. Start with a CLAUDE.md covering build commands, stack details, and known gotchas. Let memory handle everything else.

**The clearest decision rule: if you'd put it in a README, put it in CLAUDE.md. ## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md is treated as authoritative project instruction — it functions as part of the system prompt and overrides default Claude behavior. Memory supplements the conversation context but does not override explicit CLAUDE.md rules. If a memory preference conflicts with a CLAUDE.md rule, the CLAUDE.md rule wins.

### Does Claude Memory sync across devices or team members?

Claude Code's auto-memory is stored locally in `~/.claude/projects/` and does not sync. Each developer builds their own memory through their own conversations. Claude.ai's memory feature syncs within your account but similarly cannot be shared with other users. For shared context, use CLAUDE.md.

### How do I migrate useful memories into CLAUDE.md?

Review your memory files at `~/.claude/projects/<project>/memory/`. Any `feedback`-type memory that applies team-wide (not just to your personal preferences) is a candidate for promotion. Copy the rule into your CLAUDE.md, then delete the memory file and its entry in MEMORY.md to avoid duplication.

### Do I need CLAUDE.md if I'm a solo developer?

Yes, for two reasons. First, CLAUDE.md is explicit and reproducible — it works identically on a new machine, in CI, or after you clear your local state. Memory is local and can be lost. Second, writing instructions forces clarity: you'll catch ambiguous rules and missing context that memory might paper over.

### How large should CLAUDE.md be?

There's no hard limit, but practical CLAUDE.md files range from 20 lines (small personal projects) to 200+ lines (complex team codebases). The constraint is the context window — every line of CLAUDE.md consumes tokens at the start of every conversation. Focus on rules Claude actually needs: build commands, coding standards, known gotchas, and workflow constraints. Remove anything Claude can derive from the code itself.
