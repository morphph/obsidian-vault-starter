# Claude Memory vs CLAUDE.md：两套记忆系统到底该用哪个？

**TL;DR：** Claude Memory 是 Anthropic 内置的**云端自动记忆**，跨对话保留个人偏好和事实；CLAUDE.md 是 Claude Code 的**项目级指令文件**，版本控制、团队共享、确定性执行。如果你在 claude.ai 上聊天，Memory 帮你省去重复自我介绍；如果你用 Claude Code 写代码，CLAUDE.md 才是真正控制 Agent 行为的手段。两者不互斥——一个管「你是谁」，一个管「这个项目怎么做」。

## 概览：Claude Memory

**Claude Memory** 是 Anthropic 在 Claude 对话产品中内置的持久化记忆系统，自动从对话中提取用户偏好、背景信息和重要事实，并在后续对话中调用。2026 年初，Anthropic 对 Memory 进行了重大升级：[免费用户全面开放，并支持从 ChatGPT 导入数据](/zh/blog/anthropic-claude-memory-upgrades-importing)。

Memory 的核心设计思路是**零配置**——你不需要手动维护任何文件，Claude 会自动判断哪些信息值得记住。它存储在 Anthropic 的云端，与你的账号绑定，跨设备、跨对话生效。典型用途包括：记住你的编程语言偏好、常用技术栈、沟通风格、项目背景等。

这套系统对非技术用户和日常对话场景非常友好，但它本质上是**个人的、隐式的、不可审计的**——你无法精确控制 Claude 记住了什么、会在何时调用。

## 概览：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级配置文件，以 Markdown 格式编写，放在项目根目录下。它不是「记忆」，而是**指令手册**——告诉 Claude Code 这个项目的编码规范、构建命令、架构约束和工作流程。

CLAUDE.md 的核心优势是**确定性和可控性**。它是一个普通的文本文件，跟代码一起提交到 Git，团队所有成员共享同一份指令。每次 Claude Code 启动会话时都会读取它，确保行为一致。正如我们在 [Claude Code 七层架构解析](/zh/blog/claude-code-seven-programmable-layers)中详细分析的，CLAUDE.md 处于整个系统的第二层——项目上下文层，直接影响 Agent 的每一个决策。

配合 [SKILL.md 技能文件](/zh/blog/9-principles-writing-claude-code-skills)使用时，CLAUDE.md 构成了一套完整的行为编程系统：CLAUDE.md 定义全局规则，SKILL.md 定义任务级指令。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 胜出方 |
|------|--------------|-----------|--------|
| **存储位置** | Anthropic 云端 | 项目仓库（Git 版本控制） | CLAUDE.md |
| **作用范围** | 个人账号级，跨所有对话 | 项目级，仅在该项目目录生效 | 各有优势 |
| **维护方式** | 自动提取，用户可手动编辑 | 手动编写，团队协作维护 | Memory |
| **可审计性** | 低——无法精确预测调用行为 | 高——纯文本，Git diff 可追踪 | CLAUDE.md |
| **团队共享** | 不支持，绑定个人账号 | 天然支持，提交到 Git 即共享 | CLAUDE.md |
| **适用产品** | claude.ai / Claude Desktop | Claude Code（终端 Agent） | 各有所属 |
| **数据导入** | 支持从 ChatGPT 导入 | 手动编写或由 /init 生成 | Memory |
| **确定性** | 概率性——Claude 自行决定何时调用 | 确定性——每次会话必定加载 | CLAUDE.md |

## 记忆机制：自动 vs 显式

这是两者最根本的差异。Claude Memory 采用**隐式提取**策略：Claude 在对话过程中自动判断哪些信息值得长期保存，用户说「我偏好 TypeScript」，它会静默记录下来。这种方式的优点是零摩擦——你不需要做任何额外工作；缺点是不透明——你不确定它到底记住了什么、会在什么时候突然「想起来」。

CLAUDE.md 则是**显式声明**：你写什么，Claude Code 就执行什么。构建命令是 `npm run build`，代码风格要求是 Tailwind v4，提交前必须跑测试——这些规则以白纸黑字的形式写在文件里，没有任何模糊空间。在 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)一文中，我们深入分析过这套双层架构：CLAUDE.md 是规则手册（确定性），Auto Memory 是自动笔记（概率性），两者在 Claude Code 内部协同工作。

对于个人日常使用，Memory 的自动化体验更好。但对于工程团队，CLAUDE.md 的确定性才是生产环境的基本要求——你不会希望 AI Agent 因为「记忆」不一致而在不同开发者的机器上产生不同的行为。

## 协作与版本控制

团队协作场景下，两者的差距最为明显。

CLAUDE.md 本质上就是一个 Git 管理的配置文件。新成员加入团队，clone 仓库就自动获得了完整的 AI 工作指令——不需要任何额外的 onboarding。修改了编码规范？提交一个 PR，团队 review 后合并，所有人的 Claude Code 立刻同步。这种模式和 `.eslintrc`、`tsconfig.json` 完全一致，工程师们早已习惯。

Claude Memory 绑定个人账号，团队成员之间无法共享。如果 Alice 的 Memory 里记住了「这个项目用 Vitest 不用 Jest」，Bob 的 Claude 并不知道。更棘手的是，Memory 的内容不可 diff——你无法 review 其他人的 AI 记忆，也无法确保团队行为一致。

实际上，很多使用 Claude Code 的[企业工程团队](/zh/blog/claude-code-enterprise-engineering-ramp-shopify-spotify)都是以 CLAUDE.md 作为 AI 工程规范的核心载体，配合 SKILL.md 文件形成标准化的 Agent 行为控制体系。

## 适用场景分析

### 什么时候用 Claude Memory

- 你主要在 claude.ai 或 Claude Desktop 上进行日常对话
- 你希望 Claude 记住个人偏好（语言风格、常用工具、背景信息）
- 你从 ChatGPT 迁移过来，想导入已有的对话记忆
- 使用场景偏个人、偏探索性，不涉及团队协作

### 什么时候用 CLAUDE.md

- 你在用 Claude Code 进行[工程开发](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)
- 项目有明确的编码规范、构建流程和质量门禁
- 需要团队所有成员的 AI Agent 行为保持一致
- 需要可审计、可 review、可回滚的 AI 配置
- 你希望精确控制 Agent 的每一条行为规则

### 两者结合使用

最常见的模式是**两者并行**。在 Claude Code 中，CLAUDE.md 管项目规则，Auto Memory 管个人偏好（比如你习惯简洁的输出风格、或者你负责后端模块而不是前端）。在 claude.ai 对话中，Memory 管持久化上下文。它们各司其职，互不冲突。

## 结论

**选择取决于你的使用场景，而非二选一。** 如果你是个人用户在 claude.ai 上聊天，Claude Memory 是开箱即用的最佳体验。如果你是工程团队用 Claude Code 写代码，**CLAUDE.md 是不可替代的核心配置**——它提供了 Memory 无法给你的确定性、可审计性和团队共享能力。最成熟的做法是两者结合：用 CLAUDE.md 管项目级的硬性规则，用 Memory 管个人级的软性偏好。想深入了解这套体系的实际配置方法，推荐阅读 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 常见问题

### Claude Memory 和 CLAUDE.md 能同时使用吗？

可以，两者在 Claude Code 中本就是共存的。CLAUDE.md 每次会话必定加载，提供项目级的确定性规则；Auto Memory 在后台补充个人偏好。两者作用于不同层级，不会冲突。在 claude.ai 网页端，只有 Memory 生效，CLAUDE.md 不参与。

### CLAUDE.md 需要手动编写吗？

可以手动编写，也可以用 Claude Code 的 `/init` 命令自动生成初始版本。实践中建议先自动生成，然后根据项目实际需求手动调整和补充。团队应将 CLAUDE.md 视为和 linter 配置同等重要的工程资产，定期 review 和更新。

### Claude Memory 存储的内容安全吗？

Memory 数据存储在 Anthropic 的云端服务器上，与你的账号绑定。你可以在设置中查看、编辑和删除已保存的记忆。但需要注意，Memory 的内容会被用于后续对话——如果你在对话中提到了敏感信息并被 Memory 捕获，建议手动检查并删除。
