# Codex Subagents vs Custom Agents：哪种多智能体方案更适合你？

**TL;DR：** **Subagents** 是 Codex 内置的轻量级任务分发机制，适合将大任务拆解成并行子任务快速执行；**Custom Agents** 则允许你定义独立的 Agent 身份、工具集和行为规则，适合构建可复用的专业化工作流。选型的关键不在"哪个更强"，而在你的任务是**一次性分解**还是**反复调用的专业角色**。

## 概述：Codex Subagents

**Subagents** 是 [Codex](/zh/blog/codex-complete-guide) 平台中内置的多任务并行执行机制。当一个主 Agent 接收到复杂任务时，它可以将任务拆分为多个子任务，并启动若干 Subagent 并行处理。每个 Subagent 继承主 Agent 的上下文和权限，在独立的沙箱环境中运行，完成后将结果汇报给主 Agent 进行汇总。

Subagents 的核心价值在于**速度**。对于代码审查、多文件重构、批量测试生成这类天然可并行的任务，Subagents 能显著缩短端到端执行时间。你不需要额外配置——Codex 的编排层会自动处理任务分配、结果收集和冲突检测。

这种模式特别适合工程团队日常的大规模代码操作场景，尤其是在 monorepo 中同时修改多个模块时效果最为明显。

## 概述：Codex Custom Agents

**Custom Agents** 是一种更重量级的方案：你定义一个拥有独立身份、专属工具集、自定义系统提示词的 Agent 角色。Custom Agent 不仅仅是"执行子任务的工人"，而是一个可以反复调用的**专业化角色**——比如"安全审计 Agent"、"文档生成 Agent"、"PR 审查 Agent"等。

Custom Agents 通过 [Agent SDK](/zh/glossary/agent-sdk) 构建，可以绑定特定的 MCP 工具、设定行为边界、定义输出格式。与 Subagents 不同，Custom Agents 拥有独立的配置文件，可以跨项目、跨团队复用。

这种方案的核心价值在于**一致性和可复用性**。当你需要确保某类任务始终按照特定标准执行时——例如每次 PR 都必须经过安全检查、每次发布都必须生成 changelog——Custom Agents 是更好的选择。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **启动方式** | 主 Agent 自动拆分派发 | 显式定义和调用 | Subagents |
| **配置复杂度** | 零配置，开箱即用 | 需要定义身份、工具集、行为规则 | Subagents |
| **上下文隔离** | 继承主 Agent 上下文 | 拥有独立的系统提示词和上下文 | Custom Agents |
| **工具绑定** | 共享主 Agent 的工具 | 可绑定专属工具集 | Custom Agents |
| **跨项目复用** | 不支持 | 支持，可作为模板分发 | Custom Agents |
| **并行执行** | 原生支持 | 需要编排层协调 | Subagents |
| **行为可控性** | 有限，依赖主 Agent 指令 | 高度可控，规则写在配置中 | Custom Agents |
| **适用任务规模** | 单次大任务的子任务分解 | 反复出现的专业化任务 | 平手 |

## 任务分解与并行执行：详细分析

Subagents 在任务并行执行上有天然优势。以一个典型场景为例：你需要对一个包含 20 个模块的 monorepo 进行依赖升级。使用 Subagents，主 Agent 可以同时启动多个子 Agent，每个负责一个模块的升级、测试和验证，最后汇总结果。整个过程不需要你写任何编排逻辑。

这种模式与 [Claude Code Agent Teams](/zh/blog/claude-code-agent-teams) 的思路类似——都是通过并行执行来压缩大任务的完成时间。区别在于 Codex 的 Subagents 运行在云端沙箱中，而 Claude Code 的 Agent Teams 运行在本地终端环境。

但 Subagents 的局限也很明显：每个 Subagent 本质上是主 Agent 的"分身"，共享同一套工具和行为模式。如果你需要不同子任务使用不同的工具集或遵循不同的规则——比如代码修改用一套规则、安全检查用另一套——Subagents 就力不从心了。

Custom Agents 在这个维度上更灵活。你可以定义一个"代码重构 Agent"只关注代码质量指标，同时定义一个"安全扫描 Agent"专注于漏洞检测，两者各司其职。虽然编排复杂度更高，但任务执行的精确度和可审计性也更强。

## 可复用性与团队协作：详细分析

Custom Agents 的杀手锏是**可复用性**。一旦你定义好一个 Custom Agent——比如一个遵循团队代码规范的"PR 审查 Agent"——它可以被团队的每个成员在每个项目中直接调用。这与 [Claude Code Skills](/zh/blog/9-principles-writing-claude-code-skills) 的理念一脉相承：将最佳实践编码为可复用的配置，而不是依赖每个人的 prompt 技巧。

在团队规模较大的场景中，Custom Agents 的价值尤为突出。想象一个 30 人的工程团队，每个人都需要进行 PR Review。如果依赖 Subagents，每次审查的质量取决于主 Agent 收到的指令质量，而指令因人而异。使用 Custom Agent，审查标准被写入配置文件，无论谁触发，执行标准一致。

不过，Custom Agents 的维护成本也更高。每次团队规范变更，你都需要更新 Custom Agent 的配置。而 Subagents 因为没有独立配置，反而避免了这个问题——你只需要更新主 Agent 的指令即可。

对于个人开发者或小团队，Subagents 的零配置优势通常更有吸引力。对于中大型团队追求工程标准化，Custom Agents 的投入才值得回报。更多关于[多智能体工作流的工程范式](/zh/blog/con-u-pour-des-workflows-multi-agents)，可以参考我们此前的深度分析。

## 什么时候选 Subagents

选择 Subagents 的典型场景：

- **一次性大规模重构**：跨多个文件或模块的批量修改，天然可并行，完成即丢弃
- **探索性任务**：不确定最终方案时，让多个 Subagent 同时尝试不同思路，对比结果
- **快速原型验证**：需要尽快跑通一个 proof of concept，不想花时间配置 Agent 角色
- **个人开发者日常工作**：任务类型多变，不值得为每类任务定义专门的 Agent

Subagents 的最大优势是**即用即走**。你不需要前期投入任何配置时间，Codex 的编排层处理所有并行逻辑。对于 [agentic coding](/zh/glossary/agentic-coding) 的入门者来说，这是最低摩擦的起步方式。

## 什么时候选 Custom Agents

选择 Custom Agents 的典型场景：

- **标准化的重复任务**：每次 PR 都要做的安全检查、每次发布都要生成的 changelog、每个新模块都要创建的脚手架
- **团队工程规范落地**：将代码审查标准、测试覆盖要求、文档格式规范编码为 Agent 配置
- **专业化工具链集成**：需要特定 Agent 绑定特定的 MCP 工具，比如安全 Agent 绑定漏洞扫描工具
- **跨项目一致性**：同一个 Custom Agent 在不同项目中保持相同的行为标准

Custom Agents 的前期投入更大，但对于[长时间运行的 Agent 工作流](/zh/blog/effective-harnesses-for-long-running-agents)来说，一致性和可审计性带来的收益远超配置成本。

## 结论

**如果你的任务是"把这个大活儿拆开快速干完"，用 Subagents。如果你的需求是"确保这类任务每次都按标准执行"，用 Custom Agents。** 两者并不互斥——成熟的团队通常两者并用：Custom Agents 定义专业化角色处理标准化流程，Subagents 负责这些角色内部的并行加速。

选型决策树很简单：这个任务你只做一次还是反复做？做一次选 Subagents，反复做选 Custom Agents。如果你正在为团队搭建 [Codex 工程化方案](/zh/blog/codex-complete-guide)，建议从 Subagents 起步快速验证价值，再逐步将高频任务封装为 Custom Agents。

## 常见问题

### Subagents 和 Custom Agents 可以混合使用吗？

可以。一个 Custom Agent 在执行任务时可以内部启动 Subagents 进行并行处理。比如一个"PR 审查 Custom Agent"可以同时派发多个 Subagent 分别检查代码风格、安全漏洞和测试覆盖率，最后汇总成一份审查报告。两种机制在架构上是互补的。

### Custom Agents 的配置复杂度高吗？

取决于你的需求。最简单的 Custom Agent 只需要一个系统提示词和工具列表，几分钟就能配好。复杂的 Custom Agent 可能涉及多步工作流编排、条件分支和输出格式约束，需要更多调试时间。建议从简单配置起步，根据实际使用反馈迭代优化。

### Subagents 的并行数量有限制吗？

Codex 平台对同时运行的 Subagent 数量有并发限制，具体取决于你的账户等级和 API 配额。对于大多数工程任务，默认并发数已经足够。如果你需要同时处理大量子任务，建议分批执行或联系 OpenAI 了解企业级配额方案。

### 哪种方案的成本更低？

Subagents 按 token 消耗计费，多个 Subagent 并行时 token 消耗是累加的。Custom Agents 的 token 消耗取决于任务复杂度和上下文长度。一般来说，对于同等任务，Custom Agents 因为有更精确的系统提示词，往往能减少不必要的 token 消耗，长期使用下成本效率更高。
