# Codex Subagents vs Custom Agents：哪种多智能体方案更适合你？

**TL;DR:** **Subagents** 是 Codex 内置的轻量级任务分发机制，适合将大任务拆分成独立子任务并行执行；**Custom Agents** 是你自己定义的专用智能体，拥有独立的 system prompt、工具集和行为约束，适合需要领域专业化的复杂工作流。如果你只需要并行加速，用 Subagents；如果你需要不同角色各司其职，用 Custom Agents。两者可以组合使用。

## 概述：Codex Subagents

Codex Subagents 是 OpenAI Codex 平台内置的子智能体机制。当主 Agent 遇到可以并行处理的独立任务时——比如同时修改多个文件、在不同模块中执行测试、或者同步进行代码审查和重构——它可以生成 Subagent 来并行执行这些任务。

Subagents 的核心价值在于**效率**。每个 Subagent 继承主 Agent 的上下文环境，但在独立的沙箱中运行，互不干扰。任务完成后，结果汇总回主 Agent。这种模式不需要额外配置，Codex 原生支持，开箱即用。

对于日常开发中的多文件重构、批量测试生成、或跨模块的代码变更，Subagents 是最直接的选择。详细的工程实践可以参考我们的 [Codex 完全指南](/zh/blog/codex-complete-guide)。

## 概述：Codex Custom Agents

Custom Agents 是你在 Codex 生态中自定义的专用智能体。每个 Custom Agent 拥有独立的 system prompt、专属工具集、特定的行为约束和输出格式。你可以把它理解为一个「角色」——安全审计员、测试工程师、文档撰写者——每个角色有自己的专业知识和工作方式。

Custom Agents 的核心价值在于**专业化**。通过 [Agent SDK](/zh/glossary/agent-sdk) 定义的 Custom Agent 可以封装团队的领域知识：编码规范、测试策略、安全检查清单。这些知识不会随着对话结束而丢失，而是固化在 Agent 定义中，团队所有成员共享同一套标准。

构建 Custom Agents 需要前期投入——定义 prompt、选择工具、设计交互流程。但对于需要反复执行的专业化任务，这个投入的回报是持续的。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **配置成本** | 零配置，原生支持 | 需要定义 prompt、工具集、约束 | Subagents |
| **并行能力** | 原生并行，自动分发 | 需要编排层协调 | Subagents |
| **专业化程度** | 继承主 Agent 能力，无定制 | 完全定制，领域专家级 | Custom Agents |
| **上下文隔离** | 沙箱隔离，共享基础上下文 | 独立上下文，精确控制 | Custom Agents |
| **可复用性** | 一次性，任务完成即销毁 | 持久定义，跨会话复用 | Custom Agents |
| **团队协作** | 个人效率工具 | 团队标准化工具 | Custom Agents |
| **学习曲线** | 几乎为零 | 需要理解 Agent SDK | Subagents |
| **适用规模** | 单任务拆分 | 多角色工作流 | Custom Agents |

## 架构差异：深度分析

Subagents 和 Custom Agents 在架构层面有本质区别，理解这个区别是选型的关键。

**Subagents 是「分身」模型。** 主 Agent 把自己的能力复制到多个子实例中，每个实例处理任务的一个切片。它们共享相同的 system prompt、相同的工具集、相同的行为模式。区别只在于输入——每个 Subagent 接收不同的子任务描述。这种模型简单、高效，但缺乏差异化：你不能让一个 Subagent 用安全审计的视角看代码，同时让另一个用性能优化的视角看同一段代码。

**Custom Agents 是「团队」模型。** 每个 Agent 是一个独立角色，有自己的专业背景和工作方式。安全 Agent 知道 OWASP Top 10，测试 Agent 了解你的测试框架偏好，文档 Agent 掌握你的技术写作规范。它们之间通过明确的接口传递信息，而不是简单的任务拆分。

在实际工程中，这个区别决定了工作流的设计方式。如果你在做一次性的大规模重构——比如把项目从 CommonJS 迁移到 ESM——Subagents 并行处理不同的模块就够了。但如果你在建设一个持续运转的开发流程——每次 PR 自动进行代码审查、安全扫描、测试验证——Custom Agents 组成的[多智能体工作流](/zh/blog/con-u-pour-des-workflows-multi-agents)才是正确的选择。

## 工程实践：深度分析

选择了架构之后，工程实践中的细节同样重要。

**Subagents 的最佳实践：** 关键在于任务拆分的粒度。太粗——失去并行的意义；太细——上下文传递的开销反而拖慢速度。经验法则是以「独立可验证的变更」为单位：一个模块的重构、一组相关测试的生成、一个 API 端点的实现。每个子任务完成后应该能独立通过编译和测试。

如果你刚开始使用 Codex，建议先从 [Codex VS Code 扩展](/zh/blog/codex-vscode)入手，在 IDE 环境中体验 Subagent 的工作方式。对于命令行用户，[Codex CLI 入门](/zh/blog/first-few-days-with-codex-cli)提供了实用的上手指南。

**Custom Agents 的最佳实践：** 关键在于 prompt 工程和工具选择。一个好的 Custom Agent 定义应该包含三个层次：

1. **角色定义**——这个 Agent 是谁，擅长什么
2. **行为约束**——什么该做、什么不该做、输出格式是什么
3. **工具授权**——哪些工具可用、哪些明确禁止

避免创建「全能」Agent。一个试图同时做代码审查、测试生成和文档撰写的 Custom Agent，不如三个各司其职的专用 Agent。这与[Agentic Coding](/zh/glossary/agentic-coding) 的核心理念一致：让每个 Agent 做它最擅长的事。

## 什么时候选 Subagents

以下场景优先考虑 Subagents：

- **一次性的大规模变更**：框架迁移、依赖升级、API 版本切换。这些任务结构明确、可独立拆分，不需要专业化的角色定义。
- **时间敏感的并行任务**：同时修改 20 个文件比逐一修改快得多。Subagents 的并行执行能力在这里是决定性优势。
- **探索性工作**：不确定最优方案时，用多个 Subagent 同时探索不同路径，然后比较结果。
- **对 Agent 系统不熟悉的团队**：Subagents 零配置、零学习成本，适合快速获得多智能体的效率收益。

一个典型用例：你要把一个 monorepo 中 15 个包的构建系统从 Webpack 切换到 Vite。每个包的迁移逻辑相似但独立，Subagent 并行处理是最优解。

## 什么时候选 Custom Agents

以下场景优先考虑 Custom Agents：

- **持续运转的工作流**：CI/CD 中的自动代码审查、每次提交前的安全扫描、发布前的文档检查。这些任务会反复执行，值得投入定义成本。
- **需要领域专业知识的任务**：安全审计需要了解攻击面，性能优化需要理解运行时特性，合规检查需要掌握行业标准。通用 Agent 做不到这个深度。
- **团队标准化需求**：当你希望所有开发者遵循相同的代码审查标准、测试覆盖要求、文档格式，Custom Agent 是最有效的执行机制。
- **多角色协作场景**：一个 Agent 写代码、一个 Agent 审查、一个 Agent 写测试——这种分工只有 Custom Agents 能实现。

一个典型用例：你的团队有严格的安全合规要求，每次 PR 必须经过安全扫描。定义一个安全审计 Custom Agent，封装你的安全检查清单和合规标准，集成到 PR 工作流中自动执行。

## 结论

**选择取决于你的需求模式：一次性任务还是持续流程、通用能力还是专业角色。**

如果你的核心需求是「把一个大任务拆快一点」——选 Subagents，零成本、立即见效。如果你的核心需求是「让不同角色各司其职，持续运转」——选 Custom Agents，前期投入换来长期回报。

最务实的做法是从 Subagents 开始。当你发现某类子任务反复出现、需要特定的专业知识、或者团队需要标准化执行时，再把它升级为 Custom Agent。两者不是互斥关系——一个 Custom Agent 内部完全可以使用 Subagents 来并行处理子任务。

## 常见问题

### Subagents 和 Custom Agents 可以一起使用吗？

可以。Custom Agent 内部可以调用 Subagents 来并行处理子任务。比如一个安全审计 Custom Agent 可以同时启动多个 Subagent 扫描不同模块的代码，汇总后生成统一的安全报告。两者是互补关系，不是替代关系。

### 创建一个 Custom Agent 需要多长时间？

从零开始定义一个可用的 Custom Agent 通常需要 1-2 小时，包括编写 system prompt、选择工具集、测试输出质量。但这是一次性投入——定义完成后，团队所有成员都能复用，后续只需要迭代优化。

### Subagents 的并行数量有限制吗？

Codex 对同时运行的 Subagent 数量有平台限制，具体取决于你的账户层级和当前的系统负载。实践中，5-10 个并行 Subagent 是比较稳定的范围。过多的并行任务可能导致上下文管理开销增大，反而降低效率。

### 不用 Agent SDK 能创建 Custom Agents 吗？

Agent SDK 是创建结构化 Custom Agents 的推荐方式，但 Codex 也支持通过配置文件和 prompt 模板定义简单的自定义行为。如果你只需要基本的角色分工，配置文件方式足够；如果需要复杂的工具集成和状态管理，Agent SDK 是更可靠的选择。
