**CLAUDE.md is a team artifact.** When you add `npm test` as a required pre-commit check, or document that the ZH content pipeline uses CJK word counting instead of space-delimited tokenization, every team member's Claude Code sessions respect those rules immediately after pulling the latest commit. Changes go through code review. The history is in `git log`. If someone adds a bad rule, you revert it like any other code change.

This makes CLAUDE.md the right home for anything that should be consistent across the team: build commands, quality gates, architectural constraints, naming conventions, and workflow expectations. It's also the right place for instructions that affect CI/CD — if your pipeline runs Claude Code in headless mode, it reads CLAUDE.md but has no access to any individual developer's Memory.

**Memory is a personal artifact.** Your correction ("don't use mocks for database tests") stays with you. Your colleague who prefers mocks in their workflow doesn't inherit your preference. Your role context ("I'm a data scientist investigating logging") shapes how Claude explains things to you without affecting how it explains things to a backend engineer on the same team.

This distinction matters for teams adopting Claude Code at scale. The pattern that works: CLAUDE.md encodes what the team has agreed on. Memory captures what each individual needs. There's no conflict because they operate at different levels — Memory personalizes behavior within the boundaries CLAUDE.md sets.

## What Belongs Where: A Decision Framework

The most common mistake teams make is putting personal preferences in CLAUDE.md (cluttering shared config with individual opinions) or putting critical rules in Memory (where they're invisible to teammates and CI).

**Put it in CLAUDE.md when:**

- It's a rule the whole team should follow — coding standards, required test commands, lint configurations
- It relates to the build or deployment process — commands, environment requirements, quality gates
- It's an architectural constraint — "don't import server-only modules in client code," "always use the `upsertKeyword()` three-argument form"
- It needs code review — any instruction that could break the build or change system behavior
- CI needs to know about it — automated pipelines read CLAUDE.md, not your personal Memory
- It's stable — the rule won't change week to week

**Put it in Memory when:**

- It's about you — your role, expertise level, communication preferences
- It's feedback on Claude's behavior — "don't summarize after every response," "I prefer single bundled PRs"
- It's time-sensitive project state — "merge freeze starts May 25," "the auth rewrite is blocked on legal review"
- It references external systems — "bugs are tracked in Linear project INGEST," "oncall dashboard is at grafana.internal/d/api-latency"
- It would rot in CLAUDE.md — deadlines pass, project priorities shift, and stale entries in a shared file create confusion

**Gray areas exist.** If you discover that every team member independently tells Claude the same thing — "we use pnpm, not npm" — that's a signal to promote it from individual Memory entries to a CLAUDE.md rule. Conversely, if a CLAUDE.md instruction only matters to one workflow that one person runs, it might belong in that person's Memory instead.

## Content Lifecycle and Staleness

One significant difference is how each system handles information decay. CLAUDE.md instructions tend to be durable — "run `npm test` before committing" stays valid for months or years. They change when the codebase changes, and the git commit that updates the code typically updates CLAUDE.md in the same PR.

Memory entries are more fragile. A project memory noting "merge freeze starts May 25" is useful for a week, then stale. A user memory capturing "I'm focused on the auth migration" becomes misleading once you move to the next project. Claude Code includes guidance to verify memories against current state before acting on them, but staleness is an inherent risk in any auto-accumulated system.

The mitigation strategy: treat Memory as context that was true when written, not as permanent truth. Claude Code already does this to some degree — it checks whether file paths mentioned in memories still exist, greps for functions before recommending them, and prefers `git log` over remembered summaries for recent repository state. But as a user, periodically reviewing your `MEMORY.md` index and pruning stale entries keeps the system accurate.

CLAUDE.md doesn't have this problem because it follows the code. When you refactor the build system, you update CLAUDE.md's build commands in the same commit. The instruction file and the codebase stay synchronized through normal development workflow.

## Memory Types and Their CLAUDE.md Equivalents

Claude Memory organizes entries into four types, each serving a purpose that partially overlaps with — but doesn't replace — CLAUDE.md.

**User memories** capture who you are: role, expertise, goals. CLAUDE.md has no equivalent because project instructions shouldn't encode individual identities. If your team's CLAUDE.md says "explain code at a senior engineer level," that forces a specific communication style on everyone. Memory lets each person set their own level.

**Feedback memories** capture behavioral corrections: "don't add trailing summaries," "prefer one bundled PR for refactors." Some of these could theoretically go in CLAUDE.md if the whole team agrees. But most feedback is personal — your preferred diff format, your tolerance for verbose output, your opinion on code comments. Memory is the right home.

**Project memories** capture dynamic state: deadlines, blockers, ongoing initiatives, who's working on what. These change too frequently for CLAUDE.md. A CLAUDE.md entry about a Q2 deadline becomes noise in Q3. Memory entries can be updated or removed without a git commit and code review cycle.

**Reference memories** capture pointers to external systems: "pipeline bugs tracked in Linear project INGEST," "API latency dashboard at this URL." Some teams put key links in CLAUDE.md, which works if the links are stable and universally relevant. But personal reference bookmarks — your monitoring dashboard, your team's Slack channel, the specific Jira board you watch — belong in Memory.

The pattern: CLAUDE.md handles the "what we do" layer. Memory handles the "who I am and what's happening now" layer. See [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) for strategies that leverage both systems.

## Configuration and Setup

**CLAUDE.md requires no setup.** Create a file named `CLAUDE.md` in your project root, write your instructions in markdown, and commit it. Claude Code reads it automatically. For global defaults, create `~/.claude/CLAUDE.md`. For subdirectory-specific rules, nest additional `CLAUDE.md` files. The hierarchy merges automatically — subdirectory rules override project-root rules, which override global rules.

**Memory setup is also automatic** — but the accumulation requires interaction. Claude Code creates the memory directory structure (`~/.claude/projects/<hash>/memory/`) the first time it writes a memory. From there, it accumulates entries by detecting when you share preference-relevant information in conversation. You can also explicitly say "remember that we use pnpm" to trigger an immediate save.

To review your memories, check the `MEMORY.md` index file in your project's memory directory. Each entry links to an individual memory file with full context and metadata. You can edit or delete these files directly — they're plain markdown.

For teams rolling out Claude Code, the recommended setup sequence is:

1. **Week 1**: Create a minimal CLAUDE.md with build commands, test commands, and the two or three most important architectural constraints
2. **Weeks 2-4**: Let Memory accumulate naturally as developers work — corrections and preferences will self-document
3. **Month 2**: Review what's accumulated in individual memories across the team. Promote recurring themes to CLAUDE.md. Prune stale project memories
4. **Ongoing**: Update CLAUDE.md alongside code changes. Let Memory handle the rest

## When to Choose Claude Memory

Choose Memory as your primary context mechanism when:

- **You're a solo developer** and the distinction between "shared" and "personal" doesn't apply — but still use CLAUDE.md for instructions you'd want preserved if you onboard a collaborator later
- **Your context is highly dynamic** — active incident response, rapidly shifting priorities, evolving project scope where weekly CLAUDE.md updates would be impractical
- **You need per-person customization** — a team where the data scientist, the frontend engineer, and the SRE all work in the same repo but need Claude to communicate differently with each
- **You're tracking external state** — links to dashboards, references to ticket systems, pointers to documentation that lives outside the repo
- **You want zero-friction capture** — saying "remember this" mid-conversation is faster than opening CLAUDE.md, editing, committing, and pushing

Memory shines in the "working memory" role: what's happening now, what I've told Claude before, what I care about as an individual. It reduces the repetition tax of re-explaining yourself every session.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary context mechanism when:

- **You're working on a team** and need consistent AI behavior across all developers — coding standards, architectural boundaries, and quality gates should never depend on individual memory
- **Your instructions affect CI/CD** — automated pipelines, pre-commit hooks, and headless Claude Code sessions read CLAUDE.md but not personal Memory
- **You need auditability** — CLAUDE.md changes are tracked in git history, reviewable in PRs, and revertable. If a bad instruction causes problems, you can trace when it was added and by whom
- **Your constraints are stable** — "always run tests before committing" doesn't change monthly. CLAUDE.md is the right home for durable rules
- **You want to onboard new team members** — a well-written CLAUDE.md means new developers get Claude Code sessions that already follow team conventions on their first day
- **You're encoding [skills](/blog/5-claude-code-skills-i-use-every-single-day)** — SKILL.md files (CLAUDE.md's task-specific cousin) define reusable workflows that should be consistent across the team

CLAUDE.md wins for anything that functions like a rule, a constraint, or a standard. If you'd put it in a team wiki or a CONTRIBUTING.md file, it belongs in CLAUDE.md.

## Verdict

**Use both — they're complementary, not competing.** CLAUDE.md is your project's shared instruction set: deterministic, version-controlled, team-wide. Memory is your personal context layer: adaptive, auto-accumulated, individual. The mistake is treating them as interchangeable when they serve fundamentally different roles.

**Start with CLAUDE.md.** It's the higher-leverage investment. A 30-line CLAUDE.md with your build commands, test requirements, and top architectural constraints improves every Claude Code session for every team member immediately. Memory accumulates naturally as you work — you don't need to front-load it.

**Promote patterns from Memory to CLAUDE.md** when you notice the same feedback appearing across multiple team members. If three developers independently tell Claude "we use pnpm, not npm," that's a CLAUDE.md rule waiting to be written. The flow from personal discovery to shared standard is one of the most effective ways to evolve your team's Claude Code configuration.

## Frequently Asked Questions

### Can Claude Memory and CLAUDE.md conflict with each other?

Yes, and when they do, CLAUDE.md wins. Claude Code treats CLAUDE.md instructions as authoritative project rules, while Memory provides supplementary context. If your Memory says "prefer mocks for testing" but CLAUDE.md says "always use integration tests against a real database," the CLAUDE.md rule takes precedence. This hierarchy is intentional — shared team standards should override individual preferences.

### Does Claude Memory sync across devices?

No. Memory files live in your local `~/.claude/projects/` directory and don't sync automatically. If you work on multiple machines, each machine accumulates its own memories independently. Some developers manually copy their memory directories, but there's no built-in sync mechanism. CLAUDE.md, by contrast, syncs naturally through git — pull the repo and you have the latest instructions.

### How many memories can Claude Code store before performance degrades?

The `MEMORY.md` index is loaded into every conversation, so a very long index can consume context window space. Practical guidance is to keep the index under 200 lines. Individual memory files are retrieved selectively based on relevance, so having dozens of well-described memory files doesn't degrade performance — only the index size matters for baseline context cost.

### Should I put API keys or secrets in CLAUDE.md or Memory?

Neither. CLAUDE.md is checked into git, making it visible to anyone with repo access. Memory files are local but stored as plain text. Use environment variables or a secrets manager for credentials. CLAUDE.md can reference the variable name — "set `DATABASE_URL` before running migrations" — without containing the value.

### Can sub-agents and CI runners access Claude Memory?

Sub-agents spawned by Claude Code inherit the parent session's loaded memories but don't have independent access to the memory directory. CI runners and headless Claude Code sessions read CLAUDE.md but not personal Memory, since Memory is tied to a specific user's filesystem. This is another reason to put CI-critical rules in CLAUDE.md rather than relying on Memory.
