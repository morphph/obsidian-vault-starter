# Claude Memory vs CLAUDE.md：两套记忆系统到底怎么选？

**TL;DR：** **Claude Memory** 是跨会话的自动持久记忆，记住你是谁、你喜欢什么；**CLAUDE.md** 是项目级的显式指令文件，定义这个项目该怎么做。两者不是竞争关系——**Memory 管「人」，CLAUDE.md 管「项目」**。最佳实践是同时使用：Memory 积累个人偏好，CLAUDE.md 固化团队规范。

## 概述：Claude Memory

**Claude Memory** 是 Claude 的自动持久记忆系统，它在对话过程中学习并记住关于你的信息——你的角色、技术偏好、工作习惯、项目背景——然后在未来的对话中自动调用这些记忆。你不需要每次开新会话都重复自我介绍。

在 Claude Code 中，这套系统被称为 **Auto Memory**，记忆文件存储在 `~/.claude/projects/` 目录下，以 Markdown 格式组织。Anthropic 在 2026 年初对记忆系统做了[重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，免费用户也能使用，还支持从 ChatGPT 导入历史数据。

Memory 的核心特点是**隐式**的：Claude 自己决定什么值得记住，什么时候调用记忆。你可以主动要求它「记住这一点」或「忘掉那件事」，但大部分时候它在后台自动运行。

## 概述：CLAUDE.md

**CLAUDE.md** 是一个检入代码仓库的 Markdown 文件，为 Claude Code 提供项目级别的指令和上下文。它的作用类似于一份「项目操作手册」——告诉 Claude 这个项目用什么技术栈、遵循什么编码规范、有哪些已知坑、构建和测试命令是什么。

CLAUDE.md 的核心特点是**显式**的：每一条规则都是开发者手动写入的，所有团队成员共享同一份文件。它和代码一起版本控制，通过 PR 审查和迭代。在 Claude Code 的[七层架构](/zh/blog/claude-code-seven-programmable-layers)中，CLAUDE.md 处于最基础的项目上下文层，是 Claude 理解你代码库的第一入口。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **存储位置** | `~/.claude/` 用户目录 | 项目根目录，检入 Git | CLAUDE.md |
| **作用范围** | 跨所有项目、所有会话 | 仅当前项目 | 各有所长 |
| **管理方式** | Claude 自动学习 + 用户口头指令 | 开发者手动编写和维护 | Memory |
| **团队共享** | 不共享，仅限个人 | 通过 Git 共享给全团队 | CLAUDE.md |
| **版本控制** | 无版本历史 | 完整 Git 历史 | CLAUDE.md |
| **内容类型** | 用户偏好、反馈、项目背景 | 构建命令、编码规范、架构约束 | 各有所长 |
| **生效时机** | 自动加载相关记忆 | 每次会话开始时全量加载 | 平手 |
| **可审计性** | 低——记忆可能过时或不准确 | 高——和代码一样可以 review | CLAUDE.md |

## 工作机制：详细对比

Claude Memory 和 CLAUDE.md 在技术实现上完全不同。Memory 采用基于文件的存储系统，每条记忆是一个独立的 Markdown 文件，带有 frontmatter 元数据（名称、描述、类型）。Claude 在对话过程中根据相关性自动检索和加载这些记忆。记忆分为四类：用户信息（user）、行为反馈（feedback）、项目上下文（project）、外部引用（reference）。

CLAUDE.md 则是一个静态文件，Claude Code 在每次会话启动时完整读取。它支持多层级结构：全局 `~/.claude/CLAUDE.md` 适用于所有项目，项目根目录的 `CLAUDE.md` 适用于当前项目，子目录的 `CLAUDE.md` 可以覆盖或补充上层规则。这种分层设计在 Claude Code 的[扩展栈架构](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中扮演关键角色。

两者的关键差异在于**可靠性模型**。CLAUDE.md 是确定性的——写了什么就执行什么，团队里每个人看到的都一样。Memory 是概率性的——Claude 可能记错、遗忘、或在不恰当的时机调用某条记忆。这意味着关键的项目规则绝不应该只依赖 Memory。

## 适用场景：详细分析

选择 Memory 还是 CLAUDE.md，取决于你要存储的信息属于「个人偏好」还是「项目规则」。

**个人偏好类信息**适合存入 Memory：你喜欢简洁的回复风格、你是前端新手但后端很资深、你讨厌代码里写注释、你的测试偏好是集成测试优先。这些信息因人而异，不需要（也不应该）强制给团队里的其他人。

**项目规则类信息**适合写入 CLAUDE.md：构建命令 `npm run build`、提交前必须通过所有测试、数据库迁移必须向后兼容、API 路由的命名规范。这些规则对所有团队成员一视同仁，必须强制执行。如果你正在系统性地使用 Claude Code 的能力体系，可以参考[这篇技巧指南](/zh/blog/5-claude-code-skills-i-use-every-single-day)了解实际工作流中如何搭配使用。

有一个灰色地带：**项目背景信息**。比如「我们正在做 auth 模块重构，法务要求改 session token 存储方式」——这条信息对当前项目很重要，但它是临时性的（重构完就没用了），而且包含决策背景（为什么要改）。这类信息可以存入 Memory 的 project 类型，也可以写入 CLAUDE.md。我的建议：如果只有你一个人需要知道，用 Memory；如果团队都需要知道，写进 CLAUDE.md。

## 什么时候选 Claude Memory

Memory 最适合以下场景：

- **个人工作流定制**：你有自己的编码风格、commit message 偏好、回复风格要求，希望 Claude 在所有项目中都遵循
- **跨项目上下文积累**：你同时维护多个项目，希望 Claude 记住你在不同项目中的角色和关注点
- **行为纠正的持久化**：你纠正过 Claude 的某个行为（比如「不要在每次回复末尾写总结」），希望它在未来的对话中也记住这个偏好
- **从其他 AI 工具迁移**：利用[记忆导入功能](/zh/blog/anthropic-claude-memory-upgrades-importing)把 ChatGPT 的对话历史带过来

## 什么时候选 CLAUDE.md

CLAUDE.md 最适合以下场景：

- **团队协作**：你需要确保所有团队成员的 Claude Code 都遵循相同的编码规范和工作流程
- **关键约束的强制执行**：构建命令、测试要求、安全规则——这些不能依赖 Claude 的「记忆」，必须每次都明确加载
- **可审计的规则变更**：通过 Git 追踪 CLAUDE.md 的每一次修改，知道谁在什么时候改了什么规则、为什么改
- **新成员 onboarding**：新人 clone 项目后，Claude Code 自动获得所有项目上下文，不需要口头传授「潜规则」

关于 CLAUDE.md 的深度用法，包括如何构建多层级的[记忆系统](/zh/blog/claude-code-memory)，推荐阅读我们的专题分析。

## 最佳实践：两者搭配使用

实际工作中，Memory 和 CLAUDE.md 不是二选一的关系。最高效的用法是：

1. **CLAUDE.md 负责「硬规则」**：构建命令、代码规范、架构约束、质量门禁。这些写进文件，版本控制，团队共享
2. **Memory 负责「软偏好」**：个人风格、沟通习惯、角色背景。这些由 Claude 自动学习和应用
3. **避免重复**：不要在 Memory 里存储已经写在 CLAUDE.md 中的规则，也不要在 CLAUDE.md 里写只对你个人有效的偏好
4. **定期清理 Memory**：Memory 会随时间积累过时信息，定期检查 `~/.claude/projects/` 目录，删除不再准确的记忆文件

**决策规则：如果这条信息丢失会导致构建失败或代码质量问题，它必须在 CLAUDE.md 里。如果丢失只会让 Claude 的回复风格不太对，放在 Memory 就行。**

## 结论

**Claude Memory 和 CLAUDE.md 解决的是两个不同的问题。** Memory 让 Claude 认识你——你的偏好、风格、背景；CLAUDE.md 让 Claude 认识你的项目——规范、约束、工作流。两者结合，才能构建出一个既懂你个人习惯、又严格遵守团队规范的 AI 编程助手。如果你只能选一个，**先写好 CLAUDE.md**——项目规则的确定性执行比个人偏好的自动学习更重要。在此基础上，再逐步让 Memory 积累你的个人上下文，参考 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)把整套体系打磨到位。

## 常见问题

### Claude Memory 会覆盖 CLAUDE.md 里的规则吗？

不会。CLAUDE.md 中的指令优先级高于 Memory 中的记忆。如果 CLAUDE.md 规定「commit message 必须用英文」，即使 Memory 里记住了你偏好中文 commit message，Claude Code 也会遵循 CLAUDE.md 的规则。两者冲突时，显式指令胜过隐式记忆。

### CLAUDE.md 有大小限制吗？

没有硬性限制，但过长的 CLAUDE.md 会占用 Claude 的上下文窗口。建议控制在 500 行以内，把详细的指令拆分到 `skills/*/SKILL.md` 文件中。Claude Code 只在需要时加载 SKILL.md，不会浪费基础上下文空间。

### 我能手动编辑 Memory 文件吗？

可以。Memory 文件是标准 Markdown，存储在 `~/.claude/projects/` 目录下。你可以直接编辑、删除或新增记忆文件。但通常更推荐通过对话让 Claude 管理——告诉它「记住这一点」或「忘掉那件事」。
