**CLAUDE.md** uses a simpler model: the file loads into Claude Code's system prompt at conversation start. There's no dynamic retrieval, no relevance filtering — everything in the file applies every time. This makes it predictable but requires discipline. A 500-line CLAUDE.md with outdated sections actively degrades Claude Code's performance by consuming context with irrelevant instructions.

The practical implication: Memory is high-volume, low-maintenance, but needs trust-but-verify. CLAUDE.md is low-volume, high-maintenance, but deterministic. For a deeper look at how these systems interact within [Claude Code's complete architecture](/blog/claude-code-complete-guide), the memory layer sits above CLAUDE.md in the prompt hierarchy — meaning Memory can add context but shouldn't contradict project rules.

## Content Boundaries: What Belongs Where

The most common mistake teams make is putting the wrong information in the wrong layer. Here's the decision framework:

**Put it in CLAUDE.md when:**
- Every developer on the team needs to follow the same rule
- The instruction relates to code standards, build processes, or quality gates
- The information changes through deliberate decisions (not organic learning)
- You'd put it in a CONTRIBUTING.md or .editorconfig if AI tools didn't exist
- It defines what Claude Code *must* or *must not* do in this project

Examples: "Run `npm test` before committing," "Never skip ESLint rules," "Use snake_case for database columns," "Deploy target is Vercel"

**Put it in Memory when:**
- It's personal to how *you* work (not the project's rules)
- It's something Claude Code learned from a correction or observation
- It provides context that helps Claude Code tailor responses to you specifically
- It references external systems or resources your workflow touches
- It captures project state that isn't derivable from the code itself

Examples: "User prefers terse responses with no trailing summaries," "Pipeline bugs tracked in Linear project INGEST," "Merge freeze begins 2026-03-05 for mobile release cut"

**Neither — use something else:**
- Current task progress → use TodoWrite within the conversation
- Implementation plan → use Plan mode
- Code patterns and architecture → derivable from the code itself
- Git history and recent changes → use `git log`

This boundary exists because Memory and CLAUDE.md serve different audiences. CLAUDE.md speaks to "any Claude Code session working on this project." Memory speaks to "Claude Code working with *this specific person*." When teams blur this line — putting personal preferences in CLAUDE.md or project rules in Memory — they get inconsistent behavior and context waste.

## Team Dynamics: Collaboration Patterns

For solo developers, the distinction between Memory and CLAUDE.md can feel academic — you're the only user either way. But the difference becomes critical at team scale.

**CLAUDE.md as team alignment:**
When five engineers work on the same repo, CLAUDE.md ensures every Claude Code session follows identical rules. Engineer A's session won't skip tests that Engineer B's session enforces. The file goes through code review, so the team explicitly agrees on what Claude Code should do. This is particularly valuable for [teams adopting Claude Code at scale](/blog/claude-code-enterprise-engineering-ramp-shopify-spotify) — onboarding a new developer means they inherit the AI configuration automatically.

**Memory as individual adaptation:**
Each team member's Memory layer captures their personal working style. A senior architect might have memories like "prefers high-level explanations, not step-by-step walkthroughs." A junior developer might accumulate corrections that help Claude Code explain things more thoroughly. Neither set of preferences belongs in CLAUDE.md because they'd conflict.

**The handoff problem:**
Memory doesn't transfer between machines or team members. If you set up a CI environment or a new laptop, Memory starts empty. CLAUDE.md, being in git, is immediately available. This makes CLAUDE.md the authoritative source for "how this project works" and Memory the personal layer for "how I work on this project."

Teams that rely heavily on Memory for project-critical information (like deployment procedures or architecture decisions) create knowledge silos. If the developer who accumulated those memories leaves, the information disappears. The rule: if losing the information would block someone else's work, it belongs in CLAUDE.md or proper documentation — not Memory.

## Maintenance and Lifecycle

**CLAUDE.md maintenance:**
CLAUDE.md requires active curation. As projects evolve, instructions become outdated. A build command changes, a linter rule gets added, a deployment target moves. Unlike code that breaks visibly when outdated, stale CLAUDE.md instructions cause subtle degradation — Claude Code might follow an old convention or skip a new quality gate.

Best practice: treat CLAUDE.md updates as part of the definition of done. When you change a build process, update CLAUDE.md in the same PR. The [documentation rules in well-maintained projects](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) enforce this explicitly — any pipeline change must update the corresponding docs.

**Memory maintenance:**
Memory is designed to be low-maintenance but isn't zero-maintenance. Claude Code can update or remove memories that it identifies as stale, but it requires verification against current state. A memory claiming "auth module uses JWT" won't self-correct if the team switched to session tokens.

The Memory system includes safeguards: before acting on a remembered fact, Claude Code should verify it against the current codebase. But this verification isn't guaranteed — especially for memories about external systems, team composition, or project priorities that can't be checked by reading files.

**Staleness risk comparison:**
- CLAUDE.md: Low staleness risk if maintained with PRs. High impact when stale (wrong rules applied globally)
- Memory: Moderate staleness risk (no review process). Lower impact (affects one person, and verification is built into the retrieval process)

## When to Choose Claude Memory

Choose Memory as your primary persistence mechanism when:

**You're a solo developer** building personal workflow efficiency. Memory accumulates your preferences, learns from your corrections, and adapts Claude Code to your style over time. You don't need the team-coordination overhead of CLAUDE.md for personal rules.

**You need cross-project context.** Memory can store references to external systems (Linear boards, Grafana dashboards, Slack channels) that span multiple repos. CLAUDE.md is scoped to one project; Memory is scoped to you across all projects.

**Information changes frequently.** Project priorities, sprint goals, team member assignments — these shift weekly. Putting them in CLAUDE.md creates noise and maintenance burden. Memory handles ephemeral project context that's useful now but might not be next month.

**You want Claude Code to learn from corrections.** When you tell Claude Code "don't write trailing summaries" or "always use integration tests here," Memory captures that as a feedback entry. The next conversation applies the correction without you repeating it. This is the primary mechanism described in [how Claude Code's memory system operates](/blog/claude-code-memory).

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary persistence mechanism when:

**Multiple people work on the project.** Any rule that should apply regardless of who's running Claude Code belongs in CLAUDE.md. Build commands, test requirements, commit conventions, deployment procedures — these are project properties, not personal preferences.

**You need deterministic, reviewable behavior.** CLAUDE.md goes through code review. You can `git blame` to see who added a rule and why. You can revert a change that caused problems. Memory offers none of this auditability.

**The information is derivable from project state but expensive to derive.** Claude Code *could* figure out your build command by reading package.json, but putting `npm run build` in CLAUDE.md saves context and eliminates ambiguity. Think of CLAUDE.md as a pre-computed index of project conventions.

**You're defining constraints and prohibitions.** "Never commit .env files." "Don't skip the linter." "Always run tests before pushing." These are guardrails that must apply universally and immediately — they can't wait for Memory to accumulate them through corrections.

**You want reproducible AI behavior.** When debugging why Claude Code did something unexpected, CLAUDE.md provides a clear chain: "The file says X, Claude did X." Memory is harder to audit because it's implicit, personal, and potentially stale.

## Advanced: Using Both Together

The most effective Claude Code configurations use both layers deliberately. Here's the pattern that works for teams using [Claude Code's full extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp):

**CLAUDE.md handles the "what":**
- Build and test commands
- Quality gates (must pass before commit)
- Code style rules
- Architecture constraints
- Deployment procedures

**Memory handles the "how I work with what":**
- Personal communication preferences (verbosity, format)
- Corrections and refinements specific to your workflow
- External system references your workflow touches
- Temporal project context (current sprint, active incidents)

**The interaction model:**
When Claude Code starts a session, it loads CLAUDE.md (project rules) and MEMORY.md (personal context) into its prompt. CLAUDE.md tells it *what the project requires*. Memory tells it *how you specifically want to interact with those requirements*. A CLAUDE.md rule might say "write tests for all new functions." Your Memory might say "user prefers table-driven tests over individual assertions." Both apply simultaneously without conflict.

**Conflict resolution:**
If Memory contradicts CLAUDE.md, CLAUDE.md wins. This is by design — project rules established through code review take precedence over individually accumulated preferences. If you find yourself wanting to override a CLAUDE.md rule through Memory, the correct action is to propose a CLAUDE.md change through a PR.

## Practical Setup Recommendations

**For new solo projects:**
Start with a minimal CLAUDE.md (build commands, test commands, key conventions) and let Memory accumulate organically. You'll naturally correct Claude Code during early sessions, and those corrections become your Memory layer. After a week, review your Memory entries and promote any that feel like project rules into CLAUDE.md.

**For team projects:**
Write CLAUDE.md first — before anyone uses Claude Code on the project. Include quality gates, style rules, and workflow requirements. Then each team member's Memory layer handles their personal adaptations. Review [how to write effective CLAUDE.md files](/blog/how-to-effectively-prompt-a-claude-code) for structural guidance.

**For existing projects adopting Claude Code:**
Audit your existing documentation (CONTRIBUTING.md, README, CI config) and extract the rules into CLAUDE.md format. Don't duplicate — reference. Then start using Claude Code and let Memory build up the contextual knowledge that your static docs don't capture.

## Verdict

**CLAUDE.md is the foundation; Memory is the personalization layer.** Every project using Claude Code should have a CLAUDE.md file — it's the minimum viable configuration that ensures consistent, rule-following AI behavior. Memory adds individual efficiency on top: your corrections stick, your preferences persist, your workflow context carries forward.

If you're forced to choose one: pick **CLAUDE.md for team projects** (consistency wins) and **Memory for personal projects** (adaptation wins). But the real answer is to use both intentionally — CLAUDE.md for shared rules, Memory for personal context. The developers getting the most from Claude Code treat CLAUDE.md like infrastructure (maintained, reviewed, essential) and Memory like a personal assistant's notebook (useful, personal, complementary). ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md rules?

No. CLAUDE.md instructions take precedence over Memory when they conflict. Memory is designed to add personal context and preferences, not override project-level rules. If you need to change a CLAUDE.md rule, modify the file through a normal code review process rather than trying to work around it through Memory corrections.

### Does Claude Memory sync across devices?

No. Claude Memory is stored locally in `~/.claude/projects/` on your machine. If you work from multiple computers, each maintains its own Memory. CLAUDE.md, being in version control, syncs automatically through git. For critical workflow information that you need everywhere, put it in CLAUDE.md or your global `~/.claude/CLAUDE.md` file.

### How much should I put in CLAUDE.md before it hurts performance?

Keep CLAUDE.md under 200 lines of actionable instructions. Every line consumes context window space in every conversation. Focus on rules that Claude Code can't derive from the codebase itself — build commands, quality gates, style exceptions, and architectural constraints. If a rule is obvious from reading the code, it doesn't need to be in CLAUDE.md.

### What happens to Memory entries when the project changes?

Memory entries can become stale when project structure, tooling, or priorities change. Claude Code is designed to verify memories against current state before acting on them, but this isn't guaranteed. Periodically review your Memory entries (check `~/.claude/projects/` for your project) and remove anything outdated. Unlike CLAUDE.md, there's no automated staleness detection.

### Should SKILL.md files go in Memory or CLAUDE.md?

Neither — [SKILL.md files](/blog/5-claude-code-skills-i-use-every-single-day) are their own persistence layer. They live in a `skills/` directory in your repo and define reusable instruction sets for specific tasks (writing tests, generating content, doing code review). CLAUDE.md references them but doesn't contain them. Memory might store preferences about *how* you invoke skills, but the skills themselves are version-controlled project assets.
