**Claude Code's approach** is hierarchical and synchronous. A parent agent receives your task, plans the decomposition, and spawns subagents with specific briefs. Each subagent gets a self-contained prompt explaining what to accomplish, what context matters, and what form the output should take. The parent waits for results (or runs agents in background with notifications), then synthesizes findings into a coherent response. This mirrors how a tech lead delegates to team members in a meeting — direct briefing, parallel execution, synchronous integration.

The typed subagent system adds a layer of safety. An `Explore` agent can read files and search but cannot edit — useful when you want research without risk. A `Plan` agent designs implementation strategies without executing them. The [Agent SDK](/glossary/agent-sdk) pattern lets you define custom agent types with specific tool access, creating guardrails that prevent subagents from taking actions outside their role.

For practical examples of this in action, our [Claude Code subagents guide](/blog/claude-code-subagents-examples) walks through real orchestration patterns: parallel test execution across modules, multi-file refactoring with isolated review, and research-then-implement workflows.

**Codex's approach** is queue-based and asynchronous. You define tasks (either manually or programmatically), each task gets its own container, and results accumulate for batch review. There's no parent agent actively orchestrating — instead, you're the orchestrator, deciding which tasks to submit and how to integrate their outputs.

This design shines for repetitive, well-defined work: "apply this migration pattern to all 12 service modules" or "write unit tests for every exported function in this package." Each task is independent, isolated, and reviewable. The [Codex for open source](/blog/codex-for-open-source) initiative demonstrates this at scale — maintainers submit batches of tasks for documentation, test coverage, or dependency updates.

The tradeoff is clear: Claude Code gives you **intelligent decomposition** (the AI decides how to break the task apart), while Codex gives you **manual decomposition with parallel execution** (you decide the breakdown, the AI executes each piece).

## Context and Memory: Detailed Analysis

How each system handles context — the information agents need to do good work — fundamentally shapes what kinds of tasks they can handle.

**Claude Code subagents** inherit context through multiple mechanisms. The parent agent can pass explicit context in the subagent's prompt (file contents, previous findings, constraints). Subagents also read the project's CLAUDE.md and relevant SKILL.md files automatically, picking up project conventions without explicit instruction. When using `isolation: "worktree"`, subagents get a full copy of the repo to work in without affecting the main working tree.

The critical advantage: subagents can reference results from other subagents in the same session. The parent agent acts as a context broker — if Agent A finds that the auth module uses JWT tokens, the parent can inform Agent B (which is refactoring the middleware) about this dependency. This emergent coordination happens naturally because the parent maintains full conversation context.

**Codex custom agents** receive context through environment setup. Each task container runs a setup script (typically `install.sh` or similar) that installs dependencies and configures the environment. The agent's system prompt defines its role and approach. Beyond that, context comes from the repository itself — the agent reads files as needed during execution.

The limitation: Codex agents in separate tasks don't share context with each other. If Task A discovers something relevant to Task B, there's no mechanism for cross-task communication. You'd need to manually bridge that gap by feeding Task A's output into Task B's prompt. For workflows that require coordination between agents, this creates friction that Claude Code's parent-child model handles automatically.

For [agentic coding](/glossary/agentic-coding) patterns that require deep project understanding — like refactoring a module while maintaining API compatibility — Claude Code's context-rich subagents have a structural advantage.

## Customization and Configuration: Detailed Analysis

Both platforms offer mechanisms for customizing agent behavior, but the approaches differ significantly in flexibility and complexity.

**Claude Code customization** operates at multiple levels. At the project level, CLAUDE.md files define conventions, constraints, and architectural decisions. At the task level, SKILL.md files encode specific workflows — how to write tests, how to review PRs, how to generate documentation. At the agent level, subagent types define tool access and behavioral constraints. This layered system means you can encode your team's standards once and have every subagent follow them automatically.

The [Claude Code extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) — Skills, Hooks, Agents, and MCP servers — provides a programmable surface for customization. Hooks trigger deterministic actions on specific events (pre-commit validation, post-edit formatting). MCP servers expose external tools and data sources. Together, these layers let you build highly customized multi-agent workflows without writing a framework from scratch.

**Codex customization** centers on the system prompt and environment configuration. You write detailed instructions for how the agent should approach work, what coding standards to follow, and what tools are available. The `AGENTS.md` file (Codex's equivalent to CLAUDE.md) defines project-level context. Custom agents are essentially specialized system prompts paired with environment setups.

Codex's simpler model has an advantage: it's easier to understand and debug. When an agent misbehaves, you edit its system prompt. There's no layered inheritance to untangle. For teams that want straightforward customization without learning a multi-layer configuration system, this simplicity is genuine value.

However, Claude Code's layered approach scales better for complex organizations. When you have 50 repositories with different conventions, the CLAUDE.md + SKILL.md + hooks system lets each repo define its own agent behavior without duplicating configuration. Codex would require maintaining separate system prompts for each repository's agents.

## Safety and Isolation: Detailed Analysis

When you let AI agents modify your codebase in parallel, safety mechanisms become critical. Both platforms take this seriously but make different architectural choices.

**Claude Code** uses a permission-based safety model. Every tool call that could be destructive (file writes, shell commands, git operations) goes through a permission check. Users can pre-approve categories of actions via settings, or approve individually at runtime. Subagents in `worktree` isolation get their own copy of the repository — changes only merge back if the subagent produces useful results, and the worktree is cleaned up automatically if no changes are made.

The typed subagent system adds behavioral constraints: an Explore agent literally cannot edit files, regardless of what its prompt says. This is enforced at the tool-access level, not just the prompt level — a structural guarantee rather than a best-effort instruction.

**Codex** uses container-based isolation. Each task runs in its own sandboxed environment with network restrictions. The agent can't access your local filesystem, other running tasks, or external services (with limited exceptions for package registries). Changes are proposed as diffs rather than applied directly — you review and approve before anything touches your actual codebase.

This containerized model provides stronger isolation guarantees. A misbehaving Codex agent can't accidentally delete files in your working directory or push to a remote branch — it physically doesn't have access. Claude Code's permission model relies on the agent respecting boundaries that are enforced at runtime but within the same process space.

For security-sensitive workflows, Codex's container isolation provides higher confidence. For interactive workflows where you want agents to actually execute changes (run tests, verify builds), Claude Code's direct access model is more practical.

## When to Choose Claude Code Subagents

Choose Claude Code's multi-agent system when your workflow requires:

- **Interactive decomposition**: You have a complex task but aren't sure how to break it down — let the parent agent decide the subtask structure based on codebase analysis
- **Cross-agent coordination**: Subtasks depend on each other's findings (refactoring module A requires understanding module B's API)
- **Iterative refinement**: You want to review intermediate results and redirect agents mid-task
- **Deep project context**: Your codebase has specific conventions encoded in CLAUDE.md and SKILL.md that agents must follow
- **Terminal-native workflows**: You live in the terminal and want multi-agent orchestration without leaving your shell

Claude Code subagents excel at the kind of work a senior engineer does: understanding a system holistically, breaking problems into coherent subtasks, maintaining awareness of dependencies, and integrating diverse solutions into a unified result. If your task requires intelligence in the decomposition itself — not just in the execution — Claude Code has the architectural advantage.

Read our [examples guide](/blog/claude-code-subagents-examples) for concrete patterns you can adopt immediately.

## When to Choose Codex Custom Agents

Choose Codex's custom agent system when your workflow requires:

- **Batch processing**: Many similar, independent tasks that don't require coordination (writing tests for 20 modules, updating docs across a monorepo)
- **Asynchronous execution**: You want to submit tasks and review results later, not babysit an interactive session
- **Strong isolation**: Security requirements demand that agents cannot access your local environment or each other's state
- **CI/CD integration**: You need agents that trigger on pull requests, issue comments, or merge events through GitHub integration
- **Team-scale review**: Multiple team members submitting and reviewing agent work through a shared interface

Codex custom agents excel at the kind of work you'd put in a task queue: well-defined, repeatable, independently verifiable. If you can write a clear brief for each task and don't need agents to talk to each other, Codex's async model lets you parallelize massively without the complexity of orchestration. The [Codex complete guide](/blog/codex-complete-guide) covers setup and configuration in detail.

## Combining Both Approaches

Many teams don't need to choose exclusively. A practical hybrid workflow:

1. Use **Claude Code subagents** for exploratory work — investigating bugs, planning architecture, prototyping solutions that require back-and-forth refinement
2. Once you've identified well-defined subtasks, submit them as **Codex custom agents** for parallel execution in isolated containers
3. Use Claude Code to review and integrate Codex's proposed diffs, resolving conflicts and ensuring coherence

This mirrors how engineering organizations work: architects design and coordinate interactively, then hand off implementation to parallel workers who execute independently.

## Verdict

**For interactive, context-rich multi-agent coding, choose [Claude Code subagents](/blog/claude-code-agent-teams).** The parent-child orchestration model, typed agent roles, and layered context system handle complex engineering tasks that require intelligent decomposition and cross-agent coordination. If your work involves understanding dependencies, maintaining conventions, and iterating on solutions, Claude Code's architecture is purpose-built for this.

**For batch, asynchronous multi-agent execution, choose Codex custom agents.** The containerized isolation, async execution model, and CI integration handle high-volume parallel tasks that are well-defined and independent. If you can write clear briefs and want fire-and-forget parallelism with strong safety guarantees, Codex delivers.

The deciding factor: do your agents need to coordinate, or just execute in parallel? Coordination demands Claude Code. Pure parallelism favors Codex.

## Frequently Asked Questions

### Can Claude Code subagents access external APIs during execution?

Yes — Claude Code subagents can access any tool available to the parent agent, including MCP servers that connect to external APIs, databases, and services. Each subagent's tool access is constrained by its type: an Explore agent gets read-only tools, while a general-purpose agent gets full access subject to user permissions.

### How many concurrent agents can Codex run?

Codex concurrency limits depend on your plan tier. Pro users can run multiple tasks simultaneously, with the exact limit varying based on current platform load. Tasks queue automatically when limits are reached. For enterprise deployments, higher concurrency is available through team and enterprise plans.

### Do Claude Code subagents share token costs with the parent agent?

Each subagent consumes its own tokens independently — prompt tokens for the context it receives and completion tokens for its output. The parent agent also uses tokens to synthesize subagent results. Total cost for a multi-agent session is the sum of all agent token usage. There's no per-agent surcharge beyond standard API billing.

### Can I use Codex custom agents with private repositories?

Yes — Codex agents can access private repositories that you've connected to your OpenAI account through GitHub integration. Each task container clones the repository at the specified branch, and proposed changes are submitted as pull requests or patches against that branch.

### Which platform handles larger codebases better for multi-agent work?

Claude Code handles large codebases through selective context — subagents receive only the information relevant to their subtask, keeping token usage manageable. Codex handles scale through full repository clones in each container, which works well for independent tasks but increases setup time for very large repos. For codebases over 100K lines, Claude Code's selective context approach is generally more efficient.
