**CLAUDE.md** is a deterministic instruction file that lives in your project root and gets checked into version control. It tells Claude Code how to behave in a specific codebase — which commands to run, which tests must pass before committing, which style conventions to follow, and which architectural decisions are non-negotiable. Every developer on the team who uses Claude Code gets the same instructions automatically.

Think of CLAUDE.md as the project's constitution. It doesn't learn or evolve on its own — someone edits it deliberately, commits the change, and the whole team gets the update on their next `git pull`. This determinism is its core strength. When your CLAUDE.md says "run `npm test` before every commit," that rule applies to every session, every developer, every time. There's no drift, no forgetting, no ambiguity.

CLAUDE.md files can exist at multiple levels: a global `~/.claude/CLAUDE.md` for personal defaults across all projects, a project-root `CLAUDE.md` for repo-wide rules, and nested CLAUDE.md files in subdirectories for module-specific instructions. Claude Code merges them hierarchically, with more specific files taking precedence. For an in-depth walkthrough of how this fits into Claude Code's broader architecture, see the [complete guide to Claude Code](/blog/claude-code-complete-guide).

## Persistence and Scope: How Each System Stores Information

Claude Memory and CLAUDE.md solve fundamentally different persistence problems, and understanding this distinction is the key to using them effectively.

**CLAUDE.md is declarative and shared.** You write explicit instructions — "use Vitest for testing," "never skip the build step," "commit messages must describe what changed" — and every Claude Code session in that project follows them. The file travels with the repo. When a new engineer clones the project and opens Claude Code, they get the same guardrails as everyone else. This is critical for team consistency: without CLAUDE.md, each developer's Claude Code sessions would interpret project conventions differently.

**Memory is learned and personal.** It accumulates from your interactions. When you correct Claude Code ("don't mock the database in integration tests — we got burned by that last quarter"), it stores that as a feedback memory. When you mention your role ("I'm a data scientist investigating our logging"), it stores that as a user memory. Over time, Claude Code builds a profile of how to work with you specifically. This profile is local to your machine and doesn't affect your teammates.

The practical implication: if your team has a rule that matters, put it in CLAUDE.md. If you have a personal preference or context that only matters for your sessions, let Memory handle it. Overlap between the two is wasted effort — and can cause confusion when Memory contradicts CLAUDE.md.

As part of Claude Code's [seven programmable layers](/blog/claude-code-seven-programmable-layers), these two systems sit at different levels of the stack. CLAUDE.md is a project-level layer that applies uniformly. Memory is a user-level layer that personalizes behavior on top of project rules.

## Content and Organization: What Goes Where

This is where most developers get confused. Both systems store text that influences Claude Code's behavior, but they're designed for categorically different information.

**CLAUDE.md should contain:**

- Build and test commands (`npm run build`, `npm test`, `npm run lint`)
- Quality gates and pre-commit requirements
- Coding style rules and architectural constraints
- File organization conventions
- Known gotchas and common mistakes specific to the codebase
- Documentation update rules
- Workflow expectations (e.g., "discuss design before implementing new features")

These are facts about the project that are true regardless of who's working on it. They rarely change, and when they do, the change should be reviewed and committed like any other code change.

**Memory should contain:**

- Your role and expertise level (senior backend engineer, data scientist, new to React)
- Communication preferences (terse responses, no trailing summaries, prefer code over explanation)
- Corrections you've given ("don't use that library," "always check the staging environment first")
- Ongoing project context (current sprint goals, who's working on what, upcoming deadlines)
- References to external systems (where bugs are tracked, which Slack channel to check, dashboard URLs)

These are facts about you and your current work context. They change frequently and don't belong in version control.

**The anti-pattern to avoid:** putting personal preferences in CLAUDE.md or putting project rules in Memory. If you add "I prefer short responses" to CLAUDE.md, every teammate gets that constraint — and they might not want it. If you store "always run tests before committing" only in Memory, a new teammate's Claude Code won't know that rule exists.

## Authoring and Maintenance: Effort Over Time

CLAUDE.md requires upfront investment. Someone on the team — usually a senior engineer or tech lead — needs to sit down and write out the project's conventions, commands, and constraints. This can take 30 minutes to several hours depending on the project's complexity. The [skills system](/blog/5-claude-code-skills-i-use-every-single-day) offers a complementary approach for task-specific instructions, but CLAUDE.md remains the foundation.

The ongoing maintenance cost is low but non-zero. When build commands change, when new quality gates are added, when architectural decisions are made — someone needs to update CLAUDE.md. The upside is that these updates go through code review, so the whole team sees and approves changes to how Claude Code behaves.

Memory requires zero upfront investment. It builds itself from your conversations. The first time you correct Claude Code, it remembers. The first time you mention your role, it files that away. Over weeks and months, your Memory accumulates into a detailed profile that makes Claude Code increasingly effective for your specific workflow.

The maintenance risk with Memory is staleness. Project memories about sprint goals or team assignments decay fast. A memory that says "the auth rewrite is the top priority" might be three months old and completely irrelevant. Claude Code is designed to verify memories against current state before acting on them, but stale memories still add noise. Periodically reviewing and pruning your `.claude/projects/` memory files keeps the system clean.

## Team Dynamics: Shared vs Individual

This is the most important practical difference between the two systems, and it drives most of the "which should I use?" decisions.

**CLAUDE.md creates team alignment.** When your CLAUDE.md says "all API endpoints must have integration tests," every developer's Claude Code enforces that rule. When it says "use snake_case for database columns," there's no drift across team members. The file is the single source of truth for how Claude Code should behave in your project, and it travels with the code.

**Memory creates individual optimization.** A junior developer's Memory might store extensive explanations about the codebase architecture, while a senior developer's Memory stores shortcuts and preferences that skip the basics. Both are valid — they just serve different users. This personalization is Memory's strength: Claude Code adapts to your level and style without anyone else needing to configure it.

The tension arises when teams want consistency but individuals want personalization. The answer is layering: CLAUDE.md handles the non-negotiable project rules, and Memory handles everything else. A developer can have a Memory that says "explain things concisely, I'm experienced with this codebase" without affecting the CLAUDE.md rule that says "run the full test suite before committing."

## When to Choose Claude Memory

Use Memory as your primary persistence layer when:

- **You're a solo developer.** Without teammates to align with, CLAUDE.md's version-control benefits matter less. Memory's automatic accumulation gives you personalization with zero authoring effort. You'll still want a minimal CLAUDE.md for build commands and quality gates, but Memory handles most of the context.

- **Your context is personal and dynamic.** Your role, your current focus areas, your communication preferences, which external tools you use — these are Memory's sweet spot. They change over time and don't affect anyone else.

- **You've been correcting Claude Code repeatedly.** If you keep telling Claude Code the same thing — "don't add comments to my code," "use integration tests, not mocks" — those corrections should be in Memory so they persist across sessions. Feedback memories are one of Memory's most valuable features.

- **You're tracking ongoing work.** Sprint goals, incident timelines, who's working on what feature — this operational context belongs in Memory with absolute dates, not in CLAUDE.md which is meant for stable project rules.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary persistence layer when:

- **You're on a team.** The moment more than one person uses Claude Code on a project, CLAUDE.md becomes essential. It ensures everyone's AI assistant follows the same rules, runs the same checks, and respects the same constraints.

- **Your rules are stable and non-negotiable.** Build commands, test requirements, coding standards, architectural constraints — these change infrequently and apply to everyone. CLAUDE.md is the right home because it's versioned, reviewable, and deterministic.

- **You need auditability.** Because CLAUDE.md is committed to git, you can see exactly when a rule was added, who added it, and why (via the commit message). Memory has no such audit trail — it's local files that evolve silently.

- **You're onboarding new team members.** A well-written CLAUDE.md means a new developer's first Claude Code session already knows the project's conventions. Without it, each new developer starts from zero and may get inconsistent behavior.

- **You want reproducible AI behavior.** If the same prompt should produce the same approach every time, regardless of who runs it, that logic belongs in CLAUDE.md. Memory introduces personal variation that may or may not be desirable.

## Verdict

**Use both — they're complementary, not competing.** CLAUDE.md is the shared foundation: project rules, build commands, quality gates, and coding standards that every team member's Claude Code must follow. Memory is the personal layer: your role, your preferences, your corrections, and your ongoing work context. The combination gives you consistent project behavior with individual optimization.

If you're forced to choose one, **start with CLAUDE.md**. A project without CLAUDE.md means Claude Code is guessing at your conventions. A project without Memory just means Claude Code doesn't know your personal preferences yet — less efficient, but not dangerous. CLAUDE.md prevents mistakes; Memory prevents repetition.

## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

No. CLAUDE.md instructions take precedence over Memory when there's a conflict. CLAUDE.md is loaded as project-level instructions that Claude Code treats as authoritative rules, while Memory provides supplementary context. If your Memory says "skip tests" but CLAUDE.md says "always run tests before committing," the CLAUDE.md rule wins.

### Does Claude Memory sync across devices?

No. Claude Memory is stored locally in your `.claude/projects/` directory and does not sync across machines. If you use Claude Code on both a laptop and a remote server, each environment maintains its own separate memory. CLAUDE.md, by contrast, syncs automatically through git when you push and pull.

### How do I migrate useful Memory entries into CLAUDE.md?

Review your `.claude/projects/` memory files periodically. Any feedback memory that reflects a project-wide rule — not just a personal preference — should be added to CLAUDE.md and removed from Memory. For example, if you corrected Claude Code about your test framework three times and it became a feedback memory, that's a signal to add a testing section to CLAUDE.md so the whole team benefits.

### Should I check Memory files into version control?

No. Memory files are personal and often contain information specific to one developer's context — their role, their preferences, their corrections. Checking them in would impose one person's working style on the entire team. Keep Memory local and use CLAUDE.md for shared conventions.

### How often should I update CLAUDE.md?

Update CLAUDE.md whenever project conventions change: new build steps, new quality gates, changed architectural rules, or newly discovered gotchas. Most active projects update CLAUDE.md a few times per month. Treat it like any other configuration file — change it deliberately, review the diff, and commit with a clear message.
