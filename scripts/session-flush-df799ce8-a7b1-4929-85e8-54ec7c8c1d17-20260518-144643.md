# Codex Subagents vs Custom Agents：多智能体编码的两条路线

**TL;DR：** Codex 中的 **Subagents** 是系统自动派生的子任务执行单元，适合标准化的并行编码场景；**Custom Agents** 是用户自定义的专用智能体，适合需要特定工具链、提示词策略或领域知识的复杂工作流。选 Subagents 图开箱即用，选 Custom Agents 图精细控制——多数团队最终会两者结合使用。

## 概述：Codex Subagents

**Subagents** 是 [OpenAI Codex](/zh/blog/codex-complete-guide) 内置的多智能体调度机制。当主 Agent 面对一个涉及多文件、多步骤的编码任务时，它会自动将任务拆解并派生多个 Subagent 并行执行。每个 Subagent 在独立的沙箱环境中运行，拥有自己的文件上下文和工具访问权限。

Subagents 的核心优势在于零配置。你不需要定义 Agent 的行为规范或工具集——Codex 根据任务语义自动决定拆分策略、上下文分配和结果合并。对于标准的代码重构、测试生成、多文件修改等场景，Subagents 能显著缩短执行时间。

这种模式与 [多智能体工作流](/zh/blog/con-u-pour-des-workflows-multi-agents) 的工程范式一致：将复杂任务分解为可并行的原子操作，由系统自动编排执行。

## 概述：Codex Custom Agents

**Custom Agents** 是用户在 Codex 生态中自行定义的专用智能体。通过配置自定义系统提示词（system prompt）、指定可用工具集、设置执行策略和约束条件，你可以创建针对特定任务的专家级 Agent。

Custom Agents 的价值在于可控性和专业化。当你的项目有特定的编码规范、安全要求或领域知识时，通用的 Subagent 可能无法满足需求。Custom Agent 允许你将这些约束编码为 Agent 的行为规范，确保输出质量符合预期。

结合 [Agent SDK](/zh/glossary/agent-sdk) 的能力，Custom Agents 可以集成外部工具、API 和数据源，构建超出标准编码范围的自动化工作流。

## 功能对比

| 特性 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **配置成本** | 零配置，系统自动调度 | 需要定义提示词、工具集、约束 | Subagents |
| **任务并行** | 原生支持，自动拆分 | 需手动编排多 Agent 协作 | Subagents |
| **行为可控性** | 由系统决定，用户控制有限 | 完全自定义，精细控制输出 | Custom Agents |
| **领域适配** | 通用编码能力 | 可注入领域知识和专业约束 | Custom Agents |
| **工具集成** | 使用 Codex 内置工具 | 可集成外部 API、MCP 服务等 | Custom Agents |
| **上下文管理** | 系统自动分配上下文 | 用户定义上下文范围和优先级 | Custom Agents |
| **学习曲线** | 低——开箱即用 | 中到高——需要理解 Agent 设计模式 | Subagents |
| **适合团队规模** | 个人到中型团队 | 有专职工程平台团队的组织 | 视情况 |

## 任务编排能力：深度分析

Subagents 和 Custom Agents 在任务编排上代表两种截然不同的哲学。

Subagents 采用**声明式编排**：你描述最终目标（"重构这个模块并更新所有相关测试"），系统自动决定如何拆分、并行执行、合并结果。这种模式的优势是降低了使用门槛——你不需要思考"应该派几个 Agent"或"每个 Agent 处理哪些文件"。对于 [Codex 新用户](/zh/blog/first-few-days-with-codex-cli) 来说，这是最自然的交互方式。

但声明式编排的代价是透明度不足。当 Subagent 的输出不符合预期时，你很难精确定位问题出在拆分策略、上下文分配还是某个子任务的执行逻辑。调试变成了黑箱操作。

Custom Agents 采用**命令式编排**：你明确定义每个 Agent 的职责边界、可用工具和执行约束。这需要更多前期设计工作，但换来的是完全可审计的执行链路。每个 Agent 的输入输出都在你的掌控之中。

在企业级场景中——尤其是涉及安全审计、合规要求或关键业务逻辑的代码修改——命令式编排的可审计性往往是硬性要求，而非可选特性。

## 定制化与扩展性：深度分析

Subagents 的定制化能力受限于 Codex 平台提供的配置项。你可以通过项目级的指令文件（如 `AGENTS.md`）向所有 Subagent 注入通用规则，但无法针对单个 Subagent 设置差异化行为。

Custom Agents 的扩展性则几乎没有上限。典型的高级用法包括：

- **代码审查 Agent**：注入团队编码规范，集成静态分析工具，自动标注不符合标准的模式
- **安全扫描 Agent**：集成 SAST/DAST 工具，根据项目威胁模型执行针对性检查
- **文档生成 Agent**：理解代码变更语义，自动更新 API 文档和变更日志
- **测试策略 Agent**：根据代码覆盖率数据和历史 Bug 分布，智能决定测试优先级

这种专业化能力是 [Agentic Coding](/zh/glossary/agentic-coding) 范式的核心价值所在。通用 Agent 完成 80% 的工作，专用 Agent 处理剩下 20% 的高价值场景。

## 什么时候选 Subagents

如果你符合以下情况，优先使用 Subagents：

- **快速上手**：刚开始使用 Codex，希望立即获得多智能体并行能力，不想花时间设计 Agent 架构
- **标准编码任务**：重构、批量修改、测试生成、跨文件重命名——这些任务的模式足够通用，系统自动调度就能胜任
- **小型团队或个人项目**：没有专职的工程平台团队来维护 Custom Agent 配置
- **探索阶段**：还在评估 Codex 的能力边界，先用 Subagents 跑出基线，再决定是否需要更精细的控制

Subagents 的最大优势是**认知负担低**。你不需要成为 Agent 设计专家就能享受多智能体编码的加速效果。

## 什么时候选 Custom Agents

如果你符合以下情况，投入 Custom Agents 的设计成本是值得的：

- **严格的代码质量要求**：团队有明确的编码规范、架构约束或安全标准，需要 Agent 严格遵守
- **特定领域知识**：项目涉及金融、医疗、法律等领域，Agent 需要理解领域术语和合规要求
- **复杂工具链集成**：需要 Agent 与内部 CI/CD 系统、监控平台或自研工具交互
- **可审计执行链路**：企业合规要求每个 AI 操作都可追溯、可审计
- **跨 Agent 协作流程**：多个专用 Agent 按固定流程协作（如：Code Agent → Review Agent → Test Agent → Deploy Agent）

Custom Agents 的 ROI 曲线是先高后低——前期设计和调优的投入会被后续每次使用摊薄。团队规模越大、使用频率越高，单次使用成本越低。

## 总结

**Subagents 和 Custom Agents 不是互斥选项，而是互补方案。** 最务实的策略是：用 Subagents 覆盖日常编码的大部分场景，用 Custom Agents 处理需要专业化控制的关键环节。

如果你刚接触 Codex 的多智能体能力，从 Subagents 开始。当你发现某类任务的输出质量反复不达标、或者需要集成外部工具时，再针对该场景构建 Custom Agent。这种渐进式策略既控制了前期投入，又保留了后续优化空间。

## 常见问题

### Subagents 能完全替代 Custom Agents 吗？

不能。Subagents 擅长标准化、可并行的编码任务，但缺乏针对特定领域或工具链的定制能力。当任务需要注入专业知识、集成外部工具或满足合规审计要求时，Custom Agents 是必要的补充。两者的定位不同，实际项目中通常结合使用。

### 构建 Custom Agent 需要多少工程投入？

取决于复杂度。一个简单的代码审查 Custom Agent（自定义提示词 + 规则集）可以在几小时内完成；一个集成了外部安全扫描工具和自动修复逻辑的 Agent 可能需要数天的设计和调优。关键是从最小可用版本开始，根据实际使用反馈迭代。

### Subagents 之间如何共享上下文？

Subagents 在独立沙箱中运行，通过主 Agent 进行上下文协调。主 Agent 负责任务拆分时的上下文分配，以及子任务完成后的结果合并。这意味着 Subagent 之间不直接通信——所有协调都通过主 Agent 中转。
