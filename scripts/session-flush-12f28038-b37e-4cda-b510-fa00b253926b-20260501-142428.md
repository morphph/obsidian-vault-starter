# Claude Memory vs CLAUDE.md：哪个才是你真正需要的上下文系统？

**TL;DR：** **CLAUDE.md** 是写给 Claude Code 的规则手册——团队共享、版本控制、确定性执行。**Claude Memory（Auto Memory）** 是 Claude Code 自动积累的个人笔记——跨会话记忆偏好、反馈和项目上下文。两者不是竞争关系，而是互补的两层：**CLAUDE.md 管「该怎么做」，Memory 管「你是谁、上次聊了什么」**。团队协作优先配 CLAUDE.md，个人效率优先用好 Memory。

## 概述：Claude Memory

**Claude Memory**（也叫 Auto Memory）是 Claude Code 的持久化记忆系统，自动将对话中的关键信息保存到本地文件系统（`~/.claude/projects/` 目录下）。它解决的问题是：每次打开新会话，Claude Code 都像失忆了一样，你得重新解释自己是谁、项目背景是什么、上次讨论到哪了。

Memory 分为四种类型：**user**（用户角色和偏好）、**feedback**（工作方式的纠正和确认）、**project**（项目动态和决策）、**reference**（外部资源指针）。它们以 Markdown 文件形式存储，通过 `MEMORY.md` 索引文件组织。Memory 是个人化的——只存在于你的本地机器上，不进 Git，不与团队共享。

关于 Claude 记忆系统的最新能力升级，可以参考我们的详细分析：[Claude 记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)。

## 概述：CLAUDE.md

**CLAUDE.md** 是放在项目根目录的指令文件，相当于给 Claude Code 写的一份「项目宪法」。每次 Claude Code 启动会话时，都会自动读取这个文件，获取项目级别的上下文和约束规则。

CLAUDE.md 的内容通常包括：技术栈说明、构建命令、代码风格约定、提交规范、已知坑点、工作流程要求等。它是确定性的——你写什么，Claude Code 就严格遵守什么。因为它是普通的 Markdown 文件，所以天然支持 Git 版本控制，整个团队共享同一份规则。

CLAUDE.md 是 [Claude Code 七层可编程架构](/zh/blog/claude-code-seven-programmable-layers) 中最基础也最重要的一层。配合 `SKILL.md` 文件，它构成了 Claude Code 行为控制的核心机制。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **存储位置** | `~/.claude/projects/` 本地目录 | 项目根目录，进 Git | CLAUDE.md |
| **共享范围** | 仅个人 | 团队所有成员 | CLAUDE.md |
| **创建方式** | Claude Code 自动生成 | 开发者手动编写 | Memory |
| **内容类型** | 用户偏好、反馈、项目动态 | 规则、命令、约定、约束 | 各有侧重 |
| **版本控制** | 不受 Git 管理 | Git 版本控制 | CLAUDE.md |
| **确定性** | 模型判断何时保存和调用 | 每次会话确定性加载 | CLAUDE.md |
| **维护成本** | 几乎零（自动） | 需要手动维护更新 | Memory |
| **跨项目** | 全局 Memory 跨项目可用 | 仅限当前项目 | Memory |

## 核心差异深度分析：确定性 vs 自适应

这是两个系统最根本的区别。

**CLAUDE.md 是确定性的规则引擎。** 你在文件里写「commit message 必须用中文」，Claude Code 就会在每次提交时严格遵守。不需要猜测、不需要学习、不需要等到第三次对话才记住。这种确定性对团队协作至关重要——新成员 clone 仓库后，Claude Code 立刻知道该遵守什么规则。

**Memory 是自适应的学习系统。** 它的价值在于积累那些无法（或不该）写成规则的上下文。比如「这个用户是数据科学家，解释代码时侧重数据流而非系统架构」、「上次讨论中用户否决了引入 Redis 的方案，原因是运维成本」。这些信息高度个人化、情境化，放在 CLAUDE.md 里既不合适也不可行。

实际使用中的关键差异：CLAUDE.md 的规则在每次会话开始时 100% 加载，而 Memory 的调用取决于模型的判断——它可能在相关时调出记忆，也可能遗漏。这意味着对于必须执行的硬性规则，CLAUDE.md 是唯一可靠的选择。详细的架构拆解请参考 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 作用域与生命周期分析

**CLAUDE.md 有三层作用域**，从宽到窄：

1. **全局层**（`~/.claude/CLAUDE.md`）：影响所有项目，适合放个人通用偏好
2. **项目层**（项目根目录 `CLAUDE.md`）：团队共享，放构建命令、代码规范、已知坑点
3. **目录层**（子目录中的 `CLAUDE.md`）：局部覆盖，用于特殊模块的特殊规则

这种层级设计让你可以精确控制规则的影响范围。

**Memory 也有作用域**，但逻辑不同：

1. **全局 Memory**（`~/.claude/memory/`）：跨项目的用户画像和通用偏好
2. **项目 Memory**（`~/.claude/projects/<project>/memory/`）：特定项目的上下文积累

Memory 没有子目录级别的粒度。它的生命周期也不同——CLAUDE.md 跟着代码库的生命周期走（代码在它就在），Memory 跟着用户走（换电脑就没了，除非手动迁移）。

## 什么时候选 CLAUDE.md

以下场景，CLAUDE.md 是明确的首选：

- **团队规范统一**：代码风格、提交格式、构建流程、质量门禁——任何需要团队所有人（包括他们的 Claude Code）都遵守的规则
- **项目入职**：新成员 clone 仓库后，Claude Code 立刻获得完整的项目上下文，不需要「调教」
- **关键约束**：「不要直接 import Next.js 模块到 pipeline 脚本」「ZH 内容必须用 CJK 分词计数」——这类容易踩的坑，必须每次会话都强制执行
- **可审查的 AI 行为**：CLAUDE.md 进版本控制，意味着 AI 的行为规则可以 code review、可以回溯历史变更

想了解更多实战用法，可以参考[我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

## 什么时候选 Claude Memory

以下场景，Memory 更合适：

- **个人工作偏好**：「我习惯简短回复，不要总结」「解释技术概念时用类比」——这些是个人风格，不该强加给团队
- **项目动态信息**：「本周四开始 merge freeze」「auth 模块重写是因为合规要求」——这类时效性强的信息放在 CLAUDE.md 里会很快过时
- **纠错与反馈**：「上次你在测试中 mock 了数据库，结果线上出问题了，以后不要 mock」——Memory 会自动记录这类反馈并在未来会话中应用
- **跨项目上下文**：你的角色、技术背景、沟通风格——这些全局 Memory 在任何项目中都有用

## 最佳实践：组合使用

真正高效的 Claude Code 用户不会二选一，而是让两个系统各司其职：

**CLAUDE.md 管规则**——确定性的、团队共享的、必须执行的。每个项目都应该有一份维护良好的 CLAUDE.md，内容包括构建命令、代码规范、已知限制和工作流程。参考 [Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills) 了解如何写好指令文件。

**Memory 管记忆**——个人化的、自适应的、无法预先规定的。让 Memory 自然积累，偶尔检查 `MEMORY.md` 索引，清理过时条目。

**避免重复**：如果某条信息同时出现在 CLAUDE.md 和 Memory 中，当它们产生冲突时会造成混乱。原则是：团队级规则只放 CLAUDE.md，个人级上下文只放 Memory。

关于 [Claude Code 记忆系统的完整机制](/zh/blog/claude-code-memory)，我们有专门的深度解析文章。

## 结论

**CLAUDE.md 和 Claude Memory 不是竞争对手，而是上下文系统的两个层次。** CLAUDE.md 是显式的项目规则层——确定性加载、团队共享、版本控制。Memory 是隐式的个人上下文层——自动积累、个人专属、跨会话持续。

如果你只能做一件事：**先写好 CLAUDE.md**。它的投入产出比最高——一次编写，团队所有人的 Claude Code 立刻受益。Memory 可以慢慢积累，但没有 CLAUDE.md 的项目，每次 Claude Code 都在裸跑。

## 常见问题

### Claude Memory 会不会覆盖 CLAUDE.md 的规则？

不会。CLAUDE.md 在每次会话开始时确定性加载，优先级高于 Memory。如果 CLAUDE.md 规定了「commit message 用英文」，即使 Memory 记录了用户偏好中文，Claude Code 也会遵守 CLAUDE.md 的规则。两者冲突时，CLAUDE.md 胜出。

### 我需要手动管理 Claude Memory 吗？

通常不需要。Claude Code 会自动判断哪些信息值得保存。但定期检查 `MEMORY.md` 索引文件是个好习惯——清理过时条目、删除不再准确的记忆。你也可以通过对话显式要求 Claude Code 记住或忘记特定信息。

### CLAUDE.md 可以放在子目录里吗？

可以。Claude Code 支持目录级别的 CLAUDE.md，子目录的规则会补充（不是覆盖）上层规则。这在 monorepo 中特别有用——根目录放通用规则，各子项目放特定规则。
