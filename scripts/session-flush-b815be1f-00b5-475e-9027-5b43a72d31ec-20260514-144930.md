**Claude Code's subagent model** is built directly into the agent runtime. When Claude Code encounters a task that benefits from decomposition, it can spawn subagents using the `Agent` tool without any external framework. Each subagent receives a prompt describing its task, runs with access to the same project files, and returns its result to the parent agent. The parent can then synthesize results and continue.

Built-in subagent types include:

- **Explore**: Fast read-only search agent for locating code, symbols, and patterns across a codebase. Optimized for breadth rather than deep analysis.
- **Plan**: Software architect agent that designs implementation strategies, identifies critical files, and considers tradeoffs before any code is written.
- **code-reviewer**: Dedicated review agent for examining code changes against project standards.
- **general-purpose**: Catch-all agent for research, multi-step tasks, and anything that doesn't fit a specialized type.

This specialization means the parent agent can dispatch the right kind of subagent for each subtask — a search agent to find relevant code, a planning agent to design the approach, and a general-purpose agent to implement the changes — all within a single session.

## Custom Agent Configuration: Detailed Analysis

Custom agents represent where the two platforms diverge most sharply in terms of developer experience and team workflows.

**In Codex**, creating a custom agent means writing Python code using the Agents SDK. You define an `Agent` class with a name, instructions, model, and tools. For multi-agent systems, you configure `Handoff` objects that specify when and how control passes between agents. This gives you full programmatic control — you can implement complex routing logic, conditional handoffs, and custom tool integrations. The tradeoff is that your agent definitions live in application code, require Python expertise, and are separate from your project repository unless you deliberately integrate them.

Here's what the Agents SDK pattern looks like:

```python
from agents import Agent, Handoff

reviewer = Agent(
    name="code_reviewer",
    instructions="Review code for security issues and style violations.",
    model="gpt-4.1",
)

implementer = Agent(
    name="implementer",
    instructions="Implement the requested feature following project conventions.",
    handoffs=[Handoff(target=reviewer, condition="after implementation is complete")],
)
```

**In Claude Code**, custom agents are markdown files placed in `.claude/agents/` within your repository. Each file defines an agent's name, description, capabilities, and behavioral instructions using frontmatter and natural language. Because these files live in your repo, they are version-controlled, reviewable in PRs, and automatically available to every team member who uses Claude Code on the project.

A custom agent definition in Claude Code looks like this:

```markdown
---
name: security-scanner
description: Scans code changes for OWASP Top 10 vulnerabilities
tools: [Read, Grep, Glob, Bash]
---

Review all changed files for security vulnerabilities.
Focus on: SQL injection, XSS, command injection, path traversal.
Report findings with file path, line number, and severity.
```

The markdown-based approach has several practical advantages for teams. Non-engineers can read and understand agent definitions. Agent behavior changes go through the same PR review process as code changes. And agents can reference project-specific context — coding standards, architecture decisions, known patterns — using the same conventions as [CLAUDE.md and SKILL.md files](/blog/claude-code-extension-stack-skills-hooks-agents-mcp).

The tradeoff is flexibility. The Agents SDK gives you Turing-complete control over agent behavior, routing, and coordination. Claude Code's markdown agents are more constrained — you define what the agent should do, but the runtime handles how it executes. For most software engineering workflows, the markdown approach covers the common cases with dramatically less setup. For complex, multi-step orchestration with conditional branching, the Agents SDK offers more power at the cost of more code.

## Isolation and Safety: Detailed Analysis

Both tools provide mechanisms to prevent agents from causing unintended damage, but their approaches reflect their architectural differences.

**Codex runs every task in a cloud sandbox.** Each container gets a fresh clone of your repository, a controlled set of environment variables, and network restrictions. Tasks cannot access your local filesystem, cannot reach arbitrary network endpoints, and cannot persist state between runs. This is the strongest isolation model of any coding agent — a misbehaving task cannot corrupt your local environment or interfere with other tasks. The result is always delivered as a proposed change (typically a pull request) that you review before merging.

**Claude Code isolates subagents using git worktrees.** When you spawn an agent with `isolation: "worktree"`, Claude Code creates a temporary linked working tree — a separate checkout of your repository where the agent can make changes without affecting your main working directory. If the agent makes no changes, the worktree is automatically cleaned up. If it does make changes, you get back the worktree path and branch name to review. This provides meaningful isolation without leaving your local machine.

Without worktree isolation, Claude Code subagents share the parent's working directory. This enables richer collaboration — a search agent can find relevant code that the implementation agent then modifies — but requires trust in the agent's behavior. Claude Code mitigates this through its permission system, where tool calls can require user approval before execution.

The practical difference: Codex's isolation is stronger but more rigid. You cannot have two Codex tasks collaborate on the same working tree. Claude Code's isolation is more flexible — you choose the isolation level per agent — but puts more responsibility on the developer to configure appropriate boundaries.

## Practical Workflow: Multi-Agent Refactoring

To illustrate the difference concretely, consider a common task: refactoring an authentication module across a large codebase. The task involves finding all auth-related code, designing a new architecture, implementing changes, and updating tests.

**With Codex**, you would either:
1. Submit the entire task as a single Codex task and let it handle everything in one container, or
2. Break it into separate tasks manually — one to audit current auth code, one to implement changes, one to update tests — and coordinate the results yourself

Option 1 is simpler but gives you less control over the approach. Option 2 gives you more control but requires you to manage the coordination, passing context from the audit task's output into the implementation task's prompt.

**With Claude Code**, the agent can decompose this automatically within a single session:
1. Spawn an **Explore** subagent to find all auth-related files and patterns
2. Spawn a **Plan** subagent to design the refactoring approach based on the Explore agent's findings
3. After reviewing the plan, spawn multiple **general-purpose** agents in parallel — each handling a different module — with worktree isolation
4. Review the changes across worktrees and merge

The key difference is that steps 1-4 happen within a single interactive session. You can review intermediate results, redirect the approach, and maintain conversational context throughout. With Codex, each task is a fire-and-forget operation — you define the task, submit it, and review the result.

## When to Choose OpenAI Codex

Choose Codex for subagent-style workflows when:

- **You need maximum isolation.** Codex's cloud sandbox model is the strongest isolation available in any coding agent. If you're working in a regulated environment where agents must not access local filesystems or network resources, Codex's container-based approach is the safer default.

- **You're already invested in the OpenAI ecosystem.** If your team uses GPT-4.1, the Agents SDK, and OpenAI's API infrastructure, building multi-agent orchestration on top of Codex keeps your stack unified. The [Agents SDK](/glossary/agent-sdk) provides a well-documented framework for custom agent systems.

- **Your tasks are naturally independent.** If your workflow consists of separate, parallelizable tasks that don't need to share intermediate state — running the same refactoring across multiple microservices, or generating test suites for independent modules — Codex's task model maps cleanly to this pattern without requiring inter-agent communication.

- **You want asynchronous, fire-and-forget execution.** Codex tasks run in the cloud and deliver results when ready. You can submit a batch of tasks and review the results hours later. This suits teams that prefer async code review workflows over interactive agent sessions.

## When to Choose Claude Code

Choose Claude Code for subagent workflows when:

- **You need real-time coordination between agents.** Claude Code's subagents can share findings, delegate subtasks, and build on each other's work within a single session. For tasks where the output of one agent informs the input of another — analysis followed by implementation followed by testing — the built-in coordination is dramatically more efficient than external orchestration.

- **You want team-shareable agent definitions without writing code.** Custom agents defined in `.claude/agents/*.md` files are readable, reviewable, and version-controlled. Any team member can create or modify agent definitions without Python expertise. This lowers the barrier to adopting multi-agent workflows across an engineering organization.

- **You need interactive control during multi-agent execution.** Claude Code lets you review intermediate results, redirect agents, and make decisions mid-workflow. If the Explore agent finds an unexpected dependency, you can adjust the Plan agent's approach before any implementation begins. Codex's task model doesn't support this kind of interactive steering.

- **You're doing local development work.** Claude Code runs in your terminal with direct access to your development environment — local databases, running services, environment configuration. Subagents inherit this context. Codex's cloud sandbox is isolated from your local environment by design, which means agents cannot interact with locally running services.

- **You want built-in agent specialization.** Rather than defining agent types from scratch, Claude Code ships with purpose-built agent types (Explore, Plan, code-reviewer) that are optimized for their specific task. This means effective multi-agent workflows with zero configuration.

## Frequently Asked Questions

### Can OpenAI Codex spawn subagents within a single task?

No. Each Codex task runs as an independent, isolated unit in its own cloud sandbox. To create multi-agent workflows, you need to use the OpenAI Agents SDK to build external orchestration that dispatches and coordinates multiple Codex tasks. The tasks themselves cannot spawn child agents or communicate with other running tasks.

### How do you define a custom agent in Claude Code?

Create a markdown file in your project's `.claude/agents/` directory with frontmatter specifying the agent's name, description, and available tools, followed by natural-language instructions for the agent's behavior. These files are version-controlled with your repository and automatically available to all team members using Claude Code on the project.

### Is the OpenAI Agents SDK the same as Codex?

No. The Agents SDK is a separate Python framework for building multi-agent systems using OpenAI's models. Codex is a cloud-based coding agent product. You can use the Agents SDK to orchestrate Codex tasks, but they are distinct products with different purposes. The [Agents SDK](/glossary/agent-sdk) handles agent coordination; Codex handles code generation and modification.

### Can Claude Code subagents run in parallel?

Yes. You can spawn multiple subagents in a single message, and they execute concurrently. Each agent can optionally run in the background with the `run_in_background` flag, allowing the parent agent to continue working while subagents complete their tasks. Results are returned when each agent finishes.

### Which tool is better for enterprise multi-agent workflows?

It depends on your constraints. Codex offers stronger isolation through cloud sandboxes, which suits compliance-heavy environments. Claude Code offers richer coordination and lower setup overhead through its built-in agent system, which suits teams that need interactive, multi-step agent workflows without building custom infrastructure.
