# Claude Memory vs CLAUDE.md：哪个才是你需要的持久化方案？

**TL;DR:** **Claude Memory**（Auto Memory）是 Claude Code 的自动笔记系统，跨会话记住你的偏好和项目上下文。**CLAUDE.md** 是显式的指令文件，版本控制、团队共享、确定性执行。**团队协作和工程规范选 CLAUDE.md**；**个人习惯和动态上下文选 Memory**。大多数认真使用 Claude Code 的开发者两个都用。

## 概述：Claude Memory

Claude Memory 是 Claude Code 内置的自动记忆系统，存储在 `~/.claude/projects/<project>/memory/` 目录下。每次对话中，Claude 会自动识别值得记住的信息——你的角色、技术偏好、项目背景、反馈修正——并写入独立的 Markdown 文件。下次开启新会话时，这些记忆会被自动加载到上下文中。

Memory 的核心特征是**被动积累**。你不需要主动维护它，Claude 在对话过程中自行判断哪些信息值得持久化。它适合存储那些难以提前预知但一旦学到就应该记住的知识：你喜欢简洁的回复风格、你的项目用 pnpm 而不是 npm、上次那个 bug 的根因是什么。

Anthropic 在 2026 年初对[记忆系统做了重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，免费用户也能使用，并且支持从 ChatGPT 导入历史数据。这让 Memory 从一个实验特性变成了正式的生产力工具。

## 概述：CLAUDE.md

**CLAUDE.md** 是放在项目根目录（或 `~/.claude/` 全局目录）的 Markdown 文件，包含你希望 Claude Code 每次都遵守的显式指令。它是 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中最基础的一层——项目级上下文注入。

CLAUDE.md 的核心特征是**主动声明**。你手动编写规则，它们会被确定性地加载到每次会话的系统提示中。典型内容包括：构建命令、代码风格要求、Git 工作流规范、已知坑点、架构约束。因为是普通文件，它天然支持 Git 版本控制，团队成员 `git pull` 就能同步最新规则。

与 Memory 不同，CLAUDE.md 不会自动生成内容。每一条规则都是你（或团队）有意识地写下的工程决策。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **创建方式** | AI 自动识别并写入 | 开发者手动编写 | 各有场景 |
| **存储位置** | `~/.claude/projects/` | 项目根目录 / `~/.claude/` | CLAUDE.md |
| **版本控制** | 不在 Git 中 | 随项目代码一起版本控制 | CLAUDE.md |
| **团队共享** | 仅个人可见 | 团队共享、PR 可审查 | CLAUDE.md |
| **内容类型** | 偏好、反馈、动态上下文 | 规则、命令、架构约束 | 各有场景 |
| **确定性** | 概率性——AI 决定记什么 | 确定性——写什么加载什么 | CLAUDE.md |
| **维护成本** | 几乎为零 | 需要手动更新 | Memory |
| **可审计性** | 低——散落在多个文件中 | 高——单文件、可 diff | CLAUDE.md |

## 持久化机制：详细分析

这是两者最根本的区别。理解这一点，后面的选择就清晰了。

**CLAUDE.md 是声明式的**。你写下 `npm run build` 是构建命令、提交前必须跑测试、不要用 `any` 类型——这些规则每次会话都会被完整加载，不存在"忘记"的可能。它的行为模式类似于 `.eslintrc` 或 `.editorconfig`：团队约定，机器执行。

**Memory 是归纳式的**。Claude 在对话中观察你的行为和反馈，提取模式，写入记忆文件。比如你纠正了它三次"不要在测试里 mock 数据库"，它会保存这条反馈记忆。但这个过程是概率性的——同样的对话，Claude 可能选择记住不同的细节。

实际影响：如果一条规则对代码质量至关重要（比如"所有 API 路由必须做输入校验"），放在 CLAUDE.md 里。如果是个人偏好（比如"我喜欢用 `const` 而不是 `let`"），让 Memory 自动学习就行。

在 [Claude Code 的扩展栈架构](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中，CLAUDE.md 和 Memory 分别处于不同的层级：CLAUDE.md 属于基础上下文层，Memory 属于个性化适配层。两者协同工作，而不是互相替代。

## 团队协作：详细分析

这是 CLAUDE.md 的绝对优势领域。

CLAUDE.md 是一个普通文件，提交到 Git 仓库中。新成员 clone 项目后，Claude Code 自动加载团队的所有工程规范。修改规则通过 PR 审查，有变更历史，有回滚能力。这意味着团队对"AI 应该怎么写代码"这个问题有了一个可版本化的共识文档。

Memory 是纯个人的。它存储在你本地的 `~/.claude/` 目录下，不会被 push 到远程仓库。你的 Memory 里可能记着"这个项目的部署用的是 Vercel"，但新同事的 Memory 里什么都没有。

结合 [Claude Code Skills](/zh/blog/9-principles-writing-claude-code-skills) 体系来看，CLAUDE.md 定义"是什么"和"不能做什么"，SKILL.md 定义"怎么做"，Memory 定义"这个人喜欢怎么做"。三者各有分工。

对于团队场景，推荐的做法是：

1. 把所有强制性规则放进 CLAUDE.md（构建命令、代码规范、Git 工作流）
2. 把任务级模板放进 SKILL.md（PR 审查流程、测试生成模板）
3. 让 Memory 自然积累个人偏好（回复风格、常用快捷操作）

## 适用场景：选择 Claude Memory

Memory 在以下场景中不可替代：

- **个人工作习惯**：你偏好简洁回复、总是用特定的 Git commit 格式、喜欢先看测试再看实现——这些细碎的偏好不值得写成正式规则
- **动态项目上下文**：当前正在做的功能、上次调试到哪里了、谁负责哪个模块——这些信息变化快，手动更新 CLAUDE.md 效率太低
- **反馈修正**：你纠正 Claude 的某个行为后，Memory 自动记住，下次不会再犯。不需要你去 CLAUDE.md 里加一条"不要做 X"
- **跨项目个人偏好**：全局 Memory（`~/.claude/` 级别）让你的偏好在所有项目中生效

如果你是独立开发者，Memory 可能覆盖你 80% 的持久化需求。搭配几条关键的 CLAUDE.md 规则，就足够高效运作了。参考[这些实际技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)可以快速上手。

## 适用场景：选择 CLAUDE.md

CLAUDE.md 在以下场景中不可替代：

- **团队工程规范**：代码风格、分支策略、审查流程——必须所有人一致遵守的规则
- **构建和部署命令**：`npm run build`、`npm test`、部署脚本——这些命令不应该每次都重新推断
- **已知坑点**：项目中的技术债、特定模块的注意事项、容易踩的坑——写下来比每次都踩一遍强
- **质量门禁**：提交前必须通过的检查清单——构建成功、测试通过、lint 无报错
- **架构约束**：不要直接 import 某些模块、某些函数的参数约定、数据库 schema 变更流程

CLAUDE.md 的核心价值是**可靠性**。它不依赖 AI 的判断来决定记不记住，写了就生效，每次都生效。对于任何"违反了会出问题"的规则，CLAUDE.md 是唯一合理的选择。

## 结论

**两者不是竞争关系，而是互补关系。** CLAUDE.md 是你的工程规范手册——确定性、可版本化、团队共享。Memory 是你的个人助理笔记——自动积累、个性适配、零维护成本。

实际建议：**先写好 CLAUDE.md**。把构建命令、代码规范、已知坑点写进去，这是基础设施。然后让 Memory 自然生长，它会填补 CLAUDE.md 覆盖不到的个性化空间。如果你发现自己反复在 Memory 中看到同一条信息，考虑把它提升为 CLAUDE.md 中的正式规则。

## 常见问题

### Claude Memory 和 CLAUDE.md 可以同时使用吗？

可以，而且推荐同时使用。CLAUDE.md 提供确定性的项目规则和团队规范，Memory 提供个性化的偏好适配。两者在每次会话启动时都会被加载到上下文中，互不冲突。大多数高效的 Claude Code 用户两个都在用。

### CLAUDE.md 的内容会自动更新吗？

不会。CLAUDE.md 是一个普通的 Markdown 文件，需要开发者手动编辑和维护。这正是它的优势——每条规则都是有意识的工程决策，通过 Git 版本控制和 PR 审查来管理变更。Memory 则由 Claude 自动维护，无需手动干预。

### Memory 记住了错误的信息怎么办？

你可以直接告诉 Claude"忘掉关于 X 的记忆"，它会找到并删除对应的记忆文件。也可以手动到 `~/.claude/projects/<project>/memory/` 目录下编辑或删除文件。定期审查 Memory 内容是个好习惯，特别是在项目技术栈发生重大变化之后。
