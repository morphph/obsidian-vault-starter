# Claude Memory vs CLAUDE.md：两套记忆系统到底该用哪个？

**TL;DR：** **CLAUDE.md** 是你手写的项目规则手册，确定性加载、团队共享、版本控制；**Claude Memory**（auto memory）是 Claude Code 自动积累的个人笔记，跨会话持久化、用户私有、基于对话学习。两者不是竞争关系——**CLAUDE.md 管「项目该怎么做」，Memory 管「这个人是谁、喜欢什么」**。最佳实践是同时使用：CLAUDE.md 写团队规范，Memory 存个人偏好和上下文。

## 概述：Claude Memory

Claude Memory 是 Claude Code 的自动持久化记忆系统，存储在 `~/.claude/projects/` 目录下。它解决的核心问题是：**每次新会话都要重复告诉 Claude 你是谁、你喜欢什么风格、项目有哪些特殊约定**。

Memory 通过对话自动学习——当你纠正 Claude 的做法（"别用 mock 测数据库"）或确认某个方案有效时，Claude 会将这些信息写入 memory 文件。下次对话自动加载，无需重复。Memory 分为四种类型：用户画像（user）、行为反馈（feedback）、项目上下文（project）和外部引用（reference）。

关键特征：**用户私有**，不进 git，不与团队共享。这是你和 Claude 之间的私人笔记本。

## 概述：CLAUDE.md

**CLAUDE.md** 是放在项目根目录（或子目录）的 Markdown 文件，Claude Code 每次启动时确定性加载。它本质上是一份**给 AI 看的项目规范**——编码风格、构建命令、架构决策、禁止事项。

CLAUDE.md 的设计哲学参考了 [Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中的「项目上下文层」：它是团队共享的、版本控制的、确定性的。写在 CLAUDE.md 里的规则，团队每个人的 Claude Code 都会遵守。它和代码一起提交、一起 review、一起演进。

关键特征：**团队共享**，进 git，所有协作者的 Claude Code 都会读取并遵守。

## 功能对比

| 维度 | Claude Memory | CLAUDE.md | 优势方 |
|------|--------------|-----------|--------|
| **存储位置** | `~/.claude/projects/` 本地目录 | 项目根目录，进 git | 视场景 |
| **加载方式** | 自动加载相关记忆 | 确定性全量加载 | CLAUDE.md |
| **创建方式** | 对话中自动学习 + 手动触发 | 手动编写维护 | Memory |
| **共享范围** | 用户私有 | 团队共享 | 视场景 |
| **版本控制** | 无 | Git 管理 | CLAUDE.md |
| **内容类型** | 偏好、反馈、人物画像 | 规范、命令、架构约定 | 视场景 |
| **可审计性** | 低——分散在多个小文件中 | 高——单文件、可 code review | CLAUDE.md |
| **维护成本** | 低——自动积累 | 中——需要手动更新 | Memory |

## 核心差异：确定性 vs 学习性

这是两套系统最本质的区别。

CLAUDE.md 是**确定性的**：你写什么，Claude 就看到什么，每次都一样。如果你写了 `npm run build` 是构建命令，Claude 不会「忘记」这条规则，也不会根据心情做不同解读。这种确定性对团队协作至关重要——你不希望同事的 Claude 用不同的 lint 规则。

Memory 是**学习性的**：它从交互中提取模式。你连续三次告诉 Claude "commit message 用中文"，它会记住这个偏好。但这个记忆是概率性的——Claude 判断某条信息"值得记住"才会存储，且只对你个人生效。

实际使用中的一个常见误区：把本该写在 CLAUDE.md 里的硬性规则告诉 Memory。比如"这个项目必须用 Vitest 不能用 Jest"——这条规则应该写进 CLAUDE.md 让全团队生效，而不是靠 Memory 让 Claude「记住」。关于 Memory 系统的完整机制，参考 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 内容分工：什么该写在哪里

理解两者的分工逻辑后，决策就清晰了：

**写在 CLAUDE.md 里的内容：**
- 构建、测试、部署命令
- 代码风格和架构约定
- 项目特有的技术限制（"不要在 pipeline 脚本里 import Next.js 模块"）
- 团队工作流规范（"提交前必须通过所有测试"）
- 目录结构说明和关键文件索引

**留给 Memory 的内容：**
- 个人身份和角色（"我是后端工程师，不熟悉前端"）
- 沟通偏好（"不要在回复末尾总结"、"用中文交流"）
- 个人工作习惯（"我喜欢一个 PR 解决一个问题"）
- 跨项目的通用偏好（"commit message 用英文"）

一个简单的判断标准：**如果这条信息换个人也应该遵守，写 CLAUDE.md；如果只对你个人有意义，留给 Memory。**

## 协作场景深入分析

在真实的团队开发中，两套系统的协作模式体现在多个层面。

**场景一：新人入职**。项目的 CLAUDE.md 已经写好了构建命令、架构约定和编码规范，新人的 Claude Code 开箱即用。随着新人与 Claude 的交互，Memory 会逐渐学习到这个人的技术背景（"熟悉 Go 但第一次写 React"）和偏好（"喜欢先看测试再看实现"），提供越来越个性化的协助。

**场景二：跨项目开发**。你在三个项目之间切换，每个项目有自己的 CLAUDE.md 定义各自的规范。但你的 Memory 跨项目生效——你的身份、偏好、沟通风格始终一致。这正是两层设计的优雅之处。

**场景三：Skill 和 Hook 的配合**。在 [Claude Code 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中，CLAUDE.md 处于基础层，Skills 和 Hooks 构建在其之上。Memory 则独立于这个技术栈运行，充当跨会话的上下文补充。当你使用 [SKILL.md 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)时，CLAUDE.md 定义「怎么执行 Skill」，Memory 记住「这个用户上次执行这个 Skill 时的反馈和调整」。

## 适用场景：选择 Claude Memory

选择依赖 Memory 的场景：

- **个人开发者独立项目**：不需要团队协作，Memory 的自动学习足够覆盖大部分需求，省去手写 CLAUDE.md 的维护成本
- **跨项目通用偏好**：你希望所有项目中 Claude 都记住你的风格——这只有 Memory 能做到
- **快速迭代探索期**：项目初期规范还没定型，先靠 Memory 积累经验，等模式稳定后再提炼进 CLAUDE.md
- **非技术用户**：不想手动维护配置文件，[用自然对话的方式](/zh/blog/how-to-effectively-prompt-a-claude-code)让 Claude 自动学习偏好

## 适用场景：选择 CLAUDE.md

选择重点维护 CLAUDE.md 的场景：

- **团队项目**：多人协作必须有统一规范，Memory 的个人性质无法满足
- **CI/CD 环境**：自动化流水线中 Claude Code 没有个人 Memory，只能依赖 CLAUDE.md
- **高合规要求**：需要审计和版本控制的环境，CLAUDE.md 的 Git 管理提供完整变更历史
- **开源项目**：贡献者使用自己的 Claude Code 时，CLAUDE.md 确保一致的代码质量标准
- **复杂架构项目**：当项目有大量非显而易见的约定时（参考 [Claude Code 不只是编程工具](/zh/blog/claude-code-is-not-a-coding-tool)的讨论），CLAUDE.md 是传递这些隐性知识的最可靠方式

## 结论

**CLAUDE.md 和 Memory 不是二选一的关系，而是两个互补的层。** CLAUDE.md 是项目的宪法——确定性、共享、可审计。Memory 是个人助手的笔记本——自动、私有、跨项目。最佳实践很简单：团队规范写 CLAUDE.md，个人偏好交给 Memory。两者同时使用时，Claude Code 既遵守项目规范，又理解你个人的工作风格。

如果你正在纠结从哪里开始，**先写 CLAUDE.md**。哪怕只写三行——构建命令、测试命令、一条最重要的代码规范——就能显著提升 Claude Code 的输出质量。Memory 会在后续使用中自然积累，不需要刻意维护。

## 常见问题

### Claude Memory 和 CLAUDE.md 会冲突吗？

不会直接冲突，但可能出现信息不一致。CLAUDE.md 的优先级更高——它作为系统指令加载，Memory 作为参考上下文。如果 CLAUDE.md 写了"用 ESLint"而 Memory 记录了"用户说用 Biome"，Claude 会优先遵守 CLAUDE.md。最佳做法是将确定性规则统一写进 CLAUDE.md，避免通过 Memory 传递硬性约定。

### Memory 存了哪些内容，怎么查看和管理？

Memory 文件存储在 `~/.claude/projects/` 对应项目目录下，以 Markdown 格式保存。你可以直接阅读和编辑这些文件，也可以在对话中告诉 Claude "忘记某条记忆"来删除特定条目。每个 memory 文件包含 frontmatter 元数据（名称、描述、类型）和具体内容。

### 团队中每个人的 Memory 不同，会导致输出不一致吗？

会有差异，但通常是良性的。团队统一行为由 CLAUDE.md 保证，个人差异（如偏好的解释详细程度、commit message 风格）由 Memory 个性化。这类似于团队有统一编码规范，但每个人的 IDE 配置不同——核心行为一致，个人体验可以不同。
