**CLAUDE.md is team-scoped.** When you check a CLAUDE.md file into your repository, every developer who clones the repo and runs Claude Code gets the same instructions. This is essential for maintaining consistency in how AI handles your codebase. Code review standards, commit message formats, testing requirements, architectural boundaries — these are team decisions that shouldn't vary by individual. If your project requires running `validate-pipeline.ts` before committing pipeline changes, that belongs in CLAUDE.md where it's enforced for everyone.

The multi-scope layering adds flexibility. A monorepo can have a root CLAUDE.md with organization-wide rules ("never skip failing tests") plus subdirectory files with component-specific constraints ("this package uses Vitest, not Jest"). User-level `~/.claude/CLAUDE.md` files handle cross-project personal defaults without polluting the team config. As described in Claude Code's [seven programmable layers](/blog/claude-code-seven-programmable-layers), CLAUDE.md sits at the project configuration layer — above the model but below conversation-level instructions.

**Claude Memory is user-scoped.** Your memories are stored locally on your machine under `~/.claude/projects/`, organized by project directory path. Your colleague working on the same repo has their own separate memory store. This is by design — Memory captures personal context that would be noise in a shared config file. Your role ("I'm a frontend engineer new to the backend"), your correction history ("stop adding trailing summaries"), your working style ("prefer single bundled PRs for refactors") — these are individual preferences, not team rules.

Memory also captures cross-project knowledge in its user-type entries. If you tell Claude you're a senior Go engineer who's new to React, that context applies regardless of which project you're working in. CLAUDE.md can't do this — it's anchored to a specific repository.

**The anti-pattern to avoid:** putting personal preferences in CLAUDE.md (where they clutter the team config) or putting team rules in Memory (where they won't apply to other developers). A clear rule of thumb: if removing the instruction would affect code quality for any team member, it belongs in CLAUDE.md. If removing it would only affect your personal experience, it belongs in Memory.

## Content Types: What Goes Where

Deciding what to store in each system becomes straightforward once you understand what each is designed for. Here's a practical breakdown by content type.

**Always CLAUDE.md:**
- Build and test commands (`npm run build`, `npm test`)
- Quality gates and pre-commit requirements
- Coding conventions (naming, file organization, import rules)
- Architectural constraints ("don't import Next.js modules in pipeline scripts")
- Known gotchas that affect code correctness ("use CJK word count for Chinese content")
- Environment-specific setup instructions
- Documentation update requirements

**Always Memory:**
- Your role and expertise level
- Corrections to Claude's behavior ("don't add emoji to commit messages")
- Confirmed approaches ("yes, single PR was the right call here")
- Project status and deadlines ("merge freeze starts March 5")
- Pointers to external resources ("bugs tracked in Linear INGEST project")
- Cross-project preferences that follow you between repos

**Judgment calls:**
- Project-specific context that's temporary (Memory — it has project-type entries with expiration awareness)
- Team conventions that emerged from discussion but aren't formal rules yet (CLAUDE.md — make them formal)
- Debugging patterns specific to the codebase (neither — these belong in code comments or docs, not AI instruction layers)

The [Claude Code memory system documentation](/blog/claude-code-memory) provides additional guidance on the boundary between these systems, including examples of what Memory explicitly excludes: code patterns derivable from reading the codebase, git history available via `git log`, and anything already documented in CLAUDE.md files.

## Maintenance and Evolution

Both systems require ongoing maintenance, but the maintenance models differ significantly.

**CLAUDE.md maintenance is deliberate.** You edit it like any other source file — in a text editor, reviewed in PRs, discussed with your team. When your project adds a new pipeline step or changes its testing framework, you update CLAUDE.md as part of the same PR. This makes maintenance visible and accountable. The downside is that CLAUDE.md can drift out of sync with reality if developers forget to update it. A rule referencing a deleted test runner or an outdated build command doesn't just fail to help — it actively misleads Claude.

Best practice: treat CLAUDE.md updates as part of your definition of done. If you change a build command, update CLAUDE.md in the same commit. Some teams add a CI check that flags CLAUDE.md staleness by looking for referenced commands that no longer exist in `package.json`.

**Memory maintenance is semi-automatic.** Claude creates, updates, and removes memory entries based on conversation signals. If you tell Claude to "forget" something, it finds and removes the relevant entry. If new information contradicts an existing memory, Claude is designed to trust current observations over stale memories. The MEMORY.md index keeps entries discoverable, with a 200-line soft limit that encourages conciseness.

The risk with Memory is accumulation without cleanup. Over months of use, you might build up dozens of memory entries, some obsolete (a project deadline that passed), some contradictory (an old preference superseded by a new one), some redundant (the same feedback captured slightly differently). Periodic review of your `~/.claude/projects/` directory helps, but there's no built-in "memory hygiene" workflow.

Memory entries can also become stale in a subtler way: a memory that names a specific function or file path is a claim about what existed when the memory was written. Functions get renamed, files get moved, flags get removed. The system addresses this by requiring verification before acting on memories that reference specific code artifacts — but this verification adds overhead to every session where stale memories are loaded.

## Integration With Other Claude Code Layers

Claude Code's [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) includes multiple programmable layers beyond CLAUDE.md and Memory. Understanding how these layers interact clarifies when each persistence mechanism adds value.

**CLAUDE.md + Skills:** SKILL.md files in `skills/*/` directories define task-specific instructions — how to write tests, generate content, or review PRs. CLAUDE.md provides the global context these skills operate within. A skill for newsletter generation references the conventions defined in CLAUDE.md (word count rules, editorial voice, quality gates). Memory doesn't interact with skills directly, but it can store feedback about how well a particular skill performed ("the newsletter skill works better when you provide 5 example headlines").

**CLAUDE.md + Hooks:** Hooks are deterministic shell commands that fire on Claude Code events (pre-commit, post-edit, tool-call). CLAUDE.md defines the rules; hooks enforce them programmatically. Memory doesn't interact with hooks at all — it operates at the conversation layer, not the event layer.

**Memory + Conversation Context:** Memory's primary integration point is the conversation itself. It provides background context that shapes Claude's behavior — your preferences, your project's current state, your corrections from past sessions. This context supplements whatever CLAUDE.md provides with personal, temporal, and cross-project knowledge.

The layering means you rarely need to choose between CLAUDE.md and Memory for the same piece of information. They serve different layers of the stack, and the cases where you'd genuinely debate "should this go in CLAUDE.md or Memory?" are rare. When they do arise, the team-vs-personal and deterministic-vs-contextual distinctions resolve the question.

## When to Choose Claude Memory

Choose Claude Memory when the information is personal, temporal, or cross-project:

- **You're the only person who needs this context.** Your role, your expertise gaps, your correction history — these are about you, not the project. Putting them in CLAUDE.md adds noise for your teammates.
- **The information has a shelf life.** Project deadlines, current sprint priorities, temporary workarounds — these change frequently. Memory entries with project-type metadata capture the "why" and "when" so you can judge staleness later. CLAUDE.md changes require commits and reviews, making it too heavyweight for temporary context.
- **You want Claude to learn from your corrections.** When you tell Claude "don't mock the database in integration tests," Memory captures not just the rule but the reason ("we got burned when mocked tests passed but the prod migration failed"). This reasoning helps Claude apply the spirit of the correction in edge cases, not just the letter.
- **The context spans multiple projects.** If you work across several repos and want Claude to know your general preferences everywhere, Memory's user-type entries carry that context without requiring you to duplicate instructions in every project's CLAUDE.md.

## When to Choose CLAUDE.md

Choose CLAUDE.md when the information is shared, stable, or safety-critical:

- **The rule must apply to every team member.** Build commands, testing requirements, architectural constraints — if violating the rule would produce broken code regardless of who's coding, it belongs in the version-controlled file where it's enforced for everyone.
- **Auditability matters.** CLAUDE.md changes show up in git history. You can see who added a rule, when, and why (from the commit message). Memory entries have no history — they're created, updated, or deleted without a trace. For compliance-sensitive projects, CLAUDE.md's audit trail is non-negotiable.
- **The instruction must fire every session.** CLAUDE.md loads unconditionally. Memory entries are loaded based on relevance matching. If you need Claude to always run the build before committing, always use a specific test framework, or always follow a specific import convention, CLAUDE.md guarantees it.
- **You want PR-reviewable AI instructions.** When you add a new rule to CLAUDE.md, your team can discuss whether it's correct, too strict, or missing edge cases. This collaborative refinement process doesn't exist for Memory — it's a private, unreviewed store.

## Verdict

**Use both — they're complementary, not competing.** CLAUDE.md is your project's source of truth for how Claude should behave in this codebase, shared across the team and enforced deterministically. Claude Memory is your personal layer — it captures your preferences, corrections, and cross-project context so you don't repeat yourself across sessions.

The decision rule is simple: **if it's a team rule, it goes in CLAUDE.md; if it's personal context, it goes in Memory.** For the rare gray areas, ask: "Would removing this instruction affect my teammate's code quality?" If yes, CLAUDE.md. If no, Memory.

Start with CLAUDE.md — it's the foundation. Add Memory naturally as you work with Claude Code and accumulate corrections and preferences. Over time, you'll develop a layered instruction system where the project defines the rules and your personal memory refines how Claude applies them to your specific workflow. ## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

No. CLAUDE.md instructions load deterministically and take precedence over Memory entries. If CLAUDE.md says "always run tests before committing" and a Memory entry says "skip tests for quick fixes," Claude follows CLAUDE.md. Memory provides supplementary context — it informs Claude's judgment but doesn't override explicit project rules.

### Does Claude Memory sync across devices?

Claude Memory is stored locally in `~/.claude/projects/` on your machine. It does not sync automatically across devices or cloud environments. If you work on multiple machines, each will develop its own memory store. Anthropic has been [working on memory importing capabilities](/blog/anthropic-claude-memory-upgrades-importing) to reduce this friction, but full cross-device sync is not yet available as of early 2026.

### Should I check CLAUDE.md into version control?

Yes — that's its primary design purpose. CLAUDE.md in version control ensures every developer gets the same Claude Code behavior. The only exception is `~/.claude/CLAUDE.md`, which is your personal user-level config and lives outside any repository. Project-level CLAUDE.md files should always be committed and reviewed like any other source file.

### How many Memory entries is too many?

The MEMORY.md index has a soft limit of 200 lines. Beyond that, entries may be truncated from Claude's context window. In practice, 20-40 well-maintained memory entries cover most developers' needs. Review your memory directory periodically and remove entries that reference outdated deadlines, renamed files, or superseded preferences.

### Can I use CLAUDE.md without Claude Memory?

Yes. CLAUDE.md works independently — Claude Code reads it at session start regardless of whether any Memory entries exist. Many developers start with just CLAUDE.md and only begin accumulating Memory entries after weeks of regular Claude Code use. The two systems are additive, not dependent.
