**Claude Memory is dynamic and probabilistic.** Claude Code decides when to create, update, and retrieve memories based on conversational signals. If you say "I'm a senior backend engineer," Claude Code writes a user memory. If you say "stop adding trailing summaries," it writes a feedback memory. But the system uses judgment — it might not save something you consider important, or it might save something you consider ephemeral. You can explicitly instruct it to remember or forget, but the default behavior is autonomous.

This distinction has practical implications. CLAUDE.md is auditable — you can `cat CLAUDE.md` and see exactly what Claude Code will read. Memory is discoverable but less predictable — you can browse `~/.claude/projects/<project>/memory/` to see what's stored, but you can't guarantee which memories Claude Code will retrieve in a given conversation. The MEMORY.md index helps, but retrieval depends on relevance matching.

For teams that need compliance or reproducibility guarantees, CLAUDE.md is the only viable option. For solo developers who want Claude Code to learn and adapt over time, Memory provides a more fluid experience.

## Scope and Sharing: Team vs Individual Context

The sharpest difference between these two systems is who sees the information.

**CLAUDE.md is a team artifact.** When you add `npm run build` as a required pre-commit step in CLAUDE.md, every developer using Claude Code in that repo follows the same rule. When you document that "ZH content must use CJK word count, not English space-delimited counting," that constraint applies universally. This makes CLAUDE.md the right place for anything that should be true regardless of who is working in the repo: build commands, test requirements, architecture decisions, style guidelines, and known pitfalls.

**Claude Memory is a personal artifact.** Your memory that you're a data scientist investigating logging is irrelevant to your teammate who's a frontend engineer fixing a CSS bug. Your preference for terse responses without summaries is a personal communication style, not a project constraint. Memory captures these individual differences without polluting the shared project configuration.

This separation matters more than it might seem. Without it, you'd face an uncomfortable choice: either stuff personal preferences into CLAUDE.md (cluttering it for everyone) or lose your personal context every time you start a new conversation. The two-layer system lets the project speak for itself through CLAUDE.md while letting each developer customize their individual experience through Memory.

In practice, teams that adopt Claude Code effectively use CLAUDE.md as the "onboarding document for the AI" — it contains everything a new developer (or a new Claude Code session) needs to know about the project. Memory then acts as the "relationship layer" — the accumulated understanding between you and Claude Code that makes it more effective the more you use it. Our [deep dive into the Claude Code memory system](/blog/claude-code-memory) covers the technical implementation in detail.

## Content Strategy: What Goes Where

Deciding what belongs in CLAUDE.md versus Memory is straightforward once you apply two rules:

**Rule 1: If a teammate needs it, it goes in CLAUDE.md.** Build commands, test requirements, linting rules, deployment procedures, architecture constraints, known bugs, and project conventions all belong in CLAUDE.md. These are facts about the project, not about you.

**Rule 2: If it's about you or your current work, it goes in Memory.** Your role, your expertise areas, your communication preferences, feedback you've given Claude Code about its behavior, and context about your current task — these belong in Memory.

Here's a practical breakdown:

**Belongs in CLAUDE.md:**
- `npm run build` must pass before any commit
- Don't import Next.js modules in pipeline scripts (server-only)
- Chinese content uses CJK word count
- API routes live in `src/app/api/`
- Commit messages must describe what changed
- The newsletter pipeline runs stages 1-7 sequentially
- Known bug: `upsertKeyword()` requires three arguments

**Belongs in Claude Memory:**
- User is a senior backend engineer, new to the frontend
- User prefers single bundled PRs over many small ones
- User wants terse responses with no trailing summaries
- Current sprint goal: migrate auth middleware by March 15
- Pipeline bugs tracked in Linear project "INGEST"
- The Grafana board at `grafana.internal/d/api-latency` is the oncall dashboard

**Grey areas exist.** A project decision like "we chose SQLite over Postgres because of deployment simplicity" could go in either place. If the decision affects how Claude Code should write code (e.g., "use SQLite syntax, not Postgres"), put it in CLAUDE.md. If it's background context that helps Claude Code understand your architecture but doesn't change its behavior, Memory is fine.

The [skills system](/blog/5-claude-code-skills-i-use-every-single-day) adds a third layer for task-specific instructions — reusable SKILL.md files that encode how Claude Code should approach particular workflows like writing tests or generating content. Skills sit alongside CLAUDE.md as checked-in, shared artifacts, but scoped to specific tasks rather than the whole project.

## Lifecycle and Maintenance: How Each System Ages

**CLAUDE.md requires deliberate maintenance.** As your project evolves — new build steps, deprecated modules, changed conventions — someone needs to update CLAUDE.md. Stale instructions are worse than no instructions: if CLAUDE.md says "run `npm run test:integration`" but that script was renamed to `npm run test:e2e` six months ago, Claude Code will fail and waste time debugging. The best teams treat CLAUDE.md updates as part of their PR process, the same way they update README files or API docs.

**Claude Memory self-maintains to a degree.** When Claude Code detects that a memory might be stale — a file it references no longer exists, a function it names can't be found — it's designed to verify before acting on the memory. You can also explicitly tell Claude Code to forget something, and it will remove the memory file and update the index. However, memories can still go stale silently, especially project memories about deadlines, sprint goals, or team assignments that change frequently.

Both systems benefit from periodic review. For CLAUDE.md, add it to your quarterly "repo hygiene" checklist. For Memory, browse `~/.claude/projects/<project>/memory/` occasionally and delete files that no longer apply. The discipline required is similar to maintaining any documentation — the difference is that CLAUDE.md is a team responsibility and Memory is a personal one.

## The Hierarchy: How They Work Together

Claude Code loads both systems at conversation start, but they have different priority levels within the overall instruction hierarchy. The [seven programmable layers](/blog/claude-code-seven-programmable-layers) of Claude Code stack as follows: system prompt (Anthropic's base instructions) → global CLAUDE.md (`~/.claude/CLAUDE.md`) → project CLAUDE.md → subdirectory CLAUDE.md → Memory → conversation context → skill files.

In practice, this means:

1. **CLAUDE.md instructions take precedence over Memory.** If CLAUDE.md says "always use TypeScript strict mode" and your Memory says "user prefers loose typing," CLAUDE.md wins. This is by design — shared project constraints should override personal preferences when they conflict.

2. **Memory supplements CLAUDE.md.** Memory fills in the gaps that CLAUDE.md doesn't cover — your role, your communication style, the external systems relevant to your work. It doesn't override project rules; it adds personal context on top of them.

3. **Neither replaces conversation context.** What you say in the current conversation takes priority over both stored systems. If you say "ignore the build step for now, I'm experimenting," Claude Code will comply even if CLAUDE.md says builds are mandatory.

This hierarchy means the two systems are complementary, not competitive. CLAUDE.md sets the floor — the minimum standards and shared context for the project. Memory raises the ceiling — making Claude Code more effective for you specifically, over time.

## When to Choose Claude Memory

Use Claude Memory as your primary persistence layer when:

- **You work solo.** Without teammates to share instructions with, CLAUDE.md provides less value. Memory captures everything you need — your preferences, your workflow, your project context — in a system that updates automatically as you work.
- **You want Claude Code to adapt to your style.** Memory learns from your corrections. If you consistently tell Claude Code to write shorter commit messages or prefer functional patterns over classes, those feedback memories accumulate and shape future behavior.
- **You work across many projects.** Global user memories (your role, your expertise, your communication preferences) follow you everywhere. You don't need to re-explain your background in each new repo.
- **Your context is ephemeral.** Sprint goals, current incidents, team assignments, and deadline pressures change weekly. Memory is designed for this kind of information — easy to create, easy to update, easy to forget when it's no longer relevant.
- **You rely on external systems.** Pointers to Linear boards, Grafana dashboards, Slack channels, and documentation wikis live naturally in reference memories. CLAUDE.md could hold these too, but they're often personal to your workflow rather than universal to the team.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary persistence layer when:

- **You work on a team.** Any instruction that should apply to every developer using Claude Code in the repo belongs in CLAUDE.md. Build commands, test requirements, coding standards, known bugs — these are project truths, not personal preferences.
- **You need reproducibility.** CLAUDE.md is deterministic and auditable. You can review it in PRs, diff it over time, and guarantee that every Claude Code session receives identical instructions. For regulated industries or high-stakes codebases, this predictability is non-negotiable.
- **You want to onboard new developers faster.** A well-written CLAUDE.md acts as both an AI configuration file and a project guide. New team members get productive faster because Claude Code already knows the project conventions.
- **You have complex build or deployment requirements.** If your project has multiple build targets, conditional test suites, or deployment steps that vary by environment, CLAUDE.md ensures Claude Code handles them correctly every time. Memory might lose or misremember these details over time.
- **Your project has architectural constraints that are easy to violate.** "Don't import Next.js modules in pipeline scripts" or "always pass three arguments to `upsertKeyword()`" — these rules exist because someone hit the bug before. CLAUDE.md prevents the AI from repeating the mistake. For a practical guide to writing these kinds of effective instructions, see our [principles for writing Claude Code skills](/blog/9-principles-writing-claude-code-skills).

## Verdict

**Use both.** CLAUDE.md and Claude Memory are not alternatives — they are complementary layers in Claude Code's persistence system, and trying to replace one with the other leads to either cluttered project files or lost team context.

**Start with CLAUDE.md.** Add your build commands, test requirements, and coding standards. Commit it to the repo. This is the foundation — it ensures Claude Code works correctly in your project regardless of who's using it.

**Let Memory build naturally.** As you work with Claude Code, it will accumulate memories about your role, your preferences, and your corrections. Don't try to front-load Memory with information; let it emerge from real conversations. Review it periodically to prune stale entries.

**The decision rule is simple:** if removing a piece of context would break Claude Code's behavior for your teammates, it belongs in CLAUDE.md. If it only affects your personal experience, let Memory handle it. ## Frequently Asked Questions

### Can Claude Code modify CLAUDE.md files?

No. CLAUDE.md is read-only from Claude Code's perspective. Claude Code loads the file at conversation start and treats it as authoritative instructions, but it cannot edit the file. You maintain CLAUDE.md manually, typically through your normal git workflow — editing, committing, and reviewing changes in pull requests.

### Does Claude Memory sync across devices?

No. Claude Memory is stored locally in `~/.claude/` on your machine. If you use Claude Code on multiple devices, each device maintains its own separate memory. There is no built-in sync mechanism. If you need the same context on multiple machines, CLAUDE.md is the better option since it travels with the repository.

### What happens when CLAUDE.md and Memory contradict each other?

CLAUDE.md takes precedence. Project-level instructions in CLAUDE.md override personal context stored in Memory. This is intentional — shared team constraints should win over individual preferences. If you find frequent conflicts, it usually means personal preferences are being stored as Memory when they should be project rules in CLAUDE.md, or vice versa.

### How large can CLAUDE.md be before it hurts performance?

There is no hard limit, but CLAUDE.md consumes context window tokens. A CLAUDE.md file over 2,000-3,000 words starts competing with your actual conversation for context space. Keep it focused on instructions Claude Code needs to follow, not general documentation. Move detailed explanations to linked docs and keep CLAUDE.md as a concise reference.

### Should I commit Memory files to the repo?

No. Memory files are personal and stored outside the repository in your home directory. Committing them would mix personal preferences with shared project configuration, defeating the purpose of the two-layer separation. If you find a memory that the whole team needs, move that information into CLAUDE.md instead.
