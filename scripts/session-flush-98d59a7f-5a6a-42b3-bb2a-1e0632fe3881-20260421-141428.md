**CLAUDE.md is team infrastructure.** When you commit a CLAUDE.md to your repo, every developer who uses Claude Code on that project gets the same instructions. This is powerful for enforcing consistency — coding standards, required quality gates, forbidden patterns, known gotchas. It's also reviewable: a teammate can open a PR to update the CLAUDE.md, and the team can discuss whether the new rule makes sense. This makes CLAUDE.md the right place for anything that represents a team decision.

Many teams structure their CLAUDE.md as a living engineering standards document. Build and test commands go at the top. Workflow rules (like "new feature → discuss design first") come next. Known gotchas and pitfalls sit at the bottom. The [complete guide to Claude Code](/blog/claude-code-complete-guide) covers how to structure these files for maximum effectiveness.

**Claude Memory is personal context.** It never leaves your machine and is never committed to the repo. This makes it the right home for information about *you* — your role, your expertise level, your preferences for how Claude should communicate. It's also where project-specific knowledge that doesn't belong in the repo lives: "we're freezing merges next Thursday for a release," or "the API team tracks bugs in Linear project INGEST."

This separation matters in practice. Consider a scenario: you prefer terse responses with no trailing summaries. That's a feedback memory — it changes how Claude talks to *you*, not how it talks to your teammates. Putting "don't summarize at the end of responses" in the shared CLAUDE.md would impose your preference on the entire team. Letting Memory learn it keeps the preference personal.

The hierarchy also handles edge cases well. A global `~/.claude/CLAUDE.md` lets you set personal rules that apply across all projects (like your git workflow preferences), while project-level CLAUDE.md handles repo-specific rules. Memory sits on top of both, adding learned context that neither instruction file captures.

## Content Types: What Goes Where

Choosing between Memory and CLAUDE.md often comes down to classifying the information you want to persist. Here's a practical decision framework.

### Always CLAUDE.md

- **Build and test commands**: `npm run build`, `npm test`, `npm run lint` — these are facts about the repo, not about you
- **Quality gates**: "All tests must pass before commit" — team-enforced rules
- **Architecture constraints**: "Don't import Next.js modules in pipeline scripts" — repo-specific gotchas
- **Style guidelines**: "Newsletter tone: sharp tech insider briefing a busy founder" — editorial standards
- **Workflow rules**: "New feature → discuss design first, get approval before coding" — process decisions
- **Known bugs and workarounds**: "ZH content must use CJK word count, not English space tokenization" — repo-specific knowledge

### Always Memory

- **Your role and expertise**: "Senior backend engineer, first time touching the React frontend" — shapes how Claude explains things to you
- **Communication preferences**: "Don't add trailing summaries" or "Prefer one bundled PR over many small ones" — personal style
- **External system pointers**: "Pipeline bugs tracked in Linear project INGEST" — organizational knowledge
- **Temporal project state**: "Merge freeze starts April 25 for mobile release" — time-bound context
- **Confirmed approaches**: "Single bundled PR was the right call for this refactor area" — learned judgment calls

### Judgment Calls

Some information sits at the boundary. A decision about which ORM to use might start as a project memory ("we decided on Drizzle because of type safety") and later migrate to CLAUDE.md if it becomes a repo-wide constraint. The general principle: if it needs to be enforced for the team, promote it to CLAUDE.md. If it's contextual background that helps Claude make better suggestions, Memory is sufficient.

For teams building out their full Claude Code configuration, CLAUDE.md and Memory are just two layers. [Skills files](/blog/5-claude-code-skills-i-use-every-single-day) encode task-specific instructions, and [hooks](/blog/claude-code-hooks-mastery) add deterministic automation triggers. The full stack works together.

## Lifecycle and Maintenance

**CLAUDE.md maintenance is explicit.** You edit the file, commit it, push it. It changes when you change it, and the git history shows exactly when and why each rule was added. This is its strength — no drift, no surprises. But it's also a responsibility: if your CLAUDE.md falls out of date, Claude Code follows stale instructions with full confidence. Teams that treat CLAUDE.md as a living document (updating it alongside code changes) get the most value. Teams that write it once and forget it eventually find Claude Code following rules that no longer apply.

**Claude Memory maintenance is mostly automatic.** Claude Code creates, updates, and occasionally removes memories based on conversation signals. But "mostly automatic" means "sometimes wrong." Memory can accumulate contradictory entries if your preferences change. It can record temporary context as if it were permanent. And it can miss corrections that should override earlier memories.

The practical maintenance burden differs significantly:

- **CLAUDE.md**: Review quarterly. Update when you change build tools, adopt new conventions, or discover new gotchas. Delete rules for things that no longer exist. Low frequency, medium effort per update.
- **Memory**: Check `MEMORY.md` occasionally to see what Claude has learned. Delete memories that are stale or wrong. Ask Claude to "forget" things that no longer apply. Higher frequency, lower effort per update.

One underappreciated maintenance pattern: **promoting memories to CLAUDE.md**. If you notice Claude Memory has learned something important enough that the whole team should know it — say, a subtle bug in the deployment pipeline — extract that knowledge from your personal memory and add it to the shared CLAUDE.md. This is how individual learning becomes team knowledge.

## Context Window Impact

Both systems consume context window space, but they do it differently. Understanding this matters for large projects where context is a scarce resource.

**CLAUDE.md loads fully every time.** A 500-line CLAUDE.md means 500 lines of context consumed in every conversation, whether or not all those instructions are relevant to the current task. This creates a practical incentive to keep CLAUDE.md concise. Teams with sprawling CLAUDE.md files sometimes find that Claude Code's responses degrade — not because the instructions are wrong, but because the context window is crowded.

**Memory loads selectively.** The `MEMORY.md` index (capped at 200 lines) always loads, but individual memory files are only retrieved when judged relevant. This means 50 memory files might only consume 10 lines of index space plus the content of 2-3 relevant files in any given conversation. Memory is more context-efficient for storing large amounts of heterogeneous information.

The practical implication: **move low-frequency reference information out of CLAUDE.md and into Memory** (or into skill files that only load when invoked). Your CLAUDE.md should contain instructions that apply to nearly every conversation. Niche gotchas that only matter for specific subsystems are better stored as project memories or in dedicated [skill files](/blog/9-principles-writing-claude-code-skills).

## Migration Patterns: From Memory-Heavy to CLAUDE.md-Heavy (and Vice Versa)

Teams typically start with one system dominant and later rebalance. Here are the two common migration paths.

### Memory-heavy → adding CLAUDE.md

This happens when a solo developer starts using Claude Code, builds up a rich Memory, and then onboards teammates. The new teammates get none of the learned context. The fix: audit your `MEMORY.md`, identify entries that represent team knowledge (not personal preferences), and migrate them to a CLAUDE.md that gets committed to the repo.

**Steps:**
1. Read your `MEMORY.md` index — identify memories typed as "project" or "reference"
2. For each: ask "would a new teammate need to know this?" If yes, add it to CLAUDE.md
3. Remove the migrated memories to avoid duplication
4. Commit the new CLAUDE.md and ask teammates to review it

### CLAUDE.md-heavy → adopting Memory

This happens when a team has a detailed CLAUDE.md but individual developers find themselves repeating personal context every session. The fix: let Memory handle the personal layer.

**Steps:**
1. Identify CLAUDE.md entries that are personal preferences (communication style, commit conventions, preferred tools)
2. Remove them from the shared CLAUDE.md
3. Either save them as memories explicitly or let Claude learn them naturally over a few sessions
4. If needed, create a personal `~/.claude/CLAUDE.md` for preferences that apply across all your projects

## When to Choose Claude Memory

Use Claude Memory as your primary persistence layer when:

- **You're a solo developer** who doesn't need team-shared rules. Memory accumulates your preferences naturally without maintaining a separate file.
- **Context changes frequently.** If your project state shifts week to week (sprint goals, active incidents, merge freezes), Memory handles temporal context better than a file you'd have to manually update constantly.
- **You want low-maintenance persistence.** Memory auto-learns. If you'd rather not maintain a configuration file, Memory captures the 80% case automatically.
- **You have many external system references.** Pointers to Slack channels, Linear projects, Grafana dashboards, and wiki pages are better as typed reference memories than cluttering a CLAUDE.md.

Memory's limitation: it's personal and non-deterministic. If you need Claude Code to *always* follow a specific rule, Memory isn't reliable enough — sometimes the relevant memory doesn't surface, or an outdated one takes precedence.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary persistence layer when:

- **You work on a team.** Any rule that everyone must follow belongs in a committed, reviewed CLAUDE.md. There is no alternative for team-shared deterministic instructions.
- **Consistency is critical.** Quality gates, build commands, and architecture constraints must fire every time, not when the relevance engine decides to retrieve them.
- **You want auditability.** CLAUDE.md has git history. You can see who added a rule, when, and why. Memory has no changelog.
- **Your project has non-obvious gotchas.** If new developers regularly hit the same pitfalls (importing server modules in client code, using wrong word-count methods for CJK text), CLAUDE.md catches these deterministically.
- **You're building a production pipeline.** Automated Claude Code runs (CI, scheduled tasks) need deterministic instructions. Memory is session-based and personal — it doesn't exist in a CI environment.

## Verdict

**Use both.** This isn't a cop-out — Claude Memory and CLAUDE.md serve genuinely different functions, and the most effective Claude Code setups layer them intentionally.

**CLAUDE.md is your team's engineering constitution** — deterministic, version-controlled, reviewed. Put build commands, quality gates, style rules, architecture constraints, and known gotchas here. Every team using Claude Code should have a CLAUDE.md from day one.

**Claude Memory is your personal adaptation layer** — learned, selective, low-maintenance. Let it capture your role, communication preferences, confirmed approaches, and pointers to external systems. It makes Claude Code feel like it knows you, not just your codebase.

The decision rule: **if removing this context would cause a bug or policy violation, it belongs in CLAUDE.md. If removing it would just make Claude Code slightly less helpful to you personally, Memory is fine.** Start with a solid CLAUDE.md, then let Memory build up naturally. Review both periodically, and promote important memories to CLAUDE.md when they represent team-level knowledge.

## Frequently Asked Questions

### Can Claude Memory and CLAUDE.md conflict with each other?

Yes. If your CLAUDE.md says "use Prettier for formatting" but a stale memory records "the team switched to Biome," Claude Code may get confused. CLAUDE.md takes precedence as system-level instructions, but memory can influence Claude's reasoning in subtle ways. The fix: periodically audit your `MEMORY.md` index and delete memories that contradict current CLAUDE.md rules.

### Does Claude Memory persist if I delete and re-clone the repo?

Memory is stored in `~/.claude/projects/` keyed by the project directory path, not the git repo. If you clone to the same path, existing memories apply. If you clone to a different path, Claude Code starts fresh. CLAUDE.md always persists because it's committed to the repo itself — any clone gets it automatically.

### Should I check Memory files into version control?

No. Memory files are personal and often contain context specific to your role and preferences. They live in `.claude/` directories that should be gitignored. If a memory contains team-relevant knowledge, migrate it to CLAUDE.md instead. Anthropic's [memory upgrade features](/blog/anthropic-claude-memory-upgrades-importing) are improving how memory transfers across environments, but the principle remains: shared knowledge goes in CLAUDE.md, personal context stays in Memory.

### How large should my CLAUDE.md be?

Keep it under 200 lines for most projects. Every line consumes context window space in every conversation. If your CLAUDE.md exceeds 300 lines, split task-specific instructions into [skill files](/blog/5-claude-code-skills-i-use-every-single-day) that only load when invoked. Reserve CLAUDE.md for instructions that apply to virtually every conversation.

### Can I use CLAUDE.md without Claude Memory enabled?

Yes. CLAUDE.md is independent of the Memory system — it's just a markdown file that Claude Code reads. Memory is a separate feature that can be enabled or disabled. Many teams start with CLAUDE.md only and add Memory later once individual developers want personalized context.
