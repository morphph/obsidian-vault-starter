**The most common mistake developers make is putting everything in CLAUDE.md or ignoring Memory entirely.** Each system has a clear domain, and mixing them creates either bloated instruction files or lost personal context. Here's how to decide.

### Put It in CLAUDE.md When:

The information is **project-scoped, team-shared, and stable**. If another developer needs the same instruction, and it won't change based on who's asking, it belongs in CLAUDE.md.

Concrete examples:

- Build and test commands (`npm run build`, `npm test`)
- Code style rules ("Use snake_case for Python, camelCase for TypeScript")
- Architecture constraints ("Never import Next.js modules in pipeline scripts")
- Required quality gates ("All tests must pass before commit")
- Project-specific gotchas ("ZH content must use CJK word count, not English space tokenization")
- Workflow rules ("New feature → discuss design first, get approval before coding")

These are rules that apply regardless of who's using Claude Code. They're the project's engineering standards encoded for AI consumption.

### Put It in Memory When:

The information is **personal, evolving, or conversation-derived**. If it changes based on who the developer is, or if Claude learned it during a conversation rather than being told upfront, it belongs in Memory.

Concrete examples:

- Your role and expertise ("Data scientist investigating logging infrastructure")
- Behavioral corrections ("Stop summarizing at the end of every response")
- Positive reinforcements ("Single bundled PR was the right call for this kind of refactor")
- Ongoing project context with deadlines ("Auth middleware rewrite driven by legal compliance, not tech debt")
- External system pointers ("Grafana dashboard at grafana.internal/d/api-latency is what oncall watches")

These are things CLAUDE.md can't capture because they're per-developer, or because they emerge from the work rather than being defined upfront.

### The Gray Zone

Some context falls between the two systems. Team conventions that started as one person's preference, project context that might benefit everyone, debugging findings that could prevent future issues. The rule of thumb: **if you'd put it in a team wiki, put it in CLAUDE.md. If you'd keep it in your personal notes, let it go to Memory.**

## How They Work Together

**In practice, CLAUDE.md and Memory form a two-layer context system where CLAUDE.md sets the floor and Memory personalizes the ceiling.** This isn't accidental — it's how Claude Code's architecture is designed to work.

### Layer 1: CLAUDE.md Sets Universal Rules

When Claude enters a project, it reads the CLAUDE.md hierarchy top to bottom: global defaults → project root → directory-specific. This establishes the baseline: what tools to use, what standards to follow, what to never do. Every developer on the team operates from this same baseline.

### Layer 2: Memory Personalizes the Experience

On top of that baseline, Claude loads the developer's personal Memory. This might adjust the communication style ("terse, no trailing summaries"), add role-specific context ("senior backend engineer, new to the frontend"), or carry forward corrections from previous sessions ("don't use mocks in integration tests").

### Example: The Same Project, Two Developers

**Developer A** (backend lead): CLAUDE.md tells Claude to run tests before committing. Memory tells Claude that Developer A prefers single bundled PRs for refactors, hates verbose explanations, and is tracking a compliance-driven auth rewrite with a deadline.

**Developer B** (junior frontend dev): Same CLAUDE.md, same test requirement. But Memory tells Claude that Developer B is new to React, prefers detailed explanations with examples, and was told to focus on the dashboard component this sprint.

Same project rules, completely different AI experiences. That's the two-layer design working as intended.

## Common Mistakes

### Mistake 1: Dumping Personal Preferences into CLAUDE.md

When one developer adds "always explain code changes in detail" to the project CLAUDE.md, it affects everyone — including the senior engineer who finds verbose explanations annoying. Personal preferences belong in Memory, not in shared configuration.

### Mistake 2: Ignoring Memory Entirely

Some developers treat every Claude Code session as a fresh start, re-explaining their role and re-correcting behavioral issues every time. Memory exists specifically to eliminate this friction. If you find yourself repeating the same correction, tell Claude to remember it.

### Mistake 3: Using Memory for Build Commands

Memory is not version-controlled and not shared. If you put build commands or architecture rules in Memory instead of CLAUDE.md, new team members won't get them, and there's no review process when they change. Anything the whole team needs to know goes in CLAUDE.md.

### Mistake 4: Never Updating CLAUDE.md

CLAUDE.md isn't write-once. As your project evolves — new build steps, deprecated patterns, changed conventions — the file needs to evolve with it. Treat it like any other code: update it in PRs, review changes, keep it current. Stale CLAUDE.md instructions cause the same problems as stale documentation.

## Migration and Maintenance

**Both systems require different maintenance approaches, and understanding the maintenance burden helps you decide how much to invest in each.**

### CLAUDE.md Maintenance

CLAUDE.md is manually maintained. This means:

- **Initial setup**: Run `/init` to generate a starter file, then customize it with your project's actual standards
- **Ongoing updates**: Edit the file when conventions change, add rules when you notice Claude making repeated mistakes, remove rules that no longer apply
- **Review process**: Changes go through PRs like code — this is a feature, not a burden

The maintenance cost scales with project complexity. A solo project might have a 20-line CLAUDE.md that rarely changes. A large team's monorepo might have multiple directory-level files that need coordination.

### Memory Maintenance

Memory is primarily auto-maintained by Claude, but benefits from occasional human review:

- **Auto-save**: Claude writes memories during conversations when it learns something worth persisting
- **Manual triggers**: Tell Claude "remember that I prefer X" to force a save
- **Cleanup**: Periodically review `~/.claude/projects/<path>/memory/` to remove stale or incorrect memories
- **No review process**: Memory changes don't go through PRs, which means they can drift without oversight

The latest [memory upgrades from Anthropic](/blog/anthropic-claude-memory-upgrades-importing) added importing capabilities, letting you bootstrap Memory from existing context rather than building it up conversation by conversation.

## Advanced Patterns

### Pattern 1: CLAUDE.md References Memory Categories

Some teams structure their CLAUDE.md to explicitly mention what should go to Memory versus what's covered in the file itself. For example, adding a section like "User-specific preferences (commit style, verbosity, role context) should be saved to Memory, not added to this file."

### Pattern 2: Skill Files as a Third Layer

Beyond CLAUDE.md and Memory, [skill files](/blog/5-claude-code-skills-i-use-every-single-day) (`skills/*/SKILL.md`) add a task-specific instruction layer. The full context hierarchy becomes: CLAUDE.md (project rules) → Memory (personal context) → SKILL.md (task-specific instructions). Each layer is more specific than the last.

### Pattern 3: Memory-Informed CLAUDE.md Updates

When you notice Claude's Memory accumulating the same correction across multiple developers (via team discussion), that's a signal to promote the rule to CLAUDE.md. Memory serves as a canary for missing project-level instructions.

## When to Choose Claude Memory

Use Memory as your primary context mechanism when:

- You're a **solo developer** who wants Claude to learn your working style over time
- You need **cross-project personal preferences** that follow you regardless of which repo you're in
- You're tracking **evolving project context** — ongoing initiatives, shifting deadlines, key decisions that aren't in the code
- You want Claude to **stop repeating mistakes** without manually editing configuration files
- You're **evaluating Claude Code** and want a low-friction way to improve session quality before investing in CLAUDE.md

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary context mechanism when:

- You're on a **team** and need consistent AI behavior across all developers
- You have **hard project constraints** — build requirements, testing gates, coding standards — that must never be skipped
- You want **version-controlled, reviewable AI instructions** with git blame and PR review
- You need **directory-scoped rules** in a monorepo where different packages have different conventions
- You want **deterministic behavior** — the same input always produces the same baseline behavior

## Verdict

**Use both.** CLAUDE.md and Claude Memory are complementary systems, not alternatives. **CLAUDE.md is your team's engineering standards encoded for AI** — deterministic, version-controlled, shared. **Claude Memory is your personal AI context** — adaptive, auto-maintained, individual. Skipping either one means leaving value on the table.

Start with CLAUDE.md. Run `/init`, customize the generated file with your actual build commands and coding standards, and check it into git. That gives every Claude Code session a solid foundation. Then let Memory build naturally — correct Claude when it does something wrong, mention your role and preferences, and let the auto-save system capture it. Within a few sessions, you'll have a personalized AI coding experience that no amount of CLAUDE.md configuration could replicate alone.

## Frequently Asked Questions

### Can Claude Memory override CLAUDE.md instructions?

Memory and CLAUDE.md are both loaded into context, but CLAUDE.md instructions are treated as authoritative project rules. If a Memory entry conflicts with a CLAUDE.md rule, Claude should follow CLAUDE.md. In practice, conflicts are rare because the systems cover different domains — project rules versus personal preferences.

### Does Claude Memory sync across devices?

No. Claude Memory is stored locally in `~/.claude/` directories on your machine. If you use Claude Code on multiple computers, each maintains its own Memory. Anthropic has introduced [memory importing features](/blog/anthropic-claude-memory-upgrades-importing) that let you bootstrap Memory from exported context, but there's no automatic cross-device sync.

### How do I see what Claude has saved in Memory?

Browse the `~/.claude/projects/<project-path>/memory/` directory. Each memory is a markdown file with frontmatter (name, description, type) and content. The `MEMORY.md` file in that directory serves as an index of all stored memories. You can edit or delete any file directly.

### Should I check Memory files into git?

No. Memory files are personal to each developer and are gitignored by default. Checking them in would impose one developer's preferences on the entire team — exactly what CLAUDE.md is designed for. If a Memory entry seems universally useful, promote the rule to CLAUDE.md instead.

### What happens if CLAUDE.md gets too long?

Claude Code processes the entire CLAUDE.md file at session start, consuming context window tokens. Extremely long files (thousands of lines) can reduce the context available for actual work. Keep CLAUDE.md focused on rules and constraints, not documentation. If it's growing past a few hundred lines, consider splitting into directory-level CLAUDE.md files or moving detailed instructions into [skill files](/blog/9-principles-writing-claude-code-skills).
