**Claude Memory** uses a two-tier file structure. The first tier is `MEMORY.md` — an index file that gets injected into every conversation's context window. This file contains one-line pointers to individual memory files, kept under 200 lines to avoid bloating the context. The second tier is the individual memory files themselves (e.g., `user_role.md`, `feedback_testing.md`), each tagged with frontmatter specifying name, description, and type. Claude reads specific memory files on demand when the index suggests they're relevant.

Memory files follow strict type categories:

- **User memories**: Your role, expertise, and how you prefer to work. Example: "Senior backend engineer, new to React, prefers explanations framed in terms of Go analogues."
- **Feedback memories**: Corrections and confirmations from past sessions. Example: "Integration tests must hit a real database — mocks caused a production incident last quarter."
- **Project memories**: Ongoing work context not derivable from code. Example: "Merge freeze begins March 5 for mobile release cut."
- **Reference memories**: Pointers to external systems. Example: "Pipeline bugs tracked in Linear project INGEST."

**CLAUDE.md** operates as a flat instruction file loaded at session initialization. Claude Code reads it before processing your first message, treating every line as a binding instruction. There's a hierarchy: a global `~/.claude/CLAUDE.md` applies to all projects, while a project-level `CLAUDE.md` in the repo root applies to that specific codebase. Project instructions override global ones when they conflict.

The critical architectural difference: Memory is **additive and evolving** — it grows over time as Claude learns. CLAUDE.md is **declarative and stable** — it defines the rules that don't change session to session.

## Content Boundaries: What Goes Where

The most common mistake developers make is putting the wrong content in the wrong system. Here's the decision framework.

**Put it in CLAUDE.md if:**

- The entire team should follow this rule (coding standards, build commands, branch policies)
- It's derivable from or closely tied to the codebase (architecture decisions, module boundaries)
- It defines Claude's behavior deterministically (quality gates, commit requirements, testing rules)
- It should be version controlled and reviewable in PRs

**Put it in Memory if:**

- It's personal to you (your role, expertise level, communication preferences)
- It's feedback about Claude's behavior ("stop summarizing at the end of responses")
- It's temporary project state ("merge freeze until March 5", "Alice is on vacation this week")
- It references external systems not in the codebase ("bugs tracked in Linear project INGEST")
- It would clutter CLAUDE.md with individual preferences

**Don't put it in either if:**

- It's derivable by reading the current code (file structure, function signatures, imports)
- It's in git history (`git log` and `git blame` are authoritative)
- It's a debugging solution (the fix is in the code; the commit message has context)
- It's ephemeral to the current session (use conversation context or task tracking instead)

This boundary matters because CLAUDE.md bloat is a real problem. When developers dump personal preferences, temporary project state, and team-specific context into CLAUDE.md, it becomes noisy and hard to maintain. Memory exists specifically to absorb the personal and temporal context that doesn't belong in a shared instruction file.

## Team Dynamics: Shared vs Personal Persistence

For solo developers, the distinction between Memory and CLAUDE.md is mostly organizational — you could technically put everything in CLAUDE.md and it would work. The distinction becomes critical on teams.

**CLAUDE.md as the team contract.** When five engineers work on the same codebase, CLAUDE.md ensures every Claude Code session follows the same rules. Engineer A's session and Engineer B's session both run `npm test` before committing, both follow the same naming conventions, both avoid the same known pitfalls. This consistency is impossible with Memory alone because Memory is per-user.

**Memory as individual calibration.** Engineer A is a senior backend developer who wants terse responses. Engineer B is a junior frontend developer who benefits from detailed explanations. Memory lets Claude adapt to each person without forcing a lowest-common-denominator instruction set in CLAUDE.md. Engineer A's feedback memory says "no trailing summaries"; Engineer B's user memory says "explain React concepts in detail."

**The handoff pattern.** Teams that use both systems effectively treat CLAUDE.md as the onboarding document — a new team member's Claude Code session immediately knows how the project works. Memory then accumulates over the first few sessions as Claude learns the individual's working style. This is analogous to how a new hire reads the team wiki (CLAUDE.md) and then builds personal relationships and context (Memory) over time.

## Maintenance and Lifecycle

**CLAUDE.md maintenance** is deliberate and infrequent. You update it when project conventions change — a new build system, a changed testing strategy, an architectural decision. It should be reviewed in PRs like any other code artifact. Stale CLAUDE.md instructions are actively harmful because Claude follows them literally, so teams need a process to keep it current.

**Memory maintenance** is largely automatic but needs periodic pruning. Claude writes and updates memories during normal sessions. However, memories can become stale — a "merge freeze until March 5" memory is useless after March 5. Claude is designed to verify memories against current state before acting on them, but outdated memories still consume context window space.

Best practices for Memory hygiene:

- Review `MEMORY.md` monthly and remove entries that are no longer relevant
- Convert stable project memories into CLAUDE.md entries if they've proven durable
- Delete memory files for completed projects or resolved temporary states
- Keep the MEMORY.md index under 200 lines — it gets loaded into every conversation

Best practices for CLAUDE.md hygiene:

- Review quarterly or when onboarding new team members
- Remove instructions that duplicate what linters or CI already enforce
- Keep it focused — long CLAUDE.md files waste context window on instructions Claude may not need
- Use `skills/*/SKILL.md` files for task-specific instructions rather than cramming everything into CLAUDE.md

## The Skill Layer: A Third Persistence Mechanism

It's worth understanding where **SKILL.md** files fit relative to Memory and CLAUDE.md, since all three form Claude Code's persistence stack. As covered in our breakdown of [Claude Code's extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp), skill files are task-specific instruction sets stored in `skills/*/SKILL.md` within your repository.

SKILL.md files are like CLAUDE.md in that they're version controlled and shared with the team. But they're scoped to specific tasks rather than the entire project. A `skills/testing/SKILL.md` defines how Claude writes tests; a `skills/review/SKILL.md` defines how Claude reviews PRs. They activate on demand rather than loading into every session.

The three-layer model:

1. **CLAUDE.md** — always-on project instructions (loaded every session)
2. **SKILL.md** — task-specific instructions (loaded when the skill is invoked)
3. **Memory** — personal and temporal context (index loaded every session, details on demand)

Each layer has its own scope, lifecycle, and audience. Confusing them leads to duplication, bloat, and inconsistent behavior. For guidance on writing effective skill files, see [9 principles for writing great Claude Code skills](/blog/9-principles-writing-claude-code-skills).

## Common Pitfalls

**Pitfall 1: Duplicating code patterns in Memory.** Memory should not store information that's derivable from reading the codebase. "The API routes are in `src/app/api/`" is not a useful memory — Claude can discover this by reading the project. Save Memory for context that isn't in the code: "The auth module is being rewritten for compliance reasons, not tech debt."

**Pitfall 2: Personal preferences in CLAUDE.md.** "I prefer short responses" does not belong in a shared project file. That's a feedback memory. CLAUDE.md should contain instructions that apply regardless of who's running the session.

**Pitfall 3: Ignoring Memory entirely.** Some developers never engage with the Memory system, which means every new Claude Code session starts cold. They repeat the same corrections ("don't mock the database", "I'm a backend engineer, not frontend") session after session. A few minutes setting up initial memories pays for itself quickly.

**Pitfall 4: Overloading CLAUDE.md.** A 500-line CLAUDE.md file consumes significant context window space in every session, even when most instructions aren't relevant to the current task. Move task-specific instructions to SKILL.md files and temporal context to Memory.

## When to Choose Claude Memory

Use Claude Memory when you need to persist:

- **Personal working style**: Your expertise level, communication preferences, response format expectations
- **Session-to-session corrections**: Feedback you've given Claude that should carry forward ("use integration tests, not mocks")
- **Temporary project state**: Deadlines, ongoing incidents, team member availability, merge freezes
- **External system references**: Where to find documentation, which Slack channel has context, which Linear project tracks bugs
- **Relationship context**: How you and Claude work together — validated approaches, preferred workflows

Memory shines in scenarios where context is personal, temporal, or external to the codebase. If you'd feel weird committing the information to a shared repo, it belongs in Memory.

## When to Choose CLAUDE.md

Use CLAUDE.md when you need to define:

- **Build and test commands**: What to run, in what order, what must pass before committing
- **Coding standards**: Naming conventions, formatting rules, import ordering, module boundaries
- **Architectural constraints**: Which patterns to use, which to avoid, why
- **Quality gates**: What checks must pass, what's non-negotiable
- **Workflow rules**: Branch naming, commit message format, PR requirements, deployment steps
- **Known gotchas**: Project-specific traps that any developer (human or AI) should avoid

CLAUDE.md wins when the instruction is project-specific, team-wide, and stable over time. If a new team member's Claude Code session should follow this rule from their very first interaction, it belongs in CLAUDE.md.

## Verdict

**Claude Memory and CLAUDE.md are complementary, not competing.** Use CLAUDE.md as your project's shared instruction manual — build commands, coding standards, architectural decisions, quality gates. Use Memory as your personal context layer — your role, your preferences, your corrections, and temporal project state that doesn't belong in version control.

The decision rule is simple: **if every team member's Claude session should follow this rule, put it in CLAUDE.md. If it's about you or about right now, let Memory handle it.** Teams that use both effectively get consistent AI behavior across the project (CLAUDE.md) with personalized interaction for each developer (Memory). ## Frequently Asked Questions

### Can Claude Memory and CLAUDE.md conflict with each other?

Yes. If CLAUDE.md says "always write verbose commit messages" but your feedback memory says "keep commits short," Claude faces a contradiction. In practice, CLAUDE.md instructions take precedence because they're treated as authoritative project rules. Memory provides context but doesn't override explicit project instructions. If you notice conflicts, update the memory to align with CLAUDE.md or discuss with your team whether the CLAUDE.md rule needs changing.

### Does Claude Memory work across different machines?

No. Memory files are stored locally in `~/.claude/projects/` and are not synced between machines. If you develop on both a laptop and a desktop, each machine builds its own memory independently. CLAUDE.md, being version controlled, automatically syncs across machines via git. This is another reason to put durable, important instructions in CLAUDE.md rather than relying solely on Memory.

### How much context window does each system consume?

CLAUDE.md is loaded in full at session start, so a 300-line file uses that space in every conversation. Memory is more efficient — only the `MEMORY.md` index (capped at 200 lines) loads automatically, with individual memory files read on demand. For large projects, keeping CLAUDE.md focused and using SKILL.md files for task-specific instructions prevents context window waste.

### Should I version control my Memory files?

No. Memory files are personal and often contain temporal context that would be misleading to other developers. They belong in your local `.claude/` directory, which is typically gitignored. If a memory has proven stable and team-relevant over time, promote it to a CLAUDE.md entry or a SKILL.md instruction instead.

### What happens if I delete my Memory files?

Claude Code starts fresh with no personal context in your next session. You lose accumulated preferences, corrections, and project context. However, Claude will rebuild relevant memories over subsequent sessions as you interact. If you want a clean slate without losing everything, review `MEMORY.md` and selectively remove outdated entries rather than deleting all files.
