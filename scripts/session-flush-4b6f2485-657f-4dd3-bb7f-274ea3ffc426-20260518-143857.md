**Codex's task-level parallelism** works like a job queue. You submit tasks — "fix the flaky test in auth.spec.ts", "add input validation to the signup endpoint", "refactor the logger to use structured output" — and each one runs independently in its own sandboxed container. The containers have no shared state, no message-passing, and no awareness of each other. When tasks complete, you review each PR independently.

This is powerful for batch operations. If you need to fix twenty bugs across a monorepo, you dispatch twenty Codex tasks and come back to twenty PRs. The parallelism scales horizontally — OpenAI's infrastructure handles the container orchestration. But if task B depends on task A's output, you must wait for A to complete, review its changes, merge them, and then dispatch B against the updated codebase.

**Claude Code's session-level composition** works like a team of specialists in one room. The main agent can spawn a research agent to find relevant files, a planning agent to design the approach, and an implementation agent to write the code — all within one interactive session. These sub-agents can run in parallel when their work is independent, or sequentially when one depends on another's findings.

The Agent tool accepts several parameters that shape this coordination:

- **subagent_type**: Selects a specialized agent (Explore, Plan, code-reviewer, or general-purpose)
- **isolation: "worktree"**: Gives the sub-agent its own git worktree so changes don't interfere with the main workspace
- **run_in_background**: Launches the agent asynchronously while the main agent continues working
- **name**: Makes the agent addressable via SendMessage for ongoing coordination

This session-level model means Claude Code can dynamically decide to spawn agents based on what it discovers mid-task. If a refactoring task reveals an unexpected dependency, the main agent can spawn an Explore agent to investigate before proceeding. Codex cannot make this kind of mid-task decision — each task's scope is fixed at dispatch time.

The tradeoff is resource constraints. Claude Code's sub-agents all share your local machine's resources and the parent session's API usage. Spawning ten parallel sub-agents works, but spawning fifty might hit practical limits. Codex's container model has no such ceiling — each task gets its own isolated resources.

## Custom Agent Configuration: Detailed Analysis

Both platforms support customizing agent behavior, but through different mechanisms and at different levels of granularity.

**Codex custom agents** are configured primarily through system prompts and the AGENTS.md file in your repository. AGENTS.md defines project-level instructions — coding standards, testing requirements, review criteria — that apply to all Codex tasks for that repo. You can also pass task-specific instructions when dispatching individual tasks. The [Codex VS Code extension](/blog/codex-vscode) provides a UI for managing these configurations.

However, Codex does not currently support defining distinct agent "types" or "roles" that you can invoke by name. Every Codex task uses the same underlying agent with the same capabilities. Differentiation comes from the instructions you provide, not from structural specialization.

**Claude Code custom agents** operate at two levels. At the project level, CLAUDE.md and [SKILL.md files](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) define reusable instruction sets — similar to Codex's AGENTS.md but with a more granular, composable structure. At the session level, the Agent tool's subagent_type parameter selects from built-in specialized agents, each with different tool access and behavioral profiles:

- **Explore**: Read-only search agent optimized for finding files and symbols. Cannot edit code.
- **Plan**: Architecture agent that designs implementation strategies. Cannot edit code.
- **code-reviewer**: Review agent that analyzes code for issues.
- **general-purpose**: Full-capability agent for complex multi-step tasks.

You can also create effectively custom agents by combining a subagent_type with a detailed prompt that encodes role-specific instructions. The prompt acts as the agent's brief — what to do, what context matters, what format to return. This is not a formal "custom agent definition" system, but it achieves similar results through composition.

For teams building more structured multi-agent systems, Anthropic's [Agent SDK](/glossary/agent-sdk) provides a programmatic layer for defining agent behaviors, tool access, and coordination patterns. This goes beyond what either Claude Code or Codex offer out of the box — it's infrastructure for building custom agent platforms rather than a feature of the coding tool itself.

## Workflow Comparison: Real-World Scenarios

**Scenario 1: Fixing ten independent bugs**

With Codex, you dispatch ten tasks simultaneously. Each gets its own container, its own repo clone, and its own branch. You come back to ten PRs ready for review. Total wall-clock time: however long the slowest task takes. No coordination needed because the bugs are independent.

With Claude Code, you can spawn ten background agents, each working on a bug. If you use worktree isolation, they won't conflict. But they all share your machine's resources and your API usage quota. For truly independent work, Codex's horizontal scaling wins.

**Scenario 2: Refactoring a module with unknown dependencies**

With Codex, you dispatch a task to refactor the module. If the agent discovers mid-task that the module is used in unexpected places, it either handles it within its initial scope or completes with a partial solution. You review, identify the gaps, and dispatch follow-up tasks.

With Claude Code, the main agent spawns an Explore agent to map all usages of the module. Based on the findings, it spawns a Plan agent to design the refactoring approach. Then it either does the work itself or spawns implementation agents for different parts of the codebase. The entire discovery-plan-execute loop happens in one session with real-time coordination.

**Scenario 3: Building a feature that spans frontend and backend**

With Codex, you might dispatch two tasks — one for the API endpoint, one for the UI component. But they cannot coordinate on interface contracts. You typically build the API first, merge it, then dispatch the frontend task against the updated codebase.

With Claude Code, you spawn a planning agent to define the API contract, then launch backend and frontend agents in parallel with the shared contract as context. The agents work concurrently, and the main agent reviews both outputs for consistency before committing.

## When to Choose OpenAI Codex

Codex is the stronger choice when your workflow involves **batch processing of independent tasks at scale**. Specific scenarios:

- **Large bug backlogs**: Dispatch dozens of fix tasks simultaneously. Codex's container isolation means no resource contention regardless of volume.
- **Repetitive codebase-wide changes**: Migration tasks, dependency updates, or boilerplate generation across many repos. Each task is self-contained and needs no coordination.
- **Teams already on ChatGPT Enterprise**: Codex integrates into the ChatGPT workflow your team already uses. The PR-based output fits standard review processes.
- **Security-sensitive environments**: Codex's cloud containers provide hard isolation boundaries. No agent has access to your local filesystem beyond the repo clone. Read more in our [Codex for open source coverage](/blog/codex-for-open-source) about how this works for public repositories.

Codex is less suited for tasks that require iterative discovery, mid-task pivots, or real-time agent coordination. If you find yourself dispatching chains of dependent tasks — where each task's input depends on the previous task's output — you're fighting the architecture.

## When to Choose Claude Code

Claude Code is the stronger choice when your workflow requires **interactive multi-agent coordination within a single session**. Specific scenarios:

- **Complex refactoring with unknown scope**: The ability to spawn Explore agents mid-task to discover dependencies before committing to an approach is a significant advantage over Codex's fixed-scope model.
- **Tasks requiring specialized agent roles**: When you need a search agent, a planning agent, and an implementation agent working in sequence — not just parallel copies of the same agent. Our [multi-agent workflow analysis](/blog/con-u-pour-des-workflows-multi-agents) covers these patterns in depth.
- **Rapid iteration cycles**: Claude Code's sub-agents return results inline. You see findings, adjust the plan, and continue — no waiting for PRs to be created and reviewed. The feedback loop is seconds, not minutes.
- **Custom agent composition**: If you need agents with different tool access, different behavioral profiles, or different context scopes, Claude Code's subagent_type system and SKILL.md files provide that granularity. See the [extension stack guide](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) for how these layers compose.
- **Solo developers or small teams**: Claude Code's local execution model means you control the full workflow without platform dependencies. No cloud containers, no API limits beyond token usage, no vendor-specific PR format.

Claude Code is less suited for pure batch operations at high volume. If you need to run fifty identical tasks in parallel, the local execution model becomes a bottleneck.

## Pricing and Access

Codex is available to ChatGPT Pro ($200/month), Team ($30/user/month), and Enterprise subscribers. Pro users get unlimited Codex tasks; Team and Enterprise users share pooled usage. Codex recently became available to [students with $100 in free credits](/blog/codex-for-students), though with caveats around usage limits. There is no per-task pricing — it's bundled into the subscription.

Claude Code uses Anthropic's usage-based API billing. You pay per input and output token, with costs varying by model tier (Opus, Sonnet, Haiku). Sub-agents consume tokens from the same account. There is no fixed subscription for Claude Code itself — you pay for what you use. For high-volume multi-agent workflows, costs can scale quickly, but you also pay nothing during idle periods.

**The pricing tradeoff**: Codex's flat subscription is more predictable. Claude Code's per-token model is cheaper for light usage but can exceed subscription costs for heavy multi-agent workloads. Teams running dozens of agents daily should estimate token consumption before committing.

## Verdict

**For subagents and custom agents specifically, Claude Code is the more capable platform today.** Its Agent tool provides genuine in-session multi-agent coordination — typed specialists, parallel execution, named message-passing, and dynamic agent spawning based on runtime discoveries. This is a compositional agent system, not just parallel task dispatch.

**Codex's strength is horizontal scale, not agent composition.** Its container-per-task model is excellent for batch operations where you need dozens of independent agents running simultaneously without resource contention. But it does not offer subagents in the traditional sense — you cannot have one agent spawn another, pass it context, or coordinate in real time.

If your workflow is "dispatch many independent tasks and review the PRs," choose Codex. If your workflow is "orchestrate specialized agents that discover, plan, and implement together," choose Claude Code. Many teams will benefit from both — using Claude Code for complex interactive work and Codex for high-volume batch operations.

## Frequently Asked Questions

### Can Codex tasks communicate with each other during execution?

No. Each Codex task runs in an isolated container with no shared state or message-passing between tasks. Coordination between tasks must happen externally — you review one task's output and use it to inform the next task's instructions. Claude Code's named agents with SendMessage can coordinate in real time within a session.

### How many sub-agents can Claude Code run in parallel?

There is no hard-coded limit on parallel sub-agents in Claude Code, but practical limits exist. Each sub-agent consumes API tokens from your account and shares your local machine's resources. Running ten parallel agents works well for most workflows. Running fifty simultaneously would likely hit context window, memory, or API rate limits.

### Does Codex support custom agent types like Claude Code's Explore or Plan agents?

Codex does not currently differentiate agent types by role. Every Codex task uses the same underlying agent with the same capabilities. You customize behavior through task instructions and AGENTS.md configuration, not through structural specialization. All Codex agents can read, write, and execute — there is no read-only search agent equivalent.

### Can I use Codex and Claude Code together in the same project?

Yes, and many teams do. A common pattern is using Claude Code for interactive development sessions — where you need agent coordination, mid-task discovery, and rapid iteration — and dispatching Codex tasks for batch operations like fixing a backlog of bugs or running codebase-wide migrations. Both tools work with standard git workflows, so their outputs (branches and PRs) integrate naturally.

### Is the Agent SDK the same as Claude Code's Agent tool?

No. The [Agent SDK](/glossary/agent-sdk) is Anthropic's programmatic framework for building custom agent applications. Claude Code's Agent tool is a built-in feature of the Claude Code CLI that lets you spawn sub-agents within a coding session. The Agent SDK is infrastructure for developers building their own agent platforms; the Agent tool is a user-facing feature of an existing platform.
