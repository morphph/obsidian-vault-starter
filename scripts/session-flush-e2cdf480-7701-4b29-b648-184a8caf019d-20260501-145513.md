# Codex Subagents vs Custom Agents：哪种多智能体模式更适合你？

**TL;DR：** **Subagents** 是 Codex 内置的子任务分发机制，适合将一个大任务自动拆解成多个并行子任务执行；**Custom Agents** 是用户自定义的智能体配置，适合需要特定工具链、指令集或领域知识的复杂工作流。如果你的需求是加速单次任务执行，选 Subagents；如果你需要跨项目复用一套标准化的 Agent 行为，选 Custom Agents。

## 概述：Subagents

Subagents 是编程智能体平台内置的子代理机制，核心设计目标是将复杂任务拆解为可并行执行的子任务。当主 Agent 接收到一个涉及多文件、多步骤的工程任务时，它会自动生成多个 Subagent，每个 Subagent 独立处理一个子任务——比如修改模块 A 的接口、更新模块 B 的测试、重构模块 C 的日志逻辑。

Subagents 的关键特征是**平台原生**：你不需要额外配置，只要任务本身具备并行拆解的可能性，系统就会自动生成 Subagents 来加速执行。这种模式在 [Codex](/zh/blog/codex-complete-guide) 和 [Claude Code Agent Teams](/zh/blog/claude-code-agent-teams) 中都有实现，是当前主流 [Agentic Coding](/zh/glossary/agentic-coding) 平台的标配能力。

Subagents 的执行环境通常是沙箱化的，每个 Subagent 运行在独立的容器或 worktree 中，互不干扰。执行完毕后，结果汇总回主 Agent 进行整合。

## 概述：Custom Agents

Custom Agents 是用户自行定义的智能体实例，通过配置文件或 API 参数来指定 Agent 的行为规则、可用工具、系统提示词和上下文策略。与 Subagents 的"系统自动生成"不同，Custom Agents 强调的是**用户可控性**。

典型的 Custom Agent 配置包括：指定模型版本、注入领域专用的 System Prompt、绑定特定的 MCP 工具链、限制文件访问范围、设置输出格式约束等。你可以把 Custom Agent 理解为一个"预设好角色和工具箱的专家"——比如一个专门做安全审计的 Agent、一个专门生成测试用例的 Agent、或一个专门处理数据库迁移的 Agent。

在 [Claude Code 的扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中，Custom Agents 通过 Skills、Hooks 和 MCP 等层级实现深度定制。Codex 平台则通过 [Agent SDK](/zh/glossary/agent-sdk) 提供类似的自定义能力。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **创建方式** | 平台自动生成 | 用户手动配置 | 视场景 |
| **配置复杂度** | 零配置 | 需要编写配置/提示词 | Subagents |
| **工具链定制** | 继承主 Agent 工具 | 可独立绑定工具链 | Custom Agents |
| **执行模式** | 并行执行子任务 | 按需调用，串行或并行 | Subagents |
| **上下文隔离** | 沙箱隔离，独立 worktree | 可配置共享或隔离 | Custom Agents |
| **跨项目复用** | 不可复用 | 可模板化复用 | Custom Agents |
| **适用任务粒度** | 单次大任务拆解 | 长期标准化工作流 | 视场景 |
| **学习曲线** | 极低 | 中等 | Subagents |

## 任务拆解与执行：核心差异

Subagents 和 Custom Agents 最根本的区别在于**谁来决定 Agent 做什么**。

Subagents 的任务分配由主 Agent 自主决定。你给出一个高层指令——"重构这个模块并更新所有相关测试"——主 Agent 分析任务结构后，自动拆解为若干子任务并分发给 Subagents。这个过程对用户来说是透明的，你不需要手动指定"Subagent 1 处理文件 A，Subagent 2 处理文件 B"。

这种模式的优势是速度：多个 Subagent 并行工作，一个原本需要串行 10 分钟的任务可能 3 分钟就完成了。但劣势同样明显——你对拆解逻辑的控制力有限。如果主 Agent 误判了任务边界，Subagents 之间可能产生冲突（比如两个 Subagent 同时修改了同一个接口的类型定义）。

Custom Agents 的任务定义完全由用户掌控。你预先设计好 Agent 的职责边界、输入输出格式和可用工具，然后在需要时精确调用。典型用法是构建一个 [多智能体工作流](/zh/blog/con-u-pour-des-workflows-multi-agents)：代码生成 Agent → 测试 Agent → Review Agent → 部署 Agent，每个环节由专门的 Custom Agent 负责。

从工程实践看，这两种模式并不互斥。在 [Claude Code 的 Subagents 实战案例](/zh/blog/claude-code-subagents-examples)中，成熟团队往往混合使用：用 Subagents 加速单个 Custom Agent 内部的并行子任务，同时用 Custom Agents 组织跨阶段的工作流。

## 定制与复用能力

Custom Agents 在定制能力上有决定性优势。

一个配置良好的 Custom Agent 可以携带完整的上下文：领域术语表、编码规范、提示词模板、甚至通过 MCP Server 连接外部知识库。这意味着你可以创建一个"了解你项目的所有架构决策、知道哪些 API 已废弃、清楚团队的 Git 分支策略"的 Agent，然后在每次需要时直接复用。

Subagents 不具备这种持久化能力。每个 Subagent 在任务结束后即销毁，不保留状态，也无法跨任务复用配置。它本质上是一次性的计算单元。

但 Subagents 的"无配置"特性也是一种优势——降低了使用门槛。如果你只是偶尔需要加速一个大型重构任务，完全不需要花时间设计 Custom Agent 的配置架构。正如 [Codex 完全指南](/zh/blog/codex-complete-guide)中所述，Subagents 的设计哲学是"让平台替你做拆解决策"。

对于团队来说，Custom Agents 的复用价值更高。你可以将一个经过反复调优的 Security Review Agent 标准化后分享给全团队，确保每个人提交 PR 时都经过相同标准的安全检查。这种 [Skills 驱动的标准化](/zh/blog/9-principles-writing-claude-code-skills)是单靠 Subagents 无法实现的。

## 适合使用 Subagents 的场景

当你的任务满足以下条件时，优先选择 Subagents：

- **任务可自然拆解**：涉及多个独立文件或模块的修改，子任务之间没有强依赖关系
- **一次性执行**：不需要反复运行同样的 Agent 配置，只是加速当前任务
- **快速迭代阶段**：项目早期或原型阶段，代码结构还在变化，不值得投入时间设计 Custom Agent
- **个人开发者**：团队只有一个人，不存在标准化和复用需求

典型场景包括：大规模重命名变量、跨模块的 API 接口升级、批量添加类型注解、将一个大 PR 的改动拆解到多个文件并行处理。

## 适合使用 Custom Agents 的场景

当你的工作流具备以下特征时，Custom Agents 是更好的选择：

- **重复性工作流**：同一类任务反复出现（如每周的依赖升级、每次 PR 的安全审查）
- **领域专业性**：任务需要特定的行业知识、编码规范或工具链
- **团队协作**：多人团队需要统一的 Agent 行为标准
- **多阶段流水线**：任务需要多个 Agent 按顺序协作完成（生成 → 测试 → 审查 → 部署）
- **合规要求**：需要审计 Agent 的行为记录和决策过程

典型场景包括：自动化 Code Review 流水线、[长时间运行的 Agent 任务](/zh/blog/effective-harnesses-for-long-running-agents)、CI/CD 集成中的智能测试生成、以及基于 [Agent SDK](/zh/glossary/agent-sdk) 构建的企业级自动化系统。

## 结论

**如果你需要快速把一个大任务并行化，用 Subagents——零配置、开箱即用。如果你需要构建可复用、可审计、可标准化的智能体工作流，用 Custom Agents。** 两者不是替代关系，而是互补关系：成熟的工程团队通常在 Custom Agent 内部利用 Subagents 实现并行加速，同时用 Custom Agent 的配置体系保证行为一致性。

选择的核心判据是**任务的重复频率和团队规模**。个人开发者偶尔加速大任务，Subagents 足够。团队需要标准化 Agent 行为并跨项目复用，Custom Agents 是必选项。无论哪种方式，关键是理解你的 Agent 在做什么——盲目并行化并不总是更快，精心设计的 Custom Agent 也不一定比简单的 Subagent 调用更有效。匹配场景，才是正确答案。

## 常见问题

### Subagents 和 Custom Agents 可以同时使用吗？

可以，而且这是推荐做法。Custom Agent 定义工作流的整体结构和行为规范，Subagents 在每个 Custom Agent 内部实现并行加速。两层嵌套使用既保证了行为可控性，又获得了执行速度的提升。

### Subagents 会不会产生代码冲突？

存在这种可能。当多个 Subagent 修改了有依赖关系的文件时，主 Agent 在合并结果时可能遇到冲突。平台通常提供冲突检测和自动解决机制，但对于复杂的接口变更，建议在任务描述中明确指定文件边界。

### Custom Agents 的配置门槛高吗？

取决于定制深度。基础配置（指定模型、注入 System Prompt）只需几行 YAML 或 JSON。深度定制（绑定 MCP 工具、设计多阶段流水线、集成外部知识库）则需要一定的架构设计能力。建议从简单配置开始，根据实际需求逐步增加复杂度。
