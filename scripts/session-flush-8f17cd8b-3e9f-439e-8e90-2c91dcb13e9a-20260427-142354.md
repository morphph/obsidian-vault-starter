# Claude Memory vs CLAUDE.md：两套持久化机制，该用哪个？

**TL;DR：** **CLAUDE.md** 是你写给 AI 的规则手册——项目约定、编码规范、工作流程，跟代码一起提交到仓库里，团队共享。**Claude Memory** 是 AI 自己记的笔记——用户偏好、项目背景、反馈修正，自动存到本地文件系统，跨会话保留。规则放 CLAUDE.md，上下文交给 Memory。搞混了，两个都不好用。

## 概述：Claude Memory

**Claude Memory** 是 [Claude Code](/zh/glossary/agentic-coding) 内置的自动记忆系统，解决的是"每次开新会话都要重新解释一遍背景"的问题。它把你在对话中提到的用户信息、项目背景、工作偏好、反馈修正等内容，以 Markdown 文件的形式持久化到 `~/.claude/projects/` 目录下。

Memory 的核心设计是**被动积累**：你不需要手动维护，Claude 会在对话过程中识别值得记住的信息并自动保存。下次开会话时，这些记忆会被加载到上下文中，让 AI 从上次断点继续。

Memory 分为四种类型：user（用户画像）、feedback（行为修正）、project（项目动态）、reference（外部资源指针）。每条记忆存为独立文件，带 frontmatter 元数据，由一个 `MEMORY.md` 索引文件统一管理。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级配置文件，放在仓库根目录，随代码一起版本控制。它的作用是告诉 AI"在这个项目里你应该怎么工作"——构建命令、测试流程、代码风格、提交规范、已知坑点。

CLAUDE.md 的设计哲学是**显式声明**：你写什么，AI 就遵守什么。它不会自动生成，不会自动更新，完全由人类控制。团队里每个人打开 Claude Code，读到的是同一份 CLAUDE.md，得到的是一致的 AI 行为。

除了根目录的 CLAUDE.md，还有全局级的 `~/.claude/CLAUDE.md`（个人偏好，所有项目生效）和 `.claude/settings.json`（权限、hooks 配置）。三层叠加，从通用到具体。详细的[七层架构解析](/zh/blog/claude-code-seven-programmable-layers)可以帮你理解这套分层体系。

## 持久化机制详解

Claude Memory 和 CLAUDE.md 虽然都在会话开始时加载到上下文，但它们的写入逻辑完全不同。

CLAUDE.md 是静态文档。你用编辑器打开，写入规则，提交到 Git。Claude Code 每次启动时读取它，但永远不会修改它。这意味着 CLAUDE.md 的内容完全可预测——你写了什么就是什么，不会因为某次对话而被 AI 悄悄改掉。

Memory 是动态的。Claude 在对话中检测到值得记住的信息时，会创建或更新 `~/.claude/projects/` 下的 Markdown 文件。每个记忆文件带 frontmatter（name、description、type），由 `MEMORY.md` 索引。你可以随时查看、编辑或删除这些文件，但正常情况下不需要手动干预。

关键区别在于**信息的生命周期**。CLAUDE.md 里的规则是长期稳定的——"用 Vitest 跑测试"、"commit message 用中文"——这些不会因为一次对话而改变。Memory 里的内容是会变的——"用户正在做 SEO 迁移"、"上次说了不要在 PR 末尾加总结"——这些有时效性，可能过期，需要 AI 自己判断是否还适用。

我们在[Claude Code 记忆系统详解](/zh/blog/claude-code-memory)中深入分析了这套双层持久化的工程设计。

## 实际使用场景对比

理解两者的最好方式是看具体场景。

**应该放 CLAUDE.md 的内容：**
- 构建和测试命令：`npm run build`、`npm test`
- 代码风格约定：函数命名用 camelCase、commit message 格式
- 项目架构约束：不要在 pipeline 脚本里 import Next.js 模块
- 工作流程规则：提交前必须跑 build + test + lint
- 已知坑点：某个函数必须传三个参数、某个 SDK 版本要求

这些是**规则**——任何人在这个项目里用 Claude Code，都应该遵守。

**应该交给 Memory 的内容：**
- 用户背景："我是数据科学家，主要关注日志和监控"
- 工作偏好："不要在回复末尾加总结"、"重构时倾向一个大 PR"
- 项目动态："下周四代码冻结，准备移动端发版"
- 外部资源："pipeline bug 在 Linear 的 INGEST 项目里追踪"

这些是**上下文**——因人而异，随时间变化。

如果你把个人偏好塞进 CLAUDE.md，团队其他人会被你的习惯约束。如果你把项目规则放在 Memory 里，换台机器就丢了。对照[Claude Code Skills 最佳实践](/zh/blog/9-principles-writing-claude-code-skills)中的分层原则，选择正确的持久化层级。

## 选 Claude Memory 的场景

当你需要 AI **记住你是谁、你怎么工作**时，用 Memory。

独立开发者是 Memory 的最大受益者。你的编码习惯、沟通风格、项目上下文——这些不需要写成正式文档，但每次对话都要用到。Memory 让 Claude Code 逐渐了解你，减少重复解释的摩擦。

Memory 也适合追踪**短期项目状态**。"正在重构 auth 模块"、"下周要做性能优化"——这些信息有时效性，不适合固化到 CLAUDE.md 里。Memory 会自然过期或被更新，更符合这类信息的生命周期。

还有一个容易忽略的场景：**行为修正**。当你告诉 Claude"别用 mock 测数据库"或"PR 描述不要超过三行"，这些反馈被存为 feedback 类型的 Memory，下次自动生效。这比每次开会话重新说一遍高效得多。

## 选 CLAUDE.md 的场景

当你需要**团队一致性和可预测性**时，用 CLAUDE.md。

CLAUDE.md 进 Git，走 code review，团队所有人看到同一份规则。新人加入项目，打开 Claude Code 就自动继承了团队的编码规范和工作流程。这是 Memory 做不到的——Memory 是私有的，不共享。

CLAUDE.md 也是**质量门禁**的正确位置。"提交前必须跑 build + test"、"不要跳过 lint"——这些必须是硬性规则，不能依赖 AI 的记忆来执行。写在 CLAUDE.md 里，每次会话都会加载，不存在"忘了"的可能。

对于需要在[多 Agent 场景](/zh/blog/claude-code-subagents-examples)下保持一致行为的项目，CLAUDE.md 更是不可替代。Sub-agent 不会继承主 agent 的 Memory，但会读取 CLAUDE.md。

## 最佳实践：两者协同

成熟的 Claude Code 工作流同时用好这两套系统。推荐的分工：

1. **CLAUDE.md 管规则**：构建命令、测试流程、代码规范、架构约束、已知坑点。内容要求精确、稳定、团队共享
2. **Memory 管上下文**：用户画像、行为偏好、项目动态、外部资源指针。内容允许模糊、动态、因人而异
3. **不要重复**：如果一条信息已经在 CLAUDE.md 里，Memory 不需要再存一份。Memory 系统设计上会避免存储"可以从代码或 Git 历史推导出的信息"
4. **定期清理 Memory**：项目动态类的记忆会过时。养成习惯，偶尔检查 `~/.claude/projects/*/memory/` 下的文件，删除过期内容

## 结论

**CLAUDE.md 是项目的宪法，Memory 是 AI 的笔记本。** 规则、规范、约束——写 CLAUDE.md，提交到 Git，团队共享。背景、偏好、动态——交给 Memory，自动积累，跨会话保留。两者解决不同层面的问题，不存在替代关系。搞清楚每条信息该放哪里，你的 Claude Code 体验会有质的提升。

## 常见问题

### Claude Memory 会不会记错东西？

Memory 可能记录过时的信息。Claude Code 在使用记忆前会尝试验证其当前有效性——比如检查文件是否还存在、函数是否还在用。你也可以手动编辑或删除 `~/.claude/projects/*/memory/` 下的记忆文件来修正错误。

### CLAUDE.md 支持多层级吗？

支持。Claude Code 会读取三层配置：全局 `~/.claude/CLAUDE.md`（个人偏好）、项目根目录 `CLAUDE.md`（团队规则）、以及子目录中的 CLAUDE.md（模块级覆盖）。三层叠加，越具体的优先级越高。

### 团队协作时 Memory 会冲突吗？

不会。Memory 存储在每个开发者的本地文件系统中，不进 Git，不共享。每个人有自己独立的记忆空间。团队一致性靠 CLAUDE.md 保证，个人偏好靠 Memory 保留。
