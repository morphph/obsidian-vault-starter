# Claude Memory vs CLAUDE.md：两套记忆系统，到底该用哪个？

**TL;DR：** **Claude Memory** 是 Anthropic 的跨会话持久记忆——Claude 自动记住你的偏好、背景和习惯，下次对话无需重复。**CLAUDE.md** 是开发者手写的项目级指令文件，随代码仓库一起版本管理，告诉 Claude Code 这个项目该怎么干活。两者不是竞争关系，而是互补层：**Memory 管"人"，CLAUDE.md 管"项目"**。个人开发者优先配好 Memory；团队协作必须写 CLAUDE.md。

## 概览：Claude Memory

**Claude Memory** 是 Anthropic 在 Claude 产品线中内置的持久化记忆能力，让 AI 在多次对话之间保留对用户的了解。它解决的核心问题是：每次新对话都要从零开始解释"我是谁、我在做什么、我喜欢什么风格"。

Memory 的运作方式是 Claude 在对话过程中主动识别值得记住的信息——你的角色、技术栈偏好、代码风格、常用工具——并将其存储为结构化的记忆条目。下次对话时，这些记忆自动加载到上下文中，Claude 据此调整回答方式。

在 Claude Code 中，Memory 系统进一步演化为 **Auto Memory**：一套基于文件的持久记忆机制，存储在 `~/.claude/projects/` 目录下。它分为 user（用户画像）、feedback（行为修正）、project（项目动态）、reference（外部资源指针）四种类型。Claude Code 会在对话中自动写入和读取这些记忆文件。

Anthropic 在 2026 年初对 Memory 做了[重大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，包括向免费用户开放记忆功能、支持从 ChatGPT 导入历史数据，大幅降低了迁移成本。

## 概览：CLAUDE.md

**CLAUDE.md** 是一个 Markdown 格式的项目指令文件，放在代码仓库根目录，作为 Claude Code 每次启动会话时的"第一份参考资料"。它的定位不是记忆，而是**规则手册**。

CLAUDE.md 由开发者手动编写和维护，通常包含：构建命令（`npm run build`、`npm test`）、代码风格约定、架构决策、质量门禁（哪些检查必须在 commit 前通过）、已知陷阱和注意事项。它随 Git 提交，团队所有成员共享同一份指令。

在 Claude Code 的[七层架构](/zh/blog/claude-code-seven-programmable-layers)中，CLAUDE.md 处于最基础的"项目上下文层"。它不会随对话演变，不会被 Claude 自动修改——除非你明确要求。这种稳定性正是它的价值：团队中的每个人（和每个 AI 会话）都遵循相同的规则。

CLAUDE.md 还有层级结构：全局的 `~/.claude/CLAUDE.md` 适用于所有项目，项目根目录的 `CLAUDE.md` 适用于当前仓库，子目录还可以有自己的 `CLAUDE.md` 覆盖特定模块的规则。

| **维护方式** | Claude 自动写入和更新 | 开发者手动编辑 | Memory |

| **生效时机** | 自动加载，无需操作 | 每次会话启动时读取 | 平局 |

## 记忆机制深度对比

Claude Memory 和 CLAUDE.md 在底层机制上有本质区别，理解这一点才能正确使用它们。

**Memory 是动态的。** Claude Code 的 Auto Memory 在对话过程中持续运作：当你纠正它的做法（"别用 mock 测数据库"），它会自动创建一条 feedback 类型的记忆；当你提到自己的角色（"我是数据科学家"），它会存为 user 类型。这些记忆存储在 `~/.claude/projects/` 下的独立 Markdown 文件中，通过 `MEMORY.md` 索引文件管理。

**CLAUDE.md 是静态的。** 它在会话开始时被读取一次，之后不会改变。Claude Code 严格遵循其中的指令——如果你写了"commit 前必须跑 `npm test`"，它每次都会执行，不会因为"上次跑过了"就跳过。这种确定性对团队协作至关重要。

一个关键区别是**可审计性**。CLAUDE.md 可以通过 code review 审查，通过 Git blame 追溯变更历史。Memory 文件虽然也是 Markdown，但不进版本管理，只有当事人能看到。对于需要合规审计的团队来说，项目规则必须放在 CLAUDE.md 中。

## 协作场景对比

两套系统在不同团队规模下的价值差异明显。

**独立开发者**场景下，Memory 的优势更大。你的编码习惯、偏好的错误处理方式、常用的 Git 工作流——这些只需要 Claude 记住一次，之后每个项目都受益。CLAUDE.md 仍然有用（记录构建命令和项目特定规则），但 Memory 承担了大部分"让 AI 理解你"的工作。

**团队协作**场景下，CLAUDE.md 是不可替代的基础设施。考虑这个场景：五个工程师用 Claude Code 协作开发同一个项目。每个人的 Memory 里存着不同的偏好和理解——一个人习惯 kebab-case，另一个人用 camelCase。如果没有 CLAUDE.md 统一规范，同一个项目会出现风格不一致的代码。

最佳实践是**分层治理**：CLAUDE.md 定义团队必须遵守的硬性规则（代码风格、测试要求、提交规范），Memory 存储个人层面的软性偏好（解释风格、沟通习惯、领域知识）。两者冲突时，CLAUDE.md 优先——这是设计使然。

如果你对 Claude Code 的完整能力体系感兴趣，建议阅读[Claude Code 记忆系统详解](/zh/blog/claude-code-memory)，其中对 CLAUDE.md 和 Auto Memory 的交互做了更深入的拆解。

## 什么时候选 Claude Memory

以下场景中，Memory 是更合适的机制：

- **个人偏好持久化**：你喜欢简洁的回答风格、不要 emoji、commit message 用中文——这些跨项目通用，适合存在 Memory 里
- **角色和背景信息**：你的技术栈、职位、负责的业务领域——Claude 记住这些后能给出更有针对性的建议
- **行为反馈修正**：当你纠正 Claude 的某个做法，Memory 确保同样的错误不会在下次对话中重演
- **外部资源指针**：Bug 追踪在哪个 Linear 项目里、监控看哪个 Grafana 面板——这些信息不属于代码仓库，但 Claude 知道后工作更高效

Memory 的核心价值是**减少重复沟通成本**。用得越久，Claude 对你的理解越深，每次对话的启动成本越低。

## 什么时候选 CLAUDE.md

以下场景中，CLAUDE.md 是正确选择：

- **团队编码规范**：代码风格、命名约定、目录结构——所有成员（包括 AI）必须遵守的硬性规则
- **构建和部署流程**：`npm run build`、`npm test`、部署到 Vercel 的步骤——这些是项目事实，不是个人偏好
- **质量门禁**：commit 前必须通过哪些检查、哪些文件不能修改、哪些依赖不能升级
- **已知陷阱**：项目里那些踩过的坑——中文内容必须用 CJK 分词、某个函数必须传三个参数——这些必须写进 CLAUDE.md 防止 AI 重蹈覆辙
- **技能文件关联**：通过 [Skills 系统](/zh/blog/9-principles-writing-claude-code-skills)定义特定任务的执行标准，CLAUDE.md 引用这些 Skill 文件

CLAUDE.md 的核心价值是**团队一致性和可审计性**。它是代码的一部分，享受版本管理的所有好处。

## 结论

**Claude Memory 和 CLAUDE.md 不是二选一的关系，而是必须同时使用的互补层。** Memory 让 Claude 理解"你是谁"——你的角色、偏好、工作习惯；CLAUDE.md 让 Claude 理解"这个项目怎么做"——规范、流程、约束。独立开发者可以先从 Memory 入手，享受跨会话的个性化体验；团队协作则必须以 CLAUDE.md 为基础，确保一致性和可追溯性。最终目标是两者配合：CLAUDE.md 管规则，Memory 管偏好，[Skills](/zh/blog/5-claude-code-skills-i-use-every-single-day) 管任务——三层合力，让 Claude Code 真正变成你团队的一员。

## 常见问题

### Claude Memory 和 CLAUDE.md 会冲突吗？

不会直接冲突。CLAUDE.md 在 Claude Code 的优先级高于 Memory——如果 CLAUDE.md 规定用 TypeScript 写测试，即使你的 Memory 里记录了偏好 JavaScript，Claude Code 也会遵循项目规则。这是正确的设计：项目规范应该覆盖个人偏好。

### Memory 的内容可以导出或共享给团队吗？

Memory 文件存储在本地的 `~/.claude/projects/` 目录下，是标准的 Markdown 文件，技术上可以手动复制。但它的设计意图是个人使用——如果某条 Memory 对团队有价值，正确做法是将其提炼后写入 CLAUDE.md。

### CLAUDE.md 文件会不会越写越臃肿？

会的。CLAUDE.md 应该保持精炼，只包含 Claude Code 工作时真正需要的信息。历史决策记录、设计文档、详细的 API 说明应该放在独立文档中，CLAUDE.md 只引用路径。建议定期审查，删除过时的条目。
