# Claude Memory vs CLAUDE.md：两套持久化系统怎么选？

**TL;DR:** **Claude Memory**（Auto Memory）是跨会话的自动笔记本——Claude 在对话中观察到重要信息后自动记录，下次对话时读取。**CLAUDE.md** 是项目级的规则手册——你手动编写，定义编码规范、工作流约束和项目上下文。两者不是竞争关系，而是互补：CLAUDE.md 管"怎么做"，Memory 管"记住了什么"。如果只能选一个先配，**先写 CLAUDE.md**——它对输出质量的影响更直接、更可控。

## 概述：Claude Memory

Claude Memory 是 Claude Code 的自动持久化记忆系统。它以文件形式存储在 `~/.claude/projects/<project>/memory/` 目录下，每条记忆是一个独立的 Markdown 文件，带有 frontmatter 标注类型（user、feedback、project、reference）。

核心机制：Claude 在对话过程中识别到值得长期保留的信息——比如你的技术偏好、项目决策、反馈修正——会自动写入记忆文件。下一次会话启动时，系统加载 `MEMORY.md` 索引，Claude 据此决定读取哪些记忆。

Memory 解决的问题是**上下文断裂**：每次新对话，Claude 都是"刚走进房间的新同事"。有了 Memory，它能记住你是资深后端工程师、你讨厌过度抽象、你的部署流程需要先跑 validate-pipeline。这些信息不属于任何单一项目，而是关于你和你的工作方式。

详细解析见我们的 [Claude Code 记忆系统详解](/zh/blog/claude-code-memory)。

## 概述：CLAUDE.md

CLAUDE.md 是 Claude Code 的项目级配置文件，位于项目根目录。它是一个纯 Markdown 文件，内容完全由你手动编写和维护。Claude Code 每次启动时自动读取，作为系统级指令注入上下文。

CLAUDE.md 解决的问题是**项目规范的一致性执行**。与其每次对话都重复"我们用 Tailwind v4"、"commit 前必须跑 build + test"、"中文内容用 CJK 分词计数"，不如写一次，永久生效。它本质上是一份"AI 工程师入职手册"——新人（Claude）第一天读完就知道该怎么干活。

CLAUDE.md 通常包含：构建命令、质量门禁、代码风格约定、已知陷阱、工作流规则。它是确定性的、版本控制的、团队共享的。关于如何有效编写项目指令，参考 [如何高效给 Claude Code 写提示词](/zh/blog/how-to-effectively-prompt-a-claude-code)。

|------|--------------|-----------|--------|
| **存储位置** | `~/.claude/projects/*/memory/` | 项目根目录 | Tie |
| **作用域** | 用户级（跨项目可共享） | 项目级（跟随 repo） | 各有所长 |
| **创建方式** | Claude 自动写入 | 人工手动编写 | CLAUDE.md（可控） |
| **版本控制** | 通常不入 git | 入 git，团队共享 | CLAUDE.md |

| **团队协作** | 个人专属 | 团队共享 | CLAUDE.md |

## 核心差异：规则 vs 记忆

这是理解两者关系的关键维度。CLAUDE.md 是**规则**——"你必须这样做"。Memory 是**记忆**——"我记得你说过这个"。

CLAUDE.md 的内容具有强制性。当你写下"commit 前必须跑 `npm run build` 和 `npm test`"，Claude 每次都会遵守，没有例外。它是确定性的行为约束，类似于 `.eslintrc` 对代码风格的约束——不是建议，是规则。

Memory 的内容具有参考性。当 Claude 记住"这个用户偏好单个大 PR 而非多个小 PR"，它会在决策时参考这条信息，但不会盲目执行——如果当前场景明显不适合，它会做出不同判断。Memory 更像是一位了解你的同事的判断力，而非机械的规则执行。

这意味着：如果某个约束对项目质量至关重要、不容违反，写进 CLAUDE.md。如果某个偏好是"一般情况下这样更好"，让 Memory 自然积累。更多关于 Claude Code 扩展体系的分层设计，参见 [Claude Code 扩展栈拆解](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)。

## 协作机制：两者如何配合

实际使用中，Memory 和 CLAUDE.md 形成分层系统：

**第一层：CLAUDE.md**（项目规范）
- 构建/测试命令
- 代码风格和架构约定
- 已知陷阱和 gotchas
- 文档更新规则

**第二层：Memory**（个人上下文）
- 你的角色和技术背景
- 你的沟通偏好（简洁 vs 详细）
- 过往反馈（"别这样做"、"上次这样很好"）
- 项目的非代码状态（谁负责什么、截止日期）

这种分层的好处是：CLAUDE.md 对团队所有人一致，Memory 对每个人个性化。同一个 repo，资深工程师和新人 intern 共享同一份 CLAUDE.md，但 Claude 基于各自的 Memory 用不同方式与他们协作。

关于 Claude 记忆系统的最新升级（包括跨平台导入），参见 [Claude 记忆系统大升级](/zh/blog/anthropic-claude-memory-upgrades-importing)。

## 什么时候该选 CLAUDE.md

以下场景，把信息写进 CLAUDE.md：

- **团队共享的规则**：编码规范、必须通过的质量门禁、部署流程——所有人都必须遵守的约束
- **项目特有的命令**：`npm run dev`、`npm test`、部署脚本——不因人而异
- **已知陷阱**：容易踩坑的地方、非直觉的约束（如"ZH 内容必须用 CJK 分词"）
- **架构决策**：技术栈选型、目录结构约定、API 设计原则
- **不容商量的规则**：比如"不允许本地有未提交的改动"、"不要在 pipeline 脚本里 import Next.js 模块"

简单判断标准：如果这条信息**换一个开发者也必须知道**，写 CLAUDE.md。

想了解 Claude Code 的完整能力框架，推荐阅读 [Claude Code 七层架构深度解析](/zh/blog/claude-code-seven-programmable-layers)。

## 什么时候该依赖 Memory

以下场景，让 Memory 自然积累：

- **个人偏好**：你喜欢简洁回复还是详细解释、commit message 的风格偏好
- **角色上下文**：你是前端还是后端、你在这个项目负责什么模块
- **反馈修正**：你纠正过 Claude 的某个行为（"别在每个回复末尾总结"）
- **项目动态**：某个功能的截止日期、某个同事负责的模块、当前 sprint 的重点
- **外部引用**：bug tracker 的位置、设计文档的链接、监控面板的 URL

简单判断标准：如果这条信息**只对你个人有意义**，或者**会随时间变化**，让 Memory 处理。

## 常见误区

**误区一：用 Memory 代替 CLAUDE.md**

有些开发者不写 CLAUDE.md，期望 Claude 通过 Memory 逐渐"学会"项目规范。问题是：Memory 是概率性的——Claude 可能记住，也可能遗漏。关键规则必须写进 CLAUDE.md 以确保 100% 执行。

**误区二：在 CLAUDE.md 里写个人偏好**

如果你在 CLAUDE.md 里写"回复要简洁"，团队其他人可能需要详细解释。个人偏好属于 Memory 层。

**误区三：重复存储**

不需要把 CLAUDE.md 里的规则也存一份到 Memory。Claude 启动时两者都会加载——重复只会浪费上下文窗口。

## 结论

**CLAUDE.md 是地基，Memory 是装修。** 先建好确定性的规则层（CLAUDE.md），再让个性化的记忆层（Memory）自然生长。两者协同工作时，Claude Code 既能遵守项目规范，又能理解你的个人工作方式。

如果你刚开始使用 Claude Code，优先投入时间写好 CLAUDE.md——它的 ROI 最高，一次编写，团队受益。Memory 则让你在日常使用中逐渐获得越来越个性化的协作体验。关于如何把这两套系统用到极致，参考 [我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

## 常见问题

### Claude Memory 的内容会被团队其他人看到吗？

不会。Memory 存储在用户本地的 `~/.claude/` 目录下，属于个人数据，不入版本控制，不会同步到远程仓库。团队成员各自有独立的 Memory 系统。

### CLAUDE.md 支持条件逻辑吗？

CLAUDE.md 是纯文本，不支持编程式的条件分支。但你可以用自然语言描述条件规则，比如"如果是 pipeline 相关改动，必须先跑 validate-pipeline.ts"。Claude 会理解并执行这类自然语言条件。

### 两者冲突时谁优先？

CLAUDE.md 的优先级更高。它作为系统级指令加载，具有强制性。如果 Memory 记录的偏好与 CLAUDE.md 的规则冲突，Claude 会遵循 CLAUDE.md。

### Memory 文件可以手动编辑吗？

可以。Memory 文件是标准 Markdown，你可以直接编辑、删除或新增。但通常建议通过对话让 Claude 自行管理——直接告诉它"记住这个"或"忘掉那个"即可。
