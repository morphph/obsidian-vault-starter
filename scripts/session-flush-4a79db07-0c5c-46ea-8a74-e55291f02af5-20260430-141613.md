# Claude Memory vs CLAUDE.md：哪种记忆机制更适合你的工作流？

**TL;DR：** **CLAUDE.md** 是团队共享的规则手册——确定性强、版本可控、适合编码规范和项目约束。**Claude Memory** 是个人化的自动笔记——对话驱动、跨会话持久、适合记住你的偏好和上下文。两者不是替代关系，而是互补的双层记忆架构。团队规范写进 CLAUDE.md，个人习惯交给 Memory。

## 概述：Claude Memory

**Claude Memory**（自动记忆）是 Claude Code 的对话级持久化系统，它会在你与 Claude 交互的过程中自动提取并保存关键信息——你的角色、技术偏好、项目背景、工作反馈。这些记忆以 Markdown 文件的形式存储在 `~/.claude/projects/` 目录下，每次新会话启动时自动加载。

Memory 的核心价值在于「用了就记住」。你不需要每次重复告诉 Claude「我是后端工程师」「这个项目用 Tailwind v4」「不要在测试里 mock 数据库」。告诉一次，它就永远知道。Anthropic 在 2026 年初对[记忆系统进行了大幅升级](/zh/blog/anthropic-claude-memory-upgrades-importing)，免费用户也能使用，还支持从 ChatGPT 导入历史数据。

Memory 按类型分为四类：用户画像（user）、工作反馈（feedback）、项目上下文（project）、外部引用（reference）。每条记忆都是独立的 Markdown 文件，带有 frontmatter 元数据，通过一个 `MEMORY.md` 索引文件进行管理。

## 概述：CLAUDE.md

**CLAUDE.md** 是 Claude Code 的项目级指令文件——一份签入代码仓库、随 Git 版本管理的规则手册。它告诉 Claude 这个项目是什么、该怎么构建、哪些事绝对不能做。每次 Claude Code 启动时，它会自动读取工作目录中的 CLAUDE.md 并将其内容注入系统提示。

CLAUDE.md 的设计哲学是「声明式规则」。你写下 `npm run build` 必须在每次提交前通过、不要在 pipeline 脚本里导入 Next.js 模块、中文内容必须用 CJK 字数统计——Claude 就会严格遵守。和 Memory 不同，CLAUDE.md 是团队共享的：每个克隆这个仓库的开发者，都会拿到同一份规则。

CLAUDE.md 还支持层级结构：全局 `~/.claude/CLAUDE.md` 定义个人通用规则，项目根目录的 `CLAUDE.md` 定义项目规则，子目录的 CLAUDE.md 可以覆盖上层设置。这种设计参考了 `.gitignore` 的级联逻辑，在[Claude Code 七层架构](/zh/blog/claude-code-seven-programmable-layers)中处于最基础的配置层。

## 记忆机制：深度分析

两种系统的底层机制完全不同，这决定了它们各自的最佳使用场景。

**Claude Memory 的工作原理：** Memory 是一个文件系统驱动的知识库。每条记忆是一个独立的 `.md` 文件，包含 `name`、`description`、`type` 三个 frontmatter 字段。一个 `MEMORY.md` 索引文件列出所有记忆的标题和链接。每次会话开始，Claude 读取 `MEMORY.md` 索引，根据当前对话上下文判断哪些记忆相关并加载。这意味着 Memory 是**按需召回**的——不是所有记忆都会出现在每次对话中。

**CLAUDE.md 的工作原理：** CLAUDE.md 的内容在每次会话开始时**全量注入**系统提示。不存在「是否相关」的判断——所有规则一律生效。这保证了确定性：你写的每条规则，Claude 每次都能看到。代价是 CLAUDE.md 不能太长，否则会占用宝贵的上下文窗口空间。

实际上，这两者在 [Claude Code 的扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)中处于不同层级。CLAUDE.md 是基础配置层，提供硬性约束；Memory 是上下文适配层，提供个性化调优。

## 团队协作：关键差异

这是两种机制最根本的分歧。

**CLAUDE.md 是团队基础设施。** 当你在 CLAUDE.md 中写下「提交前必须通过 `npm test`」，这条规则对所有团队成员生效。新人入职、克隆仓库、启动 Claude Code——他们自动获得和你一样的 AI 行为约束。这对于维护代码质量标准、统一工程实践至关重要。

**Memory 是个人工具。** 你的 Memory 记录了「这个用户是后端工程师，偏好简洁回答，不要在响应末尾加总结」。这些偏好只对你有意义。如果你的同事喜欢详细解释，他们的 Memory 会记录完全不同的内容。

这意味着：**团队规范永远写进 CLAUDE.md，个人偏好交给 Memory。** 如果你发现自己在 Memory 中存储了「这个项目用 Vitest 而不是 Jest」这类信息，它应该被迁移到 CLAUDE.md 中——因为这是所有团队成员都需要知道的事实。

关于 Claude Code 在团队场景下的更多实践，参考[Claude Code 记忆系统详解](/zh/blog/claude-code-memory)中对双层架构的完整分析。

## 适用场景：何时使用 Claude Memory

选择 Claude Memory 的典型场景：

- **个人偏好持久化**：你喜欢简洁回答、偏好函数式风格、不希望 Claude 在每个文件开头加注释——告诉它一次，Memory 自动记住
- **跨项目上下文**：你的角色是安全工程师，无论在哪个项目中 Claude 都应该优先关注安全问题。全局 Memory 解决这个需求
- **反馈积累**：你纠正过 Claude 的某个行为（「不要用 mock 数据库测试」），Memory 确保这个教训不会在下次会话中丢失
- **项目动态信息**：「下周四之前冻结非关键 PR」这类有时效性的信息，适合存在 Memory 而不是 CLAUDE.md

Memory 的核心优势是**零维护成本**——你正常工作，它在后台默默积累上下文。

## 适用场景：何时使用 CLAUDE.md

选择 CLAUDE.md 的典型场景：

- **构建命令和工作流**：`npm run build`、`npm test`、`npm run lint`——这些是项目级事实，每次会话都必须可用
- **编码规范和约束**：「不要在 pipeline 脚本里导入 Next.js 模块」「中文内容用 CJK 字数统计」——硬性规则需要确定性加载
- **新人引导**：CLAUDE.md 本质上是一份「AI 可读的项目文档」，新团队成员启动 Claude Code 后立刻获得完整的项目上下文
- **质量门禁**：定义提交前必须通过的检查列表，Claude 每次都会执行，不存在「忘记」的可能

如果一条规则被违反会导致构建失败或代码质量下降，它必须在 CLAUDE.md 中。关于如何设计高效的项目指令，[Claude Code 不是一个编程工具](/zh/blog/claude-code-is-not-a-coding-tool)一文有深入的方法论探讨。

## 最佳实践：双层搭配策略

最有效的用法是将两者视为互补的双层架构：

**第一层：CLAUDE.md（团队规则层）**
- 构建/测试/部署命令
- 编码规范和禁止事项
- 项目架构说明
- 文档更新规则

**第二层：Memory（个人适配层）**
- 沟通风格偏好
- 角色和专业背景
- 工作反馈记录
- 时效性项目信息

**决策规则：如果不确定该写在哪里，问自己一个问题——这条信息对团队里其他人也重要吗？** 是的话写 CLAUDE.md，不是的话交给 Memory。

一个常见的反模式是在 CLAUDE.md 中写入个人偏好（「回复要简洁」），或在 Memory 中存储项目规范（「用 Vitest 测试」）。前者会影响所有团队成员，后者可能在某次会话中不被召回。了解更多 Claude Code 的高效使用模式，可以参考[我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

## 判定

**CLAUDE.md 和 Claude Memory 解决不同问题，不存在「用哪个」的选择——你应该同时使用两者。** 把 CLAUDE.md 当作项目的宪法，把 Memory 当作你的私人秘书。CLAUDE.md 保证所有团队成员获得一致的 AI 行为；Memory 保证 AI 越用越懂你。如果你现在只用了其中一个，建议补上另一个——双层记忆架构的效果远大于单独使用任何一层。

## 常见问题

### Claude Memory 的内容会被分享给团队成员吗？

不会。Claude Memory 存储在本地 `~/.claude/projects/` 目录中，是完全私有的。只有 CLAUDE.md 通过 Git 仓库共享。如果某条信息需要团队共享，应该将其移到 CLAUDE.md 中。

### CLAUDE.md 太长会影响性能吗？

会。CLAUDE.md 的内容在每次会话中全量注入上下文窗口，过长的文件会挤压有效工作空间。建议控制在 500 行以内，将详细文档放在 `docs/` 目录中按需引用，而不是全部塞进 CLAUDE.md。

### 如果 Memory 和 CLAUDE.md 的指令冲突，哪个优先？

CLAUDE.md 优先。它作为系统级指令注入，具有更高的约束力。如果你发现 Memory 中的某条记录和 CLAUDE.md 矛盾，应该更新或删除 Memory 中的过时信息。
