**CLAUDE.md gets team-wide, stable, non-derivable instructions.** Build commands (`npm run build`), quality gates ("all tests must pass before commit"), coding conventions ("use snake_case for database columns"), and architectural constraints ("never import Next.js modules in pipeline scripts"). These rules change rarely, apply to everyone, and can't be figured out by reading the code.

**Memory gets personal, evolving, contextual knowledge.** Your role ("data scientist investigating logging"), corrections you've given ("don't mock the database in integration tests"), project status ("merge freeze starts March 5"), and external references ("pipeline bugs are tracked in Linear project INGEST"). These change frequently, apply to you specifically, and help Claude Code tailor its behavior without burdening CLAUDE.md with personal preferences.

A common mistake is overloading CLAUDE.md with information that belongs in Memory. If you add "user prefers terse responses" to a shared CLAUDE.md, every team member gets that preference applied. If you add "the auth rewrite is blocked on legal review" to CLAUDE.md, it stays there long after the review is complete unless someone remembers to remove it. Memory handles both cases better: personal preferences stay personal, and project status gets updated or removed as the situation evolves.

The reverse mistake — putting team rules in Memory — is equally problematic. If your "always run tests before committing" rule lives only in your Memory, new team members won't have it. And since Memory is managed by Claude Code, it could theoretically be modified or removed during conversation, while CLAUDE.md remains stable.

### What neither system should store

Both systems have explicit exclusions. Neither should store information derivable from the code itself — file paths, function signatures, import structures. Neither should duplicate git history. Neither should contain debugging recipes or fix instructions that belong in commit messages. The principle: CLAUDE.md and Memory store context that can't be found by reading the repo.

## Maintenance and Lifecycle

**CLAUDE.md requires deliberate maintenance.** You write it, you review it in PRs, you update it when the project evolves. This is both a strength (it's auditable and intentional) and a cost (it can go stale if no one updates it). Teams that treat CLAUDE.md like living documentation — updating it alongside the code it describes — get the most value. Teams that write it once and forget about it end up with instructions that drift from reality.

**Memory is self-maintaining but needs verification.** Claude Code creates, updates, and removes memory entries automatically. When you correct Claude Code's approach, it saves that correction as a feedback memory. When you share project context, it saves the relevant details. When information becomes stale, it should update or remove the entry. In practice, Memory entries can accumulate and occasionally become outdated — Claude Code is instructed to verify memories against current state before acting on them, but this isn't foolproof.

The maintenance tradeoff is clear: CLAUDE.md gives you full control at the cost of manual upkeep. Memory gives you automatic maintenance at the cost of occasional drift. For teams that want to learn how to effectively manage both systems, our guide on [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) covers practical patterns.

## Team Collaboration Patterns

**CLAUDE.md is the team alignment tool.** When checked into the repo, it ensures every developer using Claude Code gets the same baseline behavior. This is critical for consistency — without CLAUDE.md, one developer might get Claude Code to write tests with mocks while another gets real database integration tests. CLAUDE.md eliminates that variance.

Effective team CLAUDE.md files typically include:
- Build and test commands
- Quality gates (what must pass before committing)
- Coding style rules specific to the project
- Architectural constraints and known gotchas
- Documentation update requirements

**Memory is the individual calibration tool.** It lets Claude Code adapt to each developer's expertise level, communication style, and current focus. A senior engineer's Memory might contain "prefers terse responses, deep Go expertise, currently focused on the auth rewrite." A junior engineer's Memory might contain "new to React, benefits from explanations that connect to backend concepts they already know." Both developers work in the same repo with the same CLAUDE.md, but get appropriately different interactions.

This separation means teams don't have to choose between consistency and personalization. CLAUDE.md handles the "what we do and how we do it" layer. Memory handles the "who I am and what I need" layer.

## Real-World Configuration Patterns

### Pattern 1: Solo developer

For solo projects, the line between CLAUDE.md and Memory blurs. You could put everything in CLAUDE.md since you're the only user. But the better practice is still to separate:

- **CLAUDE.md**: Project build commands, architecture decisions, code conventions — anything you'd want preserved if you revisited the project in six months
- **Memory**: Your current focus areas, recent decisions and their rationale, what you've been debugging this week

This separation pays off when you context-switch between projects. Your Memory adapts per-project automatically, while CLAUDE.md reminds you of project-specific rules you may have forgotten.

### Pattern 2: Small team (2-5 developers)

The shared CLAUDE.md becomes essential. Include everything needed for consistent AI behavior: build commands, test requirements, commit message format, and architectural boundaries. Keep it under 200 lines — long CLAUDE.md files dilute the most important instructions.

Each developer's Memory naturally diverges based on their work. The backend developer's Memory reflects database and API context. The frontend developer's Memory reflects component patterns and styling conventions. Neither needs to coordinate their Memory with the other.

### Pattern 3: Open source project

CLAUDE.md is a contribution accelerator. New contributors who use Claude Code immediately get the project's conventions, build process, and constraints. They don't need to read a contributing guide end-to-end — CLAUDE.md front-loads the critical rules.

Memory plays a smaller role here since contributors come and go. But for maintainers, Memory accumulates valuable context about release processes, known issues, and community patterns that would be inappropriate to put in a shared CLAUDE.md.

## Interaction with Skills and Hooks

CLAUDE.md and Memory don't operate in isolation. They're part of Claude Code's broader [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp), and understanding the interactions prevents duplication.

**Skills** (`SKILL.md` files) define how Claude Code approaches specific tasks — writing tests, generating content, reviewing PRs. Skills are explicit and version-controlled like CLAUDE.md, but scoped to a specific activity rather than the whole project. If you find yourself putting task-specific instructions in CLAUDE.md ("when writing newsletters, use this tone"), consider moving them to a Skill file instead. For practical examples, see [5 Claude Code Skills I Use Every Single Day](/blog/5-claude-code-skills-i-use-every-single-day).

**Hooks** are deterministic shell commands that run at specific points in Claude Code's lifecycle — before a tool call, after a file edit, on session start. Hooks handle enforcement that CLAUDE.md can only request. CLAUDE.md says "run tests before committing." A hook actually blocks the commit if tests fail. For more on this enforcement layer, see the [Claude Code Hooks guide](/blog/claude-code-hooks-mastery).

**Memory** complements all of these by storing learned context that none of the explicit systems capture — your corrections, your preferences, your project's current state.

## Common Mistakes

**Mistake 1: Putting everything in CLAUDE.md.** Long CLAUDE.md files (500+ lines) dilute critical instructions. Claude Code processes the entire file every conversation, and important rules buried in a wall of text get less attention. Keep CLAUDE.md focused on the 20% of rules that matter 80% of the time. Move task-specific instructions to Skills.

**Mistake 2: Ignoring Memory entirely.** Some developers never correct Claude Code or share context, treating every conversation as isolated. This means Claude Code never learns their preferences and makes the same mistakes repeatedly. A few deliberate corrections early on — "don't use mocks in these tests," "I prefer concise responses" — compound across every future conversation.

**Mistake 3: Duplicating content across systems.** If CLAUDE.md says "use PostgreSQL, not SQLite" and Memory also stores "project uses PostgreSQL," the duplication creates maintenance risk. When the rule changes, one system gets updated and the other doesn't. Follow the principle: CLAUDE.md for rules, Memory for context. If it's a rule, it goes in one place.

**Mistake 4: Storing volatile information in CLAUDE.md.** Sprint goals, current blockers, deployment schedules — these change weekly. Putting them in CLAUDE.md means constant updates to a version-controlled file. Memory handles volatile information better because Claude Code can update and remove entries without requiring a commit.

## When to Choose Claude Memory

Use Memory as your primary context system when:

- You work solo and want Claude Code to adapt to your style without manually configuring files
- Your context is personal — role, preferences, communication style, expertise areas
- The information changes frequently — project status, current focus areas, recent decisions
- You want Claude Code to remember corrections without re-stating them each session
- You're tracking external references — where bugs are filed, which dashboards to check, where documentation lives

Memory shines for [agentic coding](/glossary/agentic-coding) workflows where you're delegating substantial tasks to Claude Code and want it to make good judgment calls based on accumulated context about you and the project.

## When to Choose CLAUDE.md

Use CLAUDE.md as your primary context system when:

- You work on a team and need consistent AI behavior across developers
- Your instructions are stable — build commands, quality gates, coding conventions
- You want version-controlled, auditable configuration
- The rules are non-negotiable — architectural constraints, security requirements, style enforcement
- You want new team members or contributors to get the right behavior immediately

CLAUDE.md is essential for any project where multiple people use Claude Code. Without it, each developer gets different AI behavior based on their individual prompting habits, leading to inconsistent code quality and style.

## Verdict

**Use both — they solve different problems.** CLAUDE.md is your project's constitution: shared, explicit, stable rules that every Claude Code session follows. Memory is your personal context layer: learned preferences, corrections, and evolving project knowledge that makes Claude Code smarter about how it works with you specifically.

If forced to choose one: **CLAUDE.md for teams, Memory for solo developers.** But the strongest setup uses CLAUDE.md for team standards and lets Memory handle everything personal and volatile. The [complete Claude Code guide](/blog/claude-code-complete-guide) covers how to set up both systems from scratch.

Start with a focused CLAUDE.md (under 100 lines) covering build commands, quality gates, and the top 5 rules your team enforces. Then let Memory accumulate naturally by correcting Claude Code when it gets something wrong and sharing context when it's missing information. Over a few sessions, you'll have a context system that's both consistent and personalized.

## Frequently Asked Questions

### Does Claude Memory replace CLAUDE.md?

No. Memory and CLAUDE.md serve different purposes. CLAUDE.md holds explicit, team-shared project rules — build commands, coding standards, quality gates. Memory stores personal, learned context — your role, preferences, and corrections. They load in the same conversation and complement each other. Removing either one loses a distinct type of context.

### Can teammates see my Claude Memory?

No. Claude Memory is stored locally in your `~/.claude/projects/` directory, scoped to your user account and the specific project. Teammates never see your Memory entries. This is by design — Memory contains personal preferences and context that wouldn't be appropriate to share. Team-wide instructions belong in CLAUDE.md, which is shared via version control.

### How do I migrate information between Memory and CLAUDE.md?

If a Memory entry keeps coming up across team members — the same correction, the same context — that's a signal it belongs in CLAUDE.md. Copy the rule into CLAUDE.md, commit it, and let Memory naturally stop storing it (Claude Code avoids duplicating what CLAUDE.md already covers). Going the other direction, if a CLAUDE.md rule only applies to you, move it to your user-level `~/.claude/CLAUDE.md` or let Memory capture it.

### What happens if Memory and CLAUDE.md contradict each other?

CLAUDE.md takes precedence. It loads as part of the system context and represents explicit instructions, while Memory provides supplementary context. If CLAUDE.md says "use tabs for indentation" and your Memory says "user prefers spaces," Claude Code follows CLAUDE.md. To resolve contradictions, update whichever system has the outdated information.

### How large should CLAUDE.md be?

Keep it under 200 lines. Long CLAUDE.md files dilute critical instructions — Claude Code processes the entire file every conversation, and important rules buried in hundreds of lines get less weight. Focus on build commands, quality gates, top architectural constraints, and the most important coding conventions. Move task-specific instructions to Skill files.
