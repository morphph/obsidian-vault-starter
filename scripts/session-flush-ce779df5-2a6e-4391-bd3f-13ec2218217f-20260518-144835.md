# Codex Subagents vs Custom Agents：子代理与自定义代理的实战选择

**TL;DR：** **Subagents** 是 Codex 内置的任务分解机制，适合在一个任务内自动拆分并行子任务；**Custom Agents** 是用户自定义的专用代理配置，适合团队级别的标准化工作流。如果你需要单次复杂任务的并行执行，选 Subagents；如果你需要可复用的专业化代理角色，选 Custom Agents。两者可以组合使用。

## 概述：Codex Subagents

**Codex Subagents** 是 OpenAI Codex 平台内置的子代理机制，允许一个父任务在执行过程中自动派生出多个子任务并行运行。这解决了传统线性执行模式下复杂工程任务耗时过长的问题。

Subagents 的核心思路是「分而治之」：当 Codex 判断一个任务可以被拆解时，它会自动创建子代理分别处理不同的子问题——比如同时重构三个独立模块、并行生成多个文件的测试用例，或者一边修改代码一边更新文档。每个子代理在独立的沙箱环境中运行，完成后将结果汇总到父任务。

这种机制对大型代码库尤其有价值。根据 OpenAI 的介绍，Codex 的云端架构让每个子代理都能获得独立的计算资源和完整的代码上下文，不会因为并行执行而相互干扰。更多关于 Codex 平台能力的详细分析，参见我们的 [Codex 完全指南](/zh/blog/codex-complete-guide)。

## 概述：Codex Custom Agents

**Codex Custom Agents** 是用户在 Codex 平台上自定义的专用代理配置。你可以为不同的工程场景预设指令、工具链和行为模式，创建具有特定角色和专业能力的代理。

Custom Agents 解决的是另一个问题：标准化和复用。一个团队可能需要「代码审查代理」「测试生成代理」「文档更新代理」等多种角色，每种角色有不同的指令集、关注点和质量标准。Custom Agents 允许你把这些配置固化下来，团队成员可以直接调用预设好的代理，而不是每次都从头描述需求。

这与 Claude Code 的 [Skills 系统](/zh/blog/9-principles-writing-claude-code-skills) 在理念上相似——都是通过可复用的指令配置来提升 AI 代理的一致性和输出质量。区别在于 Codex Custom Agents 运行在云端，更适合异步、团队协作的场景。

## 功能对比

| 维度 | Subagents | Custom Agents | 优势方 |
|------|-----------|---------------|--------|
| **触发方式** | 系统自动拆分 | 用户手动创建和调用 | 各有所长 |
| **并行执行** | 原生支持，自动编排 | 需手动启动多个代理 | Subagents |
| **配置复用** | 临时生成，任务结束即销毁 | 持久化配置，跨任务复用 | Custom Agents |
| **指令定制** | 继承父任务指令 | 完全自定义指令集 | Custom Agents |
| **适用粒度** | 单个任务内的子问题 | 团队级别的工作流角色 | 各有所长 |
| **上手难度** | 零配置，自动触发 | 需要前期配置投入 | Subagents |
| **一致性控制** | 依赖模型判断 | 用户完全掌控 | Custom Agents |

## 任务分解能力：详细分析

Subagents 最大的优势在于自动化的任务分解。当你向 Codex 提交一个涉及多文件、多模块的复杂任务时，系统会分析任务结构，识别可并行执行的部分，自动派生子代理。

这种自动编排在以下场景表现突出：

- **跨模块重构**：同时修改多个相互独立的模块，每个子代理处理一个模块
- **批量测试生成**：为多个文件并行生成单元测试
- **代码迁移**：同时处理多个文件的 API 升级或语法迁移

但 Subagents 的局限也很明显：你无法精确控制拆分策略。模型会自行决定哪些部分可以并行、如何分配子任务。如果任务之间存在隐含的依赖关系，自动拆分可能导致冲突或不一致。

相比之下，Custom Agents 在任务分解上需要更多人工介入，但提供了更精确的控制。你可以明确指定每个代理负责的范围、使用的工具和遵循的规则。关于[多代理工作流的工程范式](/zh/blog/con-u-pour-des-workflows-multi-agents)，我们有更深入的分析。

## 可复用性与团队协作：详细分析

Custom Agents 的核心价值在于可复用性。一次配置，反复使用。

假设你的团队有以下标准流程：每次 PR 提交前需要代码审查、安全扫描、文档同步。你可以创建三个 Custom Agents：

1. **Code Reviewer Agent**：关注代码质量、性能模式、错误处理
2. **Security Scanner Agent**：检查 OWASP Top 10 漏洞、依赖安全
3. **Doc Sync Agent**：确保代码变更与文档保持一致

这些代理配置一旦定义好，团队中任何人都可以直接使用，不需要每次重新描述需求。这在大型团队中尤其重要——它确保了 AI 辅助工作流的一致性，不会因为不同成员的提示词风格不同而产生差异化的输出。

Subagents 则不具备这种持久化的复用能力。它们是临时的、一次性的——为当前任务服务，任务结束后就不再存在。如果你在 Codex 中多次执行类似的复杂任务，每次的子代理拆分策略可能不同。

这与 Claude Code 的 [Agent Teams](/zh/blog/claude-code-agent-teams) 机制形成了有趣的对照：Claude Code 的 Agent Teams 也支持并行子代理，但它结合了 CLAUDE.md 和 SKILL.md 的持久化配置，在自动编排和可复用性之间找到了平衡点。

## 什么时候选 Subagents

Subagents 是你的默认选择，当：

- **任务规模大但结构清晰**：重构、批量修改、多文件测试生成等天然可并行的任务
- **你不想花时间配置**：直接提交任务，让 Codex 自动决定如何拆分
- **一次性任务**：不需要反复执行的复杂操作
- **探索性工作**：不确定最优拆分策略，让模型先试一试

Subagents 的优势是零摩擦。你不需要预先思考任务应该如何分解，Codex 会根据代码结构和任务描述自动判断。对于[刚接触 Codex 的开发者](/zh/blog/first-few-days-with-codex-cli)来说，这大幅降低了上手门槛。

## 什么时候选 Custom Agents

Custom Agents 是更好的选择，当：

- **工作流需要标准化**：团队有固定的代码审查、测试、部署流程
- **输出质量要求高且一致**：同类任务每次都需要遵循相同的规则和标准
- **需要精确控制代理行为**：指定工具链、限制操作范围、定义输出格式
- **跨团队协作**：不同团队成员需要使用相同的 AI 工作流

如果你已经在使用 Claude Code 的 [Skills 系统](/zh/blog/5-claude-code-skills-i-use-every-single-day)，Custom Agents 的理念会很熟悉——本质上都是把「好的提示词」固化为可复用的配置。

## 组合使用：最佳实践

Subagents 和 Custom Agents 并不互斥。实际生产环境中，最有效的策略是组合使用：

1. **用 Custom Agents 定义角色**：创建符合团队标准的专用代理
2. **在 Custom Agent 内利用 Subagents 并行执行**：当自定义代理处理复杂任务时，允许它自动派生子代理
3. **用 Custom Agent 做质量门控**：让专用的审查代理检查 Subagents 的输出

这种分层架构——Custom Agents 负责「做什么」和「怎么做」，Subagents 负责「如何高效执行」——在[多代理编程实战](/zh/blog/claude-code-subagents-examples)中已经被验证为有效模式。

## 结论

**如果你是个人开发者处理复杂任务，Subagents 的自动并行能力让你事半功倍。如果你是团队负责人需要标准化 AI 工作流，Custom Agents 的可复用配置才是正确投资。** 大多数成熟团队最终会走向组合使用：Custom Agents 定义标准，Subagents 加速执行。

需要注意的是，Codex 平台仍在快速迭代中，Subagents 和 Custom Agents 的具体能力边界可能随版本更新而变化。建议关注 OpenAI 官方文档获取最新信息。关于 [Codex 与 Claude Code 在代理架构上的不同取舍](/zh/blog/codex-complete-guide)，我们的深度分析提供了更完整的视角。

## 常见问题

### Codex Subagents 需要手动配置吗？

不需要。Subagents 由 Codex 自动触发和管理。当系统判断任务可以并行拆分时，会自动创建子代理。用户无需预先配置，但也无法精确控制拆分策略。这是 Subagents「零配置」优势的来源，也是其灵活性受限的原因。

### Custom Agents 和 Claude Code 的 SKILL.md 有什么区别？

两者理念相似——都是通过持久化配置提升 AI 代理的一致性。主要区别在于运行环境：Custom Agents 在 Codex 云端异步执行，适合团队协作和 CI/CD 集成；[SKILL.md](/zh/glossary/agentic-coding) 运行在本地终端的 Claude Code 中，适合个人开发者的实时交互场景。

### 两者可以同时使用吗？

可以。推荐的做法是用 Custom Agents 定义工作流角色和质量标准，同时允许这些自定义代理在执行复杂任务时自动触发 Subagents 进行并行处理。这样既有标准化保障，又能享受并行执行的效率提升。
