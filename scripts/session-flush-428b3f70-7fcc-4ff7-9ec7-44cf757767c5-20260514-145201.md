**Codex's model** treats each task as a fully independent unit. When you submit a task to Codex, it spins up a sandboxed container with your repo, runs the agent to completion, and returns a diff. If you submit five tasks, you get five independent containers running in parallel. There is no mechanism for Task A to ask Task B a question, share an intermediate result, or coordinate on a shared resource like a database schema. Each task sees only the repo snapshot it started with.

This independence is both Codex's strength and limitation. It means tasks cannot interfere with each other — no race conditions, no conflicting edits to resolve mid-execution. But it also means you cannot model workflows where one agent's output feeds another's input without manual intervention. If Agent A needs to generate an interface definition before Agent B writes the implementation, you must run them sequentially and manually pass context between runs.

**Claude Code's model** uses a parent-child architecture. The main Claude Code session acts as an orchestrator that can spawn specialized subagents using the Agent tool. Each subagent receives a prompt from the parent, does its work, and returns a result that the parent incorporates into its ongoing context. The parent can then spawn additional subagents based on those results, creating dynamic multi-step workflows.

Claude Code ships with [built-in agent types](/blog/claude-code-agent-teams): **Explore** for fast read-only codebase search, **Plan** for architecture design, and a general-purpose type for research and execution. Each type has access to different tool subsets — Explore agents cannot edit files, Plan agents cannot write code — which enforces separation of concerns at the architecture level.

The practical impact: in Claude Code, you can say "search the codebase for all API endpoints, then design a test plan, then implement the tests" and the orchestrator will spawn an Explore agent to find endpoints, pass those results to a Plan agent for test design, and then delegate implementation to a coding agent. Each step builds on the previous one within a single session. In Codex, you would need to run three separate tasks with manual context bridging between them.

## Custom Agent Definitions: Detailed Analysis

Customizing agent behavior is where the two platforms diverge most sharply, and it is the core consideration when evaluating how to use subagents and custom agents in Codex versus Claude Code.

**Codex** allows customization primarily through the task prompt and repository-level configuration. You can set a `AGENTS.md` file (analogous to Claude Code's `CLAUDE.md`) that provides project context to every Codex task. Beyond that, customization happens at the prompt level — you write detailed instructions for each task describing what the agent should do, which files to focus on, and what conventions to follow. There is no formal mechanism to define reusable agent "types" with constrained toolsets or behavioral profiles.

Codex's customization is effective but coarse-grained. You can tell a task "only modify test files" in the prompt, but there is no enforcement mechanism — the agent operates in a full sandbox with write access to the entire repo. Discipline comes from prompt quality, not architectural constraints.

**Claude Code** offers a layered customization system documented in detail in our [extension stack deep-dive](/blog/claude-code-extension-stack-skills-hooks-agents-mcp). Custom agents can be defined with specific:

- **Toolsets**: Restrict which tools (file editing, shell access, web search) a subagent can use
- **Model selection**: Choose different Claude models (Opus, Sonnet, Haiku) for different agent types based on capability-cost tradeoffs
- **System prompts**: Define behavioral instructions that shape how the agent approaches its task
- **Isolation modes**: Run subagents in git worktrees for safe parallel file editing without conflicts

The [SKILL.md system](/blog/5-claude-code-skills-i-use-every-single-day) extends this further. Skills are reusable instruction files that encode specific workflows — a "test-writer" skill, a "security-reviewer" skill, a "code-simplifier" skill. When you invoke a skill, it effectively creates a custom agent persona with domain-specific instructions. Our guide on [writing effective skills](/blog/9-principles-writing-claude-code-skills-i-use-every-single-day) covers the principles behind making these reliable.

The practical difference: Codex gives you a powerful general-purpose agent you customize per-task through prompting. Claude Code gives you an agent framework where you define specialized agents with enforced constraints, reusable across sessions and team members.

## Execution Environment and Safety

**Codex** runs every task in a cloud sandbox with network disabled by default. This is a significant safety advantage — a misbehaving agent cannot accidentally `rm -rf /` your local machine, push to production, or exfiltrate code. The sandbox boundary means Codex tasks are inherently safe to run unattended. You review the resulting diff and decide whether to merge, just like reviewing a pull request from a colleague.

**Claude Code** runs locally with real-time permission controls. Every potentially destructive action — shell commands, file writes, git operations — requires user approval unless explicitly allow-listed. The [hooks system](/blog/claude-code-hooks-a-complete-guide-to-automating-your-ai-coding-workflow) adds a deterministic safety layer: you can define shell commands that run before or after specific tool calls, blocking dangerous operations programmatically. Subagents in Claude Code inherit the parent session's permission model, meaning a subagent cannot escalate privileges beyond what the user has approved.

The tradeoff is clear: Codex's sandbox model is safer for fire-and-forget workflows where you do not want to supervise execution. Claude Code's local model is more powerful — subagents can run your actual build tools, access your real database, and test against your local environment — but requires active supervision or carefully configured permission boundaries.

For teams evaluating how to **use subagents and custom agents in Codex** for automated pipelines (CI-triggered code generation, batch refactoring across repos), Codex's sandboxing makes it the safer choice for unsupervised execution. For interactive development sessions where subagents need access to your actual dev environment, Claude Code's local execution model is essential.

## Workflow Patterns: When Each Approach Shines

### Pattern 1: Batch Refactoring

**Codex wins.** You have 20 microservices that need a dependency updated, API signatures changed, and tests adjusted. Fire off 20 Codex tasks, one per service, and review the PRs as they complete. Each sandbox gets a clean repo state, and there is no coordination needed between tasks.

Doing this in Claude Code would require either sequential execution (slow) or using git worktree isolation for parallel subagents (possible but more complex to orchestrate). The [worktree architecture](/blog/anatomy-of-git-worktree-add) makes this technically feasible, but Codex's cloud-native parallelism is purpose-built for this pattern.

### Pattern 2: Feature Implementation With Discovery

**Claude Code wins.** You need to add a new API endpoint that touches the database layer, the service layer, and the frontend. First you need to understand the existing patterns, then design the approach, then implement across multiple files.

Claude Code's subagent orchestration handles this naturally: spawn an Explore agent to map the existing codebase patterns, pass findings to a Plan agent for architecture design, get user approval on the plan, then delegate implementation to a coding agent. The parent orchestrator maintains context across all phases. In Codex, you would need to run the discovery task, manually read the output, formulate the next task prompt incorporating those findings, and repeat — a significantly slower feedback loop.

### Pattern 3: Code Review Automation

**Tie — different strengths.** Codex can review PRs asynchronously, leaving comments and suggestions through its GitHub integration. Claude Code's [review capabilities](/blog/claude-code-review-agents) operate in real-time within your terminal, with subagents analyzing different aspects of a change in parallel (security, performance, style).

Codex's async model fits better into existing PR workflows. Claude Code's real-time model gives faster feedback during active development. Many teams benefit from both — Claude Code for pre-commit review, Codex for post-push CI review.

### Pattern 4: Multi-Agent Research and Implementation

**Claude Code wins decisively.** Complex tasks requiring multiple specialized agents — one searching documentation, another analyzing code patterns, a third writing implementations — play to Claude Code's orchestration model. The parent agent can dynamically decide which subagent to spawn based on intermediate results, creating adaptive workflows that respond to what the codebase actually contains.

Codex's independent-task model cannot replicate this without human-in-the-loop orchestration between each step. If your workflow requires agents to share context, coordinate outputs, or make decisions based on each other's findings, Claude Code's architecture is fundamentally better suited.

- **Team scale**: Multiple team members can submit tasks to a shared Codex queue

## When to Choose Claude Code Subagents

Choose Claude Code when your multi-agent workflow requires coordination, context sharing, or interactive oversight:

- **Interactive development sessions**: You want to steer the agent as it works, not just review output

- **Local environment access**: Subagents need to run your build tools, test suites, or access local services
- **Skill-based workflows**: Reusable agent personas encoded as [SKILL.md files](/blog/claude-code-extension-stack-skills-hooks-agents-mcp)
- **Dynamic orchestration**: The parent agent decides what to do next based on subagent results

Claude Code's [agent team system](/blog/claude-code-agent-teams) is purpose-built for workflows where the whole is greater than the sum of the parts — where coordination between agents creates value that independent parallel tasks cannot match.

## Frequently Asked Questions

### Can Codex subagents share context with each other?

No. Each Codex task runs in an independent sandbox with its own repository checkout. There is no shared memory, message-passing, or coordination mechanism between concurrent tasks. To pass context between tasks, you must run them sequentially and manually incorporate results into the next task's prompt.

### How many subagents can Claude Code run in parallel?

Claude Code can spawn multiple subagents concurrently by including multiple Agent tool calls in a single response. Each subagent gets its own context window and tool access. For file-editing subagents, git worktree isolation prevents conflicts. The practical limit depends on API rate limits and the complexity of each subagent's task.

### Is it possible to define custom agent types in Codex?

Codex does not support formally defined agent types with constrained toolsets. Customization happens through task-level prompts and repository-level configuration files. You can instruct a Codex task to behave as a specific type of agent in the prompt, but there is no enforcement mechanism preventing it from acting outside those instructions.

### Which platform is cheaper for multi-agent workflows?

It depends on volume and workflow type. Codex is bundled with ChatGPT Pro ($200/month) and Team subscriptions, making it cost-effective for teams already on those plans. Claude Code uses per-token API billing, which can be cheaper for light usage but scales with consumption. For heavy multi-agent workflows with many parallel subagents, compare your projected token usage against Codex's flat subscription cost.

### Can I use both Codex and Claude Code subagents in the same project?

Yes. The tools operate independently — Codex runs in the cloud against your GitHub repository, while Claude Code runs locally in your terminal. A common pattern is using Claude Code subagents for interactive development and Codex for automated batch tasks in CI. There is no conflict between the two, though you should coordinate to avoid overlapping changes to the same files.
