**CLAUDE.md** is committed to your repository. Anything in it is visible to anyone with repo access. Never put secrets, API keys, personal information, or sensitive business logic in CLAUDE.md. It's appropriate for *instructions*, not *credentials*.

**Claude Memory** is stored locally in `~/.claude/projects/`. It's not committed to version control and doesn't leave your machine (unless your home directory is synced to a cloud service). This makes it more appropriate for personal context, but it still shouldn't contain secrets — Memory files are plain-text markdown, readable by anyone with filesystem access.

Neither system is designed for secret storage. Use environment variables, secret managers, or encrypted vaults for credentials.

## Migration: Moving Between Systems

If you've been putting everything in CLAUDE.md and want to adopt Memory, or vice versa, here's how to migrate cleanly.

### From CLAUDE.md to Memory

Review your CLAUDE.md for content that's personal rather than project-level. Common candidates:

- "I prefer..." statements
- Developer-specific workflow notes
- Temporary project context ("we're currently...")
- References to your personal tools or environments

Tell Claude to save these as appropriate memory types, then remove them from CLAUDE.md. Commit the slimmer CLAUDE.md.

### From Memory to CLAUDE.md

Review your memory directory for content that the whole team would benefit from. Common candidates:

- Gotchas you discovered that aren't documented
- Build or test commands that aren't in CLAUDE.md
- Architecture decisions that should be team knowledge
- Convention corrections that apply to the project, not just you

Add these to CLAUDE.md, commit, and delete the corresponding memory files.

## When to Choose Claude Memory

Choose Memory when the context is:

- **Personal**: Your role, expertise level, communication preferences
- **Conversational**: Feedback you've given Claude about its behavior ("don't do X", "keep doing Y")
- **Transient**: Project facts with a shelf life — current sprint goals, temporary environment issues, ongoing incidents
- **Individual workflow**: How you prefer Claude to structure commits, PRs, or explanations
- **External references**: Links to your team's Linear project, Slack channel, or Grafana dashboard

Memory shines when you want Claude to *learn from working with you* — adapting over time without you maintaining a config file.

## When to Choose CLAUDE.md

Choose CLAUDE.md when the context is:

- **Universal**: Build commands, test commands, deployment steps — facts that apply to everyone
- **Stable**: Architectural constraints, coding conventions, style guides — rules that change slowly
- **Critical**: Quality gates, pre-commit requirements, security constraints — rules that must never be skipped
- **Structural**: Project layout explanations, module responsibilities, known gotchas — the project's "onboarding doc" for AI
- **Reviewable**: Anything that benefits from PR review, `git blame`, or change history

CLAUDE.md shines when you want Claude to follow *project rules consistently* — the same way for every developer, every session, with changes tracked in version control.

## Verdict

**Use both.** CLAUDE.md and Claude Memory are complementary systems, not competitors. CLAUDE.md is your project's constitution — the shared, reviewed, version-controlled foundation. Memory is your personal relationship with Claude — the accumulated context of how you work together.

**If you're starting from scratch**, write a CLAUDE.md first. Include your build commands, test commands, key conventions, and known gotchas. This gives Claude the baseline it needs to be useful. Memory will build itself naturally as you work — Claude learns your preferences from corrections and confirmations.

**If you're already using CLAUDE.md for everything**, audit it for personal content and migrate those pieces to Memory. Your CLAUDE.md should read like a project onboarding doc, not a personal notebook.

**If you're relying entirely on Memory**, create a CLAUDE.md for the project fundamentals. Memory is great for adaptation, but a project needs a stable foundation that doesn't depend on one developer's conversation history.

## Frequently Asked Questions

### Can Claude Memory override instructions in CLAUDE.md?

Memory and CLAUDE.md occupy different roles — CLAUDE.md provides project instructions while Memory provides personal context. In practice, Claude treats CLAUDE.md as authoritative project rules and Memory as supplementary context. If a Memory entry contradicts a CLAUDE.md instruction, CLAUDE.md takes priority because it represents the team's deliberate, reviewed decisions.

### Does Claude Memory sync across devices?

No. Claude Memory is stored locally in `~/.claude/projects/` on your machine. If you work from multiple computers, each will have its own independent memory. CLAUDE.md, by contrast, syncs automatically through version control — clone the repo, and you get the CLAUDE.md. There is no built-in mechanism to sync Memory files between machines, though you could manually copy the directory.

### How do I see what Claude has saved in Memory?

Browse the files in `~/.claude/projects/<project-id>/memory/`. Each memory is a separate markdown file with typed frontmatter. The `MEMORY.md` index file lists all memories with one-line descriptions. You can also ask Claude directly — "what do you remember about me?" or "show me your memories" — and it will read and summarize its memory files.

### Should I commit CLAUDE.md to a public repository?

Yes, with care. CLAUDE.md should contain project instructions, not secrets. Review it before committing to ensure it doesn't include API keys, internal URLs, employee names, or sensitive business logic. For public repos, CLAUDE.md serves double duty: it helps contributors' Claude sessions follow your project's conventions, which improves the quality of AI-assisted contributions.

### Can I disable Claude Memory entirely?

You can delete the memory directory or ask Claude not to save memories during a session. However, completely disabling Memory means Claude starts every conversation without personal context — it won't remember your preferences, past corrections, or workflow decisions. For most developers, the better approach is periodic memory hygiene: reviewing and pruning outdated entries rather than disabling the system entirely.
