# Claude Code 子代理 vs OpenAI Codex 代理：多 Agent 编排方案对比

**TL;DR：** 两套系统解决同一个问题——让 AI 编程 Agent 学会"分工"——但路径截然不同。**Claude Code 子代理**是本地终端内的即时分身，主 Agent 在对话中直接 spawn 子 Agent，共享文件系统和 Git 仓库，适合需要快速并行探索或隔离实验的工作流。**OpenAI Codex** 走的是云端沙箱路线，每个任务在独立容器中运行，天然隔离但需要显式同步。选 Claude Code 如果你要灵活的本地编排；选 Codex 如果你要严格隔离的云端并行。

## 概述：Claude Code 子代理系统

Claude Code 的子代理（Subagents）是 Anthropic [Agent Teams](/zh/blog/claude-code-agent-teams) 架构的核心能力。主 Agent 在终端会话中通过 `Agent` 工具直接 spawn 子代理，每个子代理有独立的上下文窗口和工具权限，但共享底层文件系统。

这套设计的关键优势是**零部署成本**——子代理运行在你的本地终端，不需要容器化或云端基础设施。你可以给子代理指定不同的角色类型（Explore、Plan、codex-rescue 等），也可以通过 `isolation: "worktree"` 让子代理在 Git worktree 中工作，实现文件级别的隔离。整个流程对用户透明：主 Agent 分配任务，子代理执行后返回结果，主 Agent 综合决策。

从定价角度看，子代理调用按 API token 消耗计费，没有额外的平台费用。具体的子代理使用模式和实战案例，可以参考我们的 [Claude Code 子代理实战指南](/zh/blog/claude-code-subagents-examples)。

## 概述：OpenAI Codex 代理系统

[OpenAI Codex](/zh/blog/codex-complete-guide) 采用完全不同的架构思路。每个编程任务在云端独立沙箱中运行——一个隔离的容器环境，包含代码仓库快照、依赖安装和运行时。Codex 的"多代理"能力体现为**多任务并行**：你可以同时提交多个任务，每个任务在各自的沙箱中独立执行。

Codex 的自定义代理（Custom Agents）通过 `AGENTS.md` 文件配置项目级指令，类似于 Claude Code 的 `CLAUDE.md`。你可以定义代理的行为规范、编码标准和任务约束。2026 年 Codex 还推出了 [VS Code 扩展](/zh/blog/codex-vscode)，让你在 IDE 内直接启动代理任务。

定价方面，Codex 对 ChatGPT Pro（$200/月）和 Plus（$20/月）用户包含不同额度的使用量，Enterprise 和 Team 计划也有独立配额。[面向学生的免费额度](/zh/blog/codex-for-students)降低了入门门槛。

## 功能对比

| 功能维度 | Claude Code 子代理 | OpenAI Codex 代理 | 优势方 |
|---------|-------------------|-------------------|--------|
| **运行环境** | 本地终端 | 云端沙箱容器 | 各有利弊 |
| **任务隔离** | 可选（worktree 模式） | 默认隔离 | Codex |
| **并行能力** | 主 Agent 内 spawn 多个子代理 | 同时提交多个独立任务 | 平手 |
| **上下文共享** | 共享文件系统和 Git 状态 | 各任务独立快照 | Claude Code |
| **自定义指令** | CLAUDE.md + SKILL.md | AGENTS.md | Claude Code |
| **子代理类型** | 多种内置类型（Explore、Plan 等） | 通用任务执行 | Claude Code |
| **结果整合** | 主 Agent 自动综合 | 手动 review 各任务 PR | Claude Code |
| **网络访问** | 取决于本地环境 | 沙箱内受限 | Claude Code |
| **定价模式** | 按 token 计费 | 订阅制 + 配额 | 取决于用量 |

## 架构差异：本地编排 vs 云端沙箱

Claude Code 和 Codex 在多 Agent 编排上的根本分歧在于**信任模型**。

Claude Code 选择信任本地环境。子代理直接访问你的文件系统、终端和 Git 仓库，这意味着它们能读取完整的项目上下文，执行任意 shell 命令，甚至直接推送代码。这种设计的好处是灵活性极高——子代理之间可以实时协调，主 Agent 能根据子代理的发现动态调整策略。缺点是安全边界模糊：一个子代理的误操作可能影响整个工作目录。

Codex 选择了相反的路径：**默认不信任**。每个任务在隔离容器中运行，没有网络访问（除非显式配置），不能影响其他任务的环境。这对企业场景极具吸引力——你可以让多个开发者同时提交任务，互不干扰。但代价是灵活性：任务之间无法实时通信，结果整合需要人工介入。

从 [Agent 扩展栈](/zh/blog/claude-code-extension-stack-skills-hooks-agents-mcp)的角度看，Claude Code 的子代理是其四层架构（Skills、Hooks、Agents、MCP）中的关键一层，与其他三层紧密集成。Codex 的代理系统则相对独立，主要通过 AGENTS.md 和任务描述来配置行为。

## 自定义代理能力：深度 vs 简洁

Claude Code 的自定义能力建立在多层配置系统上。[CLAUDE.md 规则手册](/zh/blog/claude-code-memory)定义项目级约束，[SKILL.md 文件](/zh/blog/9-principles-writing-claude-code-skills)编码特定任务的执行标准，Hooks 在工具调用前后插入自动化逻辑。子代理可以继承这些配置，也可以通过 `subagent_type` 参数选择特化角色——Explore 用于代码搜索、Plan 用于架构设计、codex-rescue 用于复杂调试。

这套系统的深度令人印象深刻，但学习曲线也更陡峭。你需要理解 CLAUDE.md、SKILL.md、Hooks、MCP 四层的交互方式，才能充分发挥子代理的能力。关于 Skills 的实际效果，我们在 [Skills 是否真的能提升 Agent 输出质量](/zh/blog/do-skills-actually-improve-your-agents-output)中做过详细分析。

Codex 的自定义方案更简洁：一个 AGENTS.md 文件，加上任务描述中的自然语言指令。没有多层配置，没有 Hook 系统，没有角色类型。对于"我只想让 AI 帮我改几个 bug"的场景，这种简洁是优势。但当你需要精细控制代理行为——比如"搜索时只读不写"或"生成代码后自动运行测试"——Codex 的配置能力就显得不够用。

## 适用场景：选 Claude Code 子代理

以下场景 Claude Code 子代理更合适：

**大型重构任务。** 你需要一个 Explore 子代理先扫描全仓库找出所有引用点，一个 Plan 子代理设计重构方案，主 Agent 执行实际修改。这种多角色协作在 Codex 中很难实现，因为各任务之间无法共享发现。

**需要即时反馈的探索性工作。** 子代理的结果直接返回给主 Agent，主 Agent 可以根据结果调整后续指令。这种交互式工作流比 Codex 的"提交-等待-review"模式快得多。

**已有 Claude Code 工作流的团队。** 如果你已经投入了 CLAUDE.md 和 SKILL.md 的配置，子代理能直接复用这些配置，边际成本几乎为零。具体实践可以参考 [我每天都在用的 5 个 Claude Code 技巧](/zh/blog/5-claude-code-skills-i-use-every-single-day)。

**本地环境依赖强的项目。** 需要访问本地数据库、内网服务或特殊硬件的场景，云端沙箱无法满足。

## 适用场景：选 OpenAI Codex 代理

以下场景 Codex 更合适：

**批量独立任务。** 你有 20 个互不相关的 bug 需要修复，每个都是独立的小任务。Codex 的多任务并行在这种场景下效率极高——同时提交 20 个任务，每个在独立沙箱中运行，互不干扰。

**企业合规要求严格的环境。** 云端沙箱的隔离特性意味着代理无法访问非授权资源。对于需要审计 AI 行为的企业场景，Codex 的安全模型更容易通过合规审查。

**团队协作场景。** 多个开发者可以同时向 Codex 提交任务，共享同一个代码仓库但互不干扰。[Codex for Open Source](/zh/blog/codex-for-open-source) 的模式特别适合开源项目的贡献者管理。

**偏好 VS Code 工作流的开发者。** 通过 [Codex VS Code 扩展](/zh/blog/codex-vscode)，你可以在编辑器内直接启动和管理代理任务，不需要切换到终端。

## 结论

**如果你需要灵活的、交互式的多 Agent 编排，Claude Code 子代理是更强大的选择。** 它的多层配置系统、多角色子代理类型和本地执行模式，让你能精细控制 Agent 的行为，适合复杂的工程任务和需要实时协调的工作流。

**如果你需要简单可靠的并行任务执行，Codex 的云端沙箱模式更省心。** 隔离性好、配置简单、企业友好，适合批量独立任务和安全要求高的环境。

两者并不互斥——不少团队同时使用 Claude Code 处理复杂编排任务，用 Codex 处理批量小任务。关键是理解你的具体需求：编排深度还是执行简洁，本地灵活还是云端安全。更多关于 [Codex 的完整能力](/zh/blog/codex-complete-guide)和 [多 Agent 工作流设计](/zh/blog/con-u-pour-des-workflows-multi-agents)的分析，请参考我们的深度文章。

## 常见问题

### Claude Code 子代理和 Codex 任务能同时使用吗？

可以。两者运行在不同环境中——Claude Code 在本地终端，Codex 在云端沙箱。你可以在 Claude Code 中处理需要深度上下文的编排任务，同时在 Codex 中提交独立的批量任务，两者互不干扰。

### Codex 的 AGENTS.md 和 Claude Code 的 CLAUDE.md 有什么区别？

功能类似但深度不同。AGENTS.md 提供项目级的自然语言指令，定义代理的基本行为规范。CLAUDE.md 是更完整的配置系统，配合 SKILL.md、Hooks 和 MCP 形成四层扩展栈，支持更精细的行为控制和自动化逻辑。

### 哪个方案对企业团队更友好？

取决于团队规模和安全要求。Codex 的云端沙箱天然适合需要审计和隔离的企业环境，订阅制定价也更容易预算。Claude Code 子代理的 token 计费模式在大量使用时成本更灵活，但需要团队统一配置本地环境和权限管理。
