**Claude Code's orchestration model** operates like a team standup happening in real-time. The parent agent maintains a high-level plan, delegates specific research or implementation subtasks to child agents, and integrates their findings into a coherent whole. When you ask Claude Code to refactor a module, it might spawn an Explore agent to map all callers, a Plan agent to design the new interface, and multiple general-purpose agents to implement changes across different files — all coordinated through a single session.

This coordination means Claude Code subagents can handle tasks with dependencies. Agent A's findings inform Agent B's work, which constrains Agent C's implementation. The parent agent manages this dependency graph implicitly, launching agents in the right order and passing context between them. For complex refactoring that touches shared interfaces, this coordination is essential.

As documented in our [Claude Code subagents examples guide](/blog/claude-code-subagents-examples), practical patterns include launching research agents in parallel to gather information, then sequentially executing implementation based on combined findings. The key insight is that subagents are not autonomous — they're extensions of the parent agent's plan.

**Codex's execution model** is closer to a task queue. You submit work items, each runs independently in its own environment, and you review the results. There's no parent agent coordinating the work — you are the orchestrator. Each custom agent configuration defines what a task looks like (instructions, environment, tools), but doesn't define how multiple tasks relate to each other.

This makes Codex excellent for embarrassingly parallel work: "fix this bug in repo A, add this feature to repo B, write tests for module C." Each task is self-contained, and you review and merge results as they complete. But for work that requires cross-cutting coordination — renaming a type that appears in 50 files with complex import relationships — the lack of inter-task communication becomes a limitation.

Codex compensates with strong safety guarantees. Since each task is sandboxed, you can fire off speculative work without risk. If an agent goes off-track, you simply discard the diff. With Claude Code subagents, a poorly-instructed agent can make changes to your working directory that you then need to undo.

## Customization and Configuration: Detailed Analysis

Both systems allow you to define custom agent behaviors, but the mechanisms and depth differ substantially.

**Claude Code custom agents** live as definition files in your repository's `.claude/agents/` directory. Each agent definition specifies a name, description, available tools, and behavioral instructions. These definitions travel with your codebase — when a teammate clones the repo, they get the same custom agents. Combined with [SKILL.md files](/blog/claude-code-subagents-examples) that encode task-specific instructions, teams can build sophisticated multi-agent workflows that enforce their engineering standards.

The customization extends to agent types. Beyond the built-in types (Explore, Plan, general-purpose), you can define agents with specific tool restrictions, model overrides, and isolation settings. An agent configured with `isolation: "worktree"` runs in a separate git worktree — getting filesystem isolation without leaving the local machine. This gives you a middle ground between Claude Code's default shared-state model and Codex's full container isolation.

Custom agents in Claude Code also integrate with the broader [extension stack](/blog/claude-code-extension-stack-skills-hooks-agents-mcp) — hooks can trigger before or after agent actions, MCP servers provide external data sources, and skills define quality standards. The composability is a strength for teams that invest in configuration, but adds complexity for simpler use cases.

**Codex custom agents** are configured through the Codex interface rather than repository files. You define an agent with a name, system instructions, and optional environment setup commands. The configuration is simpler — closer to a prompt template than a full agent definition. You might create a "Test Writer" custom agent that always sets up the test framework and follows your testing conventions, or a "Code Reviewer" that applies your team's style guide.

The simplicity has advantages. Codex custom agents are accessible to developers who don't want to learn a configuration DSL. You write instructions in natural language, specify any setup commands the container should run, and you're done. There's no file to commit, no schema to follow, no integration points to wire up.

However, Codex custom agents don't compose the way Claude Code's do. You can't define an agent that spawns sub-agents, or create a workflow where one agent's output feeds into another. Each custom agent is a standalone task executor. For workflows that need chaining — "analyze the codebase, then plan the migration, then implement it" — you run sequential Codex tasks manually, carrying context between them yourself.

## Developer Workflow: Practical Patterns

Understanding how each system fits into daily development work matters more than architecture diagrams. Here's how real usage differs.

**With Claude Code subagents**, your workflow stays in the terminal. You describe a high-level task, and the agent decides when to delegate. You might say "refactor the authentication module to use the new session store" and watch as Claude Code spawns an Explore agent to map the auth module's surface area, uses the results to plan the migration, then implements changes across multiple files — all without you switching context.

For teams, the custom agent definitions become part of your engineering culture. A [senior engineer's workflow](/blog/claude-code-is-not-a-coding-tool) might include a "PR Prep" custom agent that runs linting, checks test coverage, validates documentation, and drafts a PR description — encoding a multi-step checklist as a single invocable agent.

**With Codex custom agents**, the workflow is more async. You open the Codex interface, select your custom agent (or the default), describe the task, and submit. Codex works in the background — you can close the tab, work on something else, and come back when the task completes. This async model works well for developers juggling multiple projects or handling a backlog of independent tasks.

The [Codex VS Code extension](/blog/codex-vscode) bridges the gap slightly by bringing task submission into your editor, but the fundamental model remains async: submit task, wait for result, review diff.

## Isolation and Safety Models

Safety guarantees affect how confidently you can delegate to agents, which directly impacts productivity.

**Claude Code subagents** operate in your local environment by default. They can read and write files, execute shell commands, and modify git state — subject to the permission system. The worktree isolation option provides git-level separation (changes happen on a temporary branch in a linked working tree), but the agent still has full system access within that worktree.

This means trust is essential. You need confidence that your agent definitions and instructions produce safe behavior. The [hooks system](/blog/claude-code-hooks-a-complete-guide-to-automating-your-ai-coding-workflow) adds a safety layer — you can define pre-execution hooks that validate commands before they run, or post-execution hooks that check results. But the responsibility for safety configuration falls on you.

**Codex custom agents** run in throwaway containers that are destroyed after task completion. The agent cannot access your local filesystem, cannot reach internal network resources (unless explicitly configured), and cannot persist state between tasks. This is a strict security boundary — even a completely misbehaving agent cannot damage anything beyond its container.

For enterprise teams with strict security requirements, Codex's isolation model is often the deciding factor. No amount of configuration can make a local agent as safe as a fully sandboxed one. But the isolation comes with costs: the agent can't access your local dev environment's state, can't run integration tests against local services, and can't interact with locally-running databases.

## When to Choose Claude Code Subagents

Claude Code's subagent system excels when your work requires **coordination, context sharing, and iterative refinement** within a single session.

**Best scenarios:**

- **Complex refactoring** spanning multiple interconnected modules where changes must be consistent
- **Architecture exploration** where research findings directly inform implementation decisions
- **Code review workflows** that need to check multiple aspects (security, performance, style) and synthesize a unified report
- **Iterative development** where you're building something incrementally and each step depends on the previous result
- **Team standardization** where custom agent definitions encode your engineering practices into the repository

Claude Code subagents are ideal for developers who prefer staying in the terminal, want real-time visibility into agent work, and need the agents to build on each other's outputs. If your tasks have dependencies, Claude Code's orchestration handles them natively.

The [agentic coding](/glossary/agentic-coding) paradigm that Claude Code embodies works best when you treat the agent system as a collaborative team rather than a task queue. You maintain awareness of what agents are doing and can redirect mid-execution.

## When to Choose Codex Custom Agents

Codex custom agents excel when your work is **parallelizable, self-contained, and benefits from strong isolation guarantees**.

**Best scenarios:**

- **Batch processing** independent tasks across multiple repositories or modules
- **Speculative exploration** where you want to try multiple approaches simultaneously without risk
- **Security-sensitive environments** where agents must not have local system access
- **Async workflows** where you submit tasks and review results later
- **Simple customization needs** where you want template-level configuration without repository-level infrastructure
- **Team onboarding** where less technical team members need to use agents without understanding the underlying system

Codex custom agents are ideal for teams that want [cloud-based AI coding](/blog/codex-complete-guide) without managing local agent infrastructure. The fire-and-forget model works well for teams with large backlogs of independent tasks — bug fixes, test additions, documentation updates — that don't depend on each other.

If your primary concern is safety and you're willing to trade coordination capabilities for isolation guarantees, Codex's containerized model provides stronger boundaries than any local agent system can offer.

## Verdict

The choice between Claude Code subagents and Codex custom agents isn't about which is "better" — it's about matching the tool to your workflow pattern.

**Choose Claude Code subagents** if you need agents that collaborate, share context, and build on each other's work in real-time. The orchestration model handles complex, dependency-rich tasks that would require multiple manual round-trips with Codex. Teams that invest in custom agent definitions and [skills](/blog/5-claude-code-skills-i-use-every-single-day) get compounding returns as their agent configurations encode more of their engineering practices.

**Choose Codex custom agents** if you need safe, parallel, async task execution with minimal setup. The containerized model is unbeatable for isolation, and the simpler configuration UX means faster adoption for teams. For independent tasks that don't need cross-agent coordination, Codex's approach has less overhead.

**For many teams, the answer is both.** Use Claude Code subagents for interactive development sessions where coordination matters, and Codex custom agents for batch processing independent tasks in the background. The tools serve different points on the autonomy-vs-control spectrum and complement rather than replace each other.

## Frequently Asked Questions

### Can Codex custom agents communicate with each other during execution?

No. Each Codex task runs in an isolated container with no mechanism for inter-task communication. If your workflow requires coordination between agents, you must orchestrate manually by reviewing one task's output before submitting the next, or use Claude Code's subagent system which supports real-time inter-agent messaging.

### Do Claude Code subagents require additional cost beyond the base Claude Code usage?

Claude Code subagents consume tokens from the same billing account as the parent agent. Each subagent uses its own context window, so spawning multiple agents increases total token usage proportionally. There is no separate per-agent fee — the cost scales with total tokens processed across all agents in a session.

### Can I define custom agent types in Claude Code that persist across projects?

Yes. Custom agent definitions can be placed in your user-level `.claude/agents/` directory to make them available across all projects, or in a project's `.claude/agents/` directory to scope them to that specific codebase. Project-level definitions are committed to version control and shared with your team automatically.

### Is Codex's container environment customizable beyond instructions?

Codex containers start with a standard Linux environment and clone your repository. You can specify setup commands that run before the agent starts working — installing dependencies, configuring tools, setting environment variables. However, you cannot customize the base container image or install system-level packages that require root access beyond what the default environment provides.

### Which approach handles large monorepos better?

Claude Code subagents have an advantage in monorepos because they share filesystem state and can coordinate changes across packages. A parent agent can spawn subagents for different packages while maintaining consistency. Codex tasks each clone the full repo independently, which means changes across packages require separate tasks that you manually ensure are compatible — a more error-prone process for tightly coupled monorepo packages.
