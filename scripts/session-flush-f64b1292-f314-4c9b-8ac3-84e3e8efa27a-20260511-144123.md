**Claude Code operates on an interactive orchestration model.** When you ask Claude Code to spawn subagents, they run as child processes within your terminal session. The parent agent maintains awareness of what each subagent is doing, can route information between them, and synthesizes their outputs before taking action. You see the work happening and can redirect it mid-stream.

This architecture excels when tasks are interdependent. Consider refactoring a shared utility module: one subagent needs to understand all call sites, another needs to design the new API, and a third needs to update tests. These aren't independent — the test updates depend on the new API design, which depends on understanding current usage. Claude Code's parent-child model handles this naturally because the parent orchestrates the sequence and passes context between agents.

The practical implication: **Codex subagents scale horizontally** (more independent tasks in parallel), while **Claude Code subagents scale vertically** (deeper coordination on complex tasks).

## Custom Agent Configuration: How Each Platform Lets You Specialize

Both platforms let you create specialized agents, but the mechanisms differ significantly.

### Codex Custom Agents

In Codex, a custom agent is primarily a set of system instructions paired with environment configuration. You define what the agent should do, what tools it can access, and how its sandbox should be set up. These configurations persist — you can reuse the same custom agent across multiple tasks and projects.

Codex custom agents are well-suited for repeatable workflows: a "test writer" agent that always follows your testing conventions, a "migration" agent that knows your framework's patterns, or a "security reviewer" agent that checks for specific vulnerability classes. Because each runs in an isolated sandbox, there's no risk of one agent's actions interfering with another's.

The limitation is flexibility. Codex custom agents are defined upfront and execute in isolation. They can't dynamically decide to spawn additional sub-tasks or adjust their approach based on what another agent found. Each agent runs its instructions to completion within its sandbox.

### Claude Code Custom Agents

Claude Code takes a more compositional approach to custom agents. The [Agent SDK](/glossary/agent-sdk) and built-in subagent system offer several mechanisms for specialization.

**Built-in subagent types** cover common workflows. The **Explore** agent is optimized for fast read-only codebase search — it can find files by pattern, grep for symbols, and answer "where is X defined" questions without needing full code review context. The **Plan** agent designs implementation strategies, identifies critical files, and considers architectural tradeoffs. These specialized types have different tool access and behavioral constraints tuned to their purpose.

**SKILL.md files** let you define domain-specific agent behaviors that travel with your repository. A [SKILL.md for code review](/blog/claude-code-subagents-examples) might specify which files to check, what patterns to flag, and how to format findings. When Claude Code spawns a subagent with a specific skill, that agent inherits the skill's instructions alongside its general capabilities.

**Dynamic orchestration** is the biggest differentiator. The parent Claude Code session can decide at runtime how many subagents to spawn, what type each should be, and how to combine their results. If an Explore agent finds unexpected complexity in a module, the parent can spawn an additional Plan agent to design the refactoring approach before proceeding. This adaptive behavior is impossible in Codex's pre-defined task model.

For teams building [reusable skill libraries](/blog/5-claude-code-skills-i-use-every-single-day), Claude Code's SKILL.md system integrates custom agent definitions directly into version control alongside the code they operate on. This means agent behaviors evolve with the codebase — when conventions change, the skills update in the same PR.

## Practical Workflows: Side-by-Side Scenarios

### Scenario 1: Bug Triage Across a Large Codebase

**With Codex:** Create a custom "bug investigator" agent. For each bug report, dispatch a separate Codex task with the bug description and relevant context. Each task runs in parallel, investigates the bug in its own sandbox, and returns findings or a proposed fix. Review all results in the Codex dashboard. Best when bugs are independent and well-described.

**With Claude Code:** Ask the parent agent to investigate. It spawns Explore subagents in parallel to search for relevant code patterns, then synthesizes their findings to identify root causes. If one bug turns out to be related to another, the parent can redirect investigation in real time. Best when bugs might be interconnected or need human judgment during triage.

### Scenario 2: Multi-File Feature Implementation

**With Codex:** Break the feature into independent components. Dispatch each component as a separate task to a custom agent configured with your architecture guidelines. Review and merge the results, resolving any integration issues manually. Works well when components have clean boundaries.

**With Claude Code:** Describe the full feature to the parent agent. It spawns a Plan subagent to design the architecture, then spawns implementation subagents for each component — but coordinates shared interfaces between them. The parent reviews all changes holistically before committing. Works well when components need to agree on shared types, APIs, or database schema.

### Scenario 3: Test Generation at Scale

**With Codex:** Create a "test writer" custom agent with your testing conventions. Dispatch one task per module — each generates tests independently. All tasks run in parallel in the cloud. Collect results and run the full test suite to check for conflicts. Highly efficient for large codebases with modular structure.

**With Claude Code:** Point the parent agent at the modules needing tests. It spawns subagents in parallel (one per module), each generating tests. The parent then runs the full test suite locally, identifies failures or conflicts, and iterates. Slower for large batches but catches integration issues immediately.

## Environment and Isolation

Codex provides strong isolation by default — each task runs in its own container with a fresh repository clone. This eliminates the possibility of agents interfering with each other's work, corrupting local state, or introducing race conditions on shared files. For teams concerned about safety and reproducibility, this is a significant advantage.

Claude Code subagents share the local filesystem by default. This enables rich coordination — agents can see each other's changes and build on them — but introduces the risk of conflicts when multiple agents edit the same files simultaneously. Claude Code mitigates this with **git worktree isolation**, where subagents operate on separate git worktrees that are merged back after completion. However, this is opt-in, not the default behavior.

For enterprise environments with strict security requirements, Codex's cloud sandbox model provides a clearer security boundary. Each agent operates within a defined container, and you can control exactly what tools and network access it has. Claude Code's local execution model means agents have the same filesystem access as the user running them — powerful but requiring trust in the agent's behavior.

## Scaling and Cost Considerations

**Codex scales with cloud resources.** Dispatching 50 parallel tasks doesn't tax your local machine — each runs on cloud infrastructure. This makes Codex the better choice for high-volume batch operations: large-scale migrations, codebase-wide refactoring, or generating content across many files. The constraint is cost — each cloud sandbox consumes compute resources, and heavy parallel usage adds up quickly on usage-based plans.

**Claude Code scales with local resources.** Spawning many subagents means more concurrent API calls and local process overhead. On a typical development machine, 3-5 parallel subagents is practical; beyond that, you'll hit context window limits and local resource constraints. The advantage is that you're only paying for API tokens, not cloud sandbox time — for focused, high-coordination tasks, this is often more cost-effective.

For teams evaluating cost, the calculation depends on workflow shape. If you regularly need 10+ parallel independent tasks, Codex's cloud model will likely be more efficient. If you need 2-4 coordinated agents working on a complex problem, Claude Code's local model avoids the overhead of cloud sandbox provisioning.

## When to Choose Codex Subagents

Choose Codex custom agents and subagents when:

- **Tasks are independent and well-defined.** Each task has clear inputs and expected outputs with minimal dependencies on other tasks. Bug fixes, test generation, and file-by-file migrations fit this pattern.
- **You need high parallelism.** Dozens of tasks running simultaneously without local resource constraints. Codex's cloud sandboxes handle this natively.
- **Isolation matters.** You want strong guarantees that agents can't interfere with each other or corrupt shared state. Each Codex sandbox is a clean environment.
- **Async fits your workflow.** You're comfortable dispatching tasks and reviewing results later, rather than watching execution in real time. This works well for teams that batch work overnight or across time zones.
- **You're already in the OpenAI ecosystem.** If your team uses ChatGPT Pro, Team, or Enterprise, Codex integrates directly into your existing workflow with minimal additional setup.

## When to Choose Claude Code Subagents

Choose Claude Code agent teams when:

- **Tasks require coordination.** The work involves shared interfaces, cross-cutting concerns, or decisions that depend on findings from multiple investigation threads. Claude Code's parent-child orchestration handles this naturally.
- **You want real-time oversight.** You need to see what agents are doing, intervene when they go off track, and provide additional context mid-execution. Claude Code's interactive model keeps you in the loop.
- **Custom agent behaviors should live in version control.** SKILL.md files travel with your repo, meaning agent configurations evolve alongside the code. This is critical for teams with [well-defined engineering standards](/blog/9-principles-writing-claude-code-skills) that change over time.
- **You're working on a single complex problem.** Refactoring a tightly coupled module, debugging a subtle integration issue, or designing a new architecture — tasks where depth of analysis matters more than breadth of parallelism.
- **Local execution is preferred.** Your codebase can't be sent to cloud sandboxes (security policy, air-gapped environments, proprietary code restrictions).

## Verdict

**For batch-style, independent task delegation, Codex subagents win.** The cloud sandbox model handles high parallelism cleanly, and custom agent configurations make repeatable workflows easy to define. If your multi-agent needs look like "run this 20 times on different inputs," Codex is the more natural fit.

**For coordinated, context-rich multi-agent workflows, Claude Code agent teams win.** The parent-child orchestration model, specialized subagent types, and SKILL.md integration create a more powerful system for complex engineering tasks where agents need to share context and build on each other's work.

Many teams will benefit from using both. Use Codex for high-volume batch operations and overnight task queues. Use Claude Code for interactive sessions where you're actively working through a hard problem with AI assistance. The tools complement rather than compete — they optimize for different shapes of work. ## Frequently Asked Questions

### Can Codex subagents communicate with each other during execution?

No. Each Codex task runs in an isolated cloud sandbox with no mechanism for inter-agent communication. If you need agents to share intermediate results, you must collect outputs from completed tasks and dispatch follow-up tasks with that context. This is by design — isolation guarantees reproducibility and prevents interference.

### How many Claude Code subagents can run in parallel?

Claude Code can spawn multiple subagents in a single message, and they execute concurrently. Practical limits depend on your local machine resources and API rate limits rather than a hard cap. For most development machines, 3-5 parallel subagents is practical. Using the `run_in_background` parameter lets the parent continue working while subagents complete independently.

### Do I need separate pricing plans for subagents on either platform?

Codex subagent usage is included in your ChatGPT Pro, Team, or Enterprise plan, though heavy usage consumes your allocated compute budget faster. Claude Code subagents consume API tokens from your Anthropic account — each subagent call is billed the same as any other Claude API usage. Neither platform charges a separate fee specifically for multi-agent functionality.

### Can I use custom agents in Codex with private repositories?

Yes. Codex integrates with GitHub and can access private repositories you've authorized. Each cloud sandbox clones the repository at task creation time, so it works with the latest state of whatever branch you specify. Repository access follows your existing GitHub permissions — Codex doesn't require additional access beyond what you grant through the GitHub integration.

### Which approach is better for enterprise teams?

It depends on the constraint. Codex's cloud sandbox model offers stronger isolation guarantees and scales without taxing developer machines — advantages for large teams with diverse, parallelizable workloads. Claude Code's local execution model avoids sending code to cloud sandboxes and keeps agent behavior versioned in the repo via SKILL.md — advantages for teams with strict data residency requirements or complex coordination needs. Evaluate based on your security policy, workflow patterns, and existing tool ecosystem.
