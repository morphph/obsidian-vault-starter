**CLAUDE.md enables team-wide AI consistency.** When your team checks a CLAUDE.md file into the repository, every developer's Claude Code session follows the same rules. Code review standards, naming conventions, test requirements, forbidden patterns — these become enforceable across every AI-assisted commit. A new team member who clones the repo immediately gets Claude Code behavior calibrated to your team's standards. This is particularly valuable for [agentic coding](/glossary/agentic-coding) workflows where Claude Code operates with significant autonomy.

Consider a concrete example: your team decides that all database queries must use parameterized statements, never string interpolation. Put this in CLAUDE.md, and every Claude Code session on your team follows it. Put it in one developer's Memory, and only their sessions follow it — everyone else's Claude Code might still generate vulnerable queries.

**Memory captures individual expertise.** A senior backend engineer and a junior frontend developer interact with the same codebase differently. The senior engineer might correct Claude Code's approach to connection pooling; the frontend developer might teach it about the team's component naming conventions. These corrections live in their respective Memory systems, gradually optimizing Claude Code's behavior for each person's workflow.

Memory also handles role-specific context. A data scientist investigating logging has different needs than a platform engineer refactoring the auth layer. Memory adapts Claude Code's explanations, suggestions, and assumptions to each individual without requiring changes to the shared CLAUDE.md.

The tradeoff is clear: **CLAUDE.md for what the team agrees on, Memory for what the individual needs.** Teams that put individual preferences in CLAUDE.md end up with bloated, contradictory instruction files. Developers who put team rules in Memory end up with inconsistent AI behavior across the team.

## What Belongs Where: A Decision Framework

The most common mistake developers make with Claude Code's context systems is putting information in the wrong layer. Here's a concrete framework for deciding where something belongs.

### Put it in CLAUDE.md if:

- **It's a rule the whole team must follow.** Build commands, test requirements, lint configurations, deployment procedures. Example: "Run `npm test` before every commit."
- **It defines project architecture.** Tech stack, directory structure, module boundaries, API conventions. Example: "This is a Next.js 16 app with SQLite. Frontend in `src/`, pipeline scripts in `scripts/`."
- **It prevents known mistakes.** Gotchas, footguns, and constraints that apply regardless of who's coding. Example: "Never import Next.js modules in pipeline scripts — they're server-only."
- **It should be reviewable in PRs.** If changing this instruction warrants team discussion, it belongs in version control.
- **It must load every session.** Critical instructions that cannot be missed.

### Put it in Memory if:

- **It's about you, not the project.** Your role, expertise level, communication preferences. Example: "User is a senior Go engineer, new to React."
- **It's a correction to Claude Code's behavior.** When you teach Claude Code to do something differently, Memory captures that feedback for future sessions. Example: "Don't mock the database in integration tests."
- **It's about ongoing work with a shelf life.** Sprint goals, incident context, migration timelines. Example: "Merge freeze begins March 5 for mobile release."
- **It points to external resources.** Slack channels, Linear projects, Grafana dashboards. Example: "Pipeline bugs are tracked in Linear project INGEST."
- **It would clutter CLAUDE.md for other developers.** If it's useful to you but irrelevant or confusing to teammates.

### Common misplacements and their consequences

**Putting personal preferences in CLAUDE.md**: You like terse responses with no trailing summaries. Adding "Don't summarize what you just did" to CLAUDE.md imposes your communication style on the whole team. A new developer who benefits from Claude Code's explanations loses that help. Put this in Memory as a feedback correction instead.

**Putting build rules in Memory**: "Always run the validation pipeline before committing" is a rule Memory might not surface in every session. If Claude Code skips it during a refactoring task because Memory didn't judge it relevant, you've shipped unvalidated code. CLAUDE.md guarantees this instruction loads.

**Putting project status in CLAUDE.md**: "We're migrating from PostgreSQL to SQLite — don't touch the old Postgres code" is temporal context that will become stale. In CLAUDE.md, it lingers after the migration completes, confusing future sessions. Memory handles temporal context better because it can be updated or removed as projects evolve.

## Content Lifecycle and Maintenance

Both systems require different maintenance strategies, and understanding their lifecycles prevents context rot.

**CLAUDE.md maintenance is like documentation maintenance.** It should be updated when the codebase changes — new build steps, new conventions, new architectural decisions. Stale CLAUDE.md instructions actively mislead Claude Code. If your CLAUDE.md still references a build command you changed three months ago, Claude Code will fail on every build attempt. The remedy is treating CLAUDE.md updates as part of your normal PR process: when you change a convention, update the CLAUDE.md in the same commit.

**Memory maintenance is largely automatic but requires periodic review.** Claude Code creates and updates memories based on conversations, but it can't detect when a memory becomes outdated because the codebase changed independently. A memory that says "the auth module uses JWT tokens" becomes misleading after you migrate to session-based auth. Periodically reviewing your memory files — especially project-type memories — prevents Claude Code from acting on stale information.

Memory files include frontmatter with type and description fields, making them searchable and categorizable. When a memory conflicts with current reality, Claude Code is designed to trust what it observes in the code over what memory says — but the conflict still costs a reasoning step. Clean memories mean faster, more accurate sessions.

**Practical maintenance schedule:**
- **CLAUDE.md**: Update in the same PR as the change it describes. Review quarterly for drift.
- **Memory**: Review project and reference memories monthly. User and feedback memories rarely go stale.

## Scaling to Larger Teams and Projects

The dynamics shift as team size and codebase complexity grow, and both systems have different scaling characteristics.

**CLAUDE.md scales with team discipline.** A 3-person startup can maintain a single CLAUDE.md easily. A 50-person engineering org with multiple services needs a more structured approach. Claude Code supports CLAUDE.md files at multiple directory levels — a root-level file for organization-wide standards and subdirectory files for service-specific instructions. This mirrors how teams already structure their documentation.

The risk at scale is CLAUDE.md bloat. When everyone adds their team's conventions, the file becomes a wall of text that Claude Code must process every session. The recommended pattern is keeping the root CLAUDE.md focused on universal rules (build, test, deploy, architectural invariants) and pushing team-specific guidance into subdirectory CLAUDE.md files or SKILL.md files. Our guide on [how to effectively prompt Claude Code](/blog/how-to-effectively-prompt-a-claude-code) covers structuring instructions for maximum clarity.

**Memory scales naturally per-individual but doesn't compound across the team.** If one developer discovers an important pattern and it lives only in their Memory, the insight stays siloed. Teams that rely heavily on Memory for critical context create invisible knowledge gaps. The solution is promotion: when a memory proves valuable, promote the underlying rule to CLAUDE.md so the whole team benefits.

This promotion workflow — discover in Memory, validate through use, promote to CLAUDE.md — is the most effective pattern for teams evolving their AI coding practices. It lets individuals experiment with corrections and preferences without immediately committing the team, then graduates proven insights to shared configuration.

## When to Choose Claude Memory

Choose Claude Memory as your primary context layer when:

- **You're a solo developer.** Without team coordination needs, Memory's automatic capture is more convenient than maintaining CLAUDE.md manually. Your corrections and preferences accumulate naturally.
- **You're onboarding to a new codebase.** Memory captures your questions, Claude Code's corrections, and the contextual understanding you build over time — creating a personal knowledge base.
- **Your workflow involves many external systems.** Memory's reference type is designed for pointers to Slack channels, Linear projects, monitoring dashboards, and other external resources that don't belong in a repo file.
- **You need Claude Code to adapt to your communication style.** Feedback memories shape how Claude Code responds to you — terse or detailed, code-first or explanation-first.

Memory's strength is low-friction context building. You don't have to decide upfront what's important — [Claude Code learns from your interactions](/blog/whats-so-special-about-the-claude-code) and retains what matters.

## When to Choose CLAUDE.md

Choose CLAUDE.md as your primary context layer when:

- **You're working on a team.** Any instruction that affects code quality, conventions, or architecture must be in CLAUDE.md to ensure consistency across all developers.
- **You need deterministic, reproducible behavior.** CLAUDE.md loads completely every session. There's no relevance filter, no selective loading. Critical rules always apply.
- **You want version-controlled AI behavior.** CLAUDE.md changes show up in git history. You can review, revert, and discuss instruction changes the same way you discuss code changes.
- **You're setting up a new project.** Start with CLAUDE.md to establish the foundation: stack, build commands, conventions, known pitfalls. Memory can't help until you've had conversations.
- **You need to enforce quality gates.** "All tests must pass before commit" is a CLAUDE.md rule, not a Memory suggestion. Our coverage of [5 Claude Code skills used daily](/blog/5-claude-code-skills-i-use-every-single-day) shows how CLAUDE.md and skills work together to enforce standards.

## Verdict

**Use both, and use them for what they're designed for.** CLAUDE.md is your project's constitution — shared, explicit, version-controlled rules that every Claude Code session must follow. Claude Memory is your personal assistant's notebook — adaptive, individual, and accumulated through real interactions. The strongest Claude Code setup combines a well-maintained CLAUDE.md with active Memory, using the promotion workflow to graduate proven insights from personal to shared context.

If you're forced to choose just one: **start with CLAUDE.md**. A project without Memory still works well — Claude Code just doesn't remember your preferences across sessions. A project without CLAUDE.md means Claude Code has no project context at all and must rediscover your stack, conventions, and constraints every time. CLAUDE.md provides the foundation; Memory adds personalization on top.

## Frequently Asked Questions

### Can Claude Memory and CLAUDE.md conflict with each other?

Yes, and when they do, CLAUDE.md takes priority because it loads deterministically at session start. If your Memory says "use tabs for indentation" but CLAUDE.md specifies spaces, Claude Code follows CLAUDE.md. To avoid confusion, keep explicit rules in CLAUDE.md and use Memory for context that doesn't contradict shared instructions.

### Does Claude Memory sync across devices?

No. Claude Memory is stored locally on your machine in `~/.claude/projects/`. If you switch to a new laptop, your memories don't transfer automatically. CLAUDE.md, being a repository file, travels with your code via git — clone the repo on any machine and the instructions are there.

### How do I migrate useful Memory entries to CLAUDE.md?

Review your memory files in `~/.claude/projects/` periodically. When a feedback or project memory contains a rule that would benefit the whole team — like "never mock the database in integration tests" — add that rule to your project's CLAUDE.md and submit it as a PR. Then remove or update the memory to avoid duplication.

### Is there a size limit for CLAUDE.md?

There's no hard character limit, but practical limits matter. CLAUDE.md loads fully into Claude Code's context window every session, consuming tokens that could be used for your actual task. Keep it focused — typically under 500 lines. If it grows beyond that, split service-specific instructions into subdirectory CLAUDE.md files or dedicated SKILL.md files.

### Do I need CLAUDE.md if I'm the only developer?

It's still recommended. CLAUDE.md provides deterministic context that loads every session without relying on relevance matching. Even solo developers benefit from having build commands, architectural notes, and known gotchas available in every Claude Code session. Memory complements this but doesn't replace the guarantee of always-loaded instructions.
