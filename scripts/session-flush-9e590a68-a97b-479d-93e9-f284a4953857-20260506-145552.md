# Codex Subagents vs Custom Agents：多智能体工作流的两条路径

**TL;DR：** **Subagents** 是 Codex 内置的任务分发机制，适合将大任务拆解为并行子任务、快速获得结果；**Custom Agents** 则允许你定义专属的 agent 配置——指定工具集、系统指令和执行边界，适合需要标准化、可复用的团队工作流。简单并行用 Subagents，复杂流程用 Custom Agents，实际项目中两者经常组合使用。

## 概述：Codex Subagents

Codex Subagents 是 OpenAI [Codex 平台](/zh/blog/codex-complete-guide)内置的子智能体机制。当一个 Codex 任务涉及多个独立的子目标时——比如同时重构三个模块、并行生成不同组件的测试——主 agent 可以自动或手动拆分出多个 subagent，每个 subagent 在独立的沙箱环境中执行，互不干扰。

Subagents 的核心价值是**并行加速**。传统的单 agent 顺序执行在面对大型代码库时效率低下，subagents 让你用时间换吞吐量。每个 subagent 继承主 agent 的代码库上下文，但拥有独立的执行环境和工具调用权限。任务完成后，结果汇总回主 agent 进行整合。

这种模式的局限在于：subagents 的行为由系统默认配置决定，你对单个 subagent 的指令控制粒度有限。如果你需要某个 subagent 使用特定的代码风格、跳过某些文件、或调用特定的外部工具，内置 subagent 机制可能不够灵活。

## 概述：Codex Custom Agents

Custom Agents 是 Codex 提供的自定义 agent 配置能力。你可以为特定类型的任务创建预设的 agent 模板，定义该 agent 的系统指令、可用工具集、执行约束和输出格式。

Custom Agents 的核心价值是**标准化和可复用性**。团队可以创建一个「代码审查 agent」、一个「测试生成 agent」、一个「文档更新 agent」，每个都有精确定义的行为边界。这对企业团队尤其重要——确保 AI 的行为符合团队规范，而非依赖每次手动输入提示词。

与 subagents 不同，custom agents 的配置是持久化的。你定义一次，团队里的任何人都可以调用。这类似于 Claude Code 生态中的 [SKILL.md](/zh/glossary/agentic-coding) 概念——将 AI 行为编码为可版本控制的配置文件。

## 功能对比

| 维度 | Subagents | Custom Agents | 胜出方 |
|------|-----------|---------------|--------|
| **启动方式** | 系统自动拆分或用户手动触发 | 预配置模板，按需调用 | 各有适用场景 |
| **配置灵活度** | 低——继承主 agent 默认配置 | 高——自定义指令、工具、约束 | Custom Agents |
| **并行执行** | 原生支持，多 subagent 同时运行 | 需要手动编排或配合 subagent 使用 | Subagents |
| **可复用性** | 临时性，用完即弃 | 持久化模板，团队共享 | Custom Agents |
| **上手成本** | 低——开箱即用 | 中——需要前期配置 | Subagents |
| **执行隔离** | 独立沙箱环境 | 独立沙箱环境 | 平局 |
| **结果整合** | 自动汇总回主 agent | 需手动或通过编排层整合 | Subagents |
| **团队协作** | 个人使用为主 | 团队级标准化 | Custom Agents |

## 并行执行能力：深度分析

Subagents 在并行执行方面具有原生优势。当你向 Codex 提交一个涉及多模块的任务时，系统会分析任务依赖图，将无依赖关系的子任务分配给不同的 subagent 并行处理。这种机制在[多智能体工作流](/zh/blog/con-u-pour-des-workflows-multi-agents)场景下尤为关键。

一个典型的使用场景：你需要为一个包含 8 个独立服务模块的 monorepo 添加单元测试。单 agent 顺序执行可能需要 30 分钟以上，而 subagents 并行处理可以将时间压缩到单个模块的执行时长——大约 5-8 分钟。每个 subagent 获取完整的代码库快照，但只操作自己负责的模块目录。

Custom Agents 本身不直接提供并行能力，但可以通过编排层——比如在 CI/CD 中同时启动多个 custom agent 任务——实现类似效果。这种方式的优势在于每个并行任务都遵循预定义的行为规范，结果一致性更高。

需要注意的是，subagents 的并行执行受限于账户的并发配额。[Codex 平台](/zh/blog/codex-complete-guide)对不同层级的用户设有不同的并发上限，超出配额的 subagent 会进入队列等待。

## 配置与标准化：深度分析

Custom Agents 的真正威力在于将 AI 行为「工程化」。以代码审查场景为例：你可以创建一个 custom agent，指定它只关注安全漏洞和性能问题，忽略代码风格（因为团队已有 linter），输出格式固定为 JSON 以便集成到 CI 流水线。这种精确控制是 subagents 做不到的。

对于已经在使用 [Agent SDK](/zh/glossary/agent-sdk) 构建工具链的团队，custom agents 的配置模式会非常熟悉。核心概念一致——定义 agent 的能力边界、可用工具和行为约束。区别在于 Codex custom agents 运行在云端沙箱中，不需要你自建执行环境。

Custom Agents 的配置通常包含以下要素：
- **系统指令**：定义 agent 的角色、输出规范和行为边界
- **工具白名单**：限制 agent 可以调用的工具（shell 命令、文件操作、网络请求等）
- **文件作用域**：指定 agent 可以读写的目录范围
- **输出模板**：定义结果的格式化方式

这种配置化的方式与 Claude Code 的 [Skills 系统](/zh/blog/9-principles-writing-claude-code-skills)有异曲同工之处——都是将 AI 行为从随机的提示词驱动转变为可控、可审计的工程实践。

## 什么时候选 Subagents

以下场景优先使用 Subagents：

- **大规模并行任务**：需要同时对多个模块执行相同类型的操作（批量重构、批量测试生成、批量文档更新）
- **探索性任务**：不确定最优方案时，让多个 subagent 尝试不同策略，比较结果
- **一次性任务**：不需要复用的临时性工作，配置 custom agent 的成本不值得
- **快速原型**：需要快速验证一个想法，不关心执行过程的标准化

Subagents 的核心优势是**低摩擦**——你不需要任何前置配置，直接描述任务，系统自动分配。对于个人开发者或小型项目，这通常就足够了。

## 什么时候选 Custom Agents

以下场景优先使用 Custom Agents：

- **团队标准化**：需要确保所有成员的 AI 辅助工作遵循相同规范
- **流水线集成**：需要将 agent 任务嵌入 CI/CD 或其他自动化工作流，输出格式必须稳定
- **安全合规**：需要严格限制 agent 的工具访问权限和文件操作范围
- **高频重复任务**：每天都要执行的代码审查、测试、部署前检查等

Custom Agents 的前期投入更高，但长期收益显著。特别是在 [Codex 企业场景](/zh/blog/codex-complete-guide)中，custom agents 是实现 AI 工程化的关键工具。

## 两者组合使用

实际项目中，subagents 和 custom agents 并不是非此即彼的选择。最有效的模式是**用 custom agents 定义标准化的执行单元，用 subagents 编排并行执行**。

例如：你有一个「安全审计 custom agent」和一个「性能分析 custom agent」。当一个 PR 提交时，CI 系统同时启动这两个 custom agent 作为 subagent 并行运行，各自生成报告，最后由主 agent 汇总成一份综合审查意见。

这种组合模式类似于 [Claude Code 的多 Agent 协作架构](/zh/blog/claude-code-subagents-examples)——不同的 agent 承担不同的职责，通过编排层协调执行。

## 结论

**如果你只是需要加速单次任务的执行，Subagents 是更简单的选择**——零配置、自动分发、结果自动整合。**如果你的目标是建立可复用、可审计、团队共享的 AI 工作流，Custom Agents 才是正确的投资方向。** 大多数成熟团队最终会走向两者结合：custom agents 定义「做什么、怎么做」，subagents 负责「并行做、快速做」。关键不在于选哪个，而在于理解你当前最需要解决的是效率问题还是标准化问题。

## 常见问题

### Subagents 之间可以共享状态吗？

每个 subagent 运行在独立的沙箱中，不能直接共享运行时状态。它们共享的是代码库的初始快照——所有 subagent 从相同的代码状态开始执行，但各自的修改相互隔离，最终由主 agent 合并结果。如果两个 subagent 修改了同一个文件，需要在合并阶段处理冲突。

### Custom Agents 的配置可以版本控制吗？

可以，而且推荐这样做。Custom agent 的配置本质上是结构化的文本定义（通常是 JSON 或 YAML 格式），可以存储在代码仓库中，纳入正常的版本控制和代码审查流程。这也是 custom agents 相比手动提示词的一大优势——配置变更有迹可循。

### Subagents 和 Custom Agents 对 API 配额的消耗方式有区别吗？

每个 subagent 和每个 custom agent 实例都独立消耗 token 配额。并行运行 5 个 subagent 会同时产生 5 份 token 消耗。区别在于 custom agents 通常有更精确的工具和文件范围限制，可能会产生更少的无效 token 消耗，因为 agent 不会浪费时间探索不相关的代码路径。
