# Vibe Coding 教会我的一件事：你不需要先学会，才能开始做

---

你有没有过这种经历——

你用 AI 帮你写了一套自动化管线，每天自动跑，采集数据、生成内容、发布。

听起来很爽对吧？

但实际情况是这样的：

管线一共 5 个 stage，跑完大概要 20-30 分钟。这段时间你干嘛呢？你只能等。因为它是交互式的，跑到一半可能要你确认权限、approve tool call，你不能离开。你就坐在那儿，盯着终端，看它一行一行地跑。

好不容易跑完了。然后你打开 review report，发现——Newsletter 少了一个 section，博客字数超标，SEO 页面格式不对。

你心想：早知道第二步就出问题了，后面三步全白跑了。

但没办法，因为管线是从头到尾一条线，中间没有任何检查点。

然后你又花 30 分钟手动 debug，手动重跑出问题的那个 stage，再手动检查一遍。

**一个本该 20 分钟搞定的事，实际花了一个多小时，而且你全程都被钉在屏幕前。**

每天都是这样。

---

我之前就是这个状态。

我的管线长这样：

```
daily-pipeline.sh（盲跑）
  → collect-news.ts
  → write-newsletter.ts
  → write-blog.ts
  → generate-seo.ts
  → import-video-blog.ts
  → git commit + push
  → (完了之后你手动看 review report 才知道质量怎样)
```

没有中间校验。每个 stage 跑完就直接冲下一个，不管产出质量。

最痛的点不只是"质量不行"，而是：

1. **你必须全程在场** — 交互式运行，Claude 随时可能问你权限、要你 approve，你不能走开去做别的事
2. **错误发现得太晚** — 第二步就出了问题，但你要等五步全跑完才知道，后面的全是浪费
3. **重跑成本很高** — 发现问题后要重新开 session、定位问题、手动重跑，又是一轮 20 分钟
4. **时间碎片化** — 你本来只想花 20 分钟跑管线，结果一个小时过去了，中间什么正事都没法做

这不是自动化，这是**半自动化加全手动兜底**。

---

直到有一天，我在 vibe coding 的过程中，偶然发现了两个能力：

**Agent 子任务** 和 **Shell 校验脚本**。

说实话，这两个概念我之前完全不知道。我不是科班出身，没学过什么编排器、什么子进程管理。

但就是在用 Claude 帮我解决问题的过程中，它告诉我：

> "你不需要等全部跑完再检查。你可以让我在每个 stage 跑完之后，立刻跑一个校验脚本，通过了才继续，没通过就自动重试。"

我当时就觉得：**这不就是我一直想要的吗？**

---

改完之后的流程是这样的：

我对 Claude 说一句话："跑今天的管线。"

然后它自己编排：

```
Claude（主 Agent / 编排器）
│
├── Sub-Agent 1：跑 collect-news.ts
│   └── 跑 validate-pipeline.ts --step=collect
│   └── ✅ 45 items, 5 tiers → 通过，继续
│
├── Sub-Agent 2：跑 write-newsletter.ts
│   └── 跑 validate-pipeline.ts --step=newsletter
│   └── ❌ EN newsletter 缺少 MODEL LITERACY section
│   └── 自动重试（最多 2 次）
│   └── 重试后：✅ 通过，继续
│
├── Sub-Agent 3：跑 write-blog.ts
│   └── 跑 validate-pipeline.ts --step=blog
│   └── ✅ 2 posts, 通过
│
└── 最终汇报："今天管线全部通过。Newsletter 重试了 1 次。"
```

**我什么都没做。** 我可以去冲杯咖啡，去回几条消息，去做别的事情。它自己跑、自己检查、自己修、自己报告。等我回来，桌面上就一条消息告诉我结果。

如果全部通过 — 我连 review report 都不用看。如果有问题 — 它已经帮我重试过了，实在修不了才找我，而且精确告诉我哪个 stage、什么问题。

之前一个小时被钉在屏幕前，现在变成了一句指令 + 等通知。**我的时间从"被占用"变成了"被释放"。**

两种方式的区别，一张表说清楚：

```
┌──────────┬──────────────────────────┬──────────────────────────────┐
│          │          之前            │            之后              │
├──────────┼──────────────────────────┼──────────────────────────────┤
│ 编排器   │ shell 脚本（盲跑）      │ Claude Agent（有判断力）     │
├──────────┼──────────────────────────┼──────────────────────────────┤
│ 校验时机 │ 全部跑完后手动看        │ 每个 stage 跑完立即校验      │
├──────────┼──────────────────────────┼──────────────────────────────┤
│ 失败处理 │ 手动开 session 调试     │ Agent 自动重试，超限才找你   │
├──────────┼──────────────────────────┼──────────────────────────────┤
│ 校验脚本 │ 有，但没人调用          │ 同一个脚本，被自动调用       │
└──────────┴──────────────────────────┴──────────────────────────────┘
```

---

你知道这件事最让我触动的是什么吗？

不是效率提升——虽然确实快了很多。

而是：**我是通过 vibe coding，反过来学会了这些工程概念。**

什么是编排器？什么是子任务？什么是 stage gate？什么是校验脚本？

这些东西教科书上都有，但我是在解决真实问题的过程中，自然地学会了。

不是先学、再做。是先做、遇到痛点、AI 告诉我有更好的方式、然后我理解了背后的原理。

---

所以我想说的是：

Vibe coding 不只是"让 AI 帮你写代码"。

它是一种**新的学习路径**。

你不需要先读完一本《系统设计》才能写管线编排。你只需要有一个真实的问题，然后在解决问题的过程中，概念自然会出现。

**痛点驱动学习，AI 加速理解。**

---

如果你也在 vibe coding，也在用 AI 帮你搭各种自动化——

下次遇到"每次都要手动兜底"的情况，试试想一下：

这个手动步骤，能不能变成一个校验脚本？能不能让 Agent 自己判断通过还是失败？

你可能会像我一样，在解决问题的同时，顺便学会了一个你从没想过要学的概念。

这就是 vibe coding 最有意思的地方。
