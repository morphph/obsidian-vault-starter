# Source: Claude Code `/goal` + Agent View official docs (2026-05-13 fetch)

## Summary
Two new Claude Code features captured from official Anthropic documentation: **`/goal`** (session-scoped autonomous turn-loop with a Haiku evaluator) and **Agent View** (research-preview TUI for managing many parallel background sessions hosted by a per-user supervisor process). Together they form the orchestration + loop-driver pair: agent view is where many sessions live; `/goal` is what each session loops on.

## Origin
- `/goal` docs: https://code.claude.com/docs/en/goal
- Agent view docs: https://code.claude.com/docs/en/agent-view

## What's in here
- Full `/goal` mechanics: setting, status, clearing, resuming; the prompt-based Stop-hook implementation; evaluator transcript-only constraint; condition-writing guidance; comparison vs `/loop` / Stop hooks / auto mode.
- Full Agent View mechanics: opening, layout, status icons (color = state, shape = process), peek/attach/background flows, dispatch entry points (from view, `/bg`, `claude --bg`), filters, keyboard shortcuts, worktree isolation, supervisor-process hosting model, state persistence paths, shell management commands (`claude attach`/`stop`/`respawn`/`rm`/`logs`), per-session model and permission overrides.
- Research-preview limitations: rate limits apply per session, sessions don't survive sleep, worktrees deleted with sessions.

## Key non-obvious facts
1. `/goal`'s evaluator is **transcript-only** — no tool access. Conditions must be checkable from what Claude has surfaced. Single biggest gotcha.
2. `/goal` is implemented as a session-scoped **prompt-based Stop hook**, inheriting all hook-system trust/permission constraints (blocked by `disableAllHooks`).
3. Agent View's background sessions are hosted by a **per-user supervisor process** independent of any terminal — sessions survive shell close, not machine sleep. `claude respawn --all` revives after sleep.
4. Each agent view session auto-isolates via a git worktree under `.claude/worktrees/`. Outside a git repo there is NO isolation — parallel writes conflict.
5. Per-session rate-limit consumption — N parallel sessions = N× quota burn.
6. The row summary line is generated by a **Haiku-class model**, refreshed at most every 15s + per turn-end. Billed at session-normal rates.
7. `←` on an empty prompt from ANY session backgrounds it and opens agent view with it pre-selected — this is the muted "context-switch shortcut" of the whole feature.
8. Sessions auto-stop their process after ~1h idle to save resources; reattach starts a fresh process from saved state with slight delay.
