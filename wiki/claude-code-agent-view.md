# Agent View — One screen for all background sessions

## Summary
[[claude-code]] research-preview TUI (v2.1.139+, announced 2026-05-11) that replaces the tmux-grid pattern for running parallel Claude sessions. Opened with `claude agents` or `←` from any session. Background sessions are hosted by a **per-user supervisor process** independent of any terminal, so dispatched work continues after you close your shell. Each row shows state, last activity, and PR status; peek+reply inline, attach for full conversation, dispatch new sessions from one input. The orchestration screen above [[claude-code-goal]] (which is the per-session loop driver).

## Details

### Opening + leaving
- `claude agents` from shell.
- `←` on an empty prompt from any session — backgrounds current session AND opens agent view with it pre-selected. Disable in `/config`.
- `Esc` returns to shell; sessions keep running.

### Layout — grouped by state
- **Pinned** (`Ctrl+T`) · **Ready for review** (has open PR) · **Needs input** · **Working** · **Completed** (includes failed + stopped).
- `Ctrl+S` toggles between state grouping and directory grouping.
- Older completed sessions fold into `… N more`. Failures and PR-opening sessions always stay visible.

### Status icons — color = state, shape = process
| Color    | State       | Meaning |
|----------|-------------|---------|
| Animated | Working     | Actively running |
| Yellow   | Needs input | Awaiting question/permission |
| Dimmed   | Idle        | Ready for next prompt |
| Green    | Completed   | Done successfully |
| Red      | Failed      | Errored |
| Grey     | Stopped     | `Ctrl+X` or `claude stop` |

| Shape | Process |
|-------|---------|
| `✻` / animated `✽` | Alive, replies immediately |
| `∙` | Process exited (auto-stopped after ~1h idle); you can still peek/reply/attach — restarts on demand |
| `✢` | A `/loop` session sleeping between iterations; run count + countdown shown |

### PR status dot (right edge of row)
Yellow = waiting/failed · Green = checks passed · Purple = merged · Grey = draft/closed. Hyperlinked in terminals that support it.

### Row summary
Generated by **Haiku-class model**, refreshes at most every 15s + once per turn-end. Billed under session's normal data terms.

### Three interaction modes
1. **Peek** (`Space`) — side panel with last output, blocking question, or opened PR. Reply inline; multi-choice → press number; `!` prefix = Bash. `↑`/`↓` peek adjacent rows.
2. **Attach** (`Enter` or `→`) — full session takeover; Claude posts a recap of what happened while away. `←` on empty prompt detaches.
3. **Background** — `/bg` inside a session, or `←` on empty prompt. Detaching never stops a session. `/stop` to actually end one.

### Dispatch — three entry points
- **From agent view**: type prompt + Enter. Each prompt = its own new session (not a follow-up).
- **From inside a session**: `/bg` or `/bg <extra instruction>`. Backgrounding starts a fresh process from saved conversation; running subagents do NOT transfer.
- **From shell**: `claude --bg "<prompt>"`. Optional `--agent <name>` to run a subagent as main. Prints short ID + management commands.

### Dispatch prefixes (from input)
| Input | Effect |
|-------|--------|
| `<subagent-name> ...` | Runs that subagent as session's main agent |
| `@<subagent>` | Mention anywhere |
| `@<repo>` | Run in sibling repo under parent dir |
| `/<skill>` | Suggest a skill |
| `#<PR>` | Select existing PR-session instead of dispatching |
| `Shift+Enter` | Dispatch and attach immediately |
| `Ctrl+G` | Open prompt in `$EDITOR` |

### Filters (type in input)
- `a:<agent>` — sessions running a specific agent
- `s:<state>` — `s:working`, `s:blocked` (= anything waiting on you), etc.
- `#<PR>` or PR URL — the session on that PR

### File isolation
Every background session moves to an isolated **git worktree under `.claude/worktrees/`** before editing. Skipped only if already inside `.claude/worktrees/`, not a git repo, or writes are outside the working dir. **Outside a git repo, parallel writes conflict** — avoid. Worktree deleted with the session — commit/push first.

### Supervisor process
- Per-user process, separate from your terminal, hosts all background sessions.
- Starts on first background/dispatch; no manual management.
- Same credentials; no extra network connections beyond the model API.
- Auto-updates: watches the Claude Code binary on disk; restarts into new version preserving sessions.
- After ~1h idle a session's process stops to free resources; transcript + state persist; next interaction restarts it.

State on disk:
- `~/.claude/daemon.log`
- `~/.claude/daemon/roster.json` (running sessions, reconnect after restart)
- `~/.claude/jobs/<id>/state.json` (per-session state)

`CLAUDE_CONFIG_DIR` overrides the location — separate supervisor with its own sessions.

### Shell management
| Command | Purpose |
|---------|---------|
| `claude agents` | Open agent view |
| `claude attach <id>` | Attach in current terminal |
| `claude logs <id>` | Print recent output |
| `claude stop <id>` | Stop session |
| `claude respawn <id>` | Restart stopped session, conversation intact |
| `claude respawn --all` | Restart everything (e.g., after machine sleep) |
| `claude rm <id>` | Remove from list, clean worktree if no uncommitted changes |

### Per-session model and permission overrides
- Header shows dispatch default model.
- Override per-session: `--model` with `claude --bg`, `/model` inside attached session, or subagent frontmatter `model:` field.
- Permission mode from session's directory. `bypassPermissions` / `auto` via `--permission-mode` refused until accepted interactively at least once (guard against unwatched sessions).

### Limitations (research preview)
- **Rate limits apply per session** — 10 parallel = 10× quota burn.
- **Local execution** — sessions stop on sleep/shutdown; `claude respawn --all` to revive.
- **Worktrees deleted with session** — push or merge first.
- Prompts under 4 chars rejected (anti-stray-keystroke).
- Earlier versions failed to open in Bedrock/Vertex/Foundry — `claude update` to fix.

### Opt-out
- Setting: `disableAgentView: true`
- Env: `CLAUDE_CODE_DISABLE_AGENT_VIEW`
- Managed settings can enforce.

## Comparison with other parallel-execution primitives
| Primitive | Best for |
|-----------|----------|
| Agent view | Many independent sessions side-by-side with peek/reply/attach UI |
| Subagents | One session dispatching specialized sub-tasks within its own turn |
| Agent teams | Sessions that need to message each other |
| Worktrees | Filesystem isolation for parallel edits (used under the hood by agent view) |
| Claude Code on web | Sessions hosted on Anthropic infra, not local |

## Pattern: agent view + `/goal`

The natural pairing — dispatch many `/goal` sessions, each working toward a transcript-checkable condition; the agent-view row updates as the evaluator decides yes/no; you step in only when a row says `Needs input` or `Completed`. This is the closest Claude Code currently gets to fire-and-forget parallel agents.

```bash
claude --bg "/goal migrate src/legacy/* to async/await; tsc clean; tests pass — or stop after 25 turns"
claude --bg "/goal triage all bug-labeled issues with a root-cause hypothesis comment, until queue empty — or stop after 20 turns"
```

## Anti-patterns
- Dispatching many sessions without setting a budget — research preview's per-session quota burn compounds fast.
- Editing the same files in parallel sessions outside a git repo — no worktree isolation.
- Deleting a session before merging/pushing — worktree contents go with it.
- Treating peek as a transcript viewer — it shows last output + blocking question only; attach for full history.
- Using `←`-to-background without saving in-flight subagent work — fresh background process doesn't inherit running subagents.
