## 开篇 / 论点

写完前作之后，发现自己对 Agent 底层的理解不够深入，加上团队 Agent 业务落地经验，整理成这篇文章。覆盖：**控制流、上下文工程、工具设计、记忆、多 Agent 组织、评测、追踪和安全**，最后用 OpenClaw 实现把设计原则串起来。

整理后有几处判断改变：
- **更贵的模型带来的提升没那么大**
- **Harness 和验证测试质量对成功率影响更大**
- 调试 Agent 行为时**优先检查工具定义** —— 多数工具选错都出在描述不准
- **评测系统本身的问题，往往比 Agent 出问题更难发现**

## 1. Agent Loop 基本运转

Agent Loop 核心实现抽象后**不到 20 行代码**：感知 → 决策 → 行动 → 反馈循环，直到模型返回纯文本。

**结构稳定，新能力通常叠加在循环外部，不改循环内部。** 新能力只通过三种方式接入：
- 扩展工具集和 handler
- 调整系统提示结构
- 把状态外化到文件或数据库

> "不应该让循环体本身变成一个巨大的状态机。模型负责推理，外部系统负责状态和边界。"

### Workflow vs Agent —— Anthropic 的区分
- **Workflow**：执行路径由代码预先写死
- **Agent**：由 LLM 动态决定下一步

核心区别在**控制权掌握在谁手里**。很多标着 Agent 的产品深入看其实更接近 Workflow。两者并无高下，关键是给任务找合适的方案。

### 五种模式（大多数 AI 系统都是这五种的组合）
1. **Prompt Chaining**（提示链）：任务拆顺序步骤，每步处理上一步输出，可加代码检查点。适合生成后翻译、先写大纲再写正文。
2. **Routing**（路由）：分类输入定向到不同流程。简单走轻量模型、复杂走强模型。
3. **Parallelization**（并行）：分段法（拆独立子任务并发）+ 投票法（同任务跑多次取共识）。适合高风险决策或多视角。
4. **Orchestrator-Workers**：中央 LLM 动态分解任务，委派工作者，综合结果。nanobot 的 spawn 工具 + learn-claude-code 子 Agent 是原型。
5. **Evaluator-Optimizer**：生成器产出，评估器反馈，循环到达标。适合翻译、创意写作这类质量标准难精确定义的任务。

## 2. 为什么 Harness 比模型更关键

**Harness** = 围绕 Agent 构建的测试、验证与约束基础设施。至少四个部分：
- **验收基线**
- **执行边界**
- **反馈信号**
- **回退手段**

模型重要，但能否稳定运行往往取决于外围工程条件。在**高可验证任务**（代码编写）最成立；**弱验证任务**（开放研究、多轮协商）里模型上限本身仍更关键。

### OpenAI 的 Agent-First Development
**3 个工程师 5 个月写了百万行代码，将近 1500 个 PR，传统开发速度的 10 倍。** 不是模型有多强，是工程决策做对了：

- **知识必须存在于代码库本身** —— 外部文档对运行中的 Agent 不可见
- **AGENTS.md 只保留约 100 行作为索引**，细节拆到各 docs 目录按需引用

## 3. 上下文工程

### 四层结构 (防 Context Rot)
- **常驻层**：系统提示、工具定义、长期不变的规则
- **按需层**：Skills、工具结果（延迟加载）
- **运行时**：当前时间、渠道 ID、用户偏好（每轮按需拼）
- **记忆**：跨会话经验写入 MEMORY.md，需要时读取

**凡是可以通过 Hooks、代码规则或工具约束表达的内容，都应交给外部系统处理，而不是让模型反复读取。**

### 上下文压缩三种方式
1. **滑动窗口**：丢弃旧消息，成本极低，丢早期上下文。适合简短对话
2. **LLM 摘要**：模型生成总结，成本中等，丢细节保留决策。适合长任务
   - 进阶：**branch summarization** —— 明确保留架构决策、未完成任务、关键约束
3. **工具结果替换**：占位符替换原始输出，成本极低，适合工具调用密集型
   - `micro_compact` 每轮替换旧工具输出
   - `auto_compact` 上下文超阈值自动触发

### Prompt Caching 机制
KV cache 命中前提是**精确前缀匹配**，不是相似就能触发，任何 token 不同都会破坏匹配。**缓存友好的设计核心是稳定性。**

- 系统提示、工具定义、长文档：天然适合缓存
- 动态信息（当前时间、用户输入、工具结果）放后面，不影响前缀稳定性
- Skills 延迟加载：按需注入不破坏系统提示前缀，而是追加在稳定前缀之后
- **反直觉：稳定的大系统提示比频繁变动的小提示实际成本更低** —— 写入成本付一次，后续读取折扣可达 90%

### Skills 设计

**核心思路：**
- Skill 描述足够短（避免常驻 token 持续涨）
- 像**路由条件**而不是功能介绍：什么时候用、什么时候不要用、产出物是什么
- 直接写法：**Use when / Don't use when** + 几条反例

很多路由失败不是模型能力问题，**是边界写得不清楚**。

调用规则在系统提示里写明确：
- 每次回复前先扫描 available_skills
- 有明确匹配再读对应 SKILL.md
- 多个匹配优先选最具体的
- **一次只加载一个**

**反例（anti-examples）不是可选项**：没有反例时准确率从 73% 掉到 53%，加上反例后升到 85%，响应时间还降了 18.1%。

**两个写法陷阱：**
1. **字数**：描述符长度对路由准确率差距不大，但每个启用的 Skill 描述符**常驻上下文**，Skill 一多累积成本大
2. **精度**：描述太短（"help with backend"）等于任何后端工作都能触发，路由会乱。**"何时该用我" 比 "我能做什么" 重要得多。**

**Skills 三个反模式：**
- 工作手册全塞进 Skill 正文而不是拆 supporting files
- 一个 Skill 试图覆盖 review、deploy、debug、incident 五件事
- 有副作用的 Skill 没有显式限制调用时机

### Skills vs MCP
- MCP 把完整结果直接返回，更容易吃掉上下文预算
- CLI + 单句描述的 Skill 更接近模型熟悉的调用方式，可过滤、可拼接
- MCP 适用场景：**Playwright 这类需要维护状态的任务**

## 4. 长任务 —— Initializer Agent + Coding Agent

长任务最常见的失败：**session 结束时任务还没做完**，compaction 也挡不住两类问题：
- 单 session 上下文先耗尽
- 只做完一部分，下轮无法准确恢复，过早判断完成

**更稳定的做法**：拆 Initializer Agent + Coding Agent 协作。适合代码生成、应用搭建、重构迁移这类单 session 做不完但可拆成一批可验证子任务的工作。

- **Initializer Agent**：第一轮运行一次，生成 `feature-list.json`、初始 git commit、`claude-progress.txt`。**先把任务变成可持久化的外部状态**
- **Coding Agent**：循环执行多个 session，每次从 `claude-progress.txt` + git log 恢复现场，定位当前任务，实现功能，跑测试，更新 passes 字段，提交退出

**进度放文件里，不放上下文里。** 功能清单用 JSON 不用 Markdown（结构化更适合模型稳定修改）。**feature-list.json 里所有功能 `passes: true` 时任务才算完成。**

### 任务状态显式记录为外部控制对象
- 同一时间只能有一个 `in_progress`
- 每完成一步先更新状态再继续
- 必要时加轻量校正（连续多轮未更新自动注入 `<reminder>`）

### 异步 subprocess + 通知队列
不要把整个 loop 改造成 async runtime。**慢速 subprocess 放后台线程，通过通知队列在下一轮 LLM 调用前注入结果。** 主循环每轮开始前检查新结果。

## 5. 多 Agent

### 指挥者 vs 统筹者 (两种工作模式)
- **指挥者模式**：同步协作，人与单个 Agent 紧密互动，每轮调决策。session 结束 context 就没了，产出物短暂
- **统筹者模式**：异步委派，人在开始时设目标，多个 Agent 并行，最后审查产出。**人只在起点和终点出现，中间产出变成分支、PR 这类可持久化工件**

**多 Agent 主要价值不在多开模型，而在把人的持续参与变成对工件的最终审核。**

### 三样东西必须先有 (协议、任务图、隔离边界)
- **JSONL inbox 协议** 主 Agent → 子 Agent 派发，子 Agent 只回摘要
- **`.tasks/`** 记录任务图和依赖关系
- **`.worktrees/`** 隔离每个子 Agent 的文件修改

**顺序：协议先定，隔离先做，再谈协作和并行。**

### 错误放大与交叉验证
多 Agent 频繁互动时错误层层放大：Agent A 偏 → B 强化 → C 叠加 → 所有 Agent 收敛到同一个高置信度的错误结论。

**交叉验证**让某个 Agent 独立判断（不顺着前面结论走）。顺序：先有可持久化任务图 → 再有身份队友 → 再有结构化协议 → 最后加交叉验证或外部反馈（独立第二 Agent、单元测试、编译器或人工审查）。

### 子 Agent 两个基本限制
- **深度限制**：防止无限递归生成孙 Agent
- **最小系统提示**：只给 Tooling / Workspace / Runtime 三节，**不带 Skills 和 Memory**（避免权限外泄 + 保隔离）

## 6. 评测

**评测核心 = 测试用例 + 评分标准 + 自动验证。** 难点不是有没有分数，是分数能不能反映真实质量。

### 三组概念要记
- **task / trial / grader** —— 测什么、跑多少次、怎么打分
- **transcript（完整执行记录）vs outcome（环境最终结果）** —— 评测不能只看一边
- **agent harness（被评测的运行框架）vs evaluation harness（评测基础设施）**

### Pass@k vs Pass^k —— 不能混用
- **Pass@k** 适合开发阶段回答"理论上能不能做到"
- **Pass^k** 适合上线前回答"已有功能有没有被改坏"

混用容易误判：回归测试过松漏问题，能力评测过严让每次小改动都告警。

### 三种评分器 (确定性递减)
1. **代码评分器**：字符串匹配、单测、结构比对。确定性最高。
2. **模型评分器**：按 rubric 打分、答案对比、多模型投票
3. **人工评分器**：专家抽样、标注校准。可靠但慢，适合建立基准

**有明确答案优先用代码评分器。**

### Transcript vs Outcome (Anthropic 机票预订例子)
Opus 4.5 在一次运行中发现航空公司政策漏洞，为用户找到更便宜方案。**按预设路径打分会判失败，按最终结果是更好方案。** 两类都覆盖才能看清楚。

### Bootstrap 评测
- **20-50 个真实失败案例**就够启动
- 来源优先选已经在手动检查的内容
- 判断标准：**两个领域专家拿同一个案例独立判断，结论不一致 → 验收标准还没写清楚，先解决定义**

### 评测系统自己的问题 (容易和模型退化混淆)
- 运行环境资源不足导致进程被杀
- 评分器本身有 bug 把正确判成失败
- 测试用例和生产场景脱节
- 只看聚合分数漏掉某类任务系统性变差

**看到评测分数下降，先查环境，再动 Agent。**

### 评测套件饱和 = 不能反映真实能力边界
"当通过率接近 100% 时补充更难的任务" 当常规工作。

## 7. 可观测性

**Trace 是排查前提。** 没有完整记录，失败案例无法稳定复现。传统只监控延迟和错误率的 APM 帮助有限 —— 接口层一切正常，问题出在模型某轮的错误决策。

### 两层评测
1. **人工抽样标注** —— 基于规则采错误案例、长对话、用户负反馈
2. **LLM 自动评估** —— 以第一层为校准基准

**两层必须一起用。**

### 在线评测采样策略 (10-20% Trace，按规则路由，不随机)
- **用户明确不满意**：100% 进队列
- **Token 消耗超阈值**：往往代表 Agent 在绕圈子
- **每天固定时间段随机采**：保持对正常流量覆盖

### 事件流做底座
Agent Loop 在 `tool_start`、`tool_end`、`turn_end` 三个节点发出事件。**完整 Trace 同步落盘，再分发给日志系统、UI 更新、在线评测、人工审查队列。一次发布多路消费，主循环不需要为下游改代码。**

## 8. 用 OpenClaw 看 Agent 如何落地

### 五层架构 (从上到下)
1. **Gateway**：WebSocket 服务，统一路由消息
2. **Channel 适配器**：23+ 渠道统一接口，新增渠道不改 Agent 代码
3. **Pi Agent**：维护主循环、会话状态、调度
4. **工具集**：shell / fs / web / browser / MCP，按 ACI 原则设计
5. **上下文 + 记忆**：Skills 延迟加载 + MEMORY.md，50% token 阈值自动整合

加定时任务后系统不再只有用户消息一个入口 → **MessageBus 在渠道和 Agent 之间加一层**。Channel 只管收发，AgentLoop 只管处理。

### 关键工程决策
- `dispatch` 不做 `await` —— 不同 session 的消息并发处理
- **同一 session 内消息必须串行**（否则并发写历史和触发 compact 有竞态）
- 每个 sessionKey 维护队列或 mutex
- **session 由 AgentLoop 统一管理，不下沉到 Channel 层** —— 换 Discord / Lark / 飞书核心代码不动

### 系统提示分层加载
- 顺序从下到上：平台与运行时信息 → 身份层 → 记忆层 → Skills 层 → 运行时注入
- 对应文件：`SOUL.md` / `AGENTS.md` / `TOOLS.md` / `USER.md` / `MEMORY.md` + Skills 索引

### 三种触发模式
- **普通会话**：完整系统提示
- **子 Agent**：只基础运行时信息，不带记忆和 Skills
- **heartbeat**：单独加载 `HEARTBEAT.md`，每 5 分钟轮询待处理任务

**长任务额外加一行身份重申压住任务漂移。**

### Cron + Heartbeat
- **cron**：按计划直接触发 Agent
- **heartbeat**：每 5 分钟轮询待处理任务
- 都不等用户发消息

### 崩溃恢复 (任务超过半小时 = 必选项)
任务进度写到磁盘，重启后从断点继续。

## 9. 安全

### Shell 权限三件事必须先到位
- 谁能用、能在哪用、做了什么可追踪
- Shell 工具强制路径检查，越出工作空间直接报错

### Prompt Injection: source-sink 防御
单靠输入过滤挡不住。**重点不是识别所有攻击，而是让 Agent 即使被注入也没机会把危险动作执行出来：**

- **不给 Agent 不需要的工具**（没 sink，source 注入也无法落地）
- **向第三方传信息 / 写操作执行前必须用户确认**
- **外部内容进入上下文时显式标注来源**，声明哪些不可信
- **关键操作引入独立 LLM 复核**

### 模型服务 fallback
Anthropic 503 / OpenAI 限速很常见 → Provider 挂了自动切下一个。

## 10. 八个常见反模式

1. **系统提示当知识库** → 关键规则被忽略。约定留提示，**知识移 Skills**
2. **工具数量失控** → 频繁选错。**合并重叠工具，明确命名空间**
3. **验证闭环缺失** → 说完成了没法验证。**每类任务绑验收标准**
4. **多 Agent 无边界** → 状态漂移，故障归因难。**明确角色权限，worktree 隔离**
5. **记忆不整合** → 第 20 轮决策质量下降。**监控 token，超阈值自动触发整合**
6. **没有评测** → 改地方不知道有没有回归。**失败案例立刻转测试用例**
7. **过早引入多 Agent** → 协调开销超并行收益。**先验证单 Agent 上限再扩展**
8. **约束靠期望不靠机制** → 规则在文档里 Agent 选择性遵守。**改用工具验证 / Linter / Hook**

## 总结要点

- Agent 核心 = 感知/决策/行动/反馈稳定循环
- Harness（验收基线、执行边界、反馈信号、回退手段）往往比模型本身更决定收敛
- 上下文工程重点是防 Context Rot
- 工具设计按 ACI 原则
- 记忆分四类：工作记忆、程序性记忆、情景记忆、语义记忆
- 长任务靠 Initializer + Coding Agent + 文件系统状态
- 多 Agent：协议先于协作
- 评测：Pass@k 验证能力边界，Pass^k 保证上线质量
- 可观测性：Trace 是前提，两层评测一起用
- **真正让 Agent 跑稳的不是更复杂的循环，是消息解耦、状态外化、分层提示、记忆整合和安全边界这些工程细节**
