## Core Concept

Ralph is a looping technique for autonomous code generation using LLMs. At its simplest:

```bash
while :; do cat PROMPT.md | claude-code ; done
```

The technique treats AI as a monolithic, single-process agent that iteratively implements features one task per loop. Huntley describes it as having "deterministically bad" behavior in an unpredictable world, requiring skilled operator guidance rather than tool blame.

## Key Technical Principles

**One Thing Per Loop**: Each iteration addresses exactly one feature. This conserves context window (~170k tokens) and prevents the agent from diverging.

**Context Stack Allocation**: Every loop maintains consistent files:
- `@fix_plan.md` (prioritized todo list)
- `@specs/*` (technical specifications)
- `@AGENT.md` (build/execution instructions)

**Subagent Parallelization**: The main agent schedules multiple subagents for parallel work (up to 500 for research, but only 1 for validation/testing) to avoid context bloat.

**Deterministic Backpressure**: Each loop includes immediate testing and validation, with feedback loops ensuring the agent self-corrects through successive iterations.

## Practical Results

A case study showed a $50,000 contract delivered for $297 using this method. The technique has successfully built CURSED, a new production-grade programming language with compiler, standard library, and LLVM integration — all generated by AI without training data for that language.

## Implementation Strategy

**Phase One - Generation**: Code creation guided by specifications and standard libraries (ensuring consistent patterns).

**Phase Two - Validation**: Immediate testing through:
- Unit tests for changed code
- Static analysis (Dialyzer, Pyright for dynamic languages)
- Compilation checks
- Custom security/performance scanners

**Iterative Refinement**: When Ralph fails, the operator "tunes" by adding instructional signs (prompt refinements) rather than blaming the tool.

## Critical Implementation Details

**Search Before Assuming**:
```
Before making changes search codebase (don't assume not implemented) using subagents. Think hard.
```

**Capture Test Intent**: Document why tests matter for future loops lacking that context.

**Avoid Placeholders**:
```
DO NOT IMPLEMENT PLACEHOLDER OR SIMPLE IMPLEMENTATIONS. WE WANT FULL
IMPLEMENTATIONS. DO IT OR I WILL YELL AT YOU
```

**Self-Learning**: The agent updates `@AGENT.md` when discovering better build/execution approaches.

## Prompt Example (CURSED Compiler)

The current production prompt directs Ralph to:
- Study specifications and source code
- Implement 10 prioritized missing features per loop
- Run tests immediately after changes
- Update `@fix_plan.md` with learnings
- Commit only when tests pass
- Create git tags on successful builds

## Limitations & Honest Assessment

Huntley emphasizes Ralph works best for **greenfield projects** only. Retrofitting existing codebases is problematic. The codebase will contain garbage, temporary files, and "unspecified latent behaviours" until production release.

Senior engineer oversight remains essential — the technique amplifies skilled guidance but doesn't eliminate the need for experienced practitioners. Success requires understanding why Ralph fails and adjusting prompts accordingly, not abandoning the approach.

## Success Metrics

- Achieves ~90% completion through automated loops
- Dramatically reduces timeline (weeks of engineer output in hours)
- Maintains code correctness through continuous validation
- Creates self-documenting specifications that guide future iterations

## Philosophy

"Tune like a guitar" — When Ralph misbehaves, add instructional signs (prompt refinements) rather than blaming the tool. The operator's skill in prompt engineering and specification writing is the critical success factor.
